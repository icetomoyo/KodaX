## Version Context

### Origin

v0.7.18 的规划来源于一次系统性的 KodaX vs Claude Code 全面代码对比分析 (2026-04-11)。

对比结论：

> **KodaX 在"智能内核"层面（AMA 任务路由、Repo Intelligence、多 Provider、结构化管理协议）已超越 Claude Code；但在"工程外壳"层面（Hook 丰富度、Cost 追踪、MCP 协议完整度、Permission 精细度）存在真实差距。**

v0.7.18 的定位是：**在不违背极简哲学的前提下，补齐工程外壳的关键能力。** 位于 v0.7.30（TUI/Memory 内部架构成熟）和 v0.8.0（Protocol V2 / Search / 压缩锚）之间。

### Design Principles

所有 v0.7.18 Feature 必须遵守：

1. **管道优先于功能** — KodaX 建管道，用户建自动化。不加 60 个命令，加 1 个 hook 管道让用户自己接。
2. **多 Provider 差异化** — 凡是与 Provider 无关的能力，参考但不复刻 Claude Code；凡是 KodaX 多 Provider 独有的能力，优先投入。
3. **增量而非重写** — 每个 Feature 在现有架构上增量构建，不引入新的包或新的重型依赖。
4. **3+ use cases 才抽象** — 严格遵守 CLAUDE.md 的抽象门槛。

## FEATURE_063: Extensible Hook & Automation Substrate

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

当前 KodaX 有 4 个内部 hook 事件 (`tool:before`, `provider:before`, `turn:settle`, `session:hydrate`)，仅支持内部函数调用。Claude Code 有 10+ 事件类型 × 4 种执行方式 (shell/LLM/HTTP/agent)。

FEATURE_063 将 hook 系统从"内部扩展点"升级为"用户可配置的自动化管道"，使用户能通过 `settings.json` 声明 hook 规则，由 KodaX 运行时执行。

### Motivation

1. **CI/CD 集成**：用户希望在工具执行后自动运行 linter、formatter、type checker
2. **Webhook 通知**：团队希望在 session 结束时发送 Slack/飞书通知
3. **安全审计**：用户希望在 bash 执行前做风险检查
4. **对比差距**：Claude Code 的 hook 系统是其自动化工作流的核心基础设施

### Goals

1. 支持 **3 种执行方式**：`command` (shell)、`http` (webhook)、`prompt` (LLM 验证)
2. 事件类型从 4 个扩展到 **8 个**，覆盖完整的工具和 session 生命周期
3. 用户通过 `~/.kodax/settings.json` 或项目级 `.kodax/settings.json` 声明 hook
4. hook 执行结果可影响工具执行（PreToolUse 返回 deny 可阻止工具）
5. hook 超时和失败不阻塞主流程（fail-open with warning）

### Non-goals

- 不实现 `agent` 类型 hook（需要 FEATURE_059 Protocol V2 支持，留 v0.8.0）
- 不实现 SSRF 防护（KodaX 面向开发者，不是 SaaS 平台）
- 不实现条件执行语法（if 条件，留后续按需）
- 不修改已有的 4 个内部 extension hook（它们继续服务内部扩展）

### Event Types

| Event | Trigger | Execution Direction |
|---|---|---|
| `PreToolUse` | 工具执行前 | 可 allow/deny/modify input |
| `PostToolUse` | 工具执行成功后 | fire-and-forget |
| `PostToolUseFailure` | 工具执行失败后 | fire-and-forget |
| `SessionStart` | session 初始化完成 | fire-and-forget |
| `SessionEnd` | session 正常结束 | fire-and-forget |
| `Compact` | context 压缩前 | fire-and-forget |
| `PromptSubmit` | 用户提交 prompt 后 | 可 modify prompt |
| `Stop` | agent 停止前 | fire-and-forget |

### Configuration Format

```jsonc
// ~/.kodax/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "matcher": "edit|write",           // 匹配工具名 (regex)
        "command": "npx prettier --write $FILE_PATH",
        "timeout": 10000
      }
    ],
    "PreToolUse": [
      {
        "type": "command",
        "matcher": "bash",
        "command": "echo $TOOL_INPUT | node ./scripts/check-bash-safety.js",
        "timeout": 5000
        // exit 0 = allow, exit 1 = deny
      }
    ],
    "SessionEnd": [
      {
        "type": "http",
        "url": "https://hooks.slack.com/services/xxx",
        "method": "POST",
        "headers": { "Content-Type": "application/json" },
        "body": "{\"text\": \"KodaX session ended: $SESSION_ID\"}"
      }
    ],
    "Stop": [
      {
        "type": "prompt",
        "prompt": "Review the changes made in this session. Are there any security concerns?",
        "model": "auto"
      }
    ]
  }
}
```

### Hook Execution Types

#### `command` — Shell 命令

```typescript
interface CommandHook {
  type: 'command';
  matcher?: string;          // 工具名 regex (仅 PreToolUse/PostToolUse)
  command: string;           // shell 命令, 支持环境变量插值
  timeout?: number;          // 默认 30000ms
  shell?: 'bash' | 'powershell';  // 默认跟随系统
}
```

环境变量注入：
- `$TOOL_NAME` — 工具名
- `$TOOL_INPUT` — 工具输入 (JSON string)
- `$TOOL_OUTPUT` — 工具输出 (仅 PostToolUse)
- `$FILE_PATH` — 涉及的文件路径 (edit/write/read)
- `$SESSION_ID` — 当前 session ID
- `$WORKING_DIR` — 当前工作目录

返回值语义 (仅 PreToolUse)：
- exit code 0 → allow
- exit code 1 → deny (stderr 作为 deny reason)
- exit code 2+ → pass (不影响, 记录 warning)

#### `http` — Webhook

```typescript
interface HttpHook {
  type: 'http';
  url: string;
  method?: 'POST' | 'PUT';   // 默认 POST
  headers?: Record<string, string>;
  body?: string;              // 支持环境变量插值, 默认发送标准 event payload
  timeout?: number;           // 默认 10000ms
}
```

标准 event payload：
```json
{
  "event": "PostToolUse",
  "toolName": "edit",
  "sessionId": "abc123",
  "timestamp": "2026-04-11T10:00:00Z",
  "data": { /* event-specific */ }
}
```

#### `prompt` — LLM 验证

```typescript
interface PromptHook {
  type: 'prompt';
  prompt: string;             // 发送给 LLM 的验证 prompt
  model?: string;             // 'auto' = 当前 provider, 或指定 provider name
  timeout?: number;           // 默认 60000ms
}
```

LLM 返回值语义 (仅 PreToolUse)：
- 响应包含 "ALLOW" → allow
- 响应包含 "DENY" → deny (响应文本作为 reason)
- 其他 → pass

### Architecture

```
User settings.json
    │
    ▼
┌──────────────────┐
│  HookRegistry    │  加载 + 验证 hook 配置
│  (packages/      │  按 event type 索引
│   coding/src/    │  按 matcher regex 过滤
│   hooks/)        │
└──────────────────┘
    │
    ▼
┌──────────────────┐
│  HookExecutor    │  执行 command/http/prompt
│                  │  超时控制 + 错误隔离
│                  │  环境变量插值
└──────────────────┘
    │
    ▼
┌──────────────────┐
│  HookResult      │  { action: 'allow'|'deny'|'pass', reason?, modifiedInput? }
└──────────────────┘
```

集成点：
- `agent.ts` → SessionStart, SessionEnd, Stop, Compact
- tool execution pipeline → PreToolUse, PostToolUse, PostToolUseFailure
- REPL input handler → PromptSubmit

- [ ] 用户在 `settings.json` 声明的 hook 在对应事件触发时执行

- [ ] 环境变量插值覆盖所有文档化的变量
- [ ] 80%+ 测试覆盖率

## FEATURE_064: Multi-Provider Cost Observatory

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

KodaX 支持 11 个 LLM Provider，但用户无法得知每次对话的实际 token 消耗和成本。Claude Code 有完整的 cost tracking（按模型费率计算 session 成本）。对于多 Provider 的 KodaX，cost tracking **比单 Provider 的 Claude Code 更加重要** — 用户需要比较不同 Provider 的性价比。

### Motivation

1. **Provider 选择依据**：用户在 zhipu-coding、kimi-code、anthropic 之间切换时，缺乏成本数据支撑
2. **AMA 成本感知**：H2 的 Planner→Generator→Evaluator 链条消耗远大于 H0 直达，用户需要可见性
3. **预算控制**：长 session 可能积累大量 token 消耗，用户需要预警
4. **对比差距**：Claude Code 有 session 级成本聚合，KodaX 没有

### Goals

1. 每次 LLM 调用记录 input/output/cache tokens 和对应成本
2. session 级成本聚合，在 status bar 实时显示
3. 内置主流 Provider 的费率表（可通过配置覆盖）
4. `/cost` 命令查看详细的 session 成本报告
5. session 结束时在 summary 中包含成本信息

### Non-goals

- 不实现账单管理或支付功能
- 不实现跨 session 的历史成本分析
- 不实现实时费率更新（内置费率表 + 用户配置覆盖足矣）
- 不实现预算硬限制（用户自行决策）

### Design

#### Cost Rate Table


#### Cost Tracker

```typescript
// packages/ai/src/cost-tracker.ts

interface TokenUsageRecord {
  readonly timestamp: number;
  readonly provider: string;
  readonly model: string;
  readonly inputTokens: number;
  readonly outputTokens: number;
  readonly cacheReadTokens: number;
  readonly cacheWriteTokens: number;
  readonly cost: number;           // USD
  readonly role?: string;          // 'scout' | 'planner' | 'generator' | 'evaluator' | undefined
}

interface SessionCostSummary {
  readonly totalCost: number;
  readonly totalInputTokens: number;
  readonly totalOutputTokens: number;
  readonly totalCacheTokens: number;
  readonly byProvider: Record<string, { cost: number; calls: number }>;
  readonly byRole: Record<string, { cost: number; calls: number }>;
  readonly records: readonly TokenUsageRecord[];
}

// Immutable cost tracker
function createCostTracker(): CostTracker;
function recordUsage(tracker: CostTracker, record: Omit<TokenUsageRecord, 'cost'>): CostTracker;
function getSummary(tracker: CostTracker): SessionCostSummary;
```

#### Integration Points

1. **Provider stream 回调**：每次 `stream()` 完成后，从 `KodaXStreamResult.usage` 提取 token 数并记录
2. **Status bar**：实时显示 `$X.XX` (累计成本)
3. **`/cost` 命令**：详细报告（按 Provider、按角色、按时间）
4. **Session end summary**：在 session 结束输出中包含成本概要

```
    ▼

    │
```

## FEATURE_065: MCP Protocol Maturity

**Status**: Partially Implemented (v0.7.16 基础已完成，剩余部分在 v0.7.18 交付)
**Priority**: Medium
**Category**: Enhancement

### Overview

KodaX 的 MCP 实现在 v0.7.16 完成了大量基础工作，现已具备完整的 stdio/SSE/Streamable HTTP 传输层、工具发现与调用链路、自动 framing 协商、以及内置工具失败时的 MCP fallback 机制。

剩余的 FEATURE_065 工作聚焦在：**OAuth 认证**（连接需要认证的 MCP server）、**Elicitation 协议**（MCP server 向用户请求输入）、以及 **ACP 链路接入**（#108）和若干收尾项（#109-#111）。

### v0.7.16 已完成的 MCP 基础工作

| 改动 | 说明 |
|------|------|
| 传输抽象层 (`transport.ts`) | `McpTransport` 接口 + stdio/SSE/Streamable HTTP 三种实现 |
| stdio 自动帧格式协商 | Content-Length → NDJSON fallback，兼容 JS 和 Python MCP SDK |
| 配置扁平化 | `mcp.servers` → `mcpServers`，`type` 可选（默认 stdio），去掉 `cacheDir` |
| 冗余参数修复 | `tools/call`/`prompts/get`/`resources/read` 移除非标准字段 |
| server→client 请求回复 | 对不支持的方法回复 `-32601` 而非静默丢弃 |
| 并行 prewarm | `Promise.allSettled` 替代串行 |
| 连接串行化 | `connectPromise` 锁 + `resetTransport()` 内部方法，防止并发踩踏 |
| 无 MCP 时工具过滤 | 没有配置 MCP 时，MCP 工具不暴露给模型 |
| Task Engine 白名单 | Scout/Planner/Evaluator/Generator 的 `allowedTools` 包含 MCP 工具 |
| 执行层 `!!runtime` 传递 | `getRuntimeActiveToolNames` 所有调用点都正确传递 `hasCapabilityRuntime` |
| Prompt context 增强 | 列出每个 server 的工具名和摘要（上限 10 个），引导模型使用 MCP 替代 |
| MCP fallback | 内置只读工具失败时自动搜索同名 MCP 工具并调用（白名单限制） |
| EPIPE 处理 | stdio stdin error handler，framing 切换不崩溃 |

### 剩余工作（v0.7.18 交付）

### Motivation

1. **OAuth 是门槛**：越来越多的 MCP server 要求 OAuth 认证（GitHub MCP、Slack MCP 等），没有 OAuth 就连不上
2. **Elicitation 是交互完整性**：MCP server 可能需要用户提供额外信息（API key、确认操作等），没有 elicitation 就无法完成这些交互
3. **#91 部分覆盖**：这个 feature 不解决"一等公民工具体系"问题（那是 FEATURE_049 的范畴），但补齐了协议层的前提条件

### Goals

1. MCP server 配置支持 OAuth 2.0 认证（Authorization Code + PKCE flow）
2. OAuth token 持久化在 `~/.kodax/mcp-tokens/` 目录
3. Token 自动刷新（refresh_token flow）
4. MCP elicitation 请求转发到用户（通过 REPL ask_user_question 通道）
5. 服务器配置支持 `auth` 字段声明认证方式

### Non-goals

- 不实现 MCP server 策略过滤（无企业需求）
- 不实现 MCP server 官方注册表（KodaX 不维护中心化注册表）
- 不实现 XAA/IdP 企业认证
- 不改变现有 MCP 工具的用户接口（mcp_search, mcp_call 等）

### OAuth Flow

```

    │
    ▼
```

## FEATURE_066: Permission Hardening

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### Overview

KodaX 当前有 3 个权限模式 (plan, accept-edits, auto-in-project)，基本覆盖日常使用。但对比 Claude Code 的 13 步决策管线后，有两个具体能力是安全底线级别的缺失：

1. **Bash 命令风险分类** — 区分 `git status`（安全）和 `rm -rf /`（危险），而非一刀切地对所有 bash 命令要求确认
2. **拒绝追踪** — 用户拒绝了一个工具后，不应在同一 session 中反复提示同类操作

FEATURE_066 只补这两个点位，不重建整个权限系统。

### Motivation

1. **用户体验**：`accept-edits` 模式下每个 bash 命令都要确认，但 `git log`、`npm test` 这类命令完全安全。用户不得不频繁按确认，打断心流。
2. **安全下限**：`auto-in-project` 模式下所有项目内操作自动执行，包括 `rm -rf`。缺少风险分类意味着全开或全关，没有中间地带。
3. **重复提示**：用户拒绝了 `bash: rm -rf node_modules` 后，agent 可能立即再次尝试同一命令，造成循环提示。

### Goals

1. Bash 命令按风险分为 3 级：`safe`（自动执行）、`normal`（遵循当前模式）、`dangerous`（总是要求确认）
2. 内置 safe/dangerous 命令模式表，用户可通过 `settings.json` 扩展
3. 拒绝追踪：同一 session 内，同类操作被拒绝后不再重复提示（最多 1 次相同请求）
4. 拒绝原因注入到 agent context（让 LLM 知道用户拒绝了什么，避免重试）

### Non-goals

- 不改变权限模式的种类（仍然是 3 个）
- 不实现 transcript classifier（auto 模式的智能决策，留后续）
- 不实现复杂规则语法（`Bash(git *)`，过度工程）
- 不实现权限持久化到磁盘（session 级别足矣）

## FEATURE_067: Parallel Task Dispatch

**Status**: Planned
**Priority**: High
**Category**: Enhancement
**Related Issue**: #92 (最小可用切片)

### Overview

KodaX 的 `runOrchestration` + `createKodaXTaskRunner` 基础设施已完备，但用户没有入口发起"多个独立任务并行执行"。#92 的核心痛点不是缺少编排引擎，而是缺少**用户入口**。

FEATURE_067 是 #92 的最小可用切片：让 Scout 识别可并行的子任务，通过已有的 orchestration 基础设施并行派发，聚合结果返回用户。

### Motivation

1. 用户常有"同时修这三个 bug"、"并行跑测试和更新文档"这类需求
2. AMA 目前是串行单任务——即使 H2 的 Planner 发现子任务可并行，也只能串行交给 Generator
3. `runOrchestration` 和 `createKodaXTaskRunner` 已经能并行运行 KodaX worker，但没有用户层面的触发路径
4. Claude Code 的 Coordinator 模式解决了这个问题，但用了更重的架构

### Goals

1. Scout 阶段识别可并行的独立子任务（基于任务描述的语义分析）
2. 通过 `runOrchestration` 并行派发子任务到独立 KodaX worker
3. 聚合各 worker 结果，返回统一的用户报告
4. 复用已有 AMA 基础设施，**不引入新的编排层**

### Non-goals

- 不实现跨 worker 通信（worker 之间完全隔离，和 Claude Code Coordinator 一样）
- 不实现 worker 状态流式可视（留 v0.8.0 的 Task UI）
- 不实现用户手动指定任务拆分（Scout 自动判断）
- 不支持有依赖关系的任务链（只支持独立并行）

### Design

#### 触发路径

```
用户: "修复 auth.ts 的类型错误，同时更新 README 的安装说明，顺便跑一下测试"
    │
    ▼
Scout 分析 → 识别 3 个独立子任务
    │
    ▼
Scout 产出 parallel dispatch directive:
  tasks:
    - "修复 auth.ts 的类型错误"
    - "更新 README 的安装说明"
    - "跑测试"
    │
    ▼
runOrchestration([
  { id: 'task-1', exec: createKodaXTaskRunner(task1) },
  { id: 'task-2', exec: createKodaXTaskRunner(task2) },
  { id: 'task-3', exec: createKodaXTaskRunner(task3) },
])
    │
    ▼
聚合结果 → 用户报告
```

#### Scout 判断标准

Scout 将任务拆分为并行子任务的条件：
- 子任务之间**无文件依赖**（不修改同一文件）
- 子任务之间**无逻辑依赖**（结果不影响彼此）
- 每个子任务可独立完成

如果不满足并行条件，Scout 走正常 H0/H1/H2 路径（不强制并行）。

#### 结果聚合

```typescript
interface ParallelDispatchResult {
  readonly tasks: readonly {
    readonly description: string;
    readonly status: 'completed' | 'failed';
    readonly summary: string;
    readonly artifacts: readonly string[];  // 修改的文件
  }[];
  readonly overallSummary: string;
}
```

## FEATURE_070: Context Engine V2 — Multi-Layer Compaction & Post-Compact Reconstruction

**Status**: Planned
**Priority**: High
**Category**: Enhancement
**Absorbs**: 069-C (Microcompaction integration — from the same context-management domain)

### Overview

KodaX 的上下文压缩已有两阶段流程（pruning → rolling summary），但存在几个实际问题：
1. Microcompact 代码已写好但从未集成到 agent loop
2. 压缩后不重建关键上下文——模型不记得操作过哪些文件
3. 摘要提示缺少预分析步骤，具体技术细节容易丢失
4. 压缩失败时回退逻辑粗暴（一刀切删 10% 消息）

FEATURE_070 针对性解决这些问题，不改变核心压缩架构，在现有流程上增量增强。

### Research Context

本 Feature 基于三方对标分析：

**Factory.ai — [Evaluating Compression](https://factory.ai/news/evaluating-compression) (2026-04)**
- Probe-based 评估框架：压缩后用 Recall/Artifact/Continuation/Decision 四类探针衡量信息保留
- **核心发现：Artifact tracking 是所有系统的最薄弱环节 (2.45/5.0)**——不能靠摘要追踪文件操作，需要 agent-level scaffolding
- 具体技术细节（endpoint path、error code）在通用摘要中最先丢失
- 优化目标应为 tokens-per-task 而非 tokens-per-request

**Claude Code CLI (c:\Works\claudecode)**
- Analysis 草稿块：`<analysis>` 预分析后剥离，零 token 成本提升摘要质量
- 后压缩文件内容注入：top-5 文件重新读取并注入
- 熔断器：3 次连续失败禁用 auto-compact
- NO_TOOLS 强制约束：防止摘要模型尝试调用工具

### Motivation

| 问题 | 证据 | 影响 |
|------|------|------|
| Microcompact 不生效 | `microcompact()` 在 agent.ts 中无调用 | 上下文增长无缓冲，突然触发全量压缩 |
| 压缩后不知道操作过什么文件 | Factory.ai Artifact 维度全线 2.19-2.45/5.0 | 模型不得不重新 read 已处理过的文件 |
| 摘要丢失精确技术细节 | Factory.ai 案例：endpoint path 在 Anthropic 摘要中丢失 | 模型重复搜索同样的错误 |
| 失败回退过于粗暴 | agent.ts:1576 — `Math.ceil(messages.length * 0.1)` | 随机丢失关键上下文 |

### Goals

1. Microcompact 生效——上下文增长从陡峭阶梯变为平缓斜率
2. 压缩后模型仍知道操作了哪些文件及其状态
3. 摘要保留精确的路径、错误码、配置值
4. 压缩失败时优雅降级而非随机丢弃
5. 用户消息（意图）在压缩中得到一等保护

### Non-goals

- 不改变触发阈值体系（triggerPercent / protectionPercent 维持现状）
- 不自动融合 Smart-Context 双轨制（保持手动流程）
- 不引入固定 token 硬编码（全部用比例控制）
- 不建 Probe 评估套件（拆为独立 Feature）
- 不增加 time-based 微压缩（turn-based 已够用）

### Design

#### 070-A: Microcompact Integration

**问题**：`microcompact()` 已在 `packages/agent/src/compaction/microcompaction.ts` 实现并测试，但 `packages/coding/src/agent.ts` 中从未调用。

**方案**：在 agent loop 的 `needsCompaction` 检查前调用 `microcompact()`，同时扩展其能力处理 thinking 块和图片。

**效果**：
- 每轮 agent 响应完毕后，20 轮前的工具输出被替换为富占位符
- Thinking 块（token 大户但信息密度极低）被清理
- 图片内容替换为描述性标记 `[Image: {path}]`
- 上下文增长变成平缓斜率，LLM 全量压缩触发频率大幅降低

注意：本项吸收原 069-C 的范围，069 只保留 069-A (Rewind) 和 069-B (Shell Completion)。

#### 070-B: Bash Placeholder Enhancement

**问题**：bash 命令的占位符目前只取首个非 flag 参数（如 `git status` → `git status`），复杂命令无法表达意图。

**方案**：提取"意图命令"——跳过前缀噪声，保留核心操作：

| 原始命令 | 当前占位符 | 改进后占位符 |
|---------|-----------|------------|
| `cd /path && git push origin main` | `[Pruned: cd /path]` | `[Pruned: git push origin main]` |
| `DOCKER_BUILDKIT=1 docker build -t app .` | `[Pruned: DOCKER_BUILDKIT=1 docker]` | `[Pruned: docker build -t app]` |
| `cat package.json \| grep version` | `[Pruned: cat package.json]` | `[Pruned: grep version package.json]` |
| `npm run test -- --coverage` | `[Pruned: npm run]` | `[Pruned: npm test --coverage]` |

规则：
- 跳过 `cd`、`export`、环境变量赋值（`VAR=val`）前缀
- 链式命令（`&&`、`\|`）取最后一个有意义的命令
- 截断到 60 字符
- 保持 `[Pruned: ...]` / `[Cleared: ...]` 自解释前缀不变

**效果**：模型在压缩后看到 `[Pruned: git push origin main]` 就知道做过什么，不需要重新执行。

#### 070-C: User Message Protection

**问题**：用户消息和助手消息在压缩中被同等对待，可能丢失用户意图。

**方案**：分层保护，不增加额外 LLM 调用：

| 用户消息长度 | 处理方式 |
|-------------|---------|
| ≤ 800 token | **永远原样保留**（不裁剪、不被摘要覆盖） |
| \> 800 token + 已进入压缩区 | 保留前 ~400 token + 末尾 ~200 token，中间替换为 `[…用户消息截断, 原始 ~{N} token, 含{内容类型}…]` |

同时在 LLM 摘要 prompt 中增加保护指令：

> 每条用户请求和决策必须在摘要中逐项保留。不能把 "用户要求修复 /api/auth/login 的 401 错误并改用 JWT" 缩减为 "用户要求修复一个错误"。

**效果**：压缩后模型始终清楚"用户到底要什么"，短消息（99% 场景）零影响。

#### 070-D: Analysis Scratchpad + Summary Prompt Precision

**问题**：LLM 直接输出结构化摘要，容易遗漏具体路径、错误码、配置值。

**方案**（两个改进合一）：

**1) Analysis 草稿块**：修改 `summary-generator.ts` 的系统提示，要求模型先在 `<analysis>` 块中逐条走查消息，然后在 `<summary>` 块中输出摘要。`generateSummary` 返回前剥离 `<analysis>` 块。

系统提示增加禁止工具调用约束：
```
CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.
Tool calls will be REJECTED and waste your only turn.
```

**2) Prompt 精确化**：将模糊的 `key files, code locations, and decisions` 替换为明确的保留清单：
- EXACT file paths, function names, and line numbers
- EXACT error messages, HTTP status codes, exception types
- API endpoints, database tables, env vars, config values
- Key decisions WITH reasoning (not just the choice)

输出格式新增 `## Files & Changes` 段：
```markdown
## Files & Changes
- **[path]**: [what was done and why]
```

**效果**：
- 模型先"慢思考"逐条检查消息，再输出摘要，减少遗漏
- analysis 块剥离后对后续上下文零 token 成本
- 精确技术细节被结构化段落强制保留

#### 070-E: Post-Compact Reconstruction

**问题**：压缩完成后只有摘要 + 受保护尾部。Artifact Ledger 追踪了所有文件操作但压缩后未注入。

**方案**：压缩完成后注入两种附件：

**1) Artifact Ledger 精简摘要**（无 LLM 调用）：

将 Artifact Ledger 渲染为系统消息注入：
```markdown
[Post-compact: recent operations]
Modified: src/auth/login.ts (edit ×2), src/config/redis.ts (write)
Read: src/middleware/session.ts, package.json, tsconfig.json
Search: grep "401" src/auth/, grep "session" src/config/
Commands: npm test, docker-compose up
```

**2) 最近修改文件内容注入**（读取磁盘）：

对 Artifact Ledger 中最近 modified 的 top-N 文件，重新读取并注入首部片段。

**预算控制**：

统一规则——**后压缩附件总预算 = 释放空间的 50%**：

```
释放空间 = triggerPercent - targetPercent (of contextWindow)
后压缩预算 = 释放空间 × 50%
```

| triggerPercent | 释放 | 后压缩预算 | 200K 窗口 | 最终占用 | 剩余 |
|---|---|---|---|---|---|
| 60% | 24% | 12% | 24K | 48% | 52% |
| 75% | 33% | 16.5% | 33K | 58.5% | 41.5% |

预算内按优先级分配：Artifact Ledger 先占 ≤15%，剩余给文件内容（单文件上限为总预算的 20%，最多 5 文件）。

**效果**：这直接解决 Factory.ai 发现的所有压缩系统的最薄弱环节（Artifact 维度 2.45/5.0）。Agent 压缩后不再需要重新 read 刚才改过的文件。

#### 070-F: Circuit Breaker + Graceful Degradation

**问题**：当前 LLM 摘要失败时回退逻辑是 `Math.ceil(messages.length * 0.1)` 一刀切删除。

**方案**：

**1) 熔断器**：连续 3 次压缩失败后禁用 auto-compact（手动 `/compact` 仍可用），下次 session 重置。

**2) 渐进降级**：失败时不删固定百分比，而是从最旧的 atomic block（tool_use + tool_result 对）开始逐个丢弃，直到 token 降到安全线。每步都保持消息结构完整性。

**效果**：失败时信息损失从"随机删 10%"变成"精确丢弃最旧最不重要的完整交互对"。无限重试循环被熔断器阻断。
