## FEATURE_023: Dual-Mode Terminal UX (Inline + Fullscreen TUI)

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. Overview

KodaX currently mixes two incompatible terminal models:

1. terminal scrollback as if it were a normal append-only CLI
2. a fullscreen-like live UI with banner, status bar, spinner, streaming, and fixed bottom regions

This creates unstable behavior when users try to inspect history while the app is still animating or streaming.

This feature records a v1.0.0 roadmap that makes the mode split explicit:

- `inline` mode: append-only, scrollback-safe, minimal redraw
- `tui` mode: fullscreen, application-owned scrolling, fixed footer/status regions

The first implementation target is intentionally smaller than a renderer rewrite:

- keep the current Ink stack for now
- stop spinner animation when KodaX is waiting for user input
- add a half-automatic history review mode
- prove the interaction model before committing to a renderer migration

### 2. Current User Pain

The highest-friction workflow today is:

1. KodaX is streaming or animating a spinner
2. KodaX asks the user to make a choice in a UI dialog
3. the user needs to review recent history before deciding
4. scrolling causes the terminal to jump to the top/banner region

At that moment KodaX is not "thinking"; it is waiting for the user. A spinner adds little value there and actively hurts readability.

### 3. Short-Term Product Decision

#### 3.1 Waiting-for-input state

When KodaX is explicitly waiting for user choice/input:

- do not show spinner animation
- do not show a busy-state indicator
- show a clear waiting-for-input state instead

This is both semantically correct and technically stabilizing.

#### 3.2 Half-automatic history review mode

Before building a full application-owned viewport, KodaX should add a half-automatic review mode.

Expected behavior:

- `PgUp` enters history review mode immediately
- entering review mode pauses spinner animation
- entering review mode pauses high-frequency streaming flush
- new streaming content continues buffering in memory
- review mode supports `PgUp`, `PgDn`, `j`, `k`, `Home`, `End`
- `Ctrl+Y` toggles review mode on/off
- `Alt+Z` acts as an ergonomic backup toggle
- `Esc` resumes live mode immediately
- `End` resumes live mode and jumps to bottom
- when already at bottom, pressing `PgDn` again resumes live mode

This delivers most of the user value without a full viewport rewrite.

### 4. Shortcut Recommendation

`Ctrl+O` is not available because KodaX already uses it for permission mode switching.

Recommended shortcut model:

- Primary entry: `PgUp`
- Explicit secondary entry: `Ctrl+Y` (toggle)
- Ergonomic backup entry: `Alt+Z` (toggle)
- Exit: `Esc`
- Exit + jump to bottom: `End`
- Alternate resume gesture: `PgDn` at bottom, then `PgDn` again

Why `Ctrl+Y`:

- it is currently unassigned in the current TUI shortcut set
- it is less collision-prone than `Ctrl+A/E/K/U/W/J`
- it stays secondary because `PgUp` should remain the discoverable main entry

Why `Alt+Z`:

- it is easy to hit with the left hand
- it provides a backup when `Ctrl+Y` is awkward in a given keyboard layout or terminal
- it should behave exactly like `Ctrl+Y`, so the shortcut model stays simple

### 5. Half-Automatic Review vs Full Viewport

#### 5.1 Half-automatic review mode

This keeps the current rendering approach mostly intact, but introduces a deliberate pause/resume UX while the user reviews history.

Benefits:

- small implementation footprint
- fits the current product stage
- removes the most frustrating interaction failure quickly

Limits:

- still depends on the terminal for part of visible-history behavior
- does not provide a true application-owned transcript viewport
- still inherits some renderer-specific quirks from Ink

#### 5.2 Full application-owned viewport

This means KodaX owns transcript scrolling directly:

- maintain the full transcript internally
- compute the visible line window from scroll offset
- render only the selected slice
- keep explicit `followBottom` state
- allow the user to inspect any point in KodaX history while live updates continue buffering

This is the "real TUI" model, but it is substantially heavier.

### 6. Renderer Strategy Notes

#### 6.1 Keep Ink for now

The current recommendation is not "stay on Ink forever". It is:

- keep Ink while business-layer improvements are still moving quickly
- prove the interaction model first
- avoid locking core transcript/scroll/output state directly into Ink primitives

This turns Ink into a transitional renderer rather than a permanent architecture choice.

#### 6.2 Rezi

Rezi is currently the strongest medium-term Node/TypeScript candidate that was researched.

Why it is promising:

- Node/TypeScript friendly
- explicit Ink compatibility path
- better renderer-level control than Ink
- realistic migration path: Ink-Compat first, native Rezi later

Why not migrate immediately:

- business-layer work still has higher priority
- the interaction model should be validated before renderer migration
- some relevant widgets/features are newer than the most mature TUI ecosystems

#### 6.3 OpenTUI

OpenTUI is attractive, especially if KodaX ever decides to become an upstream contributor.

Why it is attractive:

- modern architecture
- renderer-first design
- strong fullscreen TUI direction

Why it is not the immediate path:

- runtime/tooling adoption is still a practical constraint for the current KodaX stack
- it is better suited for a future TUI frontend layer than for an urgent stop-the-bleeding fix

#### 6.4 Ratatui / Bubble Tea

These remain the most mature long-term fullscreen TUI foundations among the options researched, but they are also the highest-cost migration targets because they imply a more complete frontend rewrite and a cross-language architecture split.

### 7. Node Ecosystem Notes

Additional Node-side alternatives explored during research:

- `pi-tui`: interesting custom TS renderer direction, worth following
- `Rezi`: best researched TS-first migration candidate
- `terminal-kit`: viable low-level option if KodaX wants to own more terminal behavior directly
- `neo-blessed`: workable, but less aligned with the desired long-term KodaX UX direction

### 8. Proposed Implementation Phases

#### Phase 1: UX stabilization in the current Ink stack

1. Hide spinner while waiting for user choice/input
2. Add `reviewing_history` state
3. Pause live animation/flush while reviewing
4. Add keyboard-driven review/resume flow
5. Show clear footer/help text for paused live updates

#### Phase 2: Renderer boundary cleanup

1. Separate mode-agnostic transcript state from renderer logic
2. Centralize terminal output ownership
3. Define scroll state explicitly (`scrollOffset`, `followBottom`, `isReviewingHistory`)
4. Reduce direct console/log writes into the live UI path

#### Phase 3: Reevaluate renderer migration

Decision gate after Phase 1 and 2 stabilize:

- stay on Ink longer
- migrate to Rezi
- explore OpenTUI contribution path
- defer heavy TUI migration until business-layer work slows down

### 9. Acceptance Criteria

- [ ] Waiting-for-input UI no longer shows spinner animation
- [ ] `PgUp` enters history review mode reliably
- [ ] `Esc` exits review mode reliably
- [ ] `End` exits review mode and jumps to bottom
- [ ] Pressing `PgDn` at the bottom resumes live mode
- [ ] Reviewing history no longer yanks the user back to the top/banner during paused live mode
- [ ] Buffered live content resumes cleanly after leaving review mode
- [ ] The design preserves a future migration path to Rezi or another renderer

### 10. Open Questions

1. Should mouse wheel support be attempted in Phase 1, or postponed until keyboard review mode feels solid?
2. Should review mode be allowed during all live streaming, or only when the UI is explicitly waiting for user interaction?
3. When resuming from review mode, should buffered content flush instantly, or be smoothed over a short catch-up animation?
