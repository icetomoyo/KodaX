## Overview

After the roadmap reset, `v1.0.0` is no longer where KodaX first "gets multi-agent".

That work moves into `v0.7.0`.

`v1.0.0` now focuses on making the engine feel mature across user-facing surfaces:

- terminal UX maturity
- surface portability
- long-running task visibility beyond one terminal window
- tool interaction maturity inside the frozen transcript shell

## FEATURE_023: Dual-Mode Terminal UX

**Status**: Planned
**Priority**: High
**Category**: Enhancement

> Source-of-truth note (2026-04-04):
> The active planned version for `FEATURE_023` is now [v0.7.30](./v0.7.30.md#feature_023-dual-mode-terminal-ux).
> This section remains as historical staging context only.

### Overview

KodaX needs a cleaner terminal interaction model that can support:

- append-only terminal usage
- full TUI-style interactive usage
- stable history inspection during long-running work

This feature is now a delivery/interaction feature, not a core engine feature.

**Handoff note (2026-04-04)**: `FEATURE_051` and `FEATURE_055` already closed the current REPL shell direction. `FEATURE_023` should now be read as a future delivery/terminal-ergonomics umbrella, not as permission to reopen the frozen visible REPL shell.

### Goals

1. Split inline and TUI assumptions clearly.
2. Improve history review and waiting-for-input behavior.
3. Reduce interaction friction during long-running tasks.
4. Preserve a migration path to stronger renderers later.

### Scope

- clearer inline vs TUI behavior
- waiting-for-input UI cleanup
- review-mode behavior
- transcript ownership and scroll-state cleanup
- future renderer-boundary cleanup that preserves the current frozen shell contract

### Non-goals

- redefining task execution semantics
- replacing the control plane
- changing role architecture
- reopening `FEATURE_051 / FEATURE_055` as a new visible-shell rollout
- adding a heavier multi-panel terminal cockpit in the name of renderer maturity
- reintroducing a second transcript/task surface that competes with the existing REPL main screen

### Acceptance criteria

- [ ] terminal behavior is clearer in both inline and TUI-style use
- [ ] history review no longer fights active rendering
- [ ] waiting-for-input state is distinct from busy-state rendering
- [ ] renderer migration remains possible later without engine rewrites

### Restored Detailed Design Record

The previous design for this feature contained UX and renderer research that should remain attached to the roadmap.

#### Current user pain retained

The main interaction failure remains:

1. KodaX is streaming or animating a spinner
2. KodaX asks the user to make a choice
3. the user needs to inspect recent history before deciding
4. scrolling fights active rendering and jumps unpredictably

The key insight retained from the earlier draft is that waiting-for-input is not the same state as busy execution.

#### Waiting-for-input state

Retained direction:

- do not show spinner animation while explicitly waiting for user choice/input
- do not present a busy-state indicator during that state
- show a clear waiting-for-input state instead

#### Half-automatic history review mode

The previous draft's staged UX still looks like the right near-term path:

- `PgUp` enters review mode
- entering review mode pauses spinner animation
- entering review mode pauses high-frequency flush/render behavior
- new streaming content buffers while the user reviews history
- `Esc` resumes live mode immediately
- `End` resumes live mode and jumps to bottom
- `PgDn` at bottom can act as a resume gesture

This remains a strong intermediate step before a full application-owned transcript viewport.

#### Shortcut guidance retained

Useful retained guidance:

- `PgUp` as the main entry
- `Ctrl+Y` as an explicit toggle
- `Alt+Z` as an ergonomic backup
- `Esc` to resume
- `End` to resume and jump to bottom

#### Review mode vs full viewport

The previous draft correctly separated:

- a lighter half-automatic review mode
- a heavier application-owned transcript viewport

That distinction should remain because it helps stage implementation without pretending the two approaches cost the same.

#### Full application-owned viewport notes retained

The heavier target from the older draft is still worth preserving as a future reference:

- maintain the full transcript internally
- compute a visible line window from scroll offset
- render only the selected slice
- keep explicit `followBottom` state
- allow history inspection while live updates buffer separately

This remains the "real TUI" end-state even if the first implementation step is lighter.

#### Renderer strategy retained

Still useful research:

- keep Ink for now while the interaction model is stabilized
- avoid overfitting core transcript state to renderer-specific assumptions
- keep Rezi as the strongest researched TS-first migration candidate
- treat OpenTUI and other deeper renderer shifts as later options, not immediate requirements

#### Renderer option notes retained

Useful older research that should remain attached to this feature:

- Rezi looked promising because it is TypeScript-friendly and offers a more realistic Ink-compatibility migration path
- OpenTUI looked attractive as a renderer-first long-term direction, but was not the right immediate stop-the-bleeding choice
- Ratatui and Bubble Tea remained strong fullscreen-TUI references, but implied a much heavier rewrite and cross-language split
- lower-level Node options such as `terminal-kit` were worth keeping in mind if KodaX later wants tighter terminal ownership

#### Acceptance and open-question history retained

The previous draft also captured useful validation questions:

- should review mode support mouse-wheel behavior immediately, or wait for keyboard flow to stabilize
- when resuming live mode, should buffered output flush instantly or with a short catch-up rhythm
- should review mode be allowed during all streaming states or mainly when KodaX is explicitly waiting for user interaction

Those questions still matter because they separate UX stabilization from renderer migration.

#### Suggested implementation phases retained

1. UX stabilization in the current Ink stack
2. renderer-boundary cleanup
3. re-evaluate renderer migration only after the interaction model is proven

## FEATURE_030: Multi-Surface Delivery

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

KodaX should not remain confined to a single terminal surface. Once the engine and control plane are stable, the product should be able to appear in:

- IDE contexts
- desktop review surfaces
- remote or web-driven monitoring surfaces

### Goals

1. Reuse one engine across multiple user-facing surfaces.
2. Preserve task fidelity across handoff boundaries.
3. Enable inspection and long-running task monitoring outside the terminal.
4. Keep approvals, evidence, and summaries portable.

### Scope

- shared session and task handoff contracts
- surface-aware but engine-neutral task presentation
- monitoring and review views outside the core REPL

### Non-goals

- inventing a second engine for non-terminal surfaces
- redefining orchestration semantics per surface
- forking task state by UI type

### Acceptance criteria

- [ ] at least one non-terminal surface can host or monitor KodaX tasks
- [ ] tasks and evidence can move across surfaces without losing fidelity
- [ ] the shared runtime remains the same engine used by CLI and REPL

### Restored Detailed Design Record

Useful direction retained from the previous draft:

#### Product goals retained

- add a serious IDE integration path for context injection and task handoff
- add a desktop review surface for richer diff and approval workflows
- add a web or remote-monitoring surface for long-running tasks
- keep sessions, `AGENTS.md`, skills, and permissions portable across surfaces

#### Technical direction retained

- define a shared session-transfer or task-handoff protocol
- support context handoff among terminal, IDE, desktop, and web
- keep engine logic surface-agnostic
- reuse shared task truth, evidence, and connector models

#### Relationship to `034`

Retained boundary:

- `034` provides headless runtime contracts and optional host adapters
- `030` owns actual IDE, desktop, and web delivery surfaces
- `030` also owns transport, monitoring UX, and surface-specific review/approval experiences

This keeps runtime and product-surface ownership separate.

## FEATURE_056: Tool Interaction Maturity and Transcript-Native Explanation Layer

**Status**: Planned
**Priority**: High
**Category**: Enhancement

> Source-of-truth note (2026-04-04):
> The active planned version for `FEATURE_056` is now [v0.7.30](./v0.7.30.md#feature_056-tool-interaction-maturity-and-transcript-native-explanation-layer).
> This section remains as historical staging context only.

### Overview

KodaX's current tool interaction is usable, but it still feels thinner and less legible than ClaudeCode in a few important ways:

- tool progress is harder to interpret at a glance
- tool failures are less clearly explained
- file-edit results do not yet feel like first-class transcript-native diffs
- action affordances still feel closer to hint text than to a mature interaction layer

This feature should improve those areas **without reopening the visible REPL shell** and without turning KodaX into a coordinator/task cockpit.

### Goals

1. Make tool progress, diff, and failure easier to understand.
2. Keep tool interaction transcript-first and summary-first.
3. Improve action affordances inside the existing shell.
4. Make critical tool/runtime/worktree moments easier to interpret without adding new panels.

### Product principles

1. Transcript-native explanation, not a new workbench.
2. Critical-node clarity over persistent visibility.
3. Richer specialization, not more surface area.
4. One main authority remains intact.

### Scope

- transcript-native tool diff rendering improvements
- clearer tool error/explanation rendering
- stronger long-running tool/shell progress explanation
- more mature message-action capability modeling inside the existing transcript shell
- clearer worktree/runtime/tool critical-node explanations through existing surfaces

### Relationship to other features

- `FEATURE_051` and `FEATURE_055` freeze the visible REPL shell; this feature must work **inside** that shell instead of reopening it.
- `FEATURE_053` provides worktree/runtime clarity at critical moments; this feature may consume those explanations but should not turn them into a persistent cockpit.
- `FEATURE_030` may later reuse the same tool explanation layer in non-terminal surfaces.

### Non-goals

- worker graph
- teammate dashboard
- task cockpit
- command-palette style tool center
- persistent inspector panels
- reopening the frozen shell surface from `FEATURE_051 / FEATURE_055`

### Acceptance criteria

- [ ] file-edit tool results can surface clearer, transcript-native diffs
- [ ] failed tool calls have a clearer explanation path inside the transcript
- [ ] long-running shell/tool progress is more legible without new persistent panels
- [ ] message actions feel more capable than plain hint text while staying lightweight
- [ ] no new control-tower UI or worker-topology surface is introduced

### Initial design direction

Useful starting directions for implementation:

- prefer specialized transcript-native explanation rows over new standalone surfaces
- improve tool-output summaries before adding any new interaction chrome
- treat worktree/runtime explanations as critical-moment guidance, not background status noise
- preserve KodaX's quieter shell posture even while improving clarity
