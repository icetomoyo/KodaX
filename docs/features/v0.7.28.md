# v0.7.28 Feature Designs

> 中文标题：`v0.7.28 Feature 设计`
>
> Version: `v0.7.28`
>
> Status: Planned (not yet released)
>
> Theme: Self-construction foundation — ConstructionRuntime + first tier Tool generation

---

## 总览

v0.7.28 开启 KodaX **自构建段**（FEATURE_087-090）。本版合并两个 feature：

- **FEATURE_087 ConstructionRuntime**：引入 stage/test/activate/revoke 四段生命周期 + Constructed-World 存储 + Resolver 合并 + policy gate
- **FEATURE_088 自构建档 2 (Tool 生成)**：让 Agent 生成 Tool 定义并通过 sandbox 测试后注册

合并理由：087 是基础设施，088 是它的**首个真实消费者**；一起做能验证基础设施的 API shape，避免 087 独立落地后 088 才发现接口需要调整。档 1 (Skill 生成) 作为既有能力的自然兑现（Agent 用 write 工具写 markdown），在本版自动可用，不单列 feature。

## FEATURE_087: ConstructionRuntime and Constructed-World Substrate

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 你愿景中 KodaX 要成为"超级智能体的底座" —— 关键是**基于 Coding 能力自我构建**
- 要实现自构建，前提是 A/B 层所有概念都是 data（FEATURE_080/081/082 已做到）
- 但仅仅"能序列化"不够 —— 还需要运行时的**注册/加载/撤销**通道，以及**安全审批**机制
- 当前 KodaX 生成 skill 文件是可行的（agent 用 write tool 写 markdown），但生成 Tool 定义、Agent 定义都没有路径落地
- 必须区分两个世界：**Builtin World**（仓库里编译的，只读）vs **Constructed World**（运行时生成的，写入 `.kodax/constructed/`）

### 设计

#### ConstructionRuntime 接口

```ts
// @kodax/core/src/construction.ts

export interface ConstructionArtifact {
  kind: 'agent' | 'tool' | 'skill' | 'handoff' | 'session-extension'
  name: string
  version: string                     // semver，支持同名多版本
  content: unknown                    // kind 对应的 data shape
  signedBy?: string                   // 可选签名
  createdAt: number
  testedAt?: number                   // 通过 sandbox test 的时间戳
  sourceAgent?: string                // 生成此 artifact 的 agent 名
}

export interface StagedHandle {
  readonly artifact: ConstructionArtifact
  readonly stagedAt: number
}

export interface TestResult {
  ok: boolean
  errors?: string[]
  warnings?: string[]
  trace?: Trace                       // 来自 @kodax/tracing
}

export interface ConstructionRuntime {
  stage(artifact: ConstructionArtifact): Promise<StagedHandle>
  test(handle: StagedHandle, input?: unknown): Promise<TestResult>
  activate(handle: StagedHandle): Promise<void>
  revoke(name: string, version: string): Promise<void>
  list(kind?: ConstructionArtifact['kind']): Promise<ConstructionArtifact[]>
}
```

#### Constructed-World 存储

```
.kodax/constructed/
├── agents/
│   ├── my-security-reviewer/1.0.0.json
│   └── my-security-reviewer/1.1.0.json
├── tools/
│   ├── pii-scanner/1.0.0.json
│   └── ...
├── skills/
│   └── (既有 skill 路径)
└── _manifest.json      —— 全量索引
```

#### Resolver 合并

```ts
export class CapabilityResolver {
  resolveAgent(name: string): Agent | undefined {
    return this.constructedAgents.get(name) ?? this.builtinAgents.get(name)
    // constructed 优先（允许用户覆盖内置）
  }
  resolveTool(name: string): Tool | undefined { /* ... */ }
  resolveSkill(name: string): Skill | undefined { /* ... */ }
}
```

#### Policy Gate

```ts
export interface ConstructionPolicy {
  onActivate(artifact: ConstructionArtifact): Promise<'approve' | 'reject' | 'ask-user'>
}

// 默认 policy：ask-user for everything
export const DefaultConstructionPolicy: ConstructionPolicy = {
  onActivate: async (art) => 'ask-user',
}

// 可配置规则：
// - 若 artifact.signedBy === 'self' && testedAt within 24h → auto-approve
// - 若 artifact.kind === 'tool' && 无网络/文件操作权限 → auto-approve
// - 其他 → ask-user
```

## FEATURE_088: Self-Construction Tier 2 — Tool Generation

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 档 1（Skill 生成）已经可用：agent 用 write tool 写 markdown 到 `~/.kodax/skills/` 或 `.kodax/skills/`，skill-loader 自动发现
- 档 2 让 agent 能**动态生成 Tool**（例如："我需要一个专门扫 OAuth scope 的工具"）
- 前提是 FEATURE_087 的 ConstructionRuntime + FEATURE_080 的 Tool-as-data

### 设计

#### Tool manifest 格式

```json
{
  "kind": "tool",
  "name": "oauth-scope-scanner",
  "version": "1.0.0",
  "content": {
    "description": "Scan a codebase for OAuth scope usage",
    "inputSchema": {
      "type": "object",
      "properties": {
        "rootPath": { "type": "string" },
        "includePatterns": { "type": "array", "items": { "type": "string" } }
      },
      "required": ["rootPath"]
    },
    "handler": {
      "kind": "script",
      "language": "typescript",
      "code": "export async function handler(input, ctx) { /* ... */ }"
    }
  }
}
```

#### 生成流程（由 Coding Agent 完成）

```
user: "生成一个扫 OAuth scope 的 tool"
   │
   ▼
Coding Agent (用 A 层原语):
   1. scaffold_tool: 根据需求写 Tool manifest（JSON + handler code）
   2. validate_tool: 静态校验 schema / handler code AST
   3. ConstructionRuntime.stage(manifest)
   4. sandbox_test: 在独立 worker / vm 中跑 handler 对一组测试输入（输出符合 schema）
   5. ConstructionRuntime.test(handle, sampleInputs)
   6. 通过 policy gate（用户确认）
   7. ConstructionRuntime.activate(handle)
   │
   ▼
New tool available via CapabilityResolver
```

#### Sandbox 测试

- Handler code 用 `vm2` / `isolated-vm` / 或新 worker thread 执行（权限隔离）
- 禁止默认访问 process.env / fs / net（需要在 manifest 声明）
- Timeout 保护（默认 30s）
- 输出必须符合 outputSchema（若声明）

#### 安全考虑

- 生成的 Tool handler 必须通过 **Guardrail** 静态检查（FEATURE_085）
- Policy gate 对 tool 生成默认 ask-user（人工审批）
- Tool 有版本，revoke 后旧版本不再可用

### Contract

**Public API（新增）**：
- `ConstructionRuntime` interface + default implementation
- `ConstructionArtifact` / `StagedHandle` / `TestResult` types
- `ConstructionPolicy` interface
- `CapabilityResolver` class
- CLI: `kodax constructed list | activate | revoke`

**Internal**：coding agent 在 Runner 有 `scaffold_tool` / `validate_tool` / `stage_construction` / `sandbox_test_tool` 等内部 tool。

### Non-Goals

- **不**支持 Agent 生成（档 3 → FEATURE_089 v0.7.31）
- **不**支持自改 role spec（档 4 → FEATURE_090 v0.7.32）
- **不**支持跨进程签名验证（本地签名够用）
- **不**自动发布 constructed artifact 到 npm / 公开 registry

### Dependencies

- **前置**：
  - FEATURE_080 (v0.7.23) — Tool / Agent 是 data
  - FEATURE_082 (v0.7.24) — `@kodax/core` 就位
  - FEATURE_083 (v0.7.24) — Tracer 记录 stage/test/activate span
  - FEATURE_085 (v0.7.26) — Guardrail 做 static check
- **后置**：
  - FEATURE_089 (v0.7.31) — Agent generation 复用 ConstructionRuntime
  - FEATURE_090 (v0.7.32) — self-modify 复用 ConstructionRuntime + versioning

### Testing

- 单元：stage / test / activate / revoke 生命周期
- 单元：Resolver 合并 builtin + constructed
- 集成：coding agent 生成一个简单 tool（如 "count lines"）→ sandbox test → activate → 后续 agent 可用
- 集成：policy gate 的 ask-user 路径（mock 用户确认）
- 安全：恶意 handler（读 /etc/passwd 之类）在 sandbox 中被阻断

### Risk

- **Sandbox 逃逸**：Node.js 层面 sandbox 不是绝对安全；对策：
  1. 默认 policy ask-user，强制人工确认
  2. Handler 有 timeout / 资源限制
  3. 未来可选 Docker / firecracker sandbox（不在本版范围）
- **版本管理**：同名多版本并存时哪个 active？规则：active 的是最新 activated 的，其他版本保留但不解析

### Timeline

v0.7.28 合并 087 + 088。ConstructionRuntime 基础设施 + Tool 生成首个 tier 一起验证发版。
