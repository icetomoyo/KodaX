## 总览

v0.7.28 开启 KodaX **自构建段**（FEATURE_087-090）。本版合并两个 feature：

- **FEATURE_087 ConstructionRuntime**：引入 stage/test/activate/revoke 四段生命周期 + Constructed-World 存储 + Resolver 合并 + policy gate
- **FEATURE_088 自构建档 2 (Tool 生成)**：让 Agent 生成 Tool 定义并通过静态检查 + 能力校验后注册

合并理由：087 是基础设施，088 是它的**首个真实消费者**；一起做能验证基础设施的 API shape，避免 087 独立落地后 088 才发现接口需要调整。档 1 (Skill 生成) 作为既有能力的自然兑现（Agent 用 write 工具写 markdown），在本版自动可用，不单列 feature。

本版另搭载两个**已实施完成**的 provider catalog 卫生 feature（与自构建主线独立）：

- **FEATURE_099 Provider Catalog Refresh**：DeepSeek V4 系列接入（v4-flash 默认 + v4-pro alt）+ kimi-code 模型 label 收敛为 `kimi-for-coding` + 删除 deprecated `deepseek-chat` / `deepseek-reasoner`
- **FEATURE_098 Per-Model Context Window & Output Token Limits**：把 `KodaXModelDescriptor.contextWindow` / `maxOutputTokens` 这两个早已声明但全代码无人读取的死字段接通运行时，让 compaction trigger 与 wire-level max_tokens 都按当前激活 model 的真实窗口算；当时按 Moonshot 文档把 `kimi.k2.5` 当作 128k 做了一次 pin（**事后核对发现 K2.5 实际 256K，已于 commit `64785de` 改正，见 FEATURE_098 节末尾的"实施后更正"**）+ 修复 `zhipu.glm-5-turbo` (128k) 偏差；自定义 provider `models[]` 升级支持描述符对象格式（兼容旧字面量）。原计划 v0.7.29 同 FEATURE_078 同版发布，因实施提前完成迁入本版

098/099 与 087/088 实施面无交集（catalog 数据 vs construction runtime），合在一版便于一次发版同时获得两条改进。

## FEATURE_087: ConstructionRuntime and Constructed-World Substrate

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 你愿景中 KodaX 要成为"超级智能体的底座" —— 关键是**基于 Coding 能力自我构建**
- 要实现自构建，前提是 A/B 层所有概念都是 data（FEATURE_080/081/082 已做到）
- 但仅仅"能序列化"不够 —— 还需要运行时的**注册/加载/撤销**通道，以及**安全审批**机制
- 当前 KodaX 生成 skill 文件是可行的（agent 用 write tool 写 markdown），但生成 Tool 定义、Agent 定义都没有路径落地
- 必须区分两个世界：**Builtin World**（仓库里编译的，只读）vs **Constructed World**（运行时生成的，写入 `.kodax/constructed/`）

### 设计

#### ConstructionRuntime 接口

```ts
// @kodax/core/src/construction.ts

export interface ConstructionArtifact {
  kind: 'agent' | 'tool' | 'skill' | 'handoff' | 'session-extension'
  name: string
  version: string                     // semver，支持同名多版本
  content: unknown                    // kind 对应的 data shape
  signedBy?: string                   // 可选签名
  createdAt: number
  testedAt?: number                   // 通过 test 阶段的时间戳
  sourceAgent?: string                // 生成此 artifact 的 agent 名
}

export interface StagedHandle {
  readonly artifact: ConstructionArtifact
  readonly stagedAt: number
}

export interface TestResult {
  ok: boolean
  errors?: string[]
  warnings?: string[]
  trace?: Trace                       // 来自 @kodax/tracing
}

export interface ConstructionRuntime {
  stage(artifact: ConstructionArtifact): Promise<StagedHandle>
  test(handle: StagedHandle, input?: unknown): Promise<TestResult>
  activate(handle: StagedHandle): Promise<void>
  revoke(name: string, version: string): Promise<void>
  list(kind?: ConstructionArtifact['kind']): Promise<ConstructionArtifact[]>
}
```

#### Constructed-World 存储

```
.kodax/constructed/
├── agents/
│   ├── my-security-reviewer/1.0.0.json
│   └── my-security-reviewer/1.1.0.json

│   └── ...
├── skills/

```

## FEATURE_088: Self-Construction Tier 2 — Tool Generation

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 档 1（Skill 生成）已经可用：agent 用 write tool 写 markdown 到 `~/.kodax/skills/` 或 `.kodax/skills/`，skill-loader 自动发现
- 档 2 让 agent 能**动态生成 Tool**（例如："我需要一个专门扫 OAuth scope 的工具"）
- 前提是 FEATURE_087 的 ConstructionRuntime + FEATURE_080 的 Tool-as-data

### 设计

#### Tool manifest 格式

```json
{
  "kind": "tool",
  "name": "oauth-scope-scanner",
  "version": "1.0.0",
  "content": {
    "description": "Scan a codebase for OAuth scope usage",
    "inputSchema": {
      "type": "object",
      "properties": {
        "rootPath": { "type": "string" },
        "includePatterns": { "type": "array", "items": { "type": "string" } }
      },
      "required": ["rootPath"]
    },
    "capabilities": {
      "tools": ["grep", "read"],
      "fs": { "read": ["{rootPath}/**"], "write": [] },
      "net": { "domains": [] },
      "env": []
    },
    "handler": {
      "kind": "script",
      "language": "typescript",
      "code": "export async function handler(input, ctx) { /* 只能用 ctx.tools.*, 不能直接 require/fetch/process.env */ }"
    }
  }
}
```

**`capabilities` 字段**是本版核心安全机制（替代代码级 sandbox）：

- `tools`：声明可通过 `ctx.tools.<name>` 调用的 **builtin tool 白名单**
- `fs`：声明可访问的文件路径模式（通过 ctx 代理，不是直接 fs API）
- `net`：声明可访问的网络域名
- `env`：声明可读的环境变量名

**Handler 是纯函数约束**：
- 输入 = `input`（通过 input_schema 校验）
- 输出 = return value（通过 outputSchema 校验）
- **禁止直接访问** Node.js API（`fs` / `net` / `process` / `require` / `import`）
- **所有 I/O 必须通过 `ctx.tools.<name>(...)` 代理**，代理调用会检查 `capabilities` 声明

#### 生成流程（由 Coding Agent 完成）

```
user: "生成一个扫 OAuth scope 的 tool"
   │
   ▼
Coding Agent (用 A 层原语):
   1. scaffold_tool: 根据需求写 Tool manifest（含 capabilities + handler code）
   2. static_check: Guardrail AST 静态校验
        - 禁用 require / import / fetch / process.env 等宿主 API
        - 所有 I/O 必须走 ctx.tools.*
        - handler 签名符合 (input, ctx) => Promise<output>
   3. ConstructionRuntime.stage(manifest)
   4. ConstructionRuntime.test(handle, sampleInputs)
        - 在主进程直接跑（已通过静态检查，无隔离需求）
        - 用 CtxProxy 注入 ctx，拦截未声明能力
        - 验证 output 符合 schema
        - 验证 provider schema 兼容
   5. 通过 policy gate（用户确认 — 默认 ask-user）
   6. ConstructionRuntime.activate(handle)
        - import handler 到主进程 TOOL_REGISTRY
        - 用 CtxProxy 包装 ctx，按 capabilities 限权
   │
   ▼
New tool available via CapabilityResolver
```

#### 安全模型：能力声明 + I/O 代理（非代码级 sandbox）

**重要决策**：v0.7.28 **不采用代码级 sandbox**（不用 `worker_threads` / `isolated-vm` / `vm2` 包裹 handler 执行）。Handler 直接 `import` 到主进程运行。

**行业先例**：调研了 Anthropic 官方 Claude Code、pi-mono、opencode 三个同类项目，**无一对插件 / 扩展代码做 JS 级沙盒**：

| 项目 | 插件代码隔离 | 依赖的安全机制 |
|------|-------------|---------------|
| Claude Code | ❌ 不隔离（skills / MCP tools 全在宿主进程） | OS 级 bash sandbox + 人工审批 |
| pi-mono | ❌ 不隔离（jiti 直接加载 extension） | OS 级 bash sandbox（macOS/Linux） |
| opencode | ❌ 不隔离（`await import()` 直入宿主） | 无（完全信任用户安装的 plugin） |

**结论**：Node.js JS 层 sandbox 不是安全边界（Node 官方文档原话），绕过成本远低于封堵成本。行业共识是"代码级不隔离 + 系统调用级隔离 + 人工审批"。

**KodaX v0.7.28 采用的四层防御**（由弱到强）：

| 层 | 机制 | 拦截什么 |
|----|------|---------|
| ① 静态检查 | Guardrail AST 检查（FEATURE_085） | `require` / `import` / `fetch` / `process.*` / 全局对象访问 |
| ② 能力声明 | Manifest `capabilities` 字段 | 明示该 handler 需要哪些 builtin tool / fs 路径 / 域名 |
| ③ Ctx 代理 | `CtxProxy` 运行时拦截 | handler 通过 `ctx.tools.*` 访问时按 capabilities 校验 |
| ④ Policy Gate | `ConstructionPolicy` 人工审批 | activate 前由用户决定信不信 |

**I/O 代理模型**：

```ts
// Handler 这么写：
export async function handler(input: Input, ctx: ToolCtx): Promise<Output> {
  //                                        ↑ CtxProxy（不是 Node API）

  // ✅ 允许：走代理（capabilities.tools 声明了 grep）
  const matches = await ctx.tools.grep({ pattern: 'oauth', path: input.rootPath })

  // ✅ 允许：logger / tracer 等无副作用工具
  ctx.logger.info(`Scanned ${matches.length} files`)

  return { matches }

  // ❌ 静态检查阶段就挂：
  // const fs = require('fs')        — Guardrail 拦截
  // fetch('https://evil.com')       — Guardrail 拦截
  // process.env.SECRET              — Guardrail 拦截
}
```

**CtxProxy 关键职责**：
- `ctx.tools.<name>()` — 调用前检查 `capabilities.tools` 白名单；调用时走已有的 `executeTool()` 路径（复用 builtin 的安全链路，如 bash 走 OS sandbox）
- `ctx.fs.read(path)` — 检查 `capabilities.fs.read` glob 匹配
- `ctx.env.get(name)` — 检查 `capabilities.env` 白名单
- 不在 capabilities 里的能力 → throw `CapabilityDeniedError`（记入 tracer）

**Timeout 保护**：handler 执行超过 `manifest.content.timeoutMs`（默认 30s）→ `AbortError`。通过 `AbortController` 实现（主进程级），**不依赖 worker termination**。

**Non-goal（显式不做）**：
- 不防御**恶意** handler（LLM 生成的代码不是对抗性攻击模型）
- 不做 V8 Isolate / 进程级隔离（留给未来可选 factory）
- 不阻止 CPU / 内存耗尽（依赖 timeout 兜底）

**升级路径**：若后续确实需要更强隔离（如企业场景、多租户场景），`SandboxHandlerFactory` 可作为**可选插件**引入 —— 本版的 `import + CtxProxy` 只是默认 factory 实现，不影响接口。

#### 安全考虑

**Guardrail 静态检查规则（FEATURE_085 扩展）**：

以下为 v0.7.28 必达的 AST 规则集（阻止 handler 绕过能力声明直接访问宿主）：

| 规则 ID | 禁止 | 原因 |
|---------|------|------|
| `no-require` | `require(...)` / `require.resolve` | 绕过 capabilities 引入任意模块 |
| `no-dynamic-import` | `import(...)` / `await import(...)` | 同上 |
| `no-global-fetch` | `fetch` / `XMLHttpRequest` / `WebSocket` | 绕过 `capabilities.net` |
| `no-process-access` | `process.*`（除白名单如 `process.cwd()`） | 绕过 `capabilities.env` |
| `no-node-globals` | `Buffer` / `__dirname` / `__filename` / `globalThis.*` | 副作用入口 |
| `no-eval` | `eval` / `new Function(...)` | 规避静态检查 |
| `no-top-level-side-effect` | handler 模块顶层 `await` / 赋值全局 | 避免 import 时就执行 |
| `require-handler-signature` | 必须导出 `async function handler(input, ctx)` | 统一 dispatch 入口 |

**其他安全机制**：

- Policy gate 对 tool 生成默认 ask-user（人工审批）
- Tool 有版本，revoke 后旧版本不再可用
- CtxProxy 运行时拦截：任何未在 capabilities 声明的 `ctx.*` 访问 → throw `CapabilityDeniedError`，记录到 tracer
- Handler 执行 AbortController + 30s timeout 默认值
- 所有 constructed tool 的调用在 tracer 里加 `source.kind='constructed'` 标签，便于事后审计

### Contract

**Public API（新增）**：
- `ConstructionRuntime` interface + default implementation
- `ConstructionArtifact` / `StagedHandle` / `TestResult` types
- `ConstructionPolicy` interface
- `CapabilityResolver` class
- CLI: `kodax constructed list | activate | revoke`

**Internal**：coding agent 在 Runner 有 `scaffold_tool` / `validate_tool`（Guardrail 静态检查）/ `stage_construction` / `test_tool`（主进程试跑 + CtxProxy 能力校验）等内部 tool。

### Dependencies

- **前置**：
  - FEATURE_080 (v0.7.23) — Tool / Agent 是 data
  - FEATURE_082 (v0.7.24) — `@kodax/core` 就位

### Testing

- 单元：stage / test / activate / revoke 生命周期
- 单元：Resolver 合并 builtin + constructed
- 单元：Guardrail 规则集逐条（`no-require` / `no-dynamic-import` / `no-global-fetch` / ...）
- 单元：CtxProxy 的能力匹配（白名单命中 / 未声明拒绝 / glob 路径匹配）
- 集成：coding agent 生成一个简单 tool（如 "count lines"）→ static_check → test → activate → 后续 agent 可用
- 集成：policy gate 的 ask-user 路径（mock 用户确认）
- 安全：恶意 handler 在 **Guardrail 静态检查阶段** 被阻断（尝试写 `require('fs')` / `fetch(...)` → AST 规则命中）
- 安全：声明 `capabilities.tools: ['grep']` 的 handler 调用 `ctx.tools.bash(...)` → CtxProxy 抛 `CapabilityDeniedError`

### Risk

- **Guardrail 绕过**：静态检查必然有盲区（字符串拼接 `const r = ['req', 'uire'].join('')` 类的混淆难以完美检测）。对策：
  1. 规则集从严，可疑模式直接 warn + 阻止 activate
  2. Policy gate 默认 ask-user，人工是最后防线
  3. Tracer 记录所有 constructed tool 调用 + capabilities，事后可审计
  4. 升级路径：未来可引入 V8 Isolate factory 作为可选第二道防线
- **CtxProxy 绕过**：handler 拿到 `ctx` 后理论上可以 `Object.getPrototypeOf(ctx.tools).<...>`。对策：
  1. CtxProxy 用 `Object.freeze` + 每次返回新对象，避免原型污染
  2. 所有 ctx.tools.* 实际调用点都走 capabilities 再校验（不依赖单次入口）
- **版本管理**：同名多版本并存时哪个 active？规则：active 的是最新 activated 的，其他版本保留但不解析
- **威胁模型明示**：本版防御的是"LLM 幻觉生成越权代码"，**不防御**对抗性攻击（恶意用户刻意写 exploit）。后者需要 V8 Isolate / 进程隔离 / OS 沙盒，不在 v0.7.28 范围

### Timeline

v0.7.28 合并 087 + 088。ConstructionRuntime 基础设施 + Tool 生成首个 tier 一起验证发版。

## LLM 集成链路（activate → LLM 可用）

FEATURE_087/088 描述了"造工具"的流水线；本节说明 `activate()` 完成之后，新 Tool **如何真正被 LLM 调用**。这是跨两个 feature 的集成契约，必须在 v0.7.28 内闭环，否则 constructed tool 停留在"存在于磁盘但 LLM 看不到"的状态。

### 现状（builtin Tool → LLM）

KodaX 现行链路（v0.7.27）：

| 环节 | 位置 |
|------|------|
| Tool 注册表 | `packages/coding/src/tools/registry.ts:42` — `TOOL_REGISTRY: Map<string, RegisteredToolDefinition[]>`（末位赢语义） |
| Turn 级快照 | `packages/coding/src/agent.ts:1968` — `getActiveToolDefinitions()` 每 turn 开始时取一次 |
| 透传给 Provider | `packages/ai/src/providers/anthropic.ts:152` — `tools as Anthropic.Messages.Tool[]` 直接进 `tools` 字段 |
| Prompt 注入 | **无** — 无 "You have access to..." 类字符串；Tool 仅通过 API 的 `tools` 字段传递 |
| Dispatch | `packages/coding/src/tools/registry.ts:938` — `executeTool(name, input, ctx)` 查 registry 并调 handler |

**关键推论**：KodaX 不做 prompt 级 Tool 注入 → constructed tool 只要转成合法的 `KodaXToolDefinition`，就与 builtin 共用同一条到 LLM 的路径。**无需改动 system prompt pipeline**。

### 集成契约（v0.7.28 新增）

Constructed Tool 能被 LLM 调用，必须打通以下 5 个环节。每个环节标注归属 Feature（087 基础设施 / 088 生成流程）。

#### 1. Handler hydration — `string → ToolHandler`（087）

Manifest 中 handler 是字符串：

```json
"handler": { "kind": "script", "language": "typescript", "code": "export async function handler(input, ctx) {...}" }
```

但 `executeTool()` 要求内存中的 `ToolHandler` 函数引用（`registry.ts:956`）。

**实现方式（路径 B）**：Handler 作为**主进程模块**直接加载，不走 worker/vm 隔离层。

```ts
// @kodax/core/src/construction.ts
export interface HandlerLoader {
  load(source: ScriptSource, capabilities: Capabilities): Promise<ToolHandler>
  unload(handler: ToolHandler): Promise<void>
}

// 默认实现
class ImportHandlerLoader implements HandlerLoader {
  async load(source, capabilities) {
    // 1. 写 handler 源码到 .kodax/constructed/tools/<name>/<version>.compiled.js
    //    （TS → JS 编译用已有的 tsx/esbuild 链路）
    const modulePath = await writeCompiledModule(source)

    // 2. 动态 import 到主进程
    const module = await import(modulePath)

    // 3. 用 CtxProxy 包装 ctx，按 capabilities 限权
    const wrappedHandler: ToolHandler = async (input, ctx) => {
      const proxiedCtx = createCtxProxy(ctx, capabilities)
      return await module.handler(input, proxiedCtx)
    }

    return wrappedHandler
  }

  async unload(handler) {
    // ESM 无法真正 unload module；从 registry 移除即可
    // 已编译 .js 文件在 revoke 时标记废弃，后续重启时不再加载
  }
}
```

**关键点**：
- Handler 跑在主进程，**和 builtin tool 共享同一内存空间**
- 安全性来自**静态检查阶段就过滤掉危险代码**（Guardrail），不靠运行时隔离
- Timeout 通过 `AbortController` + `Promise.race` 实现，不依赖进程/worker 终止
- ESM 的 module cache 不支持 unload → revoke 只移 registry 条目，已编译文件留在磁盘供审计

**为什么不用 worker_threads**：参见 "安全模型" 小节的行业调研与决策理由。接口保留 `HandlerLoader` 抽象，未来想加 worker 版本作为可选 factory 不会破坏现有代码。

#### 2. Registry 注入（087）

`activate()` 的最后一步直接写入现有 `TOOL_REGISTRY`：

```ts
registerToolInternal({
  definition: {
    name: artifact.name,
    description: artifact.content.description,
    input_schema: artifact.content.inputSchema,
    handler: hydratedHandler,   // 来自步骤 1
  },
  source: {
    kind: 'constructed',        // 新增 kind
    version: artifact.version,
    manifestPath: `.kodax/constructed/tools/${artifact.name}/${artifact.version}.json`,
  },
})
```

利用 `TOOL_REGISTRY` 现有的**栈语义**（`registry.ts:92-98` "末位赢"）天然实现"constructed 覆盖同名 builtin"。`CapabilityResolver` 在此之上提供 kind-aware 查询（"只列 constructed" / "列全部"）+ manifest 索引枚举，但**不重复实现**查找逻辑。

**Non-breaking**：现有 `RegisteredToolDefinition.source` 类型扩展一个 `'constructed'` kind，builtin 路径完全不受影响。

#### 3. 生效时机（契约）

**activate 在 turn N 完成 → turn N+1 开始时自动对 LLM 可见。**

原因：`agent.ts:1968` 的 `getActiveToolDefinitions()` 在每个 turn 边界刷新快照，无需额外触发机制。

**Non-goal**：不支持 turn 内热加载。Agent 在同一 turn 里造的工具，**同 turn 内 LLM 看不到**（否则 LLM 上下文里的 `tools` 列表与实际可调用集合不一致，会产生幻觉调用）。文档里需要明写此语义，避免 Agent 在生成代码时产生"刚造完就用"的错误期望。

#### 4. Provider schema 兼容校验（088 test 阶段的一步）

不同 provider 的 Tool schema 支持范围不同：
- Anthropic: `input_schema`（JSON Schema 子集，不支持 `$ref`、对 `oneOf` 有限制）
- OpenAI: `parameters`（JSON Schema 子集）
- 部分 provider 不支持 `allOf` / `oneOf` / `anyOf`

当前代码（`anthropic.ts:152`）是**无校验强转** `tools as Anthropic.Messages.Tool[]`，constructed tool 的不兼容 schema 会**延迟到 LLM API 调用**才 400 报错。

`ConstructionRuntime.test()` **新增**一步静态校验：

```ts
export function validateToolSchemaForProvider(
  inputSchema: JsonSchema,
  provider: 'anthropic' | 'openai' | ...,
): { ok: boolean; errors: string[] }
```

若不兼容 → 返回 `TestResult.errors`，阻止 activate。

**v0.7.28 最小实现**：支持 `anthropic`（默认 provider）。其他 provider 的校验规则在后续版本增量补齐；用户在非默认 provider 下 activate 时给出 warning 而非 block（由 policy 决定）。

#### 5. Revoke 清理（087）

`revoke(name, version)` 语义：

| 动作 | 范围 |
|------|------|
| 从 `TOOL_REGISTRY[name]` stack 中移除匹配 `source.kind==='constructed' && source.version===version` 的条目 | 必达 |
| 若 stack 清空且存在同名 builtin，下次查询回落到 builtin | 天然生效（stack 语义） |
| 调 `HandlerLoader.unload(handler)` | 必达（本版为 no-op，仅为接口对称；ESM 无法真正 unload module） |
| 已编译 `.compiled.js` 文件在 `_manifest.json` 标记 `revokedAt`，保留磁盘供审计 | 必达 |
| **不中断**正在执行的 turn | 已发起的 tool call 继续完成；下个 turn 起该 tool 不再出现在 `getActiveToolDefinitions()` 输出中 |

**跨会话影响**：revoke 发生在会话内，但已 import 的 module 留在 Node module cache；**重启 KodaX 进程后**该 version 不会再被加载（`_manifest.json` 读取时跳过 `revokedAt` 非空的条目）。

## FEATURE_099: Provider Catalog Refresh — DeepSeek V4 + Kimi-code Label Collapse

**Status**: Completed

### 背景

DeepSeek 在 2026-04 发布 V4 系列（v4-pro / v4-flash），把上下文从 128K 提升到 1M、把推理从 V3 时代的 `enable_thinking` toggle 切成 OpenAI 风格的 `reasoning_effort`，同时官方公告 `deepseek-chat` 与 `deepseek-reasoner` 两个 V3 alias 将于 **2026-07-24 弃用**。KodaX 当时的 `deepseek` provider 仍以 `deepseek-chat` 作默认 model + `native-toggle` 推理 capability + 128K 上下文，与官方下一阶段不匹配。

## FEATURE_098: Per-Model Context Window & Output Token Limits

**Status**: Completed
**Priority**: High
**Category**: Internal

### 背景

KodaX 的 `KodaXModelDescriptor` 早就声明了 `contextWindow` 与 `maxOutputTokens` 字段（[packages/ai/src/types.ts:303-310](../../packages/ai/src/types.ts)），但全代码库**无人读取**：

- [base.ts:242](../../packages/ai/src/providers/base.ts) `getContextWindow()` 只读 `this.config.contextWindow`，不接 model 参数
- [base.ts:80](../../packages/ai/src/providers/base.ts) `getEffectiveMaxOutputTokens()` 同样只读 provider 级
- 4 处 compaction 调用点（[agent.ts:1701](../../packages/coding/src/agent.ts)、[commands.ts:317](../../packages/repl/src/interactive/commands.ts)、[repl.ts:397](../../packages/repl/src/interactive/repl.ts)、[managed-task/compaction.ts:83](../../packages/coding/src/task-engine/_internal/managed-task/compaction.ts)）全部 `provider.getContextWindow?.()`，未透传当前激活 model
- wire 层（[anthropic.ts:129/455](../../packages/ai/src/providers/anthropic.ts)、[openai.ts:232](../../packages/ai/src/providers/openai.ts)）调用 `getEffectiveMaxOutputTokens()` 时同样不传 model

后果：同 provider 不同 model 真实窗口差异大时，KodaX 一刀切。已确认偏差：

| Provider | Provider 级 contextWindow | 真实差异 model |
|---|---:|---|
| `kimi` | 256_000 | ~~`k2.5` 实际 128k~~ — **2026-04-25 核对实际 256K，无偏差**（commit `64785de` 已撤销该 pin）|

| `zhipu` / `zhipu-coding` | 200_000 | `glm-5-turbo` 实际 128k |

切到 `glm-5-turbo` 时压缩 trigger 虚高 → 实际撑爆服务端（k2.5 的偏差已撤回，详见节末"实施后更正"）。

### 关键洞察：架构能力已就位

- `reasoningCapability` —— [base.ts:149](../../packages/ai/src/providers/base.ts) 的 `getConfiguredReasoningCapability(modelOverride?)` 模式

两者的查找逻辑就是要复刻的样板。**不是新增能力，是把没接进来的两根线接上**。

#### 数据层（无破坏性）

`KodaXModelDescriptor` 已存在的字段 (`contextWindow` / `maxOutputTokens`) 直接用起来。

#### 解析层（base.ts）

**`getEffectiveContextWindow(model?: string): number`**
