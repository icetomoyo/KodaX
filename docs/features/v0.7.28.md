## 总览

v0.7.28 开启 KodaX **自构建段**（FEATURE_087-090）。本版合并两个 feature：

- **FEATURE_087 ConstructionRuntime**：引入 stage/test/activate/revoke 四段生命周期 + Constructed-World 存储 + Resolver 合并 + policy gate
- **FEATURE_088 自构建档 2 (Tool 生成)**：让 Agent 生成 Tool 定义并通过静态检查 + 能力校验后注册

合并理由：087 是基础设施，088 是它的**首个真实消费者**；一起做能验证基础设施的 API shape，避免 087 独立落地后 088 才发现接口需要调整。档 1 (Skill 生成) 作为既有能力的自然兑现（Agent 用 write 工具写 markdown），在本版自动可用，不单列 feature。

本版另搭载三个**已实施完成**的 provider catalog 卫生 feature（与自构建主线独立）：

- **FEATURE_099 Provider Catalog Refresh**：DeepSeek V4 系列接入（v4-flash 默认 + v4-pro alt）+ kimi-code 模型 label 收敛为 `kimi-for-coding` + 删除 deprecated `deepseek-chat` / `deepseek-reasoner`
- **FEATURE_098 Per-Model Context Window & Output Token Limits**：把 `KodaXModelDescriptor.contextWindow` / `maxOutputTokens` 这两个早已声明但全代码无人读取的死字段接通运行时，让 compaction trigger 与 wire-level max_tokens 都按当前激活 model 的真实窗口算；当时按 Moonshot 文档把 `kimi.k2.5` 当作 128k 做了一次 pin（**事后核对发现 K2.5 实际 256K，已于 commit `64785de` 改正，见 FEATURE_098 节末尾的"实施后更正"**）+ 修复 `zhipu.glm-5-turbo` (128k) 偏差；自定义 provider `models[]` 升级支持描述符对象格式（兼容旧字面量）。原计划 v0.7.29 同 FEATURE_078 同版发布，因实施提前完成迁入本版
- **`ark-coding` provider 接入**（commits `8b5a380` + `4784dbd`）：Volcengine Ark Coding Plan 多模型 gateway（routes by request `model` field，与 `kimi-for-coding` 服务端忽略 model 不同），走 AnthropicCompat 路径自动复用 `parseToolInputWithSalvage` + 现有 stack；落地时已通过 9 个路由 model 真实回归（glm-5.1 / glm-4.7 / kimi-k2.6 / kimi-k2.5 / minimax-latest / deepseek-v3.2 / doubao-seed-2.0-{code,pro,lite}）；5 个 model 加 per-model context window pin（`kimi-k2.5/k2.6` 256K、`minimax-latest` 204800、`doubao-seed-2.0-{code,pro,lite}` 256K）。落地后 KodaX 共 13 个 provider（AnthropicCompat 6 + OpenAICompat 5 + CLI bridge 2）

以上三条与 087/088 实施面无交集（catalog 数据 / provider transport vs construction runtime），合在一版便于一次发版同时获得多条改进。

## FEATURE_087: ConstructionRuntime and Constructed-World Substrate

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 你愿景中 KodaX 要成为"超级智能体的底座" —— 关键是**基于 Coding 能力自我构建**
- 要实现自构建，前提是 A/B 层所有概念都是 data（FEATURE_080/081/082 已做到）
- 但仅仅"能序列化"不够 —— 还需要运行时的**注册/加载/撤销**通道，以及**安全审批**机制
- 当前 KodaX 生成 skill 文件是可行的（agent 用 write tool 写 markdown），但生成 Tool 定义、Agent 定义都没有路径落地
- 必须区分两个世界：**Builtin World**（仓库里编译的，只读）vs **Constructed World**（运行时生成的，写入 `.kodax/constructed/`）

### 设计

#### ConstructionRuntime 接口

```ts
// @kodax/coding/src/construction/

export interface ConstructionArtifact {
  kind: 'agent' | 'tool' | 'skill' | 'handoff' | 'session-extension'
  name: string
  version: string                     // semver，支持同名多版本
  content: unknown                    // kind 对应的 data shape
  signedBy?: string                   // 可选签名
  createdAt: number
  testedAt?: number                   // 通过 test 阶段的时间戳
  sourceAgent?: string                // 生成此 artifact 的 agent 名
}

export interface StagedHandle {
  readonly artifact: ConstructionArtifact
  readonly stagedAt: number
}

export interface TestResult {
  ok: boolean
  errors?: string[]
  warnings?: string[]
  trace?: Trace                       // 来自 @kodax/tracing
}

export interface ConstructionRuntime {
  stage(artifact: ConstructionArtifact): Promise<StagedHandle>
  test(handle: StagedHandle, input?: unknown): Promise<TestResult>
  activate(handle: StagedHandle): Promise<void>
  revoke(name: string, version: string): Promise<void>
  list(kind?: ConstructionArtifact['kind']): Promise<ConstructionArtifact[]>
}
```

#### Constructed-World 存储

```
.kodax/constructed/
├── agents/
│   ├── my-security-reviewer/1.0.0.json
│   └── my-security-reviewer/1.1.0.json
├── tools/
│   ├── pii-scanner/1.0.0.json
│   └── ...
└── skills/
    └── (既有 skill 路径)

# 每个 artifact JSON 自身携带 status 字段（'active' | 'revoked'）
# 启动时 glob('.kodax/constructed/**/*.json') 扫盘 + 过滤 status==='active'
# 不维护单独索引文件，避免双写一致性问题
```

#### Resolver 合并

复用现有 `TOOL_REGISTRY` / `AGENT_REGISTRY` 末位赢栈语义（constructed push 在 builtin 之上，自动覆盖同名）。**不引入 Resolver 类**，仅加 view-layer 函数：

```ts
// @kodax/coding/src/construction/
export function listConstructed(kind?: ArtifactKind): RegisteredCapability[]
export function findByVersion(name: string, version: string): RegisteredCapability | undefined
export function listAll(kind?: ArtifactKind): RegisteredCapability[]
```

查找逻辑天然由 registry 现有 `getActiveToolRegistration()` 等函数承担——这三个函数仅用于"按 kind / version 过滤枚举"。

#### Policy Gate

Policy 是 `activate()` 调用时的决策点，签名是函数类型——**不引入 interface / class**，按 KodaX 哲学保持轻量。

```ts
// 类型别名，便于内部模块标注
export type ConstructionPolicy = (
  artifact: ConstructionArtifact,
) => Promise<'approve' | 'reject' | 'ask-user'>

// 默认值：什么都问用户
export const defaultPolicy: ConstructionPolicy = async () => 'ask-user'
```

v0.7.28 的 policy 入口仅有两个：
1. **REPL 默认**：`packages/repl/src/common/construction-bootstrap.ts` 在 REPL 启动时绑一个走 askUser dialog 的 policy（弹窗 approve / reject）
2. **非 REPL 表面**（ACP server / 单次 CLI / 子 Agent）：未绑定 askUser → 默认 `'reject'`，避免静默激活

**为什么不做声明式配置入口**：曾计划让用户在 `kodax.config.ts` 里 export 自定义 `constructionPolicy`，复审后认为违反 KodaX 哲学（CLAUDE.md "leverage LLM intelligence" + "NEVER add configuration for hypothetical needs"）——单用户场景下 REPL dialog 已够用；企业 / CI 场景目前无真实需求。**等真有用户需要 auto-approve / 一律 reject 的开关时**，更智能的做法是加一个 `risk_mode` 三档开关（`'strict' | 'balanced' | 'trusting'`），由 KodaX 根据 capabilities 自动判断风险等级；**不让用户写 policy 函数**。详见本文档末尾"Deferred Design Decisions"。

## FEATURE_088: Self-Construction Tier 2 — Tool Generation

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 档 1（Skill 生成）已经可用：agent 用 write tool 写 markdown 到 `~/.kodax/skills/` 或 `.kodax/skills/`，skill-loader 自动发现
- 档 2 让 agent 能**动态生成 Tool**（例如："我需要一个专门扫 OAuth scope 的工具"）
- 前提是 FEATURE_087 的 ConstructionRuntime + FEATURE_080 的 Tool-as-data

### 设计

#### Tool manifest 格式

```json
{
  "kind": "tool",
  "name": "oauth-scope-scanner",
  "version": "1.0.0",
  "content": {
    "description": "Scan a codebase for OAuth scope usage",
    "inputSchema": {
      "type": "object",
      "properties": {
        "rootPath": { "type": "string" },
        "includePatterns": { "type": "array", "items": { "type": "string" } }
      },
      "required": ["rootPath"]
    },
    "capabilities": {
      "tools": ["grep", "read"]
    },
    "handler": {
      "kind": "script",
      "language": "javascript",
      "code": "export async function handler(input, ctx) { /* 只能用 ctx.tools.*, 不能直接 require/fetch/process.env */ }"
    }
  }
}
```

**`capabilities` 字段**是本版核心安全机制（替代代码级 sandbox）：

- `tools`：声明可通过 `ctx.tools.<name>` 调用的 **builtin tool 白名单**

> **设计说明**：v0.7.28 仅保留单维 `tools`。所有 I/O（文件读写、网络、环境变量）一律通过对应 builtin tool（`read` / `write` / `bash` 等）进行；handler 不会拿到独立的 `ctx.fs` / `ctx.net` / `ctx.env` 入口。
>
> **演进路径**：当未来出现"必须把同一 builtin 限制到具体路径/域名"的真实场景时，再升级 schema 为：
>
> ```json
> "tools": [
>   { "name": "read", "constraints": { "path": ["./src/**"] } },
>   "grep"
> ]
> ```
>
> 即 `string | { name, constraints }` 联合类型。v0.7.28 不实现 constraints（YAGNI），但 schema 留向前兼容空间。

**Handler 是纯函数约束**：
- 输入 = `input`（通过 input_schema 校验）
- 输出 = return value（与 builtin tool 一致，不强制 outputSchema 校验）
- **禁止直接访问** Node.js API（`fs` / `net` / `process` / `require` / `import`）
- **所有 I/O 必须通过 `ctx.tools.<name>(...)` 代理**，代理调用会检查 `capabilities` 声明

#### 生成流程（由 Coding Agent 完成）

```
user: "生成一个扫 OAuth scope 的 tool"
   │
   ▼
Coding Agent (用 A 层原语):
   1. scaffold_tool: 根据需求写 Tool manifest（含 capabilities + handler code）
   2. static_check: Guardrail AST 静态校验
        - 禁用 require / import / fetch / process.env 等宿主 API
        - 所有 I/O 必须走 ctx.tools.*
        - handler 签名符合 (input, ctx) => Promise<output>
   3. ConstructionRuntime.stage(manifest)
   4. ConstructionRuntime.test(handle, sampleInputs)
        - 在主进程直接跑（已通过静态检查，无隔离需求）
        - 用 CtxProxy 注入 ctx，拦截未声明能力
        - 验证 input_schema 的 provider 兼容性
   5. 通过 policy gate（用户确认 — 默认 ask-user）
   6. ConstructionRuntime.activate(handle)
        - import handler 到主进程 TOOL_REGISTRY
        - 用 CtxProxy 包装 ctx，按 capabilities 限权
   │
   ▼
New tool available via TOOL_REGISTRY (constructed 末位赢覆盖同名 builtin)
```

#### 安全模型：能力声明 + I/O 代理（非代码级 sandbox）

**重要决策**：v0.7.28 **不采用代码级 sandbox**（不用 `worker_threads` / `isolated-vm` / `vm2` 包裹 handler 执行）。Handler 直接 `import` 到主进程运行。

**行业先例**：调研了 Anthropic 官方 Claude Code、pi-mono、opencode 三个同类项目，**无一对插件 / 扩展代码做 JS 级沙盒**：

| 项目 | 插件代码隔离 | 依赖的安全机制 |
|------|-------------|---------------|
| Claude Code | ❌ 不隔离（skills / MCP tools 全在宿主进程） | OS 级 bash sandbox + 人工审批 |
| pi-mono | ❌ 不隔离（jiti 直接加载 extension） | OS 级 bash sandbox（macOS/Linux） |
| opencode | ❌ 不隔离（`await import()` 直入宿主） | 无（完全信任用户安装的 plugin） |

**结论**：Node.js JS 层 sandbox 不是安全边界（Node 官方文档原话），绕过成本远低于封堵成本。行业共识是"代码级不隔离 + 系统调用级隔离 + 人工审批"。

**KodaX v0.7.28 采用的四层防御**（由弱到强）：

| 层 | 机制 | 拦截什么 |
|----|------|---------|
| ① 静态检查 | Guardrail AST 检查（FEATURE_085） | `require` / `import` / `fetch` / `process.*` / 全局对象访问 |
| ② 能力声明 | Manifest `capabilities.tools` 白名单 | 明示该 handler 需要哪些 builtin tool |
| ③ Ctx 代理 | `CtxProxy` 运行时拦截 | handler 通过 `ctx.tools.*` 访问时按 capabilities 校验 |
| ④ Policy Gate | `ConstructionPolicy` 函数（默认 ask-user，可由用户配置覆盖） | activate 前由用户决定信不信 |

**I/O 代理模型**：

```ts
// Handler 这么写：
export async function handler(input: Input, ctx: ToolCtx): Promise<Output> {
  //                                        ↑ CtxProxy（不是 Node API）

  // ✅ 允许：走代理（capabilities.tools 声明了 grep）
  const matches = await ctx.tools.grep({ pattern: 'oauth', path: input.rootPath })

  // ✅ 允许：logger / tracer 等无副作用工具
  ctx.logger.info(`Scanned ${matches.length} files`)

  return { matches }

  // ❌ 静态检查阶段就挂：
  // const fs = require('fs')        — Guardrail 拦截
  // fetch('https://evil.com')       — Guardrail 拦截
  // process.env.SECRET              — Guardrail 拦截
}
```

**CtxProxy 关键职责**：
- `ctx.tools.<name>()` — 调用前检查 `capabilities.tools` 白名单；命中后走已有的 `executeTool()` 路径（复用 builtin 的安全链路，如 bash 走 OS sandbox）
- 未声明的 tool → throw `CapabilityDeniedError`（记入 tracer）
- v0.7.28 单维设计：CtxProxy 只拦一种入口；未来若引入 `constraints` schema，再扩展为参数级校验

**Timeout 保护**：handler 执行超过 `manifest.content.timeoutMs`（默认 30s）→ `AbortError`。通过 `AbortController` 实现（主进程级），**不依赖 worker termination**。

**Non-goal（显式不做）**：
- 不防御**恶意** handler（LLM 生成的代码不是对抗性攻击模型）
- 不做 V8 Isolate / 进程级隔离（留给未来可选 factory）
- 不阻止 CPU / 内存耗尽（依赖 timeout 兜底）

#### 安全考虑

**Guardrail 静态检查 = 3 条硬规则 + LLM 审查**：

KodaX 哲学要求"leverage LLM intelligence"，不为对抗手写 AST 数百条规则。本版采用**两层组合**：

**第一层：3 条硬规则（AST 即可）**——便宜、必拦、永远命中：

| 规则 ID | 禁止 | 原因 |
|---------|------|------|
| `no-eval` | `eval(...)` 调用 | 任何 eval 是对静态检查的根本规避 |
| `no-Function-constructor` | `new Function(...)` | 等价于 eval |
| `require-handler-signature` | 必须导出 `async function handler(input, ctx)` | 统一 dispatch 入口；同时让"间接 require"等模式无落脚点 |

通过这 3 条 → 进入第二层。

**第二层：LLM 静态审查**——由 Coding Agent 主流程之外的一次独立 LLM 调用执行：

```text
prompt:
  你正在审查一段即将被注册到 KodaX 工具系统的 JavaScript handler。
  这是它声明的 capabilities：{capabilities.tools 列表}
  这是它的源码：{handler.code}
  请输出 JSON：
  {
    "verdict": "safe" | "suspicious" | "dangerous",
    "concerns": [...],
    "suggested_capabilities": [...]   // LLM 觉得它实际需要的 tools
  }
  关注以下模式（含变形）：
    - 任何形式的 require / dynamic import / Function constructor
    - 字符串拼接绕过引用（如 ['req', 'uire'].join('')）
    - process.* / globalThis.* / Buffer / __dirname / __filename
    - 全局 fetch / XMLHttpRequest / WebSocket
    - 顶层副作用（顶层 await、模块加载时的全局赋值）
    - capabilities.tools 之外的 ctx 访问
```

`verdict` 处置：
- `safe` → 通过，进入 stage
- `suspicious` → 进入 policy gate（默认 ask-user，把 concerns 一并展示给用户）
- `dangerous` → 直接 reject，不进 policy gate

**为什么这样组合**：
- 硬规则拦截最廉价、最确定的攻击模式（eval/Function/签名错误），**毫秒级、零 API 成本**
- LLM 审查覆盖语义级模式（字符串拼接、间接引用、混淆），**不可能用 AST 完美写规则**
- 失败成本极低：被 LLM 判 suspicious 只是多一次 ask-user，没有"漏放过攻击"
- 不重复造轮子：**handler 本就是 LLM 写的**，让另一次 LLM 审查同类代码非常自然

**其他安全机制**：

- Policy gate 对 tool 生成默认 ask-user（人工审批）
- Tool 有版本，revoke 后旧版本不再可用
- CtxProxy 运行时拦截：任何未在 capabilities 声明的 `ctx.*` 访问 → throw `CapabilityDeniedError`，记录到 tracer
- Handler 执行 AbortController + 30s timeout 默认值
- 所有 constructed tool 的调用在 tracer 里加 `source.kind='constructed'` 标签，便于事后审计

### Contract

**Public API（新增）**：
- `ConstructionRuntime` interface + default implementation
- `ConstructionArtifact` / `StagedHandle` / `TestResult` types
- `ConstructionPolicy` 函数类型 + `defaultPolicy`（用户可在配置里覆盖）
- View-layer 函数：`listConstructed()` / `findByVersion()` / `listAll()`
- CLI 入口（已实装，`src/constructed_cli.ts` + `src/kodax_cli.ts`）：
  - `kodax <constructed-tool-name> [args...]` —— **不打开 REPL** 直接派发到已激活的构造工具，handler 输出打到 stdout。args 按 `inputSchema` 解析：`--key=value` / `--key value` / 单 positional → 第一个 required string 字段。`kodax <tool> --help` 自动生成基于 schema 的帮助
  - `kodax tools list [--all]` —— 列已激活的构造工具（`--all` 加列 builtin / extension）
  - `kodax tools inspect <name>[@<version>]` —— 打印工件 manifest JSON
  - `kodax tools revoke <name>@<version>` —— 撤销并写盘 `status='revoked'`
  - **`activate` 不暴露在 CLI**：activate 必须经 policy gate；非 REPL 表面没有 askUser dialog，policy 默认 `'reject'` —— activate_tool 必须从 REPL 调起。这是 DD §14.5.4 的契约保护，CLI 故意没有绕道

**Internal**：coding agent 在 Runner 有 `scaffold_tool` / `validate_tool`（Guardrail 静态检查）/ `stage_construction` / `test_tool`（主进程试跑 + CtxProxy 能力校验）等内部 tool。

#### Coding Agent prompt 模式（仿照 74d4aaa Phase C）

`74d4aaa feat(prompts): give KodaX a self-identity and runtime awareness` 引入了**条件性 prompt section** 模式：当 AMA controller 的某个机器状态为真（如 `fanoutAdmissible=yes`）时，向 system prompt 追加一段可执行行为描述，**不进 base prefix**（避免污染缓存 + 不在不相关任务里出现）。

v0.7.28 给"造工具模式"**仿照同一模式**新建一个 conditional section：

| 触发条件 | 注入 section |
|---------|------------|
| Coding Agent 进入 tool-construction mode（用户显式请求 / 或 Agent 判断当前任务需要造工具） | `[Tool Construction]` slot |

`[Tool Construction Mode]` 内容要点：
- 描述五步流程：`scaffold_tool` → 编辑 JSON → `validate_tool` → `stage_construction` → `test_tool` → `activate_tool`（经 policy gate）
- **明示生效语义**："activate 完成后**下个 turn** 才可见，本 turn 不要尝试调用刚造的工具"（DD §14.6.2）
- 提示 `capabilities.tools` 必须显式声明 builtin tool 白名单；handler 仅能通过 `ctx.tools.<name>(...)` 调用
- 提示 `language` 限定 `'javascript'`；AST 硬规则禁用 `eval` / `new Function`、强制 `async function handler(input, ctx)`
- 版本不可变：覆盖 active 版本会被 `stage_construction` 拒绝，强制 bump semver

**实施位置**（已完成）：`packages/coding/src/prompts/sections.ts` 注册 `tool-construction` section（slot=`specialist`，order=100）；`prompts/tool-construction.ts` 持有 `TOOL_CONSTRUCTION_PROMPT` 文本与 `shouldIncludeToolConstructionSection` 门闸；`builder.ts` 按 `KodaXContextOptions.toolConstructionMode` 注入。结构与 `74d4aaa` 的 [AMA Behavior] 对齐——条件性、按 runtime 状态启停、不进 base 缓存。

### Non-Goals

- **不**支持 Agent 生成（档 3 → FEATURE_089 v0.7.31）
- **不**支持自改 role spec（档 4 → FEATURE_090 v0.7.32）
- **不**支持跨进程签名验证（本地签名够用）
- **不**自动发布 constructed artifact 到 npm / 公开 registry

- **Guardrail 绕过**：静态检查必然有盲区，但本版组合（3 条硬规则 + LLM 审查）覆盖语义级模式。对策：

  3. Policy gate 默认 ask-user，人工是最后防线

  1. CtxProxy 用 `Object.freeze` + 每次返回新对象，避免原型污染
  2. 所有 ctx.tools.* 实际调用点都走 capabilities 再校验（不依赖单次入口）
- **版本管理**：同名多版本并存时哪个 active？规则：active 的是最新 activated 的，其他版本保留但不解析

- **Standalone binary 模式动态 import 验证（已闭环 ✓）**：KodaX 现支持 `bun --compile` 打 standalone binary 分发。`loadHandler()` 用 `await import('file:///.../<version>.js')` 加载用户工作区的 constructed handler —— 这是 ESM 标准，理论上应可工作但历史上从未跑过此路径。冒烟手测（2026-04-26）：在 `dist/binary/win-x64/kodax.exe` 里 rehydrate 一个预先 staged 的 `count_lines@1.0.0` 工件 → `kodax.exe count_lines sample.txt` 跑通 → 手 handler 通过 `ctx.tools.read()` 调用 builtin → 返回正确行数。沿途修了一个**预先存在的** binary 启动 bug：`scripts/build-binary.mjs` 把 `--define process.env.NODE_ENV="production"` 用双引号传 spawnSync，Windows arg 管道剥掉了内嵌 `"`，导致 bun 把 `production` 当 bare identifier 替换，binary 启动即崩 `ReferenceError: production is not defined`。改为单引号 JS 字符串字面量后 round-trip 完整，定义保留为字符串。
- **Rehydrate 跳过 policy gate 的 supply-chain attack vector**（known limitation, v0.7.28 不修复）：`rehydrateActiveArtifacts()` 启动时扫盘把 `status='active'` 的工件**直接** registerTool，不走 policy gate（设计意图："这台机之前已批准过，重启不该重复问"）。这导致一个攻击面：如果 `.kodax/constructed/tools/` 进入版本控制，恶意 collaborator 可以提交一个 status=active 的 .json + 恶意 handler；用户 merge + 重启 KodaX 后**无任何提示**就 register 该工具。v0.7.28 缓解措施：(1) **强烈建议** `.kodax/constructed/` 进 .gitignore——这是单用户场景的预期使用模式；(2) 多 collaborator 共享 constructed tool 的场景留给 v0.7.29+ 的 fingerprint / trusted-source 机制（恢复时校验本机签名表，新 fingerprint 强制走 policy）。当前 v0.7.28 的威胁模型明示在"LLM 幻觉" 而非"恶意 git 协作者"；如果 future 出现后者场景必须先升 fingerprint。详见 DD §14.5.3 的同一条 known-limit 记录

## LLM 集成链路（activate → LLM 可用）

FEATURE_087/088 描述了"造工具"的流水线；本节说明 `activate()` 完成之后，新 Tool **如何真正被 LLM 调用**。这是跨两个 feature 的集成契约，必须在 v0.7.28 内闭环，否则 constructed tool 停留在"存在于磁盘但 LLM 看不到"的状态。

### 现状（builtin Tool → LLM）

KodaX 现行链路（v0.7.27）：

| 环节 | 位置 |
|------|------|
| Tool 注册表 | `packages/coding/src/tools/registry.ts:42` — `TOOL_REGISTRY: Map<string, RegisteredToolDefinition[]>`（末位赢语义） |
| Turn 级快照 | `packages/coding/src/agent.ts:1968` — `getActiveToolDefinitions()` 每 turn 开始时取一次 |
| 透传给 Provider | `packages/ai/src/providers/anthropic.ts:152` — `tools as Anthropic.Messages.Tool[]` 直接进 `tools` 字段 |
| Prompt 注入 | **无** — 无 "You have access to..." 类字符串；Tool 仅通过 API 的 `tools` 字段传递 |
| Dispatch | `packages/coding/src/tools/registry.ts:938` — `executeTool(name, input, ctx)` 查 registry 并调 handler |

**关键推论**：KodaX 不做 prompt 级 Tool 注入 → constructed tool 只要转成合法的 `KodaXToolDefinition`，就与 builtin 共用同一条到 LLM 的路径。**无需改动 system prompt pipeline**。

### 集成契约（v0.7.28 新增）

Constructed Tool 能被 LLM 调用，必须打通以下 5 个环节。每个环节标注归属 Feature（087 基础设施 / 088 生成流程）。

#### 1. Handler hydration — `string → ToolHandler`（087）

Manifest 中 handler 是 JavaScript 源码字符串：

```json
"handler": { "kind": "script", "language": "javascript", "code": "export async function handler(input, ctx) {...}" }
```

但 `executeTool()` 要求内存中的 `ToolHandler` 函数引用（`registry.ts:956`）。

**实现方式（路径 B）**：Handler 作为**主进程模块**直接加载，单个纯函数完成。

```ts
// @kodax/coding/src/construction/

export async function loadHandler(
  source: ScriptSource,
  capabilities: Capabilities,
): Promise<ToolHandler> {
  // 1. 写 handler 源码到 .kodax/constructed/tools/<name>/<version>.js
  //    限定 language='javascript' → 无编译步骤，直接写盘
  const modulePath = await writeModule(source)

  // 2. 动态 import 到主进程
  const module = await import(modulePath)

  // 3. 返回 CtxProxy 包装后的 handler
  return async (input, ctx) => {
    const proxiedCtx = createCtxProxy(ctx, capabilities)
    return await module.handler(input, proxiedCtx)
  }
}
```

**关键点**：
- Handler 跑在主进程，**和 builtin tool 共享同一内存空间**
- 安全性来自**静态检查阶段就过滤掉危险代码**（3 条硬规则 + LLM 审查），不靠运行时隔离
- Timeout 通过 `AbortController` + `Promise.race` 实现，不依赖进程/worker 终止
- **限定 language='javascript'**：LLM 直接写 JS 无成本；不引入 TS 编译链路（无 esbuild/tsx 依赖、无 source map、无类型纠结）。手写 TS handler 是 0% 当前场景，等真有需求再加
- ESM 的 module cache 不支持 unload → revoke 只移 registry 条目，已 import 的 module 留在内存（重启进程才彻底清；详见 §5）
- 不引入 `HandlerLoader` 接口：当前只有一种实现，按 KodaX 哲学"need it then add it"

#### 2. Registry 注入（087）

`activate()` 的最后一步直接写入现有 `TOOL_REGISTRY`：

```ts
registerToolInternal({
  definition: {
    name: artifact.name,
    description: artifact.content.description,
    input_schema: artifact.content.inputSchema,
    handler: hydratedHandler,   // 来自步骤 1
  },
  source: {
    kind: 'constructed',        // 新增 kind
    version: artifact.version,
    manifestPath: `.kodax/constructed/tools/${artifact.name}/${artifact.version}.json`,
  },
})
```

利用 `TOOL_REGISTRY` 现有的**栈语义**（`registry.ts:92-98` "末位赢"）天然实现"constructed 覆盖同名 builtin"。`listConstructed()` / `findByVersion()` / `listAll()` 三个 view-layer 函数提供 kind-aware 查询，**不重复实现**查找逻辑。

**Non-breaking**：现有 `RegisteredToolDefinition.source` 类型扩展一个 `'constructed'` kind，builtin 路径完全不受影响。

#### 3. 生效时机（契约）

**activate 在 turn N 完成 → turn N+1 开始时自动对 LLM 可见。**

原因：`agent.ts:1968` 的 `getActiveToolDefinitions()` 在每个 turn 边界刷新快照，无需额外触发机制。

**Non-goal**：不支持 turn 内热加载。Agent 在同一 turn 里造的工具，**同 turn 内 LLM 看不到**（否则 LLM 上下文里的 `tools` 列表与实际可调用集合不一致，会产生幻觉调用）。文档里需要明写此语义，避免 Agent 在生成代码时产生"刚造完就用"的错误期望。

#### 4. Provider schema 兼容校验（088 test 阶段的一步）

不同 provider 的 Tool schema 支持范围不同：
- Anthropic: `input_schema`（JSON Schema 子集，不支持 `$ref`、对 `oneOf` 有限制）
- OpenAI: `parameters`（JSON Schema 子集）
- 部分 provider 不支持 `allOf` / `oneOf` / `anyOf`

当前代码（`anthropic.ts:152`）是**无校验强转** `tools as Anthropic.Messages.Tool[]`，constructed tool 的不兼容 schema 会**延迟到 LLM API 调用**才 400 报错。

`ConstructionRuntime.test()` **新增**一步静态校验：

```ts
export function validateToolSchemaForProvider(
  inputSchema: JsonSchema,
  provider: 'anthropic' | 'openai' | ...,
): { ok: boolean; errors: string[] }
```

若不兼容 → 返回 `TestResult.errors`，阻止 activate。

**v0.7.28 最小实现**：支持 `anthropic`（默认 provider）。其他 provider 的校验规则在后续版本增量补齐；用户在非默认 provider 下 activate 时给出 warning 而非 block（由 policy 决定）。

#### 5. Revoke 清理（087）

`revoke(name, version)` 语义：

| 动作 | 范围 |
|------|------|
| 从 `TOOL_REGISTRY[name]` stack 中移除匹配 `source.kind==='constructed' && source.version===version` 的条目 | 必达 |
| 若 stack 清空且存在同名 builtin，下次查询回落到 builtin | 天然生效（stack 语义） |
| 已加载的 module 留在 Node module cache（ESM 无 unload 能力，本版不强求清理） | 接受 |
| artifact JSON 文件的 `status` 字段改为 `'revoked'`，`revokedAt` 写入时间戳；`.js` 源文件保留供审计 | 必达 |
| **不中断**正在执行的 turn | 已发起的 tool call 继续完成；下个 turn 起该 tool 不再出现在 `getActiveToolDefinitions()` 输出中 |

**跨会话影响**：revoke 发生在会话内，但已 import 的 module 留在 Node module cache；**重启 KodaX 进程后**该 version 不会再被加载（启动时 glob 扫盘 + 过滤 `status === 'active'`）。

## FEATURE_099: Provider Catalog Refresh — DeepSeek V4 + Kimi-code Label Collapse

**Status**: Completed
**Priority**: High
**Category**: Internal

### 背景

DeepSeek 在 2026-04 发布 V4 系列（v4-pro / v4-flash），把上下文从 128K 提升到 1M、把推理从 V3 时代的 `enable_thinking` toggle 切成 OpenAI 风格的 `reasoning_effort`，同时官方公告 `deepseek-chat` 与 `deepseek-reasoner` 两个 V3 alias 将于 **2026-07-24 弃用**。KodaX 当时的 `deepseek` provider 仍以 `deepseek-chat` 作默认 model + `native-toggle` 推理 capability + 128K 上下文，与官方下一阶段不匹配。

同时盘点现有 catalog 时发现两处长期不一致：

1. `kimi-code` provider 在 [registry.ts:230](../../packages/ai/src/providers/registry.ts) 注释里已记录「服务端忽略请求里的 model 字段，永远 route 到当前 K2.x GA」，但 KodaX 仍在选择菜单里暴露 `K2.6` / `k2.5` 两个版本号 label —— 用户选哪个都不影响实际路由，反而造成「以为切到 K2.5 实际还是 K2.6」的预期错配。
2. DeepSeek 与 Kimi 都同时支持 OpenAI / Anthropic 双协议，需要做协议选择决策（避免 cache_control 损失或 fidelity 损失）。

### 设计

#### DeepSeek V4 接入决策：走 OpenAI 兼容路径

| 维度 | OpenAI 兼容 (`https://api.deepseek.com`) | Anthropic 兼容 (`https://api.deepseek.com/anthropic`) |
|---|---|---|
| Prompt Cache | ✅ 原生（命中价 ¥0.2 / ¥1 元/M） | ❌ `cache_control` 被忽略 |
| 模型 ID | v4-pro / v4-flash 直接对应 | 仅 v4-pro 显式支持，其他名自动 fallback 到 v4-flash |
| 已有 KodaX 适配 | DeepSeekProvider 继承 `KodaXOpenAICompatProvider`，最小改动 | 需新增 anthropic compat provider 路径 |
| `reasoning_effort` | 原生支持 | `output_config.effort`，`thinking.budget_tokens` 被忽略 |

1M 上下文场景下 prompt cache 是 5-12x 折扣的关键路径，Anthropic 兼容路径会丢失整个红利 → 选 OpenAI 兼容。

#### Kimi-code label 收敛决策

服务端忽略 model 字段是 KodaX 已知事实，继续暴露 `K2.6` / `k2.5` label 是不诚实的 UI。把 `kimi-code` 默认 model 改成单一 `kimi-for-coding` alias，向用户清晰传达"服务端给你最新 GA"的实际语义；旧 label 完全移除。

注：**保留 Anthropic 协议接入**而不切到 OpenAI 兼容（与 DeepSeek 决策相反）。理由：Kimi K2 系列在 Moonshot 端点是**自动前缀缓存**（无需 `cache_control` 标记），缓存红利与协议无关；KodaX 内部 agent loop 是 Anthropic 风格，工具调用 fidelity 最高；项目已经有 `kimi`(OpenAI/moonshot) + `kimi-code`(Anthropic/coding) 的有意双协议覆盖架构，切换会消除差异化价值。

#### 删除 deprecated chat/reasoner

`deepseek-chat` 与 `deepseek-reasoner` 在 2026-07-24 弃用。KodaX 提前移除而不保留兼容（v0.7.28 是合适的窗口，发版后给用户充分迁移时间）。同步清理 [openai.ts](../../packages/ai/src/providers/openai.ts) 里 `native-toggle` 分支中针对 `deepseek-chat` 的 V3 时代 thinking toggle 死代码（`extra_body.thinking = { type: 'enabled' }`）。

### Contract

**Public API**：
- `deepseek` provider default model 从 `deepseek-chat` → `deepseek-v4-flash`
- `deepseek` provider 可选 model 列表新增 `deepseek-v4-pro`，移除 `deepseek-chat` / `deepseek-reasoner`

- **不**切 kimi-code 到 OpenAI 兼容路径 —— K2 自动前缀缓存让协议选择对成本中性

## FEATURE_098: Per-Model Context Window & Output Token Limits

**Status**: Completed
**Priority**: High
**Category**: Internal

### 背景

KodaX 的 `KodaXModelDescriptor` 早就声明了 `contextWindow` 与 `maxOutputTokens` 字段（[packages/ai/src/types.ts:303-310](../../packages/ai/src/types.ts)），但全代码库**无人读取**：

- [base.ts:242](../../packages/ai/src/providers/base.ts) `getContextWindow()` 只读 `this.config.contextWindow`，不接 model 参数
- [base.ts:80](../../packages/ai/src/providers/base.ts) `getEffectiveMaxOutputTokens()` 同样只读 provider 级
- 4 处 compaction 调用点（[agent.ts:1701](../../packages/coding/src/agent.ts)、[commands.ts:317](../../packages/repl/src/interactive/commands.ts)、[repl.ts:397](../../packages/repl/src/interactive/repl.ts)、[managed-task/compaction.ts:83](../../packages/coding/src/task-engine/_internal/managed-task/compaction.ts)）全部 `provider.getContextWindow?.()`，未透传当前激活 model
- wire 层（[anthropic.ts:129/455](../../packages/ai/src/providers/anthropic.ts)、[openai.ts:232](../../packages/ai/src/providers/openai.ts)）调用 `getEffectiveMaxOutputTokens()` 时同样不传 model

后果：同 provider 不同 model 真实窗口差异大时，KodaX 一刀切。已确认偏差：

| Provider | Provider 级 contextWindow | 真实差异 model |
|---|---:|---|
| `kimi` | 256_000 | ~~`k2.5` 实际 128k~~ — **2026-04-25 核对实际 256K，无偏差**（commit `64785de` 已撤销该 pin）|
| `qwen` | 256_000 | `qwen3.5-plus` 社区报告 1M（官方未明确，本版未修） |
| `zhipu` / `zhipu-coding` | 200_000 | `glm-5-turbo` 实际 128k |

切到 `glm-5-turbo` 时压缩 trigger 虚高 → 实际撑爆服务端（k2.5 的偏差已撤回，详见节末"实施后更正"）。

### 关键洞察：架构能力已就位

项目里 **2 个对称字段**已经走 model 级 lookup 路径成功工作：

- `thinkingBudgetCap` —— anthropic.opus=28k / haiku=10k 已差异化生效（[registry.ts:188-189](../../packages/ai/src/providers/registry.ts)）
- `reasoningCapability` —— [base.ts:149](../../packages/ai/src/providers/base.ts) 的 `getConfiguredReasoningCapability(modelOverride?)` 模式

两者的查找逻辑就是要复刻的样板。**不是新增能力，是把没接进来的两根线接上**。

### 设计

#### 数据层（无破坏性）

`KodaXModelDescriptor` 已存在的字段 (`contextWindow` / `maxOutputTokens`) 直接用起来。
`KodaXCustomProviderConfig.models` 从 `string[]` 升级为 `Array<string | KodaXModelDescriptor>`：

```ts
// 兼容旧字面量
"models": ["codellama", "mistral"]

// 新对象格式
"models": [
  { "id": "codellama", "contextWindow": 16000, "maxOutputTokens": 4096 },
  { "id": "mistral",   "contextWindow": 32000 }
]
```

#### 解析层（base.ts）

升级两个 effective getter，按以下优先级查找：

**`getEffectiveContextWindow(model?: string): number`**

1. `descriptor.contextWindow`（model-level，新增）
2. `config.contextWindow`（provider-level，现有）
3. `200_000`（兜底）

注：用户级 `compaction.contextWindow` 仍由调用点叠加做 manual override，最高优先级。

**`getEffectiveMaxOutputTokens(model?: string): number`**（升级现有签名）

1. `setMaxOutputTokensOverride()` 一次性（agent escalation/recovery，现有）
2. env `KODAX_MAX_OUTPUT_TOKENS`（现有）
3. `descriptor.maxOutputTokens`（model-level，新增）
4. `config.maxOutputTokens`（provider-level，现有）
5. `KODAX_MAX_TOKENS`（兜底）

`getContextWindow()` 保留作 ergonomic 包装，内部调用 `getEffectiveContextWindow(this.config.model)`。

#### 透传层（5 处调用点）

把当前激活 model 传进 effective getter：

- [anthropic.ts:129/455](../../packages/ai/src/providers/anthropic.ts) — wire-level max_tokens
- [openai.ts:232](../../packages/ai/src/providers/openai.ts) — 同上
- [agent.ts:1701](../../packages/coding/src/agent.ts) — auto-compaction trigger
- [commands.ts:317](../../packages/repl/src/interactive/commands.ts) — `/compact` 命令
- [repl.ts:397](../../packages/repl/src/interactive/repl.ts) — REPL trigger
- [managed-task/compaction.ts:83](../../packages/coding/src/task-engine/_internal/managed-task/compaction.ts) — managed task 内部

#### 数据修复（registry.ts）

只在**真有差异**的 model 上加描述符 override，避免冗余：

- ~~`kimi.models`: `{ id: 'k2.5', contextWindow: 128_000 }`（k2.6 与 provider 级一致，不加）~~ — **commit `64785de` 撤销**：核对 K2.5 实际 256K，与 provider 级一致，无需 per-model override
- `zhipu.models` / `zhipu-coding.models`: `glm-5-turbo` 加 `contextWindow: 128_000`
- `qwen.qwen3.5-plus`：本版**不修**。社区报告 1M 但官方 dashscope 文档无明确单一来源，留作后续单独 fix。

#### 自定义 provider 同步升级

[custom-provider.ts:43-61](../../packages/ai/src/providers/custom-provider.ts) `buildProviderConfig` 接受新格式（描述符数组）+ 字面量数组兼容：

```ts
const models = custom.models?.map(entry =>
  typeof entry === 'string' ? { id: entry } : entry
);
```

[config.example.jsonc](../../config.example.jsonc) 注释更新示例 + 补充说明：用户级 `compaction.contextWindow` 仍可作为 manual override，最高优先级；新部署只在自定义 provider 没声明精确 model 数据时使用。

### Contract

## Deferred Design Decisions

> 本节登记 v0.7.28 设计阶段提出但**最终撤回 / 推迟**的设计点，便于后续重新提出时不重蹈覆辙。

### `kodax.config.ts` 自定义 `constructionPolicy`（撤回，2026-04-26）

**原设计**：用户在项目根的 `kodax.config.ts` 里 export 一个 `ConstructionPolicy` 函数，KodaX 启动时 import 它当 activate gate。示例规则：

- `signedBy === 'self'` && `Date.now() - testedAt < 24h` → `'approve'`
- 涉及 `bash` / `write` 的工具 → `'ask-user'`
- 企业 / CI 环境一律 `'reject'`

**撤回理由**：

1. **违反 KodaX 哲学**：CLAUDE.md 明文 "NEVER add configuration for hypothetical needs" / "leverage LLM intelligence"。让用户写 policy 函数 = 把心智负担丢给用户去理解 `ConstructionArtifact` shape、写 TS、维护它
2. **没有真实需求**：v0.7.28 实际用户群是单用户开发者，REPL dialog 已 100% 覆盖 approve / reject 流程；企业 / CI 部署目前不存在
3. **更智能的替代方案存在**：未来真有需求时，加一个 `risk_mode` 三档开关（`'strict' | 'balanced' | 'trusting'`）由 KodaX 自己根据 `capabilities.tools` 风险等级判断，比让用户写函数智能得多

**当前状态**：
- REPL surface：`packages/repl/src/common/construction-bootstrap.ts` 绑 askUser dialog policy（已实装）
- 非 REPL surface：默认 `'reject'`（已实装）
- 没有声明式 override 入口

**未来重新激活的触发条件**：用户**明确反馈**单一 policy 行为不满足某个生产场景。届时按 `risk_mode` 枚举开关方向做，而非函数 hook。
