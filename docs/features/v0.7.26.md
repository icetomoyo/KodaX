# v0.7.26 Feature Designs

> 中文标题：`v0.7.26 Feature 设计`
>
> Version: `v0.7.26`
>
> Status: Planned (not yet released)
>
> Theme: Runtime rewrite — Scout/Generator/Evaluator on Layer A primitives + Guardrail tri-layer runtime

---

## 总览

v0.7.26 是整条路线图**最大体量**的版本。FEATURE_084 把 Scout / Generator / Evaluator 从 task-engine.ts 内的隐式角色重写为**标准 KodaXAgent 实例**，同时用 tool-call 驱动的结构化协议**替换** fenced-block 文本协议（吸收 FEATURE_059 的 dual-track visibleText+protocolPayload 目标）。FEATURE_085 在 Agent 的 input / output / tool 三个位点补齐 Guardrail runtime。两者合并是因为 084 重写时必然 touch Agent 的这三个位点，同时挂 guardrail 钩子比分版各做一遍经济。

---

## FEATURE_084: Task Engine Phase 2 — Rewrite Scout/Generator/Evaluator on Layer A Primitives

**Status**: Planned  
**Priority**: High  
**Category**: Core

> Absorbs FEATURE_059: The dual-track `visibleText + protocolPayload` goal originally scoped for v0.8.0 is now delivered by this feature — replacing fenced-block text parsing with tool-call-driven structured protocol is a natural outcome of re-expressing Scout/Generator/Evaluator as declarative Agent instances.

### 背景

- v0.7.22-v0.7.24 完成了 task-engine 拆分、Agent primitive 引入、SA 路径 dog-food、包重组、tracing 就位 —— 所有基础到位
- AMA 路径（Scout/Generator/Evaluator）仍是 task-engine 内部的隐式角色 + fenced-block 文本协议，没有利用新原语
- FEATURE_059 原设计（v0.8.0）目标："dual-track visibleText + protocolPayload" —— 与 084 重写必须同步做
- 内部 "SA 走新 Runner / AMA 走旧路径" 的双轨状态不能长期存在

### 设计

#### Scout / Generator / Evaluator 作为 Agent 实例

```ts
// packages/coding/src/agents/coding-agents.ts

export const scoutAgent: Agent = {
  name: 'scout',
  instructions: (ctx) => buildScoutInstructions(ctx),
  tools: [read, grep, glob, bash, emit_scout_verdict],   // 含新协议工具
  handoffs: [
    { target: generatorAgent, kind: 'continuation', description: 'Upgrade to H1 — execute + evaluate' },
    { target: plannerAgent,   kind: 'continuation', description: 'Upgrade to H2 — plan + execute + evaluate' },
  ],
  reasoning: { default: 'quick', escalateOnRevise: false },  // FEATURE_078 v0.7.29 填行为
  guardrails: [scoutInputGuardrail, scoutVerdictOutputGuardrail],  // FEATURE_085 挂点
}

export const generatorAgent: Agent = {
  name: 'generator',
  instructions: (ctx) => buildGeneratorInstructions(ctx),
  tools: [write, edit, bash, read, grep, run_tests, emit_handoff],
  handoffs: [{ target: evaluatorAgent, kind: 'continuation' }],
  reasoning: { default: 'balanced', max: 'deep', escalateOnRevise: true },
}

export const evaluatorAgent: Agent = {
  name: 'evaluator',
  instructions: (ctx) => buildEvaluatorInstructions(ctx),
  tools: [read, run_tests, emit_verdict],
  handoffs: [
    { target: generatorAgent, kind: 'continuation', description: 'revise' },
    { target: plannerAgent,   kind: 'continuation', description: 'replan' },
  ],
  reasoning: { default: 'balanced', max: 'deep' },
}
```

#### 结构化协议替换（吸收 FEATURE_059）

**旧**：Scout / Planner / Evaluator 在响应文本末尾 emit fenced block：
````
```kodax-task-verdict
{
  "status": "accept",
  "summary": "Tests pass, diff is minimal"
}
```
````
parse by string scanning.

**新**：每个角色有专门的 emit tool，LLM 通过 tool call 发送结构化 payload：

```ts
// @kodax/capabilities/src/agents/tools/protocol-emitters.ts

export const emit_scout_verdict: Tool = {
  name: 'emit_scout_verdict',
  description: 'Emit the scout verdict (harness tier + downstream hints)',
  inputSchema: {
    type: 'object',
    properties: {
      harness: { enum: ['h0_direct', 'h1_execute_eval', 'h2_plan_execute_eval'] },
      scopeFacts: { type: 'array', items: { type: 'string' } },
      downstream_reasoning_hint: { type: 'string', enum: ['quick', 'balanced', 'deep'] },
      // ...
    },
    required: ['harness'],
  },
  execute: async (input, ctx) => {
    ctx.session.append({ type: 'scout-verdict', payload: input })
    return { visible: renderScoutVerdict(input), protocolPayload: input }
  },
}
```

**双轨输出**：
- **visibleText**：用户看到的自然语言解释
- **protocolPayload**：系统内部消费的结构化 JSON（`scout-verdict` / `contract` / `handoff` / `verdict` entry）

FEATURE_059 的 "degraded continuation when protocol is malformed" 在此版**不再需要** —— LLM 调用 tool 的成功/失败由 provider 保证，不再靠文本解析。

#### Harness 状态机用 Handoff 表达

`Runner` 根据 Agent.handoffs 执行角色转移。原 H0 / H1 / H2 状态机的转移逻辑变成纯 handoff 选择：

- Scout 的 `emit_scout_verdict` tool 返回 harness tier → Runner 查找对应 handoff 目标（Generator for H1, Planner for H2），触发 continuation handoff
- Evaluator 的 `emit_verdict` tool 返回 revise/replan → Runner 触发对应 handoff

原 `runManagedTask` 的 round 控制逻辑变成 Runner 的 handoff 链路 + budget tracking + managed-task durable state 读写。

#### FEATURE_076 兼容

`runManagedTask` 出口的 `context.messages` 语义（FEATURE_076）在新 Runner 架构下继续保持"干净的用户对话"形态 —— 用户只看到 Agent-as-a-whole 的最终输出，内部 handoff 产生的 worker messages 不泄漏到 `context.messages`。

## FEATURE_085: Guardrail Tri-Layer — Input / Output / Tool

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- FEATURE_080 (v0.7.23) 只引入了 Guardrail 占位类型，没有 runtime
- 外部 SDK 消费者想在边界做验证/过滤（内容审核、schema 校验、PII 过滤）没有挂点
- FEATURE_084 重写 Scout 时要 touch Agent 的 input/output/tool 三个位点 —— 同时挂 guardrail runtime 是经济选择

### 设计

#### 三档 Guardrail

```ts
// @kodax/core/src/guardrail.ts

export interface InputGuardrail extends Guardrail {
  readonly kind: 'input'
  check(input: Message[] | string, ctx: GuardrailContext): Promise<GuardrailVerdict>
}

export interface OutputGuardrail extends Guardrail {
  readonly kind: 'output'
  check(output: Message, ctx: GuardrailContext): Promise<GuardrailVerdict>
}

export interface ToolGuardrail extends Guardrail {
  readonly kind: 'tool'
  beforeTool?(tool: Tool, input: unknown, ctx: GuardrailContext): Promise<GuardrailVerdict>
  afterTool?(tool: Tool, result: unknown, ctx: GuardrailContext): Promise<GuardrailVerdict>
}

export type GuardrailVerdict =
  | { action: 'allow' }
  | { action: 'rewrite'; payload: unknown }
  | { action: 'block'; reason: string }
  | { action: 'escalate'; reason: string }    // 触发 user confirmation
```

#### 运行时钩子点

Runner 在三个位点调用 guardrails：

1. **Input**：`Runner.run(agent, input)` 启动时
2. **Output**：Agent 产生 final output 之前
3. **Tool before / after**：每次 tool execute 前后

#### 现有集成

- 当前的 `applyToolResultGuardrail`（packages/coding/src/tools/tool-result-policy.ts）归入 ToolGuardrail 的 afterTool 钩子
- permission 系统（plan/accept-edits/auto-in-project）保留为**产品级 UX**，不合并到 Guardrail（permission 与 guardrail 是不同层次）
- FEATURE_085 提供的 Guardrail 是 SDK 级能力

#### Extension 注册

```ts
// 扩展可以注册自己的 guardrail
extension.registerGuardrail(myInputGuardrail)
extension.registerGuardrail(myToolGuardrail)
```

### Contract

**Public API（新增 runtime）**：`InputGuardrail` / `OutputGuardrail` / `ToolGuardrail` / `GuardrailVerdict` / `GuardrailContext` 类型；`Runner` 接受 `guardrails` 选项；`Agent.guardrails` 字段实装。

**Public API（保留）**：`applyToolResultGuardrail` 作为 ToolGuardrail 的一个实现保留，外部 API 不变。

### Non-Goals

- **不**替换 permission 系统（permission 是产品级 UX，guardrail 是 SDK 级）
- **不**实装远程 guardrail（全部本地 inline）
- **不**引入 guardrail 的异步批处理优化（性能问题留到未来）

### Dependencies

- **前置**：FEATURE_080 (v0.7.23) Guardrail 占位类型；FEATURE_082/083 (v0.7.24) 包重组 + tracing
- **后置**：
  - FEATURE_086 (v0.7.27) 清理 legacy `applyToolResultGuardrail` 调用路径
  - FEATURE_088 (v0.7.28) 自构建 Tool gen 时 sandbox 测试用 guardrail 做静态检查

### Testing

- 单元：三档 Guardrail 的 allow / rewrite / block / escalate 各路径
- 集成：Runner 集成 Guardrail 的完整流程
- 集成：SA 路径挂 guardrail 不影响既有行为
- 集成：AMA 路径（新 Agent 运行时）挂 guardrail 不影响既有行为
- 回归：`applyToolResultGuardrail` 现有 behavior 保持

### Risk

- **延迟叠加**：三档 guardrail 顺序执行可能累积延迟；Runner 内部并行化 input + output guardrail，tool guardrail 不可并行（必须在 tool execute 前后）
- **GuardrailContext 数据量**：`ctx` 里要不要有 session / tracer / budget？选择最小必要集合，避免成为另一个"dumping ground 接口"

### Timeline

v0.7.26 合并 084 + 085。AMA 重写 + 协议替换 + Guardrail runtime 一起发版。这是整条路线图最重的一版，发版前需要：
- 完整端到端测试跑绿（所有既有 coding 工作流场景 regress）
- Scout → Generator → Evaluator → revise → accept 五条典型路径用 Span 观测链完整
- FEATURE_076 的 `context.messages` 干净用户对话契约跑通
