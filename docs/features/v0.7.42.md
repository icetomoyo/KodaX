# v0.7.42 Feature Designs

> 中文标题：`v0.7.42 Feature 设计`
>
> Version: `v0.7.42`
>
> Status: Planned (not yet released)
>
> Theme: Deep Anti-Escape Hardening — runtime detection + retry contract to stop LLMs bailing to Python/bash heredoc for large file writes

---

## 总览

v0.7.26 已经交付了三层浅层防御（P0 系统 prompt 软化、P2a `multi_edit` 工具、P2b `max_output_tokens` write-turn cap）。这三层针对的是"LLM 把大文件写拆成 bash `cat <<EOF` / Python `open().write()` 脚本"的绕行行为（以及 Windows "编码问题" 借口）。

但实战日志（Kimi-Code / MiniMax / Zhipu-Coding 等非 Anthropic provider）显示，仍有一定比例的任务在 P2b cap + P2a 工具可见后继续落进 bash 脚本写大文件的通路。原因：P0/P2a/P2b 都在"LLM 愿意合作"的假设下起作用，没有在**运行时检测到绕行行为之后做自动纠偏**。

v0.7.42 的 FEATURE_094 补上最后一层：**运行时检测 + 自动重试合同 (retry contract)**。在 `bash` 工具的 `beforeTool` 阶段识别 "生成型大文件写" 的 shell 签名，直接拒绝执行并向模型回送结构化提示，要求用 `write` 或 `multi_edit` 重写。这是 FEATURE_085 ToolGuardrail runtime 的第二个官方消费者（第一个是 FEATURE_092 Auto Mode Classifier）。

## FEATURE_094: Deep Anti-Escape Hardening — Runtime Detection and Retry Contract for Generative Large-File Writes

**Status**: Planned
**Priority**: Medium
**Category**: Core

### 背景

v0.7.26 已落地的三层防御：

| 层 | 机制 | 失效场景 |
|---|---|---|
| P0 | 系统 prompt 软化：去掉 "HARD LIMIT"，改 skeleton+edit 叙事；write 工具描述新增 anti-bash-generation 条款 + Windows 编码问题解释 | 模型不完全遵循 prompt；部分 provider 对 system 部分压得轻 |
| P2a | `multi_edit` 工具：原子批量单文件 edit，给 skeleton+edit 模式一个第一公民工具 | 模型没用工具就直接 `bash cat <<EOF`，P2a 无法保护 |
| P2b | write-turn `max_output_tokens` 封顶 8K（针对 RST-prone provider）：限制单回合可生成的 token 预算，避免长生成触发流重置 | 模型用 bash `cat <<EOF` 绕开，总输出量仍可长达 32K+，bash stdin 是独立通道不受 P2b 影响 |

线上样本（观察日期 `2026-04` 起）：Kimi-Code / MiniMax-Coding / GLM-Coding 在大文件任务上仍有 ~15% 的 bash-heredoc 绕行率。典型签名：

```bash
cat > packages/foo/bar.ts <<'EOF'
... several hundred lines ...
EOF
```

```bash
python3 <<'PY'
content = """... several hundred lines ..."""
open("packages/foo/bar.ts", "w", encoding="utf-8").write(content)
PY
```

这些调用**语义上是 "generative large-file write"**（内容不是计算得来，就是模型一次性吐出的），**形态上是 bash 工具调用**。它们具备以下共同签名：

1. 单条 bash 命令中出现 heredoc 边界标记（`<<'EOF'`、`<<EOF`、`<<PY`、`<<'PY'`）
2. heredoc body 长度 ≥ N 行（默认 N=80）或 ≥ M 字符（默认 M=3000）
3. heredoc 目标是向文件系统写入内容（`cat >`、`tee`、`python ... open(... 'w')`、`echo > file`）
4. 被写入的路径命中项目 tree（不是 `/tmp` 临时调试）

### 设计

#### Runtime 检测器

新增 `coding/src/tools/anti-escape-guardrail.ts`，作为 `ToolGuardrail` 实现（挂 `beforeTool`）：

```typescript
export const antiEscapeGuardrail: ToolGuardrail = {
  name: 'anti-escape',
  beforeTool: async (call, ctx) => {
    if (call.name !== 'bash') return undefined;
    const command = String(call.input?.command ?? '');
    const verdict = detectGenerativeLargeFileWrite(command);
    if (!verdict.flagged) return undefined;
    // Tool-level block with structured hint (the LLM sees this as the
    // tool result content; Runner surfaces it as a failed tool call so
    // the next turn can retry with write / multi_edit).
    return {
      decision: 'block',
      reason: buildRetryContractHint(verdict),
    };
  },
};
```

核心的 `detectGenerativeLargeFileWrite` 基于上面四条签名做检测，允许 env-var 关闭 (`KODAX_DISABLE_ANTI_ESCAPE_GUARDRAIL=1`)。

#### Retry Contract Hint

当 guardrail 阻断时，返回给 LLM 的 reason 不是干瘪错误，而是结构化的重试指引：

```
Blocked: this bash command is a generative large-file write (heredoc body
{LINES} lines / {CHARS} chars targeting {PATH}). Use `write` (or `multi_edit`
for skeleton+fill) instead. Reason: heredoc-in-bash bypasses write-tool
safety (encoding normalisation, path policy, CRLF preservation, atomic
backup).

Retry with:
  write {
    file_path: "{PATH}",
    content: <your heredoc body>,
  }
```

这样下一回合 LLM 有明确的纠正路径（工具名、参数名、原路径），不用再猜。

#### 白名单

下列 bash 场景**不是** "generative large-file write"，必须放行：

- heredoc body 是"计算出来的"模板（有 `${VAR}` 内插、backtick 执行、`$(...)` 子进程）
- 目标是 `/tmp/**`、`.agent/**`、`/var/**`、`node_modules/**` 等非项目 tree 路径
- 命令显式包含 shell pipe / redirection 链（`|`、`&&`、`||`）——说明内容参与 shell 组合
- heredoc body < 80 行且 < 3000 字符（小内容用 bash 也没问题）

白名单采用"加法策略"：检测器默认保守，命中任一白名单条目就放行。

#### Telemetry

每次 guardrail 触发写入一条 `[anti-escape:blocked]` debug log（复用 `emitResilienceDebug`）：

```json
{
  "provider": "kimi-code",
  "tool": "bash",
  "flag": "heredoc-large-write",
  "lines": 320,
  "chars": 12450,
  "target": "packages/repl/src/shell.tsx"
}
```

这个日志足以在 3 次触发内判断是"单次样本"还是"provider/prompt 组合问题"，为后续打磨 prompt 或调阈值提供数据。

### 验收

#### 功能

- [ ] 识别 `cat > path <<EOF ... EOF`，body ≥ 80 行即阻断
- [ ] 识别 `python3 <<PY ... open(...'w').write(...) ... PY`
- [ ] 识别 `tee path <<EOF` / `tee -a path <<EOF`
- [ ] 不阻断内容小于阈值、包含 `${VAR}`、含管道、目标在 `/tmp` 的 bash 调用
- [ ] Retry hint 里包含目标 `file_path` 和建议工具名

#### 测试

- Unit: `anti-escape-guardrail.test.ts`，覆盖四类阻断签名 + 四类白名单
- Integration: runner-driven 侧挂 guardrail 后，mock provider 发 bash `cat <<EOF` 大 body 请求，期待收到 `decision: 'block'` 并在下一回合看到 `write` 工具调用
- Provider-smoke: 选 Kimi-Code / MiniMax-Coding / GLM-Coding 三档，在已知触发样本上抽样跑 10 次，绕行率应从 ~15% 降到 < 3%

#### 回归

- 现有 bash 工具测试全绿（guardrail 默认挂但对非绕行场景零干预）
- 有 `KODAX_DISABLE_ANTI_ESCAPE_GUARDRAIL=1` 可兜底关闭
