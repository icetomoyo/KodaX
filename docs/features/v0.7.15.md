# v0.7.15 Feature Designs

> 中文标题：`v0.7.15 Feature 设计`
>
> Version: `v0.7.15`
>
> Status: Planned
>
> Theme: provider stream resilience and graceful recovery

---

## 总览

`v0.7.15` 的目标不是“把 retry 次数调大”，而是把 provider / 网络波动正式建模成一条可恢复的执行语义。

KodaX 当前已经有基础重试、stream completeness 检测和 idle watchdog，但用户体感仍然会在几类场景里突然从“正常执行”掉到“像是死掉了”：

- provider 短暂超时或断流后，被误判成 permanent failure
- 等待下一次重试时，没有明确的 countdown / next action 语义
- `request timeout`、`stream idle timeout`、`chunk timeout` 混在一起，行为不够可控
- 已经出现过部分输出或局部 tool-call 输入时，缺少“最后稳定边界”概念，恢复策略偏粗
- 某些 provider 明明给了 `Retry-After` 或 overload 信号，但当前没有统一利用

`v0.7.15` 需要把这些问题收束成一套符合 KodaX “极简且智能”哲学的系统：

- 极简：默认行为足够好，用户不需要理解 transport 细节
- 智能：系统能区分失败阶段、保留稳定状态、选择合适恢复路径
- 克制：不引入嘈杂的新模式，不把 provider 差异直接暴露给用户

## FEATURE_045: Provider Stream Resilience and Graceful Recovery

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_045` 把 provider 请求失败从“一个最终 error”升级为“一个有阶段、有边界、有恢复梯度的 runtime event”。

它覆盖的不是业务工具逻辑，也不是一般意义上的 task retry，而是：

- provider request 建连失败
- provider 接受请求后迟迟不返回首包
- streaming 中途断流 / 空闲停滞 / incomplete finish
- provider rate limit / overload
- streaming transport 降级为 non-streaming transport 的兜底恢复

### Product philosophy

#### 1. 成功恢复时尽量安静

如果失败发生在用户尚未感知到结果之前，且恢复成本很低，系统应优先自动修复，而不是把底层噪音直接抛给用户。

#### 2. 等待明显时必须可感知

一旦进入肉眼可感知的等待区间，必须明确告诉用户：

- 发生了什么
- 正在做什么
- 还要等多久
- 用户能否取消

#### 3. 永远从“最后稳定边界”恢复

系统不能因为 provider 断流，就把未完成的 assistant turn 或未完成的 tool-call 当成可靠上下文继续传播。

#### 4. 不为了自动化而重放副作用

如果本地 tool 已经执行，provider retry 不能偷偷重放这个 tool。
恢复必须以“稳定结果已落地”为前提，而不是以“最好别让用户看见失败”为前提。

#### 5. 少量配置，大量默认

只提供少数真正有价值的 resilience knobs：

- request timeout
- stream idle timeout
- chunk timeout
- max retries
- max retry delay
- non-streaming fallback 开关

其余由系统根据失败阶段和 provider 能力自动决定。

### Why now

KodaX 目前已经具备以下能力，但缺少统一收束：

- provider stream completeness 校验
- coding 层 `withRetry` 和 jittered exponential backoff
- REPL retry history item
- provider compatibility policy

问题不在于“没有零件”，而在于零件之间缺少统一语义层。

`v0.7.15` 需要补上的，就是这层 `resilience contract`。

### Scope

- 适用于 `SA` 与 `AMA`
- 适用于 CLI / Ink UI / JSON event surface
- 适用于 OpenAI-compatible、Anthropic-compatible，以及 CLI-bridge provider
- 只处理 provider / transport / stream 相关失败

### Non-goals

- 不处理通用 shell/tool 失败重试
- 不默认开启跨 provider 自动 failover
- 不承诺所有 provider 都支持 mid-stream resume
- 不引入新的 execution mode

## User Experience Contract

### State model

provider 执行路径需要显式区分以下用户可感知状态：

1. `connecting`
2. `streaming`
3. `recovering`
4. `degraded_fallback`
5. `awaiting_manual_continue`
6. `failed`

### Recovery messaging rules

#### A. 首包前失败

如果失败发生在首个 assistant delta 之前：

- 自动重试
- 使用紧凑状态文案
- 默认不刷大段错误正文

示例：

`⏳ Provider request timed out · retry 1/3 in 3s`

#### B. 已出现可见输出后失败

如果用户已经看到文本或 thinking：

- 保留当前 live transcript
- 明确标注“连接不稳，正在继续”
- 不把这段未完成输出直接当作稳定 turn 落库

示例：

`⏳ Stream interrupted after partial output · reconnecting 2/3 in 8s`

#### C. tool-call 输入流阶段失败

如果只是在 tool arguments streaming 中断，且 tool 尚未执行：

- 丢弃未完成 tool-call 作为“非稳定边界”
- 从上一稳定 turn 重新请求
- UI 明确提示“未执行任何工具”

#### D. tool 已执行后失败

如果本地 tool 已经执行并产出了稳定结果：

- 禁止自动重放 tool
- 恢复以“已有 tool_result”为边界继续
- 必要时让模型从“请基于已有 tool result 继续”恢复，而不是重做工具

#### E. 重试耗尽

如果所有自动恢复路径都失败：

- 给出简洁失败摘要
- 说明系统保留了什么状态
- 提供显式 `continue` 语义，而不是让用户猜

## Failure Taxonomy

`FEATURE_045` 引入统一失败分类，区分“是什么错”和“错在什么阶段”。

### Error classes

- `rate_limit`
- `provider_overloaded`
- `request_timeout`
- `stream_idle_timeout`
- `chunk_timeout`
- `connection_failure`
- `incomplete_stream`
- `user_abort`
- `non_retryable_provider_error`

### Failure stages

- `before_request_accepted`
- `before_first_delta`
- `mid_stream_text`
- `mid_stream_thinking`
- `mid_stream_tool_input`
- `post_tool_execution_pre_assistant_close`

恢复策略必须由 `error class x failure stage` 共同决定，不能只看 message substring。

---

## Stable Boundary Model

`FEATURE_045` 引入 “最后稳定边界（stable boundary）” 概念。

### Stable boundary definition

以下内容可以被视为稳定：

- 上一个完整 assistant message
- 已持久化的 tool result
- 已完成的 user turn
- 已完成的 compacted context snapshot

以下内容不稳定：

- live streaming text buffer
- live thinking buffer
- 尚未闭合的 tool-call JSON
- 尚未被本地执行确认的 tool intent

### Runtime artifacts

执行期间需要维护一份最小恢复快照：

- `requestId`
- `provider`
- `model`
- `attempt`
- `failureStage`
- `lastStableMessageIndex`
- `executedToolCallIds`
- `pendingToolCallIds`
- `visibleLiveTextLength`
- `visibleThinkingLength`
- `fallbackUsed`

该快照服务于恢复决策，不直接作为新用户可见产品概念。

## Recovery Ladder

`FEATURE_045` 采用固定恢复梯度，避免实现散落在各层。

### Ladder order

1. `same-request retry with fresh connection`
2. `same-turn retry from stable boundary`
3. `streaming -> non-streaming fallback`
4. `manual continue / manual intervention`

### Rules

#### Rule 1: 优先 fresh connection，而不是原连接继续读

对于 `ECONNRESET`、`socket hang up`、`other side closed` 等错误，应优先重新建立连接。

#### Rule 2: provider 要求的等待时间可以遵守，但必须有上限

当 provider 给出 `Retry-After` / `retry-after-ms` / `x-ratelimit-reset*` 时：

- 允许遵守
- 但必须经过 `maxRetryDelayMs` 裁剪
- 超限时转为用户可见状态，而不是无限安静等待

#### Rule 3: stream fallback 必须是显式的“降级”，不是隐式换协议

当 streaming 多次失败，但 provider 支持 non-streaming：

- 可以切到 non-streaming
- 必须记录 `fallbackUsed`
- UI 用一条紧凑说明标记

#### Rule 4: 任何本地副作用完成后，不再自动回滚或重放

tool 一旦执行成功，就只能基于结果继续，不再重放。

## Configuration Model

### Global config

建议新增顶层 resilience 配置：

```jsonc
{
  "providerResilience": {
    "requestTimeoutMs": 600000,
    "streamIdleTimeoutMs": 60000,
    "chunkTimeoutMs": 30000,
    "maxRetries": 3,
    "maxRetryDelayMs": 60000,
    "enableNonStreamingFallback": true
  }
}
```

### Per-provider override

允许在 custom/built-in provider 上做少量覆盖：

```jsonc
{
  "providerPolicies": {
    "newapi-anthropic": {
      "streamIdleTimeoutMs": 45000,
      "chunkTimeoutMs": 20000,
      "maxRetryDelayMs": 20000
    }
  }
}
```

### Design constraints

- 所有字段都可选
- 未配置时使用统一默认值
- 不暴露 backoff factor、jitter range 这类低层调参项

## Architecture Changes

### 1. `@kodax/ai`

职责：

- 统一 provider 失败归一化
- 解析 server retry hints
- 区分 request timeout / chunk timeout / incomplete stream
- 暴露 provider transport fallback 能力

建议新增：

- `KodaXRetryHint`
- `KodaXTransientFailure`
- provider-level `extractRetryDelay`
- provider-level `supportsNonStreamingFallback`

### 2. `@kodax/coding`

职责：

- 基于 failure stage 决定恢复路径
- 管理 stable boundary
- 确保 tool side effect 不被重放
- 统一 retry/backoff/cancel 逻辑

建议新增：

- `ProviderRecoveryCoordinator`
- `StableTurnBoundary`
- `RecoveryAction`
- `ProviderExecutionState`

### 3. `@kodax/repl`

职责：

- 展示 retry status / countdown / fallback notice
- 在 transcript 和 status bar 之间做最小噪音表达
- JSON output 输出结构化 retry event

建议新增 JSON event 字段：

- `stage`
- `reasonCode`
- `delayMs`
- `nextAt`
- `recoveryAction`
- `fallbackUsed`

## Detailed Behavior Matrix

| Failure stage | Default action | User-visible behavior |
|---|---|---|
| Before first delta | Auto retry with fresh connection | Compact retry line |
| Mid text/thinking stream | Retry from stable boundary | Keep partial live output, show reconnecting |
| Mid tool input stream, tool not executed | Drop incomplete tool intent and retry | Show “tool not executed” note |
| After tool executed, before assistant close | Continue from tool result boundary | Show “continuing after completed tool” |
| Rate limit with acceptable delay | Wait and retry | Show countdown |
| Rate limit with too-large delay | Stop auto wait, surface explicit choice | Show capped-delay notice |
| Streaming repeatedly fails, non-streaming available | Fallback to non-streaming once | Show degraded mode note |

## Telemetry and Debuggability

`FEATURE_045` 需要让 retry 可观测，但不污染默认 UX。

### Default user-facing telemetry

- retry reason
- attempt / maxAttempts
- remaining wait
- whether fallback is active

### Debug telemetry

在 debug 模式下额外记录：

- raw provider error
- normalized error class
- failure stage
- chosen recovery action
- whether fresh client / fresh connection was used
- whether server delay hint was honored or capped

## Acceptance Criteria

- [ ] `timed out` / `connection reset` / `stream stalled` / `incomplete stream` 被稳定归类为 transient 或 staged failure
- [ ] retry 等待可被用户 abort 立即打断
- [ ] request timeout / stream idle timeout / chunk timeout 可配置
- [ ] `Retry-After` 等 server delay hint 可被解析并受上限约束
- [ ] UI 和 JSON mode 都能表达 `recovering` 与 `degraded_fallback`
- [ ] incomplete tool-call 不会污染下一轮上下文
- [ ] 已执行 tool 不会因 provider retry 被自动重放
- [ ] streaming 多次失败时，支持一次受控 non-streaming fallback
- [ ] 自动恢复成功时 transcript 保持紧凑，不制造“假报错墙”
