## 总览

`v0.7.25` 的目标不是把 KodaX 做成更重的 terminal product。

这一版真正要完成的是：

- 把 transcript 做成稳定主屏
- 把输入、滚动、搜索、选择这些底层机制补成熟
- 把 footer / task / status 的表面控制在 KodaX 合适的克制档位

这意味着：

- 学 ClaudeCode 的机制成熟度
- 不复制 ClaudeCode 的重表面和可见复杂性

## FEATURE_051: Host-Aware Fullscreen TUI Substrate and Transcript UX

**Status**: Completed  
**Close-out note (2026-04-03)**: Implemented with host-aware transcript profiles, decoupled verbosity/follow/buffering state, fullscreen-or-fallback transcript control, and stable hidden-work strip integration.  
**Close-out addendum (2026-04-04)**: Final REPL TUI close-out freezes the current visible status/footer/task/message surfaces and keeps follow-up work focused on invisible substrate maturation: transcript owner, viewport/search/navigation, host-aware scroll/selection/copy, and input focus/paste/cursor behavior.  
**Handoff note (2026-04-04)**: Remaining REPL substrate hardening moved to `FEATURE_055` in `v0.9.0` and was completed there without reopening the visible shell; `051` should be read as the completed visible-shell close-out record.  
**Priority**: High  
**Category**: Enhancement

### Overview

`FEATURE_051` 把 KodaX 的 TUI 明确成三层：

1. `host-aware substrate`
2. `transcript-first interaction model`
3. `minimal, frozen visible shell`

前几轮已经完成了 transcript owner、prompt input controller、footer/task/status 的基础壳层。  
最终收尾的目标不是再长更多表面，而是把这些壳层背后的 owner、controller、host policy 和 budget 真正做稳。

### Product philosophy

#### 1. transcript 是绝对主屏

用户始终面对一个主 authority。  
transcript 是主工作面，其他都是辅助仪表。

#### 2. Adaptive inside, minimal outside

内部机制可以更智能、更 host-aware，但外部不要求用户理解系统内部组织。

#### 3. mechanism first, surface restraint

继续补 transcript / scroll / selection / input 这些 invisible substrate。  
停止扩更多长期可见的表面。

#### 4. cockpit, not control tower

KodaX 可以像 AI 工作台驾驶舱，但应该是：

- 一个主屏
- 一条克制仪表带
- 少量按需出现的 overlay

而不是：

- 更多并列 panel
- 更重 task dashboard
- 可见 worker topology

### Final close-out posture (2026-04-04)

`FEATURE_051` 在用户可见层面已经达到 KodaX 目标档位，后续收尾遵循：

- transcript 继续是唯一主屏
- 当前 `StatusBar` 的内容、顺序、符号和信息密度冻结
- 当前 footer / help / task / message action / tool row 的可见表面冻结或尽量冻结
- 后续实现只允许继续补 owner / controller / host policy / budget 这些 invisible substrate

这条口径的目的不是降低成熟度，而是让 KodaX 保持：

- 更极简
- 更智能
- 更克制
- 更少长期可见 surface

### Goals

1. 在不扩 visible surface 的前提下，把 transcript / scroll / selection / input 这几层 substrate 补到成熟。
2. 保持 transcript 仍是唯一主屏，让 `InkREPL` 继续只做 orchestration shell，而不是重新长回大 owner。
3. 让不同 terminal host 走清晰的 capability-driven 策略，尤其是 selection、copy-on-select、wheel history 和 fullscreen chrome。
4. 冻结当前 `StatusBar`、footer、task bar、message/tool surface 的用户可见档位，只允许内部整理和行为闭环。
5. 学 ClaudeCode 的机制成熟度，但不复制更重的工作台表面。

### Scope

- transcript chrome / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner、budget 和 view-model 收口
- `FEATURE_051` 文档与 feature index 的最终收尾口径同步

### Non-goals

- 复制 ClaudeCode 的重表面和可见复杂性
- 继续扩更多长期可见的 footer/help/task/message surface
- 引入 visible worker graph、task tree、teammate dashboard、persistent notification center
- 改变当前 `StatusBar`、footer、task bar、message actions、tool rows 的用户可见密度
- 用一次性 renderer rewrite 替代 owner / controller / host policy 的渐进收口

### Current pain to resolve

#### A. transcript owner 还不够强

owner 骨架已经有了，但 sticky prompt、jump-to-latest、search anchor、reveal policy 还没完全内聚到 owner 层。

#### B. scroll / selection / copy 机制还不够一体化

manual scroll、page nav、selection clear、copy-on-select、search disarm 还没完全变成一套 host-aware 规则。

#### C. input core 在 focus / paste / cursor 这些方面还没最终闭环

这类问题不会要求新 UI，但会直接影响“手感”和稳定性。

### State model

`FEATURE_051` 的状态模型在前几轮已经成型；最终收尾不再扩新的用户可见 mode，而是把现有语义彻底收口。

```ts
type TranscriptVerbosity = "compact" | "verbose";
type TranscriptFollowMode = "follow-bottom" | "browsing-history";
type LiveBufferingPolicy = "live" | "buffered-fallback";

interface TranscriptDisplayState {
  verbosity: TranscriptVerbosity;
  followMode: TranscriptFollowMode;
  buffering: LiveBufferingPolicy;
  scrollAnchor: number;
  jumpToLatestAvailable: boolean;
  stickyPromptVisible: boolean;
  selectionMode: "none" | "message";
  selectedItemId?: string;
  searchMode: "closed" | "history";
  searchAnchorItemId?: string;
  currentMatchIndex: number;
}
```

最终收尾约束：

- `verbosity`、`followMode`、`buffering` 继续独立，不再彼此偷语义
- `searchMode` 只表达 transcript history search，不再挟带显示密度或 review 语义
- `currentMatchIndex = -1` 合法，表示 query 仍在，但当前 match 已被 manual scroll / page nav disarm
- `selectedItemId` 只在 host 支持 viewport-owned selection path 时进入正式交互路径

### Host policy matrix

最终收尾阶段，host classification 不再只是声明能力，而要真正驱动 selection / copy / wheel 行为。

| Host class | Fullscreen / overlay | Selection path | Copy behavior | Wheel / browse policy |
|---|---|---|---|---|
| `native_vt` | 开启 owner-driven chrome | 开启 | copy-on-select + 显式复制 | 完整 wheel history |
| `xtermjs_host` | 开启，但更保守 | 开启 | 只保留显式复制 | 保守 wheel history |
| `degraded_vt` | 不开启 fullscreen chrome | 不进入 viewport-owned selection path | 不启用 copy-on-select | 仅保守 fallback |
| `unsupported_control_host` | append-only / fallback only | 不进入 | 不启用 | 不进入 |

### Shortcut contract

快捷键集合在前几轮已基本确定；最终收尾只修规则，不再扩新的 visible action。

- `Ctrl+O`: toggle transcript verbosity
- `PgUp`: enter browsing-history
- `Ctrl+Y`: explicit history browsing toggle
- `Alt+Z`: ergonomic backup for history browsing
- `Esc`: leave browsing-history
- `End`: jump to latest and restore follow-bottom
- `c`: copy selected transcript entry
- `i`: copy selected tool input
- `v`: toggle detail for selected entry

最终收尾约束：

- 上述键位集合不再增加新的用户可见动作
- `c / i / v` 的可见性与可用性继续由 host capability 和当前 selection state 决定
- 输入层继续保证优先级：`dialog/request > prompt editing > transcript navigation > global shortcuts`

### Component responsibility split

这部分在前几轮已经部分落地；最终收尾只做“把行为真相放对地方”，不再新增表面组件。

#### `InkREPL.tsx`

- 继续负责 runtime/session state、顶层 mode、surface 出现条件和 view-model 组装
- 不重新拿回 transcript / footer / task / status 的行为真相

#### `FullscreenTranscriptLayout.tsx`

- 保留当前可见 chrome
- 继续作为 sticky header / jump-to-latest / overlay / footer 层级关系的 owner

#### `TranscriptViewport.tsx`

- 保留当前 browse/search/selection surface 的可见样子
- 继续下沉 search anchor、match stepping、selection reveal、tool row reveal 等行为真相

#### `prompt-input-controller.ts`

- 继续作为输入逻辑 owner
- 保持 `Enter/Esc/Tab/Up/Down` 的优先级和 paste / focus / cursor 行为收口
- 不把逻辑退回 `InputPrompt` / `TextInput` 条件树

#### `StatusBar.tsx`

- 可见表现冻结
- 只作为 renderer；格式化和策略继续往 view-model builder 收

### Final design direction

#### 1. Keep the current visible shell

`FEATURE_051` 的最终目标不是“再长更多 UI”，而是：

- transcript 继续是唯一主屏
- footer / help / task / message / tool 表面维持当前档位
- `StatusBar` 可见文本保持不变

#### 2. Continue strengthening invisible substrate

真正还要继续做的是：

- transcript owner / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner 与 budget 收口

#### 3. Prefer host-aware correctness over faux parity

KodaX 不追求“所有 host 看起来都一样强”，而是：

- safe host 上提供成熟行为
- weak host 上收窄到正确行为
- 不用重表面掩盖弱能力路径

## FEATURE_054: AMA-Project Convergence — Absorb Project Mode into Adaptive H2

**Status**: Planned
**Priority**: High
**Category**: Refactor

> Source-of-truth note (2026-04-04):
> The active planned version for `FEATURE_054` is now [v0.7.30](./v0.7.30.md#feature_054-ama-project-convergence-absorb-project-mode-into-adaptive-h2).
> This section remains as historical staging context only.

### Overview

`FEATURE_054` 将独立的 `Project` 模式（`/project` 命令组）完全融合进 AMA H2 的自适应流程。

当前 KodaX 存在两套独立的规划系统叠在一起：

- **Project 模式**（REPL 层）：`/project init / brainstorm / plan / next / auto / quality / verify / status`，8 个文件，独立持久化 `.agent/project/`
- **AMA 模式**（task-engine 层）：`Scout -> Planner -> Generator <-> Evaluator`，独立持久化 `.agent/managed-tasks/`

两套系统互不感知，同一任务可能产出两份计划（Project 的 `session_plan.md` 和 AMA 的 `kodax-task-contract`），违背了 KodaX 极简且智能的构建哲学。

### Product philosophy

#### 1. 用户入口极简

用户不需要学 `/project init / brainstorm / plan / auto` 这套命令。直接说需求，AMA 全自动处理。

#### 2. 智能自适应

AMA 根据请求复杂度自动选择 H0/H1/H2。H2 已经包含完整的 plan -> execute -> evaluate 流程，不需要外部命令来手动编排。

#### 3. Brainstorm 是 Planner 的内建能力

不是独立 Agent 或独立阶段。Planner 在信息不充分时使用 `ask_user_question` 进行多轮对齐，然后生成 `kodax-task-contract`。

#### 4. one main authority，不能复活第二套 project surface

`FEATURE_054` 的目标不是把旧 `Project` 模式换个壳留下来，而是彻底消除“第二套规划入口”和“第二套任务表面”：

- 不恢复 project dashboard / planner panel / project footer
- 不保留与 AMA 并行存在的 project workflow state machine
- 不让用户再学习一套 `/project ...` 交互语法

### Problem analysis

#### 矛盾 1：两层规划互相不知道对方的存在

```text
/project plan → buildProjectPlan() 确定性生成 4-phase plan
/project next → runManagedTask(taskSurface='project')
  → AMA routing 可能选 H2 → Planner 生成自己的 contract（不知道 session_plan.md 的存在）
```

结果：同一任务有两个计划。

#### 矛盾 2：Project 的 plan 是确定性的，但执行是自适应的

`buildProjectPlan()` 用关键词匹配推断复杂度，但真正执行时 AMA 的 Scout 可能判定完全不同的复杂度。两者的 Harness 级别、角色组合、预算可能完全不匹配。

#### 矛盾 3：两个独立的状态机

```text
Project workflow state: bootstrap → discovering → aligned → planned → executing → completed
AMA managed task lifecycle: created → running → [rounds] → completed/blocked
```

两个状态机没有任何同步点。

#### 矛盾 4：Brainstorm 能力的空壳化

AMA 的 `inferRequiresBrainstorm()` 只是往 prompt overlay 加了一句"请先讨论"，没有独立阶段、没有状态、没有持久化、没有结构化输出。而 Project 模式的 brainstorm 是完整的多轮对齐机制。

### Goals

1. **废弃 `/project` 命令组**——所有命令全部移除，用户不再需要学任何 `/project` 命令。
2. **Planner 吸收 brainstorm 能力**——Planner 在需求不清晰时自动进入多轮对齐模式，对齐完成后生成 contract。
3. **统一持久化**——`.agent/project/` 不再存在，所有状态统一存入 `.agent/managed-tasks/<id>/`。
4. **更新 DD.md §9.4**——将"正交维度"描述改为"融合进 AMA H2"。

### Scope

- 废弃 `packages/repl/src/interactive/project-*.ts`（8 个文件）
- 增强 Planner 的 brainstorm 能力（prompt + task-engine 层面结合）
- 持久化路径统一
- 文档更新（DD.md、FEATURE_LIST.md）

### Non-goals

- 自动迁移用户已有的 `.agent/project/` 数据（由用户自行处理）
- 修改 H0/H1 的行为（它们不涉及 Project 模式）
- 重写 task-engine.ts 的核心 managed task lifecycle
- 保留或重建独立的 project dashboard / planner surface / project-mode UX
- 在 transcript / footer / task bar 里新增 “project system” 专属表面

### Core decisions

#### Decision 1: `/project` 命令组全部废弃

| 废弃命令 | 替代方案 |
|---------|---------|
| `/project init` | 用户直接说需求，AMA 自动创建 managed task |
| `/project brainstorm` | Planner 的 brainstorm 工作模式（自动触发或用户说"先讨论"） |
| `/project plan` | AMA Planner 自动生成 contract |
| `/project next/auto` | AMA round-based execution 自动推进 |
| `/project quality/verify` | AMA Evaluator 自动检查 |
| `/project status` | 用户自然语言询问进度，AMA 从 managed task 读取回答 |

#### Decision 2: Brainstorm 是 Planner 的内建能力

Planner 接收 Scout 的 scope facts 后有两种自然过渡的工作模式：

- **需求清晰**：直接生成 `kodax-task-contract`
- **需求模糊**：用 `ask_user_question` 多轮对齐（brainstorm）→ 再生成 contract

这两种模式由 Planner 自行判断，不需要外部路由切换。

## FEATURE_073: Reference-Style Lineage and Island Model Removal

**Status**: **Cancelled (2026-04-19)**
**Priority**: ~~Medium~~
**Category**: Internal / Refactor
**Builds on**: `v0.7.20 FEATURE_072` (lineage-native compaction migration)
**Reference implementations**: pi-mono `buildSessionContext`, Claude Code `getMessagesAfterCompactBoundary`

### Cancellation rationale

After iterating the design through two rounds of adversarial challenge (local + GLM), the accumulated evidence pointed away from shipping this feature:

1. **Zero user pain.** `FEATURE_072` already closed the v0.7.18 post-compact monotonic-growth regression and made `lineage` the single source of truth for the compaction hot path. There is no live bug, user complaint, or regression that 073 resolves.
2. **No measurable improvement.** Every performance axis came back at "parity, not improvement":
   - Memory: reference-style + retargeted eviction is at parity with copy-style + `evictOldIslandMessageContent` (same `[compacted]` placeholder, same footprint).
   - Path walk cost: requires introducing `stopAtCompaction` to *avoid regression*, not to improve.
   - Reconciliation cost: 073 has to bypass `createSessionLineage` and hand-roll a replacement — new code, not less.
3. **Withdrawn user-visible benefit.** The early draft pitched "`/fork` to pre-compaction entry gives full history" as an improvement. Adversarial review surfaced a 20MB+/fork memory cost on long sessions, and the pitch was withdrawn via Design §8 `stopAtCompaction`-bounded fork. After the withdrawal, the only remaining user-visible change is a cosmetic session-tree rendering tweak (multi-island → single-chain).
4. **Residual motivation is YAGNI.** What remains is "align with pi-mono / Claude Code so *future* features (partial compaction, multi-boundary, `cache_edits`-style) become easier". Those features are not planned in any roadmap version. [docs/CLAUDE.md](../CLAUDE.md) is explicit: *"NEVER add 'flexibility' for hypothetical future needs (YAGNI)"*, *"NEVER create abstractions until you have 3+ concrete use cases"*. 073 fails both rules.
5. **Complexity price was real.** Cancelling avoids: a `compactionStyle` per-entry tag branching throughout all lineage code, a dual eviction function set (`evictOldIslandMessageContent` + `evictPreCompactionMessageContent`), a `physical` vs `logical` active-path split, a permanent mixed-shape session compat layer, a declared downgrade incompatibility, and a hand-rolled replacement for `createSessionLineage` in the compaction path.

The design document below (Overview through Open questions) is retained unchanged as a starting point for a future feature if concrete demand ever materializes. It is **not** scheduled for any release.

### Overview

After `FEATURE_072` lands, KodaX's compaction hot path reads from lineage via `getSessionMessagesFromLineage`, and post-compact attachments live natively on `KodaXSessionCompactionEntry`. The **data structure shape** and **consumer contracts** are pi-mono-compatible — but the underlying **island model** (copy-on-compact) is preserved.

`FEATURE_073` is the structural follow-up: migrate from copy-style to reference-style. Instead of every compaction creating a new `parentId: null` island with a duplicated kept tail, the lineage becomes a single linear chain (with occasional user-triggered forks); CompactionEntries become inline markers that reference the original pre-compaction entries via `firstKeptEntryId`.

### Why this is a separate feature, not part of 072

`FEATURE_072` targets the user-visible regression (monotonic context growth under AMA). Copy-style vs reference-style is invisible to users — both produce identical LLM input given the same compaction decisions. The structural change touches `archiveOldIslands`, `buildSessionTree`, and session persistence compatibility; those are user-visible paths and carry their own risk. Bundling them with 072 would have ballooned scope and delayed the growth-bug fix.

Splitting also lets 072 stabilize in production first. If copy-style turns out to have no real downside, 073 can be re-evaluated or cancelled.

### Current (copy-style) vs target (reference-style)

| Aspect | Copy-style (v0.7.20) | Reference-style (v0.7.25+) |
|---|---|---|
| `CompactionEntry.parentId` | `null` (new island root) | Last entry before compaction (inline on active path) |
| `CompactionEntry.compactionStyle` | Absent (implicit copy) | `'reference'` (explicit per-entry tag) |
| Kept tail | Copied as new entries under CompactionEntry | Original entries, re-parented onto CompactionEntry |
| Old content cleanup | `evictOldIslandMessageContent` (island-based) | `evictPreCompactionMessageContent` (compaction-boundary-based) — same placeholder, same footprint |
| `getSessionLineagePath` | Naturally stops at `parentId: null` | Uses explicit `stopAtCompaction` option on hot paths |
| `/fork` target pre-compaction | `[compacted]` placeholder after eviction | `[compacted]` placeholder after retargeted eviction — observable parity |
| `buildSessionTree` roots | One per island | One root; CompactionEntries inline; physical vs logical active path distinguished |
| `applySessionCompaction` internals | Writes new island root + duplicates kept tail via `createSessionLineage` | Writes inline CE + re-parents first kept entry; no `createSessionLineage` reconciliation |

### Goals

1. **Single-chain lineage with inline compaction markers.** Eliminate the "multiple islands per session" shape; all entries live on one primary chain with CompactionEntries inline.
2. **Reference semantics for `firstKeptEntryId`.** Field points at the original pre-compaction entry, not a new copy. Kept tail is no longer duplicated.
3. **Retarget eviction from island-based to compaction-boundary-based.** `evictOldIslandMessageContent` gets a sibling `evictPreCompactionMessageContent` that swaps message bodies on the same placeholder, following the new data shape. Memory footprint stays at parity with copy-style + eviction.
4. **Preserve observable behaviour** of `/fork`, `/rewind`, `/load-session`, session tree UI, and `/archive` across copy-style, reference-style, and mixed-shape sessions.
5. **Align with pi-mono and Claude Code** so future features (partial compaction, multi-boundary, inline-marker compactions, `cache_edits`-style) can be adopted without another round of structural surgery.

### Non-goals

- **Not changing the public shape of `KodaXSessionCompactionEntry`.** Only an optional `compactionStyle` field is added; interpretation of `parentId` and `firstKeptEntryId` changes for new entries.
- **Not changing the external API of `getSessionMessagesFromLineage`.** Function signature and return shape stay identical; only internal walk logic changes. `getSessionLineagePath` gains an optional `stopAtCompaction` parameter; default behaviour is unchanged.
- **Not changing persistence format incompatibly.** Per-entry `compactionStyle` replaces implicit `parentId === null` detection. No forced migration. Legacy sessions load through the same derivation, treated as `'copy'` by default.
- **Not supporting downgrade.** A 0.7.25+ session with reference-style entries loaded by pre-0.7.25 KodaX may have broken archival. Declared, not fixed — see Compatibility and risks §Downgrade.
- **Not introducing partial compaction, multi-boundary, or `cache_edits`.** Those become easier but are separate features.
- **Not removing `archiveOldIslands`.** It is reworked with a precise new preserve closure; the archival surface (`/archive`, slimmed lineage output) is preserved.
- **Not removing `evictOldIslandMessageContent` entirely.** Its retargeted counterpart (`evictPreCompactionMessageContent`, Design §7) handles reference-style entries; the original continues to handle legacy copy-style entries in mixed-shape sessions.
- **Not advertising a `/fork` improvement.** 073 preserves `/fork` observable behaviour. The "fork to pre-compaction entry gives full history" characteristic of naive reference-style is intentionally suppressed via scope-bounded fork (Design §8). A future feature may add opt-in `/fork --full-history`.
- **Not touching 051's visible shell.** This feature is pure internal restructure of `@kodax/agent`.

### Design

#### 1. `applySessionCompaction` — inline CompactionEntry on the active path

Today (copy-style) and 072 (copy-style, lineage-native) both create CompactionEntry with `parentId: null`, forming a new island. First-kept entry becomes a child of CompactionEntry via `createSessionLineage`-driven reconciliation over a duplicated kept tail. 073 replaces both:

```ts
// Pre-compaction: root → ... → V → W → X → Y → Z   (Z = activeEntryId)
// Compact keeps [W, X, Y, Z], cuts at V.

// After 073 (reference-style):
compactionEntry.id = <new>;
compactionEntry.parentId = V.id;                 // NOT null — links to last pre-compaction entry
compactionEntry.firstKeptEntryId = W.id;         // points at the ORIGINAL W, not a copy
compactionEntry.postCompactAttachments = [...];  // 072 field
compactionEntry.compactionStyle = 'reference';   // NEW per-entry tag (§5)

W.parentId = compactionEntry.id;  // re-parent W onto CE — CE must be inline on path

// activeEntryId = Z.id (unchanged); Z's ancestors are now Y → X → W → CE → V → ... → root
```

**Two non-obvious moves:**

1. **First-kept entry's `parentId` is re-parented onto the CompactionEntry.** Without this, CE is a sibling of W (both children of V) and `getSessionLineagePath(Z)` would never walk through it — the slicer at §2 could not find the compaction boundary. CE must be inline on the active path.
2. **`activeEntryId` stays at the original leaf Z.** It is not retargeted to CE or to a new copy. The path from Z traces Z → Y → X → W → CE → V → ... → root.

This is a mutation to an existing entry (W.parentId changes). KodaX already mutates existing entries during compaction (`evictOldIslandMessageContent` replaces message content in-place, [session-lineage.ts:652-711](../../packages/agent/src/session-lineage.ts#L652-L711)), so there is precedent; the mutation is scoped to compaction time and guarded by the reference-style write path.

#### 2. `getSessionMessagesFromLineage` — firstKeptEntryId-based slicing

> **Relies on FEATURE_072's `getContextMessagesForEntry` 1-to-1 contract (v0.7.20.md §2).** 073 assumes `getContextMessagesForEntry(entry).length ≤ 1` for every entry type. Compaction entry → `[summary]`, message → `[body]`, branch_summary → `[summary]`, archive_marker → `[]`; attachments are never emitted by `getContextMessagesForEntry` and are prepended separately by the slicer below. 073 Phase A adds a regression guard asserting this contract.

```ts
export function getSessionMessagesFromLineage(
  lineage: KodaXSessionLineage,
  targetId: string | null = lineage.activeEntryId,
): KodaXMessage[] {
  const path = getSessionLineagePath(lineage, targetId, { stopAtCompaction: true });

  // stopAtCompaction truncates the path to start at the most recent non-rewind
  // CompactionEntry if one exists. If none exists, path is full history.
  const head = path[0];
  const isCompactionHead =
    head?.type === 'compaction' && head.reason !== 'rewind';

  if (!isCompactionHead) {
    return path.flatMap(getContextMessagesForEntry).map(cloneMessage);
  }

  // Slicer: [CE's summary, ...CE's attachments, ...path-after-CE]
  // 072's 1-to-1 contract guarantees getContextMessagesForEntry(head) = [summary].
  return [
    ...getContextMessagesForEntry(head),                       // [summary]
    ...(head.postCompactAttachments ?? []).map(cloneMessage),
    ...path.slice(1).flatMap(getContextMessagesForEntry),
  ].map(cloneMessage);
}
```

**`reason !== 'rewind'` filter**: `rewindSessionLineage` creates `type: 'compaction'` entries as audit markers ([session-lineage.ts:812-823](../../packages/agent/src/session-lineage.ts#L812-L823)). Today they never appear on an active path (rewind leaves `activeEntryId = targetEntryId`; rewindEntry is a downstream sibling of target). But `setSessionLineageActiveEntry` does not reject rewind entries as navigation targets — defensive correctness requires explicit exclusion of `reason === 'rewind'` from compaction slicing, and `stopAtCompaction`'s stop condition must honour this filter.

#### 2a. `getSessionLineagePath` gains `stopAtCompaction` option

Today `getSessionLineagePath` walks from leaf via `parentId` until hitting `parentId === null`. In copy-style (072 and earlier) this naturally stops at the current island root (CompactionEntry with `parentId: null`), giving an O(current island) path. **Under reference-style, CompactionEntry.parentId is non-null**, so the walk continues to the original root — O(full history), which is a performance regression on hot paths.

Fix: parametrize the walk.

```ts
export interface LineagePathOptions {
  stopAtCompaction?: boolean;  // stop at most recent non-rewind CompactionEntry
}

export function getSessionLineagePath(
  lineage: KodaXSessionLineage,
  targetId: string | null = lineage.activeEntryId,
  options: LineagePathOptions = {},
): NavigableSessionEntry[] { ... }
```

**Hot-path callers use `stopAtCompaction: true`**:
- All six 072 compaction-path consumers (via `getSessionMessagesFromLineage`)
- Token counting / `countActiveLineageMessages`
- Iteration-end reconciliation reading from derived view

**Full-history callers explicitly opt out (omit `stopAtCompaction`)**:
- `archiveOldIslands` — needs pre-compaction entries to decide archival (§3)
- `getCommonAncestorId` / `getBranchSegment` — `/fork` and branch operations may target pre-compaction entries
- `buildSessionTree` — UI tree needs full structure (§4)

Under copy-style this option is a no-op (walk already stops at `parentId: null`). Under reference-style it is load-bearing. Phase A lands the option and switches hot-path callers; write-side changes land in Phase B.

#### 3. `archiveOldIslands` — precise preserve closure

The 140-line copy-style implementation ([session-lineage.ts:943-1085](../../packages/agent/src/session-lineage.ts#L943-L1085)) detects islands structurally via `parentId: null` roots and BFS. Reference-style has no island structure — all entries reachable from the sole `parentId: null` root. Preserving "active path" naively would preserve every pre-compaction ancestor and archive nothing.

**New preserve closure (reachability by role, not by structure):**

```
preserve =
    ∪ LLM-visible path of active leaf                              (via slicer; not raw parent chain)
    ∪ LLM-visible path of every label target                       (/load-session to a label still works)
    ∪ LLM-visible path of every persisted fork leaf                (fork branches stay navigable)
    ∪ ancestor chain of every branch_summary.fromId                (branch-summary audit trail)
    ∪ every CompactionEntry                                        (needed for reconstructable slicing)
    ∪ every non-message entry (label, branch_summary, archive_marker)  (always small; cheap to keep)
    ∪ ancestor chain of every non-message entry                    (prevents tree drift)

archive candidates = navigable entries NOT in preserve closure
```

"LLM-visible path" = what `getSessionMessagesFromLineage` would walk for a given target (i.e., slicer-truncated). This is the key departure from the copy-style implementation's "active island" BFS.

**Phase C regression gate**: on a set of v0.7.19 / 072 fixture sessions (copy-style), run both implementations and assert **byte-identical** archived entries + preserved closure. Only then does the new implementation ship as the live path.

#### 4. `buildSessionTree` — physical path vs logical LLM view

Today `buildSessionTree` marks `active = activePathIds.has(entry.id)` where `activePathIds = getSessionLineagePath(lineage).map(e => e.id)`. In copy-style this is the current island + forks, which matches user expectation ("entries on the active conversation").

Under reference-style, the same call returns the full root-to-leaf path — **including every pre-compaction ancestor that has been sliced out of the LLM view**. Marking all of them as "active" breaks the UI semantic.

**Resolution**: distinguish two reachability sets.

```ts
type ActivePathView = {
  physical: Set<string>;  // full parent-chain reachability (for structural correctness)
  logical: Set<string>;   // LLM-visible slice (for "active conversation" highlighting)
};
```

`buildSessionTree` emits both. UI chrome that marks "current conversation" uses `logical`; anything that needs structural reachability (parent-check, connectivity, tree rendering itself) uses `physical`. Under copy-style both sets are identical, preserving today's behaviour.

#### 5. Session persistence — `compactionStyle` field and explicit downgrade declaration

`KodaXSessionCompactionEntry` gains:

```ts
interface KodaXSessionCompactionEntry {
  // ... existing fields ...
  compactionStyle?: 'copy' | 'reference';  // absent = 'copy' (legacy); 'reference' = 073+ writes
}
```

Explicit per-entry tag; replaces implicit detection via `parentId === null`. Mixed-shape sessions (some entries copy-style, some reference-style) are natural and common during rollout.

**Compat read**: `getSessionMessagesFromLineage`, `archiveOldIslands`, `buildSessionTree`, and the eviction dispatcher (§7) all branch on `compactionStyle ?? 'copy'` per entry. Legacy entries without the field are treated as copy-style. Reference-style entries go through the new slicer / preserve closure / retargeted eviction.

**No `lineage.version` bump**. Per-entry tagging is sufficient; a session-wide version bump would force migration or hard-reject, and neither is needed.

**Downgrade (declared non-support)**: a session written under 0.7.25+ with `compactionStyle: 'reference'` entries, loaded by a pre-0.7.25 KodaX, will be misinterpreted — `archiveOldIslands` may not identify any island (no `parentId: null` CompactionEntries among new ones), `evictOldIslandMessageContent` may evict nothing or the wrong entries. **This regression is declared, not fixed.** Users downgrading after a 0.7.25+ session are expected to start a fresh session or accept broken archival on affected sessions; release notes call this out.

#### 6. Duplicate kept-tail cleanup (deferred)

After migration, pre-0.7.25 sessions still contain the duplicated kept-tail entries under each old CompactionEntry. They remain valid, reachable via `/fork`, and sliced out of LLM view by compat read. Physical cleanup (rewriting old sessions to drop the duplicates) is a separate garbage-collection feature if disk footprint becomes an issue — not scoped here.

#### 7. Eviction retargeting — `evictPreCompactionMessageContent`

`evictOldIslandMessageContent` exists because copy-style inflates memory with duplicated kept tails; eviction compresses old islands' message bodies to `[compacted]` placeholder ([session-lineage.ts:652-711](../../packages/agent/src/session-lineage.ts#L652-L711)). Under reference-style there is no duplication — but pre-compaction entries retain their original (potentially large) message bodies **indefinitely**, since the single-chain lineage never "closes" an old island.

Naive reference-style uses *more* memory than copy-style + eviction for long sessions: 5 rounds of compaction on a 200-entry session → reference-style keeps 200 full-sized messages; copy-style + eviction keeps ~20 full + ~180 placeholders (typically ~5× less memory).

**Solution: eviction retargets from island-based to compaction-boundary-based.**

```ts
// Today
evictOldIslandMessageContent(lineage)
// – walks entries outside the active island (parentId: null subtree)
// – replaces each message's content with [compacted] placeholder

// 073
evictPreCompactionMessageContent(lineage)
// – for each reference-style CompactionEntry on the active path, walk its
//   ancestor chain (entries before firstKeptEntryId)
// – replace each ancestor message's content with [compacted] placeholder,
//   same placeholder, same memory footprint as copy-style post-eviction
// – labels, branch_summary, and forks protect their ancestors (preserve closure §3)
```

Old islands in mixed-shape sessions still route through the original `evictOldIslandMessageContent`. New reference-style entries route through `evictPreCompactionMessageContent`. Same call site, per-entry dispatch on `compactionStyle`.

**Net effect**: reference-style memory footprint matches copy-style + eviction. `/fork` to a pre-compaction entry lands on a `[compacted]` placeholder — same as today's behaviour. The "fork to pre-compaction gets full history" characteristic of naive reference-style is **intentionally not a feature** (see §8).

#### 8. `forkSessionLineage` — default scope limited to the current LLM view

Naive reference-style `/fork` would clone the entire root-to-target path via `structuredClone`. On long sessions (5 rounds, 2000+ entries, large tool_results), a single `/fork` could consume 20MB+.

**073's scope for fork**:

- **Default behaviour (preserved, not improved)**: `/fork` without further flags behaves the same as copy-style `/fork` — it clones from the **current LLM-visible view** outward, i.e. entries from the most recent CompactionEntry (or from root if no compaction) to the target. The implementation uses `getSessionLineagePath(lineage, targetId, { stopAtCompaction: true })` as the clone basis.
- **`/fork` to a pre-compaction entry** (target sliced out of current LLM view): **explicitly out of scope for 073**. Copy-style always handled this by landing on an evicted `[compacted]` placeholder; reference-style + retargeted eviction (§7) preserves that behaviour.
- **Future `/fork --full-history` opt-in**: a later feature can add an explicit flag that accepts the memory cost; this feature does not.

**Consequence for framing**: the early draft's "fork to pre-compaction entry gives full history" pitch is **withdrawn**. Fork semantics are "preserved, not improved" — consistent with Non-goals.

#### 9. `applySessionCompaction` does not reconcile via `createSessionLineage`

072's `applySessionCompaction` (current and 072-design) internally calls `createSessionLineage(keptMessages, base)` to materialize the kept tail as message entries under the new CompactionEntry. Under copy-style this works: CompactionEntry is at `parentId: null`, `createSessionLineage` starts reconciliation at `parentId: null`, finds the summary message match (via 072's 1-to-1 contract), then walks kept messages.

**Under reference-style this path is broken**: the new CompactionEntry has `parentId = V.id` (not null), so `createSessionLineage`'s children walk from `parentId: null` never encounters it. The summary message becomes an orphan message entry at root level, and the kept tail is either duplicated or silently stranded.

**Fix**: reference-style `applySessionCompaction` does **not** call `createSessionLineage` for post-compaction reconciliation. Instead:

1. Compute the kept-tail entries directly (they are already in lineage — find them by id from the pre-compaction active path).
2. Re-parent the first kept entry onto the new CompactionEntry (§1).
3. Done. The kept-tail entries keep their existing identity, `parentId` linkage (except the re-parented first), and message content. No duplication happens.

```ts
// Reference-style applySessionCompaction (sketch)
const base = cloneLineage(lineage);
const prePath = getSessionLineagePath(base, base.activeEntryId);   // full; no stopAt
const cutIdx = findCutIndex(prePath, keptMessages);                // map messages → entries by identity
const lastPreCompaction = prePath[cutIdx - 1];                     // V
const firstKept = prePath[cutIdx];                                 // W

const compactionEntry: KodaXSessionCompactionEntry = {
  type: 'compaction',
  id: generateEntryId(),
  parentId: lastPreCompaction.id,
  firstKeptEntryId: firstKept.id,
  compactionStyle: 'reference',
  postCompactAttachments: attachments,
  summary: anchor.summary,
  // ...rest of anchor fields
};

base.entries.push(compactionEntry);
base.entries = base.entries.map(e =>
  e.id === firstKept.id ? { ...e, parentId: compactionEntry.id } : e,
);
// activeEntryId unchanged — still points at the original leaf Z.

return evictPreCompactionMessageContent(base);  // §7
```

Iteration-end lineage reconciliation (REPL `reconcileContextLineage`, `onCompactedMessages`, `recordCompletedAgentRound`) continues to call `createSessionLineage(messages, lineage)` but only over newly-appended user/assistant turns; the kept tail and CompactionEntry are already in lineage so reconciliation is append-only on new turns — it never tries to re-materialize the kept tail.

### Acceptance criteria

#### Structural invariants

- [ ] After reference-style compaction: `CompactionEntry.parentId !== null` (points at pre-compaction last entry) **AND** `CompactionEntry.compactionStyle === 'reference'`.
- [ ] **Path connectivity**: `getSessionLineagePath(lineage, activeEntryId)` contains the most-recent reference-style CompactionEntry on its path. Consequence: §2's slicer always finds CE.
- [ ] **Reference semantics for `firstKeptEntryId`**: the entry with that id exists in the lineage and was created before the CompactionEntry (timestamp check). No new copy entry is created for kept messages.
- [ ] **No kept-tail duplication**: after reference-style compaction, `lineage.entries.filter(e => e.type === 'message' && matches kept tail)` contains no duplicates.

#### Correctness

- [ ] `getSessionMessagesFromLineage` produces identical LLM-visible message content for the same logical history, regardless of whether compactions were copy-style, reference-style, or mixed-shape.
- [ ] **Slicer correctness**: `getSessionLineagePath(_, _, { stopAtCompaction: true })` in reference-style returns the path from most-recent CompactionEntry to leaf; in copy-style behaves identically to today's natural `parentId: null` stop.
- [ ] **072 contract guard**: `getContextMessagesForEntry(entry).length ≤ 1` for every entry type (compaction / message / branch_summary / archive_marker). Regression test in 073 Phase A.

#### Memory & performance

- [ ] **Memory parity**: total bytes of `context.lineage` after 5 rounds of reference-style compaction on a 200-entry session are within 10% of the same history processed by copy-style + eviction.
- [ ] **Hot-path walk cost**: `getSessionMessagesFromLineage` latency on a long reference-style session (after 5 compactions) is within 10% of the same call on a same-sized copy-style session. `stopAtCompaction: true` is the load-bearing assumption.
- [ ] **Fork memory parity**: `/fork` on a reference-style session clones no more memory than the copy-style equivalent (±10%). Default fork scope is `stopAtCompaction`-bounded.

#### Session / tree / archive

- [ ] Sessions persisted under v0.7.20 load without error; derivation handles legacy entries (absent `compactionStyle`) as copy-style.
- [ ] `archiveOldIslands` produces byte-identical archival output to the old implementation on v0.7.20 fixture sessions.
- [ ] `buildSessionTree` emits a UI tree with a single `parentId: null` root under reference-style; physical vs logical active-path distinction is available to UI callers.
- [ ] `/fork`, `/rewind`, `/load-session` observational parity on reference-style and mixed-shape sessions against copy-style regression fixtures.

#### Compatibility

- [ ] **Rewind entry-count assertion**: after rewind to target T, `getSessionMessagesFromLineage(lineage, T)` produces the same output as if the session had always ended at T. No phantom post-target entries in the LLM view.
- [ ] **Downgrade regression declared** (not fixed) in release notes: 0.7.25+ reference-style sessions may not archive correctly on pre-0.7.25 KodaX.

### Compatibility and risks

#### Session persistence

Mixed-shape sessions (some entries copy-style, some reference-style) must remain valid forever. Per-entry `compactionStyle` tag makes this unambiguous; legacy entries without the field default to `'copy'`. An eager migration is **not** scoped — it would introduce write side-effects during load.

#### Downgrade (declared non-support)

A session written under 0.7.25+ with reference-style entries, loaded by pre-0.7.25 KodaX, may have broken archival: the old `archiveOldIslands` does not recognize reference-style entries as island boundaries, and may either archive nothing or archive entries that should be preserved. This is declared, not fixed — users downgrading after creating a reference-style session are expected to either start fresh or accept degraded archival. Release notes call this out explicitly.

#### `/fork` scope (intentional)

073 deliberately keeps `/fork` within the current LLM-visible view (`stopAtCompaction`-bounded). An earlier draft pitched "fork to pre-compaction entry gives full history" as a feature improvement; that is **withdrawn**. The memory cost of full-history fork on long reference-style sessions (potentially 20MB+ per fork) is not worth the marginal UX gain. A future feature may add opt-in `/fork --full-history`.

#### `buildSessionTree` UI rendering

Session tree UI under reference-style renders as a single tree with inline CompactionEntry markers, whereas copy-style rendered multiple islands. The change is a cosmetic improvement (closer to pi-mono / Claude Code) but user-visible — visual snapshot regression in Phase C covers this.

#### `archiveOldIslands` policy rewrite

Largest risk surface. Mitigation: Phase C runs both implementations in parallel on fixture sessions and asserts **byte-identical** output on copy-style inputs before the new path ships. Reference-style inputs are validated independently.

#### Memory footprint parity

`evictPreCompactionMessageContent` (§7) is load-bearing. Without it, long reference-style sessions would retain 5-10× more memory than copy-style + eviction. The Phase B memory-parity gate (±10%) is the explicit checkpoint.

#### Reconciliation change (§9)

073 removes `createSessionLineage` from the post-compaction path. This is a behavioural change from 072 (which still uses `createSessionLineage` in `applySessionCompaction`). Iteration-end reconciliation (REPL `reconcileContextLineage`, `onCompactedMessages`, `recordCompletedAgentRound`) continues to call `createSessionLineage` but only over newly-appended user/assistant turns, which is append-only and compaction-blind. Phase B regression test: after reference-style compaction, reconciling a subsequent user/assistant turn produces a single new message entry pair, not a duplicated kept tail.

### Relationship to prior work

- `v0.7.19` surgical fixes (P1-P6) — structurally superseded by `072`; `073` continues the refactor into copy-style retirement.
- `v0.7.20 FEATURE_072` — **direct prerequisite.** Without 072, there are still flat-array compaction consumers that depend on copy-style island semantics; 073 cannot land on the current structure.
- `v0.7.25 FEATURE_051` — untouched. `073` only affects internal `@kodax/agent` lineage code; the frozen visible shell from `051` is not reopened.
- pi-mono `buildSessionContext` — reference target.
- Claude Code `getMessagesAfterCompactBoundary` — inline-marker variant of the same pattern.

### Open questions

1. **Eager migration vs indefinite compat?** Should a one-shot `/migrate-sessions` command rewrite old copy-style sessions into reference-style on user request? Leaning lazy — mixed-shape support is cheap; forced migration has no clear payoff.
2. **`stopAtCompaction` overhead on short sessions** (no compaction on path): Phase A benchmark should confirm the option is zero-cost when no compaction entries are present. Expected: a single extra type check per step, negligible.
3. **When can legacy `evictOldIslandMessageContent` be fully removed?** Needs a signal that legacy copy-style sessions have rolled over — likely a version-bump criterion tied to a deprecation window. Out of scope for 073.
4. **Should a future feature add `/fork --full-history`?** Left open. Not needed for 073; pure UX decision downstream.
5. **`compactionStyle` default when mixed-shape sessions gain a third style in future** (e.g. partial compaction): the field leaves room for that — `'copy' | 'reference' | ...`. No decision needed now, but the field is intentionally an open enum.

## FEATURE_076: Managed Task Round Boundary — User Conversation Preservation

**Status**: InProgress  
**Started**: 2026-04-20  
**Priority**: High  
**Category**: Internal / Behavior  
**Builds on**: v0.7.19 Scout ceiling clamp fix (commit `3efdb7b`), `FEATURE_061` (Scout-First AMA Architecture)  
**Reference implementations**: Claude Code turn-based transcript persistence, pi-mono `buildSessionContext`

### Scope decisions (2026-04-20 philosophy review)

Four implementation decisions confirmed before TDD start:

**Q1 — Placeholder detection (reuse existing structured field, zero blast radius)**:
The original design proposed string-matching `"Task is running"` / `"Scout is completing..."` via `isStatusPlaceholderText`. Review discovered that the existing `result.managedTask?.verdict?.status` field (type `KodaXTaskStatus = 'planned' | 'running' | 'blocked' | 'failed' | 'completed'`) is **already** the authoritative signal. All 3 placeholder summary sites in [task-engine.ts:2163, 2244, 2343](../../packages/coding/src/task-engine.ts) simultaneously set `verdict.status = 'running'`, and `lastText` is derived from `verdict.summary` at [task-engine.ts:5326](../../packages/coding/src/task-engine.ts#L5326). **Decision**: replace string matching with `verdict.status` check — no new field on `KodaXResult`, no construction-site changes.

Per-status reshape behavior:
- `completed` → reshape as `{user, assistant: finalText}`
- `blocked` → reshape (blocked reason IS the user-facing answer — e.g. "需要 OAuth 授权")
- `failed` → reshape (error message IS the user-facing answer)
- `running` → fallback (placeholder, not a real answer)
- `planned` → fallback (should not reach runManagedTask exit; defensive)

**Q2 — Token snapshot full recompute**:
Original design used `rebaseContextTokenSnapshot(cleanMessages, oldSnapshot)` — partial reuse of old snapshot. Dropped in favor of full `recomputeContextTokenSnapshot(cleanMessages)`. Simpler; eliminates the drift class that A/B token-meter bugs were already symptoms of.

**Q3 — Fork mode pulled into 076**:
Original design marked InkREPL fork-mode user-message handling ([InkREPL.tsx:5744-5757](../../packages/repl/src/ui/InkREPL.tsx#L5744-L5757)) as orthogonal non-goal. Reversed: 5/6 paths clean + 1 path dirty is its own consistency bug. Fork path also goes through `reshapeToUserConversation`, with fork-specific user-message handling.

**Q4 — Historical session load-time normalization**:
Original design said "Not migrating historical sessions." Reversed: sessions saved under v0.7.24 and earlier carry execution-trace-shaped `messages`; on `/load-session` restore the first post-upgrade round still sees dirty initialMessages, contradicting acceptance criterion #1. Added: detect role-prompt-shaped leading messages on session load and run a one-time normalization pass.

### Overview

After a managed task round finishes, `context.messages = result.messages`. Today `result.messages` is whatever internal message array the last-returning path produced:

- **SA / H0**: Scout's own `runKodaX` messages. Starts with `initialMessages` and ends with Scout's role-prompt-wrapped user question plus tool calls and final text. The original user prompt is buried inside the Scout role prompt; subsequent rounds see a heavily wrapped "user" turn.
- **H1 / H2**: the terminal worker (usually Evaluator) ran in an isolated session. Its `result.messages` looks like `[{ role: 'user', content: "You are the Evaluator role..." }, { role: 'assistant', content: verdict block }]`. Nothing about the actual user question or the conversation history survives.

Either way, the saved conversation is **the agent's execution trace, not the user's dialog**. `FEATURE_076` reshapes the `runManagedTask` exit so `context.messages` always comes back as a clean user-facing dialog:

```
[
  ...previousUserConversation,                 // preserved from before this round
  { role: 'user', content: plainUserPrompt },  // this round's question (not role-wrapped)
  { role: 'assistant', content: finalAnswer }, // this round's user-facing answer
]
```

### Why this is a separate feature

The v0.7.19 Scout ceiling clamp fix (`3efdb7b`, `fa4708f`) unblocked the immediate symptom — H1/H2 no longer returns a synthetic 1-message result when Scout escalates on read-only tasks. But the underlying shape of `result.messages` still carries worker execution traces, not the user's dialog. The fix was scoped to "make H1/H2 runnable"; this feature is scoped to "make `context.messages` semantically correct across rounds".

These are separable because:

- The clamp fix changes control flow (which branch runs, which workers are created). It is load-bearing and needed immediate delivery.
- The round-boundary reshape changes the **shape of returned messages**. It touches a larger downstream surface (REPL consumers, session persistence, artifact ledger extraction, `reconcileContextLineage`, token accounting) and benefits from a design-first pass.

Shipping them separately also lets the clamp fix stabilize in production before the reshape lands.

### Current pain

#### A. Multi-turn conversation incoherence

**Scenario**: user asks "审查内存泄漏", H1 pipeline runs, Evaluator produces the final answer. `context.messages` is overwritten with Evaluator's isolated session messages. User asks "那 `session-lineage.ts` 里具体哪里？". The new Scout starts round 2 with `initialMessages` that read:

```
user: "You are the Evaluator role. Review the Generator's output..."
assistant: "<verdict block>"
user: "那 session-lineage.ts 里具体哪里？"
```

Scout cannot infer what was previously asked or answered. It starts investigation from scratch, often asking the user to re-state the task.

#### B. Token accounting drift

After H1/H2, `context.contextTokenSnapshot.currentTokens` reflects Evaluator's internal session (typically small). Users see the token meter drop sharply and interpret it as "context was cleared". Session bookkeeping becomes decorrelated from the visible dialog.

#### C. H0 path also carries the Scout role prompt as a "user" message

Even on the H0 success path, `result.messages` ends with `[..., { role: 'user', content: scout_role_prompt_wrapping_Q }, ..., { role: 'assistant', content: summary }]`. The wrapping prompt re-enters the next round as part of the conversation, adding several K tokens of boilerplate per round and distracting the LLM from the actual dialog.

#### D. Session persistence stores execution traces

`context.messages` is persisted via `storage.save({ messages })` at the end of every round. Loading a saved session reproduces the execution-trace shape, so `/load-session` followed by a follow-up question inherits the same incoherence.

### Design philosophy

The core principle: **`context.messages` represents the user's conversation, not the agent's execution trace**.

Separation of concerns:

- **User dialog** → `context.messages` → clean, readable, token-efficient
- **Execution trace** (worker role prompts, tool calls, verdict blocks, handoff payloads) → persisted to `.agent/managed-tasks/<id>/` artifacts — inspectable for debugging, not polluting conversation context
- **UI streaming** (live tool call animations, thinking deltas, progress events) → REPL's `persistedUiHistoryRef` — already independent of `messages`

### Goals

1. `context.messages` after any `runManagedTask` round is a clean dialog of `{user, assistant}` pairs, regardless of SA / H0 / H1 / H2 / resume path.
2. Preserve all pre-round user conversation turns exactly — do not rewrite history, only append this round's `{user, assistant}`.
3. The assistant turn carries the user-facing final answer (derived from `result.lastText` or equivalent), not worker execution details.
4. Artifact references surfaced during internal execution remain accessible through `result.artifactLedger`, so REPL consumers relying on `extractArtifactLedger` do not regress.
5. Provide observable behaviour uniform across all managed task paths: H0 no longer carries Scout role prompt wrapping; H1/H2 no longer carries Evaluator session; resume and SA paths follow the same contract.

### Non-goals

- **Not changing REPL UI streaming.** Tool call animations, thinking deltas, and progress events stay on the existing independent channel.
- **Not altering on-disk execution artifacts.** `.agent/managed-tasks/<id>/` continues to persist full execution traces for debugging and audit.
- **Not redesigning `KodaXResult`.** `messages` stays on `KodaXResult`. `artifactLedger` stays on `KodaXResult`. No new top-level fields (reuses existing `managedTask.verdict.status` per Q1).
- **Not changing intra-run worker handoff.** `FEATURE_046` already covers worker-to-worker handoff contracts. `076` only reshapes the round exit returned to callers.
- **Not introducing multi-surface awareness.** Same reshape applies to `cli` / `repl` / `project` / `acp` surfaces.

> **Scope additions per 2026-04-20 review** (originally non-goals, pulled in):
> - Fork-mode user-message handling (Q3)
> - Load-time normalization for pre-v0.7.25 sessions (Q4)

### Design direction

#### 1. Single reshape at the `runManagedTask` outer boundary

Split `runManagedTask` into an outer wrapper and an inner `executeRunManagedTask`:

```ts
export async function runManagedTask(
  options: KodaXOptions,
  prompt: string,
): Promise<KodaXResult> {
  const result = await executeRunManagedTask(options, prompt);
  return reshapeToUserConversation(result, options, prompt);
}
```

The inner function keeps the existing SA / H0 / H1 / H2 / resume branches untouched. The reshape is the single seam.

#### 2. `reshapeToUserConversation` shape (finalized per Q1-Q4)

```ts
function reshapeToUserConversation(
  result: KodaXResult,
  options: KodaXOptions,
  prompt: string,
): KodaXResult {
  if (!result.messages) return result;

  const originalInitialMessages = options.session?.initialMessages ?? [];
  const inputArtifacts = options.context?.inputArtifacts;
  const finalText = extractFinalAssistantText(result);

  // Debug-preserve (Q1): only fall back when verdict.status indicates the
  // task has NOT converged on a user-facing answer. Uses the existing
  // structured `KodaXTaskStatus` field — no new result-level field, no
  // string matching on placeholder summaries.
  //   completed / blocked / failed  → reshape (all three carry a real
  //                                    user-facing answer)
  //   running / planned             → fallback (placeholder or invalid
  //                                    boundary state)
  // Additional guard: interrupted with empty finalText still fallback.
  if (
    isUnconvergedVerdict(result.managedTask?.verdict?.status)
    || (result.interrupted && !finalText)
  ) {
    return result;
  }

  // Pre-extract artifact ledger BEFORE reshape: tool_result blocks disappear
  // from messages after reshape. Both InkREPL (reads result.artifactLedger) and
  // CLI REPL (reads result.messages via extractArtifactLedger — must also be
  // updated to prefer result.artifactLedger, see §7) benefit.
  const preservedArtifactLedger =
    result.artifactLedger ?? extractArtifactLedger(result.messages);

  const cleanMessages = buildUserFacingMessages(
    originalInitialMessages,
    prompt,
    finalText,
    inputArtifacts,
  );

  // Q2: Full recompute (not rebase). Simpler, eliminates the drift class
  // that token-meter bugs were already symptoms of.
  const recomputedSnapshot = result.contextTokenSnapshot
    ? recomputeContextTokenSnapshot(cleanMessages, result.contextTokenSnapshot)
    : undefined;

  return {
    ...result,
    messages: cleanMessages,
    artifactLedger: preservedArtifactLedger,
    contextTokenSnapshot: recomputedSnapshot,
  };
}

// Q1: Check the existing structured verdict status instead of string-matching
// the placeholder summary.
function isUnconvergedVerdict(status?: KodaXTaskStatus): boolean {
  return status === 'running' || status === 'planned';
}
```

`isUnconvergedVerdict` is a trivial helper over the existing `KodaXTaskStatus` enum — no new field on `KodaXResult`, no construction-site changes. `recomputeContextTokenSnapshot` is a new helper that tokenizes `cleanMessages` and produces a fresh snapshot (preserving non-token fields like `maxTokens` from the old snapshot).

#### 3. `buildUserFacingMessages` with REPL dedup

```ts
function buildUserFacingMessages(
  initial: readonly KodaXMessage[],
  prompt: string,
  assistantText: string,
  inputArtifacts?: readonly KodaXInputArtifact[],
): KodaXMessage[] {
  const lastMsg = initial[initial.length - 1];
  const alreadyHasPrompt =
    lastMsg?.role === 'user'
    && extractComparableUserMessageText(lastMsg) === prompt;

  const userMsg: KodaXMessage = {
    role: 'user',
    content: buildPromptMessageContent(prompt, inputArtifacts),
  };
  const assistantMsg: KodaXMessage = { role: 'assistant', content: assistantText };

  return alreadyHasPrompt ? [...initial, assistantMsg] : [...initial, userMsg, assistantMsg];
}
```

Dedup handles the CLI REPL path where the user message is pre-pushed to `context.messages` before `runManagedTask` is called, so we do not duplicate the turn.

#### 4. `extractFinalAssistantText` priority

```
1. result.lastText (all paths fill this with the user-facing answer)
2. extractMessageText(result) (reads last assistant turn from messages)
3. empty string (and a fallback assistant placeholder upstream)
```

#### 5. Pre-extract `artifactLedger` before reshape

`extractArtifactLedger` walks `messages` looking for tool_result blocks. After reshape, tool_result blocks are no longer in `messages`. Run the extraction **before** reshaping and hang the result on `result.artifactLedger`, which REPL already reads.

#### 6. Behaviour per path after reshape

| Path | Before (`result.messages` tail) | After |
|---|---|---|
| SA | Direct agent's messages including tool calls | `[...initial, {user: prompt}, {assistant: finalText}]` |
| H0 (Scout direct) | Scout's messages; last user turn is role-prompt-wrapped | Same clean dialog shape |
| H1 | Evaluator's isolated session messages | Same clean dialog shape |
| H2 | Evaluator's isolated session messages | Same clean dialog shape |
| Resume | Whatever the resumed worker produced | Same clean dialog shape (or debug-preserve if `finalText` is a status placeholder) |
| Interrupted (no text) | Execution trace at interruption point | Preserved (debug-friendly exception) |
| Interrupted (placeholder text) | Execution trace + placeholder verdict | Preserved — placeholder is not user-visible answer |

#### 7. CLI REPL artifact consumer update

`reshape` pre-extracts `result.artifactLedger`, but the CLI REPL (non-Ink) consumer still reads `extractArtifactLedger(result.messages)` at two sites:

```ts
// packages/repl/src/interactive/repl.ts:1161 — /esc edit path
// packages/repl/src/interactive/repl.ts:1272 — main agent round path
context.artifactLedger = mergeArtifactLedger(
  context.artifactLedger ?? [],
  extractArtifactLedger(result.messages),   // ← empty after reshape
);
```

These must switch to read `result.artifactLedger` with a messages fallback:

```ts
context.artifactLedger = mergeArtifactLedger(
  context.artifactLedger ?? [],
  result.artifactLedger ?? extractArtifactLedger(result.messages),
);
```

InkREPL is already safe — it updates `context.artifactLedger` via the `onCompactedMessages` callback and reads directly from `result.artifactLedger` when it flows through `recordCompletedAgentRound`. The fix is CLI-REPL-only.

#### 8. `buildFallbackManagedResult` contract note

`buildFallbackManagedResult` in `task-engine.ts` calls `replaceLastAssistantMessage(workerResult.messages, finalText)` as part of constructing the return value. After 076, the outer reshape overwrites `messages` entirely — the `replaceLastAssistantMessage` work is effectively wasted. This is not a bug: the function's meaningful output is `lastText`, which the reshape consumes. Add a documentation comment at the function header so future maintainers do not rely on the `messages` shape that `buildFallbackManagedResult` produces.

### Edge cases and decisions

| Case | Handling |
|---|---|
| First round in a session (`initialMessages = []`) | Result: `[{user: prompt}, {assistant: answer}]` |
| InkREPL path (no pre-push) | Dedup check is false → push both user and assistant |
| CLI REPL path (pre-push) | Dedup check is true → push assistant only |
| Multi-modal user prompt (text + image blocks) | `extractComparableUserMessageText` filters text blocks; dedup compares text portions. Phase A unit test covers this |
| Interrupted with partial text | Reshape normally; partial text becomes the assistant content; `result.interrupted` flag preserved |
| Interrupted with no text | Keep original `result.messages` (debug preserve) |
| Unconverged verdict (`running` / `planned`) | Keep original `result.messages` — Q1 per-status policy; placeholder summary is not a user-facing answer |
| `blocked` verdict | Reshape — blocked reason IS a valid user-facing answer (e.g. "需要 OAuth 授权才能继续") |
| `failed` verdict | Reshape — error message IS a valid user-facing answer |
| Empty `lastText` on `completed` | Synthesize fallback assistant text (`"Task completed without a textual summary."`); never produce an empty assistant turn |
| `result.messages` is undefined | Pass through without reshape |
| SA fast-path (agentMode=sa) | Also wrapped — keeps the contract uniform |

| Fork mode in InkREPL (`prepared.mode === "fork"`) **(Q3: pulled into scope)** | Reshape runs at `runManagedTask` exit as usual. The fork-specific REPL handling at `InkREPL.tsx:5744-5757` is updated to rely on reshape's clean `result.messages` — the user's fork prompt now lands in `context.messages` consistently across all 6 paths (SA/H0/H1/H2/resume/fork) |

### Risks and mitigations

#### Risk 1: Downstream consumers depend on execution-trace shape

`extractArtifactLedger`, `reconcileContextLineage`, `appendLastAssistantToHistory`, `extractTitle` all walk `context.messages`.

- `extractArtifactLedger` → **CLI REPL calls this directly at `repl.ts:1161` and `:1272`**, bypassing `result.artifactLedger`. After reshape, `messages` has no tool_result blocks → returns empty. **Mitigation**: update both call sites to `result.artifactLedger ?? extractArtifactLedger(result.messages)` (see §7 and Phase C). **InkREPL is already safe** — it updates artifact ledger via `onCompactedMessages` + reads from `result.artifactLedger` in `recordCompletedAgentRound`.
- `reconcileContextLineage` → works on any valid `{user, assistant}` sequence; the clean dialog shape is strictly simpler. No change required.
- `appendLastAssistantToHistory` → reads the last assistant turn; clean dialog contains exactly that. Behaviour improves (no verdict-block noise).
- `extractTitle` → reads the first user turn; now reads the real user question instead of the Scout role prompt. Titles improve.

#### Risk 2: Provider API calls lose mid-turn tool results across rounds

Today's H0 path carries Scout's tool_result blocks into the next round, giving the next LLM call direct access to grep outputs, file contents, etc. After reshape, the next round only sees the finalText summary.

- Mitigation: accepted trade-off. The finalText summarizes what was learned. If the next round needs the same tool data, Scout re-runs the relevant queries — expected behaviour for a fresh round. This aligns with how chat-based LLMs have always worked.
- Side benefit: dramatically smaller context on round N+1, faster and cheaper API calls.

#### Risk 3: Protocol blocks (`kodax-task-contract`, `kodax-task-verdict`, `kodax-task-handoff`) are no longer in `messages`

These blocks are internal orchestration contracts between Scout / Planner / Generator / Evaluator within a round. They are never consumed across rounds. `FEATURE_046` (completed) already treats them as round-local.

- Mitigation: none needed. Confirmed they are round-local.

#### Risk 4: Historical sessions have execution-trace messages

Users with saved sessions from pre-v0.7.25 versions will have execution-trace-shaped `context.messages` on reload.

- **Q4 mitigation (in scope)**: Session `load` path runs a one-time `normalizeLoadedSessionMessages` pass that detects role-prompt-shaped leading messages (starts with `"You are the Scout role..."` / `"You are the Evaluator role..."` heuristic plus verdict block suffix) and collapses them back to a plain `{user, assistant}` shape by preserving the original user prompt (recovered from Scout role-prompt payload section, if present) and the final assistant text. If the session cannot be disambiguated safely, log a one-line warning and load as-is (the next round's reshape will converge).

#### Risk 5: `/fork`, `/rewind`, `/tree` semantics

These commands walk the lineage. Lineage entries are driven by `reconcileContextLineage(result.messages)`, which constructs lineage entries from `{user, assistant}` turns.

- Mitigation: the clean dialog shape has clearer lineage (one entry per user turn, one per assistant turn), matching user intuition. Snapshot tests validate fork/rewind targets remain stable.

## FEATURE_058: Transcript Native Scrollback Dump

**Status**: InProgress  
**Started**: 2026-04-20  
**Priority**: Medium  
**Category**: Enhancement  
**Builds on**: FEATURE_051 (fullscreen TUI substrate — `AlternateScreen`, `FullscreenLayout`, `ScrollBox`, `renderer-runtime` all landed)

> **Version move note (2026-04-20)**: Originally planned for v0.8.0. Moved up to v0.7.25 after philosophy review concluded (a) 051 substrate prerequisites are all code-complete; (b) 058 validates 051's dump path contract — doing them adjacent is better than "build substrate, wait 4 versions for first consumer"; (c) 058 scope is small, risk is low; (d) v0.8.0 is already over-packed with 9 features. Core v0.7.25 theme ("transcript UX maturity + round boundary cleanup") actually gets more coherent with 058 in scope.

### Overview

`FEATURE_058` adds a Claude-style transcript escape hatch that dumps the current transcript view back into the terminal's native scrollback.

This is explicitly a **transcript reader convenience**, not a second transcript product surface. The goal is to let users temporarily leave renderer-owned viewport semantics when they want native terminal review, copy, or paging behavior.

### Goals

1. Add a transcript-only shortcut to dump the current transcript into native terminal scrollback.
2. Keep the primary transcript experience renderer-owned and fullscreen-first on capable hosts.
3. Avoid replaying stale managed frames back into terminal history during the dump flow.
4. Preserve KodaX's existing banner/status branding while borrowing Claude's native-scrollback escape hatch.

### Scope

- transcript-only dump shortcut (keybinding to be confirmed during implementation; candidate: `Ctrl+S` for "scrollback" or `Shift+PgUp` in transcript mode)
- native scrollback render path for dumped transcript output (plain-text rendering of current transcript content, not live UI frames)
- host-safe handoff between fullscreen transcript mode and terminal scrollback (exit alt-screen → write text → re-enter alt-screen on user command)
- transcript footer/help updates for the dump shortcut

### Non-goals

- do not add transcript `$VISUAL/$EDITOR` launching in this feature
- do not replace the fullscreen transcript browser with main-screen as the default path
- do not revive transcript item-palette semantics as the primary interaction model
- do not add Claude's transcript `v` editor shortcut

### Design direction

- **Alternate-screen handoff**: use the existing `AlternateScreen.tsx` exit sequence — temporarily leave fullscreen, emit a serialized transcript dump on stdout, optionally re-enter fullscreen on any key. No new low-level escape sequences.
- **Content serialization**: walk the REPL's `persistedUiHistoryRef` or equivalent message store and emit plain-text lines (role tag + content + tool-call summary). Do **not** replay ANSI frames — those contain cursor positioning and progress animations that leak into scrollback as visual garbage. This is the "no stale frame replay" non-goal.
- **Re-enter strategy**: on `AlternateScreen` re-entry, the renderer repaints from current React state — no need to "restore" the pre-dump view; whatever is current is correct.
- **Keybinding**: scoped to transcript mode (not default prompt surface). Rationale: default fullscreen REPL already supports native mouse scroll via 057's substrate work — no dump needed there.

### Acceptance criteria

- [ ] Transcript mode exposes a dedicated dump-to-scrollback shortcut; footer/help shows the hint.
- [ ] Dumping transcript content does not replay stale fullscreen UI frames into terminal history (serialized as plain text, not ANSI frames).
- [ ] Transcript can return cleanly to fullscreen/live mode after the dump path is used (no mode lockout, no visual artifacts).
- [ ] KodaX does not add Claude's transcript `v` editor shortcut.
- [ ] Non-capable hosts (where AlternateScreen is not entered) behave no differently — the shortcut is a no-op or shows a "not available" hint.
- [ ] Unit tests: transcript serialization emits plain-text lines; ANSI escape sequences are stripped.

### Relationship to other features

- **FEATURE_051 (Completed)** — direct prerequisite. 051 shipped `AlternateScreen`, `FullscreenLayout`, `ScrollBox`, `renderer-runtime` — all four primitives 058 needs.
- **FEATURE_057 (v0.7.30, InProgress)** — no dependency. 057 is about *default fullscreen* interaction (wheel/drag/copy ownership), while 058 is an *explicit escape hatch* out of fullscreen. Non-overlapping. 058 non-goal ("do not replace fullscreen browser with main-screen as default") directly aligns with 057's reserve-main-screen-for-escape posture.
- **FEATURE_075 (v0.7.25)** — no dependency. Both reuse `ScrollBox` but for different surfaces (plan approval dialog vs transcript view).
