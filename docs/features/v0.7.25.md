## 总览

`v0.7.25` 的目标不是把 KodaX 做成更重的 terminal product。

这一版真正要完成的是：

- 把 transcript 做成稳定主屏
- 把输入、滚动、搜索、选择这些底层机制补成熟
- 把 footer / task / status 的表面控制在 KodaX 合适的克制档位

这意味着：

- 学 ClaudeCode 的机制成熟度
- 不复制 ClaudeCode 的重表面和可见复杂性

## FEATURE_051: Host-Aware Fullscreen TUI Substrate and Transcript UX

**Status**: Completed  
**Close-out note (2026-04-03)**: Implemented with host-aware transcript profiles, decoupled verbosity/follow/buffering state, fullscreen-or-fallback transcript control, and stable hidden-work strip integration.  
**Close-out addendum (2026-04-04)**: Final REPL TUI close-out freezes the current visible status/footer/task/message surfaces and keeps follow-up work focused on invisible substrate maturation: transcript owner, viewport/search/navigation, host-aware scroll/selection/copy, and input focus/paste/cursor behavior.  
**Handoff note (2026-04-04)**: Remaining REPL substrate hardening now belongs to `FEATURE_055` in `v0.9.0`; `051` should be read as the completed visible-shell close-out record.  
**Priority**: High  
**Category**: Enhancement

### Overview

`FEATURE_051` 把 KodaX 的 TUI 明确成三层：

1. `host-aware substrate`
2. `transcript-first interaction model`
3. `minimal, frozen visible shell`

前几轮已经完成了 transcript owner、prompt input controller、footer/task/status 的基础壳层。  
最终收尾的目标不是再长更多表面，而是把这些壳层背后的 owner、controller、host policy 和 budget 真正做稳。

### Product philosophy

#### 1. transcript 是绝对主屏

用户始终面对一个主 authority。  
transcript 是主工作面，其他都是辅助仪表。

#### 2. Adaptive inside, minimal outside

内部机制可以更智能、更 host-aware，但外部不要求用户理解系统内部组织。

#### 3. mechanism first, surface restraint

继续补 transcript / scroll / selection / input 这些 invisible substrate。  
停止扩更多长期可见的表面。

#### 4. cockpit, not control tower

KodaX 可以像 AI 工作台驾驶舱，但应该是：

- 一个主屏
- 一条克制仪表带
- 少量按需出现的 overlay

而不是：

- 更多并列 panel
- 更重 task dashboard
- 可见 worker topology

### Final close-out posture (2026-04-04)

`FEATURE_051` 在用户可见层面已经达到 KodaX 目标档位，后续收尾遵循：

- transcript 继续是唯一主屏
- 当前 `StatusBar` 的内容、顺序、符号和信息密度冻结
- 当前 footer / help / task / message action / tool row 的可见表面冻结或尽量冻结
- 后续实现只允许继续补 owner / controller / host policy / budget 这些 invisible substrate

这条口径的目的不是降低成熟度，而是让 KodaX 保持：

- 更极简
- 更智能
- 更克制
- 更少长期可见 surface

### Goals

1. 在不扩 visible surface 的前提下，把 transcript / scroll / selection / input 这几层 substrate 补到成熟。
2. 保持 transcript 仍是唯一主屏，让 `InkREPL` 继续只做 orchestration shell，而不是重新长回大 owner。
3. 让不同 terminal host 走清晰的 capability-driven 策略，尤其是 selection、copy-on-select、wheel history 和 fullscreen chrome。
4. 冻结当前 `StatusBar`、footer、task bar、message/tool surface 的用户可见档位，只允许内部整理和行为闭环。
5. 学 ClaudeCode 的机制成熟度，但不复制更重的工作台表面。

### Scope

- transcript chrome / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner、budget 和 view-model 收口
- `FEATURE_051` 文档与 feature index 的最终收尾口径同步

### Non-goals

- 复制 ClaudeCode 的重表面和可见复杂性
- 继续扩更多长期可见的 footer/help/task/message surface
- 引入 visible worker graph、task tree、teammate dashboard、persistent notification center
- 改变当前 `StatusBar`、footer、task bar、message actions、tool rows 的用户可见密度
- 用一次性 renderer rewrite 替代 owner / controller / host policy 的渐进收口

### Current pain to resolve

#### A. transcript owner 还不够强

owner 骨架已经有了，但 sticky prompt、jump-to-latest、search anchor、reveal policy 还没完全内聚到 owner 层。

#### B. scroll / selection / copy 机制还不够一体化

manual scroll、page nav、selection clear、copy-on-select、search disarm 还没完全变成一套 host-aware 规则。

#### C. input core 在 focus / paste / cursor 这些方面还没最终闭环

这类问题不会要求新 UI，但会直接影响“手感”和稳定性。

### State model

`FEATURE_051` 的状态模型在前几轮已经成型；最终收尾不再扩新的用户可见 mode，而是把现有语义彻底收口。

```ts
type TranscriptVerbosity = "compact" | "verbose";
type TranscriptFollowMode = "follow-bottom" | "browsing-history";
type LiveBufferingPolicy = "live" | "buffered-fallback";

interface TranscriptDisplayState {
  verbosity: TranscriptVerbosity;
  followMode: TranscriptFollowMode;
  buffering: LiveBufferingPolicy;
  scrollAnchor: number;
  jumpToLatestAvailable: boolean;
  stickyPromptVisible: boolean;
  selectionMode: "none" | "message";
  selectedItemId?: string;
  searchMode: "closed" | "history";
  searchAnchorItemId?: string;
  currentMatchIndex: number;
}
```

最终收尾约束：

- `verbosity`、`followMode`、`buffering` 继续独立，不再彼此偷语义
- `searchMode` 只表达 transcript history search，不再挟带显示密度或 review 语义
- `currentMatchIndex = -1` 合法，表示 query 仍在，但当前 match 已被 manual scroll / page nav disarm
- `selectedItemId` 只在 host 支持 viewport-owned selection path 时进入正式交互路径

### Host policy matrix

最终收尾阶段，host classification 不再只是声明能力，而要真正驱动 selection / copy / wheel 行为。

| Host class | Fullscreen / overlay | Selection path | Copy behavior | Wheel / browse policy |
|---|---|---|---|---|
| `native_vt` | 开启 owner-driven chrome | 开启 | copy-on-select + 显式复制 | 完整 wheel history |
| `xtermjs_host` | 开启，但更保守 | 开启 | 只保留显式复制 | 保守 wheel history |
| `degraded_vt` | 不开启 fullscreen chrome | 不进入 viewport-owned selection path | 不启用 copy-on-select | 仅保守 fallback |
| `unsupported_control_host` | append-only / fallback only | 不进入 | 不启用 | 不进入 |

### Shortcut contract

快捷键集合在前几轮已基本确定；最终收尾只修规则，不再扩新的 visible action。

- `Ctrl+O`: toggle transcript verbosity
- `PgUp`: enter browsing-history
- `Ctrl+Y`: explicit history browsing toggle
- `Alt+Z`: ergonomic backup for history browsing
- `Esc`: leave browsing-history
- `End`: jump to latest and restore follow-bottom
- `c`: copy selected transcript entry
- `i`: copy selected tool input
- `v`: toggle detail for selected entry

最终收尾约束：

- 上述键位集合不再增加新的用户可见动作
- `c / i / v` 的可见性与可用性继续由 host capability 和当前 selection state 决定
- 输入层继续保证优先级：`dialog/request > prompt editing > transcript navigation > global shortcuts`

### Component responsibility split

这部分在前几轮已经部分落地；最终收尾只做“把行为真相放对地方”，不再新增表面组件。

#### `InkREPL.tsx`

- 继续负责 runtime/session state、顶层 mode、surface 出现条件和 view-model 组装
- 不重新拿回 transcript / footer / task / status 的行为真相

#### `FullscreenTranscriptLayout.tsx`

- 保留当前可见 chrome
- 继续作为 sticky header / jump-to-latest / overlay / footer 层级关系的 owner

#### `TranscriptViewport.tsx`

- 保留当前 browse/search/selection surface 的可见样子
- 继续下沉 search anchor、match stepping、selection reveal、tool row reveal 等行为真相

#### `prompt-input-controller.ts`

- 继续作为输入逻辑 owner
- 保持 `Enter/Esc/Tab/Up/Down` 的优先级和 paste / focus / cursor 行为收口
- 不把逻辑退回 `InputPrompt` / `TextInput` 条件树

#### `StatusBar.tsx`

- 可见表现冻结
- 只作为 renderer；格式化和策略继续往 view-model builder 收

### Final design direction

#### 1. Keep the current visible shell

`FEATURE_051` 的最终目标不是“再长更多 UI”，而是：

- transcript 继续是唯一主屏
- footer / help / task / message / tool 表面维持当前档位
- `StatusBar` 可见文本保持不变

#### 2. Continue strengthening invisible substrate

真正还要继续做的是：

- transcript owner / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner 与 budget 收口

#### 3. Prefer host-aware correctness over faux parity

KodaX 不追求“所有 host 看起来都一样强”，而是：

- safe host 上提供成熟行为
- weak host 上收窄到正确行为
- 不用重表面掩盖弱能力路径

## FEATURE_054: AMA-Project Convergence — Absorb Project Mode into Adaptive H2

**Status**: Planned
**Priority**: High
**Category**: Refactor

### Overview

`FEATURE_054` 将独立的 `Project` 模式（`/project` 命令组）完全融合进 AMA H2 的自适应流程。

当前 KodaX 存在两套独立的规划系统叠在一起：

- **Project 模式**（REPL 层）：`/project init / brainstorm / plan / next / auto / quality / verify / status`，8 个文件，独立持久化 `.agent/project/`
- **AMA 模式**（task-engine 层）：`Scout -> Planner -> Generator <-> Evaluator`，独立持久化 `.agent/managed-tasks/`

两套系统互不感知，同一任务可能产出两份计划（Project 的 `session_plan.md` 和 AMA 的 `kodax-task-contract`），违背了 KodaX 极简且智能的构建哲学。

### Product philosophy

#### 1. 用户入口极简

用户不需要学 `/project init / brainstorm / plan / auto` 这套命令。直接说需求，AMA 全自动处理。

#### 2. 智能自适应

AMA 根据请求复杂度自动选择 H0/H1/H2。H2 已经包含完整的 plan -> execute -> evaluate 流程，不需要外部命令来手动编排。

#### 3. Brainstorm 是 Planner 的内建能力

不是独立 Agent 或独立阶段。Planner 在信息不充分时使用 `ask_user_question` 进行多轮对齐，然后生成 `kodax-task-contract`。

### Problem analysis

#### 矛盾 1：两层规划互相不知道对方的存在

```text
/project plan → buildProjectPlan() 确定性生成 4-phase plan
/project next → runManagedTask(taskSurface='project')
  → AMA routing 可能选 H2 → Planner 生成自己的 contract（不知道 session_plan.md 的存在）
```

结果：同一任务有两个计划。

#### 矛盾 2：Project 的 plan 是确定性的，但执行是自适应的

`buildProjectPlan()` 用关键词匹配推断复杂度，但真正执行时 AMA 的 Scout 可能判定完全不同的复杂度。两者的 Harness 级别、角色组合、预算可能完全不匹配。

#### 矛盾 3：两个独立的状态机

```text
Project workflow state: bootstrap → discovering → aligned → planned → executing → completed
AMA managed task lifecycle: created → running → [rounds] → completed/blocked
```

两个状态机没有任何同步点。

#### 矛盾 4：Brainstorm 能力的空壳化

AMA 的 `inferRequiresBrainstorm()` 只是往 prompt overlay 加了一句"请先讨论"，没有独立阶段、没有状态、没有持久化、没有结构化输出。而 Project 模式的 brainstorm 是完整的多轮对齐机制。

### Goals

1. **废弃 `/project` 命令组**——所有命令全部移除，用户不再需要学任何 `/project` 命令。
2. **Planner 吸收 brainstorm 能力**——Planner 在需求不清晰时自动进入多轮对齐模式，对齐完成后生成 contract。
3. **统一持久化**——`.agent/project/` 不再存在，所有状态统一存入 `.agent/managed-tasks/<id>/`。
4. **更新 DD.md §9.4**——将"正交维度"描述改为"融合进 AMA H2"。

### Scope

- 废弃 `packages/repl/src/interactive/project-*.ts`（8 个文件）
- 增强 Planner 的 brainstorm 能力（prompt + task-engine 层面结合）
- 持久化路径统一
- 文档更新（DD.md、FEATURE_LIST.md）

### Non-goals

- 自动迁移用户已有的 `.agent/project/` 数据（由用户自行处理）
- 修改 H0/H1 的行为（它们不涉及 Project 模式）
- 重写 task-engine.ts 的核心 managed task lifecycle

### Core decisions

#### Decision 1: `/project` 命令组全部废弃

| 废弃命令 | 替代方案 |
|---------|---------|
| `/project init` | 用户直接说需求，AMA 自动创建 managed task |
| `/project brainstorm` | Planner 的 brainstorm 工作模式（自动触发或用户说"先讨论"） |
| `/project plan` | AMA Planner 自动生成 contract |
| `/project next/auto` | AMA round-based execution 自动推进 |
| `/project quality/verify` | AMA Evaluator 自动检查 |
| `/project status` | 用户自然语言询问进度，AMA 从 managed task 读取回答 |

#### Decision 2: Brainstorm 是 Planner 的内建能力

Planner 接收 Scout 的 scope facts 后有两种自然过渡的工作模式：

- **需求清晰**：直接生成 `kodax-task-contract`
- **需求模糊**：用 `ask_user_question` 多轮对齐（brainstorm）→ 再生成 contract

这两种模式由 Planner 自行判断，不需要外部路由切换。

`inferRequiresBrainstorm()` 的输出从"prompt overlay 一句话"改为"提示 Planner 需要多轮对齐"。

**实现层面是两种结合**：
- **Planner prompt**：加入 brainstorm 能力描述——"如果需求不清晰，先用 `ask_user_question` 对齐再生成 contract"
- **task-engine 层面**：在 `requiresBrainstorm=true` 时为 Planner 提供 `ask_user_question` 工具权限和更宽松的交互预算

不需要新增角色/Agent，只需要强化 Planner 的能力。

#### Decision 3: 持久化统一到 managed-tasks

| 数据 | 原位置 | 新位置 | 处理 |
|------|--------|--------|------|
| alignment | `.agent/project/alignment.md` | `.agent/managed-tasks/<id>/alignment.md` | 迁入 |
| brainstorm 记录 | `.agent/project/brainstorm/` | `.agent/managed-tasks/<id>/brainstorm.md` | 合并迁入 |
| evidence | `.agent/project/evidence/` | `.agent/managed-tasks/<id>/evidence/` | 迁入 |
| checkpoints | `.agent/project/checkpoints/` | `.agent/managed-tasks/<id>/checkpoints/` | 迁入 |
| session plan | `.agent/project/session_plan.md` | 被 `contract.json` 替代 | 废弃 |
| control-state / project-state | `.agent/project/` | 被 managed task lifecycle 替代 | 废弃 |
| harness config / runs | `.agent/project/harness/` | 被 AMA 自适应替代 | 废弃 |
| lightweight run record | `.agent/project/lightweight-run.json` | 不再需要 | 废弃 |

已存在的 `.agent/project/` 目录由用户自行处理，系统不做自动迁移。
