## 总览

`v0.7.25` 的目标不是把 KodaX 做成更重的 terminal product。

这一版真正要完成的是：

- 把 transcript 做成稳定主屏
- 把输入、滚动、搜索、选择这些底层机制补成熟
- 把 footer / task / status 的表面控制在 KodaX 合适的克制档位

这意味着：

- 学 ClaudeCode 的机制成熟度
- 不复制 ClaudeCode 的重表面和可见复杂性

## FEATURE_051: Host-Aware Fullscreen TUI Substrate and Transcript UX

**Status**: Completed  
**Close-out note (2026-04-03)**: Implemented with host-aware transcript profiles, decoupled verbosity/follow/buffering state, fullscreen-or-fallback transcript control, and stable hidden-work strip integration.  
**Close-out addendum (2026-04-04)**: Final REPL TUI close-out freezes the current visible status/footer/task/message surfaces and keeps follow-up work focused on invisible substrate maturation: transcript owner, viewport/search/navigation, host-aware scroll/selection/copy, and input focus/paste/cursor behavior.  
**Handoff note (2026-04-04)**: Remaining REPL substrate hardening moved to `FEATURE_055` in `v0.9.0` and was completed there without reopening the visible shell; `051` should be read as the completed visible-shell close-out record.  
**Priority**: High  
**Category**: Enhancement

### Overview

`FEATURE_051` 把 KodaX 的 TUI 明确成三层：

1. `host-aware substrate`
2. `transcript-first interaction model`
3. `minimal, frozen visible shell`

前几轮已经完成了 transcript owner、prompt input controller、footer/task/status 的基础壳层。  
最终收尾的目标不是再长更多表面，而是把这些壳层背后的 owner、controller、host policy 和 budget 真正做稳。

### Product philosophy

#### 1. transcript 是绝对主屏

用户始终面对一个主 authority。  
transcript 是主工作面，其他都是辅助仪表。

#### 2. Adaptive inside, minimal outside

内部机制可以更智能、更 host-aware，但外部不要求用户理解系统内部组织。

#### 3. mechanism first, surface restraint

继续补 transcript / scroll / selection / input 这些 invisible substrate。  
停止扩更多长期可见的表面。

#### 4. cockpit, not control tower

KodaX 可以像 AI 工作台驾驶舱，但应该是：

- 一个主屏
- 一条克制仪表带
- 少量按需出现的 overlay

而不是：

- 更多并列 panel
- 更重 task dashboard
- 可见 worker topology

### Final close-out posture (2026-04-04)

`FEATURE_051` 在用户可见层面已经达到 KodaX 目标档位，后续收尾遵循：

- transcript 继续是唯一主屏
- 当前 `StatusBar` 的内容、顺序、符号和信息密度冻结
- 当前 footer / help / task / message action / tool row 的可见表面冻结或尽量冻结
- 后续实现只允许继续补 owner / controller / host policy / budget 这些 invisible substrate

这条口径的目的不是降低成熟度，而是让 KodaX 保持：

- 更极简
- 更智能
- 更克制
- 更少长期可见 surface

### Goals

1. 在不扩 visible surface 的前提下，把 transcript / scroll / selection / input 这几层 substrate 补到成熟。
2. 保持 transcript 仍是唯一主屏，让 `InkREPL` 继续只做 orchestration shell，而不是重新长回大 owner。
3. 让不同 terminal host 走清晰的 capability-driven 策略，尤其是 selection、copy-on-select、wheel history 和 fullscreen chrome。
4. 冻结当前 `StatusBar`、footer、task bar、message/tool surface 的用户可见档位，只允许内部整理和行为闭环。
5. 学 ClaudeCode 的机制成熟度，但不复制更重的工作台表面。

### Scope

- transcript chrome / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner、budget 和 view-model 收口
- `FEATURE_051` 文档与 feature index 的最终收尾口径同步

### Non-goals

- 复制 ClaudeCode 的重表面和可见复杂性
- 继续扩更多长期可见的 footer/help/task/message surface
- 引入 visible worker graph、task tree、teammate dashboard、persistent notification center
- 改变当前 `StatusBar`、footer、task bar、message actions、tool rows 的用户可见密度
- 用一次性 renderer rewrite 替代 owner / controller / host policy 的渐进收口

### Current pain to resolve

#### A. transcript owner 还不够强

owner 骨架已经有了，但 sticky prompt、jump-to-latest、search anchor、reveal policy 还没完全内聚到 owner 层。

#### B. scroll / selection / copy 机制还不够一体化

manual scroll、page nav、selection clear、copy-on-select、search disarm 还没完全变成一套 host-aware 规则。

#### C. input core 在 focus / paste / cursor 这些方面还没最终闭环

这类问题不会要求新 UI，但会直接影响“手感”和稳定性。

### State model

`FEATURE_051` 的状态模型在前几轮已经成型；最终收尾不再扩新的用户可见 mode，而是把现有语义彻底收口。

```ts
type TranscriptVerbosity = "compact" | "verbose";
type TranscriptFollowMode = "follow-bottom" | "browsing-history";
type LiveBufferingPolicy = "live" | "buffered-fallback";

interface TranscriptDisplayState {
  verbosity: TranscriptVerbosity;
  followMode: TranscriptFollowMode;
  buffering: LiveBufferingPolicy;
  scrollAnchor: number;
  jumpToLatestAvailable: boolean;
  stickyPromptVisible: boolean;
  selectionMode: "none" | "message";
  selectedItemId?: string;
  searchMode: "closed" | "history";
  searchAnchorItemId?: string;
  currentMatchIndex: number;
}
```

最终收尾约束：

- `verbosity`、`followMode`、`buffering` 继续独立，不再彼此偷语义
- `searchMode` 只表达 transcript history search，不再挟带显示密度或 review 语义
- `currentMatchIndex = -1` 合法，表示 query 仍在，但当前 match 已被 manual scroll / page nav disarm
- `selectedItemId` 只在 host 支持 viewport-owned selection path 时进入正式交互路径

### Host policy matrix

最终收尾阶段，host classification 不再只是声明能力，而要真正驱动 selection / copy / wheel 行为。

| Host class | Fullscreen / overlay | Selection path | Copy behavior | Wheel / browse policy |
|---|---|---|---|---|
| `native_vt` | 开启 owner-driven chrome | 开启 | copy-on-select + 显式复制 | 完整 wheel history |
| `xtermjs_host` | 开启，但更保守 | 开启 | 只保留显式复制 | 保守 wheel history |
| `degraded_vt` | 不开启 fullscreen chrome | 不进入 viewport-owned selection path | 不启用 copy-on-select | 仅保守 fallback |
| `unsupported_control_host` | append-only / fallback only | 不进入 | 不启用 | 不进入 |

### Shortcut contract

快捷键集合在前几轮已基本确定；最终收尾只修规则，不再扩新的 visible action。

- `Ctrl+O`: toggle transcript verbosity
- `PgUp`: enter browsing-history
- `Ctrl+Y`: explicit history browsing toggle
- `Alt+Z`: ergonomic backup for history browsing
- `Esc`: leave browsing-history
- `End`: jump to latest and restore follow-bottom
- `c`: copy selected transcript entry
- `i`: copy selected tool input
- `v`: toggle detail for selected entry

最终收尾约束：

- 上述键位集合不再增加新的用户可见动作
- `c / i / v` 的可见性与可用性继续由 host capability 和当前 selection state 决定
- 输入层继续保证优先级：`dialog/request > prompt editing > transcript navigation > global shortcuts`

### Component responsibility split

这部分在前几轮已经部分落地；最终收尾只做“把行为真相放对地方”，不再新增表面组件。

#### `InkREPL.tsx`

- 继续负责 runtime/session state、顶层 mode、surface 出现条件和 view-model 组装
- 不重新拿回 transcript / footer / task / status 的行为真相

#### `FullscreenTranscriptLayout.tsx`

- 保留当前可见 chrome
- 继续作为 sticky header / jump-to-latest / overlay / footer 层级关系的 owner

#### `TranscriptViewport.tsx`

- 保留当前 browse/search/selection surface 的可见样子
- 继续下沉 search anchor、match stepping、selection reveal、tool row reveal 等行为真相

#### `prompt-input-controller.ts`

- 继续作为输入逻辑 owner
- 保持 `Enter/Esc/Tab/Up/Down` 的优先级和 paste / focus / cursor 行为收口
- 不把逻辑退回 `InputPrompt` / `TextInput` 条件树

#### `StatusBar.tsx`

- 可见表现冻结
- 只作为 renderer；格式化和策略继续往 view-model builder 收

### Final design direction

#### 1. Keep the current visible shell

`FEATURE_051` 的最终目标不是“再长更多 UI”，而是：

- transcript 继续是唯一主屏
- footer / help / task / message / tool 表面维持当前档位
- `StatusBar` 可见文本保持不变

#### 2. Continue strengthening invisible substrate

真正还要继续做的是：

- transcript owner / viewport owner 的剩余闭环
- host-aware scroll / selection / copy-on-select 规则
- input focus / paste / cursor substrate
- footer / task / status 的内部 owner 与 budget 收口

#### 3. Prefer host-aware correctness over faux parity

KodaX 不追求“所有 host 看起来都一样强”，而是：

- safe host 上提供成熟行为
- weak host 上收窄到正确行为
- 不用重表面掩盖弱能力路径

## FEATURE_054: AMA-Project Convergence — Absorb Project Mode into Adaptive H2

**Status**: Planned
**Priority**: High
**Category**: Refactor

> Source-of-truth note (2026-04-04):
> The active planned version for `FEATURE_054` is now [v0.7.30](./v0.7.30.md#feature_054-ama-project-convergence-absorb-project-mode-into-adaptive-h2).
> This section remains as historical staging context only.

### Overview

`FEATURE_054` 将独立的 `Project` 模式（`/project` 命令组）完全融合进 AMA H2 的自适应流程。

当前 KodaX 存在两套独立的规划系统叠在一起：

- **Project 模式**（REPL 层）：`/project init / brainstorm / plan / next / auto / quality / verify / status`，8 个文件，独立持久化 `.agent/project/`
- **AMA 模式**（task-engine 层）：`Scout -> Planner -> Generator <-> Evaluator`，独立持久化 `.agent/managed-tasks/`

两套系统互不感知，同一任务可能产出两份计划（Project 的 `session_plan.md` 和 AMA 的 `kodax-task-contract`），违背了 KodaX 极简且智能的构建哲学。

### Product philosophy

#### 1. 用户入口极简

用户不需要学 `/project init / brainstorm / plan / auto` 这套命令。直接说需求，AMA 全自动处理。

#### 2. 智能自适应

AMA 根据请求复杂度自动选择 H0/H1/H2。H2 已经包含完整的 plan -> execute -> evaluate 流程，不需要外部命令来手动编排。

#### 3. Brainstorm 是 Planner 的内建能力

不是独立 Agent 或独立阶段。Planner 在信息不充分时使用 `ask_user_question` 进行多轮对齐，然后生成 `kodax-task-contract`。

#### 4. one main authority，不能复活第二套 project surface

`FEATURE_054` 的目标不是把旧 `Project` 模式换个壳留下来，而是彻底消除“第二套规划入口”和“第二套任务表面”：

- 不恢复 project dashboard / planner panel / project footer
- 不保留与 AMA 并行存在的 project workflow state machine
- 不让用户再学习一套 `/project ...` 交互语法

### Problem analysis

#### 矛盾 1：两层规划互相不知道对方的存在

```text
/project plan → buildProjectPlan() 确定性生成 4-phase plan
/project next → runManagedTask(taskSurface='project')
  → AMA routing 可能选 H2 → Planner 生成自己的 contract（不知道 session_plan.md 的存在）
```

结果：同一任务有两个计划。

#### 矛盾 2：Project 的 plan 是确定性的，但执行是自适应的

`buildProjectPlan()` 用关键词匹配推断复杂度，但真正执行时 AMA 的 Scout 可能判定完全不同的复杂度。两者的 Harness 级别、角色组合、预算可能完全不匹配。

#### 矛盾 3：两个独立的状态机

```text
Project workflow state: bootstrap → discovering → aligned → planned → executing → completed
AMA managed task lifecycle: created → running → [rounds] → completed/blocked
```

两个状态机没有任何同步点。

#### 矛盾 4：Brainstorm 能力的空壳化

AMA 的 `inferRequiresBrainstorm()` 只是往 prompt overlay 加了一句"请先讨论"，没有独立阶段、没有状态、没有持久化、没有结构化输出。而 Project 模式的 brainstorm 是完整的多轮对齐机制。

### Goals

1. **废弃 `/project` 命令组**——所有命令全部移除，用户不再需要学任何 `/project` 命令。
2. **Planner 吸收 brainstorm 能力**——Planner 在需求不清晰时自动进入多轮对齐模式，对齐完成后生成 contract。
3. **统一持久化**——`.agent/project/` 不再存在，所有状态统一存入 `.agent/managed-tasks/<id>/`。
4. **更新 DD.md §9.4**——将"正交维度"描述改为"融合进 AMA H2"。

### Scope

- 废弃 `packages/repl/src/interactive/project-*.ts`（8 个文件）
- 增强 Planner 的 brainstorm 能力（prompt + task-engine 层面结合）
- 持久化路径统一
- 文档更新（DD.md、FEATURE_LIST.md）

### Non-goals

- 自动迁移用户已有的 `.agent/project/` 数据（由用户自行处理）
- 修改 H0/H1 的行为（它们不涉及 Project 模式）
- 重写 task-engine.ts 的核心 managed task lifecycle
- 保留或重建独立的 project dashboard / planner surface / project-mode UX
- 在 transcript / footer / task bar 里新增 “project system” 专属表面

### Core decisions

#### Decision 1: `/project` 命令组全部废弃

| 废弃命令 | 替代方案 |
|---------|---------|
| `/project init` | 用户直接说需求，AMA 自动创建 managed task |
| `/project brainstorm` | Planner 的 brainstorm 工作模式（自动触发或用户说"先讨论"） |
| `/project plan` | AMA Planner 自动生成 contract |
| `/project next/auto` | AMA round-based execution 自动推进 |
| `/project quality/verify` | AMA Evaluator 自动检查 |
| `/project status` | 用户自然语言询问进度，AMA 从 managed task 读取回答 |

#### Decision 2: Brainstorm 是 Planner 的内建能力

Planner 接收 Scout 的 scope facts 后有两种自然过渡的工作模式：

- **需求清晰**：直接生成 `kodax-task-contract`
- **需求模糊**：用 `ask_user_question` 多轮对齐（brainstorm）→ 再生成 contract

这两种模式由 Planner 自行判断，不需要外部路由切换。

`inferRequiresBrainstorm()` 的输出从"prompt overlay 一句话"改为"提示 Planner 需要多轮对齐"。

**实现层面是两种结合**：
- **Planner prompt**：加入 brainstorm 能力描述——"如果需求不清晰，先用 `ask_user_question` 对齐再生成 contract"
- **task-engine 层面**：在 `requiresBrainstorm=true` 时为 Planner 提供 `ask_user_question` 工具权限和更宽松的交互预算

不需要新增角色/Agent，只需要强化 Planner 的能力。

#### Decision 3: 持久化统一到 managed-tasks

| 数据 | 原位置 | 新位置 | 处理 |
|------|--------|--------|------|
| alignment | `.agent/project/alignment.md` | `.agent/managed-tasks/<id>/alignment.md` | 迁入 |
| brainstorm 记录 | `.agent/project/brainstorm/` | `.agent/managed-tasks/<id>/brainstorm.md` | 合并迁入 |
| evidence | `.agent/project/evidence/` | `.agent/managed-tasks/<id>/evidence/` | 迁入 |
| checkpoints | `.agent/project/checkpoints/` | `.agent/managed-tasks/<id>/checkpoints/` | 迁入 |
| session plan | `.agent/project/session_plan.md` | 被 `contract.json` 替代 | 废弃 |

## FEATURE_073: Reference-Style Lineage and Island Model Removal

**Status**: Planned
**Priority**: Medium
**Category**: Internal / Refactor
**Builds on**: `v0.7.20 FEATURE_072` (lineage-native compaction migration)
**Reference implementations**: pi-mono `buildSessionContext`, Claude Code `getMessagesAfterCompactBoundary`

### Overview

After `FEATURE_072` lands, KodaX's compaction hot path reads from lineage via `getSessionMessagesFromLineage`, and post-compact attachments live natively on `KodaXSessionCompactionEntry`. The **data structure shape** and **consumer contracts** are pi-mono-compatible — but the underlying **island model** (copy-on-compact) is preserved.

`FEATURE_073` is the structural follow-up: migrate from copy-style to reference-style. Instead of every compaction creating a new `parentId: null` island with a duplicated kept tail, the lineage becomes a single linear chain (with occasional user-triggered forks); CompactionEntries become inline markers that reference the original pre-compaction entries via `firstKeptEntryId`.

### Why this is a separate feature, not part of 072

`FEATURE_072` targets the user-visible regression (monotonic context growth under AMA). Copy-style vs reference-style is invisible to users — both produce identical LLM input given the same compaction decisions. The structural change touches `archiveOldIslands`, `buildSessionTree`, and session persistence compatibility; those are user-visible paths and carry their own risk. Bundling them with 072 would have ballooned scope and delayed the growth-bug fix.

Splitting also lets 072 stabilize in production first. If copy-style turns out to have no real downside, 073 can be re-evaluated or cancelled.

### Current (copy-style) vs target (reference-style)

| Aspect | Copy-style (v0.7.20) | Reference-style (v0.7.25+) |
|---|---|---|
| `CompactionEntry.parentId` | `null` (new island root) | Last entry before compaction |
| Kept tail | Copied as new entries under CompactionEntry | Original entries, referenced via `firstKeptEntryId` |
| Old content cleanup | `evictOldIslandMessageContent` replaces message body with `[compacted]` placeholder | No cleanup needed — sliced out of LLM view by derivation |
| `/fork` to pre-compaction entry | Target entry may be a placeholder after eviction | Target entry physically intact |
| `buildSessionTree` roots | One per island | One (plus user-created forks) |
| Multi-round memory growth | Each round duplicates kept tail | Flat — kept tail referenced, not copied |

### Goals

1. **Single-chain lineage.** Eliminate the "multiple islands per session" shape; all entries live on one primary chain (forks remain user-visible branches).
2. **Reference semantics for `firstKeptEntryId`.** Field points at the original pre-compaction entry, not a new copy.
3. **Remove `evictOldIslandMessageContent` from the main compaction path.** Old entries remain physical; slicing in `getSessionMessagesFromLineage` keeps them out of the LLM view.
4. **Preserve `/fork`, `/rewind`, `/load-session` observable behavior.** `/fork` to a pre-compaction entry becomes semantically clean (original entry still exists).
5. **Align with pi-mono and Claude Code** so future features (partial compaction, multi-boundary, inline-marker compactions, `cache_edits`-style) can be adopted without another round of structural surgery.

### Non-goals

- **Not changing the public shape of `KodaXSessionCompactionEntry`.** Only internal interpretation of `parentId` and `firstKeptEntryId` changes.
- **Not changing the external API of `getSessionMessagesFromLineage`.** Function signature and return shape stay identical; only internal walk logic changes.
- **Not changing persistence format incompatibly.** Old sessions load through a compat read path; new compactions write the new parent-linking. No forced migration.
- **Not introducing partial compaction, multi-boundary, or `cache_edits`.** Those become easier but are separate features.
- **Not removing `archiveOldIslands`.** It is reworked, not removed — archival policy becomes "entries before latest compaction not reachable via label/branch", but the archival surface (`/archive`, slimmed lineage output) is preserved.
- **Not touching 051's visible shell.** This feature is pure internal restructure of `@kodax/agent`.

### Design

#### 1. `applySessionCompaction` — no new island

```ts
// Before (copy-style, v0.7.20)
compactionEntry.parentId = null;
// createSessionLineage then duplicates kept tail under compactionEntry
// evictOldIslandMessageContent sweeps old-island message bodies

// After (reference-style, v0.7.25+)
compactionEntry.parentId = lastPreCompactionEntry.id;
compactionEntry.firstKeptEntryId = <first original entry in kept range>.id;
compactionEntry.postCompactAttachments = [...];  // 072 field, unchanged
// No kept-tail duplication; old entries stay in place
```

#### 2. `getSessionMessagesFromLineage` — slicing semantics

```ts
// Pseudocode
const path = getSessionLineagePath(lineage, targetId);
const lastCompactionIdx = findLastIndex(
  path,
  e => e.type === 'compaction' && e.reason !== 'rewind',
);

if (lastCompactionIdx === -1) {
  return path.flatMap(getContextMessagesForEntry);
}

const compaction = path[lastCompactionIdx];
const firstKeptIdx = compaction.firstKeptEntryId
  ? path.findIndex(e => e.id === compaction.firstKeptEntryId)
  : -1;

// Start point: firstKept if present, else entry after compaction marker
const tailStart = firstKeptIdx >= 0 ? firstKeptIdx : lastCompactionIdx + 1;

return [
  summaryMessage(compaction.summary),
  ...(compaction.postCompactAttachments ?? []),
  ...path.slice(tailStart).flatMap(getContextMessagesForEntry),
];
```

**Note**: `reason !== 'rewind'` filter — `rewindSessionLineage` uses `type: 'compaction'` entries as audit markers ([session-lineage.ts:812-823](../../packages/agent/src/session-lineage.ts#L812-L823)); they must not trigger compaction slicing.

#### 3. `archiveOldIslands` reworked

The "island = `parentId: null` subtree" heuristic is replaced with a reachability-based policy on a single chain:

```
Archive candidates = entries before the latest compaction's firstKeptEntryId that are NOT reachable via:
  - any active leaf's path
  - any label target's ancestor chain
  - any branch_summary's fromId
  - any fork branch's ancestor chain
```

Same observable behavior (archive old stuff, keep what's referenced) but implemented on a single chain with explicit reachability, not structural island detection.

#### 4. `buildSessionTree` root identification

Today: walk `lineage.entries`, pick entries with `parentId === null` as roots. Each compaction creates a new root.

New: a reference-style session has exactly one original root entry (`parentId === null`). Compaction entries appear inline on the chain. Forks produce branches off their shared parent, same as today. Visual rendering shifts from "multiple trees stacked" to "one tree with inline compaction markers" — closer to how pi-mono / Claude Code present session history.

#### 5. Session persistence compatibility

Old sessions (v0.7.20 and earlier) have CompactionEntries with `parentId: null`. Compat read path:

- On load, `getSessionMessagesFromLineage` detects CompactionEntries with `parentId === null` and treats them as historical island boundaries — the slicing algorithm handles both shapes transparently.
- On next compaction, the new CompactionEntry uses reference-style linking. Old entries on disk stay copy-style; new ones are reference-style. Mixed-shape sessions are valid indefinitely.
- No eager rewrite. No forced migration.
- `evictOldIslandMessageContent` is kept as a legacy path, invoked only when a new CompactionEntry is created in a session that still has copy-style history. Eventually (after all sessions have rolled over), it becomes dead code.

#### 6. Duplicate kept-tail cleanup (deferred)

After migration, old sessions still contain the duplicated kept-tail entries under each pre-0.7.25 CompactionEntry. They remain valid and reachable via `/fork`. Cleanup is a separate garbage-collection feature if disk footprint becomes an issue — not scoped here.

### Phasing

| Phase | Scope | Ship criteria |
|---|---|---|
| A | Add compat read in `getSessionMessagesFromLineage`: handle both `parentId: null` CompactionEntries (copy-style) and parent-linked ones (reference-style) transparently. No write-side changes yet. | Unit tests: both session shapes produce identical LLM view. Zero behaviour change for existing callers. |
| B | Switch `applySessionCompaction` to reference-style writes. `firstKeptEntryId` now points at an original entry, not a copy. `evictOldIslandMessageContent` becomes a no-op on reference-style entries but still runs on copy-style legacy sessions. | Unit tests: new compactions produce reference-style lineage; sessions saved under v0.7.20 still load and function. |
| C | Rework `buildSessionTree` for single-chain rendering. Rework `archiveOldIslands` policy to reachability-based on a single chain. | `/list-sessions`, `/archive`, session tree UI all produce expected output on both session shapes. Multi-round compaction memory benchmark shows flat `context.lineage.entries.length` growth per compaction. |
| D | Cleanup: `evictOldIslandMessageContent` removed from the main compaction path (kept only as a legacy-session helper, or scheduled for deletion once legacy sessions are believed to have rolled over). Optional warn log on load for copy-style sessions. | Code simplification; no behaviour change. |

### Acceptance criteria

- [ ] After migration, `applySessionCompaction` produces `CompactionEntry.parentId !== null` (points at pre-compaction last entry) for all new compactions.
- [ ] `firstKeptEntryId` points at an **existing** entry in the pre-compaction path, not a newly-created copy. Unit test: `lineage.entries.some(e => e.id === compaction.firstKeptEntryId && e.timestamp < compaction.timestamp)` is true.
- [ ] `getSessionMessagesFromLineage` produces identical LLM-visible message content for the same logical history, regardless of whether compactions were copy-style or reference-style.
- [ ] `/fork` to a pre-compaction entry is observationally identical whether the session was created under copy-style or reference-style compactions, and the target entry is never a `[compacted]` placeholder on reference-style sessions.
- [ ] **Memory benchmark:** multi-round compaction (≥5 rounds) increases `context.lineage.entries.length` by O(1) per round under reference-style, versus O(kept_tail_size) under copy-style.
- [ ] Sessions persisted under v0.7.20 load without error and without eager rewrite; derivation handles mixed shapes.
- [ ] `archiveOldIslands` produces the same archival output (entries archived, markers inserted, preserve closure) as the old policy, validated against regression fixtures that predate the change.
- [ ] `buildSessionTree` produces a UI tree with exactly one `parentId: null` root (plus any user-created forks) for any reference-style session; copy-style legacy sessions still render.

### Compatibility and risks

#### Session persistence

Mixed-shape sessions (some compactions copy-style, some reference-style) must remain valid forever. Compat read is the primary mechanism. An eager migration is **not** scoped; it would introduce write side-effects during load.

#### `/fork` semantics subtle change

Today, `/fork` to a pre-compaction entry may land on an evicted placeholder (message content is `[compacted]`). Reference-style restores access to the original entry content. Verify no test or caller implicitly depends on the eviction behavior. Expected: none — eviction was intended as memory relief, not as a semantic gate.

#### `buildSessionTree` UI regression risk

Session tree UI renders shift from N trees per session to 1 tree with inline markers. Visual snapshot tests needed. The change is cosmetic improvement, but it is user-visible — add a regression note in the release.

#### `archiveOldIslands` policy rewrite

Largest surface change in this feature. Mitigation: parallel-run comparison in Phase C — given a fixture session, run both old policy (on copy-style) and new policy (on an equivalently-shaped reference-style version), assert identical archive output.

#### Memory relief for legacy sessions

`evictOldIslandMessageContent` relieves memory pressure on long-running copy-style sessions. Removing it entirely before all legacy sessions roll over could regress memory footprint for users who never trigger a post-migration compaction. Phase D cleanup is therefore conditional; retained as a legacy helper indefinitely until a separate GC feature handles it.

### Relationship to prior work

- `v0.7.19` surgical fixes (P1-P6) — structurally superseded by `072`; `073` continues the refactor into copy-style retirement.
- `v0.7.20 FEATURE_072` — **direct prerequisite.** Without 072, there are still flat-array compaction consumers that depend on copy-style island semantics; 073 cannot land on the current structure.
- `v0.7.25 FEATURE_051` — untouched. `073` only affects internal `@kodax/agent` lineage code; the frozen visible shell from `051` is not reopened.
- pi-mono `buildSessionContext` — reference target.
- Claude Code `getMessagesAfterCompactBoundary` — inline-marker variant of the same pattern.

### Open questions

1. **Eager migration vs indefinite compat?** Should a one-shot `/migrate-sessions` command rewrite old copy-style sessions into reference-style on user request, or is lazy compat sufficient forever? Leaning lazy.
2. **Does compat read add measurable overhead on session load?** Benchmark before Phase A ships; if overhead is zero, lazy compat is the obvious choice.
3. **When can `evictOldIslandMessageContent` be fully deleted?** Needs a signal that legacy copy-style sessions have rolled over — likely a version-bump criterion, not a time-based one.

## FEATURE_075: Plan Approval Dialog Scroll and Editor Integration

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement
**Builds on**: v0.7.20 `FEATURE_074` (exit_plan_mode tool)
**Reference**: Claude Code `ExitPlanModePermissionRequest.tsx` — fullscreen sticky-footer pattern

### Origin

`FEATURE_074` introduced `exit_plan_mode` as the approval entry point for plan-mode exit. The v0.7.20 implementation renders the plan inside the existing yellow-bordered confirm dialog ([DialogSurface.tsx](../../packages/repl/src/ui/components/DialogSurface.tsx)), which has no scroll handling and no height cap. Long plans exceed terminal height in InkREPL and get clipped off the top. v0.7.20 ships a **head + tail truncation** (12 lines + notice + 2 lines) as a pragmatic fallback — the user sees initial steps and final verdict but loses the middle.

Readline REPL is unaffected because native terminal scroll handles long output naturally.

### Goals

1. **Scrollable plan area** in InkREPL. Arrow keys / Page Up / Page Down scroll the plan inside the dialog while approval buttons stay pinned.
2. **External editor integration**: Ctrl+G (or equivalent) opens the full plan in `$EDITOR` for users who want to read/search the whole thing or paste into their notes.
3. **Markdown rendering** of the plan (optional, check dependency cost). Plans frequently use bullet lists, headings, code blocks — raw-text rendering is usable but not pleasant.
4. **Retire the head+tail truncation** from `buildToolConfirmationDisplay` — truncation becomes unnecessary when scroll works.

### Non-goals

- Not rebuilding the generic confirm dialog. Scroll applies only to `exit_plan_mode` (and any future tool with similarly long content).
- Not matching every Claude Code approval option (Yes + auto-mode, Yes + bypass permissions, etc.). KodaX's mode model is simpler; keep it simple.
- Not adding plan editing mid-approval (Claude Code supports editing the plan before accepting). Revision flow stays "reject → LLM revises".

### Design direction

- Follow Claude Code's ScrollBox + FullscreenLayout bottom-slot pattern ([reference](c:/Works/claudecode/src/components/permissions/ExitPlanModePermissionRequest/ExitPlanModePermissionRequest.tsx)). Evaluate whether KodaX's existing fullscreen substrate (FEATURE_051 / FEATURE_057) already provides the right primitives, or whether a standalone component is cleaner.
- External editor: spawn `$EDITOR` with a temp file containing the plan; resume dialog on editor exit. Keep readline parity — readline users already get native scroll, so the editor integration primarily benefits InkREPL.
- Markdown: if FEATURE_058 (Transcript Native Scrollback Dump) lands a markdown renderer first, reuse. Otherwise defer to a separate iteration.

### Acceptance criteria

- [ ] A 100-line plan in InkREPL is fully navigable via arrow / PageUp / PageDown without approval buttons scrolling off-screen.
- [ ] `Ctrl+G` opens the plan in `$EDITOR` (configurable); editor exit returns control to the dialog with approval state preserved.
- [ ] `buildToolConfirmationDisplay.truncatePlanForDisplay` is removed once scroll is live.
- [ ] No regression for short plans — they still render inline without chrome overhead.
