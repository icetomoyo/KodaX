# v0.7.45 Feature Designs

> 中文标题：`v0.7.45 Feature 设计`
>
> Version: `v0.7.45`
>
> Status: Planned (not yet released)
>
> Theme: Adaptive Multi-Provider Orchestration — 把 KodaX 从"provider 兼容层"升级为"可观测、可验证、可配置、逐步自适应的 orchestration runtime"。核心思想是利用不同 provider/model 的真实能力差异（context 长度、代码能力、tool-call 稳定性、视觉、价格、延迟、语言风格），在 agent / coding agent 工作流的不同阶段做分工，关键节点做并行 fan-out 并用客观信号裁决，但**严格不做**实时 LLM router、task 内跨 provider 接力、全程多模型投票。第一优先级是 telemetry，所有后续优化都从真实数据里长出来。

---

## 总览

KodaX 已支持 11 家 provider（Anthropic / OpenAI / Google / Zhipu / Kimi / MiniMax / DeepSeek / 等），现状是**用户在配置层决定一次"用哪个"，task 全程单 provider 单 model 直跑**。这种"兼容层"用法没有利用各 provider/model 的真实能力差异：长上下文 vs 短上下文、强代码 vs 弱代码、tool-call 稳定 vs 不稳定、视觉 vs 纯文本、便宜快 vs 贵慢、推理强 vs 通用。

愿景：让 KodaX 在 agent / coding agent 工作流的不同**阶段**用不同 provider/model 来做事，关键节点（review / diagnosis / 多策略生成）做并行 fan-out，并用客观信号（测试 / typecheck / lint / build / patch apply / 运行结果）裁决。终态是一个"理解什么时候该用谁"的 orchestration runtime，而不是"把请求转发给用户选定 provider"的 compatibility shim。

但**愿景不等于一上来就做超级大脑**。本 feature 的核心是**正确的实施顺序**：先 telemetry，再 fan-out + 客观裁决，再 stage-level routing，再 fallback，最后才谈 data-driven adaptive。任何颠倒顺序的尝试（先做 LLM router、先全程并行投票、先做 capability profile 不打 trace）都会落入"看起来很聪明、实际很难控"的陷阱。

## FEATURE_102: Adaptive Multi-Provider Orchestration Runtime

**Status**: Planned
**Priority**: High
**Category**: Internal / Core

### 背景

#### 现状

`packages/ai/` 已抽象出多 provider 调用层；`packages/coding/` 的 task-engine、Scout / Generator / Evaluator / Planner 等 subagent 都通过统一接口调用 LLM；FEATURE_080-085 (v0.7.23-v0.7.26) 引入了 `Agent` / `Handoff` / `Runner` / `Guardrail` / `Tracer` / `Span` 等 Layer A primitive；FEATURE_100 (v0.7.29) 把 SA / AMA 双底座统一到单一 substrate executor。这意味着 v0.7.30 之后，KodaX 已经具备**让不同 stage 用不同 provider/model 的结构基础**（Agent declaration 上的 model slot、Runner-level guardrail / tracer 注册栈），但这块能力**还没有被组织化使用**。

#### 不同 provider/model 的真实能力差异

来自实战经验和厂商文档的常识级差异（不依赖 marketing claim，需要 telemetry 验证）：

| 维度 | 强项典型 | 弱项典型 |
|---|---|---|
| 长上下文 | Gemini / Qwen-Long / Kimi | 大多数 200k 以下模型 |
| 代码生成 | Sonnet / Opus / DeepSeek / GLM-Coding / Kimi-Code | 通用聊天模型 |
| 代码审查盲区互补 | 不同家族独立训练 → 错位错误 | 同家族同时漏过同一类问题 |
| Tool-call 稳定性 | Anthropic / OpenAI 成熟生态 | 部分国产 provider JSON 偶尔不闭合 / 工具协议偶发偏差 |
| Vision / Multimodal | Anthropic Vision / GPT-4V / Gemini Vision | 纯文本模型 |
| 大批量分类 / 摘要 | DeepSeek / Haiku / 便宜快速档 | 用 Opus 做 grep 筛选是浪费 |
| 中文表达 / 本土语境 | Zhipu / Kimi / MiniMax / Qwen | 部分英文为主模型在长中文输出上偏弱 |

KodaX 当前**完全不利用**这些差异；用户选 Sonnet 就全程 Sonnet，选 DeepSeek 就全程 DeepSeek。

#### Coding agent 比通用聊天更适合多模型编排的根本原因

通用聊天的多模型应用通常退化为"投票"或"风格融合"，质量提升不明显。但 coding agent 不一样：**它有客观裁决信号**——

- patch 是否能 apply
- tests 是否通过
- typecheck 是否通过
- lint 是否干净
- build 是否成功
- 运行结果是否符合预期
- review finding 是否能被复现

这些信号让多模型系统从"谁说得更像真的"变成"谁的结果经得起现实验证"。这是**多模型编排在 coding 域比聊天域更值得做**的根本原因。

### 核心判断（不能违反的设计原则）

1. **多模型收益主要来自"分工"和"盲区互补"，不是来自全程投票。** 不同模型的优势必须通过 capability tag / stage routing 显式使用，而不是让模型"自由发挥"或多家齐发投票。
2. **Coding agent 的 judge 优先使用客观信号，而不是另一个 LLM。** 测试 / typecheck / lint / build / patch apply / 运行结果 / finding 可复现性 排在 LLM judge 之前。
3. **第一优先级是 telemetry / trace。** 没有真实数据之前，任何"这个阶段用某模型更好"的判断都只是经验或占卜。Capability profile 必须能被 telemetry 修正，否则 tag 只是主观印象。
4. **Provider 切换必须发生在 stage boundary（结构化中间产物边界），不在连续 tool loop 中间。** 切换点不是时间边界，而是 artifact 边界（repo summary / plan / diff / test output / review findings / diagnosis report / task state JSON）。**单 task 内跨 provider 反复接力会让 prompt cache 命中归零、repo 上下文重 tokenize、tool-call 协议反复翻译，成本可能直接抵消"用便宜模型省钱"的收益。**
5. **不传完整聊天历史，传结构化中间产物。** stage 之间的交接走干净的 typed payload，不要试图把整段 conversation history 在 provider 间搬运。
6. **弱模型只做"可丢弃、可验证、可被纠正"的辅助产物。** grep 筛选、批量分类、日志摘要、候选文件排序、生成待验证假设可以用便宜模型；但**不能产出关键的不可验证决策（如架构 plan）后让强模型盲目执行**——链条强度等于最弱一环，弱模型的 plan 缺陷会被强模型快速放大成更多 buggy 代码。
7. **短期不做 LLM router。** 实时 LLM router 会带来：额外延迟、额外成本、新的错误来源、不可复现的调度行为、难以解释"为什么这次选了这个模型"。初期一律用：静态规则 + capability flag + 用户可覆盖 + telemetry 记录 + 明确 fallback。等数据足够后，再谈 scoring / bandit / adaptive。

### 实施路线（4 个 Phase，顺序硬性不可调换）

#### Phase 0 — Instrumentation / Trace（基础设施）

**目标**：不是立刻变聪明，而是让未来所有优化都"可证明、可回放、可比较"。

**记录字段（建议最小集）**：
- `task_id` / `trace_id` / `parent_trace_id`
- `stage` / `subagent_role`（scout / planner / generator / evaluator / reviewer / classifier / 等）
- `provider` / `model` / `capability_request`
- `prompt_tokens` / `completion_tokens` / `cache_hit` / `cache_write`
- `latency_ms` / `cost_usd`
- `tool_call_total` / `tool_call_failed` / `tool_call_failure_reason`
- `fallback_count` / `retry_count`
- `triggered_rework`（是否触发返工）
- `objective_signals`：测试 / lint / typecheck / build 的原始结果（pass/fail/n_a）
- `user_outcome_event`：accept / partial_accept / reject / revert / cherry_pick / manual_edit_before_commit

**实操要点**：
- **存储**：单用户 CLI 场景，本地 SQLite 或 JSONL 即可，**不需要 OTel** 这套企业级基础设施。建议方向：复用现有 `@kodax/tracing` 的 `Tracer` / `Span` 抽象（FEATURE_083 v0.7.24 已落地），新增本地 `TracingProcessor` 子类做持久化。
- **注入层**：调用层强制打基础字段（provider/model/tokens/latency/tool_result），subagent 提供 `stage` tag。这是兼顾一致性（基础字段不漏）和业务语义（stage 由调用方提供）的中间方案。
- **只记原始事件，不急着定义 derived metric**。"成功 / 失败 / 返工"这些信号噪声大（patch 过测但下周被 revert 算什么？用户 cherry-pick 了 60% 算什么？AI 输出后用户手改再 commit AI 的"贡献"是什么？），derived metric 等数据攒起来后再定义，避免 reward 设计错误污染未来 adaptive 决策。
- **隐私 / 本地优先**：trace 数据完全 local-only，默认不上报；提供配置开关（默认开，可关闭）。

**验收标准（Phase 0 起步版）**：
- [ ] 所有 subagent / LLM 调用都被 trace 记录，可通过本地存储查询
- [ ] Trace schema 文档化，且支持后续追加字段而不破坏旧数据
- [ ] 至少能回答两个问题：(1) 各 stage 实际使用了哪些 model；(2) tool-call 在不同 provider 上的失败率分布
- [ ] 配置中提供 telemetry 开关（默认开，可关闭）

#### Phase 1 — Review Fan-out + Objective Arbiter（最高确定性收益）

**目标**：在 review / critique / test failure diagnosis 等"分歧有价值"的节点做并行 fan-out，用客观信号裁决可验证项，用 judge model 合并不可验证项。

**Reviewer 输出结构化（必须）**：
```typescript
interface ReviewerFinding {
  type: 'bug' | 'security' | 'style' | 'perf' | 'design' | 'doc';
  severity: 'critical' | 'high' | 'medium' | 'low';
  file: string;
  line?: number;
  evidence: string;
  verifiable: boolean;        // 是否可被客观信号验证
  verification_command?: string; // 建议的验证命令（npm test / tsc --noEmit / 等）
  confidence: number;         // 0-1
}
```

**裁决机制**：
- 可验证 finding（`verifiable: true`） → 跑 `verification_command`，用真实结果裁决（true positive 保留，false positive 丢弃）
- 不可验证 finding → 主模型或 judge model 合并；冲突时倾向最小风险方案，或显式上抛给用户
- **不做"多模型投票决定代码对错"——用现实裁决，不用民主裁决**

**配置示例**：
```yaml
review:
  parallel: true
  models:
    - { provider: anthropic, model: sonnet-4-6, role: code-reviewer }
    - { provider: openai, model: gpt-5-codex, role: security-reviewer }
    - { provider: deepseek, model: v4-coder, role: cheap-second-opinion }
  arbiter:
    verifiable: objective-signals
    unverifiable: judge-model
```

#### Phase 2 — Stage-level Capability Routing（约束版）

**目标**：仅做收益明确的 5 类阶段级路由，不做全自动调度。

| Capability | 路由目标（示例） | 适用场景 |
|---|---|---|
| `long_context_summary` | 长上下文模型（Gemini / Qwen-Long / Kimi） | 跨多文件 / 跨整个 repo 摘要 |
| `vision` | vision-capable 模型 | 截图诊断 / UI diff / 图片任务 |
| `cheap_classify` | 便宜快速档（DeepSeek / Haiku） | grep 结果筛选 / 候选文件排序 / 大批量分类 |
| `main_coding_loop` | 稳定的强代码模型（Sonnet / Opus / 用户配置主模型） | 实际改代码 / Plan / Generate |
| `review` | 多个不同家族 reviewer 并行 | review / critique / diagnosis |

**Capability profile 要点**：
- **初期人工 seed**，但**必须能被 telemetry 修正**——厂商 marketing claim 不可信，最终要靠真实数据
- **早期数据稀疏问题**：(provider × model × stage × task_type) 笛卡尔积下每格能有几条数据？早期肯定不够做频率主义判断。**Phase 2 阶段**保持人工 seed + 用户 override，**不引入**自动 profile 修正；自动修正留给 Phase 4。
- **稀疏统计的应对方向**（Phase 4 才用）：Bayesian prior + shrinkage（无数据时回落到默认主模型），不要直接频率主义。

**Stage 边界判定问题**：
tool loop 可能在 mid-execution 触发 re-plan（Evaluator revise / replan）。需要明确：
- 哪些事件算 "stage 切换"（Scout → Planner / Planner → Generator / Generator → Evaluator / Evaluator verdict → Generator revise / 等）
- 哪些事件**不算**（同一 Generator 内的多轮 tool call）
- 切换时传递的"结构化中间产物"是什么（plan diff / contract / verdict / 等具体 JSON shape）

#### Phase 3 — Fallback / Health Checks（用户感知强、风险低）

**触发条件**：
- Provider 限流 / 超时 / 5xx → 自动降级到次选
- Tool-call 反复失败（连续 N 次 JSON 闭合错误等）→ 切换到 tool-call 更稳定的 provider
- Context overflow（接近 model context window 上限） → 切换到长上下文模型
- 质量异常（连续 N 次输出空 / 拒答 / 偏离任务） → 切换或上抛

**关键约束**：
- 切换时**保留结构化 artifact**，不重放完整 history（呼应原则 5）
- 切换必须有 trace 记录（呼应 Phase 0），让用户事后能追溯"为什么这次最终用了 model X 而不是配置的 model Y"
- 默认开启 fallback，但允许用户禁用（`fallback: disabled` / `fallback: warn-only`）

#### Phase 4 — Data-driven Adaptive（基于 Phase 0 数据）

**前提**：Phase 0 telemetry 已积累至少 N 个 task / N 周数据（具体阈值实施时根据真实使用量定）。

**回答的问题（举例）**：
- Plan 阶段用更贵模型是否真的提高成功率？还是 5x 价格只换来 1.1x 质量？
- Review fan-out 多加一个 reviewer 是否还有边际收益？
- 哪些 task_type 适合 cheap model？哪些必须用主模型？
- 哪些 provider 在 KodaX 的实际 tool 协议下失败率显著高于平均？
- 哪些模型在 H2 长任务下导致返工次数显著高于 H1 短任务？

**引入的机制**：
- **Scoring**：基于 cost / latency / objective signal pass rate / rework rate 的多维评分
- **Contextual bandit**：在 capability profile 候选模型间根据上下文特征做软选择
- **Bayesian shrinkage**：应对稀疏单元格，无足够数据时回落到默认
- **成本-质量动态权衡**：暴露 `budget` 配置（economy / balanced / quality），不同档位走不同评分曲线

**Phase 4 不会做的事**：
- ❌ LLM router 实时决策（多一层延迟 + 不可复现 + 不可解释，留在 "明确不做" 列表里）
- ❌ "智能"自动 capability profile 重写（必须有可解释的 scoring 公式 + 用户可见可覆盖）

### 配置接口（草案，后续 Phase 实施时再细化）

```yaml
orchestration:
  mode: adaptive            # 早期实际是 static_rules + telemetry，Phase 4 后才真正 adaptive
  budget: balanced          # economy / balanced / quality
  telemetry:
    enabled: true
    storage: sqlite         # sqlite / jsonl
    redact_user_input: false  # 是否在落盘前 redact 用户内容
  coding:
    planning: strong-reasoning
    exploration: fast-cheap
    implementation: code-specialist
    review:
      parallel: true
      models:
        - code-reviewer
        - security-reviewer
    fallback:
      on_tool_error: stable-tool-model
      on_context_overflow: long-context-model
      on_provider_5xx: secondary
```

`capability` 别名（`strong-reasoning` / `fast-cheap` / `code-specialist` / `stable-tool-model` / `long-context-model` / 等）解析到具体 `provider:model` 由 capability profile 决定，profile 文件位置和 schema 在 Phase 2 实施时定义。

### 明确不做（避免被未来误解）

- ❌ Task 内频繁跨 provider 接力（prompt cache 归零 + tool-call 协议翻译 + 上下文重 tokenize）
- ❌ LLM router 实时决定每一步（额外延迟 / 不可复现 / 不可解释 / 新增错误源）
- ❌ 全程多模型投票（贵 / 慢 / 噪声大 / 收益不确定）
- ❌ 弱模型产出关键不可验证 plan，强模型盲目执行（链条强度等于最弱一环）
- ❌ 没有 telemetry 就声称 adaptive（Phase 0 是后续所有 Phase 的硬前置）
- ❌ 把多 provider 编排做成全局智能大脑（应该从可证明有收益的局部增强长出来）

### 风险与待解决问题（实施时需正面回答）

1. **结构化中间产物 vs 工作记忆 narrative**：tool 调用历史中"为什么这条路走不通"的 reasoning trace 不是干净 artifact，但常被下一阶段需要。需要设计 "传 artifact + 精简 narrative" 的接口（narrative 怎么精简又是一个 LLM 任务，递归性需要克制）。
2. **Telemetry 信号设计**：`triggered_rework` / `user_outcome_event` 这些信号噪声大，Phase 0 阶段先记原始事件，derived metric 留待数据攒起来后再定义。
3. **稀疏统计问题**：`(provider × model × stage × task_type)` 笛卡尔积下早期数据稀疏，Phase 4 必须使用 Bayesian shrinkage / 软先验，不能直接频率主义。
4. **Stage 边界本身可能是动态的**：tool loop mid-execution 触发 re-plan 时，"什么算 stage 切换"的判定规则需要明确，并与 FEATURE_100 后的 Runner Frame 对齐。
5. **用户体验可解释性**：用户需要知道"为什么这一步用了某个模型 / 为什么多花了钱"。trace 必须可被 user-facing 命令查询（比如 `kodax trace --task <id>`）。
6. **依赖 / 配合的既有 feature**：FEATURE_080-085 (Layer A primitive + Tracer + Span)、FEATURE_083 (Tracing 子系统)、FEATURE_085 (ToolGuardrail runtime)、FEATURE_100 (SA Runner Frame Adoption) 是本 feature 的结构基础，本 feature 不重做，只在它们之上组织化使用。

### 一句话总结

KodaX 的多 provider 编排应该从"可证明有收益的局部增强"长出来，而不是一开始设计成全局智能大脑。**先 instrument，再在 review / diagnosis 这类高收益节点 fan-out 用客观信号裁决，然后做 stage-level capability routing，最后基于真实数据走向 adaptive。** 这样既保留愿景，又避开跨 provider 接力 / LLM router / 全程投票这些高复杂度低确定性的坑。
