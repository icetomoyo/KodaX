## FEATURE_061: Scout-First AMA Architecture Simplification

**Status**: Planned
**Priority**: High
**Category**: Refactor

### Overview

`FEATURE_061` simplifies the AMA managed task architecture from a multi-layer routing pipeline into a Scout-centric model where Scout is both the decision-maker and the first worker.

The current AMA pipeline has accumulated significant complexity:

- 4-5 pre-Scout decision layers (Intent Gate → LLM routing → Review Floor → Guardrails → Bypass check)
- Scout as a pure router that doesn't do actual work
- context breaks between every role transition (Scout → Generator → Evaluator each start from scratch)
- subagent parallelism locked behind 3 hardcoded Tactical Flow variants
- H0/H1/H2 naming with a state machine for promotion/demotion/guardrails

This violates KodaX's core philosophy: **极致轻量化 — minimalist and intelligent**.

### Motivation

Anthropic's [harness design article](https://www.anthropic.com/engineering/harness-design-long-running-apps) establishes a key principle:

> "Every component in a harness encodes an assumption about what the model can't do on its own."

With Opus 4.6, many assumptions encoded in the current AMA pipeline no longer hold:

- models can self-route reliably without a separate routing pass before Scout
- models can produce consistent structured output without 6-field cross-validation
- models don't need context resets between roles (the article explicitly found this with Opus 4.6)
- a 3-agent pattern (Planner → Generator → Evaluator) is sufficient for most long-running tasks

### Goals

1. Make Scout the single AMA entry point after Intent Gate — no pre-routing LLM call, no review floor, no bypass check.
2. Let Scout both judge complexity and do actual work (H0 tasks are completed by Scout directly).
3. Preserve context across role transitions: Scout upgrades into Generator or Planner rather than handing off to a fresh agent.
4. Give every core role (Scout/Planner/Generator/Evaluator) the ability to spawn subagents for parallel work.
5. Simplify the harness model from named profiles with a state machine to a pure worker-chain composition.
6. Maintain the Generator/Evaluator separation (proven valuable for independent verification).

### Non-goals

- changing user-visible mode semantics (SA / AMA remain the only user choices)
- removing Intent Gate (it's zero-cost regex, it stays)
- removing repo routing signals (Scout still needs them)
- making Evaluator inherit Generator's full context (Evaluator must stay independent)
- reviving visible team mode or worker topology UI

### Target architecture

```
User prompt
    │
    ▼
┌───────────────┐
│  Intent Gate  │  regex/rules, zero cost
│  (sync)       │  SA → runDirectKodaX, done
│               │  AMA → read repo signals, enter Scout
└───────────────┘
    │ AMA
    ▼
╔═══════════════════════════════════════════════════════╗
║  SCOUT (single LLM session, sole AMA entry point)    ║
║                                                       ║
║  Capabilities:                                        ║
║  1. Assess task complexity and harness needs          ║
║  2. H0: complete task directly (may spawn subagents)  ║
║  3. H1/H2: produce handoff directive and upgrade      ║
╚═══════════════════════════════════════════════════════╝
    │
    ├── H0: Scout completes directly → return result
    │
    ├── H1: Scout upgrades to Generator
    │   │
    │   ▼
    │   Generator (inherits Scout context, may spawn subagents)
    │   │
    │   ▼
    │   Evaluator (independent context: task + deliverable only)
    │     ├── accept → output
    │     ├── revise → Generator fixes (1x) → re-evaluate
    │     │     ├── accept → output
    │     │     └── not ok → escalate to H2
    │     └── escalate → escalate to H2
    │
    └── H2: Scout upgrades to Planner
        │
        ▼
        Planner (inherits Scout context, may spawn subagents)
        → plan self-check (1x self-correction opportunity)
        │
        ▼
        Generator (new context: task + plan artifact, may spawn subagents)
        │
        ▼
        Evaluator (independent context: task + plan + deliverable)
          ├── accept → output
          ├── revise → Generator fixes (1x) → re-evaluate
          └── replan → Planner replans (1x) → re-execute
```

### Context passing rules

| From → To | Method | Reason |
|-----------|--------|--------|
| Scout → Generator (H1) | same session continues | Scout already read diff/code, don't lose context |
| Scout → Planner (H2) | same session continues | same reason |
| Planner → Generator | new session + plan artifact | Generator only needs the plan, not Planner's reasoning |
| Generator → Evaluator | new session + task + deliverable | Evaluator needs independent perspective |
| Evaluator → Generator (revise) | resume Generator session + feedback | Generator needs prior context to fix targeted issues |
| Evaluator → Planner (replan) | resume Planner session + feedback | Planner needs prior context to revise plan |

### Subagent model

Every core role (Scout/Planner/Generator/Evaluator) may spawn subagents:

- subagents are assistants that return results to the parent role
- subagents do not make harness decisions (no escalation, no evaluation)
- subagents receive scoped context (only the sub-task, not the full conversation)
- the decision to spawn subagents is made by the role itself, not hardcoded admission rules
- subagents run in parallel via `runOrchestration` and consolidate before the role produces its output

### H1 retry and escalation policy

- Evaluator may `revise` once: Generator gets specific feedback, fixes, resubmits
- If second evaluation still fails: escalate to H2 (add Planner), do not retry further at H1
- Evaluator may `escalate` directly on first pass if the problem clearly needs planning

Rationale: if H1 can't solve it in 2 attempts, the issue is task understanding, not execution quality. Adding a Planner is more productive than retrying.

### H2 retry and replan policy

- Evaluator distinguishes `revise` (execution issue) from `replan` (plan issue)
- `revise` → Generator fixes (1x) → re-evaluate
- `replan` → Planner revises plan (1x) → Generator re-executes → Evaluator re-evaluates
- maximum 1 replan cycle to prevent infinite loops
- Planner includes a self-check step before handing off to Generator (1x self-correction)

Rationale: most plan issues only surface during execution. The self-check catches obvious problems; the replan path catches deeper ones.

### What gets deleted

| Component | Reason |
|-----------|--------|
| `createReasoningPlan` LLM routing call | Scout handles routing |
| `applyCurrentDiffReviewRoutingFloor` | Scout assesses review scope directly |
| `resolveManagedHarnessGuardrail` + guardrail system | folded into Scout's judgment |
| `shouldBypassScoutForManagedH0` | all AMA goes through Scout |
| `runTacticalReviewFlow` / `runTacticalInvestigationFlow` / `runTacticalLookupFlow` | replaced by role-level subagent capability |
| `validateScoutDirectiveConsistency` + 6-field cross-validation | Scout does work directly, no complex structured output to validate |
| H0/H1/H2 naming + promotion/demotion state machine | replaced by worker-chain composition |
| budget zone partitioning + per-role iteration limits | simplified to total cap + round limit |

### What gets preserved

| Component | Reason |
|-----------|--------|
| Intent Gate (regex) | zero cost, filters trivial tasks |
| repo routing signals | Scout needs repo context |
| Generator/Evaluator separation | self-evaluation is unreliable (proven by Anthropic article) |
| artifact-backed handoff (FEATURE_046) | proven value, keeps evidence flow trustworthy |
| `runOrchestration` infrastructure | needed for subagent parallelism |
| `runDirectKodaX` worker execution | core LLM call mechanism |

### Relationship to other features

- `FEATURE_059` (Managed Task Structured Protocol V2): protocol design should be finalized after this architecture lands; the dual-track model simplifies when Scout does actual work
- `FEATURE_052` (Dual-Profile AMA Harness): design thesis preserved (users see SA/AMA only); internal Tactical/Managed boundary simplified
- `FEATURE_046` (AMA Handoff Integrity): artifact-backed handoff preserved but handoff paths simplified
- `FEATURE_047` (Invisible Adaptive Parallelism): fan-out capability moves from Tactical Flows to role-level subagents

### Acceptance criteria

- [ ] AMA entry goes directly from Intent Gate to Scout with no intermediate LLM routing call
- [ ] Scout can complete H0 tasks end-to-end (both judgment and execution)
- [ ] H1 flow preserves Scout context into Generator phase
- [ ] H2 flow preserves Scout context into Planner phase
- [ ] Evaluator receives independent context (task + deliverable only, no Generator reasoning)
- [ ] every core role can spawn subagents for parallel work
- [ ] H1 revise limited to 1 attempt, then escalates to H2
- [ ] H2 supports both `revise` (execution) and `replan` (plan) feedback paths
- [ ] 3 Tactical Flow functions consolidated into role-level subagent capability
- [ ] existing eval suite passes with no regression
- [ ] total pre-Scout routing code reduced by >60%

### Phase 1: Remove Pre-Scout Routing Layers

**Goal**: Scout becomes the direct AMA entry point after Intent Gate. No LLM routing call, no bypass check, no harness guardrails before Scout.

**Risk**: Low-Medium (mostly deleting code, but tests need updating)

#### Step 1A: Remove LLM routing from `createReasoningPlan`

**File**: `reasoning.ts`

**What to change**:
- Function `createReasoningPlan` (line 1261): currently has 3 branches:
  1. `!intentGate.shouldUseModelRouter` → uses `buildFallbackRoutingDecision` (heuristic, no LLM)
  2. `mode === 'auto'` → calls `routeTaskWithLLM` (**LLM call — DELETE THIS BRANCH**)
  3. else → uses `buildFallbackRoutingDecision`
- Make ALL paths use `buildFallbackRoutingDecision`. The function signature stays the same.
- Keep `routeTaskWithLLM` function body for now (dead code, clean up later)
- Add routing note: `'Pre-Scout LLM routing disabled (FEATURE_061 Phase 1); Scout is the routing authority.'`

**Why**: Scout will be the sole routing authority. The pre-Scout LLM call is redundant.

**Test impact**: Tests in `task-engine.test.ts` mock `createReasoningPlan` at the import level, so changing its implementation doesn't break mocks. Check `reasoning.test.ts` for any tests calling the real function.

#### Step 1B: Delete `shouldBypassScoutForManagedH0` and its callsite

**File**: `task-engine.ts`

**What to delete**:
- Function `shouldBypassScoutForManagedH0` (line 869-899): determines if a task should skip Scout entirely
- Its callsite in `runManagedTask` (lines 10892-10919): the `if (shouldBypassScoutForManagedH0(...))` block that returns `runDirectKodaX(...)` directly
- Also delete the `shouldKeepLookupDirect` variable (lines 10892-10897) which only gates the bypass

**Why**: All AMA tasks must go through Scout. No bypass.

**Test impact**: Tests expecting H0 tasks to bypass Scout need updating:
- `'keeps obvious AMA H0 tasks on the direct path without scout'` → change to expect Scout is called
- Any test that asserts `mockRunDirectKodaX` was called exactly once for a simple H0 task

#### Step 1C: Remove `resolveManagedHarnessGuardrail`, simplify `applyManagedHarnessGuardrailsToPlan`

**File**: `task-engine.ts`

**What to delete**:
- Function `resolveManagedHarnessGuardrail` (lines 2157-2188): computes minimum harness based on `needsIndependentQA`, `mutationSurface`, etc.

**What to simplify**:
- Function `applyManagedHarnessGuardrailsToPlan` (lines 2190-2250): currently calls `resolveManagedHarnessGuardrail` and forces harness upgrade. Change to a pass-through that only attaches `reviewTarget` to the decision without enforcing any harness floor.

New implementation:
```typescript
function applyManagedHarnessGuardrailsToPlan(
  plan: ReasoningPlan,
  reviewTarget: ManagedReviewTarget,
): { plan: ReasoningPlan; routingOverrideReason?: string } {
  const decisionWithTarget = cloneRoutingDecisionWithReviewTarget(plan.decision, reviewTarget);
  if (decisionWithTarget === plan.decision) {
    return { plan };
  }
  return {
    plan: {
      ...plan,
      decision: decisionWithTarget,
      amaControllerDecision: buildAmaControllerDecision(decisionWithTarget),
      promptOverlay: buildPromptOverlay(
        decisionWithTarget,
        plan.providerPolicy?.routingNotes,
        plan.providerPolicy,
        buildAmaControllerDecision(decisionWithTarget),
      ),
    },
  };
}
```

**Why**: Harness decisions are now Scout's job, not pre-Scout logic.

**Test impact**:
- `'keeps a minimum of H1 when the task explicitly requires independent verification'` → update: Scout (not guardrails) determines harness based on `needsIndependentQA` context
- `'keeps a minimum of H2 for high-risk system-level overwrite work'` → update: Scout determines harness based on risk context

#### Step 1D: Simplify `applyCurrentDiffReviewRoutingFloor`

**File**: `task-engine.ts`

**What to change**:
- Function `applyCurrentDiffReviewRoutingFloor` (lines 2351-2428): currently computes reviewTarget + reviewScale, then calls `applyManagedHarnessGuardrailsToPlan` to enforce harness floors.
- Simplify: keep `inferReviewTarget()` and `deriveFallbackReviewScale()` (Scout needs this context), but stop enforcing harness floors. Just attach reviewTarget and reviewScale to the decision as informational context.

**Why**: Review target/scale info is useful for Scout's judgment, but shouldn't force pre-Scout harness decisions.

**Test impact**:
- `'allows large current-diff reviews to stay on H0 when Scout provides complete direct-review evidence'` → verify Scout receives reviewScale in context

#### Step 1E: Update tests and verify build

**Tests to update** (in `task-engine.test.ts`):

1. Tests expecting Scout bypass → now expect Scout call
2. Tests expecting guardrail enforcement → now expect Scout receives context and makes decision
3. Verify all `mockRunDirectKodaX` setups include Scout response handling

**Verification**:
```bash

npm test -- --filter packages/coding
```

### Phase 3: Context Continuation Across Role Upgrades

**Goal**: When Scout upgrades to Generator (H1) or Planner (H2), preserve context from the Scout session.

**Risk**: Medium-High (core flow change)

#### Step 3A: Design context transfer mechanism

**Option A — Same session continuation** (recommended):
- Scout and Generator/Planner share a single `runDirectKodaX` session
- Simplest, zero context loss
- Opus 4.6 handles long sessions well (per Anthropic article)

**Option B — Full context transfer** (fallback):
- Scout session ends, key findings serialized into context bundle
- New session with bundle injected

#### Step 3B: Modify `runManagedScoutStage` to optionally continue session

- Function `runManagedScoutStage` (line 9244): add option to keep session open for H1/H2
- Return type extends to include `session` or `continuation` handle

#### Step 3C: Modify H1/H2 execution to accept continued session

- `executeManagedTaskRound` (line 10437): add path to continue an existing session for the first worker

#### Step 3D: Ensure Evaluator gets independent context

- Evaluator always starts a fresh session with: task original text + deliverable artifact

#### Step 3E: Implement revise/replan session resumption

- `revise` → resume Generator session with feedback
- `replan` → resume Planner session with feedback

#### Step 3F: Update tests

- Test context inheritance for H1/H2
- Test Evaluator independence
- Test revise/replan session resumption
