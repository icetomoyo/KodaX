# v0.7.36 Feature Designs

> 中文标题：`v0.7.36 Feature 设计`
>
> Version: `v0.7.36`
>
> Status: Planned (not yet released)
>
> Theme: AMA Harness V2 — 把 H0/H1/H2 三档 + Scout/Planner/Generator/Evaluator 四角色简化为**单一 Worker role + 按需 fork Verification subagent**，对齐工业界（Claude Code / OpenCode / pi-mono / Warp）共识，从结构上解决 FEATURE_097 计划列表在 review 类任务上看不到的根因。

---

## 总览

KodaX 当前 AMA harness 是一条 4 节点流水线：

```
Scout → Planner → Generator → Evaluator
```

由 [`managed-protocol.ts:118-148`](../../packages/coding/src/managed-protocol.ts#L118-L148) 的 `H0_DIRECT / H1_EXECUTE_EVAL / H2_PLAN_EXECUTE_EVAL` 三档枚举驱动。FEATURE_084（v0.7.26）把它从文本协议升级到 Layer A 原语；FEATURE_097（v0.7.34）把 `execution_obligations` 接到 TodoListSurface。

但实战上这个结构有 5 个连锁矛盾：

1. **Plan 被劈成两半**——Scout 的 `execution_obligations` 是真实 plan（TodoStore seed 来自这），Planner 只产出 `success_criteria + constraints`。两个 role 干一件事的两半（[role-prompt.ts:627-669](../../packages/coding/src/task-engine/_internal/managed-task/role-prompt.ts#L627-L669)）。
2. **FEATURE_107 实测证明 Planner 没创造价值**——`benchmark/datasets/h2-plan-execute-boundary/` 跑出 H2-A vs H2-B = 0pp delta，Planner 引入的"信息压缩"假设被否决。
3. **Generator 在 H2 是 dispatcher 不是执行者**——Generator H2 模式下 fan-out 给写子 Agent，自己不写文件。这意味着 H2 实际是 5+ 节点而非 4 节点。Issue 124 显示这层 fanout 触发率偏低。
4. **Evaluator 在管线尾部不在 plan 节点**——当前 Evaluator 只在 Generator 全部跑完后跑一次；用户要求的"每个计划节点根据需要加 evaluator"无法表达。
5. **Replan = 全量 reset**——`runner-driven.ts` 中 `revise + next_harness=H2` 触发 `todoStore.reset()` 清空全部 todo，破坏感强。

更关键的是：**这套结构和 FEATURE_097 的"计划列表可见性"诉求结构性打架**。32c5205（FEATURE_097 P0-followup）用 prompt-only 强制 Scout "EARLY emit"，但 Scout 的 prior 是"verdict = 调研后产物"，prompt 软对抗一直压不住——review 类任务尤其严重（因为 review 天然先调查后定论）。

业界研究（4/4 codebase 一致）显示：

| 维度 | Claude Code | pi-mono | OpenCode | Warp | 共识 |
|---|---|---|---|---|---|
| 角色架构 | 单主线程 + 按需 fork subagent | 单核心 + extension subagent | 单 loop + named-agent 池 | 单主 agent + 类型化 sub-agent | **单 primary + 按需子任务** |
| Plan-first 强制 | 否（system prompt + tool nudge） | 否（用户 `/plan`） | 否（用户选 plan agent） | 否（LLM 自决调 SuggestPlan） | **从不强制** |
| Trivial fast path | 同一 query loop | 同一 runLoop | 同一 runLoop | 同一 send_request | **无 fast path** |
| Per-step LLM-as-judge | 无；末端 verification subagent | 无 | 无 | 无 | **从不存在** |
| Per-step deterministic check | TodoWriteTool 的 structural nudge | beforeToolCall hook | doom-loop 检测 | enum check | **存在但都是确定性** |
| Todo 可见 | TodoWrite → AppState reactive | plan-mode widget | todo.updated bus | UpdateTodos protocol | **实时驱动 UI 是统一答案** |

KodaX 当前 4 角色 + 3 档的设计在工业界**找不到对应物**。最接近的 Claude Code 是单主线程 + 按需 fork verification subagent，而且 verification 不是逐步触发是末端 nudge。

**FEATURE_114 的设计核心**：

```
Worker（单一 role，单一 loop，无阶段）
   + Verification（按需 fork 的 read-only subagent）
   + 1 个 PLANNED harness（trivial 是 plan=1 步的自然形式）
```

净变化：4 role → 2 role，3 harness → 1 harness，2 份 plan 数据 → 1 份（TodoStore 即 canonical plan），分类决策点 → 0。**全是减法**，零新抽象。

## FEATURE_114: AMA Harness V2 — Single Worker Role + On-Demand Verification Subagent

**Status**: Planned
**Priority**: High
**Category**: Internal / Architecture

### 背景

#### 当前 H2 的 5 个结构性矛盾

详见上方"总览"——这里只补一个关键观察：**32c5205 commit message 自己已经写明了 prompt-only fix 的局限**：

> "The model knew the rule existed but had no temporal anchor strong enough to override the 'verdict = outcome' semantic prior."

这是 KodaX 自己的 retrospective：prompt 是软约束，prior 太强就会失败。FEATURE_097 在 review 类任务上的反复回归，本质是**结构上让 prompt 替结构干活**——`H1_EXECUTE_EVAL` 这个名字写着"无 PLAN"，prompt 却要求"假装在 plan 然后立刻执行"，prompt 和 harness 自己打架。

#### v0.7.48 引用的 AHE 论文（arxiv:2604.25850）相关数据

FEATURE_109（v0.7.48）的 background 引用了 AHE 论文实证：**system prompt 单独 -2.3pp，prose 不 transfer 跨 family**。这给当前 prompt-only fix 又一个证据：跨 provider（GLM / Kimi / Claude / GPT）的 prompt 表现会漂移，结构性约束才能跨家族 transfer。

#### 工业界 4 家 codebase 的共识形状

见上方对照表。所有 4 家的核心架构都是：**1 个 primary agent + 按需 fork 子任务 + 实时 todo UI**。Plan / verification 都是工具，不是流水线阶段。

### 设计

#### 顶层结构

```
┌──────────────────────────────────────────────────────┐
│ Worker（单一 primary role）                           │
│                                                      │
│ 单一 system prompt（合并 Scout+Planner+Generator）    │
│                                                      │
│ 单一 runner loop：                                    │
│   while (not done):                                  │
│     llm_response = call_llm(...)                     │
│     for tool_call in llm_response.tool_calls:        │
│       result = execute_tool(tool_call)               │
│       if tool == 'todo_update' and step.evaluator:   │
│         result += run_deterministic_check(evaluator) │
│       if N reads without todo_update:                │
│         result += structural_nudge_reminder()        │
│     done = llm_response.finish == 'stop'             │
│                                                      │
│ 工具集：read/grep/glob/write/edit/multi_edit/bash/   │
│         todo_update/dispatch_child(write|verify)     │
└──────────────────────────────────────────────────────┘

      ↓ Worker 调 dispatch_child(subagent='verification')
        或 runner 末端 nudge（3+ todo 关闭后追加提醒）

┌──────────────────────────────────────────────────────┐
│ Verification（按需 fork 的 read-only subagent）       │
│                                                      │
│ 独立 system prompt（精简、对立验证）                  │
│ 独立 child session（可选独立 model）                  │
│                                                      │
│ 工具集：read/grep/glob/bash(read-only)               │
│   禁：write/edit/multi_edit                          │
│                                                      │
│ 输出：VERDICT: PASS / PARTIAL / FAIL + 证据列表        │
└──────────────────────────────────────────────────────┘
```

#### Plan-first 实现：prompt + nudge，不是 gate

**system prompt（Worker）核心条款**（伪代码）：

```
You are the Worker — KodaX's single primary agent.

For non-trivial tasks (>=2 distinct execution steps, or any task touching
>=2 files/areas/feature threads), your FIRST tool call MUST be
todo_update to commit your plan.

Trivial tasks (single typo fix, single-line edit, single-question lookup,
pure conversational answer) skip todo_update — answer or execute directly.

If you find mid-task that what looked trivial is actually multi-step,
call todo_update at that moment to retrofit the plan.

todo_update items have an optional evaluator hint:
  { content, status, evaluator: 'none' | 'build' | 'test' | 'lint' }
Mark milestone steps with the appropriate deterministic check; leave
routine steps as 'none'.
```

**runner-side structural nudge**（不阻断、只提醒）：

```
After tool_call, if:
  - tool count since last todo_update >= 5
  - and at least one read/grep/glob has fired
  - and no todo_update has fired this session
Append to tool_result:
  "[nudge] You have read N files without committing a plan.
   If this task is non-trivial, call todo_update now."
```

**为什么不做 runner gate**：4/4 业界 codebase 都不做 gate。Claude Code（最规范的）也只用 system prompt + TodoWriteTool 的 nudge。Gate 会破坏 LLM 的自然推理流，且在 trivial 误判上代价大。

#### Per-step evaluator：仅 deterministic，不接 LLM-as-judge

`TodoItem` schema 扩展（`packages/coding/src/task-engine/todo-store.ts`）：

```ts
interface TodoItem {
  id: string;
  content: string;
  status: 'pending' | 'in_progress' | 'completed';
  evaluator?: 'build' | 'test' | 'lint' | 'eval-case:X';  // 可选
}
```

**触发**：runner 看到 `todo_update` 把某个 item 从 `in_progress` → `completed`，且该 item 有 `evaluator`，自动跑对应 deterministic check：

| evaluator 值 | 命令 | 失败处理 |
|---|---|---|
| `'build'` | `npm run build`（项目级，超时 90s） | 失败 → tool result 追加 stderr，标 status='failed'，让 LLM 自决修补 |
| `'test'` | `npm run test --` 或 `npx vitest run <path>`（item 携带 path） | 同上 |
| `'lint'` | `npm run lint` 或 `tsc --noEmit` | 同上 |
| `'eval-case:X'` | 调 FEATURE_104 prompt eval harness 跑指定 case | 仅在 prompt-改动相关的 plan 节点触发 |

**为什么不做 `evaluator: 'judge'` 走 LLM-as-judge**：4/4 业界都不做。Claude Code 的 verification subagent 是末端一次而非 per-step，且它内部主要靠跑 build/test/lint 而非 LLM judging。Per-step LLM-judge 会让每个任务的 LLM call 数翻倍，违反 KodaX `CLAUDE.md` 的 minimalist 原则。

#### Verification 角色：Claude Code 风格的末端验证

仿 Claude Code [`verificationAgent.ts`](file:///c:/Works/claudecode/src/tools/AgentTool/built-in/verificationAgent.ts) 模式：

- **触发条件 A**（结构性 nudge）：runner 检测到 `>=3 完成的 todo` 且没有 verification subagent fork 记录，在最近一次 `todo_update` 的 tool_result 追加：
  ```
  [nudge] You have completed 3+ plan steps. Consider dispatching a
  verification subagent to audit changes before final answer.
  ```
- **触发条件 B**（显式调用）：Worker LLM 主动调 `dispatch_child_task(subagent='verification', prompt=<具体审计要求>)` tool。

**Verification subagent 行为**：

- 独立 system prompt（强调"对立验证、找漏洞、不护短"）
- 工具白名单：`read / grep / glob / bash(read-only)`
- 工具黑名单：`write / edit / multi_edit / shell mutation`
- 输出强制 schema：`{ verdict: 'PASS' | 'PARTIAL' | 'FAIL', evidence: string[], blocking_issues: string[] }`
- 结果作为 user-role message 回 Worker，Worker 自决是否修补 / 继续 / 上交

#### Replan：iterative，不 reset

当前 H2 的 `revise + next_harness=H2` 走 `todoStore.reset()` 全炸。V2 替换为 iterative replan：

| 操作 | 实现 | UI 反映 |
|---|---|---|
| **adjust** | LLM 调 `todo_update` 把当前 in_progress item 改 description / 改 evaluator | 该 item 内容变更，状态保留 |
| **insert** | LLM 调 `todo_update` 在当前 item 后插新 item（带 `inserted_after` 标记） | 新 item 出现在列表中，注 "added mid-task" |
| **drop** | LLM 调 `todo_update` 把某 pending item 标 `cancelled`（schema 加这一态） | 该 item 划线显示 |
| **branch** | LLM 调 `todo_update` 一次性增删多 item | 列表实时刷新 |

**reset 仅作为兜底**：当 Verification 输出 `FAIL` 且 Worker 决定 "推翻重来"，可以显式调一个新工具 `todo_reset(reason)`——不是 evaluator 自动触发。

#### Trivial 路径：自然形式，零额外机制

不写任何 fast path 代码。Trivial 任务的天然形式是：

- LLM 看 system prompt 判断"这是 trivial"
- 直接答 / 直接调一次 tool 结束
- 不调 `todo_update`
- runner loop 在 `finish: 'stop'` 时退出

UI 表现：TodoListSurface 在没有 todo 时**完全不渲染**（既有行为）。trivial 任务用户看不到任何 plan UI，符合预期。

### 验收标准

1. **V2 单 role 跑通基本三类任务**：trivial answer / 多步实现 / review/audit。三类都能正常 finish，trivial 不调 `todo_update`，多步任务首个 tool call 是 `todo_update`，review 任务能在前 1-2 turn emit 计划列表（不再后置）。
2. **FEATURE_097 在 V2 下结构稳定**：跑 [`bench/datasets/`](../../benchmark/datasets/) 中所有 review 类 case，TodoListSurface 在前 2 turn 内出现的比率 ≥ 95%（V1 当前数据约 60-70%）。
3. **Per-step evaluator 触发正确**：todo item 标 `evaluator: 'build'`，状态变 completed 时 runner 自动跑 `npm run build`；失败时 tool result 含 stderr，LLM 能在下一轮基于此修补。
4. **Verification subagent 末端 nudge 工作**：Worker 完成 3+ todo 后，下一次 `todo_update` 的 tool result 含 verification 提醒；Worker 调 `dispatch_child_task(subagent='verification')` 后能拿到 PASS/PARTIAL/FAIL verdict。
5. **Replan iterative 不丢数据**：mid-task 调 `todo_update` 增删 step，TodoListSurface 增量更新（不全量 reset），UI 注明 "added mid-task" / "cancelled"。
6. **Feature flag 双轨**：`KODAX_HARNESS_V2=false` 完全等价于当前 v0.7.34/35 行为（所有现有 test 必须绿）；`KODAX_HARNESS_V2=true` 走 V2 全新路径。
7. **Prompt eval 三类 baseline**：`tests/harness-v2-baseline.eval.ts` 在 FEATURE_104 alias 表 8 alias 上 mean ≥ 80% 通过率（与 FEATURE_106/111 同一阈值）。
8. **跨 family 验证**（呼应 FEATURE_109 AHE 数据）：V2 在 GLM-Coding / Kimi-Code / Claude / GPT-4o 四家上 baseline eval 通过率 max-min spread ≤ 15pp。这是结构性约束应有的 cross-family transfer 表现。
9. **无回归**：`npm run test` / `npm run build` 全绿；既有 H0/H1/H2 路径在 flag=false 时表现不变。

### 非目标

- **不** 引入 per-step LLM-as-judge（只接 deterministic build/test/lint）
- **不** 在 runner 层强制 plan-first（不做工具白名单 gate；用 prompt + nudge）
- **不** 为 trivial 任务做 fast path 代码分支（自然形式由 LLM 自决）
- **不** 引入 PauseToken / 状态机 / 新协议层（V2 只是把 4 角色合并 + 增 evaluator hint 字段）
- **不** 与 FEATURE_111 soft-pause 互斥（两者并存且 V2 下 soft-pause 更好用）
- **不** 与 FEATURE_092 auto-mode 互斥（guardrail 是工具层、V2 是 role 层）
- **不** 在 v0.7.36 删任何旧 role prompt 文件（直到 v0.7.45 才删）
- **不** 持久化 TodoStore（V2 暂不引入，留在 FEATURE_113 v0.8.2 一起做）

### 与其他 feature 的关系

- **FEATURE_097**（v0.7.34，realtime todo list）：V2 是 097 的**结构性根因修复**——097 的 review 任务可见性问题在 V2 下结构上不再可能发生（无 H1 这一档，Scout/Planner/Generator 三角色合并消除了 emit 时机歧义）。
- **FEATURE_110**（v0.7.34，删旧 plan-mode）：FEATURE_114 是 110 简化路线的延续——110 删 path 1，114 删 H1 这一档。
