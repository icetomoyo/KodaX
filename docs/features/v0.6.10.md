## FEATURE_024: Project Harness - Action-Level Verified Execution

**Status**: Completed | **Priority**: High | **Category**: Enhancement | **Created**: 2026-03-18

### 1. Overview

Project Mode already has strong project artifacts, deterministic planning, and quality reporting, but it still lacks an action-level verifier in the execution loop. The current gap is not better prompting; it is the absence of a command-layer judge that can decide whether a proposed or completed action is legal, relevant, and sufficient to count as real progress.

This feature adds a Phase 1 `Project Harness` that wraps `/project next` and `/project auto` with deterministic verification. The harness becomes the first step toward proof-carrying completion for Project Mode.

### 1.1 Current Implementation Status

Implemented in the current slice:

- `.agent/project/` runtime artifact migration with legacy `.kodax` read compatibility
- protected-artifact blocking for `feature_list.json` and harness-owned files, including shell/interpreter write attempts
- verifier-gated `/project next` and `/project auto`
- `/project verify` with current-workspace deterministic re-verification
- harness run/evidence persistence and manual-override evidence
- proof-carrying completion report parsing plus stricter evidence completeness checks
- current-workspace rule-source discovery plus config refresh when project docs or scripts change
- declarative invariant compilation for test evidence, doc evidence, and workspace dependency checks
- active-feature relevance checks, unrelated-diff detection, and repeated-failure escalation
- structured failure codes and deterministic retry feedback
- lightweight checkpoint metadata and session-tree node capture for each persisted harness run
- feature-step and session-plan checklist coverage gates for proof-carrying completion
- declarative exception allowlists and custom repair-playbook policy in generated harness config

Still pending for later slices:

- richer architecture-boundary rules beyond the current keyword-based invariant heuristics
- full action-trace replay verification instead of deterministic workspace re-checks
- richer critic/refiner loops beyond declarative playbooks and one retry
- automatic checkpoint/rollback orchestration and branching harness search from later roadmap phases

### 1.2 KodaX-Practice Completion Boundary

Must complete for this feature under KodaX's minimal-but-smart philosophy:

- one more high-value deterministic architecture invariant
- stable `/project next`, `/project auto`, and `/project verify` semantics that no longer depend on agent self-declared completion
- clear feature documentation and completion criteria aligned with proof-carrying execution

Good to complete when it stays declarative and low-complexity:

- declarative refiner behavior
- repair playbook management
- more explicit evaluator output
- finer repo-specific exception and checklist synthesis

Not required for completing this feature in KodaX style:

- action-trace replay verification
- automatic rollback orchestration
- branch search or candidate-search orchestration
- model-generated executable harness code

### 2. Product Decisions

1. Phase 1 targets `harness-as-action-verifier`, not `harness-as-policy`
2. Only `/project next` and `/project auto` get strong completion gating in this phase
3. Only one new Project command is added: `/project verify [#index|--last]`
4. Successful execution remains automatic by default; verification and evidence collection should not require routine manual confirmation
5. Automatic self-repair is limited to one retry after a retryable verification failure
6. `feature_list.json` and `PROGRESS.md` remain repository-root project truth files
7. Project runtime artifacts move to `.agent/project/`
8. `.kodax/` remains reserved for KodaX control-plane files and should no longer absorb Project runtime churn
9. Project Harness reads rules from the current workspace project, not from KodaX control-plane files

### 3. Artifact Layout

#### 3.1 Project truth in repository root

- `feature_list.json`
- `PROGRESS.md`

These files remain in the root because they are user-visible handoff artifacts and the existing coding flow already treats them as the long-running project contract.

#### 3.2 Project runtime artifacts in `.agent/project/`

- `.agent/project/session_plan.md`
- `.agent/project/brainstorm/...`
- `.agent/project/checkpoints/index.jsonl`
- `.agent/project/session-tree/nodes.jsonl`
- `.agent/project/harness/config.generated.json`
- `.agent/project/harness/runs.jsonl`
- `.agent/project/harness/evidence/*.json`

These files are machine-owned runtime artifacts. They should be writable within the project boundary without forcing repeated writes into `.kodax/`.

#### 3.3 KodaX control plane in `.kodax/`

- `.kodax/config.local.json`
- `.kodax/AGENTS.md`
- `.kodax/skills/...`
- `.kodax/commands/...`

Project Mode should not store high-churn runtime artifacts here.

#### 3.4 Rule-source boundary

Project Harness should compile invariants from the current workspace project, for example:

- `<workspace>/AGENTS.md`
- `<workspace>/docs/HLD.md`
- `<workspace>/docs/ADR/**/*.md`
- `<workspace>/package.json` and local validation scripts

Project Harness should not treat `.kodax/**` or user-level KodaX control files as general project-invariant sources.

When KodaX is self-hosting inside its own repository, those repository docs are valid rule sources only because the current workspace project is KodaX itself.

### 4. Harness Model

Phase 1 introduces a deterministic `Project Harness` with three concerns:

1. Hard legality
   - project boundary checks
   - protected artifact checks
   - permission mode checks
   - dangerous action gating
2. Project invariants
   - active-feature scope checks
   - repo-specific rules discovered from the current workspace project docs and config
   - architecture and workflow rules compiled into declarative config
   - feature-step and session-plan completion checklist coverage
3. Goal progress
   - changed-file relevance
   - checks/test/quality deltas
   - completion evidence completeness
   - repeated-failure and stall detection

The runtime verifier is deterministic. LLM usage in this phase is limited to discovery-time generation of a declarative harness config, not free-form executable policy code.

### 5. Command Surface

#### 5.1 Existing commands with new behavior

- `/project next`
- `/project auto`

New command-layer rules:

- the agent may implement the feature, but it does not directly mark the feature as completed
- the command layer runs automatic verification after the attempt
- only the command layer may update completion state after verification passes

#### 5.2 New command

- `/project verify [#index|--last]`

Purpose:

- rerun deterministic verification for the last attempt or a selected feature
- print legality, checks, evidence completeness, quality delta, violations, repair hints, and final decision

#### 5.3 Explicitly out of scope for Phase 1

- `/project checkpoint`
- `/project rollback`
- `/project execute`
- harness branching or Thompson-sampling-style candidate search

### 6. Execution State Machine

Phase 1 execution flow for `/project next` and `/project auto`:

1. Select active feature
2. Load or generate harness config
3. Capture pre-run snapshot
   - active feature
   - session plan
   - quality baseline
4. Run implementation attempt with harness action gating
5. Run automatic verification
   - touched files
   - required checks
   - evidence completeness
   - quality delta
6. Decide outcome
   - `verified_complete`
   - `retryable_failure`
   - `needs_review`
   - `blocked`
7. If `retryable_failure`, inject structured critic feedback and retry once automatically
8. If retry succeeds, mark complete and continue
9. If retry fails again, stop without marking complete and surface `needs_review`

This keeps the green path fully automatic while still preventing false completion.

### 7. Prompt Changes

Only minimal prompt changes are required in Phase 1.

#### 7.1 Feature execution prompt updates

Replace direct completion instructions with:

- focus only on the active feature
- prefer minimal relevant edits
- add tests for behavior changes
- update `PROGRESS.md` with evidence and blockers
- do not directly mark the feature as done; completion is decided by the verifier

#### 7.2 Retry prompt updates

When a feature fails verification in a retryable way, prepend a structured critic summary from the harness run log before the retry attempt.

### 8. Acceptance Criteria

1. During `/project next` and `/project auto`, the agent cannot directly complete a feature by editing `feature_list.json`
2. Successful features are still completed automatically after checks and evidence verification pass
3. Failed verification does not mark the feature done
4. One automatic self-repair retry is attempted for retryable failures
5. Runtime Project artifacts are written under `.agent/project/`, not `.kodax/`
6. `/project verify` can reproduce the deterministic verification summary for the last run or a selected feature

### 9. Out of Scope

- automatic checkpoint/rollback orchestration
- Session Tree branch search
- model-generated executable harness code
- large prompt refactors for brainstorm/plan/status/quality
- migrating `feature_list.json` or `PROGRESS.md` into `.agent/`
