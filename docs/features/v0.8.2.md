## FEATURE_113: TodoList JSON / CLI Surface

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement
**Implementation surface**: `src/cli/json-output.ts` + `src/cli/commands/todos.ts` + `packages/coding/src/types.ts` 既有 `KodaXEvents.onTodoUpdate` consumer 注册点

### 背景

FEATURE_097 (v0.7.34) 在 KodaX AMA Runner 内部已经维护一份完整的 TodoList in-memory store，单进程 CLI 用 `KodaXEvents.onTodoUpdate?(items: readonly TodoItem[])` 单路事件回调把状态推到 Ink REPL 顶层 React state，渲染为 TodoListSurface。**该数据从未对外暴露**：

- `--json` mode（`src/cli/json-output.ts`）只输出 final answer + transcript，不输出中间 todo 状态
- 没有独立 CLI 命令 query 当前 todo 状态
- 上游 agent / IDE 扩展 / CI wrapper 无法消费 KodaX 的进度信号

v0.7.34 ship 时用户明确确认"对外 surface 暂时不加，但记入未来 feature"——这就是 FEATURE_113 的来源。

### 用户痛点（待 v0.7.34 落地后真实数据驱动）

设计稿 placeholder 阶段先列假设痛点，实际需要 v0.7.34 ship 后通过 telemetry / 真实用户反馈确认：

1. **CI / 自动化 wrapper**：把 KodaX 跑在 GitHub Actions / Jenkins，希望日志里看到 progress checkpoints 而非纯 spinner blob
2. **上游 agent 嵌入**：另一个 agent（例如 KodaX 自己的 dispatch_child_task 或第三方 host 通过 `clients/repointel/` 集成）调 KodaX 时希望拿到子任务的进度心跳
3. **IDE / VSCode 扩展**：在 sidebar 显示 KodaX 当前进度，对齐 Claude Code 的 IDE integration 体验

### 设计

**待 FEATURE_097 落地后根据真实使用场景细化**。当前规划的两个 surface：

#### Surface A — JSON Mode 增强

`--json` 输出流加入 `todo_update` event type（在现有 `tool_use` / `text` / `repo_intelligence_trace` 之外）：

```jsonc
{"type": "todo_update", "items": [
  {"id": "todo_1", "content": "Run migration tests", "status": "completed"},
  {"id": "todo_2", "content": "Update type definitions", "status": "in_progress"},
  ...
]}
```

每次 `KodaXEvents.onTodoUpdate` 触发时输出一行 JSON。流式消费者（jq / shell pipeline）可按 `select(.type=="todo_update")` 过滤。

#### Surface B — `kodax todos` CLI 命令（可选）

```bash
kodax todos              # one-shot: print current todo state from active session
kodax todos --watch      # streaming: print on every update
kodax todos --session ID # query specific session
```

实现：从 KodaX 既有 session lineage / IPC channel 读 todo state（具体 IPC 机制留待 FEATURE_097 落地后看哪种最合适）。

### 显式前置依赖

- FEATURE_097 (v0.7.34) 必须先 ship —— 没有内部 TodoList store + onTodoUpdate 事件，外部 surface 无数据可消费
- FEATURE_097 落地后至少 1-2 个 minor 版本的真实使用数据 —— 验证 todo 数据形态稳定 + 收集真实 wrapper / IDE 集成需求

### 显式不做

- **不**自动激活：FEATURE_097 落地不会自动启用 JSON / CLI surface。本 feature 是显式选择项
- **不**复刻 Claude Code 的 V2 swarm IPC：单进程 CLI 不需要 fs.watch + lockfile
- **不**在 v0.7.34 ship 前预留 placeholder API：YAGNI，当前 `KodaXEvents.onTodoUpdate` 已是 SDK 出口的稳定原语，FEATURE_113 落地时直接消费即可

### 与 FEATURE_097 的关系

完全 additive。FEATURE_097 是 KodaX 内部 UX，FEATURE_113 是对外集成 surface。两者共用同一份 in-memory TodoList store，不存在协议分叉。
