## 总览

v0.7.29 单一主题：**消除 SA / AMA 两套独立的 agent 执行路径**，把全部 agent 调用收敛到同一个 Runner 帧、同一个 executor、同一套 Layer-A primitives。

用户层 SA / AMA 切换保留为永久产品决策（ADR-003 / ADR-012）。本版只动实现层 —— 把"用户可切换"和"实现必须双轨"在历史中被合并的两件事重新拆开。

变更面：

- `agent.ts` 的 `runKodaX` 不再是独立 SA 入口；其 substrate 能力（provider loop、microcompact、edit recovery、extension runtime、cost tracker、session snapshot、reasoning resolution、tool guardrail）拆分到 `agent-runtime/`，其 SA-specific 行为（auto-reroute 等）下沉为 `defaultCodingAgent` 的 declarable middleware。
- `task-engine.ts` dispatcher 简化为：根据 `agentMode` 选择喂 `defaultCodingAgent`（SA topology）还是 `scoutAgent`（AMA topology），body 不分叉。
- v0.7.23 FEATURE_080 的 "Option Y" preset dispatcher facade 升级为真实 Runner 帧入口，shim 删除。
- AMA 路径（`runner-driven.ts`）当前 13 处 `// legacy parity restore` / `// inadvertently dropped` 注释所标记的失踪能力在统一底座下天然消失。

## FEATURE_100: SA Runner Frame Adoption & Capability Unification

**Status**: Planned
**Priority**: Critical
**Category**: Internal / Refactor

### 背景

#### 当前双底座的事实

- **SA 路径**：`packages/coding/src/agent.ts:1433` `runKodaX`，3164 行手写 provider loop。`task-engine.ts` 在 `agentMode === 'sa'` 时直接调用 `runDirectKodaX`。
- **AMA 路径**：`packages/coding/src/task-engine/runner-driven.ts:3698` `runManagedTaskViaRunner`，用 `Runner.run` + Layer-A primitives（FEATURE_084 v0.7.26 引入），**显式绕开 `runKodaX`**。
- **Subagent 路径**：`packages/coding/src/child-executor.ts` lazy-import `runKodaX`，子任务跑的是 SA body。

`packages/coding/src/agent.ts:2197-2204` 当前注释：

> "This file is the SA-mode entry (`runDirectKodaX`) and runner-driven.ts is the AMA-mode path... They are two parallel execution modes with **no convergence plan**, but they share the same provider stack and therefore see the same thinking-mode errors. Per CLAUDE.md 'abstract only after 3+ real cases', 2 call sites stay duplicated. v0.7.28."

这条注释准确描述了状态，但把它定性为"design"是事后合理化。本 ADR-020 把 "no convergence plan" 关闭。

#### SA body 内的 7 类独有能力（AMA 完全没有）

这些今天只在 `runKodaX` 体内存在，AMA worker 拿不到。其中 auto-reroute 经产品决策确认应在 AMA 也可用（理由：harder tasks deserve deeper thinking，符合 ADR-003 "harness on demand" 哲学）：

| 能力 | 位置 | 当前状态 | 新架构归属 |
|---|---|---|---|
| microcompact（每轮轻量清理） | `agent.ts:1731` | SA-only | Class 1 substrate middleware |
| edit anchor recovery + 写入阻断 | `agent.ts:913-1001` | SA-only | Class 1 substrate middleware |
| mutation scope reflection | `agent.ts:795-810` | SA-only | Class 1 substrate middleware |
| pre/post-tool judges | `agent.ts:1651` | SA-only | Class 1 substrate middleware |
| extensionRuntime per-turn 队列消费 | `agent.ts:366-502` | SA-only | Class 1 substrate middleware |
| autoResume / 会话生命周期 | `agent.ts:1465-1500` | SA-only | Class 1 substrate middleware |
| auto-reroute（mid-iteration depth escalation + task family reroute） | `agent.ts:1076-1244, :2442, :2788` | SA-only（runner-driven 不调 `runKodaX`） | declarable opt-in middleware，default-on for `defaultCodingAgent` 与 `generatorAgent` |

auto-reroute 的归属说明：grep 验证 `agent.ts` 中**没有任何 `agentMode` 门控**，门控条件是 `effectiveReasoningPlan.mode === 'auto'`。今天 AMA 没有 auto-reroute 是结构副作用（runner-driven 绕开 `runKodaX`），不是设计排除。Evaluator 是 post-iteration verdict，不替代 mid-iteration adaptive recovery；让 generatorAgent 也能 mid-iteration 升档符合 ADR-003 "harness on demand" 哲学（任务难度真实变化时升档，不被 ceiling 固定）。

| FEATURE_089 Constructed Agent Invocation | v0.7.31 | `await Runner.run(constructedAgent, input)` 显式调用 | SA 无机制调用 Agent 实例 |

| FEATURE_092 AutoMode ToolGuardrail | v0.7.33 | 注册到 Runner 默认 guardrail 栈 | SA 不进 Runner 栈 → 安全分类器对 SA 无效 |

### 设计

#### 模型：Option α — 单一 executor + 多 Agent declaration

**Layer A — substrate（共享，单一实现）**：

```
provider loop · tool dispatch · history 管理 · microcompact · edit recovery
extension runtime · ToolGuardrail runtime · reasoning resolution
trace + span emission · session snapshot · cost tracking · onSessionStart/Complete/Error
permission gate · cleanupIncompleteToolCalls · validateAndFixToolHistory
buildPromptMessageContent · saveSessionSnapshot
```

所有 Agent 共用同一个 executor，在 substrate 之上跑。

**Layer B — Agent declaration（多份）**：

```ts
interface AgentDeclaration {
  name: string                                  // role name
  systemPrompt: string | PromptBuilder          // role prompt
  handoffs?: HandoffConfig[]                    // null for SA / leaf agents
  reasoning?: AgentReasoningProfile             // FEATURE_078 v0.7.30 填充语义
  tools?: ToolSliceSelector                     // 此 Agent 可见的工具子集
  middleware?: {                                // declarable opt-in middleware
    autoReroute?: boolean                       // default false; defaultCodingAgent + generatorAgent 默认 true
    mutationScopeReflection?: boolean           // default false; defaultCodingAgent 默认 true
    preAnswerJudge?: boolean                    // default false; defaultCodingAgent 默认 true
    postToolJudge?: boolean                     // default false; defaultCodingAgent + generatorAgent 默认 true
  }
}
```

**Layer C — dispatcher（薄层）**：

```ts
// task-engine.ts 简化后
function dispatch(options: KodaXOptions, prompt: string): Promise<KodaXResult> {
  const agent = options.agentMode === 'sa'
    ? defaultCodingAgent
    : scoutAgent  // Scout 自带 handoff 链到 Planner / Generator / Evaluator
  return Runner.run(agent, prompt, buildRunnerCtx(options))
}
```

#### 能力归类

**Class 1 — Substrate Middleware（共享，无条件运行）**：

provider loop kernel、tool dispatch loop、history 管理、microcompact、edit anchor recovery、mutation scope reflection（hookable）、extensionRuntime 队列消费、autoResume / session lifecycle、`cleanupIncompleteToolCalls`、`validateAndFixToolHistory`、`buildPromptMessageContent` 多模态、`saveSessionSnapshot`、`CostTracker`、tri-state 权限 gate、`onSessionStart` / `onStreamEnd` / `onComplete` / `onError` / `onProviderRateLimit` 事件发射、repoIntelligenceContext 注入。

→ 落到 `packages/coding/src/agent-runtime/middleware/*.ts`，每个能力一个文件。

**Class 2 — Runner-Level Lifecycle / Guardrail（共享，注册式）**：

ToolGuardrail 注册栈（085 引入）、reasoning profile resolution（078 v0.7.30 填充行为）、max_tokens 升级阶梯（085 已部分迁移）、`Trace` / `Span` emission。

→ 由 `packages/core/src/runner.ts` 的 `Runner.run` 在进入帧时挂载，所有 Agent 透明享用。

**Class 3 — Declarable Opt-In Middleware（按 Agent 声明启用）**：

auto-reroute、pre-answer judge、post-tool judge、mutation reflection。

→ 在 `Agent.middleware` 字段声明启用；substrate executor 按声明决定挂哪些钩子。`defaultCodingAgent` 和 `generatorAgent` 启用 auto-reroute（保持 SA 当前行为 + 给 AMA Generator 补能力，对齐 ADR-003 "harness on demand"：harder tasks deserve deeper reasoning regardless of mode）。

#### Agent 声明示例

```ts
// packages/coding/src/agents/default-coding-agent.ts
export const defaultCodingAgent: AgentDeclaration = {
  name: 'default-coding',
  systemPrompt: buildDefaultCodingPrompt,
  reasoning: { default: 'balanced', max: 'deep', escalateOnRevise: true },
  // SA 无 handoffs
  middleware: {
    autoReroute: true,
    mutationScopeReflection: true,
    preAnswerJudge: true,
    postToolJudge: true,
  },
}
```

```ts
// packages/coding/src/agents/scout-agent.ts (FEATURE_084 已存在的 Agent，本版补 middleware)
export const scoutAgent: AgentDeclaration = {
  name: 'scout',
  systemPrompt: buildScoutPrompt,
  reasoning: { default: 'quick', max: 'balanced', escalateOnRevise: false },
  handoffs: [plannerHandoff, generatorHandoff, evaluatorHandoff],
  middleware: {
    autoReroute: false,  // Scout turn 通常很短，不需要
    mutationScopeReflection: false,
    preAnswerJudge: false,
    postToolJudge: false,
  },
}

export const generatorAgent: AgentDeclaration = {
  name: 'generator',
  systemPrompt: buildGeneratorPrompt,
  reasoning: { default: 'balanced', max: 'deep', escalateOnRevise: true },
  handoffs: [evaluatorHandoff],
  middleware: {
    autoReroute: true,             // 让 Generator 也能 mid-iteration 升档应对更难任务
    mutationScopeReflection: false, // Evaluator 接管 mutation scope 判定
    preAnswerJudge: false,
    postToolJudge: true,
  },
}
```

#### 与 v0.7.23 Option Y 的关系

Option Y 是"Runner.run 注册 dispatcher，body 不动"的 facade pattern：表面上 SA 走了 Runner，实际还是直跑 `runKodaX`，guardrail 栈、reasoning resolution、Agent declaration metadata 都没生效。

本版**显式废除 Option Y**：

- `packages/coding/src/coding-preset.ts` 中的 dispatcher 注册（FEATURE_080 引入）改为真正进入 Runner 帧 → 解析 Agent declaration → 跑 substrate executor。
- 不保留兼容 alias、不保留 legacy flag、不保留双轨过渡。
- `runDirectKodaX` 符号删除；`runKodaX` 名义保留为 substrate executor 的薄壳（返回 `Runner.run(defaultCodingAgent, prompt, ctx)`），便于外部 SDK 消费者无 breaking。

### AMA Convergence as Side-Effect（不是附带好处，是 P3/P4 硬验收点）

FEATURE_084（v0.7.26）当时的 punt 是"只重写 Scout/Generator/Evaluator 为 Agent 实例，SA body 不动"。直接后果：AMA 的 `task-engine/runner-driven.ts` 因为不能 `import runKodaX`，只能把 SA body 里的能力**逐项手动复刻**。在 v0.7.28 这一版的 codebase 里，`grep -E "parity restore|v[0-9]\\.[0-9]\\.[0-9]+ parity:|Legacy agent\\.ts:|Legacy parity"` 在 `runner-driven.ts` 里命中 **35 行**，对应约 24 个独立 restore 点。

### TurnContext type design

新文件：`packages/coding/src/agent-runtime/turn-context.ts`

```typescript
// AbortSignal is a global in Node 18+ — no import needed.
import type {
  KodaXOptions, KodaXEvents, KodaXToolExecutionContext,
  KodaXContextTokenSnapshot, KodaXMessage, KodaXManagedProtocolPayload,
  SessionErrorMetadata,
} from '../types.js';
import type { CompactionConfig } from '@kodax/agent';
import type { CostTracker } from '@kodax/ai';
import type { RuntimeSessionState } from './runtime-session-state.js';
import type { ReasoningPlan } from '../reasoning.js';
import type { ReasoningExecutionState } from './reasoning-plan-entry.js';

/**
 * Single immutable value threaded through every step. Each step returns
 * the next ctx via spread `{ ...ctx, field: next }`. Side-effect callbacks
 * (events, persistence) live on StepCallbacks, not on ctx.
 */
export interface TurnContext {
  // ── STABLE (set at session-start, never replaced) ───────────────────────
  readonly options: KodaXOptions;
  readonly events: KodaXEvents;
  readonly maxIter: number;
  readonly sessionId: string;
  readonly executionCwd: string;
  readonly toolCtx: KodaXToolExecutionContext;       // backups Map: see §exception (a)
  readonly compactionConfig: CompactionConfig;
  readonly prompt: string;
  readonly title: string;                             // frozen after first persist
  readonly errorMetadata: SessionErrorMetadata | undefined;
  readonly managedProtocolPayload: { current: KodaXManagedProtocolPayload | undefined };

  // ── PER_TURN (advanced at iteration boundaries) ──────────────────────────
  readonly iter: number;
  readonly messages: readonly KodaXMessage[];
  readonly sessionState: RuntimeSessionState;        // see §exception (b)
  readonly currentExecution: ReasoningExecutionState;
  readonly reasoningPlan: ReasoningPlan;
  readonly contextTokenSnapshot: KodaXContextTokenSnapshot;
  readonly costTracker: CostTracker;
  readonly compactConsecutiveFailures: number;
  readonly managedProtocolContinueAttempted: boolean;
  readonly incompleteRetryCount: number;
  readonly maxTokensRetryCount: number;
  readonly preAnswerJudgeConsumed: boolean;
  readonly postToolJudgeConsumed: boolean;
  readonly autoFollowUpCount: number;
  readonly autoDepthEscalationCount: number;
  readonly autoTaskRerouteCount: number;
  readonly lastText: string;
  readonly limitReached: boolean;
}

/** Side-effect channel — never on TurnContext. */
export interface StepCallbacks {
  emit: KodaXEvents;
  signal: AbortSignal | undefined;
  persistSession: (ctx: TurnContext) => Promise<void>;  // wraps storage errors internally
}

/** A substrate executor step. MUST NOT mutate ctx in place. */
export type Step = (ctx: TurnContext, callbacks: StepCallbacks) => Promise<TurnContext>;

/** Outcome variants returned by the per-turn step pipeline. */
export type TurnOutcome =
  | { kind: 'continue'; ctx: TurnContext }
  | { kind: 'complete'; ctx: TurnContext; signal?: 'COMPLETE' | 'BLOCKED' | 'DECIDE' }
  | { kind: 'interrupted'; ctx: TurnContext }
  | { kind: 'error'; ctx: TurnContext; error: Error }
  | { kind: 'limit_reached'; ctx: TurnContext };
```

#### Step signature 选 async-with-callbacks 而非 pure

- pure `(ctx) => ctx` 不行：`provider.stream` / `intelligentCompact` 是异步的，且 stream 回调要在流进行中字符级触发。
- `(ctx) => { next, events }` 队列化事件不行：streaming delta 必须实时到 REPL，不能 batch。
- `async (ctx, callbacks) => ctx` 与 P2 已有的 `applyProviderPrepareHook` 模式一致。

#### Immutability boundary（三个明确例外）

**例外 (a) — `toolCtx.backups: Map<string, string>`**：write tool 执行时 in-place 增长。每步深拷贝 O(n) 不可行，且没有 step 需要观察"工具执行前的 backup 快照"。规则：`toolCtx` 字段引用稳定，`toolCtx.backups` 允许就地变更。在 JSDoc 标 `@mutable-exception(a)`。

**例外 (b) — `sessionState: RuntimeSessionState`**：扩展回调通过 `createExtensionRuntimeSessionController` 持有引用，重绑控制器超出 P3 范围。`sessionState` 引用在 frame 生命期内恒定，扩展回调允许就地修改其内部 Map/Set（`extensionState` / `editRecoveryAttempts` / `blockedEditWrites`）。

**例外 (c) — `managedProtocolPayload: { current: ... }`**：wrapper 对象被 `toolCtx.emitManagedProtocol` 闭包捕获。回调通过 `mergeManagedProtocolPayload` 在流式过程中替换 `.current`。wrapper 引用稳定，仅 `.current` 字段被该闭包替换。

**Streaming buffers**：`onTextDelta` / `onThinkingDelta` 字符级回调不碰 ctx，本地变量积累，stream settle 后 join 一次性写入 `ctx.lastText`。

**AbortSignal**：仅在 `callbacks.signal` 上，绝不在 ctx 上（不可序列化、不可恢复）。

### Cross-cutting risks（不在原 inventory 中）

**R1 — Stream 回调内 token snapshot 累积**。`onRateLimit` 的 `usage` metadata 当前通过闭包捕获 `let contextTokenSnapshot` 累积。抽出后 `BoundaryTrackerSession` 必须维护一个 mutable `tokenAccumulator`，在 `executeProviderStream` 返回前 merge 到 `ctx.contextTokenSnapshot`，否则 `emitIterationEnd` 报错的 token 数。新增 5 行 merger 函数 + 专门测试。

**R2 — 步骤间 abort 检查**。`await stepA → await stepB` 之间若 `signal.aborted`，必须在每个 step boundary 重检（不能只在 `dispatchTools` 里检）。否则 stream 返回工具块后、dispatch 启动前 Ctrl+C 会被吞。CAP-TOOL-CANCEL-002 必须 active 验证。

**R3 — Judge consumed latch 的 reroute 重置**。`preAnswerJudgeConsumed` / `postToolJudgeConsumed` 是 session-scoped latch。当 `maybeAdvanceAutoReroute` 切换 routing decision 时，consumed 必须清零让 judge 在新分支上重新触发。否则首次决策后所有后续轮 judge 静默跳过。新增 `CAP-PRE-ANSWER-JUDGE-CONSUMED-RESET-001`。

**R4 — `auto*Count` 三计数器作用域**。当前设计放在 `TurnContext` per-turn 字段（auto-reroute 在 default-coding-agent 中始终启用，简化处理）。但严格来说这是 Class 3 middleware（auto-reroute）私有状态。若未来 agent declaration 支持禁用 auto-reroute，这三个计数器需迁移至 middleware 自己的 state container —— 否则禁用 middleware 的 declaration 仍会"占位"计数。本风险在 P3.6 之后、middleware-opt-in 落地前必须复审。

**R5 — `messages` vs `providerMessages` 分歧**。`streamStep` 内 sanitize-thinking-and-retry 链上有 mid-retry 的 `providerMessages` 和 settled `messages` 两份，commit 到 `ctx.messages` 必须只在 `break` 后用 settled 值。否则 ctx 持有部分 sanitized 历史，后续 dispatch 会读错。

**R6 — `title` 冻结契约无运行时强制**。TS `readonly` 是编译期。在 `persistSession` wrapper 内 debug 模式断言 title 跨调用不变（warning 即可，不抛）。

**R7 — `isNewSession` 三处歧义**。当前 3 处用 `messages.length === 1`，含义实为"首轮 / 无前置 assistant 轮 / 压缩后只剩 summary"三种。P3 必须审计每个 call site，明确是 `ctx.iter === 0` 还是 `ctx.messages.length === 1`，差异点：恢复会话 + 压缩后。`buildAutoRepoIntelligenceContext` 是关键消费方。

**R8 — CAP-013 storage 失败隔离前置到 P3.0**。`saveSessionSnapshot` 当前不裹 `storage.save` try/catch —— 必须在 `StepCallbacks.persistSession` 包装层吸收，否则错误路径上的 persist 会覆盖原始 error。这是 CAP-013-003 的 P3-deferred 项，必须 P3.0 实现，不能后置。

**R9 — `reasoningPlan` 与 `currentExecution` 必须原子替换**。`maybeAdvanceAutoReroute` 同时返回新的两者；任何只更新一个的 helper 都禁止。在 `auto-reroute.ts` 内定义 `applyRerouteState(ctx, rerouteResult): TurnContext` 作为唯一合法路径。
