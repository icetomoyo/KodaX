## 总览

v0.7.29 单一主题：**消除 SA / AMA 两套独立的 agent 执行路径**，把全部 agent 调用收敛到同一个 Runner 帧、同一个 executor、同一套 Layer-A primitives。

用户层 SA / AMA 切换保留为永久产品决策（ADR-003 / ADR-012）。本版只动实现层 —— 把"用户可切换"和"实现必须双轨"在历史中被合并的两件事重新拆开。

变更面：

- `agent.ts` 的 `runKodaX` 不再是独立 SA 入口；其 substrate 能力（provider loop、microcompact、edit recovery、extension runtime、cost tracker、session snapshot、reasoning resolution、tool guardrail）拆分到 `agent-runtime/`，其 SA-specific 行为（auto-reroute 等）下沉为 `defaultCodingAgent` 的 declarable middleware。
- `task-engine.ts` dispatcher 简化为：根据 `agentMode` 选择喂 `defaultCodingAgent`（SA topology）还是 `scoutAgent`（AMA topology），body 不分叉。
- v0.7.23 FEATURE_080 的 "Option Y" preset dispatcher facade 升级为真实 Runner 帧入口，shim 删除。
- AMA 路径（`runner-driven.ts`）当前 13 处 `// legacy parity restore` / `// inadvertently dropped` 注释所标记的失踪能力在统一底座下天然消失。

## FEATURE_100: SA Runner Frame Adoption & Capability Unification

**Status**: Planned
**Priority**: Critical
**Category**: Internal / Refactor

### 背景

#### 当前双底座的事实

- **SA 路径**：`packages/coding/src/agent.ts:1433` `runKodaX`，3164 行手写 provider loop。`task-engine.ts` 在 `agentMode === 'sa'` 时直接调用 `runDirectKodaX`。
- **AMA 路径**：`packages/coding/src/task-engine/runner-driven.ts:3698` `runManagedTaskViaRunner`，用 `Runner.run` + Layer-A primitives（FEATURE_084 v0.7.26 引入），**显式绕开 `runKodaX`**。
- **Subagent 路径**：`packages/coding/src/child-executor.ts` lazy-import `runKodaX`，子任务跑的是 SA body。

`packages/coding/src/agent.ts:2197-2204` 当前注释：

> "This file is the SA-mode entry (`runDirectKodaX`) and runner-driven.ts is the AMA-mode path... They are two parallel execution modes with **no convergence plan**, but they share the same provider stack and therefore see the same thinking-mode errors. Per CLAUDE.md 'abstract only after 3+ real cases', 2 call sites stay duplicated. v0.7.28."

这条注释准确描述了状态，但把它定性为"design"是事后合理化。本 ADR-020 把 "no convergence plan" 关闭。

这些今天只在 `runKodaX` 体内存在，AMA worker 拿不到。其中 auto-reroute 经产品决策确认应在 AMA 也可用（理由：harder tasks deserve deeper thinking，符合 ADR-003 "harness on demand" 哲学）：

| 能力 | 位置 | 当前状态 | 新架构归属 |
|---|---|---|---|

| edit anchor recovery + 写入阻断 | `agent.ts:913-1001` | SA-only | Class 1 substrate middleware |
| mutation scope reflection | `agent.ts:795-810` | SA-only | Class 1 substrate middleware |
| pre/post-tool judges | `agent.ts:1651` | SA-only | Class 1 substrate middleware |

auto-reroute 的归属说明：grep 验证 `agent.ts` 中**没有任何 `agentMode` 门控**，门控条件是 `effectiveReasoningPlan.mode === 'auto'`。今天 AMA 没有 auto-reroute 是结构副作用（runner-driven 绕开 `runKodaX`），不是设计排除。Evaluator 是 post-iteration verdict，不替代 mid-iteration adaptive recovery；让 generatorAgent 也能 mid-iteration 升档符合 ADR-003 "harness on demand" 哲学（任务难度真实变化时升档，不被 ceiling 固定）。

### 设计

#### 模型：Option α — 单一 executor + 多 Agent declaration

**Layer A — substrate（共享，单一实现）**：

```
provider loop · tool dispatch · history 管理 · microcompact · edit recovery
extension runtime · ToolGuardrail runtime · reasoning resolution
trace + span emission · session snapshot · cost tracking · onSessionStart/Complete/Error
permission gate · cleanupIncompleteToolCalls · validateAndFixToolHistory
buildPromptMessageContent · saveSessionSnapshot
```

所有 Agent 共用同一个 executor，在 substrate 之上跑。

**Layer B — Agent declaration（多份）**：

```ts
interface AgentDeclaration {
  name: string                                  // role name
  systemPrompt: string | PromptBuilder          // role prompt
  handoffs?: HandoffConfig[]                    // null for SA / leaf agents
  reasoning?: AgentReasoningProfile             // FEATURE_078 v0.7.30 填充语义
  tools?: ToolSliceSelector                     // 此 Agent 可见的工具子集
  middleware?: {                                // declarable opt-in middleware
    autoReroute?: boolean                       // default false; defaultCodingAgent + generatorAgent 默认 true
    mutationScopeReflection?: boolean           // default false; defaultCodingAgent 默认 true
    preAnswerJudge?: boolean                    // default false; defaultCodingAgent 默认 true
    postToolJudge?: boolean                     // default false; defaultCodingAgent + generatorAgent 默认 true
  }
}
```

**Layer C — dispatcher（薄层）**：

```ts
// task-engine.ts 简化后
function dispatch(options: KodaXOptions, prompt: string): Promise<KodaXResult> {
  const agent = options.agentMode === 'sa'
    ? defaultCodingAgent
    : scoutAgent  // Scout 自带 handoff 链到 Planner / Generator / Evaluator
  return Runner.run(agent, prompt, buildRunnerCtx(options))
}
```

#### 能力归类

**Class 1 — Substrate Middleware（共享，无条件运行）**：

provider loop kernel、tool dispatch loop、history 管理、microcompact、edit anchor recovery、mutation scope reflection（hookable）、extensionRuntime 队列消费、autoResume / session lifecycle、`cleanupIncompleteToolCalls`、`validateAndFixToolHistory`、`buildPromptMessageContent` 多模态、`saveSessionSnapshot`、`CostTracker`、tri-state 权限 gate、`onSessionStart` / `onStreamEnd` / `onComplete` / `onError` / `onProviderRateLimit` 事件发射、repoIntelligenceContext 注入。

→ 落到 `packages/coding/src/agent-runtime/middleware/*.ts`，每个能力一个文件。

**Class 2 — Runner-Level Lifecycle / Guardrail（共享，注册式）**：

ToolGuardrail 注册栈（085 引入）、reasoning profile resolution（078 v0.7.30 填充行为）、max_tokens 升级阶梯（085 已部分迁移）、`Trace` / `Span` emission。

→ 由 `packages/core/src/runner.ts` 的 `Runner.run` 在进入帧时挂载，所有 Agent 透明享用。

**Class 3 — Declarable Opt-In Middleware（按 Agent 声明启用）**：

auto-reroute、pre-answer judge、post-tool judge、mutation reflection。

→ 在 `Agent.middleware` 字段声明启用；substrate executor 按声明决定挂哪些钩子。`defaultCodingAgent` 和 `generatorAgent` 启用 auto-reroute（保持 SA 当前行为 + 给 AMA Generator 补能力，对齐 ADR-003 "harness on demand"：harder tasks deserve deeper reasoning regardless of mode）。

#### Agent 声明示例

```ts
// packages/coding/src/agents/default-coding-agent.ts
export const defaultCodingAgent: AgentDeclaration = {
  name: 'default-coding',
  systemPrompt: buildDefaultCodingPrompt,
  reasoning: { default: 'balanced', max: 'deep', escalateOnRevise: true },
  // SA 无 handoffs
  middleware: {
    autoReroute: true,
    mutationScopeReflection: true,
    preAnswerJudge: true,
    postToolJudge: true,
  },
}
```

```ts
// packages/coding/src/agents/scout-agent.ts (FEATURE_084 已存在的 Agent，本版补 middleware)
export const scoutAgent: AgentDeclaration = {
  name: 'scout',
  systemPrompt: buildScoutPrompt,
  reasoning: { default: 'quick', max: 'balanced', escalateOnRevise: false },
  handoffs: [plannerHandoff, generatorHandoff, evaluatorHandoff],
  middleware: {
    autoReroute: false,  // Scout turn 通常很短，不需要
    mutationScopeReflection: false,
    preAnswerJudge: false,
    postToolJudge: false,
  },
}

export const generatorAgent: AgentDeclaration = {
  name: 'generator',
  systemPrompt: buildGeneratorPrompt,
  reasoning: { default: 'balanced', max: 'deep', escalateOnRevise: true },
  handoffs: [evaluatorHandoff],
  middleware: {
    autoReroute: true,             // 让 Generator 也能 mid-iteration 升档应对更难任务
    mutationScopeReflection: false, // Evaluator 接管 mutation scope 判定
    preAnswerJudge: false,
    postToolJudge: true,
  },
}
```

#### 与 v0.7.23 Option Y 的关系

Option Y 是"Runner.run 注册 dispatcher，body 不动"的 facade pattern：表面上 SA 走了 Runner，实际还是直跑 `runKodaX`，guardrail 栈、reasoning resolution、Agent declaration metadata 都没生效。

本版**显式废除 Option Y**：

- `packages/coding/src/coding-preset.ts` 中的 dispatcher 注册（FEATURE_080 引入）改为真正进入 Runner 帧 → 解析 Agent declaration → 跑 substrate executor。
- 不保留兼容 alias、不保留 legacy flag、不保留双轨过渡。
- `runDirectKodaX` 符号删除；`runKodaX` 名义保留为 substrate executor 的薄壳（返回 `Runner.run(defaultCodingAgent, prompt, ctx)`），便于外部 SDK 消费者无 breaking。

复用 v0.7.27 引入的 dispatch eval baseline（commit `35a9b67`）。重构 PR 跑一遍 baseline，分数下降不得超过 noise threshold（-2%）。下降必须能定位到具体 capability 没接好。

#### 机制 5 — Phased Merge Strategy（P1-P4 硬 gate）

P1 → P2 → P3 → P4 严格顺序，phase 内可多 PR，phase 间硬 gate。每个 phase 通过条件如上。

#### 加强 A — Inventory 增加"已知失踪能力"基础集（P1 启动时）

P1 启动时**先做 runner-driven.ts 的 parity-restore 注释挖掘**：

```bash
  packages/coding/src/task-engine/runner-driven.ts
```

每条命中是 FEATURE_084 当年漏掉过的能力，标为 `HIGH_RISK_PARITY` 进 inventory，golden trace 必须有专门 case 覆盖。理由：这些是已被实证容易在重构中漏掉的能力，不是想象中的风险。

### TurnContext type design

新文件：`packages/coding/src/agent-runtime/turn-context.ts`

```typescript
// AbortSignal is a global in Node 18+ — no import needed.
import type {
  KodaXOptions, KodaXEvents, KodaXToolExecutionContext,
  KodaXContextTokenSnapshot, KodaXMessage, KodaXManagedProtocolPayload,
  SessionErrorMetadata,
} from '../types.js';
import type { CompactionConfig } from '@kodax/agent';
import type { CostTracker } from '@kodax/ai';
import type { RuntimeSessionState } from './runtime-session-state.js';
import type { ReasoningPlan } from '../reasoning.js';
import type { ReasoningExecutionState } from './reasoning-plan-entry.js';

/**
 * Single immutable value threaded through every step. Each step returns
 * the next ctx via spread `{ ...ctx, field: next }`. Side-effect callbacks
 * (events, persistence) live on StepCallbacks, not on ctx.
 */
export interface TurnContext {
  // ── STABLE (set at session-start, never replaced) ───────────────────────
  readonly options: KodaXOptions;
  readonly events: KodaXEvents;
  readonly maxIter: number;
  readonly sessionId: string;
  readonly executionCwd: string;
  readonly toolCtx: KodaXToolExecutionContext;       // backups Map: see §exception (a)
  readonly compactionConfig: CompactionConfig;
  readonly prompt: string;
  readonly title: string;                             // frozen after first persist
  readonly errorMetadata: SessionErrorMetadata | undefined;
  readonly managedProtocolPayload: { current: KodaXManagedProtocolPayload | undefined };

  // ── PER_TURN (advanced at iteration boundaries) ──────────────────────────
  readonly iter: number;
  readonly messages: readonly KodaXMessage[];
  readonly sessionState: RuntimeSessionState;        // see §exception (b)
  readonly currentExecution: ReasoningExecutionState;
  readonly reasoningPlan: ReasoningPlan;
  readonly contextTokenSnapshot: KodaXContextTokenSnapshot;
  readonly costTracker: CostTracker;
  readonly compactConsecutiveFailures: number;
  readonly managedProtocolContinueAttempted: boolean;
  readonly incompleteRetryCount: number;
  readonly maxTokensRetryCount: number;
  readonly preAnswerJudgeConsumed: boolean;
  readonly postToolJudgeConsumed: boolean;
  readonly autoFollowUpCount: number;
  readonly autoDepthEscalationCount: number;
  readonly autoTaskRerouteCount: number;
  readonly lastText: string;
  readonly limitReached: boolean;
}

/** Side-effect channel — never on TurnContext. */
export interface StepCallbacks {
  emit: KodaXEvents;
  signal: AbortSignal | undefined;
  persistSession: (ctx: TurnContext) => Promise<void>;  // wraps storage errors internally
}

/** A substrate executor step. MUST NOT mutate ctx in place. */
export type Step = (ctx: TurnContext, callbacks: StepCallbacks) => Promise<TurnContext>;

/** Outcome variants returned by the per-turn step pipeline. */
export type TurnOutcome =
  | { kind: 'continue'; ctx: TurnContext }
  | { kind: 'complete'; ctx: TurnContext; signal?: 'COMPLETE' | 'BLOCKED' | 'DECIDE' }
  | { kind: 'interrupted'; ctx: TurnContext }
  | { kind: 'error'; ctx: TurnContext; error: Error }
  | { kind: 'limit_reached'; ctx: TurnContext };
```

#### Step signature 选 async-with-callbacks 而非 pure

- pure `(ctx) => ctx` 不行：`provider.stream` / `intelligentCompact` 是异步的，且 stream 回调要在流进行中字符级触发。
- `(ctx) => { next, events }` 队列化事件不行：streaming delta 必须实时到 REPL，不能 batch。
- `async (ctx, callbacks) => ctx` 与 P2 已有的 `applyProviderPrepareHook` 模式一致。

#### Immutability boundary（三个明确例外）

**例外 (a) — `toolCtx.backups: Map<string, string>`**：write tool 执行时 in-place 增长。每步深拷贝 O(n) 不可行，且没有 step 需要观察"工具执行前的 backup 快照"。规则：`toolCtx` 字段引用稳定，`toolCtx.backups` 允许就地变更。在 JSDoc 标 `@mutable-exception(a)`。

**例外 (b) — `sessionState: RuntimeSessionState`**：扩展回调通过 `createExtensionRuntimeSessionController` 持有引用，重绑控制器超出 P3 范围。`sessionState` 引用在 frame 生命期内恒定，扩展回调允许就地修改其内部 Map/Set（`extensionState` / `editRecoveryAttempts` / `blockedEditWrites`）。

**例外 (c) — `managedProtocolPayload: { current: ... }`**：wrapper 对象被 `toolCtx.emitManagedProtocol` 闭包捕获。回调通过 `mergeManagedProtocolPayload` 在流式过程中替换 `.current`。wrapper 引用稳定，仅 `.current` 字段被该闭包替换。

**Streaming buffers**：`onTextDelta` / `onThinkingDelta` 字符级回调不碰 ctx，本地变量积累，stream settle 后 join 一次性写入 `ctx.lastText`。

**AbortSignal**：仅在 `callbacks.signal` 上，绝不在 ctx 上（不可序列化、不可恢复）。

## FEATURE_078: Role-Aware Reasoning Profiles

**Status**: Implemented (v0.7.29)
**Priority**: High
**Category**: Internal
**Dependencies**: FEATURE_100 (本版) — Agent declaration `reasoning` slot

> Migration note (2026-04-27): originally planned for v0.7.35, then v0.7.29
> (Plan B compression), then 2026-04-26 顺延 v0.7.30, finally 2026-04-27
> 撤回顺延 — landed in v0.7.29 alongside FEATURE_100. Rationale: the
> SA `defaultCodingAgent.reasoning` slot is already wired by FEATURE_100;
> shipping the slot's behavior in a separate version would leave an
> "infrastructure-without-runtime" gap. v0.7.30 's FEATURE_057 Track F
> (TUI cell-level diff) is unaffected — zero overlap with this feature.

### 背景

Pre-FEATURE_078 reasoning model was **flat single-tier**: `--reasoning <mode>` /
`options.reasoningMode` pinned **all** roles (Scout, Planner, Generator,
Evaluator, SA) to the same depth. Three problems:

1. Scout's role is fast scope analysis — running it at user's `deep` ceiling
   wastes tokens on a role designed for triage
2. Generator's code-writing and Evaluator's evidence-checking have different
   thinking-depth needs; one global knob can't express both
3. The user's `--reasoning deep` intent is "I want deep thinking when it
   helps", not "Scout judging task complexity should also be deep"

FEATURE_080 (v0.7.23) introduced `Agent.reasoning: AgentReasoningProfile` as
a placeholder; FEATURE_084 (v0.7.26) prefilled per-role profiles for
Scout/Planner/Generator/Evaluator; FEATURE_100 (this version) attached
`defaultCodingAgent.reasoning` for SA. All prerequisites in place.

### 设计 — 4 级决策链

```
L1 用户意图上限 (User Ceiling + Bias)
      │  --reasoning / options.reasoningMode / config.json reasoningCeiling
      │  语义：硬上限 + 默认偏好
      ▼
L2 角色默认档位 (Agent.reasoning.default + .max)
      │  Scout: { default: 'quick',    max: 'balanced', escalateOnRevise: false }
      │  Planner: { default: 'balanced', max: 'deep',  escalateOnRevise: true  }
      │  Generator: { default: 'balanced', max: 'deep', escalateOnRevise: true  }
      │  Evaluator: { default: 'balanced', max: 'deep', escalateOnRevise: false }
      │  defaultCodingAgent (SA): { default: 'balanced', max: 'deep', escalateOnRevise: true }
      ▼
L3 Scout 下游建议 (KodaXManagedScoutPayload.downstreamReasoningHint)
      │  非绑定；只有 Scout 在 emit_scout_verdict 时显式设置才进入链路
      ▼
L4 Evaluator / revise-loop 动态升档 (escalateThinkingDepth(_, ceiling))
      │  升档 clamp 至 L1 上限；ceiling='off' 时 escalate 是 no-op
      ▼
final per-turn reasoning depth
```

**Final = clamp(clamp(L3 ?? L2.default, L2.max), L1)**

`L1 = 'off'` 是 hard kill switch — 不论 L2/L3 给什么，effective 永远是 `off`。
`L1 = 'auto'` 是 "let the system pick" — 等价于 `ceilingDepth = 'high'`，escalate 不被 clamp。

### Contract

**Public API（新增 / 变更）**:
- `AgentReasoningProfile` type (v0.7.23 introduced as placeholder, v0.7.29 starts being honored)
- `resolveRoleReasoning(role, userCeiling, profile?, scoutHint?)` — 新提，L1-L4 chain entry

**Public API（语义变更）**:
- `--reasoning <mode>` 语义从 "all roles use this" 收紧为 "ceiling + bias for default"

## FEATURE_103: Scout Calibration & L5 User-Followup Escalate

> Discovered during v0.7.29 review of FEATURE_078; per the user's
> instruction "既然是当前版本发现的且相关的问题，单独加一个 feature 来记录但当前版本处理掉",
> implemented in-version as a discrete feature.

### 背景

FEATURE_078 落地后回顾发现两个 calibration 问题：

1. **Scout 默认深度与职责不匹配**：v0.7.16 FEATURE_061 之后 Scout 不再是单纯分类器——它判断 H0/H1/H2（决定整个 run 的 topology）、直接执行 H0、emit `executionObligations[]`、emit FEATURE_078 `downstream_reasoning_hint`（meta-reasoning：替 Generator 推理"它需要多深"）。`default: 'quick'` 是 v0.7.26 FEATURE_084 重写时按"分类器时代"沿用的旧值，与今日 Scout 实际承担的级联决策职责不符。

2. **L1-L4 决策链覆盖了"系统自检"但没覆盖"用户施压"**：L4 `escalateThinkingDepth(_, ceiling)` 在 Evaluator 返回 `revise` 时升档，覆盖了**任务内**的不满（系统自己说不行）；但**任务间**——用户回来说"这个不对吧"——今天没有任何升档机制。理想形态应该是 5 层而不是 4 层。

两件事在同一个反思链上，合并为 FEATURE_103 一并修正。

### 设计

#### 第 1 件：Scout profile 校准

```ts
// 旧
reasoning: { default: 'quick',    max: 'balanced', escalateOnRevise: false }
// 新（FEATURE_103）
reasoning: { default: 'balanced', max: 'deep',     escalateOnRevise: false }
```

- `default: balanced`：Scout 是级联决策点，地板提到 balanced 与 SA / Planner / Generator / Evaluator 一致
- `max: deep`：用户显式 `--reasoning deep` 时 Scout 也跟上（之前被 `max: balanced` 卡死）
- `escalateOnRevise: false`（保持）：Scout 没有 revise 回路，emit 一次就 handoff

代价：Scout 每次 AMA 入口多一档思考；收益：分类准确率 + obligation 质量 + downstream hint 准确度都提升，下游因 Scout 误判浪费一整轮的概率下降。

#### 第 2 件：L5 user-followup escalate

把 FEATURE_078 的 4 层链扩展到 5 层：

```
L1  user ceiling                                 (--reasoning <mode>)
L2  agent declaration default + max              (Agent.reasoning)
L3  Scout downstream hint                        (downstream_reasoning_hint)
L4  evaluator-revise escalate (任务内)           (escalateThinkingDepth)
L5  user-followup escalate (任务间)  ← FEATURE_103
```

L5 触发条件（命中任一即升档单步）：

| 类别 | 触发条件 | 例子 |
|---|---|---|
| **doubt**（怀疑/反驳） | prompt 含怀疑标记词 **AND** session 有 prior assistant turn | "这个不对吧" / "are you sure" / "搞错了" |
| **deepen**（加深要求） | prompt 含加深标记词，**不要求**有 prior turn | "仔细分析" / "think harder" / "再想想" |

doubt 类别的 prior-turn 门是反 false-positive：第一轮带"wrong"的 prompt 可能只是引用文本（"wrong file path 报错怎么办"），还没有 assistant 回答可以反驳。deepen 类别不需要——用户首轮就说"仔细看看"，意图明确。

升档规则：单步 rank+1，被 deep 顶住；`off` 是 kill switch，不升。

```
off (sacrosanct)   ─┐
auto → quick        │  rank +1 once
quick → balanced    │  per (runKodaX | runManagedTaskViaRunner) call
balanced → deep     │
deep (fixed point) ─┘
```

关键不变量：
- L5 **不**直接跳到 max（避免一次用完头寸）；多轮持续不满才会从 balanced→deep 之后保持 deep
- L5 **不**绕过 L1 hard cap：用户显式 `--reasoning balanced` 时 L5 仍可升到 deep——这个看似破例其实不是，L5 调整的就是 L1 本身（"用户原来说 balanced，但又说不对，那本意应该是再深点"），不是 L2/L3 偷偷绕过 L1
- L5 **不**重叠 L4：L4 在任务内多轮间触发；L5 在任务边界（runKodaX / runManagedTaskViaRunner 入口）触发一次

| `coding/task-engine/runner-driven.ts:runManagedTaskViaRunner` | 同上，wrap inner |

**词典**（保守覆盖；歧义短语如 bare "wrong" / "not" 故意不收，避免对引用文本 / 代码标识符 false-positive）：

- doubt（中文）：`不对` / `错了` / `有问题` / `真的吗` / `你确定` / `不是这样` / `弄错了` / `搞错了` / `搞反了` / `这不对` / `不正确` / `答错` / `回答错`
- doubt（英文）：`that's wrong` / `that is wrong` / `that's not right` / `that is not right` / `are you sure` / `not really` / `this is wrong` / `this is incorrect` / `that's incorrect` / `that is incorrect` / `you're wrong` / `you are wrong`
- deepen（中文）：`仔细` / `深入` / `认真` / `再看看` / `再想想` / `想清楚` / `用心` / `深度分析` / `仔细分析` / `认真分析` / `彻底`
- deepen（英文）：`think harder` / `think more carefully` / `look more carefully` / `dig deeper` / `be thorough` / `more careful` / `more carefully` / `reconsider` / `reexamine` / `re-examine`

匹配规则：substring，case-insensitive（双向：原文 + lower-cased 各匹配一次，CJK 字符 .toLowerCase 不变所以不影响）。doubt 在 deepen 之前检查（doubt 命中先返回，避免 "不对，请仔细看看" 被误归为 deepen）。

### Testing

- **单元测试**（`reasoning-feature-103.test.ts`，30 active）：覆盖 doubt 中英文 / deepen 中英文 / null 路径 / round-gate 行为 / `escalateUserCeiling` 5 档 invariants / `applyFollowupEscalation` end-to-end / `applyFollowupEscalationToOptions` 集成
- **回归**：FEATURE_078 21 + CAP-094 7 + coding-agents 15 全绿；全仓 npm test 仅遗留 4 个 v0.7.29-pre-existing flake（2 tracker-consistency metadata drift + 2 provider-policy parallel-run flake，单跑通过）
- **反向审计 grep gate**（`legacy parity restore` / `inadvertently dropped` / `inadvertently lost` / `preserves SA semantic`）：4 类全 0 命中

不涉及 prompt 修改（仅深度档位调整），故无需 prompt eval。Prompt eval 基础设施作为独立的 FEATURE_104 单独立项。

### Risk / Mitigation

| Risk | Mitigation |
|---|---|
| Scout balanced 起步显著拉高 AMA 入口 token | 真实代价 1 档，且 Scout 每次只跑一次；下游因误判减少 1 整轮的代价远高于 1 档 Scout |
| L5 doubt 词典 false-positive | (a) doubt 类别强制要求 prior assistant turn 才生效；(b) 词典只收明确的多词短语（不收 bare "wrong" / "not"）；(c) 升档单步，不直接跳 max |
| L5 与 L4 触发冲突 | L4 在 task 内多轮，L5 在 task 边界一次；不会同一 round 两次升档 |
| 用户显式 `--reasoning balanced` 被 L5 突破 | 这是 L5 设计的核心语义——L5 调整的就是 L1 本身（"用户表达了想再深点"），不是绕过 L1。如果用户真的想锁死 balanced，应该用 `--reasoning off` 后单独 prompt 控制——但那是边缘 case |
| L5 应用后多次 re-resolve 重复升档 | `applyFollowupEscalationToOptions` 返回新 options 一次性写入 `reasoningMode`，所有下游 `resolveReasoningMode(options)` 看到的都是已升档值；不会双倍升档 |

### Non-Goals

- **不**做 topic-continuity 检测（"是否在追问同一个问题"是经典 hard 问题，做不好不如不做）
- **不**让 L5 一步跳 max（多轮持续不满才考虑别的策略）
- **不**重写 L1-L4（FEATURE_078 的契约不变，L5 是纯 additive 第 5 层）
- **不**触碰 `--reasoning off` 的 kill switch 语义（off 永远 off）

## FEATURE_104: Prompt-Eval Harness Module (v2 — Quantitative Benchmark)

> 用户在 v0.7.29 review 期间提出："未来做 prompt 任何改动的时候，都依据
> provider/model 的测试情况来看 prompt 怎么改进更好，也就是这个可以当一个
> 专门用来测试 prompt 的模块。"
>
> v1 落地后用户分享了 LiveCanvas 项目里的一份 prompt benchmark recipe
> （headline 结果：单点 prompt 改动把关键 correctness assertion 通过率
> 从 22% 提到 100%，跨 8 个 provider × 3 runs），指出 v1 实现可能不够。
> 据此把 FEATURE_104 升到 v2，一个版本内兑现。

### 背景

仓库内此前已有 7 个 `tests/*.eval.ts` 文件（identity-roundtrip、dispatch-prompt-comparison、construction-llm-review 等），跑真实 LLM provider 验证 prompt 行为。`vitest.eval.config.ts` 提供 `npm run test:eval` 入口，眼缺三件事：

1. **provider/model 信息每个 eval 文件都自己 inline 一份**：v0.7.28 FEATURE_099 catalog refresh 把 `glm-5` 改名为 `glm-5.1`、删掉 `deepseek-chat`，得逐个 eval 文件改一遍——drift 风险高
2. **没有共享的 alias 短名**：用户希望以 `zhipu/glm51` / `ds/v4flash` 这种短名引用，便于在 eval 表头和日志里读
3. **没有"convention"约束**：哪些 PR 必须配 eval，哪些不需要，写 eval 该用什么模式（one-shot vs A/B），没有公认规范

FEATURE_104 把这三件事一次性收敛为 `benchmark/harness/` 模块（v0.7.29 后期 restructure：`tests/prompt-eval/` → `benchmark/`，docs + datasets 分目录）。

### v1 设计（已落，留给 lightweight probe 用；v0.7.29 后期 restructure 后位于 benchmark/）


### v2 升级（按 LiveCanvas recipe 补缺口，但 KodaX 场景做 1 项关键 deviation）

LiveCanvas recipe 里 6 个 anti-pattern 与 v1/v2 对照：
1. Speed 线性从 0 计分 → **KodaX 不适用**（见下方关键 deviation）
2. Style 评分但没 correctness → ✗ v1 judge 是 flat pass/fail → ✓ v2 加 `JudgeCategory` 维度
3. 不持久化 raw output → ✗ v1 只在内存返回 → ✓ v2 加 `persist.ts`
4. 单 run 决策 → ✗ v1 每个 cell 跑一次 → ✓ v2 默认 n=3 + variance 跟踪
5. assertion 太窄 → ✓ judges 工厂支持，可控
6. 整段 prompt 重写 → 文档/工作流约束，由 README 兜住

#### KodaX 关键 deviation：质量是唯一打分维度，延迟不参与排名

LiveCanvas 是交互式 UI 生成器——用户在等结果，慢但对的 provider 比快但对的体验差。所以 LiveCanvas recipe 把 quality 和 speed 合成 composite。

KodaX 不一样：用户按 Enter 后可以去吃饭，**慢但正确严格优于快但错误**。把 latency 折算进排名分数会导致快但错的方案被过度奖励，违反 coding agent 的本质。所以：

- 删除 `speedScore` / `DEFAULT_SPEED_IDEAL_MS` / `DEFAULT_SPEED_CEILING_MS` / `DEFAULT_COMPOSITE_WEIGHTS`
- `BenchmarkCellSummary` 移除 `speed` 和 `composite` 字段
- `variantsDominantOnEveryModel` 改用 `quality` 比较
- REPORT.md §3 标题从 "composite" 改为 "quality 0-100"
- REPORT.md §5 标题从 "Time analysis" 改为 "Latency observed (informational only — not part of scoring)"
- REPORT.md §7 ranking 改用 quality 平均值

`durationMs` 和 `duration` 统计仍保留，但是**纯诊断**用——比如 spotting 某个 provider 挂了 10 分钟还在转圈。


### v2 关键设计决策

| 决策 | 取舍 |
|---|---|
| n=3 默认 | LiveCanvas recipe "minimum for variance to be meaningful"。1 run 是 anti-pattern 4 |
| **质量是唯一打分维度** | KodaX coding agent 场景：慢但对严格优于快但错。LiveCanvas 的 composite (quality + speed) 不适用——会奖励快错方案 |
| Latency 仍跟踪但只做诊断 | 看 outlier（hang 住的 provider）有用，但不进 rank |
| `format` category 卡门 | 不解析就是 0 quality，不能让格式错误的 output 拿其他维度的分 |
| `correctness` 是 judge 默认 category | 推动 casual judge 写法朝最重要的维度走 |
| Persistence 路径默认 `benchmark/results/<timestamp>/` | 时间戳隔离、`.gitignore` 兜底（commit 是 opt-in 用作 baseline）；docs 和 datasets 在 `benchmark/` 同级目录 version-tracked，结果不 track |
| Failure patterns 按频率排序在 §8 | recipe "the top of this list is your prompt-improvement opportunity" |
| 3-point 内不算赢 | 内嵌为 §7 prose caveat。harness 不强制断言（eval-file 自决） |
| `variantsDominantOnEveryModel` 按 quality 而不是 composite | 严格 quality ≥ 才算"v2 真的赢了"，避免单 cell 的 lucky win |

### 跳过的（KodaX 不需要 vs LiveCanvas）

- **Speed scoring + composite ranking**：LiveCanvas 把 latency 折算进排名（0.85 × quality + 0.15 × speed）。KodaX 是 coding agent 场景，慢但对严格优于快但错，把 latency 进 rank 会奖励错误。**质量是唯一指标**
- **AST-based scorer**：LiveCanvas 用 Babel 做 React/JSX 结构判断；KodaX 的 prompt 输出多样性更高，regex + structural 判 MVP 够用，3+ 真实 case 后再考虑
- **Routing table generation**：LiveCanvas 自动生成 per-slot 路由表；KodaX 路由是另一个 feature 的事
- **Token / cost tracking**：eval 是手动 trigger，不在 budget 内；recipe 也建议跳过
- **LLM-as-judge**：少于 3 真实 case，不抽

## Appendix A: FEATURE_100 Capability Inventory

> Originally a sibling document (`v0.7.29-capability-inventory.md`); merged here on 2026-05-01 to comply with the `features/v{VERSION}.md` convention. Cross-referenced from [§FEATURE_100 P1](#p1--capability-inventory--goldens-lock无代码变更).
>
> Status: P1 in progress — see [Coverage Status](#coverage-status) for what's locked vs. pending.
>
> Last updated: 2026-04-26

This inventory enumerates every capability `runKodaX` (`packages/coding/src/agent.ts`) provides today. Each entry is a CAP-XXX record. Refactor PRs in P2-P4 must demonstrate that every CAP entry has a wire-up in the new architecture (substrate middleware / Runner-level lifecycle / Agent declaration middleware) — or an explicit ADR-rationale for non-support.

**This document is a P3-merge gate**: a refactor PR that fails to wire any CAP entry cannot merge.

---

### Methodology

1. **Source mining**: `packages/coding/src/agent.ts` (3164 lines, current SA entry) is read function-by-function. Helpers and `runKodaX` body branches are recorded as separate CAP entries when they represent independent capability boundaries (independent triggers / independent outputs).
2. **Parity-restore mining**: `packages/coding/src/task-engine/runner-driven.ts` is grep'd for `legacy.*runKodaX | inadvertently dropped | parity restore | Legacy agent\.ts | matching legacy`. Each match marks a capability that was de-facto SA-exclusive (because AMA route bypassed `runKodaX`) and got patched back. These entries carry `Risk: HIGH_RISK_PARITY` — they are the empirical baseline of "things that have already been lost in a similar refactor".
3. **Cross-validation**: each CAP entry must cite at least one `file:line` source location, **independently verified by grep against current `HEAD` at the timestamp shown next to the citation**. Legacy line numbers from runner-driven.ts comments (written at v0.7.26 / FEATURE_084 time) are treated as historical references only — agent.ts has been touched substantially since (FEATURE_085, FEATURE_086, v0.7.27/v0.7.28 commits), so any number copied from a legacy comment is re-verified before being recorded here. When a legacy citation has drifted, the inventory records both: the verified current line + a note indicating the legacy reference is stale.
4. **Utility-level helpers excluded**: pure data-shape constructors and trivial classification helpers (e.g., `createToolResultBlock` at `agent.ts:873` — a 9-line constructor returning `KodaXToolResultBlock`) are NOT recorded as CAP entries. The boundary is **independent capability vs. shared utility**: a capability has an independent trigger condition / independent observable effect; a utility is a shared shape constructor or pure predicate that has no behavior of its own. Utility-level helpers travel with their consumers in the migration. If a future reader is uncertain whether a particular helper warrants a CAP entry, default to including it — over-inventorying is safer than under-inventorying.

---

### Format spec

```
### CAP-XXX: <name>

- **Class**: 1 (substrate middleware) | 2 (Runner-level lifecycle/guardrail) | 3 (declarable opt-in middleware) | (deferred / non-supported with rationale)
- **Risk**: HIGH_RISK_PARITY | MEDIUM | LOW
- **Current location**: `path:line[-line]`
- **Trigger condition**: when this capability fires today
- **Inputs consumed**: state / args read
- **Outputs produced**: state / events / side effects emitted
- **Time-ordering constraint**: if it must run before/after some other capability, state it
- **Migration target**: new file path under `packages/coding/src/agent-runtime/...`
```
