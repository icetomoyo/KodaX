## 总览

v0.7.29 单一主题：把 `reasoning` 从**单个全局档位**改造成**分层决策链**。

动机来自一个悖论：Scout 要决定 harness 和下游 reasoning，但 Scout 自己也需要 reasoning 才能思考 —— 因此 Scout 不能成为自己 reasoning 档位的唯一决定者。与此同时，单一 `--reasoning` 全局档又压平了"Scout 要快 / Generator 要深 / Evaluator 要准"这些合理差异。

这一版把 reasoning 做成"用户给上限 + 角色声明默认 + Scout 给下游建议 + Evaluator 可动态升档"的四级模型，复用既有 `escalateThinkingDepth` 机制，把它串到 Agent 声明式定义（FEATURE_080 已引入）里。

## FEATURE_078: Role-Aware Reasoning Profiles

**Status**: Planned  
**Priority**: High  
**Category**: Internal

### 背景

当前 reasoning 模型是**平面单档**：

- CLI 暴露 `--reasoning off | auto | quick | balanced | deep` + `--thinking` 别名
- 所有角色（Scout / Planner / Generator / Evaluator）共享一个档位
- Provider 层有能力映射（budget / effort / toggle / none）
- FEATURE_021 引入了 thinking budget，FEATURE_039 把 budget 透明化给用户，但都是全局层级

**问题**：
1. Scout 是 AMA 的路由权威（FEATURE_061），Scout 的 reasoning 档位必须在 Scout 跑之前就确定，不能由 Scout 自己决定
2. "所有下游角色用同一档"不符合常识 — Generator 写代码和 Evaluator 核对证据对思考深度的需求不同
3. 用户 `--reasoning deep` 的真实意图是"我想要深度思考体验"，而不是"Scout 判题也要 deep"（会浪费 token）

FEATURE_080 (v0.7.23) 已引入 `Agent.reasoning: AgentReasoningProfile` 字段作为占位；FEATURE_084 (v0.7.26) 已把 Scout / Generator / Evaluator 重写成 Agent 实例 —— 所有前置到位。

### 设计

#### 四级决策链

```
L1 用户意图上限 (User Ceiling)
      │  CLI --reasoning / API options.reasoning
      │  意图：我最多允许到这一档；也偏好这个档作为默认
      ▼
L2 角色默认档位 (Role Default)
      │  Agent definition 的 reasoning: { default, max? } 字段
      │  Scout: default=quick
      │  Planner: default=balanced
      │  Generator: default=balanced
      │  Evaluator: default=balanced
      ▼
L3 Scout 下游建议 (Scout Hint, optional)
      │  Scout 输出 downstream_reasoning_hint
      │  非绑定；是 L4 的输入信号
      ▼
L4 Evaluator / revise-loop 动态升档 (Dynamic Escalation)
      │  复用 escalateThinkingDepth
      │  升档时 clamp 到 L1 ceiling，永不越界
      ▼
最终 per-round reasoning depth
```

**关键语义**：
- L1 是 **ceiling + bias**，不是"所有角色都用这档"
- L2 是 **Agent declaration 的一部分**，不是 task-engine 的逻辑
- L3 是 **hint，不是 binding**
- L4 **ceiling-clamped**，永远 `effective = min(L2_or_L3_or_L4, L1)`

#### Agent 定义字段（FEATURE_080 已预留）

```ts
export interface AgentReasoningProfile {
  default: ReasoningDepth           // 'off' | 'quick' | 'balanced' | 'deep'
  max?: ReasoningDepth              // 本 agent 愿意升到多深，默认 = userCeiling
  escalateOnRevise?: boolean        // 被 Evaluator 打回时是否自动升一档
}
```

#### 默认 profile（预置到 coding preset）

| Agent | default | max | escalateOnRevise |
|-------|---------|-----|------------------|
| Scout | quick | balanced | false |
| Planner | balanced | deep | true |
| Generator | balanced | deep | true |
| Evaluator | balanced | deep | false |

#### CLI / API 兼容

- `--reasoning <mode>` 语义收紧为 **ceiling + default bias**；CLI help 文案更新
- `--thinking` 继续作为 `--reasoning auto` 的别名
- Provider `reasoning-overrides` 系统不变（provider 能力映射正交）
- 旧 `config.json` 的 `reasoningMode` 自动 map 为 `reasoningCeiling`

#### 实现切片

1. **contract**：`AgentReasoningProfile` 类型已在 FEATURE_080 引入；本版只实装行为
2. **resolver**：`resolveReasoningMode` 改为 `resolveRoleReasoning(role, userCeiling, agentProfile, scoutHint?)`；保留旧函数作为 shim（v0.7.40 清理）
3. **Scout 输出**：managed-protocol 的 scout payload（FEATURE_084 已改成结构化）增加 `downstream_reasoning_hint` 字段
4. **escalate**：`escalateThinkingDepth` 增加 `ceiling` 参数，默认不越过
5. **migration**：config.json `reasoningMode` → `reasoningCeiling` 自动迁移（启动时静默完成，标 warning 一次）
6. **docs**：README / docs/ADR 更新 reasoning 章节

### Contract

**Public API**（新增）：
- `AgentReasoningProfile` type（已 v0.7.23 预留，本版实装行为）
- `resolveRoleReasoning(...)` 作为 L1-L4 链的统一入口

**Public API（变更）**：
- `--reasoning` 语义：从"all roles use this"收紧为"ceiling + bias"
- 对终端用户行为透明（默认 profile 产生的效果和旧的近似），但外部 SDK 消费者若直接读 resolveReasoningMode 行为会微调

**Internal**：
- `reasoning.ts` 的 `resolveReasoningMode` 标 `@deprecated`，内部重定向到新 resolver
- `escalateThinkingDepth` 签名增加 `ceiling?: ReasoningDepth`

### Non-Goals

- **不**引入 per-task reasoning override（未来需求）
- **不**改 provider 的 native capability mapping（`reasoning-overrides.ts` 职责，正交）
- **不**让用户直接配置每个角色的 reasoning（角色化是系统内部决策）

### Dependencies

- **前置**：
  - FEATURE_080 (v0.7.23) — `AgentReasoningProfile` 字段引入
  - FEATURE_084 (v0.7.26) — Scout / Generator / Evaluator 已是 Agent 实例，reasoning 字段可填
- **后置**：
  - FEATURE_089 (v0.7.31) — Agent generation 时可生成带 reasoning profile 的 Agent
  - FEATURE_090 (v0.7.32) — self-modify 可以改 reasoning profile

### Testing

- 单元：`resolveRoleReasoning` 覆盖 L1-L4 组合（ceiling 压低、角色 default 被 hint 覆盖、escalate 被 clamp）
- 集成：Scout → Generator 一次 revise 升档路径，验证 ceiling clamp 生效
- 迁移：旧 config.json 读入后产生的 effective depth 和旧版本 snapshot 一致

### Migration

- 用户 zero-action（config / flag 自动兼容）
- Extension / SDK 消费者直接调用 `resolveReasoningMode` 会见到 deprecation 警告，迁移到 `resolveRoleReasoning`

### Risk

- **主要风险**：L1 ceiling 压低时，某些 revise loop 可能需要升档但被 clamp，任务卡住
  - 缓解：ceiling 达到后 Evaluator 收到信号，可 abandon 或 handoff 给 Planner；不静默失败
- **次要风险**：默认 profile 初始值偏差
  - 缓解：一次 end-to-end eval 跑（借用 /eval 框架）验证默认值合理

### Timeline

v0.7.29 仅此一 feature。completion + eval regression 后发版。
