# v0.7.29 Feature Designs

> 中文标题：`v0.7.29 Feature 设计`
>
> Version: `v0.7.29`
>
> Status: Planned (not yet released)
>
> Theme: Per-dimension layering — role-aware reasoning profiles + model-aware runtime limits

> Migration note (2026-04-19):
> `FEATURE_078` was originally planned for `v0.7.35`. Under Plan B compression, it is moved to `v0.7.29` — immediately after `FEATURE_084/085` AMA rewrite (v0.7.26) and before self-construction tier 3 (FEATURE_089 v0.7.31). This ordering lets reasoning profiles settle on the new Agent primitive before Agent generation starts using them.

---

## 总览

v0.7.29 单一主题：把 `reasoning` 从**单个全局档位**改造成**分层决策链**。

动机来自一个悖论：Scout 要决定 harness 和下游 reasoning，但 Scout 自己也需要 reasoning 才能思考 —— 因此 Scout 不能成为自己 reasoning 档位的唯一决定者。与此同时，单一 `--reasoning` 全局档又压平了"Scout 要快 / Generator 要深 / Evaluator 要准"这些合理差异。

这一版把 reasoning 做成"用户给上限 + 角色声明默认 + Scout 给下游建议 + Evaluator 可动态升档"的四级模型，复用既有 `escalateThinkingDepth` 机制，把它串到 Agent 声明式定义（FEATURE_080 已引入）里。

## FEATURE_078: Role-Aware Reasoning Profiles

**Status**: Planned  
**Priority**: High  
**Category**: Internal

### 背景

当前 reasoning 模型是**平面单档**：

- CLI 暴露 `--reasoning off | auto | quick | balanced | deep` + `--thinking` 别名
- 所有角色（Scout / Planner / Generator / Evaluator）共享一个档位
- Provider 层有能力映射（budget / effort / toggle / none）
- FEATURE_021 引入了 thinking budget，FEATURE_039 把 budget 透明化给用户，但都是全局层级

**问题**：
1. Scout 是 AMA 的路由权威（FEATURE_061），Scout 的 reasoning 档位必须在 Scout 跑之前就确定，不能由 Scout 自己决定
2. "所有下游角色用同一档"不符合常识 — Generator 写代码和 Evaluator 核对证据对思考深度的需求不同
3. 用户 `--reasoning deep` 的真实意图是"我想要深度思考体验"，而不是"Scout 判题也要 deep"（会浪费 token）

FEATURE_080 (v0.7.23) 已引入 `Agent.reasoning: AgentReasoningProfile` 字段作为占位；FEATURE_084 (v0.7.26) 已把 Scout / Generator / Evaluator 重写成 Agent 实例 —— 所有前置到位。

### 设计

#### 四级决策链

```
L1 用户意图上限 (User Ceiling)
      │  CLI --reasoning / API options.reasoning
      │  意图：我最多允许到这一档；也偏好这个档作为默认
      ▼
L2 角色默认档位 (Role Default)
      │  Agent definition 的 reasoning: { default, max? } 字段
      │  Scout: default=quick
      │  Planner: default=balanced
      │  Generator: default=balanced
      │  Evaluator: default=balanced
      ▼
L3 Scout 下游建议 (Scout Hint, optional)
      │  Scout 输出 downstream_reasoning_hint
      │  非绑定；是 L4 的输入信号
      ▼
L4 Evaluator / revise-loop 动态升档 (Dynamic Escalation)
      │  复用 escalateThinkingDepth
      │  升档时 clamp 到 L1 ceiling，永不越界
      ▼
最终 per-round reasoning depth
```

**关键语义**：
- L1 是 **ceiling + bias**，不是"所有角色都用这档"
- L2 是 **Agent declaration 的一部分**，不是 task-engine 的逻辑
- L3 是 **hint，不是 binding**
- L4 **ceiling-clamped**，永远 `effective = min(L2_or_L3_or_L4, L1)`

#### Agent 定义字段（FEATURE_080 已预留）

```ts
export interface AgentReasoningProfile {
  default: ReasoningDepth           // 'off' | 'quick' | 'balanced' | 'deep'
  max?: ReasoningDepth              // 本 agent 愿意升到多深，默认 = userCeiling
  escalateOnRevise?: boolean        // 被 Evaluator 打回时是否自动升一档
}
```

#### 默认 profile（预置到 coding preset）

| Agent | default | max | escalateOnRevise |
|-------|---------|-----|------------------|
| Scout | quick | balanced | false |
| Planner | balanced | deep | true |
| Generator | balanced | deep | true |
| Evaluator | balanced | deep | false |

#### CLI / API 兼容

- `--reasoning <mode>` 语义收紧为 **ceiling + default bias**；CLI help 文案更新
- `--thinking` 继续作为 `--reasoning auto` 的别名
- Provider `reasoning-overrides` 系统不变（provider 能力映射正交）
- 旧 `config.json` 的 `reasoningMode` 自动 map 为 `reasoningCeiling`

#### 实现切片

1. **contract**：`AgentReasoningProfile` 类型已在 FEATURE_080 引入；本版只实装行为
2. **resolver**：`resolveReasoningMode` 改为 `resolveRoleReasoning(role, userCeiling, agentProfile, scoutHint?)`；保留旧函数作为 shim（v0.7.40 清理）
3. **Scout 输出**：managed-protocol 的 scout payload（FEATURE_084 已改成结构化）增加 `downstream_reasoning_hint` 字段
4. **escalate**：`escalateThinkingDepth` 增加 `ceiling` 参数，默认不越过
5. **migration**：config.json `reasoningMode` → `reasoningCeiling` 自动迁移（启动时静默完成，标 warning 一次）
6. **docs**：README / docs/ADR 更新 reasoning 章节

### Contract

**Public API**（新增）：
- `AgentReasoningProfile` type（已 v0.7.23 预留，本版实装行为）
- `resolveRoleReasoning(...)` 作为 L1-L4 链的统一入口

**Public API（变更）**：
- `--reasoning` 语义：从"all roles use this"收紧为"ceiling + bias"
- 对终端用户行为透明（默认 profile 产生的效果和旧的近似），但外部 SDK 消费者若直接读 resolveReasoningMode 行为会微调

**Internal**：
- `reasoning.ts` 的 `resolveReasoningMode` 标 `@deprecated`，内部重定向到新 resolver
- `escalateThinkingDepth` 签名增加 `ceiling?: ReasoningDepth`

### Non-Goals

- **不**引入 per-task reasoning override（未来需求）
- **不**改 provider 的 native capability mapping（`reasoning-overrides.ts` 职责，正交）
- **不**让用户直接配置每个角色的 reasoning（角色化是系统内部决策）

### Dependencies

- **前置**：
  - FEATURE_080 (v0.7.23) — `AgentReasoningProfile` 字段引入
  - FEATURE_084 (v0.7.26) — Scout / Generator / Evaluator 已是 Agent 实例，reasoning 字段可填
- **后置**：
  - FEATURE_089 (v0.7.31) — Agent generation 时可生成带 reasoning profile 的 Agent
  - FEATURE_090 (v0.7.32) — self-modify 可以改 reasoning profile

### Testing

- 单元：`resolveRoleReasoning` 覆盖 L1-L4 组合（ceiling 压低、角色 default 被 hint 覆盖、escalate 被 clamp）
- 集成：Scout → Generator 一次 revise 升档路径，验证 ceiling clamp 生效
- 迁移：旧 config.json 读入后产生的 effective depth 和旧版本 snapshot 一致

### Migration

- 用户 zero-action（config / flag 自动兼容）
- Extension / SDK 消费者直接调用 `resolveReasoningMode` 会见到 deprecation 警告，迁移到 `resolveRoleReasoning`

### Risk

- **主要风险**：L1 ceiling 压低时，某些 revise loop 可能需要升档但被 clamp，任务卡住
  - 缓解：ceiling 达到后 Evaluator 收到信号，可 abandon 或 handoff 给 Planner；不静默失败
- **次要风险**：默认 profile 初始值偏差
  - 缓解：一次 end-to-end eval 跑（借用 /eval 框架）验证默认值合理

### Timeline

v0.7.29 仅此一 feature。completion + eval regression 后发版。

## FEATURE_098: Per-Model Context Window & Output Token Limits

**Status**: Planned
**Priority**: High
**Category**: Internal

### 背景

KodaX 的 `KodaXModelDescriptor` 早就声明了 `contextWindow` 与 `maxOutputTokens` 字段（[packages/ai/src/types.ts:303-310](../../packages/ai/src/types.ts)），但全代码库**无人读取**：

- [base.ts:242](../../packages/ai/src/providers/base.ts) `getContextWindow()` 只读 `this.config.contextWindow`，不接 model 参数
- [base.ts:80](../../packages/ai/src/providers/base.ts) `getEffectiveMaxOutputTokens()` 同样只读 provider 级
- 4 处 compaction 调用点（[agent.ts:1701](../../packages/coding/src/agent.ts)、[commands.ts:317](../../packages/repl/src/interactive/commands.ts)、[repl.ts:397](../../packages/repl/src/interactive/repl.ts)、[managed-task/compaction.ts:83](../../packages/coding/src/task-engine/_internal/managed-task/compaction.ts)）全部 `provider.getContextWindow?.()`，未透传当前激活 model
- wire 层（[anthropic.ts:129/455](../../packages/ai/src/providers/anthropic.ts)、[openai.ts:232](../../packages/ai/src/providers/openai.ts)）调用 `getEffectiveMaxOutputTokens()` 时同样不传 model

后果：同 provider 不同 model 真实窗口差异大时，KodaX 一刀切。当前已确认的偏差：

| Provider | Provider 级 contextWindow | 真实差异 model |
|---|---:|---|
| `kimi` | 256_000 | `k2.5` 实际 128k |
| `qwen` | 256_000 | `qwen3.5-plus` 实际 1M |
| `zhipu` / `zhipu-coding` | 200_000 | `glm-5-turbo` 实际 128k |

切到 `k2.5` / `glm-5-turbo` 时压缩 trigger 虚高 → 实际撑爆服务端；切到 `qwen3.5-plus` 时白白浪费 75% 潜在窗口。

### 关键洞察：架构能力已就位

项目里 **2 个对称字段**已经走 model 级 lookup 路径成功工作：

- `thinkingBudgetCap` —— anthropic.opus=28k / haiku=10k 已差异化生效（[registry.ts:188-189](../../packages/ai/src/providers/registry.ts)）
- `reasoningCapability` —— [base.ts:149](../../packages/ai/src/providers/base.ts) 的 `getConfiguredReasoningCapability(modelOverride?)` 模式

两者的查找逻辑就是要复刻的样板。**不是新增能力，是把没接进来的两根线接上**。

### 设计

#### 数据层（无破坏性）

`KodaXModelDescriptor` 已存在的字段 (`contextWindow` / `maxOutputTokens`) 直接用起来。
`KodaXCustomProviderConfig.models` 从 `string[]` 升级为 `Array<string | KodaXModelDescriptor>`：

```ts
// 兼容旧字面量
"models": ["codellama", "mistral"]

// 新对象格式
"models": [
  { "id": "codellama", "contextWindow": 16000, "maxOutputTokens": 4096 },
  { "id": "mistral",   "contextWindow": 32000 }
]
```

#### 解析层（base.ts）

升级两个 effective getter，按以下优先级查找：

**`getEffectiveContextWindow(model?: string): number`**

1. `descriptor.contextWindow`（model-level，新增）
2. `config.contextWindow`（provider-level，现有）
3. `200_000`（兜底）

注：用户级 `compaction.contextWindow` 仍由调用点叠加做 manual override，最高优先级。

**`getEffectiveMaxOutputTokens(model?: string): number`**（升级现有签名）

1. `setMaxOutputTokensOverride()` 一次性（agent escalation/recovery，现有）
2. env `KODAX_MAX_OUTPUT_TOKENS`（现有）
3. `descriptor.maxOutputTokens`（model-level，新增）
4. `config.maxOutputTokens`（provider-level，现有）
5. `KODAX_MAX_TOKENS`（兜底）

`getContextWindow()` 保留作 ergonomic 包装，内部调用 `getEffectiveContextWindow(this.config.model)`。

#### 透传层（5 处调用点）

把当前激活 model 传进 effective getter：

- [anthropic.ts:129/455](../../packages/ai/src/providers/anthropic.ts) — wire-level max_tokens
- [openai.ts:232](../../packages/ai/src/providers/openai.ts) — 同上
- [agent.ts:1701](../../packages/coding/src/agent.ts) — auto-compaction trigger
- [commands.ts:317](../../packages/repl/src/interactive/commands.ts) — `/compact` 命令
- [repl.ts:397](../../packages/repl/src/interactive/repl.ts) — REPL trigger
- [managed-task/compaction.ts:83](../../packages/coding/src/task-engine/_internal/managed-task/compaction.ts) — managed task 内部

#### 数据修复（registry.ts）

只在**真有差异**的 model 上加描述符 override，避免冗余：

- `kimi.models`: `{ id: 'k2.5', contextWindow: 128_000 }`（k2.6 与 provider 级一致，不加）
- `qwen.models`: `[{ id: 'qwen3.5-plus', contextWindow: 1_000_000 }]`
- `zhipu.models` / `zhipu-coding.models`: `glm-5-turbo` 加 `contextWindow: 128_000`

#### 自定义 provider 同步升级

[custom-provider.ts:43-61](../../packages/ai/src/providers/custom-provider.ts) `buildProviderConfig` 接受新格式（描述符数组）+ 字面量数组兼容：

```ts
const models = custom.models?.map(entry =>
  typeof entry === 'string' ? { id: entry } : entry
);
```

[config.example.jsonc](../../config.example.jsonc) 注释更新示例 + 补充说明：用户级 `compaction.contextWindow` 仍可作为 manual override，最高优先级；新部署只在自定义 provider 没声明精确 model 数据时使用。

### 实施切片（6 commit）

切片之间无强依赖，每个独立可测、可回滚：

1. `feat(ai): per-model contextWindow/maxOutputTokens lookup in base.ts`
2. `feat(ai): allow KodaXModelDescriptor objects in custom provider models[]`
3. `fix(ai): pin true context windows for kimi.k2.5 / qwen / zhipu.glm-5-turbo`
4. `refactor(coding,repl): pass active model at compaction call sites`
5. `refactor(ai): pass active model at wire layer (anthropic/openai)`
6. `docs: clarify compaction.contextWindow as manual override`

### 测试

- `base.test.ts`：`getEffectiveContextWindow` / `getEffectiveMaxOutputTokens` 的 model 级 fallback 链
- `registry.test.ts`：kimi.k2.5、qwen.qwen3.5-plus、zhipu.glm-5-turbo 的 per-model 数据断言
- `compaction.test.ts`：trigger 在切换 model 后正确变化（小窗 model 应该更早触发）
- `custom-providers.test.ts`：描述符格式 + 旧字面量兼容

### Risk

- **主要风险**：现有用户的 `compaction.contextWindow` 配置语义变化（从"覆盖 provider"到"manual override"），但行为完全一致 —— 仍然是最高优先级
- **次要风险**：自定义 provider 的 `models[]` 格式扩展可能让用户配置混淆。缓解：兼容旧字面量；新格式仅在用户主动用对象时启用

### Migration

- 内置 provider 切换 model 时窗口自动正确（无需用户改配置）
- 自定义 provider 想用新能力：把 `"models": ["x", "y"]` 改成对象数组并填字段；旧配置不报错
- `compaction.contextWindow` 用户配置无破坏性变更

### Timeline

与 FEATURE_078 同版发布。两者实施面无交集（reasoning vs limits），可并行推进。
