## FEATURE_032: JSON 输出模式 (--mode json)

**Status**: Completed
**Priority**: Medium
**Category**: Enhancement

### 1. 需求概述

类比 pi-mono 的 `--mode json` 能力，为 KodaX 增加 JSON Lines 输出模式。将 agent 运行期间的所有事件（文本增量、工具调用、工具结果、错误、完成等）以 JSONL 格式输出到 stdout，便于外部程序解析和集成。

**Positioning**:
- 这是 KodaX 面向 shell、脚本和 CI 的轻量 JSONL 输出模式
- 它不承担 ACP/RPC 这类双向协议职责；编辑器和 IDE 集成继续使用 ACP Server（FEATURE_040）
- 事件 schema 以脚本消费稳定性和可读性优先，而不是直接暴露内部运行时事件模型

**Goals**:
1. CLI 标志 `--mode json` 启用 JSON 输出模式（与 `-p` print 模式互斥）
2. 结构化事件类型覆盖完整 agent 生命周期
3. 与现有 `runKodaX` events API 对齐，JSON 事件为 events 的序列化形式
4. 错误输出到 stderr，JSONL 数据输出到 stdout（管道友好）
5. 退出码语义：0=成功, 1=错误

**User Scenarios**:
- CI/CD 流水线中解析 agent 输出
- 外部脚本/程序集成 KodaX
- 调试和审计 agent 行为
- 与 Pi 的 JSON 模式保持相近心智模型，但不追求协议级兼容

**Inspired by**: [pi-mono --mode json](https://github.com/badlogic/pi-mono)

### 2. 影响范围

**需要创建的文件**:
- `packages/repl/src/ui/json-events.ts` - JSONL 事件输出器
- `packages/repl/src/ui/json-events.test.ts` - JSON 输出序列化测试
- `src/cli_option_helpers.test.ts` - CLI 模式校验测试

**需要修改的文件**:
- `src/kodax_cli.ts` - 增加 `--mode json` 选项、帮助文案、模式校验、最终 `run.result` 输出
- `src/cli_option_helpers.ts` - 输出模式解析、JSON 模式合法性校验、事件接线、session 处理
- `packages/repl/src/ui/index.ts` - 导出 JSON 事件输出器

**不需要修改的文件**:
- `packages/coding/src/agent.ts` - 继续复用现有 `KodaXEvents` 生命周期事件
- `packages/coding/src/types.ts` - 现有事件接口已足够承载 JSON 模式

### 3. 技术方案

#### 3.1 CLI 模式选择

- 新增 `--mode json`
- 默认仍为文本模式，无需显式指定 `--mode text`
- `--mode json` 与 `-p/--print` 互斥，提示用户改用位置参数传 prompt

#### 3.2 JSONL 事件输出

- 新增 `createJsonEvents()`，将 `runKodaX` 生命周期事件逐条序列化为 JSON Lines
- `stdout` 仅输出 JSONL 数据，便于脚本管道消费
- `stderr` 输出结构化错误记录，避免破坏 `stdout` 的 JSON 流
- 不复用 ACP session update 命名；CLI JSON mode 与 ACP server 明确分层

#### 3.3 模式约束

JSON 模式 v1 仅支持单次非交互执行，显式拒绝以下场景：

- `--init`
- `--team`
- `--auto-continue`
- `--session list/delete/delete-all`
- 不带 id 的 `--resume`
- 缺少位置参数 prompt

#### 3.4 最终结果事件

- 在根 CLI 成功返回后额外输出一条 `run.result`
- 用于向脚本方暴露 `success`、`signal`、`sessionId`、`limitReached` 等摘要元数据
- 保持细粒度生命周期事件仍来自 `KodaXEvents`

### 4. 接口契约

```bash
kodax --mode json "inspect auth flow"
kodax --mode json -c "continue the migration"
kodax --mode json -r session123 "review the latest changes"
```

示例 JSONL：

```json
{"type":"session.start","provider":"openai","sessionId":"session-123"}
{"type":"iteration.start","iter":1,"maxIter":200}
{"type":"text.delta","text":"Here is the summary..."}
{"type":"tool.start","id":"tool-1","name":"read","input":{"path":"README.md"}}
{"type":"tool.result","id":"tool-1","name":"read","content":"..."}
{"type":"iteration.end","iter":1,"maxIter":200,"tokenCount":42,"tokenSource":"estimate"}
{"type":"complete"}
{"type":"run.result","success":true,"sessionId":"session-123","limitReached":false}
```

### 5. 实现步骤

1. 为 root CLI 增加 `--mode json` 选项与帮助文案
2. 在 `cli_option_helpers.ts` 中增加输出模式解析和模式合法性校验
3. 新增 `createJsonEvents()`，把现有 `KodaXEvents` 映射为 JSONL
4. 将 JSON 输出器接入 `createKodaXOptions()`
5. 在 CLI 主流程追加 `run.result` 事件
6. 增加 JSON 输出器和 CLI 校验的单元测试
7. 通过 targeted tests + TypeScript build 验证

### 6. 验收标准

- [x] CLI 标志 `--mode json` 启用 JSON 输出模式
- [x] `--mode json` 与 `-p/--print` 互斥
- [x] 生命周期事件以 JSONL 写入 `stdout`
- [x] 错误记录写入 `stderr`
- [x] JSON 模式在不支持的 CLI 场景下给出明确错误
- [x] 为 JSON 输出序列化和模式校验补充单元测试

## FEATURE_037: API Token Usage 真实值优先 + 估算回退

**Status**: Completed
**Priority**: Medium
**Category**: Enhancement

### 1. 需求概述

建立统一的 Token Usage 获取策略：当 API 响应中包含 token usage 字段时，优先使用 API 返回的真实 token 数量；当 API 未返回有效 usage 字段时，自动回退到现有的 `js-tiktoken` 客户端估算。

**核心目标**:
1. **消除估算误差** — 使用 API 返回的真实 token 用量替代客户端估算，提升 compaction/context 管理的精确度
2. **零降级风险** — 所有现有 Provider 均支持返回 usage，估算仅作为极端情况的兜底
3. **统一接口** — 扩展 `KodaXStreamResult` 添加 `usage` 字段，所有 Provider 走同一数据通道
4. **优先 + 回退** — 消费端优先使用真实 usage，仅在缺失时回退到 `estimateTokens()`

**背景**:
- 所有 Provider API 均支持返回 usage 数据，但 KodaX 当前**零提取**
- CLI bridge 解析器（Gemini/Codex）已提取 usage，但在 `pseudo-acp-server.ts` 的 ACP 协议转换中**丢失**
- compaction/context 管理完全依赖客户端 `js-tiktoken` 估算，与 API 实际 token 计数存在偏差
- `KodaXStreamResult` 接口**没有 usage 字段**，真实 token 数据无法传播到上层

### 2. 影响范围

**需要修改的文件**:

| 文件 | 改动说明 |
|------|---------|
| `packages/ai/src/types.ts` | `KodaXStreamResult` 添加可选 `usage` 字段 |
| `packages/ai/src/providers/anthropic.ts` | 从 `message_start` + `message_delta` 提取 usage |
| `packages/ai/src/providers/openai.ts` | 添加 `stream_options`，提取最终 chunk usage |
| `packages/ai/src/cli-events/pseudo-acp-server.ts` | 修复 `CLICompleteEvent` usage 丢失 |
| `packages/ai/src/cli-events/acp-base.ts` | 接收并传播 usage 数据 |
| `packages/agent/src/tokenizer.ts` | 添加 `getTokens(messages, usage?)` 优先使用真实值 |
| `packages/agent/src/compaction/compaction.ts` | 优先使用 `usage.totalTokens` |

**不需要修改的文件**:
- `packages/ai/src/cli-events/gemini-parser.ts` — 已正确解析 usage
- `packages/ai/src/cli-events/codex-parser.ts` — 已正确解析 usage
- `packages/ai/src/cli-events/types.ts` — `CLICompleteEvent.usage` 接口已定义

### 3. 技术方案

#### 3.1 KodaXStreamResult 扩展

```typescript
// packages/ai/src/types.ts

export interface KodaXUsage {
  inputTokens: number;
  outputTokens: number;
  totalTokens: number;
}

export interface KodaXStreamResult {
  textBlocks: KodaXTextBlock[];
  toolBlocks: KodaXToolUseBlock[];
  thinkingBlocks: (KodaXThinkingBlock | KodaXRedactedThinkingBlock)[];
  usage?: KodaXUsage;  // 可选，API 未返回时为 undefined
}
```

#### 3.2 Anthropic 协议族 usage 提取

Anthropic streaming 的 usage 分散在两个事件中：

- `message_start` → `message.usage.input_tokens`（包含缓存命中信息）
- `message_delta` → `usage.output_tokens`

```typescript
// packages/ai/src/providers/anthropic.ts

// 在 stream() 方法中增加局部变量追踪 usage
let inputTokens = 0;
let outputTokens = 0;

// message_start 事件
if (event.type === 'message_start') {
  inputTokens = event.message.usage?.input_tokens ?? 0;
}

// message_delta 事件
if (event.type === 'message_delta') {
  outputTokens = event.usage?.output_tokens ?? 0;
}

// 流结束后返回 usage
return {
  textBlocks,
  toolBlocks,
  thinkingBlocks,
  usage: (inputTokens > 0 || outputTokens > 0) ? {
    inputTokens,
    outputTokens,
    totalTokens: inputTokens + outputTokens,
  } : undefined,
};
```

#### 3.3 OpenAI 协议族 usage 提取

OpenAI 需要显式启用 `stream_options.include_usage`，usage 出现在最终 chunk 中：

```typescript
// packages/ai/src/providers/openai.ts

// 创建请求时添加 stream_options
const stream = await this.client.chat.completions.create({
  ...params,
  stream: true,
  stream_options: { include_usage: true },  // 新增
});

// 流处理中追踪 usage
let usage: KodaXUsage | undefined;

for await (const chunk of stream) {
  // ... 现有 delta 处理逻辑 ...

  // 提取 usage（通常在最终 chunk）
  if (chunk.usage) {
    usage = {
      inputTokens: chunk.usage.prompt_tokens,
      outputTokens: chunk.usage.completion_tokens,
      totalTokens: chunk.usage.total_tokens,
    };
  }
}

return {
  textBlocks,
  toolBlocks,
  thinkingBlocks: [],
  usage,
};
```

#### 3.4 CLI Bridge usage 传播修复

当前 `pseudo-acp-server.ts` 的 `mapToAcpNotification` 对 `complete` 事件返回 `null`，丢失了 usage 数据。

修复方案：在 `CLICompleteEvent` 上携带 usage，通过 ACP session 层传递给 `KodaXAcpProvider`：

```typescript
// packages/ai/src/cli-events/pseudo-acp-server.ts

// complete 事件时将 usage 存储在 session 级别
case 'complete':
    this.pendingUsage = event.usage;
    break;

// 在会话结束时将 usage 传播给 provider
// 通过 AcpClient 的 close/receive 机制传递
```

```typescript
// packages/ai/src/cli-events/acp-base.ts

// stream() 返回时携带 usage
return {
  textBlocks,
  toolBlocks,
  thinkingBlocks,
  usage: this.collectedUsage,  // 从 ACP bridge 收集的 usage
};
```

#### 3.5 消费端优先 + 回退策略

```typescript
// packages/agent/src/tokenizer.ts

export function getTokenCount(
  messages: KodaXMessage[],
  usage?: KodaXUsage,
): number {
  // 优先使用 API 返回的真实值
  if (usage?.totalTokens && usage.totalTokens > 0) {
    return usage.totalTokens;
  }
  // 回退到客户端估算
  return estimateTokens(messages);
}
```

```typescript
// packages/agent/src/compaction/compaction.ts

// compaction 判断中优先使用 usage
function needsCompaction(
  messages: KodaXMessage[],
  threshold: number,
  usage?: KodaXUsage,
): boolean {
  const tokens = getTokenCount(messages, usage);
  return tokens >= threshold;
}
```

### 4. 接口契约

## FEATURE_041: Tool Output Guardrails and Context Overflow Protection

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### Overview

This feature moves the protection boundary forward from compaction to tool output handling so a single oversized tool result can no longer blow up the conversation history before compaction gets a chance to help.

### Implemented

1. Shared tool-output truncation utilities
   - Added `packages/coding/src/tools/truncate.ts`
   - Added shared head/tail truncation, byte/line limits, line shortening, and spill-file helpers

2. Global tool-result guardrail before history writes
   - Added `packages/coding/src/tools/tool-result-policy.ts`
   - Applied the guardrail in `packages/coding/src/agent.ts` before tool results are emitted to the UI or written back into `messages`

3. Safer `read`
   - Reworked `packages/coding/src/tools/read.ts`
   - Added binary-file rejection
   - Added bounded streaming reads instead of whole-file `readFile()`
   - Added large-file preflight guidance and explicit `offset/limit` continuation hints
   - Added long-line shortening

4. Safer `bash`
   - Reworked `packages/coding/src/tools/bash.ts`
   - Replaced unbounded stdout/stderr accumulation with bounded tail capture
   - Added tail-oriented truncation for large command output
   - Included stderr in timeout previews

5. Safer `write` / `edit` / `grep`
   - `write` and `edit` now summarize oversized diffs and spill the full diff when needed
   - `grep` now shortens overly long match lines and truncates oversized result sets with follow-up hints

6. Tool contract and prompt guidance updates
   - Updated tool descriptions in `packages/coding/src/tools/registry.ts`
   - Updated `packages/coding/src/prompts/system.ts` to steer the model toward `grep` + segmented `read`, and toward narrower follow-up calls after truncation hints

### Validation

- `npm run build:packages`
- `npx tsc -p tsconfig.json --noEmit`
- `npx vitest run packages/coding/src/tools/bash.test.ts packages/coding/src/tools/truncate.test.ts packages/coding/src/tools/tool-result-policy.test.ts packages/coding/src/tools/read.test.ts packages/coding/src/tools/grep.test.ts tests/kodax_core.test.ts tests/interactive.test.ts`

### Notes

- Compaction still exists as a downstream safety mechanism, but it is no longer the only line of defense against oversized tool output.
- Spill files are written to the user-level KodaX tool-results directory by default, not into the project workspace.
