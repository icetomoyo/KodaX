# v0.7.29 Capability Inventory

This inventory enumerates every capability `runKodaX` (`packages/coding/src/agent.ts`) provides today. Each entry is a CAP-XXX record. Refactor PRs in P2-P4 must demonstrate that every CAP entry has a wire-up in the new architecture (substrate middleware / Runner-level lifecycle / Agent declaration middleware) — or an explicit ADR-rationale for non-support.

**This document is a P3-merge gate**: a refactor PR that fails to wire any CAP entry cannot merge.

---

## Methodology

1. **Source mining**: `packages/coding/src/agent.ts` (3164 lines, current SA entry) is read function-by-function. Helpers and `runKodaX` body branches are recorded as separate CAP entries when they represent independent capability boundaries (independent triggers / independent outputs).
2. **Parity-restore mining**: `packages/coding/src/task-engine/runner-driven.ts` is grep'd for `legacy.*runKodaX | inadvertently dropped | parity restore | Legacy agent\.ts | matching legacy`. Each match marks a capability that was de-facto SA-exclusive (because AMA route bypassed `runKodaX`) and got patched back. These entries carry `Risk: HIGH_RISK_PARITY` — they are the empirical baseline of "things that have already been lost in a similar refactor".
3. **Cross-validation**: each CAP entry must cite at least one `file:line` source location, **independently verified by grep against current `HEAD` at the timestamp shown next to the citation**. Legacy line numbers from runner-driven.ts comments (written at v0.7.26 / FEATURE_084 time) are treated as historical references only — agent.ts has been touched substantially since (FEATURE_085, FEATURE_086, v0.7.27/v0.7.28 commits), so any number copied from a legacy comment is re-verified before being recorded here. When a legacy citation has drifted, the inventory records both: the verified current line + a note indicating the legacy reference is stale.
4. **Utility-level helpers excluded**: pure data-shape constructors and trivial classification helpers (e.g., `createToolResultBlock` at `agent.ts:873` — a 9-line constructor returning `KodaXToolResultBlock`) are NOT recorded as CAP entries. The boundary is **independent capability vs. shared utility**: a capability has an independent trigger condition / independent observable effect; a utility is a shared shape constructor or pure predicate that has no behavior of its own. Utility-level helpers travel with their consumers in the migration. If a future reader is uncertain whether a particular helper warrants a CAP entry, default to including it — over-inventorying is safer than under-inventorying.

## Format spec

```
### CAP-XXX: <name>

- **Class**: 1 (substrate middleware) | 2 (Runner-level lifecycle/guardrail) | 3 (declarable opt-in middleware) | (deferred / non-supported with rationale)
- **Risk**: HIGH_RISK_PARITY | MEDIUM | LOW
- **Current location**: `path:line[-line]`
- **Trigger condition**: when this capability fires today
- **Inputs consumed**: state / args read
- **Outputs produced**: state / events / side effects emitted
- **Time-ordering constraint**: if it must run before/after some other capability, state it
- **Migration target**: new file path under `packages/coding/src/agent-runtime/...`
- **Wire-up contract**: how the new substrate / Agent declaration / Runner frame invokes it
- **Default for**: which Agent declarations enable it (only relevant for Class 3)
- **Test obligation**: golden trace coverage required? contract test ID?
```
