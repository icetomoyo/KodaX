>

> Theme: AMA Visibility Polish — 把 Claude Code 风格的"可见计划列表"引入 KodaX AMA Runner，让用户在长任务运行时实时看到 Scout 产出的 obligation 逐项推进，与既有 BackgroundTaskBar / repo-intelligence trace / spinner 共同组成完整的 AMA observability 面板。

## 总览

主路线 Plan B 在 `v0.7.33` 落地 Auto Mode Classifier 后告一段落。KodaX 在 `v0.7.26` 后的 AMA 架构已经**稳定成熟**（Scout → Planner → Generator ⇄ Evaluator），但 UX 层的**可见性**仍比 Claude Code 弱一档 —— 用户看到的是 spinner + 一小条 BackgroundTaskBar pill，长任务中间"黑盒感"明显。

对照 Claude Code 的 `TodoWriteTool` / `TaskListV2` 组件，关键设计不在于"多复杂的任务追踪系统"，而在于一件事：**让用户在每一次模型推进时都看到"还剩多少、当前在做什么、哪几项已完成"**。这是解决长任务焦虑最廉价、收益最高的一个 UX 点。

KodaX 天然有落地优势：Scout 在入口阶段已经产出 `skillMap.executionObligations: string[]`（`packages/coding/src/task-engine/runner-driven.ts:346`），**这已经是一个现成的"计划列表"**，只是从未被 UI 消费。v0.7.34 的全部工作就是把这条既有数据接到 UI，并补齐"状态推进 + 视觉规范"两块。

## FEATURE_097: AMA Runner Realtime Todo List — Claude-Aligned Visibility Surface

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 背景

#### 当前 AMA 可见性现状

一个 H2 任务启动后，用户看到的是：

```
⠋ Planning...
  [ Scout active ]  [ Scanning 3 modules ]
  ▸ repo intel: resolved entrypoint packages/ai/src/providers/openai.ts
  ▸ tool: read_file packages/ai/src/providers/openai.ts (1842 lines)
  ▸ tool: grep "reasoning_budget" (12 matches)
  ...
```

- **Spinner**：一个旋转字符 + 单行 phase 文本（`Planning`/`Generating`/`Evaluating`）
- **BackgroundTaskBar**（`packages/repl/src/ui/components/BackgroundTaskBar.tsx`）：短 pill，最多 3 条活动状态
- **repo-intelligence trace**（FEATURE_086 刚合并）：一行一条的工具调用/文件读取进度流
- **transcript streaming**：模型 token 流

信息全都有，但**缺一个结构化的"本任务要做哪几件事、做到第几件了"**视图。用户长时间等待期间的主要焦虑来自"不知道还剩多少"。

#### Claude Code 的做法

`c:/Works/claudecode` 里有两套并行 Todo 系统（V1 TodoWrite + V2 TaskCreate/Update），核心视觉规范一致：

- **符号**：`☐` pending / `⏺` 或 `*` in_progress / `✓` completed
- **双形式文本**：`content`（指令形，"Run tests"）用于静态展示；`activeForm`（进行时，"Running tests"）用于 spinner / in_progress 项
- **颜色**：pending dim / in_progress cyan bold / completed green
- **layout**：列表左侧一条竖向装订线，右上角 `N/M completed` 计数
- **消失策略**：所有项 completed 后 **5 秒延迟**再隐藏（`useTasksV2.ts`），给用户时间感知完成

关键设计哲学（**本 feature 严格对齐**）：

> **显示与否完全由"是否有 todo 项"决定，不做基于任务复杂度的人工折叠**。

在 Claude Code 里，模型**自主**决定何时调 `TodoWrite` —— 简单任务（单步 / 两步）模型就不调，列表自然不出现；复杂任务调了就出现。PROMPT 里写的是"3+ 步任务建议使用"，但**代码层面没有任何基于复杂度的折叠逻辑**。这是合理的：UI 不应该做"任务难度"的判断，该由产生数据的那一层（模型）决定。

#### KodaX 的天然映射

Scout 的 `skillMap.executionObligations` 已经扮演了"计划列表种子"的角色：

- **H0**：Scout 通常产出 0-2 条 obligation（或直接跳过执行返回答案）
- **H1**：Scout 产出 2-5 条 obligation，由 Generator 逐条执行
- **H2**：Planner 把 Scout 的 obligation 细化为 contract，可能产出 5-10 条

这意味着**不需要在 UI 层做 H0/H1/H2 硬编码折叠** —— 复杂度的差异天然反映在 obligation 数量上。和 Claude Code 同样一条规则：**0-1 条时不渲染，2+ 条时统一展开**。H0/H1 的用户**照样**能看到计划列表，不会因为不是 H2 就被降级为纯 spinner。

### 设计

#### 数据模型（`@kodax/coding`）

```ts
// packages/coding/src/types.ts
export type TodoStatus =
  | "pending"
  | "in_progress"
  | "completed"
  | "failed"
  | "skipped";

export interface TodoItem {
  id: string;
  content: string;              // 指令形，例如 "Run migration tests"
  activeForm: string;           // 进行时，例如 "Running migration tests"
  status: TodoStatus;
  owner?: string;               // "main" | `child_${taskId}` —— 用于 child task 并行
  sourceObligationId?: string;  // 关联 Scout/Planner 产出的 obligation 索引
}

export type TodoList = readonly TodoItem[];
```

**`content` / `activeForm` 双形式**：对齐 Claude Code，避免 UI 层临时改写文本。生成时由 Scout/Planner 同时给出两种形式；若缺 `activeForm`，fallback 为 `"Working on: " + content`。

**状态语义**：
- `pending` → `in_progress` → `completed | failed | skipped`
- `failed` 专为 Evaluator `revise` verdict 设计，下一轮开始时可重置为 `pending`
- `skipped` 专为 Scout 中途发现某条 obligation 不必要（例如 H2 Planner 合并了两条）

**"一次一个 in_progress" 约束**：**按 `owner` 分组**执行（每个 owner 一个 in_progress）。跨 owner 允许并行（对应 `dispatch_child_task` 场景）。约束走 **prompt 层**，代码层不 enforce（与 Claude Code 对齐）。

#### 后端 Store（新建 `packages/coding/src/task-engine/todo-store.ts`）

```ts
interface TodoStore {
  init(obligations: TodoInit[]): void;         // Scout 入口种入
  updateStatus(id: string, status: TodoStatus, note?: string): void;
  replace(items: TodoItem[]): void;            // Planner H2 细化时全量替换
  getAll(): readonly TodoItem[];
  snapshot(): TodoList;                        // for session 持久化（Phase 4）
  reset(): void;                               // 新任务开始时清空
}
```

- **纯 in-memory**：不引入 fs、不引入 lockfile。KodaX 是单进程 CLI，Claude Code V2 的 fs.watch + proper-lockfile 是 swarm 多进程场景，本 feature 不需要
- **挂载点**：`runManagedTaskViaRunner` 作用域内，与 managed task 生命周期对齐
- **订阅**：当任意 item 变化时，触发 `events.onTodoUpdate(items)` 回调（下一节）

#### 事件管线（复刻 FEATURE_086 `onRepoIntelligenceTrace`）

`KodaXEvents`（`packages/coding/src/types.ts:158`）新增：

```ts
export interface KodaXEvents {
  // ... 既有
  onRepoIntelligenceTrace?(event: RepoIntelligenceTraceEvent): void;
  onTodoUpdate?(items: readonly TodoItem[]): void;  // 新增
}
```

`KodaXManagedTaskStatusEvent`（`types.ts:640` 附近）扩展：

```ts
export interface KodaXManagedTaskStatusEvent {
  phase: "planning" | "generating" | "evaluating" | "completed" | ...;
  summary?: string;
  todoItems?: readonly TodoItem[];  // 新增：随 status 事件附带最新 todo 状态
}
```

两路并行：
1. **独立事件**：`onTodoUpdate` 每次 item 变化时触发（低延迟，UI 实时更新）
2. **依附事件**：`onManagedTaskStatus` 在 phase 切换时带上 todoItems 快照（兜底，确保遗漏的订阅者也能同步）

#### 初始种入流程

`runner-driven.ts` 在 Scout verdict 处理点（`emit_scout_verdict` 附近）：

```ts
// 伪代码，关键逻辑
const obligations = scoutVerdict.skillMap.executionObligations ?? [];

if (obligations.length >= 2) {                   // ← 0-1 条不种入，UI 自然不显示
  const initialTodos = obligations.map((ob, idx) => ({
    id: `todo_${idx + 1}`,
    content: ob.label,                            // Scout 产出时必须带 label
    activeForm: ob.activeForm ?? `Working on: ${ob.label}`,
    status: "pending" as const,
    owner: "main",
    sourceObligationId: String(idx),
  }));
  todoStore.init(initialTodos);
  events.onTodoUpdate?.(todoStore.getAll());
}
```

**`obligations.length >= 2` 这个门槛**是本 feature 唯一的显示决策点 —— 完全对齐 Claude Code "由产生数据的层自行决定是否值得展示"的哲学。H0/H1/H2 都走这同一条路径，没有任何硬编码的复杂度分档。

#### Generator/Planner 主动推进（新工具 `TodoUpdate`）

新建 `packages/coding/src/tools/todo-update.ts`：

```ts
export const todoUpdateTool: KodaXTool = {
  name: "todo_update",
  description:
    "Update the status of a planned todo item. Call this every time you " +
    "complete a major step so the user can see progress. Rules: " +
    "(1) Set status=in_progress BEFORE starting work on an item. " +
    "(2) Set status=completed AFTER finishing that item. " +
    "(3) Only ONE item may be in_progress per owner at any time. " +
    "(4) Use status=failed if an attempt clearly failed and needs retry.",
  parameters: {
    id: { type: "string", required: true },
    status: {
      type: "string",
      enum: ["in_progress", "completed", "failed", "skipped"],
      required: true,
    },
    note: { type: "string", required: false },
  },
  execute: async ({ id, status, note }, ctx) => {
    ctx.todoStore.updateStatus(id, status as TodoStatus, note);
    ctx.events.onTodoUpdate?.(ctx.todoStore.getAll());
    return { ok: true };
  },
};
```

该工具**默认注入 Generator 和 Planner 的工具集**（Scout 和 Evaluator 不注入 —— Scout 已经在 verdict 里产出了初始列表，Evaluator 的状态推进由 Runner 自动代劳）。

**Evaluator 自动代劳规则**（Runner 层）：
- `accept` verdict：剩余所有 `pending` 项自动标 `completed`（用户能一次性看到整批完成）
- `revise` verdict：Evaluator 点名的 obligation 对应 item 标 `failed`；下一轮 Generator 开始前重置为 `pending`
- `replan` verdict：整个列表 reset，等待 Planner 重新 `replace()`

#### LLM 行为兜底（避免模型忘记调 `todo_update`）

三层兜底：

1. **System prompt 强提示**：在 Generator/Planner 的 role prompt 里硬编码"每完成一个 item 必须 call `todo_update`"
2. **节流提醒**（Phase 4，本 feature 可选落地）：连续 N = 8 个 iteration 模型未调 `todo_update`，Runner 在下一轮 prompt 追加 `<system-reminder>` 带未完成列表（参考 Claude Code `getTodoReminderAttachments` `TURNS_SINCE_WRITE=10`）
3. **Runner 层启发式推断**：若 Generator 调了 `write_file`、`edit_file`、`run_tests` 等 high-signal 工具且目标路径/命令命中某 item 的关键词，Runner 侧自动把该 item 从 `pending` 推到 `in_progress`（作为视觉兜底，即使模型忘调工具用户也能看到进度）

#### UI 层（新建 `packages/repl/src/ui/components/TodoListSurface.tsx`）

**视觉规范（严格对齐 Claude Code TaskListV2，但加 KodaX 视口约束）**：

> **视口约束**：列表行数**上限 6 行**（含可选的 summary 折叠行）。Claude Code 视觉上不做硬截断是因为模型一般产出 5-8 条 todo，观感就 5-6 行；KodaX 保留同样的视觉感受但**加一条硬规则**：无论 Scout/Planner 产出多少 obligation（10 条、20 条甚至更多），TodoListSurface 实际渲染永远不超过 6 行。超出部分用 summary 行折叠（见下方 "滚动窗口策略"）。这样 spinner + TodoListSurface + BackgroundTaskBar 的总占屏始终 ≤ 8 行，不压缩 transcript 可见区域。

**正常场景（obligation 数 ≤ 6）：全部展开，一条一行**

```
┌─ 示例：7 条 obligation，有 1 条摘要+ 其它全显示 ───────────┐
│                                                          │
│ ⠋ Generating...                          3/7 completed   │
│ ▏ ✓ Locate test fixtures                                 │
│ ▏ ⚡ Running migration tests                             │
│ ▏ ☐ Update type definitions                              │
│ ▏ ☐ Regenerate snapshot                                  │
│ ▏ ☐ Verify end-to-end flow                               │
│ ▏ ✓ 2 done                          ← 头部折叠摘要行     │
│                                                          │
│ [ Scout handoff ] [ 3 files touched ]   ← BackgroundTaskBar │
└──────────────────────────────────────────────────────────┘
```

**长列表场景（obligation 数 > 6）：滚动窗口 + 首尾折叠**

```
┌─ 示例：总 12 条 obligation，当前 in_progress 是第 5 条 ───┐
│                                                          │
│ ⠋ Generating...                         4/12 completed   │
│ ▏ ✓ 3 done                          ← 顶部折叠（过去完成） │
│ ▏ ✓ Locate test fixtures                                 │
│ ▏ ⚡ Running migration tests         ← current（始终可见） │
│ ▏ ☐ Update type definitions                              │
│ ▏ ☐ Regenerate snapshot                                  │
│ ▏ ☐ +5 more                         ← 底部折叠（未来待做） │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

关键元素：

| 元素 | 规范 |
|---|---|
| 左侧竖向装订线 | Unicode `▏`（U+258F），灰色 dim，和 Claude Code 的 border 对齐 |
| pending 符号 | `☐`（U+2610），dim 灰 |
| in_progress 符号 | `⚡` 或 `●`（Claude Code 用 `⏺`；KodaX 选一致的动效字符），cyan + bold |
| completed 符号 | `✓`（U+2713），green |
| failed 符号 | `✗`（U+2717），red |
| skipped 符号 | `⊘`（U+2298），dim 灰 |
| pending / completed 行文本 | 显示 `content`（指令形） |
| in_progress 行文本 | 显示 `activeForm`（进行时） |
| failed 行文本 | 显示 `content` + 红色小字 note（如有） |
| 右上角计数 | `{completedCount}/{totalCount} completed`，dim |
| spinner 行 spinner 字符 | 复用现有 `spinner-frames.ts`，不新增 |

**挂载点**：`InkREPL.tsx` 的 `PromptFooter` 内，**位置**：spinner 下方、BackgroundTaskBar 上方（独立新 slot）。相关锚点：`promptActivityViewModel.showSpinner` 为 true 时才挂载 TodoListSurface。

**消失策略**（对齐 Claude Code 5 秒延迟）：
- 所有 item 为 `completed | skipped` 时，启动 5 秒 timer
- timer 内有任何新 todo 事件 → 取消 timer（任务还在动）
- 5 秒后 `todoItems` 置空 → 组件自然卸载

#### View-Model（新建 `packages/repl/src/ui/view-models/todo-plan.ts`）

```ts
const MAX_VISIBLE_ROWS = 6;   // 含折叠摘要行在内的硬上限

type TodoRowKind = "item" | "summary_done" | "summary_pending";

interface TodoRow {
  kind: TodoRowKind;
  id?: string;                  // kind === "item" 时存在
  symbol: string;
  symbolColor: "dim" | "cyan" | "green" | "red" | "gray";
  text: string;                 // "Running tests" 或 "3 done" / "+5 more"
  isActive: boolean;            // in_progress → true
}

interface TodoPlanViewModel {
  shouldRender: boolean;        // items.length >= 2 && !hiddenAfterGracePeriod
  rows: TodoRow[];              // 最多 MAX_VISIBLE_ROWS 行
  completedCount: number;
  totalCount: number;
}

export function buildTodoPlanViewModel(
  items: readonly TodoItem[],
  opts: { now: number; lastAllCompletedAt: number | null },
): TodoPlanViewModel { ... }
```

**自动跟随滚动算法**（`items.length > MAX_VISIBLE_ROWS` 时启用）：

> **重要澄清**：这里的"滚动"指 **每当某个 item 从 `in_progress` 转到 `completed`、下一个 item 进入 `in_progress` 时，UI 自动把视口向下挪一行，让新的 current 对齐到可见位置**。**不是用户可交互的滚动**（不绑快捷键、不响应方向键/鼠标滚轮）。Claude Code 也是同样做法：用户不需要操作，视口自己跟随进度走。若用户想回看全部历史轨迹，在 transcript / tool call 历史里追溯（未来若有强需求，可在 Phase 4 加手动展开热键）。

**锚点计算**：

1. **锚点行 (anchor)**：
   - 优先 = 第一个 `in_progress` item 的索引
   - 无 in_progress 时 = 第一个 `pending` 的索引（任务刚开始/刚被 evaluator 重置）
   - 全部 completed 时 = 最后一个 completed 的索引（5 秒消失前的终态显示）

2. **可视窗口布局**（除 1-2 条折叠摘要外，剩 4-5 行给真实 item）：
   - 锚点**之前**最多 1 条（最近完成的一条，让用户看到"刚做完啥"）
   - 锚点**本身**1 条（current）
   - 锚点**之后**最多 2 条（接下来要做的 pending）

3. **折叠摘要行（始终占行数预算）**：
   - 若锚点之前还有**未显示**的 completed 项 ≥ 1 条：**顶部**插入 `✓ N done`
   - 若锚点之后还有**未显示**的 pending 项 ≥ 1 条：**底部**插入 `☐ +N more`
   - 两条折叠行都不占"4 条真实 item"的预算，作为独立 row 叠加（但总行数仍受 `MAX_VISIBLE_ROWS=6` 约束，视需要压缩真实 item）

4. **failed 项优先级**：若存在 failed 且未在窗口内，强制把其中**最新**一条替换窗口内最近的 pending 位置（让用户立刻看到失败告警）

**窗口前移的具体时机**：

视口的"跟随滚动"不是动画效果（Ink 不支持流畅动画），而是**每次 view-model 重建时重新计算锚点 → 重新渲染**。用户观感就是：
- 一条 `in_progress` → `completed` 时，view-model 锚点自动变为下一条 item
- 顶部折叠摘要 `✓ N done` 的数字 **+1**
- 窗口整体看起来"向上挪了一行"（因为完成的从窗口里移出去，新 pending 从下方进来）
- 全程无动画，直接 re-render

**折叠行样式**：
- `✓ N done`：symbolColor=green，text 用 dim，不响应 id 查询
- `☐ +N more`：symbolColor=dim，text dim
- `kind !== "item"`，不对应真实 TodoItem

**边界**：
- `totalCount ≤ 6`：不启用自动滚动窗口，全部展开；若 completed 多于 4 条也可选在顶部合并折叠成 `✓ N done` 减少噪音（默认只在 `totalCount > 4` 时合并）
- `totalCount = 0-1`：`shouldRender = false`，组件不挂载

所有显示决策集中在 view-model，组件层保持纯渲染。单元测试覆盖：全 pending / 全 completed / 锚点在头部 / 锚点在尾部 / 含 failed / 总数 = 2/6/7/20 几个档位；**自动前移场景**（in_progress 从 item[3] → item[4] 时锚点迁移 + 折叠行数字变化）必测。

#### 事件路由注意

`emitInfoItemToCorrectLayer`（`InkREPL.tsx:1555`）的现存规则要求 managed foreground 期间的 info 条目路由到 worker 层。**TodoListSurface 不走这条**：todo 状态是**跨 worker 的全局任务进度**，挂在 InkREPL 顶层 React state (`todoItems: TodoItem[]`) 而非 `managedForegroundLedger`。实现时在 `onTodoUpdate` handler 里**直接** `setTodoItems(items)`，跳过 `emitInfoItemToCorrectLayer`。

### 验收标准

1. **Scout 自动种入**：AMA H0/H1/H2 任何一档任务启动后，若 Scout 产出 ≥ 2 条 `executionObligations`，spinner 下方立刻出现 TodoListSurface，全部 `pending`；少于 2 条时不渲染（对齐 Claude Code "任务太简单不干扰"）。
2. **Generator 实时推进**：Generator 每调一次 `todo_update`，对应 item 符号 / 颜色 / 文本（content ↔ activeForm 切换）实时变更，无需等 phase 切换。
3. **Evaluator verdict 自动收尾**：
   - `accept` → 所有剩余 `pending` 转 `completed`
   - `revise` → 受影响 item 变 `failed`（红色 ✗ + note），下一轮开始前重置 `pending`
   - `replan` → TodoListSurface 整体 reset，Planner 重新 `replace()` 后出现新列表
4. **视觉对齐 Claude Code**：符号（☐ / ⚡ / ✓ / ✗ / ⊘）+ 颜色（dim / cyan bold / green / red / gray）+ 左侧装订线 `▏` + 右上角 `N/M completed` 全部就位。
5. **双形式文本**：`in_progress` 行显示 `activeForm`（进行时），其它状态显示 `content`（指令形）。
6. **5 秒延迟消失**：全部 completed / skipped 后 5 秒消失；5 秒内任意 todo 事件取消 timer 继续显示。
7. **Child task 并行**：`dispatch_child_task` 场景下 owner 分组渲染（`main` + `child_<id>` 各自占一块），每组独立"一个 in_progress"约束，可并行跑。
8. **H0/H1/H2 行为一致**：H0 若产出 2 条 obligation 也显示（H1 H2 更不例外），UI 层无任何复杂度分档逻辑。
9. **视口占用 ≤ 6 行**：任何 obligation 数量下（2 条、7 条、20 条），TodoListSurface 实际渲染不超过 6 行。超出时通过顶部 `✓ N done` + 底部 `☐ +N more` 折叠摘要行维持上限。Claude Code 观感（5-6 行短列表）被严格保持。
10. **完成后自动跟随滚动**：当某一项从 `in_progress` 转 `completed`、下一项进入 `in_progress` 时，视口自动前移 —— 顶部折叠 `✓ N done` 数字 +1，窗口内 current 锚点自动对齐到新 in_progress 行。**全程无需用户按键或鼠标操作**。E2E 场景：总 12 条 obligation，Generator 依次调 `todo_update` 推进 1→12，每次推进后 re-render 锚点正确移动。
11. **与既有 UI 不冲突**：TodoListSurface 挂载时不挤掉 spinner / BackgroundTaskBar / repo-intel trace；PromptFooter 布局正常；spinner + TodoListSurface + BackgroundTaskBar 的总占屏始终 ≤ 8 行。
12. **无回归**：`npm run test` / `npm run build` 全绿；既有 AMA 测试（`runner-driven.test.ts`、`InkREPL` E2E）通过。

### 非目标

- **不** 做 fs 持久化（Claude Code V2 的 `~/.claude/tasks/*.json` + fs.watch + proper-lockfile）。KodaX 是单进程 CLI，纯 in-memory 足够
- **不** 做 todo 依赖图（V2 的 `blocks` / `blockedBy`）。KodaX 的 DAG 已由 Scout/Planner/Evaluator 表达，todo 层不再重复
- **不** 做跨会话持久化（resume 时恢复 todo 列表）。放 Phase 4 独立考虑，不在 v0.7.34 范围
- **不** 复刻 Claude Code 的 V2 `TaskCreate` / `TaskUpdate` 工具细分（v2 有 id 生成、高水位、`claimTask({ checkAgentBusy })` 等 swarm 专用能力）。KodaX 只需一个简化工具 `todo_update`，创建交由 Scout verdict 自动完成
- **不** 做基于 H0/H1/H2 的硬编码折叠/展开策略。复杂度分层**由 Scout 产出的 obligation 数量自然决定**，不由 UI 判断
- **不** 做**用户交互式滚动**（键盘 up/down、鼠标滚轮、PageUp/PageDown 展开全量列表等）。视口"滚动"仅指**自动跟随当前 in_progress 的 re-render 式窗口前移**，对齐 Claude Code。用户想看全量历史可追溯 transcript 里的 tool call 记录；若未来出现强需求，再用独立 feature 加手动展开热键
- **不** 把 todo 事件走 `emitInfoItemToCorrectLayer` 路由 / 进入 history。Todo 是 PromptFooter 层的活动状态，不是对话历史
- **不** 改动 `BackgroundTaskBar` 的 pill 风格。两者是并列组件，各司其职
