## 总览

主路线 Plan B 在 `v0.7.33` 落地 Auto Mode Classifier 后告一段落。v0.7.34 做两件事：

**FEATURE_110（前置清理）**：KodaX 当前同时存在两套 plan-mode 实现且会真实打架——路径 1（`/plan` slash 命令 + `runWithPlanMode` + readline/chalk + plan-storage 持久化，v0.3.1 时代）已被 FEATURE_074 v0.7.20 引入的路径 2（`PermissionMode="plan"` + `exit_plan_mode` tool + Ink-native PlanScrollPanel 审批 UI）完全替代但从未删除。路径 1 的 `~/.kodax/plans/*.json` 持久化、自家文本协议解析、步级 y/n confirm 全部跟主线 InkREPL UI / FEATURE_072 lineage / FEATURE_074 permission gate 脱节，KNOWN_ISSUES 已记录"plan-mode 路径不接 pendingInputs 队列"两个版本未修。本 feature 一个 PR 砍 ~460 行净删除，无新增代码。

**FEATURE_097（主交付）**：KodaX 在 `v0.7.26` 后的 AMA 架构已经**稳定成熟**（Scout → Planner → Generator ⇄ Evaluator），但 UX 层的**可见性**仍比 Claude Code 弱一档 —— 用户看到的是 spinner + 一小条 BackgroundTaskBar pill，长任务中间"黑盒感"明显。对照 Claude Code 的 `TodoWriteTool` / `TaskListV2` 组件，关键设计不在于"多复杂的任务追踪系统"，而在于一件事：**让用户在每一次模型推进时都看到"还剩多少、当前在做什么、哪几项已完成"**。这是解决长任务焦虑最廉价、收益最高的一个 UX 点。KodaX 天然有落地优势：Scout 在入口阶段已经产出 `skillMap.executionObligations: string[]`（`packages/coding/src/task-engine/runner-driven.ts:346`），**这已经是一个现成的"计划列表"**，只是从未被 UI 消费。v0.7.34 的全部工作就是把这条既有数据接到 UI，并补齐"状态推进 + 视觉规范"两块。

**FEATURE_112（路由修复）**：用户实战反馈 KodaX 在 read-only 重度调研类任务（"为什么 X 这么设计 / 解释整条数据流 / 跨模块审查"）上 Scout 永远 H0 单飞，从来不升 H1（让 Evaluator audit）也从来不派 child（早期分流）。三轮独立调研 + 一轮对抗评审定位四点系统性偏差：(a) `deriveTopologyCeiling` 只看 `mutationSurface + assuranceIntent`，没有 complexity 维度，read-only 复杂任务天花板被锁死在 H0；(b) `fanoutReason` 对 `primaryTask=unknown` 任务硬写"No high-value shard class detected"，对 LLM 是负面 dispatch 信号；(c) Scout role-prompt 的 SCOPE COMMITMENT 只覆盖 mutation scope（"≥2 files OR project from scratch"），没有对称的 investigation scope 触发条件；(d) `topologyCeiling: H1_EXECUTE_EVAL` 在 decisionSummary 里是裸值，Scout 没有任何提示告诉它"H1 = Evaluator 可 audit"。修复对称 FEATURE_106 的成功路径——纯 prompt + 路由 heuristic，**零跨包改动**，~21 LoC + 4 cell × 3 alias eval 数据集。

**实施顺序**：FEATURE_110 先 land（清理风险低、依赖 0），FEATURE_097 后 land（其 Phase 4+ "Plan Mode 联动"延伸只针对路径 2，无双写顾虑）。FEATURE_112 与前两者完全正交，可以并行 land 不需顺序约束。

## FEATURE_110: Remove Legacy Plan Mode (Path 1)

**Status**: Released (v0.7.34, 2026-05-04, commit `6b1df35`)
**Priority**: High
**Category**: Refactor / Cleanup

### 背景

KodaX 当前 plan-mode 有两套并存实现：

**路径 1（v0.3.1 老）**：
- 入口：`/plan once <task>` / `/plan on|off|list|resume|clear` slash 命令 + InkREPL `[planMode, setPlanMode]` boolean state
- 实现：`packages/repl/src/common/plan-mode.ts`（281 行）+ `packages/repl/src/common/plan-storage.ts`（77 行）
- UI：chalk + readline.question + console.log，**完全绕过 Ink REPL UI**
- 持久化：`~/.kodax/plans/*.json`，独立 schema 跟主线 session lineage / task-contract artifact 都没接
- 流程：先 `runManagedTask({maxIter:1})` 生成 plan 文本 → 自家 regex 解析 `1. [READ] desc - target` → 步级 `confirm("Execute this step?")` y/n
- 时代：v0.3.1（2024 早期），wizard 风格

**路径 2（v0.7.20 FEATURE_074 新）**：
- 入口：`PermissionMode = "plan" | "accept-edits" | "auto" | "auto-in-project"`，通过 `setPermissionMode` callback 切（[InkREPL.tsx:6504](../../packages/repl/src/ui/InkREPL.tsx#L6504)）
- 实现：[`exit-plan-mode.ts`](../../packages/coding/src/tools/exit-plan-mode.ts) tool + `managedRunContext.planModeBlockCheck` live predicate closure（[InkREPL.tsx:5645](../../packages/repl/src/ui/InkREPL.tsx#L5645)）
- UI：InkREPL 内 PlanScrollPanel + ScrollBox + FullscreenLayout（FEATURE_075 v0.7.25 在打磨）
- 流程：LLM 在 plan permission mode 下用全部正常工具规划，写操作被 permission gate 阻断 → 最后调 `exit_plan_mode(plan)` → 弹审批 dialog → approve → permission mode 翻到 `accept-edits` → 继续
- 时代：v0.7.20（2026），permission-mode 风格
- Parent-only：child agent 被 `CHILD_EXCLUDE_TOOLS_BASE` 过滤
- Live predicate：mid-run 切 plan↔accept-edits 立即生效

### 两条路径会打架的真实场景

路径 1 用一个布尔 `planMode`，路径 2 用 `PermissionMode` 枚举，两者各自的 state 互不感知：

1. 用户先 `/plan on`（路径 1 `planMode=true`），又把 `PermissionMode` 切到 `"plan"`（路径 2）
2. 用户提 task → [InkREPL.tsx:6097](../../packages/repl/src/ui/InkREPL.tsx#L6097) 走 `runWithPlanMode` 分支
3. Wizard 内部调 `runManagedTask` 跑某一步（[plan-mode.ts:152](../../packages/repl/src/common/plan-mode.ts#L152)），它**没有显式覆盖 permissionMode**，继承了 session 的 `"plan"`
4. 这一步如果是 `write_file`，被路径 2 的 `planModeBlockCheck` 阻断 → 返回错误
5. 但 wizard 自己的 `confirm("Execute this step?")` 已经 y 通过了
6. 用户体验：明明 confirm yes 了，却得到 `[Tool Error] plan mode blocks write`

KNOWN_ISSUES.md 记录的"plan-mode 路径（[InkREPL.tsx:6466](../../packages/repl/src/ui/InkREPL.tsx#L6466)）不接 pendingInputs 队列"两个版本未修，是路径 1 已无人维护的另一个铁证。

## FEATURE_097: AMA Runner Realtime Todo List — Claude-Aligned Visibility Surface

**Status**: Released (v0.7.34, 2026-05-04, commit `a974c57`)
**Priority**: High
**Category**: Enhancement

### 背景

#### 当前 AMA 可见性现状

一个 H2 任务启动后，用户看到的是：

```
⠋ Planning...
  [ Scout active ]  [ Scanning 3 modules ]
  ▸ repo intel: resolved entrypoint packages/ai/src/providers/openai.ts
  ▸ tool: read_file packages/ai/src/providers/openai.ts (1842 lines)
  ▸ tool: grep "reasoning_budget" (12 matches)
  ...
```

- **Spinner**：一个旋转字符 + 单行 phase 文本（`Planning`/`Generating`/`Evaluating`）
- **BackgroundTaskBar**（`packages/repl/src/ui/components/BackgroundTaskBar.tsx`）：短 pill，最多 3 条活动状态
- **repo-intelligence trace**（FEATURE_086 刚合并）：一行一条的工具调用/文件读取进度流
- **transcript streaming**：模型 token 流

信息全都有，但**缺一个结构化的"本任务要做哪几件事、做到第几件了"**视图。用户长时间等待期间的主要焦虑来自"不知道还剩多少"。

#### Claude Code 的做法

`c:/Works/claudecode` 里有两套并行 Todo 系统（V1 TodoWrite + V2 TaskCreate/Update），核心视觉规范一致：

- **符号**：`☐` pending / `⏺` 或 `*` in_progress / `✓` completed
- **双形式文本**：`content`（指令形，"Run tests"）用于静态展示；`activeForm`（进行时，"Running tests"）用于 spinner / in_progress 项
- **颜色**：pending dim / in_progress cyan bold / completed green
- **layout**：列表左侧一条竖向装订线，右上角 `N/M completed` 计数
- **消失策略**：所有项 completed 后 **5 秒延迟**再隐藏（`useTasksV2.ts`），给用户时间感知完成

关键设计哲学（**本 feature 严格对齐**）：

> **显示与否完全由"是否有 todo 项"决定，不做基于任务复杂度的人工折叠**。

在 Claude Code 里，模型**自主**决定何时调 `TodoWrite` —— 简单任务（单步 / 两步）模型就不调，列表自然不出现；复杂任务调了就出现。PROMPT 里写的是"3+ 步任务建议使用"，但**代码层面没有任何基于复杂度的折叠逻辑**。这是合理的：UI 不应该做"任务难度"的判断，该由产生数据的那一层（模型）决定。

#### KodaX 的天然映射

Scout 的 `skillMap.executionObligations` 已经扮演了"计划列表种子"的角色：

- **H0**：Scout 通常产出 0-2 条 obligation（或直接跳过执行返回答案）
- **H1**：Scout 产出 2-5 条 obligation，由 Generator 逐条执行
- **H2**：Planner 把 Scout 的 obligation 细化为 contract，可能产出 5-10 条

这意味着**不需要在 UI 层做 H0/H1/H2 硬编码折叠** —— 复杂度的差异天然反映在 obligation 数量上。和 Claude Code 同样一条规则：**0-1 条时不渲染，2+ 条时统一展开**。H0/H1 的用户**照样**能看到计划列表，不会因为不是 H2 就被降级为纯 spinner。

### 设计

#### 数据模型（`@kodax/coding`）

> **简化决策（2026-05-03 用户确认）**：放弃 Claude Code 的 `content` / `activeForm` 双形式，只保留 `content` 单字段。Scout 既有的 `executionObligations: string[]` (managed-protocol.ts:339, types.ts:482/1079) 直接复用为 todo 文案来源 —— **零 prompt eval 触发**（不要求 Scout 同时产出指令形 + 进行时两种文本）、零 schema 迁移、Generator/Planner 也只产文本一份。视觉上的"进行时感觉"完全靠状态符号（`●` cyan bold）+ 颜色 + 装订线表达，不靠语法变形。

```ts
// packages/coding/src/types.ts
export type TodoStatus =
  | "pending"
  | "in_progress"
  | "completed"
  | "failed"
  | "skipped";

export interface TodoItem {
  id: string;
  content: string;              // 来自 Scout 既有 executionObligations: string[]，零变形
  status: TodoStatus;
  owner?: string;               // "main" | `child_${taskId}` —— child task 并行用
  sourceObligationIndex?: number;
}

export type TodoList = readonly TodoItem[];
```

**状态语义**：
- `pending` → `in_progress` → `completed | failed | skipped`
- `failed` 专为 Evaluator `revise` verdict 设计：当前所有 `in_progress` 项一律标 `failed`（带 `Evaluator requested revision` note），下一轮 Generator 开始前自动 reset 回 `pending`。**不**要求 Evaluator 协议层知道 todo ID（详见后文 §5 决策细节 ①）
- `skipped` 专为 Planner H2 path 合并 obligation 的场景

**"一次一个 in_progress" 约束**：按 `owner` 分组（每个 owner 一个 in_progress），跨 owner 允许并行（`dispatch_child_task` 场景）。约束走 **prompt 层**，代码层不 enforce（与 Claude Code 对齐）。

#### 后端 Store（新建 `packages/coding/src/task-engine/todo-store.ts`）

```ts
interface TodoStore {
  init(obligations: TodoInit[]): void;         // Scout 入口种入
  updateStatus(id: string, status: TodoStatus, note?: string): void;
  replace(items: TodoItem[]): void;            // Planner H2 细化时全量替换
  getAll(): readonly TodoItem[];
  snapshot(): TodoList;                        // for session 持久化（Phase 4）
  reset(): void;                               // 新任务开始时清空
}
```

- **纯 in-memory**：不引入 fs、不引入 lockfile。KodaX 是单进程 CLI，Claude Code V2 的 fs.watch + proper-lockfile 是 swarm 多进程场景，本 feature 不需要
- **挂载点**：`runManagedTaskViaRunner` 作用域内，与 managed task 生命周期对齐
- **订阅**：当任意 item 变化时，触发 `events.onTodoUpdate(items)` 回调（下一节）

#### 事件管线（复刻 FEATURE_086 `onRepoIntelligenceTrace`）

`KodaXEvents`（`packages/coding/src/types.ts:158`）新增：

```ts
export interface KodaXEvents {
  // ... 既有
  onRepoIntelligenceTrace?(event: RepoIntelligenceTraceEvent): void;
  onTodoUpdate?(items: readonly TodoItem[]): void;  // 新增
}
```

**单路事件（不做双轨）**：

```ts
// 仅 onTodoUpdate 一路
events.onTodoUpdate?.(todoStore.getAll());
```

**为什么不做双轨**：原方案曾设计 `KodaXManagedTaskStatusEvent.todoItems?` 作为 phase 切换时的快照兜底，但 KodaX 是单进程 CLI，所有消费者（InkREPL）都在同一个进程里，不存在"订阅者遗漏"的真实场景；FEATURE_086 的 `onRepoIntelligenceTrace` 单路至今未见问题，是先例。`onTodoUpdate` 触发频率本就高于 phase 切换，订阅者在第一次回调时就已同步，phase 事件不参与 UI 渲染，第二轨是预防性防御而非应对已知缺陷——按 KodaX 极简哲学砍掉。`KodaXManagedTaskStatusEvent` 不增加 `todoItems` 字段。

#### 初始种入流程

`runner-driven.ts` 在 Scout verdict 处理点（`emit_scout_verdict` 附近）：

```ts
// 伪代码，关键逻辑
const obligations = scoutVerdict.skillMap.executionObligations ?? [];

if (obligations.length >= 2) {                   // ← 0-1 条不种入，UI 自然不显示
  const initialTodos = obligations.map((ob, idx) => ({
    id: `todo_${idx + 1}`,
    content: ob,                                  // executionObligations 是 string[]，直接用
    status: "pending" as const,
    owner: "main",

```

**`obligations.length >= 2` 这个门槛**是本 feature 唯一的显示决策点 —— 完全对齐 Claude Code "由产生数据的层自行决定是否值得展示"的哲学。**H0/H1/H2 共用同一规则，没有任何硬编码的复杂度分档**。

#### Scout phase visibility（设计意图说明）

GLM external review (2026-05-03) 提出："`emit_scout_verdict` 是 Scout 最后一步才调用的工具——Scout 做完所有调查/分析后才产出 `executionObligations`。这是否意味着 Scout 调研期间用户看不到任何结构化信息？"

**这是设计意图而非 bug**，原因：

1. **Scout 阶段在形成计划，不是执行计划**——在计划还没形成时强行显示一个计划列表是误导（Scout 自己都不知道要做哪几件）
2. **Scout 阶段的可见性已经由 3 个既有 surface 覆盖**：
   - Spinner 文字 (`Planning...` / `Scouting...`)
   - BackgroundTaskBar 短 pill
   - FEATURE_086 `onRepoIntelligenceTrace` 一行一条的 "reading X.ts / grepping Y / scanning Z" 流
3. **TodoListSurface 出现的时机** = Scout 完成 → 真正进入 long phase（Generator / Planner / H0 executor 继续）时——正好是用户最焦虑的"长任务执行"段
4. **未来若需要"调研期间也想看 sub-plan"的强需求**：可在 v0.8.x 加 `emit_scout_progress(partial_obligations)` 工具让 Scout 中途发布临时计划列表，但当前 0 真实数据驱动，YAGNI

**H0 短任务边界**：如果 H0 任务总耗时 < 5s（简单查询），TodoListSurface 渲染 → 5s linger → 消失，用户基本只看到一闪。这是**可接受边界**——这种任务通常 `obligations.length < 2` 直接不渲染；偶尔 ≥2 时 5s linger 提供"事后回顾"视图也有价值。**不**为这个边缘场景加复杂度（早渲染、不消失、不同步骤数特殊处理等）。

**用户痛点：H0 多步任务能不能也有 todo list？**

用户 2026-05-03 提出："除了非常简单的 QA 和几步就完成的任务外，其他即使是 H0 也可以有计划列表"。当前规则**已经满足**这个直觉：

| 场景 | obligations 实际产出 | TodoListSurface |
|---|---|---|
| H0 单文件改注释 | 0-1 条 | 不渲染 ✓ |
| H0 简单 QA "foo.ts:45 是什么" | 0 条 | 不渲染 ✓ |
| H0 多步任务（rename + 更新 3 处 caller + 跑 lint） | 2-3 条 | 渲染 ✓ |
| H1 调研类 | 2-5 条 | 渲染 ✓ |
| H2 复杂任务 | 5-10 条 | 渲染 ✓ |

**唯一前置：Scout role-prompt 鼓励 H0 多步任务也产出 obligations**。当前 Scout role-prompt 没有"H0 不产 obligations"的硬规则，但也没有"H0 多步任务请产 obligations"的鼓励，实际行为依赖模型自由发挥。**本 feature 在 Scout role-prompt 加 H0 mini-planner 措辞，强度由 release-gate eval 决定（详见 §H0 Mini-Planner 强度 A/B eval）**。

##### Ship 变体（Scout role-prompt 加这段）

```
For any task that requires ≥ 2 distinct execution steps (whether at H0_DIRECT,
H1_EXECUTE_EVAL, or H2_PLAN_EXECUTE_EVAL), populate executionObligations with
one entry per step BEFORE calling emit_scout_verdict.

Examples of "distinct execution steps":
  - Editing files in different modules
  - Refactor + verification (e.g., rename + run tests)
  - Multiple changes to the same file when each is independent

Examples of NOT distinct steps (do NOT split into multiple obligations):
  - Reading a file before editing it (preparation, not a step)
  - "Think about X" (reasoning, not a step)
  - Single-token typo fixes (single action)

After emit_scout_verdict, when continuing as H0_DIRECT executor, call
todo_update at each transition (pending → in_progress → completed) so the
user sees real-time progress. This gives the user a visible plan and forces
you to think through the full scope before acting.
```

#### Generator/Planner 主动推进（新工具 `TodoUpdate`）

新建 `packages/coding/src/tools/todo-update.ts`：

```ts
export const todoUpdateTool: KodaXTool = {
  name: "todo_update",
  description:
    "Update the status of a planned todo item. Call this every time you " +
    "complete a major step so the user can see progress. Rules: " +
    "(1) Set status=in_progress BEFORE starting work on an item. " +
    "(2) Set status=completed AFTER finishing that item. " +
    "(3) Only ONE item may be in_progress per owner at any time. " +
    "(4) Use status=failed if an attempt clearly failed and needs retry.",
  parameters: {
    id: { type: "string", required: true },
    status: {
      type: "string",
      enum: ["in_progress", "completed", "failed", "skipped"],
      required: true,
    },
    note: { type: "string", required: false },
  },
  execute: async ({ id, status, note }, ctx) => {
    if (!ctx.todoStore.has(id)) {
      // §5 决策细节 ⑤：unknown id 返回 valid id 列表，不抛异常、不静默成功
      return {
        ok: false,
        reason:
          `Unknown todo id: ${id}. ` +
          `Current valid ids: ${ctx.todoStore.allIds().join(', ')}.`,
      };
    }
    ctx.todoStore.updateStatus(id, status as TodoStatus, note);
    ctx.events.onTodoUpdate?.(ctx.todoStore.getAll());
    return { ok: true };
  },
};
```

**工具注入规则**（2026-05-03 修正，原稿"Scout 不注入"是 H0 时机 bug）：

| Agent | 注入 `todo_update`？ | role-prompt 触发条件 |
|---|---|---|
| **Scout** | ✅ 注入 | **仅在 emit_scout_verdict 之后，且作为 H0_DIRECT executor 继续执行时**调用。Scout 调研阶段（emit_scout_verdict 之前）禁止调 —— 此时还没有 todo 列表存在 |
| **Generator** | ✅ 注入 | 每完成 obligation 必须调（H1/H2 path） |
| **Planner** | ✅ 注入 | 细化 obligation 后用 `replace()` 重置列表（H2 path） |
| **Evaluator** | ❌ 不注入 | 状态推进由 Runner 在 verdict 处理处自动代劳（accept/revise/replan，详见 §5 决策细节 ①） |

**H0 时机修复说明**：emit_scout_verdict 是 Scout 调研阶段的**结尾**（产出 verdict + executionObligations），但对 H0 path 而言它同时是 Scout **执行阶段的开始**——Scout 继续作为 executor 调工具完成任务。原稿"Scout 不注入"会导致 H0 任务全程没有 todo 推进，5 秒消失前用户只看到全 pending 状态。修复：注入并通过 role-prompt 限定调用时机。**FEATURE_104 触发**：Scout role-prompt 加这条限定 → 必须配 eval，dataset 至少 1 H0 多步 case 验证 Scout 在 emit_scout_verdict 后正确推进 todo。

**Evaluator 自动代劳规则**（Runner 层 —— 不动 Evaluator 协议，详见 §5 决策细节 ①）：
- `accept` verdict：所有 **非终态**（`pending` 和 `in_progress`）项自动标 `completed`。Evaluator 接受意味着工作已完成，模型若忘记调 `todo_update` 收尾的 `in_progress` 项也由 Runner 兜底闭合。终态（`completed` / `failed` / `skipped`）保持不变
- `revise` verdict：当前所有 `in_progress` 项一律标 `failed`（注 `Evaluator requested revision`），下一轮 Generator 开始前自动 reset 回 `pending`
- `replan` verdict：整个列表 reset，等待 Planner 重新 `replace()`

#### LLM 行为兜底（避免模型忘记调 `todo_update`）

**两层兜底**（v0.7.30 设计稿曾列三层，启发式那层经 2026-05-03 决策砍掉，详见 §5 决策细节 ③）：

1. **System prompt 强提示**：在 Generator/Planner 的 role prompt 里硬编码"每完成一个 item 必须 call `todo_update`"。**FEATURE_104 触发**：必须配 eval。
2. **节流 reminder**（本 feature 必交付，不延后）：详见 §5 决策细节 ②。常量 `TURNS_SINCE_TODO_UPDATE_REMINDER = 8`，模型连续 N round 没调 `todo_update` 时下一轮注入 `<system-reminder>` 带未完成列表，参考 Claude Code `getTodoReminderAttachments`。**FEATURE_104 触发**：reminder 文本是注入 LLM 的 prompt → 必须配 eval。

#### UI 层（新建 `packages/repl/src/ui/components/TodoListSurface.tsx`）

**视觉规范（严格对齐 Claude Code TaskListV2，但加 KodaX 视口约束）**：

> **视口约束**：列表行数**上限 6 行**（含可选的 summary 折叠行）。Claude Code 视觉上不做硬截断是因为模型一般产出 5-8 条 todo，观感就 5-6 行；KodaX 保留同样的视觉感受但**加一条硬规则**：无论 Scout/Planner 产出多少 obligation（10 条、20 条甚至更多），TodoListSurface 实际渲染永远不超过 6 行。超出部分用 summary 行折叠（见下方 "滚动窗口策略"）。这样 spinner + TodoListSurface + BackgroundTaskBar 的总占屏始终 ≤ 8 行，不压缩 transcript 可见区域。

**消失策略**（对齐 Claude Code 5 秒延迟）：
- 所有 item 为 `completed | skipped` 时，启动 5 秒 timer
- timer 内有任何新 todo 事件 → 取消 timer（任务还在动）
- 5 秒后 `todoItems` 置空 → 组件自然卸载

**自动跟随滚动算法**（`items.length > MAX_VISIBLE_ROWS` 时启用）：

> **重要澄清**：这里的"滚动"指 **每当某个 item 从 `in_progress` 转到 `completed`、下一个 item 进入 `in_progress` 时，UI 自动把视口向下挪一行，让新的 current 对齐到可见位置**。**不是用户可交互的滚动**（不绑快捷键、不响应方向键/鼠标滚轮）。Claude Code 也是同样做法：用户不需要操作，视口自己跟随进度走。若用户想回看全部历史轨迹，在 transcript / tool call 历史里追溯（未来若有强需求，可在 Phase 4 加手动展开热键）。

**锚点计算**：

1. **锚点行 (anchor)**：
   - 优先 = 第一个 `in_progress` item 的索引
   - 无 in_progress 时 = 第一个 `pending` 的索引（任务刚开始/刚被 evaluator 重置）
   - 全部 completed 时 = 最后一个 completed 的索引（5 秒消失前的终态显示）

2. **可视窗口布局**（除 1-2 条折叠摘要外，剩 4-5 行给真实 item）：
   - 锚点**之前**最多 1 条（最近完成的一条，让用户看到"刚做完啥"）
   - 锚点**本身**1 条（current）
   - 锚点**之后**最多 2 条（接下来要做的 pending）

3. **折叠摘要行（始终占行数预算）**：
   - 若锚点之前还有**未显示**的 completed 项 ≥ 1 条：**顶部**插入 `✓ N done`
   - 若锚点之后还有**未显示**的 pending 项 ≥ 1 条：**底部**插入 `☐ +N more`
   - 两条折叠行都不占"4 条真实 item"的预算，作为独立 row 叠加（但总行数仍受 `MAX_VISIBLE_ROWS=6` 约束，视需要压缩真实 item）

4. **failed 项优先级**：若存在 failed 且未在窗口内，强制把其中**最新**一条替换窗口内最近的 pending 位置（让用户立刻看到失败告警）

**窗口前移的具体时机**：

视口的"跟随滚动"不是动画效果（Ink 不支持流畅动画），而是**每次 view-model 重建时重新计算锚点 → 重新渲染**。用户观感就是：
- 一条 `in_progress` → `completed` 时，view-model 锚点自动变为下一条 item
- 顶部折叠摘要 `✓ N done` 的数字 **+1**
- 窗口整体看起来"向上挪了一行"（因为完成的从窗口里移出去，新 pending 从下方进来）
- 全程无动画，直接 re-render

**折叠行样式**：
- `✓ N done`：symbolColor=green，text 用 dim，不响应 id 查询
- `☐ +N more`：symbolColor=dim，text dim
- `kind !== "item"`，不对应真实 TodoItem

**边界**：
- `totalCount ≤ 6`：不启用自动滚动窗口，全部展开；若 completed 多于 4 条也可选在顶部合并折叠成 `✓ N done` 减少噪音（默认只在 `totalCount > 4` 时合并）
- `totalCount = 0-1`：`shouldRender = false`，组件不挂载

所有显示决策集中在 view-model，组件层保持纯渲染。单元测试覆盖：全 pending / 全 completed / 锚点在头部 / 锚点在尾部 / 含 failed / 总数 = 2/6/7/20 几个档位；**自动前移场景**（in_progress 从 item[3] → item[4] 时锚点迁移 + 折叠行数字变化）必测。

#### 事件路由注意

`emitInfoItemToCorrectLayer`（`InkREPL.tsx:1555`）的现存规则要求 managed foreground 期间的 info 条目路由到 worker 层。**TodoListSurface 不走这条**：todo 状态是**跨 worker 的全局任务进度**，挂在 InkREPL 顶层 React state (`todoItems: TodoItem[]`) 而非 `managedForegroundLedger`。实现时在 `onTodoUpdate` handler 里**直接** `setTodoItems(items)`，跳过 `emitInfoItemToCorrectLayer`。

### §5 决策细节（2026-05-03 用户确认）

设计稿初稿对 5 个边角问题留了"概念占位"，落地前必须给出明确处理。这 5 个问题用户已逐一拍板。

#### ① Evaluator → todo ID 映射 —— 不映射，按 in_progress 集合处理

**问题**：Evaluator 返回 `revise` verdict 时，怎么把 verdict 反向映射到具体 TodoItem 标 `failed`？

**否决方案**：
- 在 Evaluator 协议加 `failed_obligation_ids: string[]` —— 要改 `protocol-emitters` + Evaluator 还得知道完整 todo ID 集合（要把 ID 喂进 prompt）+ 触发 FEATURE_104 prompt eval。引入"协议层 + UI 层"双 source of truth。
- 启发式 string match Evaluator note ↔ todo content —— 模糊匹配永远做不对。

**采纳方案**：**revise 时把所有当前 `in_progress` 项一律标 `failed`**，注 `Evaluator requested revision`；下一轮 Generator 开始前所有 `failed` 项 auto-reset 为 `pending`。

理由：
- 不动 Evaluator 协议 → 零 prompt eval 触发
- "本轮没满足质量门槛 → 下一轮重做" 是 revise 的本来语义，不需要 Evaluator 自己挑出哪些
- 用户视觉反馈完整：● Running migration tests → ✗ Run migration tests (Evaluator requested revision) → ☐ Run migration tests （下一轮）
- 若 v0.8.x 真出现 3+ use case 需要按 obligation 粒度反馈，再加协议字段（YAGNI）

#### ② Layer 2 节流提醒 —— 本 feature 必交付，不延后

**作用域 + 计数规则**：
- 计数器 **per managed-task**（不 per agent）：Generator → Planner handoff 不重置，避免 handoff 边界刚好 reset 让 reminder 永远不发
- 增量：每个 Runner round 结束时，若该 round 内**没**调 `todo_update`，counter +1
- 重置：调一次 `todo_update` → 0；phase 切换（Scout → Planner → Generator → Evaluator） → 0

**阈值**：`const TURNS_SINCE_TODO_UPDATE_REMINDER = 8`（KodaX `packages/coding/src/task-engine/todo-throttle-reminder.ts` 文件级常量，不暴露 config——常量与节流逻辑同模块，便于集中演进）。Claude Code `TURNS_SINCE_WRITE = 10` 是参考起点；KodaX 一个 round 含一次 tool call + 模型思考，比 Claude Code 单轮重一档，8 是经验起点。**未来按 telemetry 调常量值即可**。

**reminder 文本**（`<system-reminder>` 注入下一轮 user message 前）：

```
<system-reminder>
You have not called todo_update in 8 iterations. Pending items:
- todo_3: Update type definitions
- todo_4: Regenerate snapshot
- todo_5: Verify end-to-end flow
If you have started or finished any of these, call todo_update now.
</system-reminder>
```

**前置门**：todo list 为空（obligations < 2 没种入）时不发 reminder（避免对纯简单任务噪声）。

**FEATURE_104 触发**：reminder 文本是注入 LLM 上下文的 prompt 内容 → **必须配 eval**。dataset 设计：模型在 reminder 注入后下一轮，至少 1 个 alias 上正确补调 `todo_update` 推进进度。

#### ③ Layer 3 启发式推断 —— 砍掉，不实现

**为什么 v0.7.30 设计稿初版列了 Layer 3，本版砍掉**：

1. **触发条件天然模糊**："write_file 路径命中某 item 关键词" —— item content 是自然语言（"Update type definitions"），关键词提取做不出健壮规则；正则化又破坏 Scout 的语义自由度
2. **三层 over-engineering**：Layer 1 (prompt) + Layer 2 (reminder) 已经覆盖"模型主动推进 + 模型遗忘补救"两条主路径；Layer 3 只是给"两条都失败"兜底，但失败之后用户最多看到 stale UI 两轮（直到下一次 Layer 2 reminder 推进），可接受降级
3. **3+ use case 规则**（CLAUDE.md "NEVER abstract until 3+ concrete use cases"）：v0.7.34 落地后 0 真实数据，先不抽象
4. **未来路径**：若 v0.7.35-37 telemetry 显示 stale UI > 2min 占比 > 5%，再单独立 feature 加启发式

**砍 Layer 3** = 验收标准、影响范围、单测列表里全删除"Runner 启发式 in_progress 推断"，节省 ~80 LoC + 关键词维护成本。

#### ④ Resume 行为 —— TodoList 是 task-scoped，不 session-scoped

**问题**：`kodax -c` 恢复历史 session 时，之前的 todo list 怎么处理？

**采纳方案**：**TodoList 生命周期 = managed-task 生命周期，不进 session lineage snapshot**。

具体语义：
- 一个 session 由多个 task 组成，每个 task = 一次 managed-task run
- TodoListSurface 在 Scout 入口种入时挂载，所有项 completed 后 5 秒延迟消失后自动卸载
- Resume session → 用户开始**新** task → 新 Scout → 新 obligations → 新 TodoList
- 旧 task 的 todo 进度**保留在 transcript history**（用户翻 history 时能看到 `[todo_update] todo_3: completed` 的工具调用日志），不在 TodoListSurface 里复活
- **`KodaXSessionCompactionEntry` / `lineage.ts` 零改动**

**为什么不做 session-scoped resume**：
- 跨 session resume 只对"任务还没做完就退出"场景有意义；KodaX 当前 task 在一次 run 内闭合（Scout → Generator → Evaluator → final answer），mid-task abort 占比极低
- 真做需要 TodoList snapshot 进 lineage，要扩 schema、要处理 obligation 漂移（resume 时 Scout 又 emit 一份新 obligations，new ID 集合 ≠ old ID 集合）—— 复杂度爆炸
- v0.7.34 后若有 3+ 用户报告"resume 后想看上次的 todo 列表"，再单立 feature

#### ⑤ Unknown id 处理 —— 返回 ok:false + valid id 列表

**场景**：Generator 调 `todo_update({id: "todo_99", status: "completed"})` 但 `todo_99` 不存在（模型 hallucination / Planner replace() 后的 stale ID）。

**采纳方案**（已在前文 `todo_update` 工具实现里 inline）：

```ts
if (!ctx.todoStore.has(id)) {
  return {
    ok: false,
    reason:
      `Unknown todo id: ${id}. ` +
      `Current valid ids: ${ctx.todoStore.allIds().join(', ')}.`,
  };
}
```

理由：
- **不抛异常**：一次 unknown id 不该让 Runner crash
- **不 silently no-op**：会让 LLM 认为推进成功 → UI 永远停在 stale 状态
- **返回 valid id 列表**：模型下一轮自我纠正
- **trace 已记录**：所有 tool call 输入输出已在 trace 里，需要 telemetry 时 grep 即可，不新增 `onTodoUpdateError` 事件（YAGNI）

**Planner replace() 的 race**：Planner 调 `replace(newItems)` 后旧 ID 全部失效。Generator 下一轮调 stale ID → 走 unknown id 路径 → 模型从 reason 文本看到新 valid id 列表 → 自适应。无锁、无 versioning、无 race condition。

**测试 / eval 覆盖**（用户 2026-05-03 要求验证此方案正确性）：

1. **Hermetic 单测**（`packages/coding/src/tools/todo-update.test.ts`，无 LLM）：
   - case A: 调 `todo_update({id:"todo_99",...})` 时 store 无此 id → 返回 `{ok:false, reason:/unknown.*todo.*id/i}` 且 reason 包含全部 valid id
   - case B: store 已有 todo_1/todo_2/todo_3 → 调 `todo_update({id:"todo_2",...})` 返回 `{ok:true}` 且 store 状态正确更新
   - case C: Planner replace([newItem with id=newA]) 后旧 todo_1 失效 → 调 `todo_update({id:"todo_1",...})` 返回 `{ok:false}` 且 reason 列出 newA
   - case D: empty store → 调 `todo_update({id:"todo_1",...})` 返回 `{ok:false}` 且 reason 显示 valid id 列表为空（明确告诉模型当前没有任何 todo）

2. **Eval case**（`tests/feature-097-unknown-id-recovery.eval.ts`，**FEATURE_104 触发**）：
   - prompt 模型在一个故意被注入"hallucinated id" 的 conversation 里——前一轮 todo_update 返回 `{ok:false, reason:"Unknown todo id: todo_99. Current valid ids: todo_1, todo_2"}`，验证模型下一轮**正确选用** `todo_1` 或 `todo_2` 之一推进，而不是继续 hallucinate 或卡住
   - 8 alias mean ≥80% 通过率（与 FEATURE_106 同阈值）
   - 不通过 → 说明 reason 文案需要更明确的引导（e.g., 加 "→ Please retry with one of:" 等指令性文字），迭代直到达标

### 验收标准

1. **Scout 自动种入**：AMA H0/H1/H2 任何一档任务启动后，若 Scout 产出 ≥ 2 条 `executionObligations`，spinner 下方立刻出现 TodoListSurface，全部 `pending`；少于 2 条时不渲染（对齐 Claude Code "任务太简单不干扰"）。
2. **Generator 实时推进**：Generator 每调一次 `todo_update`，对应 item 符号 / 颜色 / 文本（content ↔ activeForm 切换）实时变更，无需等 phase 切换。
3. **Evaluator verdict 自动收尾**：
   - `accept` → 所有剩余 `pending` 转 `completed`
   - `revise` → 受影响 item 变 `failed`（红色 ✗ + note），下一轮开始前重置 `pending`
   - `replan` → TodoListSurface 整体 reset，Planner 重新 `replace()` 后出现新列表
4. **视觉对齐 Claude Code**：符号（☐ / ⚡ / ✓ / ✗ / ⊘）+ 颜色（dim / cyan bold / green / red / gray）+ 左侧装订线 `▏` + 右上角 `N/M completed` 全部就位。
5. **统一文本来源**：所有状态行显示 `content`（来自 Scout `executionObligations: string[]`，零变形），状态语义靠符号 + 颜色 + 装订线表达。
6. **5 秒延迟消失**：全部 completed / skipped 后 5 秒消失；5 秒内任意 todo 事件取消 timer 继续显示。
7. **Child task 并行**：`dispatch_child_task` 场景下 owner 分组渲染（`main` + `child_<id>` 各自占一块），每组独立"一个 in_progress"约束，可并行跑。
8. **H0/H1/H2 行为一致**：H0 若产出 2 条 obligation 也显示（H1 H2 更不例外），UI 层无任何复杂度分档逻辑。
9. **视口占用 ≤ 6 行**：任何 obligation 数量下（2 条、7 条、20 条），TodoListSurface 实际渲染不超过 6 行。超出时通过顶部 `✓ N done` + 底部 `☐ +N more` 折叠摘要行维持上限。Claude Code 观感（5-6 行短列表）被严格保持。
10. **完成后自动跟随滚动**：当某一项从 `in_progress` 转 `completed`、下一项进入 `in_progress` 时，视口自动前移 —— 顶部折叠 `✓ N done` 数字 +1，窗口内 current 锚点自动对齐到新 in_progress 行。**全程无需用户按键或鼠标操作**。E2E 场景：总 12 条 obligation，Generator 依次调 `todo_update` 推进 1→12，每次推进后 re-render 锚点正确移动。
11. **与既有 UI 不冲突**：TodoListSurface 挂载时不挤掉 spinner / BackgroundTaskBar / repo-intel trace；PromptFooter 布局正常；spinner + TodoListSurface + BackgroundTaskBar 的总占屏始终 ≤ 8 行。
12. **无回归**：`npm run test` / `npm run build` 全绿；既有 AMA 测试（`runner-driven.test.ts`、`InkREPL` E2E）通过。
13. **H0 Mini-Planner 强度 release gate**：`tests/feature-097-h0-mini-planner-strength.eval.ts` 必须跑过 8 alias × 4 case × 2 variant = 64 cells（参考 FEATURE_106/112 同量级），按决策矩阵选 ship 的 prompt 变体。release 前 4 个量化指标全部满足阈值（详见 §H0 Mini-Planner 强度 A/B Eval）。
14. **Unknown id 自我恢复**：`tests/feature-097-unknown-id-recovery.eval.ts` 在 8 alias mean ≥80% 通过率（详见 §5 决策细节 ⑤）。

### 非目标

- **不** 做 fs 持久化（Claude Code V2 的 `~/.claude/tasks/*.json` + fs.watch + proper-lockfile）。KodaX 是单进程 CLI，纯 in-memory 足够
- **不** 做 todo 依赖图（V2 的 `blocks` / `blockedBy`）。KodaX 的 DAG 已由 Scout/Planner/Evaluator 表达，todo 层不再重复
- **不** 做跨会话持久化（resume 时恢复 todo 列表）。放 Phase 4 独立考虑，不在 v0.7.34 范围
- **不** 复刻 Claude Code 的 V2 `TaskCreate` / `TaskUpdate` 工具细分（v2 有 id 生成、高水位、`claimTask({ checkAgentBusy })` 等 swarm 专用能力）。KodaX 只需一个简化工具 `todo_update`，创建交由 Scout verdict 自动完成
- **不** 做基于 H0/H1/H2 的硬编码折叠/展开策略。复杂度分层**由 Scout 产出的 obligation 数量自然决定**，不由 UI 判断
- **不** 做**用户交互式滚动**（键盘 up/down、鼠标滚轮、PageUp/PageDown 展开全量列表等）。视口"滚动"仅指**自动跟随当前 in_progress 的 re-render 式窗口前移**，对齐 Claude Code。用户想看全量历史可追溯 transcript 里的 tool call 记录；若未来出现强需求，再用独立 feature 加手动展开热键
- **不** 把 todo 事件走 `emitInfoItemToCorrectLayer` 路由 / 进入 history。Todo 是 PromptFooter 层的活动状态，不是对话历史
- **不** 改动 `BackgroundTaskBar` 的 pill 风格。两者是并列组件，各司其职

## FEATURE_112: Investigation-Scale-Aware Routing — Read-Scope Fix

**Status**: Released (v0.7.34, 2026-05-04, commit `b5ff2b0`)
**Priority**: High
**Category**: Routing / AMA
**Implementation surface**: `packages/coding/src/reasoning.ts` + `packages/coding/src/task-engine/_internal/managed-task/role-prompt.ts` + `benchmark/datasets/read-scope-routing/` + `tests/feature-112-read-scope-routing.eval.ts`. **零跨包改动、零新基础设施、零 child-executor 改动**。

### 用户痛点

实战反馈：KodaX 在以下场景下表现"只有 Scout 一个人在做"：
- "为什么 KodaX 路由系统在 read-only 任务上总是不升级"——Scout 跑 15 工具调用，最后 H0 单飞回答
- "解释整条 AMA 数据流" / "审查 packages/coding 的依赖结构"——Scout 读 8+ 文件，无 evaluator audit
- 多线程明显的任务（"审计三个独立子系统的安全性"）——Scout 不派 child，深 dive 第一个，剩下两个被挤压

三轮独立调研（含对抗评审）定位的根因：**KodaX 路由系统对"read-only 重度调研任务"系统性失明**。修复对称 FEATURE_106（mutation 路径 multi_file_h0_rate 15.6% → 0%）的成功路径：纯 prompt + 路由 heuristic，不引入 runtime guardrail（避免触发时机错误 / child-executor 跨层耦合）。

### 四点根因（按权重排序）

| # | 缺陷 | 文件 | 失败模式 |
|---|------|------|---------|
| 1 | `deriveTopologyCeiling` 只看 mutationSurface + assuranceIntent | `reasoning.ts:1154` | read-only 任务无论多复杂，天花板永远 H0_DIRECT |
| 2 | `fanoutReason` 对 unknown 任务硬写 "No high-value shard class detected" | `reasoning.ts:1539-1540` | 对 LLM 是负面 dispatch 信号，鼓励 Scout 不派 child |
| 3 | Scout role-prompt 的 SCOPE COMMITMENT 只覆盖 mutation scope | `role-prompt.ts:476` | 没有对称的 investigation-scope 触发条件，模型不知何时该升 H1 |
| 4 | `topologyCeiling: H1_EXECUTE_EVAL` 是裸值，无语义说明 | `role-prompt.ts:104` | 即使 Slice 1 解锁了 H1 ceiling，Scout 不知道 H1 意味着 "Evaluator can audit" |

### 实施切分（4 Slice + 1 Eval）

#### Slice 1 — `deriveTopologyCeiling` 加 complexity 维度（reasoning.ts）

```typescript
function deriveTopologyCeiling(
  mutationSurface: KodaXMutationSurface,
  assuranceIntent: KodaXAssuranceIntent,
  complexity?: KodaXTaskComplexity,  // NEW
): KodaXHarnessProfile {
  if (mutationSurface === 'read-only' || mutationSurface === 'docs-only') {
    if (assuranceIntent === 'explicit-check') return 'H1_EXECUTE_EVAL';
    // FEATURE_112: read-only + complex/systemic → H1 ceiling.
    // 仅放宽天花板，Scout 仍持有升级决策权（FEATURE_061）。
    if (complexity === 'complex' || complexity === 'systemic') {
      return 'H1_EXECUTE_EVAL';
    }
    return 'H0_DIRECT';
  }
  return 'H2_PLAN_EXECUTE_EVAL';
}
```

两个调用点（`selectHarnessProfile` 和 `createReasoningPlanFromDecision`）都已能直接传入 `decision.complexity`。Backward-compat：`complexity` 参数 optional，省略时退化为 v0.7.33 行为。

**副作用**：两个既有 reasoning.test.ts 测试在 read-only + complex 场景下需要更新预期 ceiling（从 H0 → H1），已在同 commit 内更新并加 `// FEATURE_112` 注释说明这是 intentional behavior 改变（harnessProfile 字段不变，只动 ceiling）。

#### Slice 2 — `fanoutReason` 文案中性化（reasoning.ts）

针对 `primaryTask === 'unknown' && fanoutClass === undefined` 的 controller overlay，把硬写的 "No high-value shard class was detected for this task." 替换为中性文案：

> Task scope is unclassified; dispatch_child_task remains available if investigation threads emerge during scoping.

**关键约束**：不动 `fanoutAdmissible` 计算、不动 `resolveAmaFanoutClass` 白名单、不动 `isAmaFanoutClassActive` 的 ambiguous gate。仅替换 reason 文案——避免 evidence-scan 文案的 semantics leak（让 LLM 误以为这是 investigation 类型，错误走"广撒网式 grep"模式而不是 clarify-first）。

#### Slice 3 — SCOPE COMMITMENT 对称扩展 + H0 反向锚（role-prompt.ts）

在现有 SCOPE COMMITMENT 段（FEATURE_106 mutation rule 之后）追加 investigation-scope rule + multi-thread early-decision rule，与 FEATURE_106 完全对称；并在三条规则之前**显式声明 "Default harness is H0_DIRECT"**（Stage 1 + iteration eval 数据驱动的设计决定，详见下面"实测迭代"段）：

```
SCOPE COMMITMENT (hard rule):
  Default harness is H0_DIRECT. The triggers below are escalation conditions, not defaults —
  apply them only when one fires for the current task. Simple lookups, single-file edits, and
  pure answer questions stay at H0 even when the rules below are present in this prompt.

  • Mutation scope: If you intend to write ≥2 files OR start a project from scratch,
    call emit_scout_verdict({confirmed_harness: H1 or H2}) BEFORE the first write.
    The scope guardrail will surface belated commitments and slow you down.
  • Investigation scope: If your read-only investigation reaches ≥5 distinct files
    OR ≥8 searches without converging on a diagnosis, treat that as a signal the work
    has exceeded H0 — emit emit_scout_verdict({confirmed_harness:"H1_EXECUTE_EVAL"}) so
    an Evaluator can audit your conclusion. Continuing solo past this threshold loses
    the audit signal.
  • Multi-thread early decision: If your initial 1-2 scoping turns reveal ≥2
    independent investigation threads, prefer dispatch_child_task over deep-diving
    solo (per RULE A/B below). Dispatching AFTER you have already deep-dived is
    wasted work — decide early.
```

##### 实测迭代（Stage 1 + 两轮 iteration eval 数据驱动）

**Stage 1**（4 task × 2 variant × 8 alias = 64 cells，wall-clock 16min）首次实测：
- 三个 deep class（read-deep-systemic / read-multithread / read-unknown-heavy）alias mean H1 升级率 **+50pp**（与 FEATURE_106 mutation 路径 `multi_file_h0_rate` 15.6%→0.0% 同量级改善）。
- read-shallow-qa 上 mmx/m27 alias 出现 100%→0% 退化嫌疑。

**回归诊断**：实读 mmx/m27 输出码本，发现 stage 1 上 feature_112 的"退化"实际是输出乱码（`"packageashawn张扬你是"` hallucination），不是"误升 H1"——是 mmx 对长 prompt 的稳定性问题。

**Iteration round 1**（4 variant × 2 task × 8 alias = 64 cells）测试两条改进假设：
- `feature_112_compact`：砍 SCOPE COMMITMENT verbosity ~40%，假设 mmx 是长度敏感
- `feature_112_anchor`：保持完整 escalation 规则，前置显式 "Default is H0_DIRECT" 反向锚，假设是 H0-default 语义不够明显

read-shallow-qa H0 retain（6 effective alias，2 mimo 全 format-fail 排除）：
| Variant | H0 alias 数 |
|---|---|
| baseline (current_v0733) | 3/6 |
| feature_112 (Stage 1 prompt) | 4/6 |
| feature_112_compact | 4/6 |
| **feature_112_anchor** | **5/6** ⭐ |

**关键发现**：format pass = quality pass，**没有任何模型把 shallow-qa 真升 H1**。所有 0 都是输出乱码（format fail），stage 1 的 mmx 退化是单 run noise 而非真实分类错误。anchor 在 5/6 alias 上稳定输出 H0（多了 kimi + ark/glm51）。

**Iteration round 2**（聚焦 deep-systemic 验证 anchor 不损失主目标，1 task × 2 variant × 8 alias = 16 cells）：
- baseline: 0/6 alias 升 H1
- **feature_112_anchor: 2/6 alias** 升 H1（zhipu/glm51 + kimi）
- dominant variant: feature_112_anchor over baseline

**综合两 task 总分**：

| Variant | shallow H0 retain | deep H1 escalate | 总分 |
|---|---|---|---|
| baseline | 3/6 | 0/6 | 3 |
| feature_112 (Stage 1) | 4/6 | 3/6 | 7 |
| **feature_112_anchor** | **5/6** | 2/6 | **7** |

anchor 与 feature_112 总质量持平但分布不同 — anchor 在简单 QA 上更稳，feature_112 在 deep escalation 上单 run noise 边际优势。**用户偏好显式表达"不要让简单任务无故升级"**，且 anchor 是 feature_112 的纯添加版（零删除任何 escalation rule，逻辑上保留 deep gain），故 ship anchor 而非原 Stage 1 prompt。

**单 run noise 限制**：当前所有数据均为 1 run/cell，单 alias 单 task 数据 ±100pp 可能。Post-pilot 阶段建议升级到 3 run/cell 跑全 4 task × 4 variant × 8 alias = 128 cells × 3 = 384 cells（约 1 小时 wall-clock）以降低噪音、固化 expect.fail 阈值。本版本以 alias mean 趋势 + 总分等价为决策依据是合理的。

**为什么不做 runtime guardrail（曾经的 v2 设计）**：FEATURE_106 已经实证纯 prompt 路径在 mutation 路径上达到 15.6% → 0% 改善；mutation-reflection runtime guardrail 在 Stage 3 生产 eval 中触发率 = 0（成了"保险"，不是"主修正器"）。runtime guardrail 还有四点真实风险：(i) child-executor guardrail 转发耦合（parent ctx.guardrails 通过 dispatch_child_task 转发给 child Runner，child 读 5 文件会让 parent tracker 触发 hint）；(ii) 触发时机错误（5 文件后 nudge 时已错过早期分流时机，符合 role-prompt.ts:523 "Dispatching after deep-dive is wasted work" 警告）；(iii) compaction 摘要被 hint 稀释；(iv) tool_result rewrite 写入 session snapshot，resume 时重放风险。静态 prompt rule 触发时机更对（决策时 vs. 证据已收集后）、零基础设施、零 child-executor 改动。

#### Slice 4 — decisionSummary 给 ceiling 加语义（role-prompt.ts）

`decisionSummary` 里的 `Topology ceiling: H1_EXECUTE_EVAL` 当前是裸值，Scout 不知道这意味着 "Evaluator audit"。补一行 conditional 语义说明：

```
Topology ceiling: H1_EXECUTE_EVAL — Evaluator can audit your conclusion if you escalate to H1
Topology ceiling: H2_PLAN_EXECUTE_EVAL — Planner+Generator+Evaluator pipeline available if you escalate to H2
Topology ceiling: H0_DIRECT  // 无附加说明，避免噪声
```

闭合 Slice 1 的死锁：Slice 1 把 H1 ceiling 解锁，但 Scout 没有从裸值推断"我应当考虑 emit H1"的能力 ——Slice 4 补上这条推理链。

#### Layer 2 Eval — `benchmark/datasets/read-scope-routing/`

#### Docs-only coverage（事后补的 eval）

`deriveTopologyCeiling` 把 `docs-only` 与 `read-only` 合并到同一分支（H0 默认，complex/systemic 才升 H1）。read-scope-routing dataset 只覆盖 read-only 维度，docs-only 是"逻辑兜底"。补 `benchmark/datasets/docs-scope-routing/`（2 case × 2 variant × 8 alias = 32 cells）专门验证 docs-only 端不会乱升级。

`current_v0733`（baseline）vs `feature_112_anchor`（已上线变体），单 run/cell。
