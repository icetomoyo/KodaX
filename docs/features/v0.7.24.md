# v0.7.24 Feature Designs

> 中文标题：`v0.7.24 Feature 设计`
>
> Version: `v0.7.24`
>
> Status: Released (2026-04-20)
>
> Theme: Primitive infrastructure — package restructure + unified tracing

---

## 总览

v0.7.24 把 v0.7.23 刚定型的 Layer A 原语**搬到正确的包中**，并引入**统一 Tracer / Span 模型**。两者一起构成"原语基础设施层"：包重组给出清晰分层边界，tracing 给出跨层观测能力。合并落地避免 082 的包移动和 083 的 tracing 内容分两版各 touch 同批新建包。

## FEATURE_082: Package Restructure

**Status**: Completed  
**Started**: 2026-04-20  
**Completed**: 2026-04-20  
**Released**: v0.7.24  
**Priority**: High  
**Category**: Core

### 背景

- v0.7.23 FEATURE_080/081 临时把 Agent / Session / CompactionPolicy 放在 `@kodax/coding` 下；这违反 Layer A 的"通用基础"定位
- 现有 `@kodax/ai/cli-events/` 是 ~800 行 CLI bridge 解析逻辑（gemini-parser、codex-parser），不属于 AI 抽象层（GLM F-8）
- MCP 当前在 `packages/coding/src/capabilities/providers/mcp/`，与 coding 专属逻辑混在一起；其渐进式披露机制应独立
- 外部 SDK 消费者现在只能选 `@kodax/coding` 全家桶或 `@kodax/ai` 最薄层；中间没有"想嵌入 Agent 运行时但不要 coding preset"的入口

### 设计

#### 新包清单

```
@kodax/ai                     —— 现状保持，清掉 cli-events/ (搬到 @kodax/coding)
@kodax/core                   —— 新：Layer A 原语
                                 - Agent / Tool / Session / Handoff / Guardrail / Runner types
                                 - DefaultSummaryCompaction
                                 - CompactionPolicy 接口
                                 - Message / SessionEntry data shapes
@kodax/tracing                —— 新：独立以便外部替换 OpenTelemetry/Langfuse
                                 - FEATURE_083 填充
@kodax/session-lineage        —— 新：从 @kodax/agent 抽出 lineage 能力
                                 - LineageExtension
                                 - LineageCompaction
                                 - KodaXSessionLineage 类（向后兼容别名，v0.7.27 移除）
@kodax/mcp                    —— 新：整体搬迁 capabilities/providers/mcp/*
                                 - 保留 progressive disclosure 全部 5 模式：
                                   lazy connect / 两级描述符 / search-describe / elicitation / cache
                                 - McpCapabilityProvider, McpServerRuntime, catalog.ts 等照搬
@kodax/capabilities           —— 新：Layer B 组合能力（FEATURE_084 v0.7.26 往里填 Scout/Evaluator 等）
@kodax/skills                 —— 现状保持（已独立）
@kodax/agent                  —— 保留为 coding-specific session / extension runtime
@kodax/coding                 —— 现状保持（coding preset）
@kodax/repl                   —— 现状保持（CLI/REPL UX）
```

#### MCP 迁移关键约束

**必须保留的 5 个渐进式披露模式**（抽取时不能丢）：
1. **lazy connect**：`connect: 'lazy' | 'prewarm' | 'disabled'` per-server 配置
2. **两级描述符**：`McpCatalogItem`（轻，~10 字段）vs `McpCapabilityDescriptor`（重，含 schemas）
3. **search → describe 两步走**：`provider.search(query)` → catalog 命中 → `describe(id)` 取完整 schema
4. **elicitation**：工具调用参数不全时向用户询问补齐，而非一次给 LLM 全部参数
5. **本地 cache snapshot**：`.kodax/mcp/` 下的 catalog 缓存

**不做**：不引入 `ProgressiveCapabilitySource` 泛化接口（遵守 CLAUDE.md "3+ real cases 才抽象"规则，当前只有 MCP 一个实现）。

#### cli-events 清理 — **Deferred to FEATURE_086 (v0.7.27)**

原计划：把 `@kodax/ai/src/cli-events/` 的 10 个文件（gemini-parser、codex-parser、pseudo-acp-server 等，~800 行）整体搬到 `@kodax/coding/src/cli-bridges/` 或独立为 `@kodax/cli-bridges`。

**v0.7.24 实施时发现**：cli-events 被 `@kodax/ai` 内部的 3 个 CLI-wrapper Provider（`providers/acp-base.ts`、`providers/codex-cli.ts`、`providers/gemini-cli.ts`）直接消费。仅搬 cli-events 会导致 ai → coding 循环依赖；干净做法需要把三个 Provider 子类 + registry 静态注册表一起重构成 extensibility 注册机制，远超 082 的"包重组 + tracing"scope。

**处置**：cli-events 留在 `@kodax/ai` 不动。**清理延至 FEATURE_086 (v0.7.27)**，与其他 legacy purge 和 KodaX 前缀清理一批完成（FEATURE_086 已在那一版大动 @kodax/ai）。本版不计入 082 的 cli-events 条目。

#### 依赖方向（严格）

```
@kodax/coding ─┬─> @kodax/core
               ├─> @kodax/tracing
               ├─> @kodax/mcp
               ├─> @kodax/capabilities
               ├─> @kodax/session-lineage
               ├─> @kodax/skills
               ├─> @kodax/agent
               └─> @kodax/ai

@kodax/capabilities ──> @kodax/core, @kodax/tracing
@kodax/session-lineage ──> @kodax/core
@kodax/mcp ──> @kodax/core
@kodax/tracing ──> (无，独立)
@kodax/core ──> @kodax/ai (只引 provider types)
```

## FEATURE_083: Unified Tracer, Span, and TracingProcessor

**Status**: Completed  
**Started**: 2026-04-20  
**Completed**: 2026-04-20  
**Released**: v0.7.24  
**Priority**: High  
**Category**: Core

### 背景

- 现有 trace 是碎片化事件：`OrchestrationTraceEvent`、`RepoIntelligenceTraceEvent`、`resilience telemetry`，各自有不同 shape
- 外部 SDK 消费者想把 KodaX 的执行喂到自己的 observability（OpenTelemetry、Langfuse 等），没有统一入口
- FEATURE_084 v0.7.26 重写 Scout 时需要 tracing 作为观测基础先落地 —— 所以 083 必须在 084 之前完工
- openai-agents-python 的 `Trace` + `Span` + `TracingProcessor` 三件套是公认的好形状

### 设计

```ts
// @kodax/tracing

export interface Trace {
  readonly id: string
  readonly startedAt: number
  readonly rootSpan: Span
  readonly metadata: ReadonlyMap<string, unknown>
}

export interface Span {
  readonly id: string
  readonly traceId: string
  readonly parentId?: string
  readonly name: string
  readonly data: SpanData
  readonly startedAt: number
  endedAt?: number
  error?: SpanError
  addChild(data: SpanData): Span
  end(): void
}

export type SpanData =
  | AgentSpanData            // Runner.run(agent, ...) 一轮
  | GenerationSpanData       // 一次 LLM 调用
  | ToolCallSpanData         // 一次 tool execute
  | HandoffSpanData          // Agent 间转移
  | CompactionSpanData       // 一次 compaction
  | GuardrailSpanData        // 一次 guardrail check
  | EvidenceSpanData         // repo-intel / evidence acquisition
  | FanoutSpanData           // 并行 fanout

export interface TracingProcessor {
  onSpanStart(span: Span): void
  onSpanEnd(span: Span): void
  onTraceEnd(trace: Trace): void
  shutdown?(): Promise<void>
}

export function addTracingProcessor(p: TracingProcessor): () => void
export function setTracingProcessors(ps: TracingProcessor[]): void
```

#### 默认 Processor

- `ConsoleTracingProcessor`：开发时 stdout 输出
- `FileTracingProcessor`：写入 `.kodax/.traces/{traceId}.jsonl`

#### 收编现有 trace

把以下都迁到统一 Span 模型：
- `OrchestrationTraceEvent` → `AgentSpanData` / `HandoffSpanData`
- `RepoIntelligenceTraceEvent` → `EvidenceSpanData`
- `resilience/retry` telemetry → 嵌入到对应 Span 的 `metadata`
- `FEATURE_064` cost observatory → `GenerationSpanData.usage` 字段

原事件接口保留到 v0.7.27 FEATURE_086 前缀清理时移除。

#### Run 接口集成

```ts
Runner.run(agent, input, { tracer: myTracer })        // 可注入
Runner.run(agent, input)                              // 用默认 tracer
```

### Contract

**Public API（新增）**：`Trace`、`Span`、`SpanData` 子类、`TracingProcessor` 接口、`addTracingProcessor` / `setTracingProcessors` 函数，从 `@kodax/tracing` 导出。

**Public API（迁移）**：Agent / Handoff / Runner / Session / CompactionPolicy 等 v0.7.23 临时在 `@kodax/coding` 的 types 迁到 `@kodax/core`；`@kodax/coding` re-export 保持兼容。

**Public API（弃用）**：`OrchestrationTraceEvent` 等旧 trace 事件标 `@deprecated`，v0.7.27 移除。

**Internal**：Scout/Generator/Evaluator 在 v0.7.26 FEATURE_084 重写时就用 Span emit，tracer 是观测入口。

### Non-Goals

- **不**实装远程 trace 传输（外部 processor 自己实现）
- **不**引入 trace sampling（默认 always-on，性能问题留到未来）
- **不**改变 FEATURE_034 extension runtime 的事件订阅 API

### Dependencies

- **前置**：FEATURE_079 (v0.7.22) task-engine 拆分；FEATURE_080/081 (v0.7.23) Agent primitive 和 Session 到位
- **后置**：
  - FEATURE_084 (v0.7.26) — 重写 Scout 时 emit Span
  - FEATURE_085 (v0.7.26) — Guardrail 执行 emit GuardrailSpanData
  - FEATURE_087 (v0.7.28) — ConstructionRuntime 的 stage/test/activate 过程 emit Span

### Testing

- 单元：Span 嵌套、Trace 结构、Processor 调用时序
- 单元：默认 ConsoleTracingProcessor 输出格式稳定
- 集成：SA 路径 end-to-end 产出完整 Trace（AgentSpan 包含 GenerationSpan + ToolCallSpan）
- 迁移：现有 orchestration trace 事件与新 Span 输出等价（dual emission 过渡期）
- 第三方集成：`examples/otel-export.ts` 示例展示如何接 OpenTelemetry

### Risk

- **性能开销**：Span 创建应轻量（无 await、无序列化）；batch flush 由 processor 负责
- **API shape 锁定**：Trace / Span 标 `@experimental` 在 v0.7.x 内

### Timeline

v0.7.24 合并 082 + 083。包重组 + tracing 一起测试稳定后发版。

### 设计偏差

1. **@kodax/capabilities 删除**（FM-2）：原设计为第 5 个新包（Layer B 组合能力占位），但本版落地时仅有 `export {};` 一行、违反 CLAUDE.md "3+ real cases 才抽象" 规则。用户审核确认后删包，FEATURE_084 (v0.7.26) 真要 Scout/Planner/Generator/Evaluator 落地时再重新创建（届时一次 mkdir + package.json 即可）。最终 9 个包（原计划 10）。
2. **cli-events 延期**：原设计"搬到 coding/src/cli-bridges/ 或独立为 @kodax/cli-bridges"。实施时发现 `@kodax/ai/src/providers/{acp-base,codex-cli,gemini-cli}.ts` 内部消费 cli-events，仅搬 cli-events 会造成 ai→coding 循环依赖；干净做法需要把 3 个 Provider 子类 + registry 静态注册表一起重构，远超 082 scope。延至 FEATURE_086 (v0.7.27) 与 KodaX 前缀清理合并处理。FEATURE_086 设计文档已追加第 9 条说明。
3. **@kodax/core 添加 `Capability` 类型**：原设计未明文；实施 Slice 4 MCP 迁移时发现 `CapabilityProvider / CapabilityKind / CapabilityResult` 是 Layer A 性质（第三方 RAG / 自定义 index 都要实现），顺势从 `coding/extensions/types.ts` 抽到 `@kodax/core/capability.ts`。

### FEATURE_093 顺带完成

借包重组的 import path 大规模重绘时机，同步做了 **coding + repl 内部循环依赖解耦**（原计划立项到 v0.8.0 的 FEATURE_093）：

- **coding 包**：`types.ts ↔ extensions/runtime.ts ↔ tools/*` ~40 个循环 + `agent.ts ↔ client.ts`
  - 引入 `extensions/runtime-contract.ts` 窄接口（6 个方法），`types.ts` 用它替代 `KodaXExtensionRuntime` 全类型引用
  - agent.ts 内部对 `options.extensionRuntime` 做一次 `as KodaXExtensionRuntime` cast（单点下沉，不在各 call site 重复）
  - `child-executor.ts` 对 `runKodaX` 改用 computed-spec 动态 import 打破 tools→agent 回边
  - `agent.ts` 移除 `KodaXClient` 导入（原仅为 re-export），barrel 直接从 client.ts 导出
- **repl 包**：`tui/index.ts ↔ components/*`、`ui/shortcuts/index.ts ↔ provider/hooks`、`completers/argument-completer ↔ command-arguments`、`project-harness ↔ project-harness-core`
  - TUI components 改从 `renderer-runtime.ts` 直接引 Box/Text
  - shortcuts components 改从 `useShortcut.ts` / `ShortcutsRegistry.ts` / `defaultShortcuts.ts` 直接引（相应测试 mock 路径更新）
  - completers 的共享类型抽到新文件 `completers/types.ts`
  - project-harness 23 个 interface 抽到新文件 `project-harness-types.ts`

**最终状态**：全仓 madge 包间循环 **0 个**，包内循环 **1 个**（repl 的 `commands/builtin.ts ↔ interactive/commands.ts ↔ commands/index.ts`三角，因为 `BUILTIN_COMMANDS` 是 ~1900 行内联命令数组，移动需要大量 handler 闭包重绑定——保留给 FEATURE_093 剩余清理，不阻塞本版发版）。**从 ~50 降到 1**，~98% 消除。
