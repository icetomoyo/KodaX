## FEATURE_007: 主题系统完善

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### 1. 概述

将所有硬编码的 chalk 颜色改为使用主题系统,实现全局主题切换。

### 2. 影响范围

**需要修改的文件**:
- `packages/repl/src/interactive/commands.ts` - 使用主题颜色
- `packages/repl/src/interactive/project-commands.ts` - 使用主题颜色
- `packages/repl/src/ui/themes/index.ts` - 添加主题切换
- `packages/repl/src/ui/contexts/` - 添加 ThemeContext

### 3. 技术方案

**ThemeContext**:
```typescript
interface ThemeContextValue {
  theme: Theme;
  setTheme: (name: string) => void;

```


### 4. 接口契约

3. **迁移 commands.ts 的 chalk 调用**
4. **迁移 project-commands.ts 的 chalk 调用**
5. **添加 /theme 命令**
6. **主题配置持久化**

### 6. 验收标准

## FEATURE_018: CodeWiki - 项目知识库系统

**Status**: Planned
**Priority**: High
**Category**: New

### 1. 需求概述

在 KodaX 运行过程中持续构建和维护的项目知识库系统。

#### 核心需求

1. **自动积累** - 在代码阅读、文件分析、代码修改过程中自动丰富 wiki
2. **同步更新** - Project wiki 与代码变更保持同步
3. **快速查询** - KodaX 可随时通过 wiki 获取项目/代码细节
4. **渐进披露** - 查询时按需加载，避免占用过多 LLM 上下文

#### 关键特性

- Wiki 以目录形式存储（如 `.kodax/wiki/`）
- 支持多种知识类型：架构概览、模块说明、API 文档、依赖关系等
- 增量更新机制，只在相关代码变更时更新对应 wiki
- 智能查询接口，根据问题 relevance 返回相关片段
- 分层披露：先返回摘要，需要时再深入细节

#### 用户场景

1. **新项目接入**: KodaX 自动分析项目结构，生成初始 wiki
2. **代码理解**: 查询 "这个项目的认证流程是什么？"，返回相关架构文档
3. **上下文增强**: 在修改代码前，自动从 wiki 加载相关模块说明
4. **知识沉淀**: 每次代码变更后，自动更新相关的 wiki 条目

## FEATURE_025: Adaptive Project Intelligence Layer

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. Overview

This feature carries forward the useful half of the Project Harness roadmap without dragging KodaX into an overgrown AutoHarness-style search system. The goal is to deepen project semantics and declarative self-correction while keeping execution deterministic, reviewable, and small.

It should absorb the parts that naturally pair with `CodeWiki`:

- deeper architecture semantic rules
- stronger declarative refiner behavior
- richer evaluator output
- finer repo-specific rule and exception synthesis

It should explicitly not absorb:

- action-trace replay verification
- rollback orchestration
- branch search / Thompson sampling / harness candidate search
- model-generated executable harness code

### 2. Relationship to Other Features

- `FEATURE_018 CodeWiki` remains the long-term semantic knowledge base
- `FEATURE_025` consumes that knowledge to compile better deterministic or declarative project rules
- `FEATURE_019 Session Tree & Rollback System` remains separate because rollback/search orchestration is a later and heavier concern

This separation keeps each feature focused:

- `018` answers "what does the project know?"
- `025` answers "how should the harness use that knowledge?"
- `019` answers "how do we branch or roll back execution history?"

### 3. Scope

#### 3.1 Must-Have Scope

- deeper deterministic architecture invariants based on workspace docs and repo structure
- declarative refiner improvements built on failure codes, critic records, and playbooks
- clearer evaluator output for legality, progress, evidence, and repair state
- finer repo-specific exception and checklist synthesis from project artifacts

#### 3.2 Explicit Non-Goals

- no branch-search system
- no replay verifier
- no automatic rollback engine
- no executable policy generation

### 4. Technical Direction

#### 4.1 Semantic Rule Compilation

Compile stronger repo rules from:

- `AGENTS.md`
- `docs/HLD.md`
- `docs/ADR/**/*.md`
- workspace package structure
- CodeWiki summaries and invariants

Expected output remains declarative and bounded:

- architecture constraints
- allowed exceptions
- completion checklists
- repair playbooks

#### 4.2 Declarative Refiner

Move from "one retry with static playbooks" toward:

- failure-pattern to playbook mapping that can evolve in config
- repo-specific repair actions
- bounded rule refinement with schema validation

#### 4.3 Evaluator Output

Expose a clearer project-facing evaluator model for:

- legality
- progress confidence
- evidence completeness
- quality delta
- retry or stall risk

This should improve explainability, not create a complicated hidden score machine.

### 5. Implementation Plan

1. Extend semantic rule compilation with a few high-value architecture checks
2. Let CodeWiki summaries feed deterministic/declarative rule generation
3. Upgrade repair-playbook resolution to support repo-specific refinement safely
4. Improve evaluator output formatting and inspection paths
5. Keep all new outputs schema-bound, testable, and diff-friendly

### 6. Acceptance Criteria

- CodeWiki-derived project knowledge can strengthen harness rules without introducing executable policy code
- repo-specific semantic constraints remain deterministic or declarative
- repair behavior becomes more adaptive while staying inspectable in config and logs
- evaluator output becomes more useful for long-running automation decisions
- the feature does not introduce rollback/search/replay subsystems

## FEATURE_028: First-Class Search Retrieval and Evidence Tooling

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. Overview

KodaX needs native research and exploration tools that are stronger than raw shell usage. This feature adds structured search, retrieval, and evidence handling for both external information and large codebases.

### 2. Scope

- built-in web search
- built-in web fetch
- built-in code search beyond basic grep flows
- evidence-carrying result blocks with attribution
- search outputs reusable by CodeWiki, Project Harness, and planning flows

### 3. Design Direction

- treat search as a first-class tool family with structured outputs
- preserve attribution and evidence summaries
- improve routing for research-heavy, verification-heavy, and review-heavy tasks
- keep permission and cost controls explicit

### 4. Acceptance Criteria

- [ ] KodaX can perform structured web and code search without shell fallbacks
- [ ] search results preserve attribution and evidence summaries
- [ ] search outputs can feed project planning and verification
- [ ] search-heavy workflows become materially easier than today

## FEATURE_038: Official Sandbox Extension

**Status**: Planned
**Priority**: High
**Category**: New

### 1. Positioning

`@kodax/sandbox` 不是“权限系统 2.0”。

它应该是一个官方可选扩展包：加载以后，不去改 KodaX 的 agent loop，也不接管 model/provider，而是把少数高风险工具替换成带边界的版本。

加载前:

- KodaX 使用默认内建工具

加载后:

- `bash` -> sandboxed bash
- `write` / `edit` / `undo` -> guarded file mutation tools
- `read` / `glob` / `grep` -> bounded read/search tools
- 新增 `/sandbox` 命令与状态展示

因此它的本质是“安全版工具包”，而不是“每次执行前站在旁边审批一遍”。

### 2. Relationship to FEATURE_034

FEATURE_038 是 FEATURE_034 的第一个旗舰级官方扩展用例。

- `FEATURE_034` 负责 extension runtime、tool override、diagnostics、lifecycle/disposable 管理
- `FEATURE_038` 负责 sandbox 配置、策略解析、provider 选择、工具替换与状态命令
- `KodaXEvents` 在 038 中仍然只是 host/UI adapter，可用于日志、状态和可观测性
- 038 不应把 034 再拉回“全局 middleware / universal interception chain”的方向

换句话说，core 提供插槽，sandbox 提供策略。

### 3. Why event interception is not enough

当前代码与 034 redesign summary 已经说明了两点:

1. `on(event)` 在 v1 中应当是 observe-only by default
2. v1 不引入 universal middleware chain

同时，当前 `packages/coding/src/agent.ts` 里工具执行前仍是“若存在 `beforeToolExecute` 则询问 REPL，否则直接 `executeTool()`”的模型；`packages/coding/src/tools/bash.ts` 仍是直接 `spawn(command, [], { shell: true, cwd: process.cwd() })`。

这意味着 sandbox 的真正 enforcement 不能主要依赖 `tool:before` 一类的事件拦截，而应该发生在“工具本身被替换之后”。

v1 设计原则:

- 事件用于初始化、状态刷新、日志与清理
- 边界执行发生在 overridden tool handler 中
- REPL permission mode 继续承担审批/UX 语义，但不承担 sandbox 的主要安全语义

### 4. Scope and non-goals

#### 4.1 In scope for v0.8.0

- `@kodax/sandbox` 作为官方 optional package
- global/project 配置合并
- `off / workspace / dev` 三个 profile
- `required / best-effort` enforcement 语义
- `srt` 进程隔离 provider
- 覆盖 `bash/read/write/edit/glob/grep/undo`
- `/sandbox` 命令与状态展示

#### 4.2 Explicit non-goals for v1

- 不是新的 approval system
- 不替换 KodaX 的 provider 架构
- 不引入 universal middleware
- 不在 v1 追求 VM / whole-session 级隔离
- 不在 v1 将 sandbox 抽象成 capability provider
- 不承诺 Windows 在 v1 具备与 macOS/Linux 等价的 process isolation

### 5. Runtime boundary

#### 5.1 `@kodax/coding` core 负责什么

只负责 runtime 插槽，不负责安全策略:

- agent loop
- local tool registry
- extension loader
- command registration
- capability provider registration
- typed event bus

#### 5.2 `@kodax/sandbox` 负责什么

负责把默认工具替换成带边界的版本:

- 读取并合并 sandbox 配置
- 把配置解析成 profile / policy
- 选择 process isolation provider
- 覆盖高风险工具
- 暴露 `/sandbox` 状态与诊断

#### 5.3 `@kodax/repl` 负责什么

只负责 UI/UX:

- 加载扩展
- 显示状态
- 暴露 `/sandbox`
- 保留 plan / accept-edits / auto-in-project 等审批与交互语义

### 6. Required slots from FEATURE_034

为了让 038 以“官方扩展”而不是“侵入 core patch”的方式实现，034 需要提供下面几个插槽。

#### 6.1 Atomic local tool definitions

当前 `packages/coding/src/tools/registry.ts` 仍然分离了:

- handler registry
- `KODAX_TOOLS`
- `KODAX_TOOL_REQUIRED_PARAMS`

038 依赖 034 先把这三份 truth source 收敛成一份原子化定义。

#### 6.2 Tool override semantics

`registerTool()` 需要支持同名覆盖。

推荐语义:

- last registration wins
- runtime diagnostics/REPL warning 明确显示“哪个 tool 被哪个 extension 覆盖”

这正是 sandbox 这种“替换内建工具”场景所需要的能力。

#### 6.3 Built-in tool factories

内建工具应从直接函数导出，收敛为可复用工厂:

```typescript
createReadTool(cwd, opts?)
createWriteTool(cwd, opts?)
createEditTool(cwd, opts?)
createBashTool(cwd, opts?)
createGlobTool(cwd, opts?)
createGrepTool(cwd, opts?)
createUndoTool(cwd, opts?)
```

038 通过这些 factory 复用:

- schema
- tool description
- output/result shape
- renderer compatibility

只替换执行后端与 guard。

#### 6.4 Required params from schema

`executeTool()` 不应再依赖独立的 `KODAX_TOOL_REQUIRED_PARAMS` 表，而应直接从 schema 推导 required params。

否则 overridden tools 仍会被旧表约束，影响扩展透明覆盖。

### 8. Config and policy model

#### 8.1 User config

```typescript
type SandboxUserConfig = {
  enabled?: boolean;
  profile?: 'off' | 'workspace' | 'dev';
  provider?: 'auto' | 'srt' | 'docker' | 'none';
  enforcement?: 'required' | 'best-effort';

  filesystem?: {
    denyRead?: string[];
    allowWrite?: string[];
    denyWrite?: string[];
  };

  network?: {
    allowedDomains?: string[];
    deniedDomains?: string[];
  };

  env?: {
    allow?: string[];
    tempHome?: boolean;
  };
};
```

#### 8.2 Resolved policy

```typescript
type ResolvedSandboxPolicy = {
  enabled: boolean;
  profile: 'off' | 'workspace' | 'dev';
  provider: 'srt' | 'docker' | 'none';
  enforcement: 'required' | 'best-effort';
  workspaceRoot: string;

  filesystem: {
    denyRead: string[];
    allowWrite: string[];
    denyWrite: string[];
  };

  network: {
    allowedDomains: string[];
    deniedDomains: string[];
  };

  env: {
    allow: string[];
    tempHome: boolean;
  };

  degraded: boolean;
  reason?: string;
};
```

#### 8.3 Config files and precedence

推荐沿用 Pi 官方 sandbox extension 的配置模式:

- `~/.kodax/sandbox.json`
- `<cwd>/.kodax/sandbox.json`

优先级:

- project config 覆盖 user config
- `workspaceRoot` 优先取 git root，缺失时退回当前 cwd

#### 8.4 Recommended default profiles

`off`

- 不启用 sandbox

`workspace`

- 只允许写 `workspaceRoot` 和系统临时目录
- 禁止读 `~/.ssh`, `~/.aws`, `~/.gnupg`, `~/.kodax`
- 禁止写 `.env`, `.env.*`, `*.pem`, `*.key`, `.kodax/**`
- 默认关闭网络

`dev`

- 继承 `workspace`
- 增加保守的开发常用域名 allowlist
- 例如 npm registry、PyPI、GitHub 文档/源码等

推荐默认 profile 是 `workspace`，而不是 `dev`，避免用户默认“以为安全但实际已开放常见外网”。

### 9. Process isolation provider model

```typescript
interface ProcessIsolationProvider {
  name: 'srt' | 'docker' | 'none';

  detect(): Promise<{ supported: boolean; reason?: string }>;

  init?(policy: ResolvedSandboxPolicy): Promise<void>;

  run(spec: {
    command: string;
    cwd: string;
    timeoutMs: number;
    env: Record<string, string>;
    signal?: AbortSignal;
  }): Promise<{
    exitCode: number | null;
    stdout: string;
    stderr: string;
    blocked?: 'filesystem' | 'network' | 'provider';
  }>;

  dispose?(): Promise<void>;
}
```

#### 9.1 Provider selection

`provider: "auto"` 的推荐逻辑:

1. macOS / Linux: 优先 `srt`
2. phase 2: 若未来提供 Docker provider，可在无 SRT 时尝试 Docker
3. 都不可用: 进入 `none`

#### 9.2 Enforcement semantics

`required`

- 无可用 process isolation provider 时 fail closed
- `/sandbox` 状态与错误信息必须明确写出 provider 不可用原因

`best-effort`

- 允许退回 guard-only host execution
- 状态必须明确标记 `degraded = true`

设计原则是: 不要让用户“以为自己在 sandbox 里”。

#### 9.3 Windows reality in v1

SRT 目前并不适合作为 Windows v1 的正式 provider。

因此在 Windows 上:

- `required` -> 直接报 provider unavailable
- `best-effort` -> 允许显式降级为 host-side guards only

这不是理想态，但比隐式伪安全更诚实。

### 10. Tool override strategy

#### 10.1 `bash`: process sandbox first-class

这是最高优先级。

执行模型:

- `profile = off` 时，走原始 built-in bash
- provider = `srt` 时，走 `SrtProvider.run()`
- provider 不可用且 `enforcement = required` 时，直接返回 `SandboxUnavailable`
- provider 不可用但 `best-effort` 时，才允许 guarded host bash

原因很直接: 当前 `packages/coding/src/tools/bash.ts` 是最危险的攻击面。

#### 10.2 `write` / `edit` / `undo`: path-based mutation guard

这些工具不应通过 shell 间接实现，而应在真正写入前做宿主侧裁决:

- 目标路径是否位于 `allowWrite`
- 是否命中 `denyWrite`
- 是否命中敏感文件模式

即使未来 `bash` 已经跑在 SRT 中，这层 host-side guard 仍有价值，用于弥补 provider / OS 差异。

#### 10.3 `read` / `glob` / `grep`: bounded read/search

这三类工具主要防越界读取与大范围扫描:

- `read` 不能读取 `denyRead`
- `glob` / `grep` 只能在允许根路径下搜索
- repo 外的大范围搜索直接拒绝

#### 10.4 Why not only wrap `bash`

如果 KodaX 决定把 permission 留在 REPL/UI 层，而不把 sandbox 做进 core，那么官方 `@kodax/sandbox` 就必须把文件类工具一起接住；否则 `write/edit` 仍然可以绕开 `bash` 直接修改宿主文件，整体安全叙事不完整。

### 11. Guards

#### 11.1 `PathGuard`

职责:

- 规范化路径
- 解析相对路径与符号链接后的落点
- 判断读/写目标是否越界
- 匹配 deny glob / protected patterns

建议默认强保护:

- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `.kodax/**`
- 用户凭证目录

#### 11.2 `EnvGuard`

职责:

- 过滤传入 sandboxed process 的环境变量
- 可选启用临时 HOME
- 限制可能泄漏 host secrets 的环境变量

推荐默认:

- allowlist 少量必要变量
- `tempHome = true`

### 12. `/sandbox` command and status model

`/sandbox` 至少应展示:

- 是否启用
- 当前 profile
- 当前 provider
- enforcement 模式
- 是否 degraded
- 可写根路径
- denyRead / denyWrite 关键规则
- 网络策略摘要

示例输出:

```text
Sandbox: enabled
Profile: workspace
Provider: srt
Enforcement: required
Workspace root: /repo
Degraded: no
Write roots: /repo, /tmp
Deny read: ~/.ssh, ~/.aws, ~/.gnupg, ~/.kodax
Deny write: .env, .env.*, *.pem, *.key, .kodax/**
Network: disabled
```

### 14. Implementation phases

#### Phase 0: 034 foundation

- atomic local tool definitions
- required params inferred from schema
- same-name tool override
- built-in tool factories
- extension loading in `packages/coding`

#### Phase 1: v0.8.0 minimum viable sandbox

- `packages/sandbox` package skeleton
- config merge + profile resolution
- `PathGuard` / `EnvGuard`
- `none` provider
- `/sandbox` command
- overridden file tools

#### Phase 2: v0.8.0 process isolation completion

- `srt` provider
- sandboxed `bash`
- provider diagnostics
- degraded/fail-closed behavior
- integration tests on macOS/Linux

#### Phase 3: post-v0.8.0 follow-ups

- Docker provider
- richer REPL status bar integration
- optional read-only resources such as `sandbox://status`
- improved Windows story once a credible provider exists

### 15. Acceptance criteria

- [ ] `@kodax/sandbox` can be loaded as an optional official extension without changing the core agent loop
- [ ] `bash` runs through a process isolation provider when sandbox is enabled and provider support exists
- [ ] `write/edit/undo` are blocked outside allowed roots and on denied patterns
- [ ] `read/glob/grep` cannot exfiltrate or enumerate sensitive paths outside policy
- [ ] provider unavailable + `required` fails closed instead of silently falling back
- [ ] provider unavailable + `best-effort` surfaces an explicit degraded state
- [ ] `/sandbox` exposes the effective policy and runtime status clearly
