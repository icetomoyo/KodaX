## 中文导读

`v0.8.0` 的定位不是增加更多用户可见模式，而是把几条已经方向正确、但还分散的能力，收束成一套真正统一的 substrate：

- MCP as a runtime-owned capability broker
- AMA as a dual-profile harness family: tactical by default, managed when load-bearing
- prompt architecture as a thin, dynamic assembly system
- first-class search / fetch / semantic retrieval
- AMA convergence and invisible internal parallelism
- durable compaction, artifact recall, and optional sandbox safety

如果只看一句话，可以把它理解成：

“让 KodaX 在不增加产品表面负担的前提下，变得更会取证、更会并行、更会稳定思考。”

## FEATURE_007: Theme System Consolidation

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### Overview

KodaX still contains scattered, hard-coded visual choices. `FEATURE_007` consolidates theme behavior so that terminal surfaces can render consistently and future multi-surface work has a clean styling boundary.

### Goals

1. Replace hard-coded terminal colors with theme tokens.
2. Make theme choice explicit and persistent.
3. Keep theme state compatible with future terminal UX work.
4. Avoid tying theme behavior to legacy project-mode-only UI paths.

### Scope

- terminal theme tokens
- theme persistence
- shared theme usage across REPL commands and status surfaces
- cleanup of legacy hard-coded color paths

### Detailed design notes retained

Useful implementation direction from the previous draft:

- introduce a shared `ThemeContext`
- migrate command/status output from scattered `chalk.*` calls to theme tokens
- support `/theme`, `/theme <name>`, and `/theme list`
- persist the selected theme so terminal behavior stays stable across sessions

#### Historical draft snapshot

The older draft also recorded concrete implementation touchpoints that should remain visible:

- `packages/repl/src/interactive/commands.ts`
- `packages/repl/src/interactive/project-commands.ts`
- `packages/repl/src/ui/themes/index.ts`
- `packages/repl/src/ui/contexts/`

Useful retained implementation order:

1. create `ThemeContext`
2. inject theme access into command rendering paths
3. migrate hard-coded `chalk` calls
4. add `/theme`
5. persist theme selection

### Acceptance criteria

- [ ] terminal output uses theme tokens instead of scattered colors
- [ ] theme choice can be persisted cleanly
- [ ] theme logic is independent of old project-mode assumptions

## FEATURE_058: Transcript Native Scrollback Dump

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### Overview

`FEATURE_058` adds a Claude-style transcript escape hatch that dumps the current transcript view back into the terminal's
native scrollback.

This is explicitly a transcript reader convenience, not a second transcript product surface. The goal is to let users
temporarily leave renderer-owned viewport semantics when they want native terminal review, copy, or paging behavior.

### Goals

1. Add a transcript-only shortcut to dump the current transcript into native terminal scrollback.
2. Keep the primary transcript experience renderer-owned and fullscreen-first on capable hosts.
3. Avoid replaying stale managed frames back into terminal history during the dump flow.
4. Preserve KodaX's existing banner/status branding while borrowing Claude's native-scrollback escape hatch.

### Scope

- transcript-only dump shortcut
- native scrollback render path for dumped transcript output
- host-safe handoff between fullscreen transcript mode and terminal scrollback
- transcript footer/help updates for the dump shortcut

### Non-goals

- do not add transcript `$VISUAL/$EDITOR` launching in this feature
- do not replace the fullscreen transcript browser with main-screen as the default path
- do not revive transcript item-palette semantics as the primary interaction model

### Acceptance criteria

- [ ] transcript mode exposes a dedicated dump-to-scrollback shortcut
- [ ] dumping transcript content does not replay stale fullscreen UI frames into terminal history
- [ ] transcript can return cleanly to fullscreen/live mode after the dump path is used
- [ ] KodaX does not add Claude's transcript `v` editor shortcut as part of this feature

## FEATURE_059: Managed Task Structured Protocol V2

**Status**: Planned
**Priority**: High
**Category**: Internal

### Overview

`FEATURE_059` upgrades managed-task worker protocols from text-coupled fenced-block parsing toward a structured,
dual-track protocol model.

The immediate motivation is reliability:

- evaluator verdict drift currently causes `missing kodax-task-verdict`
- malformed or missing worker blocks can force fail-closed retries
- raw worker payloads can leak into UI, artifacts, and memory in ways that create poor failure UX

The long-term goal is to separate:

- user-visible prose
- machine-consumable protocol state

so that managed rounds can continue safely even when visible wording changes.

### Goals

1. Keep current managed-task flows compatible while reducing dependence on brittle tail-block parsing.
2. Introduce a structured protocol direction that can eventually replace fenced-block parsing for all managed workers.
3. Make missing or malformed worker protocol output degrade gracefully instead of causing protocol dead-ends or process-level instability.
4. Preserve raw worker output for debugging artifacts without forcing large raw payloads into user-visible UI or final result objects.

### Scope

- evaluator verdict hardening
- scout / planner / handoff parser hardening
- managed-worker protocol artifact separation
- design groundwork for dual-track `visibleText + protocolPayload`

### Non-goals

- do not redesign the AMA harness model in this feature
- do not change transcript/product UX beyond protocol-failure surfacing
- do not replace managed workers with a totally different orchestration engine

### Problem statement

Current managed workers still rely on visible text that ends with fenced protocol blocks:

- `kodax-task-scout`
- `kodax-task-contract`
- `kodax-task-handoff`
- `kodax-task-verdict`

This coupling creates three recurring problems:

1. minor wording or formatting drift can invalidate otherwise useful worker output
2. protocol retry / fail-closed paths can retain or replay too much raw text
3. the system must parse critical state out of long freeform responses instead of reading a dedicated protocol channel

### Phase plan

#### Phase 1: Parser hardening

- accept the last matching fenced block instead of only a strict end-of-message block
- support JSON payloads for managed worker blocks where safe
- normalize common field aliases and punctuation drift
- keep fail-closed behavior, but make protocol parsing substantially more tolerant

#### Phase 2: Safer degraded continuation

- when evaluator protocol is missing after retries, surface a degraded verification result instead of a cold protocol-only failure
- preserve raw evaluator output in artifacts while returning compact visible failure text
- avoid process-level instability even when raw worker payloads are large

#### Phase 3: Structured protocol v2

- introduce a dual-track result model:
  - `visibleText`
  - `protocolPayload`
- treat fenced blocks as compatibility fallback, not the primary source of truth
- let evaluator act as a verdict producer/classifier rather than requiring long visible prose plus a trailing verdict block

### Acceptance criteria

- [ ] evaluator verdict drift no longer commonly fails on punctuation, aliases, JSON, or trailing chatter
- [ ] scout / planner / handoff parsing are upgraded to the same tolerance level as evaluator
- [ ] malformed managed worker output cannot crash the session or flood the visible UI with raw protocol payloads
- [ ] managed protocol failures retain raw artifacts for debugging without forcing the final result object to carry large raw text
- [ ] the design path to `visibleText + protocolPayload` is documented and implementable in later phases

## FEATURE_044: Durable Compression Anchors and Artifact Recall

**Status**: Completed
**Close-out note (2026-04-03)**: Implemented with durable compaction anchors, artifact ledgers, compact memory seeds, and continuation-oriented recall helpers.
**Priority**: High
**Category**: Enhancement

### Overview

当前的 session compaction 已经具备“滚动摘要 + pruned tool result”这条基础路径，但它离“长期稳定记忆”还有一段距离：

- compaction 结果主要还是一条普通 system summary message，而不是 first-class lineage anchor
- artifact / file / decision state 过度依赖摘要文本，缺少独立 recall index
- 被裁剪的 `tool_result` 现在主要保留 tool 名称和极少量 preview，能知道“做过什么工具”，却不一定能低成本知道“操作过哪个目标”

与此同时，managed task 已经开始使用更成熟的 compact memory seed 和 artifact-first continuation 形态。这说明 KodaX 并不缺长期记忆的方向，缺的是把这套能力推广到普通 session compaction。

`FEATURE_044` 的目标，是把自动压缩从“节省 token 的摘要技巧”升级成“可回放、可检查、可评测的长期记忆层”，并明确把“tool 裁剪时保留最小目标信息”纳入正式设计边界。

### Why `v0.8.0`

把这个 feature 放在 `v0.8.0` 更合适，因为它本质上属于 knowledge / retrieval / evidence 这一层的基础能力补强：

- 它先解决“历史信息怎么以低成本保真地留下来”
- 再支撑后续 harness 和 multimodal 流程能否高质量 continuation
- 它比 `v0.9.0` 的 harness Phase 2 更偏基础设施，而不是上层 orchestration 能力

### Goals

1. 把 compaction 变成 first-class memory anchor，而不是普通 summary message。
2. 把 artifact / file / command target state 从摘要中拆出来，形成独立 ledger。
3. 在极小 token 预算下，让 pruned `tool_result` 至少保留 `action + target` 级别的信息。
4. 把 managed-task 的 compact memory seed 思路下沉到普通 session。
5. 用 continuation 质量和 recall 准确率评估压缩，而不是只看压缩率。

### Scope

- session compaction anchors
- artifact ledger / recall index
- target-aware pruned tool previews
- unified compact memory seed for ordinary sessions
- continuation-oriented compaction prompt hardening
- probe-based compression evaluation harness

### Non-goals

- 永久保存完整 shell transcript
- 把 compaction 变成新的 repo-intelligence 产品表面
- 为 `v0.8.0` 新增复杂 UI 面板

### Current gaps

#### 1. Compaction 还不是真正的 lineage node

当前 hot path 会把压缩结果写回消息数组，但不会稳定地产生一个带引用关系的 compaction entry。这样一来，长历史回溯时更像是在“重放一段被改写过的消息历史”，而不是“从一个记忆锚点继续”。

#### 2. Artifact state 过于依赖摘要文本

现在能保留的 file-op 信息仍偏窄，而且更多是 summary prompt 的副产物。它不适合承担：

- 精确 recall 哪些文件被读过 / 改过
- 在 continuation 时快速提醒模型“哪些目标值得优先回看”
- 在 branch / rewind / checkpoint 之间共享关键 artifact 线索

#### 3. Pruned tool preview 保留的信息还不够稳

对 continuation 来说，“调用过 `bash`”远不如“改过 `packages/agent/src/compaction/compaction.ts`”有价值。当前 preview 对某些工具仍只留下 tool 名称、命令首 token 或 basename，这会在长会话里丢掉最重要的“目标定位信息”。

#### 4. 普通 session 和 managed task 的 memory strategy 还未统一

managed task 已经在使用 compact memory seed、artifact files、continuation note；普通 session 仍主要依赖 summary message。两套机制没有统一，就很难把同一套长期记忆原则沉淀成共用基础设施。

#### 5. 缺少 probe-based evaluation

如果没有稳定评测，压缩机制就只能靠体感调 prompt。真正需要衡量的是：

- recall 对不对
- artifact 线索会不会丢
- continuation 会不会重复调查
- 关键决策会不会漂移

### Deliverables

#### 044-A Compaction anchors as first-class lineage entries

- 每次 compaction 生成明确的 anchor entry，而不是仅写回压缩后的 message array
- anchor 至少包含：
  - `summary`
  - `firstKeptEntryId`
  - `tokensBefore`
  - `tokensAfter`
  - `artifactLedgerRef` 或等价引用
  - `details` / `reason`
- session rewind / continue 优先从“最近 anchor + 原始 tail”重建，而不是只依赖摘要消息

#### 044-B Session artifact ledger

- 为普通 session 增加独立 artifact ledger
- ledger 记录的不是全文，而是 continuation 最需要的“可提醒线索”
- 目标类型包括：
  - file read / file modified / file created / file deleted
  - search scope / glob scope / command scope
  - check result / failure / decision / tombstone

建议的数据形态：

```ts
interface SessionArtifactLedgerEntry {
  id: string;
  kind:
    | 'file_read'
    | 'file_modified'
    | 'file_created'
    | 'file_deleted'
    | 'path_scope'
    | 'search_scope'
    | 'command_scope'
    | 'check_result'
    | 'decision'
    | 'tombstone';
  sourceTool?: string;
  action?: string;
  target: string;
  displayTarget?: string;
  summary?: string;
  sessionEntryId?: string;
  timestamp: string;
  metadata?: Record<string, unknown>;
}
```

原则是：summary 负责 continuation 叙事，ledger 负责 recall 定位。

#### 044-C Target-aware pruned tool previews

这是本次讨论新增并锁定的设计点。

当前被裁剪的 `tool_result` placeholder 不应该只告诉模型“这里曾经有一个 tool call”，而应该在极小 token 预算下，保留最重要的目标信息。设计原则如下：

1. 优先保留 `action + target`，而不是完整命令。
2. 对 continuation 来说，target 通常比 tool 名称更重要。
3. preview 是低成本保真，不是 artifact ledger 的替代品。
4. 目标信息必须优先于噪声参数、长 payload、heredoc、完整 diff。

#### Target extraction strategy

第一层：结构化字段优先

- path-like fields:
  - `path`
  - `file`
  - `files[0]`
  - `outputPath`
  - `cwd`
  - `scenePath`
  - `scriptPath`
  - `resourcePath`
- search-like fields:
  - `glob`
  - `pattern`
  - `query`
- command-like fields:
  - `command`
  - `command_line`
  - `CommandLine`

第二层：按预算压缩显示

1. 文件或目录显示“最短可区分后缀”，而不总是只留 basename。
2. 多目标场景保留主目标并附加数量，如 `(+3)`。
3. shell / bash 场景优先抽取“动词 + 目标”，而不是完整命令行。
4. 丢弃低价值 flags、长参数、内联 payload、heredoc 和大段 stdout/stderr。
5. preview 设定固定 token budget；一旦冲突，优先保留目标后缀，而不是命令前缀。

#### Preview examples

- `[Pruned: read packages/agent/src/compaction/compaction.ts]`
- `[Pruned: edit InkREPL.tsx]`
- `[Pruned: bash git diff packages/coding/src]`
- `[Pruned: bash pytest tests/auth (+3)]`
- `[Pruned: grep task-engine.ts "memoryStrategy"]`

#### Design conclusion

这项设计的结论可以明确写成一句话：

> 在自动压缩的裁剪阶段，`tool_result` 必须尽量保留“操作动作 + 目标对象”，因为这比单独保留工具名更能支撑低成本 continuation。

#### 044-D Unified compact memory seed

普通 session 需要获得与 managed task 相同原则的 compact memory seed，但形式更轻：

- current objective
- constraints / assumptions
- progress so far
- blockers / open loops
- next recommended steps
- key decisions
- important artifact targets
- explicit tombstones for failed paths

这样 continuation 恢复时，就不需要重新从整段 summary 中猜哪些信息最重要。

#### 044-E Probe-based compression evaluation harness

压缩优化必须配套评测。建议引入 4 组 probes：

1. Recall probe
   - 压缩后还能否回答历史事实
2. Artifact probe
   - 压缩后还能否定位关键文件、目录、命令目标、失败实验
3. Continuation probe
   - 压缩后继续做任务时，是否减少重复读取和重复探索
4. Decision probe
   - 压缩后是否仍记得已确认的方案、约束和放弃路线

核心指标不是“压了多少 token”，而是：

- continuation 质量是否稳定
- artifact fidelity 是否提升
- 是否减少重复调研
- 是否减少决策漂移

#### 044-F Compaction prompt contract and continuation discipline

`FEATURE_044` 不应该靠“把 compaction prompt 写得越来越长”来获得质量。KodaX 需要的是一个更窄、更稳定、面向 continuation 的 specialist prompt：

- summarize for continuation, not for documentation polish
- emit anchor fields, ledger hints, tombstones, and next-step cues
- prefer stable fielded output over verbose freeform prose
- keep compaction behavior separate from general system-prompt inflation

这也意味着 `044` 需要和 `FEATURE_050` 对齐：

- compaction prompt 应该有可快照、可 diff、可回归的 contract
- recall / artifact / continuation probe 应该覆盖 prompt drift，而不只是代码 drift

### Relationship to other features

- `FEATURE_011` 提供基础 compaction 能力
- `FEATURE_019` 提供 session tree / lineage / rewind 语义
- `FEATURE_022` 提供 managed-task / harness 路径，可复用 compact memory 设计思想
- `FEATURE_028` 与 artifact / evidence tooling 的边界相邻，但 `044` 更聚焦于 session memory
- `FEATURE_043` 关注 harness 成熟化，`044` 则补齐长期记忆与回溯质量
- `FEATURE_050` 把 compaction prompt contract 变成 snapshot / eval 的正式对象

### Acceptance criteria

- [ ] session compaction 会生成 first-class anchor entry，而不是仅改写 message array
- [ ] 普通 session 存在独立 artifact ledger，可记录关键文件、目录、命令目标、检查结果与决策
- [ ] pruned `tool_result` placeholder 默认保留 `action + target`，而不是只保留 tool 名称
- [ ] 当 basename 不足以区分目标时，preview 会自动提升为更长的唯一路径后缀
- [ ] 普通 session continuation 可以使用 compact memory seed + artifact ledger，而不只依赖摘要文本
- [ ] 长历史 continuation 中，模型对关键目标的重复读取和重复摸索显著下降
- [ ] compression evaluation harness 至少支持 Recall / Artifact / Continuation / Decision 四类 probe
- [ ] “tool 裁剪时最小保留目标信息”被写成正式设计约束，而不是仅作为 prompt 经验
- [ ] artifact ledger 与 pruned preview 之间职责清晰：前者负责 recall，后者负责低成本 continuity
- [ ] 该能力不会退化成永久保留完整命令或完整输出的高 token 方案
- [ ] compaction 使用 continuation-oriented specialist prompt，而不是继续膨胀成巨型通用总结 prompt
- [ ] compaction prompt drift 可以通过 snapshot / regression probes 被发现

## FEATURE_035: MCP Capability Provider

**Status**: Completed
**Close-out note (2026-04-03)**: Implemented as a runtime-owned MCP capability provider with catalog/runtime/provider layers, capability truth injection, and MCP-backed tool surfaces.
**Priority**: High
**Category**: New

### Overview

After design review, `FEATURE_035` belongs in `v0.8.0` because it is no longer a REPL bridge experiment. It becomes the canonical MCP path for KodaX's capability substrate, evidence transport, and prompt-time capability truth.

KodaX should be able to access MCP-backed capabilities through the shared capability runtime without pretending that every remote tool must become a built-in core tool or a user-visible subsystem.

### Goals

1. Expose MCP ecosystems through the shared capability runtime.
2. Preserve `tool / resource / prompt` as first-class capability families with typed describe semantics.
3. Keep MCP integration compatible with provenance, diagnostics, structured results, and permission-mode truth.
4. Make MCP usable by Scout, retrieval, execution, and evaluator flows without product-surface bloat.
5. Surface active MCP capability truth through dynamic prompt sections rather than by mutating the base prompt.

### Design direction

Recommended model:

- MCP capability catalog managed by the runtime
- stable capability-provider abstraction
- explicit provenance, trust, and failure reporting
- capability results shaped for evidence transport
- prompt-time injection of current MCP truth only when the provider is actually available

### Relationship to other features

- `FEATURE_034` provides runtime loading, capability registration, diagnostics, and structured transport
- `FEATURE_028` defines the retrieval / evidence boundary that MCP results should feed
- `FEATURE_029` provides provider-capability truth and disclosure principles
- `FEATURE_048` defines how active capability truth enters prompt assembly
- `FEATURE_049` provides adjacent first-class retrieval surfaces that can complement MCP resources

### Acceptance criteria

- [ ] MCP capabilities can be loaded through the shared capability runtime
- [ ] capability provenance, trust hints, and failures are visible in runtime diagnostics
- [ ] MCP does not require mutating core built-in tool definitions to exist
- [ ] capability results can participate in task evidence flows with `structuredContent` / artifacts when available
- [ ] active MCP capability truth can be surfaced through dynamic prompt sections without bloating the base prompt

### Detailed design record

The detailed design record below remains the normative implementation reference for the `v0.8.0` rollout. Earlier bridge-oriented ideas are kept only as migration context.

> 2026-03 redesign note: the normative design for FEATURE_035 is the "Redesign Summary" below. The older bridge-oriented draft is retained temporarily as historical context and should not be treated as the target implementation.

### Redesign Summary

#### Positioning

FEATURE_035 is no longer scoped as a REPL-only "MCP Bridge Runtime". It becomes the first large **Capability Provider** built on top of FEATURE_034.

Its job is:

- to represent MCP as a provider inside the capability runtime
- to support lazy discovery and typed description of tools, resources, and prompts
- to execute MCP capabilities without flattening everything into a single `mcp_call + tools.md` pattern

#### Why this redesign is needed

- A REPL-startup bridge would miss print mode, auto-continue, and other non-REPL paths that already call `runKodaX()`.
- MCP has three distinct capability families: tools, resources, and prompts. Treating them all as one markdown index throws away too much structure.
- MCP supports pagination, `list_changed` notifications, and structured tool output. A static `tools.md` file cannot be the main protocol surface.
- "Zero SDK dependency" is not user value. Correctness, spec alignment, and future support for resources/prompts matter more.

#### Runtime ownership

The MCP runtime should be session/runtime owned, not `InkREPL.tsx` owned.

```text
runKodaXSession()
  -> load extensions
  -> init capability registry
  -> init MCP capability provider(s) if enabled
  -> build model tool list
  -> run agent
```

REPL can still expose `/mcp status` and `/mcp refresh`, but it should not be the only place MCP exists.

#### Relationship to upgraded FEATURE_034

FEATURE_034 now owns the programmable runtime substrate: extension loading, hot reload, structured capability result transport, runtime control APIs, and explicit hook contracts. FEATURE_035 remains the first large capability provider built on top of that substrate.

This means 035 should not re-introduce its own:

- extension loader
- session/state/model control side channel
- provider-specific event system
- REPL-only ownership model

#### v1 model-facing surface

For a low-risk rollout, v1 may expose a small MCP-prefixed tool family to the model:

```typescript
mcp_search({
  query: string,
  server?: string,
  kind?: 'tool' | 'resource' | 'prompt',
  limit?: number,
})

mcp_describe({ id: string })

mcp_call({
  id: string,
  args: Record<string, unknown>,
})

mcp_read_resource({ id: string })
```

This is still provider-neutral internally because all real behavior lives behind the capability provider abstraction from 034.

#### Capability and catalog model

```typescript
interface McpCatalogItem {
  id: string; // mcp:github:tool:create_issue
  serverId: string;
  kind: 'tool' | 'resource' | 'prompt';
  name: string;
  title?: string;
  summary: string;
  tags?: string[];
  risk?: 'read' | 'write' | 'network' | 'exec';
  annotations?: Record<string, unknown>;
  cachedAt: string;
}

interface McpCapabilityDescriptor extends McpCatalogItem {
  inputSchema?: JsonSchema;
  outputSchema?: JsonSchema;
  promptArgsSchema?: JsonSchema;
  uri?: string;
  mimeType?: string;
}
```

JSON should be the source of truth:

```text
~/.kodax/mcp/
```

### 3. 技术方案

#### 3.1 零 SDK 的 MCP 客户端

```typescript
// packages/repl/src/bridge/mcp-client.ts
class MCPServerConnection {
  private proc: ChildProcess;
  private nextId = 1;
  private pending = new Map<number, { resolve, reject }>();
  private buffer = '';

  // 发送 JSON-RPC request，等待 response
  async send(method: string, params: Record<string, unknown>): Promise<unknown>;
  // 发送 JSON-RPC notification（无返回值）
  async notify(method: string, params: Record<string, unknown>): Promise<void>;
  // 启动连接：spawn → initialize → notifications/initialized → tools/list
  async connect(command: string, args: string[], env?: Record<string, string>): Promise<ToolDef[]>;
  // 调用工具
  async callTool(name: string, args: Record<string, unknown>): Promise<string>;
  // 优雅关闭：stdin.end → SIGTERM(2s) → SIGKILL(2s)
  async shutdown(): Promise<void>;
  // 状态
  get name(): string;
  get connected(): boolean;
}
```

关键实现点：
- stdout 用 `\n` 分割 JSON-RPC messages，处理跨 chunk 的 buffer
- stderr 直接 pipe 到 process.stderr（MCP server 的日志输出）
- 超时保护：request 30s 超时

## FEATURE_046: AMA Handoff Integrity and Final-Answer Convergence

**Status**: Completed
**Close-out note (2026-04-03)**: Implemented for the current read-only AMA tactical substrate: child results fail closed, parent reduction is ledger/artifact-first, and final answers converge through parent-owned verdicts.
**Priority**: High
**Category**: Enhancement

### Overview

The main problem with KodaX AMA is not that it lacks more roles. It is that worker-to-worker transfer is still too summary-driven, long-form review and investigation can lose evidence during handoff, and the terminal worker can still sound like commentary about another worker instead of a converged final answer.

`FEATURE_046` keeps the post-reset AMA shape intact:

- `H1` remains lightweight checked-direct
- `H2` remains `Planner -> Generator <-> Evaluator`
- the work is to make evidence flow, handoff payloads, and final-answer contracts trustworthy

### REPL posture / UI boundary

`FEATURE_046` strengthens evidence flow and parent-owned final answers, but it does not expand the REPL surface.

- user-visible execution semantics remain `SA` / `AMA`
- convergence may feed existing transcript and summary strips
- no visible worker topology, teammate panel, or task cockpit is introduced
- `FEATURE_055` later hardens the same frozen REPL shell without adding new multi-agent surface

### Goals

1. Make worker handoff artifact-backed instead of summary-only.
2. Separate user-facing status summaries from downstream worker payloads.
3. Make terminal-worker output converge into a single final answer.
4. Preserve inspectable non-terminal outputs for review and debugging.
5. Introduce explicit role-local prompt contracts without adding new user-visible ceremony.

### Scope

- dependency handoff payload upgrade
- display summary vs downstream payload separation
- final-answer convergence contract
- non-terminal transcript persistence
- artifact-backed evaluator input for review / investigation flows
- same-role summary vs full evidence responsibility split
- role-local prompts for `Scout / Planner / Generator / Evaluator`

### Non-goals

- adding a new harness level
- reviving visible team graphs or role topology
- defaulting to more rounds or more negotiation ceremony

### Design direction

#### Handoff contract

handoff should explicitly distinguish:

- `display summary`: short status text for the user
- `dependency artifact`: the real downstream payload for the next worker
- `role memory summary`: low-cost continuation seed for the same role across rounds

These should no longer be mixed together.

#### Long-form upstream output

For review, investigation, and planning flows:

- summaries can stay short
- downstream payloads should reference artifacts, structured findings, or full bundles
- evaluators should not rely on clipped summaries to make verdicts

#### Final-answer ownership

For review and investigation:

- the terminal worker should output the merged final answer
- it should not mainly say "I evaluated the generator output"
- when the evaluator changes the conclusion, it should be able to point back to evidence

#### Role-local prompt contracts

KodaX should keep the global prompt thinner and push responsibility into bounded role-local contracts:

- `Scout`: evidence gaps, capability triage, route-quality signals
- `Planner`: dependency decomposition, mutation boundary, verification plan
- `Generator`: artifact-first execution and bounded write responsibility
- `Evaluator`: verification, contradiction detection, and converged final-answer synthesis

These contracts should reinforce runtime responsibilities, not invent new product roles.

### Deliverables

#### 046-A Handoff payload schema split

Introduce an explicit split between:

- UI-facing status summary
- downstream dependency payload
- same-role continuation summary

The schema should be strong enough that downstream workers no longer need to guess whether a field is intended for human display or machine consumption.

#### 046-B Artifact-backed worker outputs

Non-terminal workers should emit outputs that are stable enough to be consumed later:

- structured findings for review / investigation
- referenced artifact paths for long-form work
- role-specific machine-readable sections where the downstream worker expects them

#### 046-C Final-answer convergence layer

Terminal-worker output should become a synthesis stage rather than commentary about prior workers:

- merge upstream findings
- resolve contradictions
- produce one user-facing answer
- retain evidence references for auditability

#### 046-D Role-local prompt modules

Prompt contracts for `Scout / Planner / Generator / Evaluator` should be represented as separate bounded modules rather than more text appended to the global system prompt.

#### 046-E Regression and inspection path

Review and investigation flows should gain a small regression set that checks:

- summary-only handoff does not silently return
- evaluator verdicts stay evidence-backed
- final answer remains converged when upstream output changes shape

## FEATURE_047: Invisible Adaptive Parallelism and Evidence-Driven Fan-Out

**Status**: Completed
**Close-out note (2026-04-03)**: Implemented with centralized branch lifecycle truth, review/investigation/lookup tactical slices, and deterministic winner-cancel limited to safe read-only investigation fan-out.
**Priority**: High
**Category**: Enhancement

### Overview

KodaX already has orchestration plumbing, and it has already decided that `--team` is not the future product story. The next step is not to make `--team` prettier. It is to turn parallelism into an internal optimization that is:

- almost invisible to the user
- selected by the runtime
- triggered only when the expected gain is real
- cancelable, reversible, and evidence-aware

### Goals

1. Let orchestration run truly parallel worker batches when the scheduler judges them worthwhile.
2. Use parallelism primarily for evidence acquisition and validation quality.
3. Keep `SA / AMA` user semantics unchanged and do not reintroduce team mode.
4. Support speculative branches with cancellation when one path wins early.
5. Merge parallel outputs back into artifact-backed verdicts rather than lossy summaries.

### Scope

- selected orchestration batches with `maxParallel > 1`
- dependency-aware worker fan-out
- evidence-shard / validation-shard / hypothesis-shard patterns
- speculative sub-branches with cancellation
- artifact merge and verdict reduction
- bounded prompt/context bundles for internal workers
- trace visibility without new user-facing topology UI

### Non-goals

- reviving `--team` as a formal product mode
- asking the user to configure worker graphs manually
- default swarm-like execution for simple tasks
- using parallelism as an excuse to inflate prompt ceremony

### Design direction

#### Good parallel targets

Prioritize:

- changed-scope / diff / module evidence sharding
- test / lint / verification sharding
- quick competition between independent hypotheses
- evaluator spot-check shards

#### Bad parallel targets

Avoid by default:

- strongly sequential edit pipelines
- tasks where the user clearly wants tight step-by-step control
- low-value "multiple roles commenting on each other"

#### Scheduler policy

The scheduler should at least consider:

- task family
- mutation surface
- dependency DAG
- evidence gap
- current budget and predicted gain
- provider capability profile

#### Prompt and context discipline

Invisible parallelism should stay cheap:

- speculative workers receive bounded evidence bundles, not the whole transcript
- worker prompts stay role- and task-local
- losing branches can be cancelled without forcing a new user-visible mode

### Deliverables

Add a real internal scheduler that decides whether a candidate fan-out is worthwhile based on:

- dependency structure
- risk and mutation surface
- expected evidence gain
- budget and provider constraints

### REPL posture / UI boundary

- fan-out shows up only through existing summary surfaces such as `AmaWorkStrip`, `BackgroundTaskBar`, and transcript live labels
- no worker graph, teammate panel, task cockpit, or revived team mode is added
- the user still experiences one main authority rather than a visible worker ensemble
- `FEATURE_055` only hardens these existing summary paths; it does not reopen the REPL shell

The scheduler should consume admissible fan-out intents from `FEATURE_052` rather than re-deciding AMA profile boundaries on its own.

#### 047-B Bounded worker-context bundles

Parallel workers should receive task-local bundles rather than a full clone of the main-thread transcript:

- relevant dependency artifacts
- targeted evidence gaps
- role-local prompt section
- cancellation metadata

#### 047-C Speculative branch lifecycle

Support:

- launch of competing branches
- early winner selection
- cancellation of losing branches
- clean merge back into the verdict pipeline

#### 047-D Parallel verdict reduction

Parallel results should converge through artifact-backed merge/reduction rather than by concatenating summaries.

#### 047-E Parallel quality evaluation

Measure whether parallelism is helping:

- does it reduce turnaround time
- does it improve evidence coverage
- does it preserve correctness
- does it avoid prompt and context blow-up

- the REPL continues to expose only `SA` and `AMA`

`v0.8.0` is already where KodaX is tightening runtime substrate, hidden parallelism, prompt boundaries, and evidence flow. This feature is the architecture layer that explains how those parts fit together:

- `046` improves handoff and final-answer convergence

Without this boundary spec, the roadmap risks continuing to mix harness levels, execution tactics, and user-mode language.

### Goals

1. Keep the user-visible execution model minimal: `SA` or `AMA`.
2. Reframe AMA as a harness family with two internal profiles: tactical by default, managed only when load-bearing.
3. Make `Child Fan-Out` an AMA tactic rather than a new mode or a new harness level.
4. Preserve the value of Anthropic-style long-running harness design without forcing heavy ceremony onto normal tasks.
5. Absorb ClaudeCode's "main agent + tactics" strengths without reviving visible team mode or teammate topology.

### Scope

- AMA profile definitions
- controller-owned profile selection and escalation policy
- child-fan-out placement within AMA
- main-authority and reduction contract
- compatibility path from today's `H0 / H1 / H2` language
- routing, policy, and regression expectations for the new framing

### Non-goals

- adding a third user-visible execution mode
- reviving `--team` or visible teammate graphs
- allowing `SA` to silently become multi-agent
- making every AMA run use `Planner -> Generator -> Evaluator`
- deleting current `H0 / H1 / H2` implementation envelopes before migration is ready

### Design thesis

Anthropic's harness article argues for heavier structure when long tasks lose coherence or when self-evaluation becomes unreliable. ClaudeCode's shipped product shows a complementary lesson: the default path can stay much lighter if one main agent can call the right tactics at the right time.

KodaX should combine those lessons rather than choosing one side:

- keep a true harness for long-running, high-assurance work
- keep the default AMA experience closer to "one main agent, smart tactics, hidden fan-out"
- escalate only when the heavier harness becomes load-bearing

### User-facing contract

The user should still only learn two primary modes:

- `SA`: strict single-agent execution
- `AMA`: adaptive harnessed execution

`AMA Tactical` and `AMA Managed` should **not** be exposed as normal top-level mode choices. They are internal AMA profiles selected by the controller.

This is important for KodaX's "minimal and intelligent" philosophy:

- fewer user concepts
- richer internal behavior
- clearer authority model

#### User control without mode explosion

Users can still influence the controller indirectly:

- choose `SA` when they want strict single-agent behavior
- choose `AMA` for adaptive execution
- express intent such as `double-check`, `review carefully`, `treat this as a long-running build`, or `strong verification`
- later, advanced debugging or evaluation flags may pin a profile explicitly, but that should remain an expert override rather than the default UX

#### Should users switch Tactical vs Managed themselves?

Default answer: no.

`AMA Tactical` and `AMA Managed` should normally be selected by the AMA controller, not by the user as everyday top-level switches.

Reason:

- exposing them as normal modes would leak implementation details into product UX
- most users care about `strict single-agent` vs `adaptive execution`, not about internal harness profile names
- KodaX should prefer fewer top-level concepts and smarter internal routing

Allowed exceptions:

- expert override flags for debugging, evals, or internal rollout
- future project-level policy knobs for testing or benchmarking

But those should remain advanced controls, not the default interaction model.

### AMA controller lifecycle

The AMA controller should own profile selection, tactic selection, and escalation.

Recommended lifecycle:

1. choose `SA` vs `AMA`
2. if `AMA`, start in `AMA Tactical` by default
3. select tactics inside tactical execution:
   - child fan-out
   - verifier pass
   - bounded planning pass
4. escalate to `AMA Managed` only if heavier harness structure becomes load-bearing
5. keep one main authority responsible for the final user answer

#### Controller vs scheduler split

`FEATURE_052` should own the controller-level decision boundary, while `FEATURE_047` should own the execution-level scheduler strategy.

That split should be explicit:

- `052` decides:
  - `SA` vs `AMA`
  - `AMA Tactical` vs `AMA Managed`
  - whether child fan-out is allowed for the current task shape
  - which tactic family is appropriate: direct, verifier, bounded planning, child fan-out
- `047` decides:
  - whether a specific fan-out is worth launching
  - how to shard work
  - how many branches to run
  - how to cancel losing branches
  - how to score cost vs gain

So `052` needs a scheduling boundary, but not the full parallel scheduler policy.

#### Default-start rule

When a task routes into AMA, the controller should prefer `AMA Tactical` unless there is strong up-front evidence that tactical execution is insufficient.

This mirrors the strongest part of ClaudeCode's default behavior:

- main agent first
- subagents and verification as tactics
- heavier coordinator/harness only when the task genuinely needs it

#### Direct-start managed rule

The controller may start directly in `AMA Managed` only when the initial task shape already makes the need obvious, for example:

- explicit multi-sprint build request
- clear need for contract and independent acceptance before implementation
- broad, long-running mutation work spanning many loosely related chunks
- user explicitly asks for a managed long-running build flow

#### Escalation rule

Escalation should be one-way by default within a run:

- `AMA Tactical -> AMA Managed`

Repeated bouncing between profiles should be treated as a design smell. If managed structure is justified, the task should stay managed.

#### What Tactical should not become

`AMA Tactical` should not gradually accrete all of the ceremony of managed execution. In particular, tactical execution should avoid by default:

- repeated contract negotiation
- default multi-round revise loops
- persistent role graphs
- visible worker topology

### ClaudeCode-inspired subagent substrate

ClaudeCode's most valuable lesson is not its visible team surface. It is the quality of its subagent substrate:

- child agents are first-class runtime objects
- child execution is isolated
- tool and MCP scope can be constrained per child
- child results return to the parent instead of becoming peer narrators
- child completion can trigger context/cache eviction rather than long-lived transcript inheritance

KodaX should absorb that substrate while keeping a more minimal user-facing story.

#### Why this belongs in `FEATURE_052`

This substrate belongs in `052` because it defines the architecture contract for AMA itself.

- `052` owns what a child agent is, where it may exist, and how it relates to the parent authority
- `047` owns when parallel fan-out is worthwhile and how scheduling/speculation/cancellation work

Without the substrate contract in `052`, `047` would only describe scheduler behavior without a clear runtime model underneath it.

#### Subagent runtime contract

Minimum runtime properties:

1. child session isolation
2. bounded context bundle instead of blind full-transcript cloning
3. scoped tool / capability / permission surface
4. artifact-first child result contract
5. parent-owned reduction and final answer
6. child completion eviction / compaction discipline

#### 1. Child session isolation

Each child agent should run in its own bounded session context:

- separate message history
- separate tool state where relevant
- separate child identity for tracing and debugging
- explicit lifecycle state: spawned, running, completed, cancelled, failed

This is one of the most important ClaudeCode lessons, and it is what makes parallel child execution safe instead of becoming shared-state chaos.

#### 2. Bounded context bundle

Children should not automatically inherit the whole main transcript.

Instead, the parent should assemble a task-local bundle such as:

- objective
- local scope
- relevant evidence artifacts
- constraints and non-goals
- expected output schema

This keeps child work cheap and prevents child execution from re-importing all main-thread noise.

#### 3. Scoped capability policy

Each child should receive only the capabilities needed for its job.

Examples:

- read-only evidence child
- validator child
- retrieval child
- disjoint-write child, but only under managed execution

This is also where permission and MCP scoping should attach.

#### 4. Artifact-first child results

Child agents should return structured results first, not long freeform transcripts.

Recommended shape:

- `claim`
- `scope`
- `evidence_refs`
- `confidence`
- `contradictions`
- `recommended_disposition`
- `artifacts`

This keeps parent reduction cheap and makes child output durable for later audit.

#### 5. Parent-owned reduction

The normal topology should remain:

```text
Parent -> Child -> Parent
```

More precisely:

- parent orchestrates
- children do bounded sub-work
- parent reduces and synthesizes
- parent owns the user-facing final answer

Reducer-child variants may exist later for very large fan-out, but they should be a specialized optimization rather than the default architecture.

#### 6. Child completion eviction discipline

When a child finishes, KodaX should preserve durable outputs but avoid promoting the entire child transcript into the main context.

Keep:

- structured result card
- referenced artifacts
- status / telemetry

Do not keep by default:

- full child reasoning transcript
- full tool noise
- entire child conversation replay in the parent context

This is the part of ClaudeCode's subagent design that most directly supports lower context pressure and better long-session memory.

### Internal model

#### `SA`

`SA` remains strict:

- one agent
- no child fan-out
- no hidden verifier worker
- no managed contract or role graph

This keeps `SA` maximally predictable and easy to reason about.

#### `AMA Tactical`

`AMA Tactical` is the default AMA profile.

Its shape is:

- one main authority agent
- optional child-fan-out
- optional verifier or checker tactic
- optional bounded planning tactic
- parent-owned synthesis and final answer

It should feel closest to ClaudeCode's default shipped behavior:

- main agent leads
- tactics are pulled in when useful
- heavy harness ceremony is avoided unless it is actually needed

Typical fits:

- review
- investigation
- read-only analysis
- docs-only work
- medium-complexity implementation that still fits one coherent main thread
- tasks where parallel validation reduces latency and context pressure

Recommended tactical tactics:

- child fan-out for review / investigation / evidence gathering
- lightweight verifier pass
- bounded planning pass when obligations or constraints need quick structuring
- tactical retrieval bursts without entering managed rounds

#### `AMA Managed`

`AMA Managed` is the heavier AMA profile.

It exists for cases where tactical execution is no longer enough to keep quality stable:

- explicit contract is needed
- independent acceptance needs to be durable
- work spans many coherent chunks or sprints
- repeated revise loops are expected
- mutation scope is broad enough that "main agent + tactics" is no longer reliable

This is the profile closest to the Anthropic harness article and to ClaudeCode's heavier coordinator path.

Its shape may include:

- explicit planning / contract negotiation
- generator / evaluator separation
- artifact-backed handoff between rounds
- managed acceptance and revise loops
- child fan-out as an auxiliary tactic inside research, implementation support, or verification

Managed-specific responsibilities:

- explicit contract or sprint boundary
- stronger acceptance ownership
- managed artifact handoff between rounds
- deliberate multi-pass refine only when evaluator or contract state requires it

### Child Fan-Out boundary

`Child Fan-Out` is not a mode. It is not a harness level. It is an execution tactic that exists **inside AMA**.

Its core job is:

- parallelize independent sub-work
- reduce wall-clock time
- reduce main-agent context pressure
- keep local tool noise out of the parent context

#### Where it belongs

- not in `SA`
- available in `AMA Tactical`
- available in `AMA Managed`

#### How it behaves

Default topology:

```text
Parent -> parallel child agents -> Parent reduction
```

The parent remains the user-facing authority. Child agents are bounded helpers, not peer narrators.

#### Default reduction rule

The normal case should be parent-side reduction:

- child agents return structured findings and artifacts
- the parent synthesizes the result
- raw child transcripts are not promoted to the main thread by default

Reducer-child behavior can exist later as an optimization for very large fan-out, but it should not be the default model.

#### Typical child-fan-out patterns

High-value patterns inspired by ClaudeCode:

- finding-level false-positive filtering
- changed-module evidence sharding
- hypothesis competition
- validator spot checks
- retrieval or repo-exploration shards

Patterns to avoid by default:

- multiple children rewriting the same mutation surface
- multiple roles generating commentary about each other
- fan-out where every child still needs the full transcript

### Tactical vs Managed boundary

The controller should treat the two AMA profiles as two different answers to two different questions.

`AMA Tactical` answers:

- can one main agent still hold the task coherently?
- can assurance be achieved with tactics rather than a full managed loop?

`AMA Managed` answers:

- has assurance become load-bearing enough that explicit contracts, independent acceptance, and structured rounds are now worth their cost?

#### Route into `AMA Tactical`

Prefer `AMA Tactical` when:

- the task still has one clear main thread
- child-fan-out can isolate local evidence work
- optional verification is enough
- the cost of full managed ceremony would outweigh the gain

#### Upgrade to `AMA Managed`

Escalate to `AMA Managed` only when one or more of these become true:

- the task requires explicit contract or sprint definition
- independent acceptance cannot stay lightweight
- repeated revise loops are likely
- mutation work spans many loosely coupled chunks
- the controller predicts coherence loss if one main thread keeps stretching
- artifact-backed handoff is now more valuable than tactical continuity

#### Ceiling rule for read-only and docs-only work

The current design rule should remain:

- read-only and docs-only work stays below the heavy managed path

In the new language, that means:

- read-only / docs-only can benefit from `AMA Tactical` plus child-fan-out and verification tactics
- they should not enter `AMA Managed` by default

### Tactic matrix

To make implementation clearer, the following matrix should guide controller choices:

| Tactic | `SA` | `AMA Tactical` | `AMA Managed` |
|---|---|---|---|
| direct main-agent execution | yes | yes | yes |
| child fan-out | no | yes | yes |
| lightweight verifier | no | yes | yes |
| bounded planning pass | no | yes | yes |
| explicit contract negotiation | no | no | yes |
| managed multi-round revise | no | no | yes |
| disjoint-write child shards | no | no | yes |

### Controller decision heuristics

The controller should consider at least these signals:

- task type: review, investigation, docs, implementation, long-running build
- mutation surface size and coupling
- expected benefit of child fan-out
- expected need for explicit acceptance
- expected number of coherent chunks or sprints
- provider / capability limits
- budget and context pressure

These heuristics are intentionally controller-level only. They determine:

- which AMA profile should own the run
- whether fan-out is even admissible
- whether the runtime should stay tactical or escalate to managed

They do **not** replace the lower-level scheduler policy from `FEATURE_047`.

#### Heuristic outcome examples

- review of a broad diff:
  - route to `AMA Tactical`
  - use finding validators and evidence shards
- medium implementation with one coherent main thread:
  - route to `AMA Tactical`
  - maybe add verifier pass
- feature build expected to span many chunks with acceptance gates:
  - route or escalate to `AMA Managed`
  - use contract, generator/evaluator separation, and managed handoff

### Controller API draft

To make implementation boundaries concrete, `052` should define the controller-side contract that sits above the `047` scheduler.

#### Recommended controller input

```ts
interface AmaControllerInput {
  userMode: 'sa' | 'ama';
  taskType:
    | 'conversation'
    | 'review'
    | 'investigation'
    | 'docs'
    | 'implementation'
    | 'long_running_build';
  mutationClass: 'none' | 'narrow' | 'medium' | 'broad';
  coherenceRisk: 'low' | 'medium' | 'high';
  acceptanceNeed: 'best_effort' | 'verified' | 'managed';
  expectedChunks: number;
  providerConstraints: {
    supportsParallelism: boolean;
    supportsLongRunningWork: boolean;
  };
  evidenceSignals: {
    benefitsFromFanout: boolean;
    needsExplicitContract: boolean;
    needsIndependentAcceptance: boolean;
  };
}
```

#### Recommended controller output

```ts
interface AmaControllerDecision {
  selectedMode: 'sa' | 'ama';
  amaProfile?: 'tactical' | 'managed';
  allowedTactics: Array<
    'direct'
    | 'child_fanout'
    | 'lightweight_verifier'
    | 'bounded_planning'
    | 'managed_contract'
    | 'managed_revise'
  >;
  fanoutPolicy?:
    | { admissible: false; reason: string }
    | {
        admissible: true;
        fanoutClass:
          | 'review_validation'
          | 'evidence_shard'
          | 'hypothesis_competition'
          | 'retrieval_shard'
          | 'managed_support';
        writePolicy: 'read_only' | 'disjoint_write';
      };
  escalationPolicy?: {
    canEscalateToManaged: boolean;
    triggers: string[];
  };
}
```

The important design point is that the controller does **not** choose branch count or detailed shard layout. It only decides profile, admissibility, and tactic envelope.

- internal helpers do not become co-equal narrators
- hidden parallelism stays hidden unless lightweight trace visibility is useful

This keeps KodaX aligned with the best part of ClaudeCode's default experience:

- richer internal execution
- simple user-facing authority

- a thinner and more stable base prompt
- dynamic sections that only describe truths that are actually active
- local specialist prompts for bounded tools and subsystems

1. Replace string-concatenation prompt building with sectionized assembly.

- prompt section registry
- effective prompt builder with explicit precedence

## FEATURE_061: Scout-First AMA Architecture Simplification

**Status**: Planned
**Priority**: High
**Category**: Refactor

### Overview

`FEATURE_061` simplifies the AMA managed task architecture from a multi-layer routing pipeline into a Scout-centric model where Scout is both the decision-maker and the first worker.

The current AMA pipeline has accumulated significant complexity:

- 4-5 pre-Scout decision layers (Intent Gate → LLM routing → Review Floor → Guardrails → Bypass check)
- Scout as a pure router that doesn't do actual work
- context breaks between every role transition (Scout → Generator → Evaluator each start from scratch)
- subagent parallelism locked behind 3 hardcoded Tactical Flow variants
- H0/H1/H2 naming with a state machine for promotion/demotion/guardrails

This violates KodaX's core philosophy: **极致轻量化 — minimalist and intelligent**.

### Motivation

Anthropic's [harness design article](https://www.anthropic.com/engineering/harness-design-long-running-apps) establishes a key principle:

> "Every component in a harness encodes an assumption about what the model can't do on its own."

With Opus 4.6, many assumptions encoded in the current AMA pipeline no longer hold:

- models can self-route reliably without a separate routing pass before Scout
- models can produce consistent structured output without 6-field cross-validation
- models don't need context resets between roles (the article explicitly found this with Opus 4.6)
- a 3-agent pattern (Planner → Generator → Evaluator) is sufficient for most long-running tasks

### Goals

1. Make Scout the single AMA entry point after Intent Gate — no pre-routing LLM call, no review floor, no bypass check.
2. Let Scout both judge complexity and do actual work (H0 tasks are completed by Scout directly).
3. Preserve context across role transitions: Scout upgrades into Generator or Planner rather than handing off to a fresh agent.
4. Give every core role (Scout/Planner/Generator/Evaluator) the ability to spawn subagents for parallel work.
5. Simplify the harness model from named profiles with a state machine to a pure worker-chain composition.
6. Maintain the Generator/Evaluator separation (proven valuable for independent verification).

### Non-goals

- changing user-visible mode semantics (SA / AMA remain the only user choices)
- removing Intent Gate (it's zero-cost regex, it stays)
- removing repo routing signals (Scout still needs them)
- making Evaluator inherit Generator's full context (Evaluator must stay independent)
- reviving visible team mode or worker topology UI

### Target architecture

```
User prompt
    │
    ▼
┌───────────────┐
│  Intent Gate  │  regex/rules, zero cost
│  (sync)       │  SA → runDirectKodaX, done
│               │  AMA → read repo signals, enter Scout
└───────────────┘
    │ AMA
    ▼
╔═══════════════════════════════════════════════════════╗
║  SCOUT (single LLM session, sole AMA entry point)    ║
║                                                       ║
║  Capabilities:                                        ║
║  1. Assess task complexity and harness needs          ║
║  2. H0: complete task directly (may spawn subagents)  ║
║  3. H1/H2: produce handoff directive and upgrade      ║
╚═══════════════════════════════════════════════════════╝
    │
    ├── H0: Scout completes directly → return result
    │
    ├── H1: Scout upgrades to Generator
    │   │
    │   ▼
    │   Generator (inherits Scout context, may spawn subagents)
    │   │
    │   ▼
    │   Evaluator (independent context: task + deliverable only)
    │     ├── accept → output
    │     ├── revise → Generator fixes (1x) → re-evaluate
    │     │     ├── accept → output
    │     │     └── not ok → escalate to H2
    │     └── escalate → escalate to H2
    │
    └── H2: Scout upgrades to Planner
        │
        ▼
        Planner (inherits Scout context, may spawn subagents)
        → plan self-check (1x self-correction opportunity)
        │
        ▼
        Generator (new context: task + plan artifact, may spawn subagents)
        │
        ▼
        Evaluator (independent context: task + plan + deliverable)
          ├── accept → output
          ├── revise → Generator fixes (1x) → re-evaluate
          └── replan → Planner replans (1x) → re-execute
```

### Context passing rules

| From → To | Method | Reason |
|-----------|--------|--------|
| Scout → Generator (H1) | same session continues | Scout already read diff/code, don't lose context |
| Scout → Planner (H2) | same session continues | same reason |
| Planner → Generator | new session + plan artifact | Generator only needs the plan, not Planner's reasoning |
| Generator → Evaluator | new session + task + deliverable | Evaluator needs independent perspective |
| Evaluator → Generator (revise) | resume Generator session + feedback | Generator needs prior context to fix targeted issues |
| Evaluator → Planner (replan) | resume Planner session + feedback | Planner needs prior context to revise plan |

### Subagent model

Every core role (Scout/Planner/Generator/Evaluator) may spawn subagents:

- subagents are assistants that return results to the parent role
- subagents do not make harness decisions (no escalation, no evaluation)
- subagents receive scoped context (only the sub-task, not the full conversation)
- the decision to spawn subagents is made by the role itself, not hardcoded admission rules
- subagents run in parallel via `runOrchestration` and consolidate before the role produces its output

### H1 retry and escalation policy

- Evaluator may `revise` once: Generator gets specific feedback, fixes, resubmits
- If second evaluation still fails: escalate to H2 (add Planner), do not retry further at H1
- Evaluator may `escalate` directly on first pass if the problem clearly needs planning

Rationale: if H1 can't solve it in 2 attempts, the issue is task understanding, not execution quality. Adding a Planner is more productive than retrying.

### H2 retry and replan policy

- Evaluator distinguishes `revise` (execution issue) from `replan` (plan issue)
- `revise` → Generator fixes (1x) → re-evaluate
- `replan` → Planner revises plan (1x) → Generator re-executes → Evaluator re-evaluates
- maximum 1 replan cycle to prevent infinite loops
- Planner includes a self-check step before handing off to Generator (1x self-correction)

Rationale: most plan issues only surface during execution. The self-check catches obvious problems; the replan path catches deeper ones.

### What gets deleted

| Component | Reason |
|-----------|--------|
| `createReasoningPlan` LLM routing call | Scout handles routing |
| `applyCurrentDiffReviewRoutingFloor` | Scout assesses review scope directly |
| `resolveManagedHarnessGuardrail` + guardrail system | folded into Scout's judgment |
| `shouldBypassScoutForManagedH0` | all AMA goes through Scout |
| `runTacticalReviewFlow` / `runTacticalInvestigationFlow` / `runTacticalLookupFlow` | replaced by role-level subagent capability |
| `validateScoutDirectiveConsistency` + 6-field cross-validation | Scout does work directly, no complex structured output to validate |
| H0/H1/H2 naming + promotion/demotion state machine | replaced by worker-chain composition |
| budget zone partitioning + per-role iteration limits | simplified to total cap + round limit |

### What gets preserved

| Component | Reason |
|-----------|--------|
| Intent Gate (regex) | zero cost, filters trivial tasks |
| repo routing signals | Scout needs repo context |
| Generator/Evaluator separation | self-evaluation is unreliable (proven by Anthropic article) |
| artifact-backed handoff (FEATURE_046) | proven value, keeps evidence flow trustworthy |
| `runOrchestration` infrastructure | needed for subagent parallelism |
| `runDirectKodaX` worker execution | core LLM call mechanism |
