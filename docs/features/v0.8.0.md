## FEATURE_007: Theme System Consolidation

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### Overview

KodaX still contains scattered, hard-coded visual choices. `FEATURE_007` consolidates theme behavior so that terminal surfaces can render consistently and future multi-surface work has a clean styling boundary.

### Goals

1. Replace hard-coded terminal colors with theme tokens.
2. Make theme choice explicit and persistent.
3. Keep theme state compatible with future terminal UX work.
4. Avoid tying theme behavior to legacy project-mode-only UI paths.

### Scope

- terminal theme tokens
- theme persistence
- shared theme usage across REPL commands and status surfaces
- cleanup of legacy hard-coded color paths

### Detailed design notes retained

Useful implementation direction from the previous draft:

- introduce a shared `ThemeContext`
- migrate command/status output from scattered `chalk.*` calls to theme tokens
- support `/theme`, `/theme <name>`, and `/theme list`
- persist the selected theme so terminal behavior stays stable across sessions

#### Historical draft snapshot

The older draft also recorded concrete implementation touchpoints that should remain visible:

- `packages/repl/src/interactive/commands.ts`
- `packages/repl/src/interactive/project-commands.ts`
- `packages/repl/src/ui/themes/index.ts`
- `packages/repl/src/ui/contexts/`

Useful retained implementation order:

1. create `ThemeContext`
2. inject theme access into command rendering paths
3. migrate hard-coded `chalk` calls
4. add `/theme`
5. persist theme selection

### Acceptance criteria

- [ ] terminal output uses theme tokens instead of scattered colors
- [ ] theme choice can be persisted cleanly
- [ ] theme logic is independent of old project-mode assumptions

## FEATURE_044: Durable Compression Anchors and Artifact Recall

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

当前的 session compaction 已经具备“滚动摘要 + pruned tool result”这条基础路径，但它离“长期稳定记忆”还有一段距离：

- compaction 结果主要还是一条普通 system summary message，而不是 first-class lineage anchor
- artifact / file / decision state 过度依赖摘要文本，缺少独立 recall index
- 被裁剪的 `tool_result` 现在主要保留 tool 名称和极少量 preview，能知道“做过什么工具”，却不一定能低成本知道“操作过哪个目标”

与此同时，managed task 已经开始使用更成熟的 compact memory seed 和 artifact-first continuation 形态。这说明 KodaX 并不缺长期记忆的方向，缺的是把这套能力推广到普通 session compaction。

`FEATURE_044` 的目标，是把自动压缩从“节省 token 的摘要技巧”升级成“可回放、可检查、可评测的长期记忆层”，并明确把“tool 裁剪时保留最小目标信息”纳入正式设计边界。

### Why `v0.8.0`

把这个 feature 放在 `v0.8.0` 更合适，因为它本质上属于 knowledge / retrieval / evidence 这一层的基础能力补强：

- 它先解决“历史信息怎么以低成本保真地留下来”
- 再支撑后续 harness 和 multimodal 流程能否高质量 continuation
- 它比 `v0.9.0` 的 harness Phase 2 更偏基础设施，而不是上层 orchestration 能力

### Goals

1. 把 compaction 变成 first-class memory anchor，而不是普通 summary message。
2. 把 artifact / file / command target state 从摘要中拆出来，形成独立 ledger。
3. 在极小 token 预算下，让 pruned `tool_result` 至少保留 `action + target` 级别的信息。
4. 把 managed-task 的 compact memory seed 思路下沉到普通 session。
5. 用 continuation 质量和 recall 准确率评估压缩，而不是只看压缩率。

### Scope

- session compaction anchors
- artifact ledger / recall index
- target-aware pruned tool previews
- unified compact memory seed for ordinary sessions
- probe-based compression evaluation harness

### Non-goals

- 永久保存完整 shell transcript
- 把 compaction 变成新的 repo-intelligence 产品表面
- 为 `v0.8.0` 新增复杂 UI 面板

### Current gaps

#### 1. Compaction 还不是真正的 lineage node

当前 hot path 会把压缩结果写回消息数组，但不会稳定地产生一个带引用关系的 compaction entry。这样一来，长历史回溯时更像是在“重放一段被改写过的消息历史”，而不是“从一个记忆锚点继续”。

#### 2. Artifact state 过于依赖摘要文本

现在能保留的 file-op 信息仍偏窄，而且更多是 summary prompt 的副产物。它不适合承担：

- 精确 recall 哪些文件被读过 / 改过
- 在 continuation 时快速提醒模型“哪些目标值得优先回看”
- 在 branch / rewind / checkpoint 之间共享关键 artifact 线索

#### 3. Pruned tool preview 保留的信息还不够稳

对 continuation 来说，“调用过 `bash`”远不如“改过 `packages/agent/src/compaction/compaction.ts`”有价值。当前 preview 对某些工具仍只留下 tool 名称、命令首 token 或 basename，这会在长会话里丢掉最重要的“目标定位信息”。

#### 4. 普通 session 和 managed task 的 memory strategy 还未统一

managed task 已经在使用 compact memory seed、artifact files、continuation note；普通 session 仍主要依赖 summary message。两套机制没有统一，就很难把同一套长期记忆原则沉淀成共用基础设施。

#### 5. 缺少 probe-based evaluation

如果没有稳定评测，压缩机制就只能靠体感调 prompt。真正需要衡量的是：

- recall 对不对
- artifact 线索会不会丢
- continuation 会不会重复调查
- 关键决策会不会漂移

### Deliverables

#### 044-A Compaction anchors as first-class lineage entries

- 每次 compaction 生成明确的 anchor entry，而不是仅写回压缩后的 message array
- anchor 至少包含：
  - `summary`
  - `firstKeptEntryId`
  - `tokensBefore`
  - `tokensAfter`
  - `artifactLedgerRef` 或等价引用
  - `details` / `reason`
- session rewind / continue 优先从“最近 anchor + 原始 tail”重建，而不是只依赖摘要消息

#### 044-B Session artifact ledger

- 为普通 session 增加独立 artifact ledger
- ledger 记录的不是全文，而是 continuation 最需要的“可提醒线索”
- 目标类型包括：
  - file read / file modified / file created / file deleted
  - search scope / glob scope / command scope
  - check result / failure / decision / tombstone

建议的数据形态：

```ts
interface SessionArtifactLedgerEntry {
  id: string;
  kind:
    | 'file_read'
    | 'file_modified'
    | 'file_created'
    | 'file_deleted'
    | 'path_scope'
    | 'search_scope'
    | 'command_scope'
    | 'check_result'
    | 'decision'
    | 'tombstone';
  sourceTool?: string;
  action?: string;
  target: string;
  displayTarget?: string;
  summary?: string;
  sessionEntryId?: string;
  timestamp: string;
  metadata?: Record<string, unknown>;
}
```

原则是：summary 负责 continuation 叙事，ledger 负责 recall 定位。

#### 044-C Target-aware pruned tool previews

这是本次讨论新增并锁定的设计点。

当前被裁剪的 `tool_result` placeholder 不应该只告诉模型“这里曾经有一个 tool call”，而应该在极小 token 预算下，保留最重要的目标信息。设计原则如下：

1. 优先保留 `action + target`，而不是完整命令。
2. 对 continuation 来说，target 通常比 tool 名称更重要。
3. preview 是低成本保真，不是 artifact ledger 的替代品。
4. 目标信息必须优先于噪声参数、长 payload、heredoc、完整 diff。

#### Target extraction strategy

第一层：结构化字段优先

- path-like fields:
  - `path`
  - `file`
  - `files[0]`
  - `outputPath`
  - `cwd`
  - `scenePath`
  - `scriptPath`
  - `resourcePath`
- search-like fields:
  - `glob`
  - `pattern`
  - `query`
- command-like fields:
  - `command`
  - `command_line`
  - `CommandLine`

第二层：按预算压缩显示

1. 文件或目录显示“最短可区分后缀”，而不总是只留 basename。
2. 多目标场景保留主目标并附加数量，如 `(+3)`。
3. shell / bash 场景优先抽取“动词 + 目标”，而不是完整命令行。
4. 丢弃低价值 flags、长参数、内联 payload、heredoc 和大段 stdout/stderr。
5. preview 设定固定 token budget；一旦冲突，优先保留目标后缀，而不是命令前缀。

#### Preview examples

- `[Pruned: read packages/agent/src/compaction/compaction.ts]`
- `[Pruned: edit InkREPL.tsx]`
- `[Pruned: bash git diff packages/coding/src]`
- `[Pruned: bash pytest tests/auth (+3)]`
- `[Pruned: grep task-engine.ts "memoryStrategy"]`

#### Design conclusion

这项设计的结论可以明确写成一句话：

> 在自动压缩的裁剪阶段，`tool_result` 必须尽量保留“操作动作 + 目标对象”，因为这比单独保留工具名更能支撑低成本 continuation。

#### 044-D Unified compact memory seed

普通 session 需要获得与 managed task 相同原则的 compact memory seed，但形式更轻：

- current objective
- constraints / assumptions
- progress so far
- blockers / open loops
- next recommended steps
- key decisions
- important artifact targets
- explicit tombstones for failed paths

这样 continuation 恢复时，就不需要重新从整段 summary 中猜哪些信息最重要。

#### 044-E Probe-based compression evaluation harness

压缩优化必须配套评测。建议引入 4 组 probes：

1. Recall probe
   - 压缩后还能否回答历史事实
2. Artifact probe
   - 压缩后还能否定位关键文件、目录、命令目标、失败实验
3. Continuation probe
   - 压缩后继续做任务时，是否减少重复读取和重复探索
4. Decision probe
   - 压缩后是否仍记得已确认的方案、约束和放弃路线

核心指标不是“压了多少 token”，而是：

- continuation 质量是否稳定
- artifact fidelity 是否提升
- 是否减少重复调研
- 是否减少决策漂移

### Relationship to other features

- `FEATURE_011` 提供基础 compaction 能力
- `FEATURE_019` 提供 session tree / lineage / rewind 语义
- `FEATURE_022` 提供 managed-task / harness 路径，可复用 compact memory 设计思想
- `FEATURE_028` 与 artifact / evidence tooling 的边界相邻，但 `044` 更聚焦于 session memory
- `FEATURE_043` 关注 harness 成熟化，`044` 则补齐长期记忆与回溯质量

### Acceptance criteria

- [ ] session compaction 会生成 first-class anchor entry，而不是仅改写 message array
- [ ] 普通 session 存在独立 artifact ledger，可记录关键文件、目录、命令目标、检查结果与决策
- [ ] pruned `tool_result` placeholder 默认保留 `action + target`，而不是只保留 tool 名称
- [ ] 当 basename 不足以区分目标时，preview 会自动提升为更长的唯一路径后缀
- [ ] 普通 session continuation 可以使用 compact memory seed + artifact ledger，而不只依赖摘要文本
- [ ] 长历史 continuation 中，模型对关键目标的重复读取和重复摸索显著下降
- [ ] compression evaluation harness 至少支持 Recall / Artifact / Continuation / Decision 四类 probe
- [ ] “tool 裁剪时最小保留目标信息”被写成正式设计约束，而不是仅作为 prompt 经验
- [ ] artifact ledger 与 pruned preview 之间职责清晰：前者负责 recall，后者负责低成本 continuity
- [ ] 该能力不会退化成永久保留完整命令或完整输出的高 token 方案

## FEATURE_035: MCP Capability Provider

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

`FEATURE_035` moves into `v0.8.0` and is reframed as a capability-provider feature on top of `FEATURE_034`, not as a one-off product hack.

KodaX should be able to access MCP-backed capabilities through the shared capability runtime without pretending that every remote tool must become a built-in core tool.

### Goals

1. Expose MCP ecosystems through the capability runtime.
2. Keep MCP integration compatible with provenance, diagnostics, and structured results.
3. Avoid dynamic ad hoc mutation of core tool truth.
4. Make MCP usable by both task execution and evidence-gathering flows.

### Design direction

Recommended model:

- MCP capability catalog managed by the runtime
- stable capability-provider abstraction
- explicit provenance and failure reporting
- capability results shaped for evidence transport

### Relationship to `034`

`034` provides:

- runtime loading
- capability registration
- diagnostics
- structured transport

`035` provides:

- MCP-specific connection and catalog behavior
- capability exposure policy
- MCP-specific ergonomics

### Acceptance criteria

- [ ] MCP capabilities can be loaded through the shared capability runtime
- [ ] capability provenance and failures are visible
- [ ] MCP does not require mutating core built-in tool definitions to exist
- [ ] MCP capability results can participate in task evidence flows

### Restored Detailed Design Record

The following detailed draft was restored from the earlier roadmap state where `FEATURE_035` temporarily lived under `v0.7.0`. It remains valuable implementation history even though the feature is now planned for `v0.8.0`.

> 2026-03 redesign note: the normative design for FEATURE_035 is the "Redesign Summary" below. The older bridge-oriented draft is retained temporarily as historical context and should not be treated as the target implementation.

### Redesign Summary

#### Positioning

FEATURE_035 is no longer scoped as a REPL-only "MCP Bridge Runtime". It becomes the first large **Capability Provider** built on top of FEATURE_034.

Its job is:

- to represent MCP as a provider inside the capability runtime
- to support lazy discovery and typed description of tools, resources, and prompts
- to execute MCP capabilities without flattening everything into a single `mcp_call + tools.md` pattern

#### Why this redesign is needed

- A REPL-startup bridge would miss print mode, auto-continue, and other non-REPL paths that already call `runKodaX()`.
- MCP has three distinct capability families: tools, resources, and prompts. Treating them all as one markdown index throws away too much structure.
- MCP supports pagination, `list_changed` notifications, and structured tool output. A static `tools.md` file cannot be the main protocol surface.
- "Zero SDK dependency" is not user value. Correctness, spec alignment, and future support for resources/prompts matter more.

#### Runtime ownership

The MCP runtime should be session/runtime owned, not `InkREPL.tsx` owned.

```text
runKodaXSession()
  -> load extensions
  -> init capability registry
  -> init MCP capability provider(s) if enabled
  -> build model tool list
  -> run agent
```

REPL can still expose `/mcp status` and `/mcp refresh`, but it should not be the only place MCP exists.

#### Relationship to upgraded FEATURE_034

FEATURE_034 now owns the programmable runtime substrate: extension loading, hot reload, structured capability result transport, runtime control APIs, and explicit hook contracts. FEATURE_035 remains the first large capability provider built on top of that substrate.

This means 035 should not re-introduce its own:

- extension loader
- session/state/model control side channel
- provider-specific event system
- REPL-only ownership model

#### v1 model-facing surface

For a low-risk rollout, v1 may expose a small MCP-prefixed tool family to the model:

```typescript
mcp_search({
  query: string,
  server?: string,
  kind?: 'tool' | 'resource' | 'prompt',
  limit?: number,
})

mcp_describe({ id: string })

mcp_call({
  id: string,
  args: Record<string, unknown>,
})

mcp_read_resource({ id: string })
```

This is still provider-neutral internally because all real behavior lives behind the capability provider abstraction from 034.

#### Capability and catalog model

```typescript
interface McpCatalogItem {
  id: string; // mcp:github:tool:create_issue
  serverId: string;
  kind: 'tool' | 'resource' | 'prompt';
  name: string;
  title?: string;
  summary: string;
  tags?: string[];
  risk?: 'read' | 'write' | 'network' | 'exec';
  annotations?: Record<string, unknown>;
  cachedAt: string;
}

interface McpCapabilityDescriptor extends McpCatalogItem {
  inputSchema?: JsonSchema;
  outputSchema?: JsonSchema;
  promptArgsSchema?: JsonSchema;
  uri?: string;
  mimeType?: string;
}
```

JSON should be the source of truth:

```text
~/.kodax/mcp/
  servers.json
  catalog/
    github.index.json
    github.items.json
    fs.index.json
    fs.items.json
  debug/
    github.tools.md
    fs.resources.md
```

Markdown is optional debug/export output, not the discovery protocol.

#### Connection and refresh strategy

Recommended config semantics:

```json
{
  "mcp": {
    "servers": {
      "github": {
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "connect": "lazy",
        "trust": "ask"
      },
      "fs": {
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"],
        "connect": "prewarm",
        "trust": "workspace"
      }
    }
  }
}
```

Semantics:

- `lazy`: connect on first search/describe/call/read
- `prewarm`: connect at session start
- `disabled`: keep config but do not activate

Refresh behavior:

- paginate through `tools/list`, `resources/list`, `prompts/list`
- cache catalog snapshots on disk
- mark provider dirty on `notifications/*/list_changed`
- refresh lazily on next search/describe
- if the server becomes unhealthy, reconnect on next use

#### Result shape and permissions

035 depends on a structured capability result shape in the runtime. MCP tool/resource outputs should not be flattened to a single string too early.

Design requirements:

- preserve `structuredContent` and declared `outputSchema` when available
- treat server annotations as hints, not as a security boundary
- align confirmation/trust behavior with KodaX permission modes
- raise confirmation level for destructive, open-world, network, or exec-like capabilities

#### Suggested file layout

```text
packages/coding/src/capabilities/providers/mcp/
  adapter.ts
  catalog.ts
  provider.ts
  runtime.ts

packages/repl/src/interactive/
  commands.ts   // /mcp status, /mcp refresh
```

#### Acceptance criteria rewrite

- [ ] MCP works in both REPL and non-REPL execution paths
- [ ] tools, resources, and prompts all enter the catalog
- [ ] `mcp_search`, `mcp_describe`, `mcp_call`, and `mcp_read_resource` work against the cached catalog/runtime
- [ ] catalog state is cached in JSON, with markdown only as optional debug output
- [ ] MCP servers reconnect on demand after failure
- [ ] the design does not require injecting every remote MCP tool directly into `KODAX_TOOLS`
- [ ] the design does not depend on "zero SDK dependency" as an architecture constraint

**Status**: Planned
**Priority**: High
**Category**: New

### 1. 概述

为 KodaX 提供 MCP 生态工具访问能力，无需内置 MCP SDK。采用持久化 Runtime 架构，在 REPL 启动时连接所有配置的 MCP server（长驻子进程），通过单一 `mcp_call` 工具暴露给 LLM，工具 schema 通过文件系统渐进披露。

**核心目标**:
1. **零 MCP SDK 依赖** — 纯 Node.js `child_process` + JSON-RPC 实现客户端（~100 行）
2. **持久化 Runtime** — REPL 启动时 spawn MCP server 子进程，session 期间保持活跃
3. **单一 mcp_call 工具** — 固定描述，不随 server 配置变化，不动态修改 KODAX_TOOLS
4. **渐进式披露** — 工具定义写入 `~/.kodax/mcp-bridge/` 用户级目录，LLM 按需 read 发现，智能在 LLM 推理而非启动时分析
5. **配置驱动** — 用户在 `~/.kodax/config.json` 中配置 `mcpServers`
6. **最小化实现** — 总计 ~325 行新增 + ~50 行修改，只做 `/mcp status` 命令

**背景**:
- KodaX 已决定不内置 MCP 支持（FEATURE_027 已删除）
- MCP 生态拥有大量高质量工具服务器（filesystem、github、postgres、brave-search 等）
- 行业标准：Claude Code、Cline、Cursor、Goose、OpenCode 全部采用持久连接模式
- 借鉴 Anthropic "Code Execution with MCP" 范式：工具定义按需加载，98.7% token 减少
- 单一 mcp_call 模式避免动态修改 KODAX_TOOLS 数组和 KODAX_TOOL_REQUIRED_PARAMS 白名单
- MCP 内容全部在用户级目录 `~/.kodax/`，不污染项目目录

### 2. 影响范围

**需要创建的文件**:
- `packages/repl/src/bridge/mcp-client.ts` — JSON-RPC over stdio 客户端（~100 行）
- `packages/repl/src/bridge/mcp-runtime.ts` — MCPRuntime 进程管理 + tools.md 生成（~120 行）
- `packages/repl/src/bridge/index.ts` — 导出（~5 行）

**需要修改的文件**:
- `packages/coding/src/tools/registry.ts` — 注册 mcp_call handler，添加到 KODAX_TOOLS 和 KODAX_TOOL_REQUIRED_PARAMS（~15 行）
- `packages/repl/src/ui/InkREPL.tsx` — 启动时调用 MCPRuntime.start()，退出时 shutdown()（~10 行）
- `packages/repl/src/common/utils.ts` — loadConfig() 返回类型添加 mcpServers 字段（~5 行）
- `packages/repl/src/interactive/commands.ts` — 注册 `/mcp status` 命令（~20 行）

**总计**: ~225 行新增 + ~50 行修改 ≈ **~325 行**

### 3. 技术方案

#### 3.1 零 SDK 的 MCP 客户端

```typescript
// packages/repl/src/bridge/mcp-client.ts
class MCPServerConnection {
  private proc: ChildProcess;
  private nextId = 1;
  private pending = new Map<number, { resolve, reject }>();
  private buffer = '';

  // 发送 JSON-RPC request，等待 response
  async send(method: string, params: Record<string, unknown>): Promise<unknown>;
  // 发送 JSON-RPC notification（无返回值）
  async notify(method: string, params: Record<string, unknown>): Promise<void>;
  // 启动连接：spawn → initialize → notifications/initialized → tools/list
  async connect(command: string, args: string[], env?: Record<string, string>): Promise<ToolDef[]>;
  // 调用工具
  async callTool(name: string, args: Record<string, unknown>): Promise<string>;
  // 优雅关闭：stdin.end → SIGTERM(2s) → SIGKILL(2s)
  async shutdown(): Promise<void>;
  // 状态
  get name(): string;
  get connected(): boolean;
}
```

关键实现点：
- stdout 用 `\n` 分割 JSON-RPC messages，处理跨 chunk 的 buffer
- stderr 直接 pipe 到 process.stderr（MCP server 的日志输出）
- 超时保护：request 30s 超时

## FEATURE_038: Official Sandbox Extension

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

`FEATURE_038` remains the official optional sandbox package built on top of `FEATURE_034`. It is not a replacement for the task engine or the permission model, and it should not pull `034` back into a universal interception architecture.

### Goals

1. Provide safer tool implementations as an optional package.
2. Keep core runtime small while offering stronger execution boundaries.
3. Make sandbox state explicit and inspectable.
4. Support future safe automation without conflating approval UX and runtime isolation.

### Scope

- optional `@kodax/sandbox` package
- safe overrides for high-risk tools
- policy resolution and status reporting
- degraded vs enforced execution semantics

### Non-goals

- replacing the entire permission system
- becoming a universal middleware framework
- owning task routing or completion semantics

### Acceptance criteria

- [ ] sandbox can load as a normal official extension
- [ ] high-risk tools can be replaced by safer implementations
- [ ] effective sandbox policy is inspectable
- [ ] task-engine architecture remains separate from sandbox policy

### Restored Detailed Design Record

The previous sandbox draft contained a lot of implementation detail that still matters. The current summary above is the normative roadmap view; the notes below remain valuable design history.

#### Positioning retained from the previous draft

`@kodax/sandbox` is not "permission system 2.0".

It should remain an optional official extension that:

- does not replace the agent loop
- does not replace the provider system
- replaces a small number of high-risk tools with bounded variants

Loading sandbox should mean:

- `bash` becomes sandboxed or guarded
- `write` / `edit` / `undo` become guarded mutation tools
- `read` / `glob` / `grep` become bounded read/search tools
- `/sandbox` exposes status and diagnostics

#### Relationship to `FEATURE_034`

Retained boundary:

- `034` owns extension runtime, hot reload, runtime control surface, tool override semantics, diagnostics, and explicit hook contracts
- `038` owns sandbox configuration, policy evaluation, process/provider choice, tool replacement, and `/sandbox` UX
- `KodaXEvents` remains a host/UI adapter rather than the main sandbox security boundary

#### Why event interception is not enough

Retained design rule:

- observe-only lifecycle events are useful for status and diagnostics
- actual enforcement should happen in overridden tool handlers and sandbox providers
- sandbox should not pull the runtime back toward a universal middleware chain

#### Scope and non-goals retained

In-scope direction that remains useful:

- official optional package
- global/project config merge
- `off / workspace / dev` profiles
- `required / best-effort` enforcement semantics
- process-isolation provider selection
- overridden high-risk tools
- `/sandbox` status and diagnostics

Non-goals retained:

- not a second approval system
- not a replacement for provider architecture
- not a universal middleware framework
- not whole-session VM isolation in v1
- not Windows parity promises before the provider story is real

#### Runtime boundary retained

Useful retained split:

- `@kodax/coding` provides runtime slots and override capability
- `@kodax/sandbox` resolves policy and provides safer tool implementations
- `@kodax/repl` remains responsible for UI/UX and approval semantics

#### Required slots from `034`

Still important prerequisites:

- atomic local tool definitions
- same-name tool override semantics
- built-in tool factories
- required params derived from schema rather than a separate table
- host-agnostic loading and reload

#### Suggested package layout retained

```text
packages/sandbox/
  src/
    extension.ts
    config.ts
    profiles.ts
    policy.ts
    status.ts
    providers/
    guards/
    tools/
    commands/
```

#### Policy and provider direction retained

Recommended defaults that still seem sound:

- default profile should be closer to `workspace` than `dev`
- `required` vs `best-effort` should be explicit
- provider availability must be surfaced honestly
- users should never be led to believe they are sandboxed when they are not

#### Tool strategy retained

Still useful:

- `bash` is the highest-priority process-isolation target
- file mutation tools need host-side path/pattern guardrails
- read/search tools need bounded traversal and sensitive-path blocking
