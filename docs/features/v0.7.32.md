## 总览

v0.7.32 是整条 Plan B 路线图的**终点**，也是**最危险的一版**。FEATURE_090 让 Agent 能修改自己的 role spec —— reasoning profile、instructions、handoff 图等。

这一档是 KodaX 自构建能力的顶端：档 1 (Skill) / 档 2 (Tool) / 档 3 (Agent) 都是**向外**造东西；档 4 是**向内**改自己。反身性让它拥有其他档位没有的风险面：版本漂移、行为发散、递归反身、安全越权。

因此本 feature 配备的安全机制比前几档都厚 —— 版本化 + 人工审批 + divergence 检测 + 反身稳定保障。

## FEATURE_090: Self-Construction Tier 4 — Agent Self-Modifying Role Spec

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 档 3 (FEATURE_089) 让一个 agent 生成**另一个** agent；档 4 让 agent 修改**自己**
- 典型用例：
  - Long-running agent 在运行中发现自己的 instructions 有盲区，生成 v2 instructions 并自我升级
  - 用户反馈"Evaluator 对 TypeScript 不敏感"，Evaluator 在下一轮自己调 reasoning profile 或 instructions
  - Coding agent 学到新的项目约定，更新自己的 agents/my-coder.json
- 反身修改有三层风险：
  1. **版本漂移**：改乱自己导致功能退化
  2. **递归反身**：改完自己又立刻触发一次修改，无限回路
  3. **安全越权**：假装"我是更高权限"来解锁功能
- 必须有专门的安全机制

### 设计

#### 修改流程

```
some-agent (自己在运行中):
   │
   │  "我想把自己的 reasoning.default 从 balanced 升到 deep"
   ▼
1. 读自己的当前 manifest
   constructed:some-agent@current (e.g., 1.2.0)
2. 生成 diff / new manifest
   constructed:some-agent@1.3.0 (新版本)
3. ConstructionRuntime.stage(new manifest)
4. validate_self_modify:
   - 检查新旧 diff 是否在允许范围（见 "修改边界"）
   - 检查没有触发反身 pattern（新 manifest 不应立即触发另一次修改）
5. sandbox_test_self_modify:
   - 用新 manifest 跑一组 regression test case
   - 对比旧 manifest 的行为，不应在已知 case 上退化
6. ConstructionRuntime.test(handle, regressionSuite)
7. policy gate: 强制 ask-user（永远不 auto-approve for self-modify）
8. ConstructionRuntime.activate(handle)
   - 旧版本不立即 revoke，保留作为 rollback 目标
9. 当前 run 结束，新版本在下一次 Runner.run 生效
   (不在当前 run 生效，避免反身循环)
```

#### 修改边界（允许改什么）

| 字段 | 允许改？ | 约束 |
|------|---------|------|
| `instructions` | ✅ | Guardrail 静态检查无 prompt injection pattern |
| `reasoning` | ✅ | 新 max 不能越过现有 userCeiling |
| `tools` | ✅ | 新加的 tool 必须已 activate 或 builtin |
| `handoffs` | ✅ | handoff target 必须已 activate；不能形成新循环 |
| `guardrails` | ✅ | 只能加不能减（安全 ratchet） |
| `model` | ⚠️ | 需要独立 policy 确认 |
| `name` | ❌ | 改名 = 新 agent，走档 3 |
| `version` | auto | 系统自动 bump |

#### 反身稳定保障

**1. 当前 run 不 swap 自己**  
新 manifest activate 后，**当前 Runner.run 继续用旧 manifest 跑完**。新 manifest 在下一次 Runner.run 生效。理由：避免一轮内"改自己 → 新 self 又改自己"的递归。

**2. Divergence 检测**  
每次 self-modify 前对比新旧 manifest，打分 divergence score：
- instructions 相似度低于阈值 → reject（"change too drastic"）
- reasoning / tools / handoffs 整体重写 → reject
- 每次 self-modify 只允许改若干字段中的一小部分

**3. Modification Budget**  
每个 agent 有 `selfModifyCount` 计数器。默认 N=3 次后必须人工 unlock（避免无限反身）。

**4. Rollback**  
旧版本保留；CLI / API 可 `kodax constructed rollback <agent-name>`。

**5. 版本审计日志**  
所有 self-modify 写入 `.kodax/constructed/_audit.jsonl`，含：时间、diff、reason、结果、rollback 信息。

#### Policy Gate 强制

- self-modify 的 policy gate **永远** ask-user，即使配置了 auto-approve 规则
- 用户看到 diff + reason + sandbox test 结果 + divergence score 之后确认
- 用户可以永久拒绝某个 agent 的 self-modify 能力（Guardrail 级禁用）

### Contract

**Public API（新增）**：
- `scaffold_self_modify` / `validate_self_modify` / `sandbox_test_self_modify` 内部 tools
- CLI: `kodax constructed rollback <name>`、`kodax constructed audit <name>`、`kodax constructed disable-self-modify <name>`
- Guardrail: `SelfModifyGuardrail`（可 per-agent 禁用）

**Internal**：
- ConstructionRuntime 对 kind='agent' + 同名 stage 操作视为 self-modify path
- Runner 检测 "current run is modifying self" 并 defer 到 next run

### Non-Goals

- **不**支持 agent 修改别的 agent 的 role spec（那是档 3 的变体 + 安全问题）
- **不**支持 builtin agent 被 self-modify（只 constructed agent 可以）
- **不**支持跨 session 的 self-modify 协作
- **不**自动学习用户反馈进行 self-modify（agent 必须显式触发）

### Dependencies

- **前置**：
  - FEATURE_089 (v0.7.31) — Agent generation 的 manifest + ConstructionRuntime 就位
  - FEATURE_078 (v0.7.29) — reasoning profile 实装，self-modify 才能改它
  - FEATURE_085 (v0.7.26) — Guardrail runtime（静态检查 instructions、SelfModifyGuardrail）
- **后置**：无（本 feature 是路线图终点）

### Testing

- 单元：divergence score 计算；修改边界 schema 校验
- 单元：modification budget 计数 / reset
- 集成：agent 触发 self-modify → sandbox test → user approval → activate → next run 生效
- 集成：尝试越过修改边界的 case 被 reject（如改 name、max 越 ceiling）
- 集成：反身递归 case 被 defer 机制拦截（同一 run 内不 swap）
- 安全：恶意 self-modify（prompt injection 进 new instructions）被 Guardrail + 用户拒绝拦截
- 审计：所有 self-modify 操作出现在 `_audit.jsonl`

### Risk

本 feature 是整条路线图**风险最高**的一 feature。风险面：

- **反身漂移**：agent 累积多次 self-modify 后行为偏离初始设计
  - 缓解：modification budget + divergence detection + rollback + 审计日志
- **越权**：agent 通过 prompt injection 假装用户确认
  - 缓解：policy gate 强制真实用户输入；Guardrail 静态检查 instructions 无越权关键词
- **安全关闭**：恶意 self-modify 把 Guardrail 移除（"只能加不能减"规则强制）
  - 缓解：修改边界 schema 检查；Guardrail ratchet
- **版本混乱**：同名多版本并存下 Resolver 行为不清晰
  - 缓解：archived 旧版本不解析，只作为 rollback 目标；active 只有一个

### Release criteria

发版前必须满足：

- [ ] 所有 5 条反身稳定保障机制实装 + 测试绿
- [ ] 至少一组端到端 self-modify scenario 跑通（例如 "evaluator 自己升 reasoning 档位"）
- [ ] 至少 3 组 adversarial case（prompt injection / 越权 / 无限反身）被拦截
- [ ] `_audit.jsonl` 格式稳定，能 replay / rollback
- [ ] 文档：docs/FEATURE_090_USER_GUIDE.md 解释 self-modify 的用法与风险

### Timeline

v0.7.32 仅此一 feature。本版是路线图终点，发版前做一次完整 eval regression + 安全审计。
