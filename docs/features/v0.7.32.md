## 总览

v0.7.32 双轨并行，两个 feature 互不依赖：

- **FEATURE_090 — Self-Construction Tier 4: Agent Self-Modifying Role Spec**：整条 Plan B 路线图的终点，也是最危险的一版。让 Agent 能修改自己的 role spec（reasoning profile / instructions / handoff 图）。反身性带来版本漂移、行为发散、递归反身、安全越权四类风险，配套机制比前几档都厚——版本化 + 人工审批 + divergence 检测 + 反身稳定保障。
- **FEATURE_107 — AMA H2 Plan-Execute Boundary Eval**：架构验证 feature，不属于 Plan B。回应 [v0.7.16 FEATURE_061](./v0.7.16.md) 的一个内部不一致——Anthropic harness 文章原则在 H1 落实了（Scout→Generator 同 session），唯独在 H2 的 `Planner → fresh Generator(+ plan artifact)` 没落实。本 feature 用 pre-registered eval 验证这一跳是否真的需要 artifact 切分，并以 H1 为参考点测算 Planner 角色本身的必要性。**eval 跑完才决定是否在代码里固化变体 B**——输了直接删 B 实装。

两条路独立交付，FEATURE_090 风险高走自己的安全审计；FEATURE_107 是数据驱动的架构清算，结果可能反向影响 v0.7.16 设计假设。

## FEATURE_090: Self-Construction Tier 4 — Agent Self-Modifying Role Spec

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 档 3 (FEATURE_089) 让一个 agent 生成**另一个** agent；档 4 让 agent 修改**自己**
- 典型用例：
  - Long-running agent 在运行中发现自己的 instructions 有盲区，生成 v2 instructions 并自我升级
  - 用户反馈"Evaluator 对 TypeScript 不敏感"，Evaluator 在下一轮自己调 reasoning profile 或 instructions
  - Coding agent 学到新的项目约定，更新自己的 agents/my-coder.json
- 反身修改有三层风险：
  1. **版本漂移**：改乱自己导致功能退化
  2. **递归反身**：改完自己又立刻触发一次修改，无限回路
  3. **安全越权**：假装"我是更高权限"来解锁功能
- 必须有专门的安全机制

### 设计

#### 修改流程

```
some-agent (自己在运行中):
   │
   │  "我想把自己的 reasoning.default 从 balanced 升到 deep"
   ▼
1. 读自己的当前 manifest
   constructed:some-agent@current (e.g., 1.2.0)
2. 生成 diff / new manifest
   constructed:some-agent@1.3.0 (新版本)
3. ConstructionRuntime.stage(new manifest)
4. validate_self_modify:
   - 检查新旧 diff 是否在允许范围（见 "修改边界"）
   - 检查没有触发反身 pattern（新 manifest 不应立即触发另一次修改）
5. sandbox_test_self_modify:
   - 用新 manifest 跑一组 regression test case
   - 对比旧 manifest 的行为，不应在已知 case 上退化
6. ConstructionRuntime.test(handle, regressionSuite)
7. policy gate: 强制 ask-user（永远不 auto-approve for self-modify）
8. ConstructionRuntime.activate(handle)
   - 旧版本不立即 revoke，保留作为 rollback 目标
9. 当前 run 结束，新版本在下一次 Runner.run 生效
   (不在当前 run 生效，避免反身循环)
```

#### 修改边界（允许改什么）

| 字段 | 允许改？ | 约束 |
|------|---------|------|
| `instructions` | ✅ | Guardrail 静态检查无 prompt injection pattern |
| `reasoning` | ✅ | 新 max 不能越过现有 userCeiling |
| `tools` | ✅ | 新加的 tool 必须已 activate 或 builtin |
| `handoffs` | ✅ | handoff target 必须已 activate；不能形成新循环 |
| `guardrails` | ✅ | 只能加不能减（安全 ratchet） |
| `model` | ⚠️ | 需要独立 policy 确认 |
| `name` | ❌ | 改名 = 新 agent，走档 3 |
| `version` | auto | 系统自动 bump |

#### 反身稳定保障

**1. 当前 run 不 swap 自己**  
新 manifest activate 后，**当前 Runner.run 继续用旧 manifest 跑完**。新 manifest 在下一次 Runner.run 生效。理由：避免一轮内"改自己 → 新 self 又改自己"的递归。

**2. Divergence 检测**  
每次 self-modify 前对比新旧 manifest，打分 divergence score：
- instructions 相似度低于阈值 → reject（"change too drastic"）
- reasoning / tools / handoffs 整体重写 → reject
- 每次 self-modify 只允许改若干字段中的一小部分

**3. Modification Budget**  
每个 agent 有 `selfModifyCount` 计数器。默认 N=3 次后必须人工 unlock（避免无限反身）。

**4. Rollback**  
旧版本保留；CLI / API 可 `kodax constructed rollback <agent-name>`。

**5. 版本审计日志**  
所有 self-modify 写入 `.kodax/constructed/_audit.jsonl`，含：时间、diff、reason、结果、rollback 信息。

#### Policy Gate 强制

- self-modify 的 policy gate **永远** ask-user，即使配置了 auto-approve 规则
- 用户看到 diff + reason + sandbox test 结果 + divergence score 之后确认
- 用户可以永久拒绝某个 agent 的 self-modify 能力（Guardrail 级禁用）

### Contract

**Public API（新增）**：
- `scaffold_self_modify` / `validate_self_modify` / `sandbox_test_self_modify` 内部 tools
- CLI: `kodax constructed rollback <name>`、`kodax constructed audit <name>`、`kodax constructed disable-self-modify <name>`
- Guardrail: `SelfModifyGuardrail`（可 per-agent 禁用）

**Internal**：
- ConstructionRuntime 对 kind='agent' + 同名 stage 操作视为 self-modify path
- Runner 检测 "current run is modifying self" 并 defer 到 next run

### Non-Goals

- **不**支持 agent 修改别的 agent 的 role spec（那是档 3 的变体 + 安全问题）
- **不**支持 builtin agent 被 self-modify（只 constructed agent 可以）
- **不**支持跨 session 的 self-modify 协作
- **不**自动学习用户反馈进行 self-modify（agent 必须显式触发）

### Dependencies

- **前置**：
  - FEATURE_089 (v0.7.31) — Agent generation 的 manifest + ConstructionRuntime 就位
  - FEATURE_078 (v0.7.29) — reasoning profile 实装，self-modify 才能改它
  - FEATURE_085 (v0.7.26) — Guardrail runtime（静态检查 instructions、SelfModifyGuardrail）
- **后置**：无（本 feature 是路线图终点）

### Testing

- 单元：divergence score 计算；修改边界 schema 校验
- 单元：modification budget 计数 / reset
- 集成：agent 触发 self-modify → sandbox test → user approval → activate → next run 生效
- 集成：尝试越过修改边界的 case 被 reject（如改 name、max 越 ceiling）
- 集成：反身递归 case 被 defer 机制拦截（同一 run 内不 swap）
- 安全：恶意 self-modify（prompt injection 进 new instructions）被 Guardrail + 用户拒绝拦截
- 审计：所有 self-modify 操作出现在 `_audit.jsonl`

### Risk

本 feature 是整条路线图**风险最高**的一 feature。风险面：

- **反身漂移**：agent 累积多次 self-modify 后行为偏离初始设计
  - 缓解：modification budget + divergence detection + rollback + 审计日志
- **越权**：agent 通过 prompt injection 假装用户确认
  - 缓解：policy gate 强制真实用户输入；Guardrail 静态检查 instructions 无越权关键词
- **安全关闭**：恶意 self-modify 把 Guardrail 移除（"只能加不能减"规则强制）
  - 缓解：修改边界 schema 检查；Guardrail ratchet
- **版本混乱**：同名多版本并存下 Resolver 行为不清晰
  - 缓解：archived 旧版本不解析，只作为 rollback 目标；active 只有一个

### Release criteria

发版前必须满足：

- [ ] 所有 5 条反身稳定保障机制实装 + 测试绿
- [ ] 至少一组端到端 self-modify scenario 跑通（例如 "evaluator 自己升 reasoning 档位"）
- [ ] 至少 3 组 adversarial case（prompt injection / 越权 / 无限反身）被拦截
- [ ] `_audit.jsonl` 格式稳定，能 replay / rollback
- [ ] 文档：docs/FEATURE_090_USER_GUIDE.md 解释 self-modify 的用法与风险

### Timeline

FEATURE_090 单独发版前做一次完整 eval regression + 安全审计。FEATURE_107 同版本搭载但独立交付，不阻塞 FEATURE_090。

## FEATURE_107: AMA H2 Plan-Execute Boundary Eval

**Status**: Planned
**Priority**: Medium
**Category**: Eval / Architecture Validation

### 背景

[v0.7.16 FEATURE_061](./v0.7.16.md) 的设计文档里同时写了两条原则，但只在一处贯彻：

1. **被引用的 Anthropic harness 原则**（[v0.7.16.md:33](./v0.7.16.md#L33)）：models don't need context resets between roles (Opus 4.6)
2. **被实际落实的 context 传递规则**（[v0.7.16.md:116-123](./v0.7.16.md#L116-L123)）：6 条 handoff 中 5 条 same-session 或 resume，唯独 `Planner → Generator` 是 `new session + plan artifact`

理由写的是"Generator only needs the plan, not Planner's reasoning"——但这句话假设 plan artifact 是无损压缩。KodaX 自身设计内有反证：

- `replan` 路径存在（最多 1 次）——显式承认 plan 在执行中会暴露 lossy
- `Planner self-check (1x)` ——承认 Planner 产出的 artifact 自身有缺陷
- H1 的 `Scout → Generator same session` 已经证明同 session 续跑可行；H2 唯独不这么做缺少独立论据

本 feature **不预设答案**，用 pre-registered eval 量化这一跳是否值得保留。

### 假设

- **H1 (主)**：H2 的 `Planner → fresh Generator(+plan artifact)` 存在可观测信息损失，表现为 Evaluator reject 率高 / replan 触发率高 / 总 token 浪费
- **H0 (零)**：plan artifact 是充分压缩，artifact 切分与同 session 续跑无显著差异

### 实验变体（4 路对比）

| 变体 | 含义 | 来源 |
|---|---|---|
| **H0-actual** | session 里实际跑的 H0_DIRECT 结果 | 免费 baseline（直接读 jsonl） |
| **H2-A** (现状) | 强制走 H2 + new-session Generator + plan artifact | 重跑 |
| **H2-B** (合并) | 强制走 H2 + same-session Generator | 重跑 |
| **H1-ref** | 强制走 H1 same-session（无 Planner） | 重跑 |

H0-actual 是 **post-data-discovery 新增**：扫描 `~/.kodax/sessions/` 发现 H2 session 实际为 0 个、H0_DIRECT 为 376 个，原"复用真实 H2 telemetry"路线不可行。改用"should-have-been-H2"候选池：从 H0_DIRECT 中过滤出疑似应该被 H2 处理的 case，重跑 4 路对比。H0-actual 直接来自 session 历史记录，无需重跑（免费 baseline）。

H1-ref 通过 eval harness 的 `forceMaxHarness: 'H1'` flag 跳过 Scout verdict，直接走 H1 same-session 路径。H1-ref 不要求代码改动 H1 自身。

不引入 C 变体（Planner 角色合并进 Generator）。H1-ref 已经回答了"Planner 角色是否必要"，不需要重写 role-prompt。

### 数据集

混合来源，**目标 25–30 个 H2-class case**：

1. **Should-have-been-H2 真实回放**（约 60%）：从 `~/.kodax/sessions/*.jsonl` 的 376 个 H0_DIRECT session 中过滤"疑似应被 H2 处理"的 case
2. **手工策划**（约 40%）：覆盖典型 H2 场景：
   - 跨包大重构（KodaX 自身代码可作素材）
   - 多文件特性新增（如新 invariant + admission hook + tests）
   - bug fix 涉及 3+ 文件协同改动
   - prompt 改写 + 同步改 eval golden

**Should-have-been-H2 过滤启发式**（P1.0 实装）：
- prompt 里出现 `多文件 / 重构 / 跨包 / refactor / multi-file / 重新设计` 等关键词
- session 涉及 ≥ 3 个 unique 文件 edit
- prompt 长度 > 200 字符且含计划/规格类语言
- session 持续 > 5 轮 tool call
- 人工 review 候选池后挑选确认 H2-class 的 case

#### Session 数据源结构

每个 jsonl 文件包含三类记录：
- `meta`（首行）：`title` / `gitRoot` / `createdAt` / `scope` / `uiHistory`
- `lineage_entry`（多条）：每条消息 + role + tool call/result + parent chain，带时间戳
- `meta_update`（偶发）：session 进度快照

可提取的关键字段：

| 字段 | 来源 | 用途 |
|---|---|---|
| user 原始 prompt | `uiHistory[0]` 或首条 user role lineage entry | 重放输入 |
| **git head SHA** | system role lineage 里的 `Repository Intelligence` 块 (`head=<sha>`) | checkout 重建 repo 状态 |
| `dirty=yes/no` | 同上 | 决定该 session 能否干净重放 |
| Scout verdict | lineage 里 Scout 角色的 thinking + `emit_scout_verdict` tool call | **H2 case 筛选条件** |
| Planner / Generator / Evaluator outputs | 后续 lineage entries | golden signal 标注的输入 |
| Execution mode | `meta.title` 前缀 `[Execution Mode: ...]` | 区分 AMA vs SA |

#### 筛选 H2 case 的流程

1. 扫 `~/.kodax/sessions/*.jsonl`（KodaX 项目当前 ~533 个）
2. 过 meta：`scope` 字段指定项目；`title` 含 AMA 标记
3. 过 lineage：找到 Scout `emit_scout_verdict` tool call 中 `harness=H2` 的 session
4. 过 dirty 状态：优先 `dirty=no`（干净重放），`dirty=yes` 次选（按需重建）
5. 标注 golden：人工 review Generator deliverable + Evaluator verdict，确认这是个有意义的 H2 case

预估筛选后 KodaX 单项目能产出 30–80 候选 case。从中挑 15–18 个进 dataset；不够用的话用户可提供其它项目的 session 数据补充。

每个 case 落到 dataset 后的字段：

```yaml
id: H2-real-001
source: ~/.kodax/sessions/20260429_092059.jsonl
prompt: <user 原始指令>
repo_fixture:
  git_head: <SHA from Repository Intelligence>
  dirty_diff: <optional patch if original was dirty>
golden_signals:
  scout_verdict_actual: H2
  must_touch_files: [...]    # 人工标
  must_not_touch: [...]      # 人工标
  acceptance_criteria: <自然语言，供 LLM judge>
expected_complexity: H2
```

### 主指标

| 指标 | 定义 | 比较方向 |
|---|---|---|
| **first-pass acceptance rate** | Evaluator 第一次就 accept 的比例 | 越高越好 |
| **final acceptance rate** | budget 内最终 accept 比例（含 revise/replan） | 越高越好 |
| **plan-intent fidelity** | LLM judge 评 deliverable 是否匹配 plan 原意（H0 成立时验证 artifact 真的充分） | 越高越好 |
| **total tokens / case** | input+output 总 token | 同质量下越低越好 |
| **replan trigger rate (A only)** | A 的 replan 触发率 | 间接反映 artifact lossy 程度 |

### 次指标

- P50 / P95 wall-clock
- Generator iteration 次数
- **context window peak**：B 是否撞窗口（关键风险信号）
- cache hit rate（B 同 session 续跑命中率会高，需要分别报）

### Pre-Registered 决策准则（4 档落档）

跑 eval **之前**就锁死，避免事后 cherry-pick：

```
档 1 — Ship B（B 胜出）⟺
  - final acceptance rate(B) ≥ final(A) − 2%
  - first-pass acceptance rate(B) ≥ first-pass(A) − 0%
  - plan-intent fidelity(B) ≥ fidelity(A) − 0%
  - total tokens(B) ≤ tokens(A) × 1.15
  - context window peak(B) < model_cap × 90% 在 ≥ 95% 的 case 上成立

档 2 — Stay on A（A 胜出）⟺
  - first-pass(A) > first-pass(B) + 5%（artifact 真有用）
  - 或 B 在 ≥ 10% case 上撞 context

档 3 — H1-ref 强 → Planner 必要性存疑 ⟺
  - H1-ref 在 first-pass + final acceptance 主指标上 ≥ A 和 B
  - 处理：**不自动**新建 feature，产出"Planner 必要性发现报告"等讨论

档 4 — 全部 H2 变体 ≤ H0-actual ⟺
  - H2-A、H2-B、H1-ref 三者中**没有**一个在主指标上明显优于 H0-actual
  - 处理：**不自动**转 KNOWN_ISSUES 或新 feature，产出"H2 升级机制有效性发现报告"等讨论；本 feature 在此情况下落到 P5.5 暂停，等用户确认后续动作

模糊地带（任何条件不满足）⟺
  - 不动代码，结果写进 ADR 标注"未决"，进 P5.5 复盘后决定后续动作
```

**触发档 3 / 档 4 的处理纪律**：FEATURE_107 不允许在没有用户拍板的情况下自动创建 KNOWN_ISSUES 条目或新 feature。所有 escalation 都先产出独立的发现报告（位置 `docs/features/v0.7.32-feature-107-findings.md`），由用户 review 后决定下一步。

### Post-Hoc Review（用户特别要求）

P5.5 复盘会做三件事：

1. **阈值合理性回查**：上述决策准则中的阈值（−2% / 1.15× / 90% / 95% / +5%）是数据前的开放性前置判断，跑完后回看每条阈值是否合理：
   - 阈值是否过于宽松导致放行了不该放行的方案？
   - 阈值是否过于严格导致漏掉了有价值信号？
   - 哪些阈值有原则依据，哪些是凭直觉拍的？
2. **样本充分性回查**：
   - 25–30 case 的标准差是否支持当前结论？需不需要扩到 50?
   - 真实 / 手工策划比例是否影响结论方向？
3. **是否需要二轮 eval**：根据数据决定要不要触发：
   - C 变体 eval（Planner 合并进 Generator）
   - 扩样本 eval
   - 跨更长 context 任务的 eval（验证 B 的 context window 风险）

P5.5 产出一份独立的 review report，作为 ADR 更新的输入。

### 实施前提

要跑 B，必须**真的实装** B 路径，不能用 prompt 拼接糊弄。

| 改动 | 文件 | 复杂度 |
|---|---|---|
| 新增编译期常量 `H2_PLANNER_GENERATOR_MERGED = false`（**不进 user config**） | `packages/coding/src/task-engine.ts` 顶部 | S |
| `runManagedTask` H2 分支新增 same-session 续跑路径 | `packages/coding/src/task-engine.ts` | M |
| 复用现有 H1 same-session 机制（FEATURE_061 Phase 3） | 同上 | S |
| B 模式下 `replan` 路径**禁用**，统一退化为 revise（保证两变体 recovery 可比） | 同上 | S |
| eval harness 加 variant 维度 + `forceMaxHarness` flag | `benchmark/harness/harness.ts` | S |
| eval harness 加 worktree 隔离封装（每 case 独立 worktree） | `benchmark/harness/worktree-runner.ts`（新建） | M |
| LLM judge 加 plan-intent fidelity 评分项 | `benchmark/harness/judges.ts` | M |
| Scan 工具：从 `~/.kodax/sessions/` 抽 should-be-H2 候选 | `benchmark/harness/scan-h2-candidates.ts`（新建） | M |

**注意**：B 路径代码是**可逆开关**——eval 输了就把 feature flag 后的整段实装删掉，不留半成品。Flag 注释里要写 `// FEATURE_107: DELETE WITH B-PATH IMPL AT P6`。

### Eval 执行隔离（关键安全要求）

**绝对原则**：eval 不允许触碰任何真实项目目录。

| 风险 | 对策 |
|---|---|
| Generator edit 真实 repo 文件 | 每 case 跑前 `git worktree add /tmp/kodax-eval-<id> <historical-sha>`；KodaX 在 worktree 内运行；跑完 `git worktree remove`；worktree **永不 merge 回主仓** |
| Generator 跑 `git push` | eval harness **禁用 push 类工具**（`bash:network` capability tier reject） |
| Generator 修改用户级配置 | 隔离 `KODAX_HOME=/tmp/kodax-eval-home-<id>` |
| Generator 跑 `npm install` 污染外部 | 限制在 worktree 内；worktree 删除时 node_modules 一并清掉 |
| 历史 SHA 已被 rebase 不存在 | 跑前 `git cat-file -e <sha>` 验证；不存在 → skip case |
| 跨项目 session（MarkXbook 等）| 同样 worktree 策略；目标 repo 不在硬盘 → skip case |
| 异常崩溃残留 worktree | harness 用 `try/finally`；启动前扫 `/tmp/kodax-eval-*` 残留并清理 |

**Release 硬条件**：
- v0.7.32 release 前，0 个 case 执行污染主仓（eval 跑完后检查主仓 `git status` 干净 + `git log` head 无变化）
- worktree 清理 leak rate ≤ 1%（极端崩溃可接受残留，但有清理脚本 `npm run benchmark:cleanup`）

### 性能退化零容忍（hard requirement，用户特别要求）

P2 引入 B-path 实装**绝对不能**让 A-path（现状 H2 路径）性能退化。即使 flag 默认 off，B-path 代码本身存在就可能引入：
- 额外分支判断的 micro-overhead
- 共享变量初始化路径变化
- import 关系扩大引发 dead-code 影响
- **同 session 续跑 prefix 增长引发的 context window 占用上升**（codex review HIGH#1 catch）

**强制门槛（含 codex review HIGH#1 修正）**：

| 时机 | 验证方式 | 通过条件 |
|---|---|---|
| P2 完成（flag=off） | 现有 task-engine.test.ts 全套跑 | 0 失败、wall-time ±2% noise band |
| P2 完成（flag=off） | ama-harness-selection eval 跑 1 round | mean wall-clock ≤ baseline × 1.02 |
| **P2 完成（flag=on，B 激活）** | **同上 + context window peak 实测** | **B-path context peak ≤ A-path × 1.10**（10% 是软上限，超出但 ≤ 90% model_cap 时记 warn 不阻断）|
| **P2 完成（flag=on）** | **B-path 同 H2-class 任务跑 3 round 取 P95** | **P95 context peak < model_cap × 0.90 在 ≥ 95% case 上成立**（同 P4 的 context risk 标准但提前到 P2 gate）|
| P4 eval（flag=on） | baseline H0/H1 任务 fanout/token/wall-time 对照 | 不变 |
| Release 前 | 完整测试 + benchmark self-test | 全绿且无 perf regression |

**关键变更**（来自 codex review）：context-window peak **从 P4 secondary metric 提升到 P2 mandatory gate**。原设计 `context window peak(B) < model_cap × 90% 在 ≥ 95% case` 只在 P4 衡量，B 实测撞 cap 的 case 被打 `B-degraded` 标签**排除而非 flag**——这是结构性回归被静默隔离。提前到 P2 gate 确保 B-path 真撞 cap 时**整个 P2 不通过**，不是把锅推给 P4 再排除。

**任何阶段发现 A-path 退化或 B-path context peak 超标** → 回退到该退化点之前的 commit，**不带任何"小退化但 ship"** 的妥协。

### Eval 信号定性（基于 codex review HIGH#3 修正）

**这个 eval 是 exploratory 不是 confirmatory**——本 dataset 量级（16-19 case × 4 variant × 8 provider）下任何 small effect (≤5%) 都不会有统计显著性。决策规则用 effect-size threshold（5% / 10% delta）而非 p-value，是工程上对样本量的诚实让步。

| 这意味着什么 | 实际处理 |
|---|---|
| ≥ 10% effect 的差异**可信** | 主决策规则按这个 |
| 5–10% 的差异**作为参考信号** | P5.5 复盘讨论但不单独触发档位 |
| < 5% 差异**视为 noise** | 不作为决策依据 |
| 任何"无差异" claim | **不能下**——sample 量不支持 reject null |

P5 输出的报告**显式标注**: "本 eval 不支持 'no difference' 结论，只支持 effect-size threshold 的方向性决策"。

### Dataset 不足时的 fallback 策略（基于 codex review HIGH#3 修正）

P1.5 + P1.5b + Pool 3 archaeology 跑完后，如果 viable case 数 < 18：

| 实际 viable 数 | 处理 | Eval 性质 |
|---|---|---|
| ≥ 18 | 按计划走 | exploratory + effect-size based |
| 12–17 | 用实际数；P5 报告标注样本量限制 | exploratory only — 不主张统计显著 |
| < 12 | **暂停 P2**，inventory 作为 KNOWN_ISSUES 标注，触发 framing 复盘 | 数据量不足以支撑任何决策 |

**12 这个数字不是统计 floor**（codex 指出严格 power analysis 需要 25+/cell），是**工程上的"还能跑出方向性信号"checkpoint**——12 case × 4 variant × 8 provider = 384 trial，能看出 ≥10% 的方向差异。

**绝不做**：编造案例、拉跨项目数据补足、降低 multi-file 阈值降标。"如果实在找不到，不做过度的事情"是用户硬约束。

### Dataset 不足时的 fallback 策略

P1.5 + P1.5b + Pool 3 archaeology 跑完后，如果 viable case 数 < 18：

| 实际 viable 数 | 处理 |
|---|---|
| ≥ 18 | 按计划走，挑 18 |
| 12–17 | **降低 dataset target 到实际数**，P5.5 复盘标注样本量限制 |
| < 12 | 暂停 P2，把现有 dataset 作为 KNOWN_ISSUES 标注 + 触发 FEATURE_107 framing 复盘（不是失败，是数据驱动的 stop signal） |

**绝不做**：编造案例、拉跨项目数据补足（与 KodaX 单项目假设不一致）、降低 multi-file 阈值降标。"如果实在找不到，不做过度的事情"是用户硬约束。

### Provider 矩阵

### 风险与缓解

| 风险 | 缓解 |
|---|---|
| Context window 爆掉让 B 看似更差 | 单独统计 context peak；爆掉的 case 标记 "B-degraded" 单列，不污染主指标 |
| Cache 行为差异让 B 看似便宜 | 同时报 cache hit 率；token 成本分别算 cached / non-cached |
| Provider 表现差异大 | 8 provider 全跑；report 分 provider 列 |
| Dataset 偏小拿到噪声 | 25–30 起步；P5.5 复盘决定是否扩到 50 |
| Evaluator 偏向某变体输出风格 | 抽 5% case 人工核对 verdict |
| H1-ref 无法 force 走 H1 路径 | eval harness 加 `forceMaxHarness` flag 直接 override Scout verdict（与生产路径正交） |
| B 同 session 让 Generator 错误延续 Planner 失误 | 这正是要测的现象之一；记录就好 |
| 单项目（KodaX）session 不够多样化 | 用户可提供其它项目的 `~/.kodax/sessions/` 数据补充；手工策划兜底比例可调 |
| `dirty=yes` session 无法干净重放 | 优先选 `dirty=no` session；`dirty=yes` 必要时用 lineage entry 重建初始 patch |

```
档 3 — H1-ref 强（Planner 必要性存疑）
  → P5.5 复盘后产出"Planner 必要性发现报告"

  → B-path flag + 实装代码默认按档 2 处理（删除）

档 4 — 全部 H2 变体 ≤ H0-actual（H2 整体有效性存疑）
  → P5.5 复盘后产出"H2 升级机制有效性发现报告"

  → 这是比 plan/execute 边界更深的问题，超出本 feature scope
  → B-path flag + 实装代码默认按档 2 处理（删除）

模糊地带（任何条件不满足）
  → 不动代码，结果写进 ADR 标注"未决"
  → P5.5 复盘后决定后续动作
```
