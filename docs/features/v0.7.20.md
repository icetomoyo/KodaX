## FEATURE_072: Lineage-Native Compaction Migration

**Status**: Completed (v0.7.20 unreleased — Phase A/B/C landed; Phase D test-guide shipped in place of P4/P6 deletion)
**Priority**: High
**Category**: Internal / Refactor
**Builds on**: v0.7.18 `FEATURE_067` (child-executor), v0.7.19 surgical compaction fixes
**Reference implementations**: Claude Code (`src/services/compact/`), pi-mono (`packages/coding-agent/src/core/session-manager.ts` + `compaction/compaction.ts`)
**Design review**: revised after adversarial review — see §Design 1b (eviction coverage), §1c (cloneForkableEntry), §2 (1-to-1 contract preservation for 073), §3 (type-separated signature), §4a (microcompact immutability), §5 (three REPL sites + compact-strategy scope exclusion), §6 (emitIterationEvents complements scope), §7a (graceful writeback), §7b (snapshot coherence — two write sites), §7c (rewind semantics)
**Downstream**: `FEATURE_073` was cancelled on 2026-04-19 after adversarial review — see [v0.7.25.md §FEATURE_073 Cancellation rationale](../features/v0.7.25.md#feature_073-reference-style-lineage-and-island-model-removal). The three invariants 073 would have depended on are still load-bearing as general hygiene: (i) `postCompactAttachments` field on `CompactionEntry` (part of 072's on-disk shape), (ii) the 6-consumer flat-messages path is documented but not migrated (deferred indefinitely with 073 cancelled — see Acceptance), (iii) `getContextMessagesForEntry`'s 1-to-1 contract, guarded by a unit test, stays as a general cleanliness rule and as a safety net if a future lineage-structure feature is ever revived.

### Version context

#### Origin — the v0.7.18 post-compact regression

In v0.7.18, `FEATURE_067`-adjacent work added **post-compact attachments**: after an LLM-based summary, the runtime re-injects an artifact-ledger summary + recently-modified file contents so the agent "remembers" what it just operated on without re-reading.

The feature is good in principle — it is directly modelled on Claude Code's `createPostCompactFileAttachments` pattern. But the KodaX implementation diverged from the reference in four subtle ways that together produced the bug users observed in production (monotonic context growth to 180k+ with no effective compaction):

1. **Proportional rather than fixed budget.** KodaX used `freedTokens × 0.5`. Claude Code uses a fixed `POST_COMPACT_TOKEN_BUDGET = 50_000` with a per-file cap of `5_000`. A single large compaction could re-inject 60k+ of content proportionally.
2. **No idempotence / dedup on re-injection.** Claude Code performs `context.readFileState.clear()` at the compaction entry point and relies on `getMessagesAfterCompactBoundary` slicing to keep prior attachments out of both the next summarization input and the next API request. KodaX injected new `[Post-compact: ...]` system messages each round without removing prior generations — over N iterations, N generations stacked.
3. **Silent summary failure path.** Claude Code throws explicitly when the summary LLM returns no usable text (`Failed to generate conversation summary - response did not contain valid text content`). KodaX's `summarizeMessages` swallowed the exception and returned `{ failed: true, summarizedMessages: 0 }`, which the caller interpreted as "compacted: true" because pruning had done something. The result: a huge new-reference array still above trigger, slipping past the `compacted === messages` graceful-degradation gate.
4. **Graceful degradation gate keyed on reference equality.** `if (compacted === messages)` was intended to mean "intelligent compact did nothing, fall back". But when partial success returns a new array with no real shrink, reference equality is `false` and the fallback never runs.

#### The v0.7.19 surgical fix

v0.7.19 (shipped as six independent bug fixes) addressed the **immediate growth bug**:

| Fix | Location | Summary |
|---|---|---|
| P1 | `agent.ts:1806-1821` | Graceful degradation gate changed to a token-count check, not reference equality |
| P2 | `agent.ts:1761-1775` | `compactConsecutiveFailures` only resets when `tokensAfter < triggerTokens`; circuit breaker now trips after 3 "partial successes" |
| P3 | `summary-generator.ts:383-391` | `generateSummary` throws on empty text, mirroring Claude Code's contract |
| P4 | `post-compact.ts:106-162` | `injectPostCompactAttachments` strips prior `[Post-compact: ...]` system messages before inserting new ones |
| P5 | `post-compact.ts:29-45` + `agent.ts:1740-1746` | Absolute caps `POST_COMPACT_TOKEN_BUDGET = 50_000` and `POST_COMPACT_MAX_TOKENS_PER_FILE = 5_000` |
| P6 | `InkREPL.tsx:6553-6566` | Run-end `finally` block rebuilds `context.contextTokenSnapshot` from local messages, clearing worker-leaked snapshots |

These fixes resolve the user-visible bug but leave the **underlying structural debt unaddressed**. `FEATURE_072` is the structural follow-up.

#### Why a structural follow-up is necessary

The v0.7.19 fixes all live on top of a **dual source-of-truth architecture** that is itself the root enabler of this class of bug:

- `context.messages` — flat array, source-of-truth for the agent loop, API send path, microcompact, needsCompaction, intelligentCompact input
- `context.lineage` — tree structure (`KodaXSessionLineage`), source-of-truth for session storage, `/fork`, `/rewind`, session-tree navigation

These two views are kept in sync through `applySessionCompaction` + `createSessionLineage`. Any code path that writes one without the other causes silent drift. The v0.7.19 dedup (P4) is a defensive string-match on `[Post-compact:` — fragile to format changes, and it only cleans flat messages, not the lineage copy that gets persisted.

**KodaX already has the infrastructure for the correct architecture.** `KodaXSessionCompactionEntry` carries `firstKeptEntryId` (identical semantics to pi-mono's `CompactionEntry`); `getSessionMessagesFromLineage(lineage, leafId)` already derives the effective LLM view (equivalent to Claude Code's `getMessagesAfterCompactBoundary`); `storage.ts`, `session-storage.ts`, and REPL `/load-session` / `/fork` / `/rewind` all consume `getSessionMessagesFromLineage`.

**But the compaction hot path does not.** `agent.ts` reads and writes flat `messages`, and `applySessionCompaction` is called as a side-effect to rebuild the lineage copy afterwards. Lineage is an output, not a source. `FEATURE_072` finishes the inversion.

### Reference implementations

#### Claude Code — boundary marker + slicing

Claude Code uses a sentinel `SystemCompactBoundaryMessage` that sits inline in the flat message array. [`getMessagesAfterCompactBoundary`](c:/Works/claudecode/src/utils/messages.ts#L4643-L4655) walks the array backwards to the most recent boundary and slices from there:

```ts
const sliced = boundaryIndex === -1 ? messages : messages.slice(boundaryIndex);
```

Post-compact attachments are appended **after** the boundary. The next compaction call receives the sliced view as input, so prior attachments never re-enter summarization. `context.readFileState.clear()` is called alongside so file-attachment state also resets.

Claude Code's strengths:
- **One invariant**: API view = post-last-boundary slice
- **Idempotent dedup**: old attachments exit the view when a new boundary is placed
- **Works across compactions**: the summary call's input is bounded by trigger - protection, not growing indefinitely

Claude Code's limitation for KodaX:
- Inline sentinel doesn't compose with KodaX's existing `KodaXSessionLineage` tree — we would end up with both a boundary marker in the array and a CompactionEntry in lineage, duplicating the "where did compaction happen" metadata.

#### pi-mono — lineage-derived messages

pi-mono (which KodaX's `session-lineage.ts` was originally modelled on) goes further: **lineage is the source, messages are derived**. [`buildSessionContext(entries, leafId)`](c:/Works/GitWorks/pi-mono/packages/coding-agent/src/core/session-manager.ts#L315-L409) walks the tree from leaf to root, finds the most recent `CompactionEntry`, and emits `summary + kept entries (from firstKeptEntryId) + post-compaction entries`:

```ts
if (compaction) {
  messages.push(createCompactionSummaryMessage(compaction.summary, ...));
  let foundFirstKept = false;
  for (let i = 0; i < compactionIdx; i++) {
    const entry = path[i];
    if (entry.id === compaction.firstKeptEntryId) foundFirstKept = true;
    if (foundFirstKept) appendMessage(entry);
  }
  // Then entries after compaction
}
```

All consumers (API send, microcompact, next compaction's input, fork/rewind) read from `buildSessionContext`. There is no flat-messages array maintained in parallel.

pi-mono's strengths:
- **Single source of truth**: impossible for the two views to drift
- **Fork/rewind are trivial**: `buildSessionContext(entries, newLeafId)` gives the correct view for any leaf
- **Multi-boundary natural**: partial compaction adds one more `CompactionEntry` without any special case

pi-mono's cost:
- Every read re-derives the array (mitigated by small per-session memoization via `byId` map)

#### Current KodaX — hybrid, biased toward flat messages

KodaX has pi-mono's **infrastructure** (entries, `firstKeptEntryId`, `getSessionMessagesFromLineage`, island eviction) but runs the **agent loop on flat messages**. Storage layer and UI-side commands use lineage derivation; the model-facing hot path does not. This asymmetry is the structural debt.

### Goals

1. **Make lineage the single source of truth for the compaction hot path.** Four consumer points in `agent.ts` switch from reading `messages` directly to reading via `getSessionMessagesFromLineage(lineage, leafId)` (or an equivalent derivation).
2. **Store post-compact attachments as first-class lineage data**, attached to the `KodaXSessionCompactionEntry` that created them — not as loose `[Post-compact: ...]` system messages. Old attachments are dropped naturally when a new `CompactionEntry` is appended.
3. **Let Scout start from the derived view**, not the parent's full flat messages. Eliminates the "Scout inherits 80k of parent history on every AMA turn" effect.
4. **Scope worker `onIterationEnd` events so they do not pollute the parent REPL's token snapshot** mid-run. The v0.7.19 finally-block cleanup becomes unnecessary once events are scoped correctly.
5. **Make the dedup structural, not string-based.** Remove the `[Post-compact:` prefix-matching dedup introduced in v0.7.19 — it becomes redundant and is a fragile layering-on-top-of-hack.
6. **Preserve all current user-visible behavior**: `/fork`, `/rewind`, `/load-session`, resume, persist, multi-worker AMA rounds must all produce identical observable behavior. This feature is an internal refactor, not a UX change.

### Non-goals

- **Not adding partial compaction** (`direction: 'from' | 'up_to'` like Claude Code's `partialCompactConversation`). The lineage structure makes future partial support natural; we scope this feature to the full-compaction path only.
- **Not adopting Claude Code's `cache_edits` API.** That feature depends on Anthropic-specific prompt-cache editing; KodaX supports multiple providers and must work uniformly.
- **Not removing `evictOldIslandMessageContent`.** The lineage memory-eviction discipline from v0.7.15-era work stays — it is orthogonal to this feature.
- **Not changing the session storage format.** `KodaXSessionCompactionEntry` gets one new optional field; old sessions load with `postCompactAttachments: undefined` and behave identically to pre-migration sessions.
- **Not changing the `SA` (single-agent) path semantics.** SA users should see zero behavior difference.
- **Not introducing multi-boundary in a single active path.** Each `applySessionCompaction` still opens a new island (parentId=null) as it does today; multi-boundary becomes possible if a future feature wants it.

### Design

#### 1. Extend `KodaXSessionCompactionEntry` with native attachments

```ts
// packages/agent/src/types.ts
export interface KodaXSessionCompactionEntry {
  type: 'compaction';
  id: string;
  parentId: string | null;
  timestamp: string;
  summary: string;
  firstKeptEntryId?: string;
  tokensBefore?: number;
  tokensAfter?: number;
  reason?: string;
  artifactLedgerId?: string;
  details?: KodaXJsonValue;
  memorySeed?: KodaXCompactMemorySeed;
  // NEW
  postCompactAttachments?: readonly KodaXMessage[];
}
```

`postCompactAttachments` holds the ledger-summary message + file-content messages that today are injected into flat messages. Keeping them inside the entry means:

- When a new `CompactionEntry` is appended, the old entry's attachments leave the active path automatically (because the new entry is the new island root with `parentId: null`).
- No string-prefix dedup needed. No cross-round accumulation possible.
- Persistence already handles arbitrary JSON in entry fields.

##### 1a. `postCompactAttachments` vs. `memorySeed` — roles and coexistence

`KodaXSessionCompactionEntry` already carries `memorySeed: KodaXCompactMemorySeed | undefined`. The two fields look conceptually similar ("payload re-injected after compaction") but serve different consumers and must not be conflated:

| Field | Consumer | Lifetime | Content |
|---|---|---|---|
| `memorySeed` | AMA worker handoff / role-resume — read by `buildManagedWorkerMemoryNote` and rendered as a synthetic system note for `memoryStrategy === 'compact'` workers | Survives across **sessions** (persisted, re-read when a worker resumes) | Compact text seed (< 1k tokens) describing what the role was doing |
| `postCompactAttachments` | Same-session API send path — read by `getSessionMessagesFromLineage` and emitted as real messages inline with the compaction summary | Lives inside the active island; replaced on next compaction | Ledger summary + recently-modified file contents (≤ 50k tokens) |

**Coexistence rule**: both fields may be present on the same `CompactionEntry`. `memorySeed` is produced by `intelligentCompact` regardless of whether attachments exist; `postCompactAttachments` is only produced when the artifact ledger is non-empty. Eviction, fork, and rewind must treat the two independently (see §1b, §1c).

##### 1b. Old-island eviction must cover `postCompactAttachments`

[`evictOldIslandMessageContent`](../../packages/agent/src/session-lineage.ts#L652-L711) today only touches `type === 'message'` entries ([L693](../../packages/agent/src/session-lineage.ts#L693) `if (entry.type !== 'message') return entry`). If `postCompactAttachments` is added without extending eviction, the old island's `CompactionEntry` keeps ≤ 50k tokens of frozen attachments **forever** — N compactions accumulate N × 50k, which is exactly the monotonic-growth pathology 072 exists to fix, in a new shape.

**Required change**: `evictOldIslandMessageContent` must also strip `postCompactAttachments` (set to `undefined`) on CompactionEntries outside the active island. `summary`, `firstKeptEntryId`, `memorySeed`, and tree-structure fields stay — only the heavy attachments leave. `memorySeed` stays because its purpose is cross-session resume; `postCompactAttachments` leaves because its purpose is same-session context and dead islands have no active same-session context.

##### 1c. `cloneForkableEntry` must be updated explicitly; `cloneEntry` inherits via spread

[`cloneEntry`](../../packages/agent/src/session-lineage.ts#L97-L102) uses `{ ...entry, details, memorySeed }` — `...entry` carries `postCompactAttachments` automatically. **No change needed.**

[`cloneForkableEntry`](../../packages/agent/src/session-lineage.ts#L731-L743) manually enumerates fields — it would silently drop `postCompactAttachments` on `/fork`, producing a branch that loses ledger context. **Must explicitly add `postCompactAttachments: entry.postCompactAttachments` to the returned object.**

Phase A adds a fork-preservation unit test asserting `/fork` across a CompactionEntry preserves both `memorySeed` and `postCompactAttachments`.

#### 2. Emit attachments at `getSessionMessagesFromLineage`, **preserve `getContextMessagesForEntry`'s 1-to-1 contract**

An earlier draft of this section folded attachments into `getContextMessagesForEntry`'s compaction branch, making it emit `[summary, ...attachments]` — **1-to-many**. That choice breaks a stable contract that `FEATURE_073` (Reference-Style Lineage) explicitly depends on ([v0.7.25.md:700-704, 787](./v0.7.25.md#feature_073-reference-style-lineage-and-island-model-removal)). 073's slicing assumes each `getContextMessagesForEntry(entry)` call returns exactly one message for compaction / message / branch_summary (and zero for archive_marker); attachments are prepended separately at the top-level slicer. Keeping 072 aligned with this contract means 073 can land without re-plumbing the derivation.

**Rule**: `getContextMessagesForEntry` stays as it is today ([session-lineage.ts:182-209](../../packages/agent/src/session-lineage.ts#L182-L209)) — compaction entry → `[summary]`, message → `[body]`, branch_summary → `[summary]`, archive_marker → `[]`. **Attachments are never emitted by this function.**

Instead, `getSessionMessagesFromLineage` is upgraded to teach it compaction awareness:

```ts
// packages/agent/src/session-lineage.ts
export function getSessionMessagesFromLineage(
  lineage: KodaXSessionLineage,
  targetId: string | null = lineage.activeEntryId,
): KodaXMessage[] {
  const path = getSessionLineagePath(lineage, targetId);
  return path.flatMap((entry) => {
    const base = getContextMessagesForEntry(entry);  // 1-to-1 contract preserved
    if (entry.type === 'compaction'
        && entry.reason !== 'rewind'                  // rewind markers don't carry attachments
        && entry.postCompactAttachments?.length) {
      return [...base, ...entry.postCompactAttachments.map(cloneMessage)];
    }
    return base;
  }).map(cloneMessage);
}
```

Under copy-style 072, the active path contains at most one non-rewind `CompactionEntry` (because `applySessionCompaction` creates a new island), so this produces `[...pre-compaction tail, summary, ...attachments, ...kept, ...subsequent]` — identical ordering to `injectPostCompactAttachments` today. Zero behaviour change at the wire level for the LLM.

**Why this composes cleanly with 073**: 073 switches from copy-style to reference-style and introduces `firstKeptEntryId`-based slicing. Its derivation ([v0.7.25.md:700-704](./v0.7.25.md#feature_073-reference-style-lineage-and-island-model-removal)) prepends `[summary, ...attachments]` for the most recent compaction and then slice-flatMaps the kept tail. **If 072 had put attachments inside `getContextMessagesForEntry`**, the 073 slicer would either double-emit (if its flatMap covered the compaction entry) or need a special skip-the-compaction-entry rule (fragile). Keeping 072's `getContextMessagesForEntry` 1-to-1 means 073 just generalizes the slicing location — no contract rework.

**Contract declaration** (for 073 to rely on): `getContextMessagesForEntry(entry).length ≤ 1` for every entry type. Tested explicitly in Phase A.

#### 3. Update `applySessionCompaction` with a **type-separated** attachments parameter

The initial draft put `postCompactAttachments` inside the `anchor` bag alongside existing metadata and relied on callers to "remember to strip attachments out of `compactedMessages` before calling". That contract is invisible to the type system — a forgetful caller silently double-stores attachments (once inline as `message` entries in lineage, once as `postCompactAttachments`), and the only way to catch it is a runtime assertion downstream.

**Type-enforce the separation instead**: make `keptMessages` and `attachments` **separate top-level parameters**, and document that `keptMessages` must be attachment-free:

```ts
export function applySessionCompaction(
  lineage: KodaXSessionLineage | undefined,
  keptMessages: KodaXMessage[],          // post-summary kept tail; MUST NOT include attachments
  attachments: readonly KodaXMessage[],   // ledger + file-content messages; written to CompactionEntry
  anchor: {
    summary: string;
    tokensBefore: number;
    tokensAfter: number;
    artifactLedgerId?: string;
    reason?: string;
    details?: KodaXJsonValue;
    memorySeed?: KodaXCompactMemorySeed;
  },
): KodaXSessionLineage {
  // keptMessages → createSessionLineage → message entries under the new CompactionEntry
  // attachments → stored on the CompactionEntry's postCompactAttachments field
}
```

Caller-side in [agent.ts:~1767](../../packages/coding/src/agent.ts#L1767), attachments are **not** injected into `compacted` anymore; they are produced separately by `buildPostCompactAttachments` + `buildFileContentMessages` and passed as the second argument.

**Phase A invariant test**: `applySessionCompaction` output never contains a `message` entry whose `content` starts with `[Post-compact:` or `[对话历史摘要]` — attachments always live on the `CompactionEntry`, never as inline messages. Guarantees the strip contract structurally, not through caller discipline.

#### 4. Migrate all flat-messages consumer points in the compaction hot path

An initial draft listed four consumers. A complete audit of `agent.ts` around the compaction block turns up **six**, plus one intentionally-not-migrated path. Enumerating all of them up front avoids a half-migrated "dual source of truth" state after Phase B:

| # | Consumer | Current location (re-verified post-v0.7.19) | Migration strategy |
|---|---|---|---|
| 1 | `needsCompaction(messages, ...)` | [agent.ts:1718](../../packages/coding/src/agent.ts#L1718) | Read `getSessionMessagesFromLineage(lineage, activeEntryId)` |
| 2 | `intelligentCompact(messages, ...)` input | [agent.ts:1727](../../packages/coding/src/agent.ts#L1727) | Same derived view |
| 3 | `microcompact(messages, config)` | [agent.ts:1707](../../packages/coding/src/agent.ts#L1707) (per-iteration entry) | **No migration needed** — already returns a new array with new objects for trimmed messages. See §4a: Phase A adds a regression guard test only. |
| 4 | `gracefulCompactDegradation(compacted, ...)` | [agent.ts:1825](../../packages/coding/src/agent.ts#L1825) | Input comes from derived view; result writes back via `applyLineageTruncation` (§7a), **not** `applySessionCompaction` |
| 5 | API send path (`messages = compacted` reassignment + downstream provider send) | [agent.ts:1840-1848](../../packages/coding/src/agent.ts#L1840-L1848) | `messages` becomes a snapshot of `getSessionMessagesFromLineage(updatedLineage, ...)` after lineage reconciliation; `validateAndFixToolHistory` still operates on the snapshot as a pre-send sanitizer (intentional, see below) |
| 6 | REPL `onCompactedMessages` handler writing `context.messages = messages` | [InkREPL.tsx:5173-5174](../../packages/repl/src/ui/InkREPL.tsx#L5173) | Keep the assignment, but the `messages` value delivered via the callback is itself the derived view; `context.messages` becomes a per-iteration snapshot of lineage, not an independently-mutated array |

Implementation sketch: introduce a single helper inside `runKodaX`:

```ts
const getActiveMessages = (): KodaXMessage[] =>
  context.lineage
    ? getSessionMessagesFromLineage(context.lineage, context.lineage.activeEntryId)
    : messages;  // legacy fallback for first-turn / no-lineage paths
```

Replace every compaction-block read of `messages` with `getActiveMessages()`. Outside the compaction block (user input / assistant response / tool result append), the local `messages` array can still be appended to during an iteration; the lineage is reconciled via `createSessionLineage(messages, prevLineage)` at iteration-end, at which point `context.messages` is rebuilt from the lineage. This keeps the per-iteration write edge cheap while making lineage the single source of truth at iteration boundaries.

**Not migrated (intentional)**: `validateAndFixToolHistory` ([agent.ts:1840](../../packages/coding/src/agent.ts#L1840)) stays transient — it is a pre-API-send sanitizer that may insert synthetic `tool_result` blocks to repair broken `tool_call_id` chains. Writing its output back into lineage would pollute persistence with synthesized repair messages. It reads the derived view, produces a cleaned array for the provider call, and that array is discarded after send. This is the current behaviour and is correct.

**Error-recovery catch path** ([agent.ts:2716-2738](../../packages/coding/src/agent.ts#L2716-L2738)): calls `cleanupIncompleteToolCalls` + `validateAndFixToolHistory` on caught exceptions, then `saveSessionSnapshot(messages=cleanedMessages)` and `contextTokenSnapshot = createEstimatedContextTokenSnapshot(cleanedMessages)`. This path is triggered by provider errors mid-turn; it does **not** run through `onCompactedMessages`. Phase C covers it with the same migration as §7b — the snapshot recomputation must read from the derived view for consistency with P6 retirement.

##### 4a. `microcompact` is already immutable — regression guard only

Re-verification against current code ([microcompaction.ts:67-151](../../packages/agent/src/compaction/microcompaction.ts#L67-L151)) shows `microcompact` **already** returns a new array with new message objects for trimmed items (line 15 comment: "Immutable: returns new array, never mutates input"; lines 136-147 build `{ ...toolResult, content: '[Cleared: ...]' }` and `{ ...msg, content: newContent }`). The initial draft of this section was based on a speculated "in-place mutate" optimization that never existed in the codebase.

**No production code change needed in Phase A.** Instead, add a regression guard unit test in `microcompaction.test.ts` asserting:

- Output array is a new reference (not `===` input)
- For any message where any block was trimmed (image → text, tool_result → `[Cleared: ...]`), the message object is a new reference (not `===` the corresponding input message)
- For unchanged messages, reference sharing is permitted (current behaviour, preserves memory)

This protects against future regressions where someone might "optimize" microcompact into in-place mutation, which would violate both the project immutability rule and the `/fork`-captures-pre-microcompact-state invariant that `cloneForkableEntry`'s `structuredClone` depends on.

#### 5. Scout `initialMessages` via derived view — three REPL-side sites + one task-engine branch

The initial draft named only `repl.ts:1109` (stale — actual line is [`1140`](../../packages/repl/src/interactive/repl.ts#L1140)). A full audit reveals **three** REPL-side sites that pass `context.messages` as worker `initialMessages`, plus a **task-engine branch** that silently overrides the REPL-provided value for `memoryStrategy === 'compact'` workers:

**REPL sites (replace `context.messages` with the derived view):**

| # | Location | Context |
|---|---|---|
| 1 | [`repl.ts:1140`](../../packages/repl/src/interactive/repl.ts#L1140) | Main agentic turn dispatch; covers Scout + most AMA worker launches |
| 2 | [`InkREPL.tsx:5276`](../../packages/repl/src/ui/InkREPL.tsx#L5276) | Parallel Ink UI path for the same dispatch — missed in the initial draft |

**Task-engine branch (intentionally unchanged — document the scope exclusion):**

[`task-engine.ts:5744`](../../packages/coding/src/task-engine.ts#L5744) — when `memoryStrategy === 'compact'`, the REPL-provided `initialMessages` is **overridden** by `buildCompactInitialMessages` ([task-engine.ts:1697-1715](../../packages/coding/src/task-engine.ts#L1697-L1715)), which synthesizes a memory-note system message from `KodaXCompactMemorySeed`. This is deliberate: `memoryStrategy: 'compact'` exists precisely to give the worker a compact role-resume seed rather than a replay of the parent's context. Deriving from lineage for this strategy would re-introduce the parent-history weight that `compact` strategy is designed to avoid.

**Scope declaration**: the §5 migration affects only workers whose `memoryStrategy !== 'compact'`. `compact`-strategy workers (which is how same-role cross-round resume is implemented today) continue to receive the synthetic memory note. `memorySeed` on the parent's `CompactionEntry` is what feeds that note; `postCompactAttachments` is not used by this path.

For Scout-continuation workers (H1/H2 Generator resuming Scout's session), the existing `continueFromSessionId` path still works: session storage load goes through `getSessionMessagesFromLineage`, which is already lineage-native.

#### 6. Scope worker `onIterationEnd` events

**Pre-existing state**: `createWorkerEvents` in [task-engine.ts:5355-5623](../../packages/coding/src/task-engine.ts#L5355-L5623) already has an `emitIterationEvents` flag (default `false`). Scout ([task-engine.ts:6306](../../packages/coding/src/task-engine.ts#L6306)) and role workers ([task-engine.ts:6202](../../packages/coding/src/task-engine.ts#L6202)) explicitly set `emitIterationEvents: true` because the user-facing "live token count during AMA" UX requires it. **Turning it off is not an option** — it would regress the AMA progress UI.

Given that, the right axis is: **workers keep firing `onIterationEnd` at the parent REPL (for live display), but the parent REPL must distinguish "this event is from a worker" and skip `context.contextTokenSnapshot = info.contextTokenSnapshot`** (at [InkREPL.tsx:5223-5225](../../packages/repl/src/ui/InkREPL.tsx#L5223-L5225)).

Extend `onIterationEnd`'s `info` with an optional `scope?: 'parent' | 'worker'` field. The REPL gates snapshot writes on `scope === 'worker'`:

```ts
onIterationEnd: (info) => {
  if (info.scope === 'worker') {
    // Worker progress: update live display but do NOT mutate
    // context.contextTokenSnapshot (which belongs to the parent).
    setLiveTokenCount(info.tokenCount);
    return;
  }
  context.contextTokenSnapshot = info.contextTokenSnapshot;
  setLiveTokenCount(info.tokenCount);
},
```

**Scope provenance — where does `scope` come from?**

Deliberately **not** inferred from the presence of `managedTask` on the execution context. Under AMA, the parent REPL itself can be a managed task, so `managedTask != null` is not a reliable worker discriminator. Instead:

- Add optional `scope?: 'parent' | 'worker'` to `runKodaX` options (or an equivalent internal execution-context field).
- Worker call-sites in [task-engine.ts](../../packages/coding/src/task-engine.ts) (Scout at line 6302, role workers at line 6193) pass `scope: 'worker'` alongside the existing `emitIterationEvents: true`.
- `runKodaX` populates `info.scope` when emitting `onIterationEnd` at **all 8 call sites** ([agent.ts:2193, 2293, 2350, 2548, 2566, 2586, 2636, 2708](../../packages/coding/src/agent.ts#L2193)) — the single helper `emitIterationEnd` at [agent.ts:1635-1643](../../packages/coding/src/agent.ts#L1635) is the one place to add the field, which propagates to all callers.
- `createWorkerEvents` gains a single line in its existing `onIterationEnd` wrapper to inject `scope: 'worker'` into the forwarded info; no other wrapping logic changes.

Absence of `scope` is treated as `'parent'`, preserving backward compatibility for any external event consumers.

Once this lands, the v0.7.19 P6 finally-block cleanup (at [InkREPL.tsx:6550-6585](../../packages/repl/src/ui/InkREPL.tsx#L6550-L6585), spanning more than the original 6553-6566 after intervening refactors) becomes a no-op and is removed in Phase D.

#### 7. Deprecate and remove the string-prefix dedup

Once lineage-native attachments land, `injectPostCompactAttachments`' stripping of existing `[Post-compact: ...]` messages (v0.7.19 P4) becomes dead code — agent.ts never reads flat messages anymore, and the attachments live inside `CompactionEntry` rather than as loose system messages.

The `injectPostCompactAttachments` function itself can be kept as a pure helper for building the attachment list, but the "strip + insert into array" behavior moves into `applySessionCompaction`.

**Deletion is gated by a regression pass (see Phase D ship criteria)** — P4 is today's belt-and-suspenders guard; removing it requires evidence that no code path can still produce stray `[Post-compact: ...]` system messages.

#### 7a. `gracefulCompactDegradation` writeback — a trim, not a new island

[`gracefulCompactDegradation`](../../packages/coding/src/agent.ts#L1820-L1836) runs when intelligent compact failed, returned partial success, or the circuit breaker is tripped. It does **atomic-block trimming**, not summary-based compaction — there is no LLM summary to write, no ledger to re-inject. Routing its output through `applySessionCompaction` (which unconditionally creates a new island with `parentId: null` and a `CompactionEntry` carrying a `summary` string) would write a degenerate entry with `summary: ''`, muddying the lineage view and the session-tree UI.

**Design**: graceful degradation writes back via a distinct function `applyLineageTruncation(lineage, keptMessages, reason)` that appends an `archive_marker`-style entry (or reuses it directly — `KodaXSessionArchiveMarkerEntry` already exists for this kind of bookkeeping) rather than a `CompactionEntry`. `getSessionMessagesFromLineage` already handles archive markers. No user-visible change; the only difference is that `/fork` and `/rewind` correctly distinguish "summary-compacted here" from "atomically trimmed here" when rendering the tree.

Phase B implements this; Phase A's type changes do not touch the graceful path.

#### 7b. `contextTokenSnapshot` must be derived from the lineage-derived view

Two write sites in `agent.ts` compute `contextTokenSnapshot` from flat `messages` today:

1. [agent.ts:1846](../../packages/coding/src/agent.ts#L1846) — post-compact snapshot refresh (`didCompactMessages` branch)
2. [agent.ts:2738](../../packages/coding/src/agent.ts#L2738) — error-recovery catch block (after `cleanupIncompleteToolCalls` + `validateAndFixToolHistory`)

Once `messages` becomes a per-iteration snapshot of lineage, both sites should compute the snapshot from `getActiveMessages()` — the same derived view that gets sent to the API. Otherwise P6 (the REPL finally-block snapshot reset) cannot be safely removed in Phase D, because the snapshot and the lineage can diverge whenever graceful degradation writes a truncation entry without emitting `onCompactedMessages`, or when the error-recovery catch rebuilds the snapshot from a cleaned-but-not-lineage-reconciled array.

Phase C change: replace both read sites with `getActiveMessages()`, and assert in a unit test that `estimateTokens(getActiveMessages()) === contextTokenSnapshot.totalTokens` at iteration end (both happy and error paths).

#### 7c. `rewindSessionLineage` semantics across a compaction boundary

[`rewindSessionLineage`](../../packages/agent/src/session-lineage.ts#L796-L817) truncates entries after a target and appends a degenerate `CompactionEntry` recording the rewind. If the rewind target **is** a `CompactionEntry` that has `postCompactAttachments`, the post-rewind view should include those attachments — the user wanted to "go back to right after the compaction", and the attachments are part of the state right after the compaction.

**Rule**: rewind preserves the target entry (including its `postCompactAttachments` and `memorySeed`). The synthetic rewind entry is appended *after* the target, not replacing it. This is already how rewind works structurally (`entries.slice(0, targetIndex + 1)`) — we just need an explicit test covering the "rewind lands on a compaction boundary" case to ensure `getSessionMessagesFromLineage` from the new leaf emits the attachments.

Phase D regression gate includes this test explicitly as part of the `/rewind across compaction boundary` criterion.

#### 8. Explicit non-goals at the code level

A short list of things this feature **intentionally does not change**, to prevent scope creep during implementation:

- **`validateAndFixToolHistory`** stays transient (pre-API-send sanitizer, not a lineage writer). See section 4's note.
- **`evictOldIslandMessageContent`** stays in the main compaction path. The copy-on-compact island model is preserved; this function still runs on every `applySessionCompaction` to release dead-island message content. Migrating away from the copy-style model would require rethinking `archiveOldIslands` and `buildSessionTree`'s root identification (both rely on `parentId: null` island roots) and is out of scope.
- **`archiveOldIslands` / `buildSessionTree` / session-tree UI root identification** — unchanged.
- **Parent ↔ worker lineage composition** — unchanged. Workers continue to own independent sessions; parent consumes worker output via `continueFromSessionId` through `ManagedWorkerSessionStorage`. Whether the parent should eventually splice worker lineages into its own tree is a separate design question (open; see below).

### Compatibility and risks

#### Session persistence

`KodaXSessionCompactionEntry` gains an optional `postCompactAttachments` field. JSON persistence and `structuredClone`-based entry cloning both handle extra fields gracefully. Sessions written by v0.7.19 lack the field; loading them yields `postCompactAttachments: undefined`, which `getSessionMessagesFromLineage`'s slicer treats as "no attachments to inline" — identical to how a post-0.7.19 compaction looks before post-compact injection runs.

Old sessions additionally have `[Post-compact: ...]` system messages living as regular `message` entries in lineage.

**Migration path**: **none. Passive cleanup only.** The v0.7.19 string-prefix dedup (P4) is kept through Phases B and C precisely to sweep these up on the next compaction. An eager on-load rewrite was considered and rejected — it would introduce a write side-effect during `/load-session` and risk data loss on abort. A user who loads a pre-0.7.20 session and exits without triggering a compaction will persist a mixed-shape lineage; this is acceptable and self-corrects on the next compaction.

#### Prompt cache interaction

Providers with prompt-cache support (Anthropic `cache_control`) depend on prefix stability. The migration changes what the API payload looks like in two places:

1. **Summary message ordering**: today, post-compact attachments follow `[对话历史摘要]`. After migration, derivation produces the same ordering `[summary, ...attachments, ...kept]`. Prefix stable.
2. **Attachment content stability across iterations**: attachments frozen on `CompactionEntry` are re-emitted verbatim until the next compaction, as opposed to being rebuilt from the ledger each iteration.

No explicit `cache_control` migration needed. **Expected outcome: cache-hit rate does not regress.** A stronger claim (e.g., "improves") requires measurement — a 10-round AMA cache-hit benchmark should be captured in Phase B and reported in the Phase C ship note. No numeric threshold is pre-committed; the bar is "no regression vs v0.7.19".

#### Memory growth

`evictOldIslandMessageContent` today only touches `type === 'message'` entries ([session-lineage.ts:693](../../packages/agent/src/session-lineage.ts#L693)). Without extension, `postCompactAttachments` on old-island CompactionEntries would **never be evicted** — N compactions accumulate N × ≤50k tokens, reproducing the monotonic-growth pathology 072 exists to fix in a new shape.

**Phase A includes the eviction extension** (§1b): strip `postCompactAttachments` on CompactionEntries outside the active island, preserve `summary` + `firstKeptEntryId` + `memorySeed`. This makes the active island the only island holding heavy attachments, consistent with how message content is already handled.

**Residual risk**: within a single active island, replacing attachments on each compaction (new island created, old island's entry evicted) means at most one generation of heavy attachments lives in memory at a time. Asserted by the integration bounded-growth test in Acceptance criteria.

#### Multi-worker coordination

Scout, H1/H2 Generator, Evaluator each call `runKodaX` with their own session (or session continuation). Each worker gets its own lineage. The migration does not change this — it only changes how the **parent** REPL's lineage is consumed, and how Scout's `initialMessages` is derived from the parent before handoff.

**Deliberately unchanged**: worker lineages are **not** spliced into the parent lineage. The parent consumes worker output via `continueFromSessionId` / `ManagedWorkerSessionStorage`, exactly as today. Whether parent should eventually splice worker lineages (enabling e.g. `/fork` to a mid-worker node from the parent REPL) is an adjacent design question — listed in Open questions, not undertaken here.

**Verification focus**: H2 Generator resumes Scout's session via `continueFromSessionId`. That path loads from `ManagedWorkerSessionStorage`, which calls `getSessionMessagesFromLineage` under the hood. Behaviour should be transparent — confirmed by a Phase C regression test rather than assumed.

#### Provider resilience

If the summary LLM persistently fails for a user's provider configuration, the v0.7.19 circuit breaker + graceful degradation remains the safety net. This feature does not change the recovery path — it only changes the steady-state shape of post-compact context.

#### Testing strategy

- **Unit**: new `session-lineage.test.ts` cases covering derivation of attachments, multi-round compaction (old attachments dropped in new island), session load with missing field.
- **Integration**: simulate the "180k monotonic growth" scenario from v0.7.18 with fault-injected summary provider — assert bounded steady-state growth.
- **Regression**: full `packages/agent`, `packages/coding`, `packages/repl` suites must stay green across every phase.
- **Manual**: multi-worker AMA run with intentional summary failures; `/fork` + `/rewind` across a compaction boundary; `/load-session` on a v0.7.19-persisted session.

### Open questions

1. **Memoization of `getSessionMessagesFromLineage`.** Settled by Phase B ship criteria: if the 500-entry p95 < 1ms benchmark fails, add `(activeEntryId, entries.length)`-keyed memoization before B ships; otherwise defer. Path-walk + flatMap is cheap for typical session sizes and `cloneMessage` is identity (post-migration still holds for the non-microcompact path), so a naive cache may be premature.
2. **`scope` field shape.** Settled: optional with absence meaning `'parent'`, populated by `runKodaX` when callers pass `scope: 'worker'` in options. Not a breaking event change.
3. **Parent ↔ worker lineage splicing.** Should parent REPL eventually splice worker lineages into its own tree (enabling `/fork` / `/rewind` across a worker-execution node from the parent)? Not in this feature's scope. Today worker lineages live in `ManagedWorkerSessionStorage` and the parent references worker outputs only through `continueFromSessionId`. Any change to that model should be a standalone feature; changing it as a side-effect of this migration would enlarge blast radius without a clear user benefit.
4. **Interaction with FEATURE_059 (Managed Task Structured Protocol V2) in `v0.8.0`.** Adjacent surfaces but shouldn't conflict; protocol v2 is about handoff payload shape, not compaction semantics.
5. **copy-style island model vs reference-style (pi-mono).** This feature keeps copy-style — `CompactionEntry` with `parentId: null` creates a new island, `evictOldIslandMessageContent` cleans old islands. ## FEATURE_074: Subagent Permission Boundary Hardening

**Status**: Completed (v0.7.20 unreleased — code merged at commit `458f333`)
**Priority**: High
**Category**: Internal / Bug fix + small refactor
**Builds on**: v0.7.16 `FEATURE_061` (AMA Scout-first), v0.7.18 `FEATURE_067` (child-executor)

> Implementation note: `set_permission_mode` is not yet deleted — it stays in the registry but the callback is no longer forwarded to `KodaXToolExecutionContext` ([agent.ts:1504-1509](../../packages/coding/src/agent.ts#L1504-L1509)), so any invocation fails closed. Full deletion (registry, prompt, command) is deferred to v0.7.21+ per the in-code note.

### Origin — observed misbehaviour

In v0.7.19, two distinct user reports show the same underlying defect:

1. A `Generator` child agent in `auto-in-project` mode silently called `set_permission_mode`. The call failed instantly with `Permission mode switching not available (callback not provided)` — no UI prompt, no recovery path, just a useless error in the trace.
2. A `Scout` child agent running while the parent REPL was in **plan mode** also called `set_permission_mode("accept-edits")` to "switch to implementation mode". Same silent failure.

Two separate issues are visible here:

- **Tool leak**: `set_permission_mode` is registered to children even though the callback that backs it is only wired at REPL command level ([repl.ts:675-681](../../packages/repl/src/interactive/repl.ts#L675-L681)). The child's `KodaXToolExecutionContext` ([agent.ts:~1497-1528](../../packages/coding/src/agent.ts#L1497-L1528)) never receives the callback, so the call is structurally guaranteed to fail.
- **Plan mode does not propagate to children**. Parent's `beforeToolExecute` enforces plan mode via `getPlanModeBlockReason` ([repl.ts:840-845](../../packages/repl/src/interactive/repl.ts#L840-L845)), but child agents have their own `runKodaX` loop ([child-executor.ts:137,202](../../packages/coding/src/child-executor.ts#L137)) and their own `beforeToolExecute` ([child-executor.ts:421-426](../../packages/coding/src/child-executor.ts#L421-L426)) that only blocks `CHILD_BLOCKED_TOOLS`. So a child running under a plan-mode parent can freely write files, run bash, and try to switch permissions.

The second issue is the load-bearing one: it means **the plan-mode contract is broken at the subagent boundary**. A user who enters plan mode expects "no side effects until I approve" — children silently violate that expectation.

A further observation during design review: users report **frequently toggling `plan ↔ accept-edits` mid-stream**. A snapshot-at-spawn design would freeze the first mode value and ignore the user's later intent — wrong on both directions (plan → accept-edits leaves children over-restricted; accept-edits → plan leaves children writing files after the user says stop). The implementation therefore uses a **live predicate closure**, evaluated at each child tool call.

### Goals

1. **Eliminate the tool leak.** `set_permission_mode` (pending deletion) and `exit_plan_mode` must never be available to a child agent's LLM. Belt-and-suspenders: also block at runtime.
2. **Propagate plan mode to children, live.** Whenever the parent is in plan mode, child tool calls are subject to `getPlanModeBlockReason`. Predicate is evaluated **at each child tool call** via a closure over live parent state — mid-run `plan ↔ accept-edits` toggles propagate into in-flight children immediately. A child-appropriate block message tells the LLM to return findings to the parent rather than try to switch modes itself.
3. **Decouple plan-mode exit from permission-mode switching.** Plan-mode exit gets its own tool, `exit_plan_mode`, which takes the finalized plan and triggers a user-approval UI flow. Tri-state callback: tool errors when called outside plan mode instead of silently no-op'ing. `set_permission_mode` is **deleted entirely** — it has been broken since introduction (callback never plumbed into the tool execution context), so no user session or prompt depends on it functioning. Keeping a broken tool as a deprecated alias only wastes LLM tool-selection surface.
4. **Preserve all current SA + AMA observable behaviour.** Single-agent path is unchanged. Parent-side plan-mode flow looks identical except the LLM is now directed to `exit_plan_mode` and the plan is actually rendered in the confirmation dialog.

### Non-goals

- **Not a capability-manifest design.** Tool filtering plus a one-line system-prompt hint is sufficient; a structured manifest is over-engineering for this defect class.
- **Not adding a child-to-parent escalation protocol.** Children that hit a plan-mode block return their findings as text; the parent agent (or user) decides what to do next. A formal `ReturnWithEscalationRequest` channel is deferred until there is concrete evidence it's needed.
- **Not changing parent-side permission semantics.** `accept-edits`, `default`, and `plan` mode behaviour at the parent is unchanged; only the *propagation* is added.
- **Not changing `KodaXToolExecutionContext` for non-permission reasons.** Only `exitPlanMode` callback is added; `setPermissionMode` callback shape is unchanged.

### Design

#### Step 1 — tool filtering (defence in depth)

Add `set_permission_mode` and `exit_plan_mode` to the child exclude list. Unify `CHILD_BLOCKED_TOOLS` with the exclude base so the two lists cannot drift again.

```ts
// packages/coding/src/child-executor.ts
const CHILD_EXCLUDE_TOOLS_BASE: readonly string[] = [
  'emit_managed_protocol',
  'dispatch_child_task',
  'ask_user_question',
  'worktree_create',
  'worktree_remove',
  'set_permission_mode',  // NEW: parent-only
  'exit_plan_mode',        // NEW: parent-only (introduced in Step 3)
];

const CHILD_BLOCKED_TOOLS = new Set<string>(CHILD_EXCLUDE_TOOLS_BASE);
```

Result: child LLM never sees the tool definition; if a future change accidentally re-registers it, runtime block kicks in.

#### Step 2 — plan mode propagation (live, not snapshot)

Plan-mode enforcement (`getPlanModeBlockReason`) lives in `packages/repl/src/permission/permission.ts`. `packages/coding` cannot depend on `repl` (reverse dep). The clean fix is **dependency injection of a live predicate closure**:

```ts
// packages/coding/src/child-executor.ts
type PlanModeBlockCheck = (tool: string, input: Record<string, unknown>) => string | null;

interface ChildExecutorOptions {
  // ...existing
  planModeBlockCheck?: PlanModeBlockCheck;
}

function buildChildEvents(
  childId: string,
  onProgress?: (status: string) => void,
  planModeBlockCheck?: PlanModeBlockCheck,
): KodaXEvents {
  return {
    beforeToolExecute: async (tool, input) => {
      if (CHILD_BLOCKED_TOOLS.has(tool)) {
        return `[Tool Error] ${tool}: Not available in child agent context.`;
      }
      if (planModeBlockCheck) {
        const reason = planModeBlockCheck(tool, input);
        if (reason) {
          return `${reason} You are a child agent inheriting plan-mode constraints. Complete investigation and return findings as text. The parent agent will request user approval for any implementation.`;
        }
      }
      return true;
    },
    // ...rest unchanged
  };
}
```

**Live semantics, not snapshot.** The predicate closes over parent state (e.g., a React ref or closure variable) and reads the parent's current mode at **each child tool call**, not at spawn. This is load-bearing:

- Users frequently toggle `plan ↔ accept-edits` mid-stream (observed in practice).
- A snapshot at spawn time would freeze the first value and ignore the user's later intent. User flips to accept-edits mid-run but in-flight children remain stuck in plan mode — contradicts "I changed my mind."
- Going the other way (spawn in accept-edits, user toggles to plan), a snapshot would let in-flight children keep writing after the user said stop. Worse.
- Live predicate = user's current intent always wins. Simpler too: one field, no spawn-time accounting.

REPL-side wiring: the REPL populates `ctx.planModeBlockCheck` with a closure over live `currentPermissionMode` / `permissionModeRef`. `dispatch_child_task` forwards `ctx.planModeBlockCheck` into `runChildAgent`.

#### Step 3 — independent `exit_plan_mode` tool, delete `set_permission_mode`

Previously, `set_permission_mode("accept-edits")` was nominally the way to leave plan mode. Two problems:

- It conflated "switch permission mode" (general) with "user approves the plan" (specific UX flow).
- Worse, it **never actually worked**: the `setPermissionMode` callback was declared on `KodaXEvents` but never plumbed into `KodaXToolExecutionContext`. Every LLM invocation returned `"Permission mode switching not available (callback not provided)"`.

Introduce a dedicated tool with a **tri-state callback** so misuse outside plan mode is surfaced as an explicit error rather than a silent no-op:

```ts
// packages/coding/src/tools/exit-plan-mode.ts
export async function toolExitPlanMode(input, ctx): Promise<string> {
  const plan = input.plan as string | undefined;
  if (!plan || typeof plan !== 'string' || plan.trim().length === 0) {
    return '[Tool Error] exit_plan_mode: Missing required parameter: plan …';
  }
  if (!ctx.exitPlanMode) {
    return '[Tool Error] exit_plan_mode: Only available in interactive REPL sessions …';
  }
  const outcome = await ctx.exitPlanMode(plan);
  if (outcome === 'not-in-plan-mode') {
    return '[Tool Error] exit_plan_mode: Not currently in plan mode. …';
  }
  return outcome === true
    ? JSON.stringify({ approved: true, note: '…proceed with implementation.' })
    : JSON.stringify({ approved: false, note: '…revise and propose again.' });
}
```

`KodaXToolExecutionContext.exitPlanMode: (plan: string) => Promise<boolean | 'not-in-plan-mode'>`. REPL behaviour:

1. If `currentPermissionMode !== 'plan'` → return `'not-in-plan-mode'` **without** showing a dialog. The tool surfaces this as an explicit error.
2. Otherwise render the plan via `buildToolConfirmationDisplay`'s new `exit_plan_mode` case (splits plan into detail lines so long plans display correctly).
3. On approval: flip `currentPermissionMode` to `accept-edits`, return `true`.
4. On rejection: leave mode as `plan`, return `false`.

The plan-mode block message in `repl.ts` and `InkREPL.tsx` is updated to direct the LLM to `exit_plan_mode`.

**`set_permission_mode` is deleted entirely.** Since the tool had never functioned end-to-end, no user session or prompt depends on it. A deprecation window would only waste LLM tool-selection surface — a broken tool in the list is a strictly worse than no tool. Removal is safe.

### Scope-adjacent bug fix — system temp as safe scratchpad

While testing `exit_plan_mode`, a longstanding semantic inconsistency surfaced in `isAlwaysConfirmPath` ([permission.ts:317](../../packages/repl/src/permission/permission.ts#L317)): **any** path outside the project root was flagged as "always confirm", regardless of mode. Consequences:

- `accept-edits` mode forced a confirmation dialog for writes to `$TMP/foo.txt`, contradicting the mode's intent.

## FEATURE_035: MCP Capability Provider (Historical staging draft; moved to `v0.8.0`)

**Status**: Planned
**Priority**: High
**Category**: New

> Migration note:
> The current source of truth for `FEATURE_035` now lives in [v0.8.0](./v0.8.0.md#feature_035-mcp-capability-provider).
> This section is retained as historical staging context only.

### Overview

`FEATURE_035` 在 `v0.7.20` 被前移，并重新明确成一件事：

KodaX 要通过共享 capability runtime 接入 MCP，而不是把 MCP 做成一套额外的重产品表面。

这意味着 MCP 在 KodaX 中的正确位置不是：

- REPL-only bridge
- 一组临时拼装的 markdown 索引
- 把所有远程能力硬塞成 built-in core tools

它的正确位置是：

- session/runtime owned capability broker
- 渐进式披露
- search / describe / execute / read / prompt 的统一 capability transport
- 和 evidence / provenance / permission mode 对齐

### Product philosophy

#### 1. Progressive disclosure, not product bloat

用户不应该先学一整套 `/mcp` 管理语义，系统应优先按任务需要暴露能力。

#### 2. Runtime-owned, not REPL-owned

MCP 不能只存在于交互入口；它必须是 `runKodaX()` 可复用的 runtime substrate。

#### 3. Catalog first, tool mutation second

MCP 原生有 `tool / resource / prompt` 三种能力族，不应被压平成一个“工具列表 markdown”。

#### 4. Evidence-carrying results

MCP 结果必须能进入 KodaX 的 evidence flow，而不是过早压成纯文本。

#### 5. Minimal surface, explicit truth

KodaX 可以提供少量 debug/status 入口，但不应复制一整套重命令、重运维、重 marketplace 的产品面。

### Goals

1. 通过共享 capability runtime 暴露 MCP 能力。
2. 保留 `tool / resource / prompt` 的原生区分。
3. 支持 lazy discovery、typed describe、structured execution result。
4. 让 Scout / retrieval / execution 都能消费 MCP 能力。
5. 把 provenance、trust、permission 边界做成一等公民。

### Scope

- MCP capability broker
- session/runtime-owned MCP provider lifecycle
- capability catalog cache and refresh
- `search / describe / execute / read / getPrompt`
- lazy / prewarm / disabled connect semantics
- provenance, diagnostics, and failure visibility
- permission-mode-aware risk / trust mapping
- evidence-friendly result shape

### Non-goals

- 不把每个 MCP server 都做成一个用户要手动操作的产品子系统
- 不把所有 MCP capability 自动提升成 core built-in tools
- 不引入 marketplace / autoupdate / policy-governance 级生态层
- 不为了 MCP 重新发明一套独立 runtime

### Relationship to existing features

- `FEATURE_034` 提供 extension / capability runtime substrate
- `FEATURE_028` 提供 retrieval / evidence 工具边界
- `FEATURE_029` 提供 provider capability truth 和 disclosure 思路

### Design direction

#### Runtime shape

推荐形态：

- MCP provider 作为 capability provider 注册到共享 runtime
- runtime 负责 provider 生命周期、catalog、diagnostics、session state
- REPL 最多只提供观测入口，不拥有核心逻辑

#### Model-facing surface

低风险 rollout 可以允许一个很小的 MCP-prefixed capability family：

- `mcp_search`
- `mcp_describe`
- `mcp_call`
- `mcp_read_resource`

但这些只是 model-facing façade；真正的 source of truth 仍然是 capability runtime。

#### Catalog and refresh

- 使用 JSON catalog 作为 source of truth
- 支持 pagination 与 `list_changed` 语义
- 支持 lazy refresh 与 unhealthy reconnect
- markdown 只作为 debug/export，而不是 discovery protocol

#### Permission and trust

- capability risk 至少区分 `read / write / network / exec`
- server annotations 只能作为 hint，不能当安全边界
- destructive / open-world / network / exec-like capability 必须抬高确认级别
- trust 必须与 KodaX permission mode 对齐，而不是 MCP 私有一套逻辑

### Acceptance criteria

- [ ] MCP capabilities can be loaded through the shared capability runtime
- [ ] KodaX preserves `tool / resource / prompt` as first-class capability kinds
- [ ] capability provenance and failures are visible in runtime diagnostics
- [ ] Scout and evidence-gathering flows can query and rank MCP-backed capabilities
- [ ] capability results preserve `structuredContent` / `evidence` / `artifacts` when available
- [ ] MCP does not require mutating core built-in tool truth to exist

## FEATURE_046: AMA Handoff Integrity and Final-Answer Convergence (Historical staging draft; moved to `v0.8.0`)

**Status**: Planned
**Priority**: High
**Category**: Enhancement

> Migration note:
> The current source of truth for `FEATURE_046` now lives in [v0.8.0](./v0.8.0.md#feature_046-ama-handoff-integrity-and-final-answer-convergence).
> This section is retained as historical staging context only.

### Overview

KodaX 当前 AMA 的主要问题，不是“角色不够多”，而是：

- worker-to-worker handoff 仍然偏摘要驱动
- 长文本 review / investigation 容易在 handoff 中丢失关键证据
- terminal worker 常常像“评论上游输出”，而不是收敛成最终答案
- non-terminal worker 的可见输出和可检查产物还不够稳定

`FEATURE_046` 的目标是把 AMA 重新钉回正确语义：

- `H1` 继续保持 lightweight checked-direct
- `H2` 继续保持 `Planner -> Generator <-> Evaluator`
- 但 evidence flow、handoff payload、final-answer contract 必须变得可信

### Product philosophy

#### 1. 不靠加角色解决问题

不回到 `H3`，也不引入新的 user-visible team graph。

#### 2. Display summary and downstream payload are different things

给用户看的摘要，不应兼任下游 worker 的真实输入。

#### 3. Final answer must converge

Evaluator 的职责是验证并收敛，而不是把 AMA 变成“生成器写一份、评审器再点评一份”。

#### 4. Evidence before confidence

下游判断必须尽量基于 artifact-backed upstream output，而不是 400 字截断摘要。

### Goals

1. 让 worker handoff 变成 artifact-backed、非损耗式。
2. 明确终端 worker 的 final-answer contract。
3. 保留 non-terminal worker 的可检查输出与 trace。
4. 降低 AMA review/investigation 的自我打架感。
5. 在不增加产品 ceremony 的前提下提高 correctness。

### Scope

- dependency handoff payload upgrade
- display summary 与 downstream payload 分离
- final-answer convergence contract
- non-terminal transcript persistence
- review / investigation flow 的 artifact-backed evaluator input
- same-role summary 与 full evidence 的职责拆分

### Non-goals

- 不增加新 harness level
- 不把用户重新拉回可见多角色拓扑
- 不默认增加更多 rounds 或更重 ceremony

### Design direction

#### Handoff contract

handoff 至少要区分三层：

- `display summary`: 状态栏 / trace / 简短概况
- `dependency artifact`: 给下游 worker 的真实载荷
- `role memory summary`: 给同角色跨轮 continuation 的低成本种子

三者不能继续混用。

#### Long-form upstream output

对于 review、investigation、planning 等长文本流：

- summary 可以短
- downstream payload 必须读 artifact / structured result / full finding bundle
- evaluator 不得只靠截断摘要做验收

#### Final-answer ownership

对于 review / investigation：

- terminal worker 输出必须是“合并后的最终答案”
- 不应再以“我评估了 generator 的结果”作为用户主输出
- 如果 evaluator 改写了结论，必须能回指对应 evidence

### Acceptance criteria

- [ ] downstream workers can consume artifact-backed upstream output rather than summary-only handoff
- [ ] long-form review and investigation flows no longer rely on clipped summaries for verdicts
- [ ] non-terminal worker output remains inspectable after task completion
- [ ] terminal worker outputs a converged final answer rather than commentary about another worker
- [ ] same-role summaries remain lightweight, but no longer substitute for full evidence payloads

## FEATURE_047: Invisible Adaptive Parallelism and Evidence-Driven Fan-Out (Historical staging draft; moved to `v0.8.0`)

**Status**: Planned
**Priority**: High
**Category**: Enhancement

> Migration note:
> The current source of truth for `FEATURE_047` now lives in [v0.8.0](./v0.8.0.md#feature_047-invisible-adaptive-parallelism-and-evidence-driven-fan-out).
> This section is retained as historical staging context only.

### Overview

KodaX 已经有 orchestration plumbing，也已经明确 `--team` 不是未来主故事。

下一步的正确方向不是把 `--team` 做漂亮，而是把并行变成一种：

- 用户几乎不可见
- runtime 自己选择
- 只在高收益节点触发
- 可取消、可回落、可追踪

的内部能力。

### Product philosophy

#### 1. Parallelism is an optimization, not a mode

用户不应该被迫理解 worker topology 才能获得并行收益。

#### 2. Default to single-agent economics

如果并行没有明显价值，就保持单路执行。

#### 3. Fan out evidence, not ceremony

优先把并行用于证据采集、验证分片、测试分片、假设竞争，而不是角色表演。

#### 4. Parallelism must be bounded

必须同时受 dependency graph、risk、budget、provider capability、task family 约束。

### Goals

1. 让 orchestration 在合适场景下支持真实并行。
2. 让并行主要服务 evidence acquisition 和 validation quality。
3. 保持 `SA / AMA` 用户语义不变，不重新引入 team mode。
4. 支持 speculative branch 和及时 cancellation。
5. 让并行结果能稳定合并进 artifact / verdict flow。

### Scope

- selected orchestration batches with `maxParallel > 1`
- dependency-aware worker fan-out
- evidence shard / validation shard / hypothesis shard patterns
- speculative sub-branches with cancellation
- artifact merge and verdict reduction
- trace visibility without new user-facing topology UI

### Non-goals

- 不恢复 `--team` 为正式产品模式
- 不让用户手动配置 worker graph
- 不对简单任务默认启用 swarm-like execution
- 不以并行名义扩大 prompt ceremony

### Design direction

#### Good parallel targets

优先支持：

- changed-scope / diff / module evidence 分片
- 测试 / lint / verification 分片
- 多个独立假设的快速竞争
- evaluator 的 spot-check shards

#### Bad parallel targets

避免默认并行：

- 强顺序依赖的 edit pipeline
- 用户明显想边走边控的任务
- 低价值的“多角色互评”

#### Scheduler policy

调度器至少应考虑：

- task family
- mutation surface
- dependency DAG
- evidence gap
- current budget and predicted gain
- provider capability profile

### Acceptance criteria

- [ ] managed-task orchestration can run selected worker batches in parallel when the scheduler judges it beneficial
- [ ] user-facing UX does not expose a revived team mode or worker topology by default
- [ ] speculative parallel branches can be cancelled cleanly when one path wins early
- [ ] parallel outputs can be merged into artifact-backed verdicts without lossy handoff
- [ ] read-only / docs-only routing ceilings remain intact even when internal parallelism exists
