## Version Context

v0.7.19 是 v0.7.18 FEATURE_067 (Child Agent Execution) 的自然延续。

FEATURE_067 在 v0.7.18 实现了 child agent 的完整执行层：
- child-executor (read-only + write fan-out)
- 结构化简报 + 锚定增量合并
- Worktree 隔离 + evaluator-assisted 合并
- AbortSignal 传播 + partial result 保留

v0.7.19 补齐**调度策略**和**边界防护**两个优化层，使 child agent 系统从"能用"变为"智能"。

## FEATURE_047: Invisible Adaptive Parallelism — Fan-Out Optimization

**Status**: Planned (v0.7.19)
**Priority**: High
**Category**: Optimization
**Builds on**: FEATURE_067 (v0.7.18 — child-executor 执行层)
**Original design**: [v0.8.0](v0.8.0.md#feature_047-invisible-adaptive-parallelism-and-evidence-driven-fan-out) (架构 close-out 2026-04-03)

### Overview

FEATURE_067 让 Scout/Generator 可以声明 childBundles 并行执行。但"何时该扇出"的决策完全依赖 LLM 判断——没有系统级的质量控制。FEATURE_047 增加三个优化层：

1. **调度策略** — 系统评估 LLM 声明的 fan-out 是否值得（避免不必要的并行开销）
2. **Winner-cancel 投机取消** — evidence-scan 场景中一个 child 找到答案即取消其他
3. **质量度量** — 追踪 fan-out 是否真的提升了结果质量和效率

### Goals

1. 调度器基于 mutation surface、evidence gap、budget 余量评估 fan-out 收益
2. 支持 `winner-cancel` 取消策略（read-only evidence-scan 场景）
3. 支持 `confidence-threshold` 取消策略（hypothesis-check 场景）
4. 追踪 fan-out 质量指标：turnaround delta、evidence coverage、token overhead

### Non-goals

- 不改变 FEATURE_067 的执行层 API
- 不改变用户可见的 SA/AMA 模式
- 不实现手动调度配置（系统自动决定）

### Design

#### 047-A: Scheduler Policy Layer

在 `executeChildAgents()` 调用前插入一个评估步骤：

```typescript
// packages/coding/src/child-scheduler.ts

interface FanoutWorthinessSignals {
  readonly fanoutClass: KodaXAmaFanoutClass;
  readonly bundleCount: number;
  readonly mutationSurface: 'read-only' | 'disjoint-write';
  readonly remainingBudget: number;
  readonly estimatedChildCost: number;  // 预估每个 child 消耗
  readonly evidenceGap: number;         // 0-1, 已有证据覆盖率
}

interface FanoutDecision {
  readonly approved: boolean;
  readonly reason: string;
  readonly adjustedMaxParallel?: number;  // 可能降低并发度
  readonly cancellationPolicy: 'none' | 'winner-cancel' | 'confidence-threshold';
}

function evaluateFanoutWorthiness(
  signals: FanoutWorthinessSignals,
): FanoutDecision;
```

决策规则（数据驱动，非硬编码）：
- `bundleCount === 1` → 不值得 fan-out（直接顺序执行）
- `estimatedChildCost * bundleCount > remainingBudget * 0.5` → budget 不够，降低 maxParallel
- `evidenceGap < 0.2` → 已有足够证据，不需要 fan-out
- `mutationSurface === 'read-only'` → 低风险，宽松批准
- `mutationSurface === 'disjoint-write'` → 高开销，需要更强信号

#### 047-B: Winner-Cancel Speculation

扩展 `child-executor.ts` 的 AbortSignal 机制：

```typescript
interface CancellationController {
  readonly policy: 'none' | 'winner-cancel' | 'confidence-threshold';
  readonly abortController: AbortController;
  onChildComplete(childId: string, result: KodaXChildAgentResult): void;
}
```

- `winner-cancel`: 第一个 child 返回 `status: 'ready'` + 非空 evidence → 取消其他 child
- `confidence-threshold`: child 返回高置信度结果 (evidence.length >= threshold) → 取消其他
- `none`: 所有 child 完成 (FEATURE_067 默认行为，不变)

#### 047-C: Quality Metrics

```typescript
interface FanoutQualityMetrics {
  readonly turnaroundDeltaMs: number;     // 并行 vs 预估顺序时间
  readonly evidenceCoverage: number;      // 0-1, 合并后证据覆盖率
  readonly tokenOverhead: number;         // fan-out 额外 token 消耗
  readonly childSuccessRate: number;      // 成功 child / 总 child
  readonly cancelledCount: number;        // 被取消的 child 数
}
```

通过 ACP events 发射，用于后续调优调度策略参数。

## FEATURE_052: Child Fan-Out Boundary Hardening

**Status**: Planned (v0.7.19)
**Priority**: Medium
**Category**: Optimization
**Builds on**: FEATURE_067 (v0.7.18), FEATURE_047 (v0.7.19)
**Original design**: [v0.8.0](v0.8.0.md#feature_052-dual-profile-ama-harness-and-child-fan-out-boundaries) (架构 close-out 2026-04-03)

### Overview

FEATURE_067 实现了 child 执行但边界防护较弱。FEATURE_052 加固边界：

1. **SA 严格模式** — SA 模式下禁止任何隐藏 child fan-out
2. **递归防护** — child 不能再 spawn child (防止无限递归)
3. **角色-Harness 矩阵强制** — 系统级保证只有合法的角色+Harness 组合才能 fan-out

### Goals

1. SA 模式下 `childBundles` 声明被静默忽略（不报错，保持极简体验）
2. Child agent 的工具集中不包含 `emit_managed_protocol` 的 childBundles 能力（递归防护）
3. 非法角色-Harness 组合的 write fan-out 被系统拒绝（如 H0 Scout 的 write bundle）
4. 边界违规通过 ACP events 记录（可审计）

### Non-goals

- 不修改 SA/AMA 的用户切换体验
- 不引入新的用户可见模式
- 不实现 child 之间的通信（保持完全隔离）

### Design

#### 052-A: SA Enforcement

```typescript
// child-executor.ts — executeChildAgents() 入口增加检查
if (currentMode === 'SA') {
  // SA 模式静默跳过 fan-out, 不影响主流程
  emitAcpEvent('fanout_skipped', { reason: 'sa_mode' });
  return { results: [], mergedFindings: [], ... };
}
```

#### 052-B: Recursive Spawn Guard

```typescript
// child-executor.ts — buildChildBriefing() 中
// Child 的工具集排除 emit_managed_protocol 的 childBundles 声明能力
// 实现方式: child 的 system prompt 中明确声明 "你不能声明 childBundles"
// + coerce 层在 child context 中过滤 childBundles 字段
```

#### 052-C: Role-Harness Matrix Enforcement

```typescript
const WRITE_FANOUT_ALLOWED: ReadonlyMap<string, readonly string[]> = new Map([
  ['H2_PLAN_EXECUTE_EVAL', ['generator']],
  // H0, H1: 没有 write fan-out 权限
]);

function validateWriteFanout(
  harness: string,
  role: string,
  bundles: KodaXChildContextBundle[],
): { allowed: KodaXChildContextBundle[]; rejected: KodaXChildContextBundle[] };
```
