## FEATURE_021: Provider-Aware Reasoning Budget Matrix

**Status**: Completed | **Priority**: High | **Created**: 2026-03-15

### 1. 概述

把当前统一的 `low=6000 / medium=12000 / high=20000` 推理预算策略升级为按 **provider + task type** 细分的默认矩阵，让 KodaX 的 reasoning 默认值更符合真实 provider 能力和任务场景。

### 2. 背景

当前系统已经具备统一的 `off / auto / quick / balanced / deep` 用户模式，但默认预算仍然主要依赖一套全局常量。

这会带来几个问题：
- `deep=20000` 更像保守工程默认值，而不是按 provider 能力校准过的值
- `openai` 走 `reasoning_effort`，不适合和 `budget_tokens` 用同一心智模型理解
- `zhipu-coding`、`anthropic`、`prompt-only` provider 的能力边界不同，却共享相近的默认映射
- review / bugfix / plan / refactor 等任务的默认强度差异还不够系统化

### 3. 目标

1. 为不同 provider 建立独立的默认推理预算或强度映射
2. 为不同 task type 建立默认深度/预算覆盖表
3. 明确 `native-budget` / `native-effort` / `native-toggle` / `prompt-only` 的差异化策略
4. 让 `deep`、`balanced` 等模式的默认值更可解释、更可调
5. 在配置模板和帮助信息中说明默认行为

### 4. 设计范围

#### 4.1 Provider 维度

首版至少覆盖：
- `anthropic`
- `openai`
- `zhipu-coding`
- `zhipu`
- `kimi-code`
- `minimax-coding`

每个 provider 需要定义：
- 默认能力类型
- 默认高/中/低推理映射
- provider cap
- 不支持原生推理时的降级策略

#### 4.2 Task Type 维度

首版按下列任务类型提供默认覆盖：
- `review`
- `bugfix`
- `edit`
- `refactor`
- `plan`
- `qa`
- `unknown`

目标方向：
- `review` 默认更保守
- `bugfix / edit` 默认中等
- `plan / refactor` 默认更深
- `unknown` 保持安全但不过度保守

### 5. 实现思路

#### 5.1 AI 层

- 扩展 `packages/ai/src/reasoning.ts`
- 支持从 provider 配置中读取 provider 级默认映射
- 在现有 `defaultThinkingBudgets` 和 `taskBudgetOverrides` 之上，明确 provider-aware 的合并顺序

#### 5.2 Provider Registry

- 更新 `packages/ai/src/providers/registry.ts`
- 给各 provider 注入首版默认预算表或默认 effort 映射
- 避免所有 provider 机械共用 `6000 / 12000 / 20000`

#### 5.3 Coding 层

- 更新 `packages/coding/src/reasoning.ts`
- 让任务路由输出和 provider/task 默认表协同工作
- 在低置信场景下走保守但不失真的 fallback

#### 5.4 文档与配置

- 更新 `config.example.jsonc`
- 明确哪些 provider 使用 token budget、哪些使用 effort、哪些只做 toggle/prompt-only
- 说明默认值和可调参数的关系

### 6. 验收标准

- 不同 provider 的默认 reasoning 映射可以独立定义
- review / bugfix / plan / refactor 等任务能使用不同默认预算
- `deep=20000` 不再作为单一全局事实存在
- 配置模板和帮助文本能解释默认行为
- 有单元测试覆盖 provider/task budget 解析

### 7. 实施检查清单

- [ ] 设计 provider 级默认预算矩阵
- [ ] 设计 task type 级默认覆盖表
- [ ] 更新 `packages/ai/src/reasoning.ts`
- [ ] 更新 `packages/ai/src/providers/registry.ts`
- [ ] 更新 `packages/coding/src/reasoning.ts`
- [ ] 补充推理预算相关测试
- [ ] 更新配置模板和帮助文档

## FEATURE_013: Command System 2.0

**Status**: Completed | **Priority**: High | **Created**: 2026-03-03 | **Started**: 2026-03-13 | **Completed**: 2026-03-16

### 1. 概述

实现 KodaX Command System 2.0，聚焦于**用户级命令发现**和**通用UI交互系统**：
- ✅ `.md` 命令文件发现（`~/.kodax/commands/` + `.kodax/commands/`）
- ✅ 通用 UI 交互组件（可用于 commands/skills/LLM tools）
- ✅ LLM 可调用的用户交互工具
- ❌ **不实现** TypeScript 扩展、动态工具注册、事件钩子（过度工程化）

### 2. 背景

#### 当前问题

**核心问题**: KodaX 无法加载用户级命令文件（类似 Claude Code 的 `~/.claude/commands/`）

**影响**:
- 用户必须手动复制命令文件到每个项目
- 无法复用个人命令
- Skills 系统虽然可用，但不够轻量

#### 对标系统对比

| 特性 | KodaX (现状) | KodaX (目标) | pi-mono | 说明 |
|------|-------------|-------------|---------|------|
| 用户级命令发现 | ❌ | ✅ `.md` 文件 | ✅ `.md` + `.ts` | **核心需求** |
| 项目级命令发现 | ❌ | ✅ `.md` 文件 | ✅ `.md` + `.ts` | 项目定制命令 |
| UI 交互组件 | ❌ | ✅ 通用系统 | ✅ ExtensionUIContext | 多场景复用 |
| LLM 工具调用 | ❌ | ✅ `ask_user` 等 | ✅ 完整 API | **重要特性** |
| TypeScript 扩展 | ❌ | ❌ 不实现 | ✅ jiti 加载 | 过度工程化 |
| 动态工具注册 | ❌ | ❌ 不实现 | ✅ registerTool | 过度工程化 |
| 事件钩子系统 | ❌ | ❌ 不实现 | ✅ EventBus | 过度工程化 |

**设计哲学**: 极简主义 + LLM 智能
- 只实现 `.md` 命令发现（不需要 TypeScript 扩展）
- UI 交互组件通用化（commands/skills/LLM tools 都可用）
- 代码量约 475 行（对比 pi-mono 数千行）

### 3. 技术方案

#### 3.1 命令发现系统

**文件**: `packages/repl/src/commands/discovery.ts` (~120 行)

**扫描目录**:
- 用户级: `~/.kodax/commands/`
- 项目级: `.kodax/commands/` (相对于工作目录)

**命令文件格式** (`.md`):
```markdown
---
name: my-command
description: My custom command
aliases: [mc, mycmd]
priority: medium
---

# Command Content

Your command prompt content here...

Supports:
- Multi-line content
- Markdown formatting
- Dynamic context injection via !`command`
```

**核心接口**:
```typescript
interface DiscoveredCommand {
  name: string;
  description: string;
  aliases?: string[];
  priority?: CommandPriority;
  content: string;
  location: 'user' | 'project';
  path: string;

function discoverCommands(baseDirs: string[]): DiscoveredCommand[];
function parseCommandFile(filePath: string): DiscoveredCommand;
function registerDiscoveredCommands(
  commands: DiscoveredCommand[],
  registry: CommandRegistry

```

**设计原则**: UI 交互是通用功能，可用于：

#### 3.3 LLM 用户交互工具

**文件**: `packages/repl/src/tools/user-interaction.ts` (~100 行)

让 LLM 能主动向用户提问，注册 3 个工具：

```typescript
// Tool 1: ask_user - 多选题
registerTool({
  name: "ask_user",
  description: "Ask the user a question with multiple choice options",
  parameters: {
    question: string,
    options: string[]
  },
  async execute(params, ctx) {
    const answer = await ctx.ui.select(params.question, params.options);
    return answer
      ? { content: [{ type: "text", text: `User selected: ${answer}` }] }
      : { content: [{ type: "text", text: "User cancelled" }] };
  }
})

// Tool 2: confirm_action - 是/否确认
registerTool({
  name: "confirm_action",
  description: "Ask the user to confirm an action",
  parameters: { message: string },
  async execute(params, ctx) {
    const confirmed = await ctx.ui.confirm(params.message);
    return {
      content: [{
        type: "text",
        text: confirmed ? "User confirmed" : "User declined"
      }]
    };
  }
})

// Tool 3: get_input - 文本输入
registerTool({
  name: "get_input",
  description: "Ask the user for text input",
  parameters: {
    prompt: string,
    defaultValue?: string
  },
  async execute(params, ctx) {
    const value = await ctx.ui.input(params.prompt, params.defaultValue);
    return value
      ? { content: [{ type: "text", text: `User input: ${value}` }] }
      : { content: [{ type: "text", text: "User cancelled" }] };
  }
})
```

#### 6.2 命令中使用 UI 交互

```typescript
// /delete-session 命令
registry.register({
  name: "delete-session",
  description: "Delete a session",
  handler: async (args, ctx, callbacks) => {
    const sessions = await callbacks.listSessions();
    const selected = await callbacks.ui.select(
      "Select session to delete",
      sessions
    );

    if (selected) {
      const confirmed = await callbacks.ui.confirm(
        `Delete session "${selected}"?`
      );
      if (confirmed) {
        await callbacks.deleteSession(selected);
        return { success: true };
      }
    }
    return { success: false, message: "Cancelled" };
  }
});
```

## FEATURE_015: Project Mode 2.0 - AI-Driven Development Workflow

**Status**: Completed | **Priority**: High | **Created**: 2026-03-07 | **Started**: 2026-03-17 | **Completed**: 2026-03-17

### 1. 概述

将 Project 模式打造为 KodaX 的核心特色，提供完整的 AI-Driven 开发工作流，使其成为区别于其他 Coding Agent 的标志性功能。

### 1.1 当前已实现切片（2026-03-17）

- `/project quality` 已可生成确定性的 workflow health / release readiness 报告，并在可用时追加 AI 质量点评。
- `/project status "<question>"` 已从占位文案升级为真正的 guided analysis，会结合 feature list、PROGRESS.md、session plan 和质量报告回答问题。
- `/project brainstorm "<topic>"` 已支持启动、继续和结束多轮头脑风暴；讨论历史会持久化到 `.kodax/projects/<session>/brainstorm/`，并支持无模型时的 fallback facilitator。
- `/project plan [#index|topic]` 已支持生成结构化实施计划，并把最新计划写入 `.kodax/session_plan.md`，供 `/project status` 和 `/project quality` 复用。
- 当前实现落点：
  - `packages/repl/src/interactive/project-brainstorm.ts`
  - `packages/repl/src/interactive/project-planner.ts`
  - `packages/repl/src/interactive/project-quality.ts`
  - `packages/repl/src/interactive/project-commands.ts`
  - `packages/repl/src/interactive/project-storage.ts`
  - `packages/repl/src/interactive/project-brainstorm.test.ts`
  - `packages/repl/src/interactive/project-planner.test.ts`
  - `packages/repl/src/interactive/project-storage.test.ts`
  - `packages/repl/src/interactive/project-quality.test.ts`
  - `packages/repl/src/interactive/project-commands.test.ts`
- `v0.6.0` 版本将本 feature 的完成标准收敛为“核心 workflow 完整可用”：
  - brainstorm
  - plan
  - execution (`/project next`, `/project auto`, `/project pause`, `/project progress`)
  - quality (`/project quality`, `/project status`, `/project analyze`)
- Graphviz/DAG、brainstorm 直接生成 feature、自动回退编排等更重增强项后移到后续 feature，避免把 `0.6.0` 卡在扩展项上。

### 2. 核心能力

#### 2.1 多轮头脑风暴 (Brainstorming Mode)

**命令**: `/project brainstorm "需求描述"`

**功能**:
- AI 自动提出探索性问题
- 质疑假设，挖掘隐藏需求
- 探索多种实现方案
- 生成决策树和方案对比表
- 记录完整讨论历史

**示例**:
```
/project brainstorm "需要实现一个用户权限系统"

AI: 让我们先明确几个关键问题：
1. 需要支持多少用户量级？
2. 权限粒度需要到什么级别？
3. 是否需要支持角色继承？
...
```

**实现**:
- 创建 `packages/repl/src/project/brainstorm.ts`
- 使用 LLM 生成探索性问题
- 存储讨论历史到 `.kodax/projects/{id}/brainstorm/`
- 支持从讨论直接生成 feature

#### 2.2 规划模式增强 (Enhanced Planning)

**命令**: `/project plan #n` 或 `/project plan "feature描述"`

**功能**:
- 自动生成三级结构计划:
  - **Phases**: 主要阶段（设计、开发、测试、部署）
  - **Milestones**: 关键里程碑
  - **Tasks**: 具体任务
- 估算每个任务的时间（基于历史数据）
- 评估复杂度（简单/中等/复杂）
- 生成依赖关系图（DAG）
- 甘特图可视化

**示例输出**:
```
## Phase 1: 设计 (Est: 2h)
- [ ] Task 1.1: 需求分析 (30min, 简单)
- [ ] Task 1.2: 技术选型 (1h, 中等)
  - 依赖: Task 1.1
- [ ] Task 1.3: 接口设计 (30min, 简单)
  - 依赖: Task 1.2

## Phase 2: 开发 (Est: 8h)
...
```

**实现**:
- 创建 `packages/repl/src/project/planner.ts`
- 集成历史数据分析（类似任务的实际耗时）
- 使用 Graphviz 生成依赖关系图
- 存储到 `.kodax/projects/{id}/plans/{feature_id}.json`

#### 2.3 长时间运行支持 (Long-Running Execution)

**命令**:
- `/project execute #n` - 启动长时间任务
- `/project checkpoint` - 保存当前进度

## FEATURE_017: 运行时用户输入插队

**Status**: Completed | **Priority**: High | **Created**: 2026-03-11 | **Started**: 2026-03-17 | **Completed**: 2026-03-17

### 1. 概述

在 KodaX 运行时（LLM 回答、流式输出或自动工具运行过程中）能够接受用户输入，并将其插入到下一轮对话中作为最新提示。

### 2. 已实现范围

- 在支持插队的对话轮次中，输入框可在 streaming 期间继续接收 follow-up 文本，并以 FIFO 队列缓存。
- 新增 `PendingInputsIndicator` 和统一的 viewport 预算，确保队列提示不会把消息区最后一行挤掉。
- follow-up 会在当前轮结束后按顺序启动新的 agent round，而不是混进当前 round 的内部 iteration。
- 排队输入支持 `Esc` 撤销最后一条、`Ctrl+C` 中断时自动清空、最多保留 5 条待处理 follow-up。
- 通过 `StreamingContext` 和 `queued-prompt-sequence` 回归测试覆盖 FIFO、上限、清空和串行接力逻辑。

**对标产品**:
- Codex CLI - inline input during streaming
- Claude Code - streaming input queue

### 2. 用户场景

#### 3.3 Agent 集成

```typescript
// packages/coding/src/agent.ts

async function* runKodaX(...) {
  while (!shouldStop) {
    // 1. 执行一轮 LLM 调用
    const response = await llm.stream(...);
    yield* processResponse(response);

    // 2. 检查是否有排队的用户输入
    if (events.hasPendingInputs?.()) {
      const pendingInputs = events.consumePendingInputs?.();
      if (pendingInputs && pendingInputs.length > 0) {
        // 3. 将排队输入作为下一轮的用户消息
        for (const input of pendingInputs) {
          messages.push({ role: 'user', content: input });
        }
        // 清空队列
        events.clearPendingInputs?.();
        // 继续循环（不增加 iteration 计数）
        continue;
      }
    }

    // 4. 正常的 iteration 逻辑
    iteration++;
    if (iteration >= maxIterations) break;
  }
}
```

**Status**: Completed | **Priority**: High | **Created**: 2026-03-13 | **Started**: 2026-03-13 | **Completed**: 2026-03-13 | **Released**: v0.5.34

### 1. 概述

实现类似 pi-mono 的 `AGENTS.md` 机制，支持项目级别的 AI 上下文规则注入。允许项目在 `.kodax/AGENTS.md` 或项目根目录的 `AGENTS.md` 中定义项目特定的规则、命令、工作流程等，这些内容会自动注入到 AI 的系统提示中。

**对标产品**: [pi-mono AGENTS.md](https://github.com/badlogic/pi-mono)

### 2. 用户场景

```markdown
- No `any` types unless absolutely necessary
- All functions must have JSDoc comments
```

### 3. 文件加载规则

#### 3.1 优先级系统

**加载顺序** (从低到高):
1. **Global Rules** - `~/.kodax/AGENTS.md`
   - 适用于所有项目的默认规则
   - **只识别 `AGENTS.md`**，不支持回退到 `CLAUDE.md`

2. **Directory Rules** - 从文件系统根到当前目录的所有 `AGENTS.md` 或 `CLAUDE.md`
   - 按 root → ... → cwd 顺序加载
   - 后加载的覆盖先加载的
   - **支持回退**: `AGENTS.md` 优先，缺失时加载 `CLAUDE.md`

3. **Project Rules** - `{projectRoot}/.kodax/AGENTS.md`
   - 仅在明确指定 `projectRoot` 时加载
   - 具有最高优先级
   - **只识别 `AGENTS.md`**，不支持回退到 `CLAUDE.md`

#### 3.2 文件识别规则

| 位置 | AGENTS.md | CLAUDE.md | 说明 |
|------|-----------|-----------|------|
| `~/.kodax/` | ✅ 识别 | ❌ 不识别 | 全局配置只支持 AGENTS.md |
| `{cwd}/.kodax/` | ✅ 识别 | ❌ 不识别 | 项目级配置只支持 AGENTS.md |
| 普通目录 | ✅ 优先 | ✅ 回退 | AGENTS.md 不存在时才加载 CLAUDE.md |

**示例**:
```
# 目录结构
/project
├── AGENTS.md            # 被加载 (Directory Rules)
├── CLAUDE.md            # 被忽略 (AGENTS.md 存在)
├── src/
│   ├── AGENTS.md        # 被加载 (Directory Rules)
│   └── lib/
```

### 4. 技术方案

#### 3.1 文件发现机制

```typescript
// packages/coding/src/context/agents-loader.ts

const CONTEXT_FILE_CANDIDATES = ["AGENTS.md", "CLAUDE.md"];

interface AgentsFile {
  path: string;
  content: string;
  scope: 'global' | 'project' | 'directory';
}

interface LoadAgentsOptions {
  cwd?: string;                    // 当前工作目录
  kodaxDir?: string;                // KodaX 全局目录
  projectRoot?: string;              // 项目根目录（可选）
}

/**
 * 加载所有 AGENTS 文件
 * 优先级：全局 < 根目录 < ... < 当前目录 < 项目级(.kodax/)
 */
function loadAgentsFiles(options?: LoadAgentsOptions): AgentsFile[] {
  const resolvedCwd = resolve(options?.cwd ?? process.cwd());
  const resolvedKodaxDir = options?.kodaxDir ?? getKodaxGlobalDir();
  const resolvedProjectRoot = options?.projectRoot ? resolve(options.projectRoot) : null;

  const contextFiles: AgentsFile[] = [];
  const seenPaths = new Set<string>();

  // 1. 全局配置 (~/.kodax/AGENTS.md)
  //    注意: 全局目录只加载 AGENTS.md，不回退到 CLAUDE.md
  const globalContext = loadAgentsFile(resolvedKodaxDir, ["AGENTS.md"]);
  if (globalContext) {
    globalContext.scope = "global";
    contextFiles.push(globalContext);
    seenPaths.add(globalContext.path);
  }

  // 2. 从根目录到当前目录遍历（只加载目录级文件）
  //    注意: 目录级文件支持 "AGENTS.md 优先，缺失时回退到 CLAUDE.md"
  const directoryFiles: AgentsFile[] = [];
  let currentDir = resolvedCwd;
  const visitedDirs = new Set<string>();

  while (true) {
    if (visitedDirs.has(currentDir)) break; // 防止循环
    visitedDirs.add(currentDir);

    // 只加载目录级 AGENTS.md 或 CLAUDE.md
    const contextFile = loadContextFileFromDir(currentDir);
    if (contextFile && !seenPaths.has(contextFile.path)) {
      contextFile.scope = "directory";
      directoryFiles.unshift(contextFile); // 从根到叶子顺序添加
      seenPaths.add(contextFile.path);
    }

    // 检查是否到达根目录
    const parsed = parse(currentDir);
    if (parsed.root === currentDir) break;

    const parentDir = resolve(currentDir, "..");
    if (parentDir === currentDir) break;
    currentDir = parentDir;
  }

  contextFiles.push(...directoryFiles);

  // 3. 项目级配置（仅当明确指定 projectRoot 时）
  //    注意: 项目级只加载 AGENTS.md，不回退到 CLAUDE.md
  if (resolvedProjectRoot) {
    const projectContext = loadAgentsFile(
      join(resolvedProjectRoot, ".kodax"),
      ["AGENTS.md"]
    );
    if (projectContext && !seenPaths.has(projectContext.path)) {
      projectContext.scope = "project";
      contextFiles.push(projectContext);
    }
  }

  return contextFiles;
}

/**
 * 辅助函数：从目录加载指定的文件（按优先级尝试）
 */
function loadAgentsFile(dir: string, filenames: readonly string[]): AgentsFile | null {
  for (const filename of filenames) {
    const filePath = join(dir, filename);
    if (existsSync(filePath)) {
      try {
        return {
          path: filePath,
          content: readFileSync(filePath, "utf-8"),
          scope: "directory", // 默认值，会被调用者覆盖
        };
      } catch (error) {
        console.warn(`[kodax:agents] Could not read ${filePath}: ${error}`);
      }
    }
  }
  return null;
}

/**
 * 辅助函数：加载目录级上下文文件（AGENTS.md 优先）
 */
function loadContextFileFromDir(dir: string): AgentsFile | null {
  return loadAgentsFile(dir, CONTEXT_FILE_CANDIDATES);
}
```

```typescript
/**
 * 加载所有 AGENTS 文件
 * 优先级：全局 < 根目录 < ... < 当前目录 < .kodax/
 */
export function loadAgentsFiles(options?: LoadAgentsOptions): AgentsFile[] {
  const resolvedCwd = options?.cwd ?? process.cwd();
  const resolvedKodaxDir = options?.kodaxDir ?? getKodaxGlobalDir();

  const contextFiles: AgentsFile[] = [];
  const seenPaths = new Set<string>();

  // 1. 加载全局配置 (~/.kodax/AGENTS.md)
  const globalContext = loadContextFileFromDir(resolvedKodaxDir);
  if (globalContext) {
    globalContext.scope = 'global';
    contextFiles.push(globalContext);
    seenPaths.add(globalContext.path);
  }

  // 2. 从根目录到当前目录遍历
  const ancestorFiles: AgentsFile[] = [];
  let currentDir = resolvedCwd;
  const root = resolve("/");

  while (true) {
    // 先检查 .kodax/AGENTS.md（更高优先级）
    const kodaxContextFile = join(currentDir, ".kodax", "AGENTS.md");
    if (existsSync(kodaxContextFile) && !seenPaths.has(kodaxContextFile)) {
      try {
        ancestorFiles.unshift({
          path: kodaxContextFile,
          content: readFileSync(kodaxContextFile, "utf-8"),
          scope: 'project'
        });
        seenPaths.add(kodaxContextFile);
      } catch (error) {
        console.warn(`Warning: Could not read ${kodaxContextFile}: ${error}`);
      }
    }

    // 再检查目录级 AGENTS.md 或 CLAUDE.md
    const contextFile = loadContextFileFromDir(currentDir);
    if (contextFile && !seenPaths.has(contextFile.path)) {
      contextFile.scope = 'directory';
      ancestorFiles.unshift(contextFile);
      seenPaths.add(contextFile.path);
    }

    if (currentDir === root) break;

    const parentDir = resolve(currentDir, "..");
    if (parentDir === currentDir) break;
    currentDir = parentDir;
  }

  contextFiles.push(...ancestorFiles);
  return contextFiles;
}

      'global': '全局规则',
      'project': '项目规则',
      'directory': '目录规则'
    }[file.scope];

    return `
## ${scopeLabel} (from ${file.path})

${file.content}
`;
  }).join('\n---\n');

  return `
---

# Project Context

${contextSections}
`;
}
```

### 1. 概述

为 KodaX 增加一层面向整体产品能力的多智能体调度基础设施，让系统能够在单次任务中安全地组织 supervisor/worker 式协作，而不是只依赖单智能体串行执行。这个能力会服务 KodaX 的整体执行模型、长任务工作流和未来的能力扩展，而不是只服务 builtin skills。

### 2. 背景

KodaX 当前已经具备一些子智能体原语，包括 skill frontmatter 中的 `context: fork`、`agent`、tool policy 和 invocation hooks。这说明底层并不是从零开始，但还缺少一层统一的调度协议来组织任务拆分、handoff、汇总和失败隔离。

如果直接把这些能力暴露成面向用户的“蜂群系统”，会很容易让产品心智变重，也会把很多未经验证的抽象提前固化。因此这项 feature 的目标不是先做一个炫的 swarm 概念，而是先做一层对 KodaX 整体能力有帮助的极简调度基础设施。

### 3. 目标

1. 提供 KodaX 原生的 supervisor/worker 执行模型与最小调度协议
2. 支持基于 workspace artifact 的任务拆分、handoff、汇总与复盘
3. 复用现有 `context: fork`、`agent`、hooks、tool policy 等原语，而不是另起一套重平台
4. 保持极简产品心智，让“多智能体”更多是内部实现能力，而不是增加用户负担的复杂概念
5. 为后续 KodaX 整体能力增强预留稳定接口，包括长任务、评测、质量闭环和协作式执行

### 4. 范围

#### 4.1 首批范围

- worker contract：输入、输出 schema、budget、timeout、tool policy
- workspace protocol：目录约定、artifact handoff、result aggregation
- supervisor executor：顺序优先、独立任务再并行、支持失败隔离
- observation surface：run metadata、handoff traces、result summaries

#### 4.2 暂不进入首批范围

- 面向普通用户的通用 swarm 概念或复杂 UI
- 无边界的 agent 自我复制和深层树状扩散
- 与特定宿主强绑定的专有编排机制

### 5. 验证策略

首批验证场景以 `skill-creator` 的 grader / analyzer / comparator 套件为试验田，用真实工作流推动协议成形。只有当这套 supervisor/worker/workspace 形状被更多 KodaX 整体能力复用时，再考虑上升为更通用的产品层能力。

### 6. 设计要点

1. Swarm 是实现细节，不是产品心智
2. Delegation 必须是 bounded delegation，只在可分解、可验证、可回收的子任务上启用
3. Handoff 以 artifact 为中心，而不是传播整段会话历史
4. 默认串行，真正独立的任务才并行
5. 保证每个 worker 都有可审计的输入、输出和退出条件

### 7. 实施检查清单

- [ ] 定义 worker contract 与执行接口
- [ ] 定义 workspace artifact 协议
- [ ] 设计 supervisor 调度流程与失败隔离
- [ ] 在 `skill-creator` 第三步中完成首个真实试验田
- [ ] 补充调度链路的测试、review 与观测输出
- [ ] 基于真实复用情况评估是否上升为 KodaX 通用产品能力
