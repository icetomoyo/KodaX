## FEATURE_086: Legacy Code Purge (子任务 A 放弃 / 子任务 B 完成)

**Status**: 子任务 B 完成；子任务 A 放弃  
**Priority**: High  
**Category**: Core

### 子任务 A 放弃决策（2026-04-24）

**原计划**：ts-morph codemod 扫全仓 190 个 `KodaX*` 类型/接口/类，统一去掉 `KodaX` 前缀（保留 `KodaXError` / `KodaXClient` / `KodaXOptions` / `KodaXEvents` 四类品牌类型）。硬切发布、同步发 `@kodax/codemod` npm 包、附 `docs/MIGRATION_0.7.27.md` 迁移指南。

**放弃理由（系统性扫描发现的硬伤）**：

1. **marquee 类型被迫必须保留前缀**——下游用户日常写的 8 个关键类型跟 `@anthropic-ai/sdk` 或 `openai` 顶层 export **namespace 碰撞**：

   | 原计划目标名 | Anthropic SDK | OpenAI SDK |
   |------|:-:|:-:|
   | `Tool` | ✅ | ✅ |
   | `Message` | ✅ | ✅ |
   | `TextBlock` | ✅ | |
   | `ContentBlock` | ✅ | |
   | `ThinkingBlock` | ✅ | |
   | `ToolUseBlock` | ✅ | |
   | `RedactedThinkingBlock` | ✅ | |
   | `Session` | | ✅（另外还跟我们自家 `@kodax/core` 的 `Session` 撞） |

   下游消费者一旦同时写 `import type { Tool } from '@kodax/ai'` + `import type { Tool } from '@anthropic-ai/sdk'` 就报 TS2300。品牌前缀的**第二层功能**（在共用 namespace 里消除歧义）本来就是关键类型必须的，不光是品牌信号。

2. **设计决定在落地前就推翻 3 条**：
   - `KodaXToolDefinition → Tool (MERGE)`：全仓没有第二个 `Tool` 类型可合并，MERGE 不成立
   - `KodaXSessionLineage → REMOVE`：`packages/agent/src/session-lineage.ts` 1202 行 + 58 处引用还在用旧 plain-data 模式，迁移到 `Session + LineageExtension` 是独立大 feature，不是 hygiene
   - `KodaXSessionCompactionEntry → CompactionEntry`：跟姐妹类型 (`MessageEntry` / `BranchSummaryEntry` / `LabelEntry` / `ArchiveMarkerEntry`) 保留 `Session` 前缀不对称

   落地前扫一轮就发现 3 条错判，说明原设计对 190 个类型**没做过逐个 review**。真正 codemod 跑起来预计还会冒出更多同类坑。

3. **收益/代价严重倒挂**：
   - 去前缀成功的只剩 **170 个边角类型**（`ProviderPolicyIssueSeverity` / `FanoutBranchTransition` / `ManagedLiveEventPresentation` 这种下游 99% 不会直接写的）
   - 代价包括：ts-morph codemod 长期维护、`@kodax/codemod` npm 包同步发布/维护、`MIGRATION_0.7.27.md` 迁移指南、私仓 KodaX-private 跨仓同步、下游消费者强制跑 codemod 或卡 v0.7.26
   - 产出了一套"有的带前缀、有的不带"的**不一致** API 表面，比现状全带前缀更糟

4. **"只增不减"红线跟 rename 动作天然冲突**：190 个类型 × 5-20 处调用点 ≈ 3000+ 编辑位点的 AST 改写，任何 1 处漏改/改错 = 下游用户的 1 个功能"没了"。而 0.7.26 刚稳定下来，团队精力不宜重新引入同类风险。

**结论**：`KodaX*` 前缀**保留作为显式的设计选择**，不是技术债。品牌识别 + namespace 歧义消除两个作用同时存在。v0.7.27 不做前缀 rename。

**回滚动作**：`git revert d23b149`（原 scaffold commit：rename-map.ts + README + MIGRATION 文档）。

### 子任务 B：Legacy 代码清除（每条走三问检查）

**门槛**：任何一条在开始动代码前必须能明确回答下面三个问题，**三问任意一个答不上来，这一条从本版移出**（不是"尽量删"，是"删得干净 or 不删"）：

1. **这段代码当前是否还在被实际调用？** —— 以 `grep` call sites 为准，不以文档说"已 deprecated"为准
2. **它提供的能力是否已被新机制完整覆盖？** —— 需要新旧**一一对应的对照表**，不允许"大致等价"模糊地带
3. **删除后回归如何保证？** —— 新路径必须有测试覆盖；测试不足的先补测试、再删代码

本版候选清单（8 项，每条的三问对照表在对应 PR 里提交）：

#### 第 8 条放弃决策（2026-04-24，三问检查结果）

原计划删除 `OrchestrationTraceEvent` 和 `KodaXRepoIntelligenceTraceEvent` 两个在 v0.7.24 FEATURE_083 时标了 `@deprecated` 的 trace 事件类型，迁移调用点到 `@kodax/tracing` 的 `AgentSpan` / `HandoffSpan` / `EvidenceSpan`。调查后**两条都过不了三问**：

**`OrchestrationTraceEvent`（定义在 `packages/coding/src/orchestration.ts:79`）**
- Q1 "还在被调用吗"：monorepo 内**零生产消费者**（9 处 `appendTrace()` 全在 `orchestration.ts` 自己内部；`runOrchestration` / `createKodaXTaskRunner` 只被自家测试调）。**但**这两者+`OrchestrationTraceEvent` 都是 `@kodax/coding` 顶层公开导出 API，从 v0.7.x 保留至今，外部是否有消费者**无法从代码判断** → 保守判断为假阳性风险
- Q2 "新机制覆盖了吗"：`@kodax/tracing` 的 `AgentSpan` / `HandoffSpan` 是 **Runner 内部 span**（单个 Runner 生命周期内），而 `OrchestrationTraceEvent` 是**跨任务编排 span**（`runOrchestration` 调度的多个 Task 之间）。两者不是同一层级抽象，**没有覆盖**
- Q3 "删除回归保障"：自家测试会挂；外部消费者未知无法保障
- **结论**：保留。`@deprecated` 标记保留，但 JSDoc 里"scheduled for removal in FEATURE_086 (v0.7.27)"的话术改成"evaluated, kept"

**`KodaXRepoIntelligenceTraceEvent`（定义在 `packages/coding/src/types.ts:377`）**
- Q1 "还在被调用吗"：✅ **5 处真实产线消费**
  - `packages/coding/src/agent.ts` — 4 处 `emitRepoIntelligenceTrace()` 调用（routing / preturn / module / impact 阶段）
  - `packages/coding/src/task-engine/_internal/managed-task/repo-intelligence.ts` — `emitManagedRepoIntelligenceTrace` 工厂
  - `packages/repl/src/ui/json-events.ts` — 序列化成 `{ type: 'repo_intelligence.trace', stage, summary, capability, trace }` JSONL 写到 stdout（**非交互 CLI 模式的脚本化协议**，下游脚本的公开约定）
  - `packages/repl/src/ui/cli-events.ts` — 交互 REPL 消费
  - `src/acp_server.ts` — ACP 服务器模式消费
- Q2 "新机制覆盖了吗"：❌ **没覆盖**。`@kodax/tracing` 的 `EvidenceSpanData` 是 `{ kind: 'evidence', source: string, queryPreview?, resultCount?, cacheHit?, error? }` 通用抽象；老 event 是 `{ stage: 'routing'|'preturn'|'module'|'impact'|'task-snapshot', summary, capability?, trace? }` 的 **repo-intelligence 专属领域 schema**（`stage` 枚举 + `capability` + `trace` 嵌套 bundle）。把 `stage` 强映射到 `source` 会丢失枚举类型安全；把 `capability`/`trace` 塞到 `attributes` bag 会丢失领域 schema
- Q3 "删除回归保障"：必须同步改 stdout JSONL schema，破坏下游脚本兼容性
- **结论**：保留。这个类型**不是历史债**，是 repo-intelligence 产品功能的一部分。原设计文档把它归入 legacy 是误判。`@deprecated` 标记**移除**（它不应该被标 deprecated）

**对红线的影响**：两条都保留意味着 `@deprecated` 标签在主干多放一阵。这没有违反"只增不减"——保留类型 = 加码向后兼容 = 增。

#### 不做项（决策记录）

- **`@kodax/ai/src/cli-events/` + CLI-wrapper provider 跨包迁移** (GLM F-8, FEATURE_082 deferred)：设计稿 v1 曾计划把 `cli-events/` + `acp-base` / `codex-cli` / `gemini-cli` 三个 Provider 一起搬到 `@kodax/coding/src/cli-bridges/`，并把 static registry 改为 coding 侧 side-effect 注册。**经重新评估不做**，理由：
  - `cli-events/` 是同包内三个 CLI-wrapper Provider 的**内部实现细节**，不对外暴露
  - 三者在同一包内（`@kodax/ai/src/providers/` + `@kodax/ai/src/cli-events/`），**不存在循环依赖**
  - 外部消费者感知到的 API 仍是 `createProvider()` / provider registry，cli-events 不可见
  - "职责外溢"属于 ADR-001 **定位洁癖**问题（`@kodax/ai` 是否必须只包含 pure completion），不是**功能/正确性**问题
  - 违反 KodaX CLAUDE.md "3+ concrete use cases 才抽象"原则：无实际用户痛点（包尺寸、第三方单独引 CLI-bridge 的需求），跨包迁移风险（provider 注册时序、static registry side-effect、extension runtime 变更）没有对等收益
  - GLM F-8 标记为"已评估，按当前现状合规，若未来出现第三方单独引用 CLI-bridge 的具体需求再考虑"

#### "只增不减"的安全保障机制

子任务 A 放弃后，v0.7.27 只做子任务 B 里真正的"legacy 代码删除"，"只增不减"的保障完全落在三问检查 + 功能对等对照表上：

1. **三问门槛**（见"子任务 B"小节）——Q1 调用存在 / Q2 新机制覆盖 / Q3 删除回归保障，任一答不上就移出本版
2. **功能对等对照表**（子任务 B 核心交付物）——每一条 legacy 删除项在 PR 描述里列新旧一一对应：`compactMessages()` → FEATURE_079 的统一 normalize 管道；`--team` → FEATURE_027 `--agent-mode ama/sa`；`/project` → FEATURE_061 AMA Scout-first 等。对照表写不出来 = 三问第 2 问没过 = 不删
3. **第 8 条决策**——原计划删除的两个 trace 事件调查后过不了三问，保留。详见上方"第 8 条放弃决策"

### Contract

## FEATURE_091: Repo-Intelligence Protocol Package Extraction

**Status**: Planned  
**Priority**: High  
**Category**: Internal

### 背景

- 公仓 KodaX 的 `packages/coding/src/repo-intelligence/premium-contract.ts` 是公仓-私仓-第三方 host 三方共享的契约
- 当前私仓 `KodaX-private/packages/repointel-core/vendor/repo-intelligence/public-bridge.js` 是**手工 vendor**（build 时从公仓复制）
- 这种 vendor 方式有三个问题：
  1. 契约变更时两边 build 时序敏感
  2. 没有 npm 语义化版本保护
  3. 第三方 host（codex/claude/opencode，见 `clients/repointel/`）只能复制引用，无法依赖一个稳定 npm 包
- 用户需求：保留 KodaX-private 的闭源价值，同时让 repointel 可被 KodaX 和第三方 coding agent 单独使用

### 设计

#### 新包：`@kodax/repointel-protocol`

内容（~200 行，纯类型 + 常量）：
- `REPOINTEL_CONTRACT_VERSION` 常量
- `RepoIntelligenceHost` / `RepoIntelligenceIntent` / `RepointelCommand` enum types
- `RepointelRequestPayload` / `RepointelRpcRequest` / `RepointelRpcResponse` interfaces
- `RepointelStatus` / `RepointelTraceInfo` 辅助类型

#### 消费者迁移

**消费者 1：公仓 KodaX**
- `packages/coding/src/repo-intelligence/premium-contract.ts` 内容迁到新包
- `premium-contract.ts` 改为 `export * from '@kodax/repointel-protocol'`（保留路径向后兼容，v0.7.40 前移除）
- `premium-client.ts` 引 `@kodax/repointel-protocol`

**消费者 2：私仓 KodaX-private**
- `repointel-core/vendor/repo-intelligence/public-bridge.js` 删除
- `repointel-core` / `repointel-daemon` / `repointel-cli` 三个 package.json 加 `@kodax/repointel-protocol` 依赖
- 导入路径从 `./vendor/repo-intelligence/*` 改为 `@kodax/repointel-protocol`
- `scripts/vendor-seed.mjs` 或类似 vendor 脚本移除

**消费者 3：第三方 host `clients/repointel/`**
- `install.mjs` / `doctor.mjs` / `demo.mjs` 改为可选依赖 `@kodax/repointel-protocol`（用于 schema 校验）
- `SKILL.md` 文档更新接入方式

#### 闭源价值保留

- 协议包只有**类型 + 常量**，不暴露私仓实现逻辑（engine、semantic index、cache、routing 仍在 `KodaX-private`）
- `premium-client.ts` 仍调用本地 daemon RPC endpoint，不暴露 daemon 的内部行为
- 第三方 host 能通过协议包理解"如何调 daemon"，但无法复现 daemon 本身

### Contract

**Public API（新 npm 包）**：`@kodax/repointel-protocol` 首个版本 `v1.0.0`，语义化版本保护契约变更。

**Internal**：
- 公仓 `premium-contract.ts` 变成 re-export shim
- 私仓三 package.json 增加 dep
- 第三方 `clients/repointel/scripts/*.mjs` 可选升级

### Non-Goals

- **不**改变 daemon 协议语义（只是把协议定义搬家）
- **不**合并公仓-私仓代码库
- **不**发布 daemon 的闭源内容

### Dependencies

- **前置**：无硬依赖（契约抽取独立于 SDK 基底重组）
- **后置**：无硬后置

### Testing

- 单元：协议包类型在公仓 `premium-client` 使用等价于旧 `premium-contract.ts`
- 集成：私仓 daemon 拆掉 vendor、用 npm dep 后，`repointel-daemon` + `repointel-cli` 全测试绿
- 集成：公仓 `repo-intelligence/query.ts` 对 daemon 的 RPC 调用结果与 v0.7.26 等价
- 集成：`clients/repointel/scripts/demo.mjs` 在接入新协议包后仍能跑 demo flow

### Risk

- **跨仓 coordination**：公仓先发 `@kodax/repointel-protocol@1.0.0`，私仓后升级依赖；需要短时间内完成两边 release 以避免跨仓不一致
- **向后兼容**：公仓保留 `premium-contract.ts` re-export 路径一段时间；私仓 vendor 移除可以稍晚

### Timeline

v0.7.27 合并 086 + 091。前缀清理 + 协议包抽取分别走各自 PR，可并行推进；本版发版前两边都要绿。
