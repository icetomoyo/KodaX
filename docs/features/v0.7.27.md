# v0.7.27 Feature Designs

> 中文标题：`v0.7.27 Feature 设计`
>
> Version: `v0.7.27`
>
> Status: Planned (not yet released)

---

## 总览

v0.7.27 是 SDK 基底段的**尾声**。两件事都是结构性 hygiene 工作，风险域互不相交（前缀清理在 coding/core 核心代码，协议抽取在 repo-intelligence），可以同版本完成：

- **FEATURE_086**：借用户量少的窗口**一次性**删除 `KodaX*` 前缀 + 清除 legacy 代码。**硬切**（不保留 `@deprecated` 类型别名作为技术债式回退层），"只增不减"的保障靠 codemod 精确度、功能对等对照表、外部 codemod 同步发布三件事承担
- **FEATURE_091**：把 `premium-contract.ts` 抽成独立 npm 包，解决当前公仓-私仓手工 vendor 的跨仓脆弱性

## FEATURE_086: KodaX Prefix Cleanup and Legacy Purge

**Status**: Planned  
**Priority**: High  
**Category**: Core

### 背景

- 当前 public API 有 100+ `KodaX*` 前缀类型（`KodaXMessage`、`KodaXSessionCompactionEntry`、`KodaXAmaControllerDecision`……）
- 品牌前缀让通用概念（Message / Session / Tool）带上"vendor 专属"暗示，劝退 SDK 消费者
- 同行对比（openai-agents-python / pi-mono）都是 `Agent / Tool / Session` 简名
- 用户量少的窗口是**唯一低成本**的 rename 时机；用户已确认"趁现在搞掉，不走长期 deprecated"
- GLM 指出的 legacy：`compactMessages()` (F-9)、`--team` CLI 参数（I-2）、README `/project` 流程（I-1）、归一化函数重复（F-4、F-6）
- 红线：**相比 v0.7.22 之前只能增不能减**。任何 rename 或删除都必须靠"新路径完整覆盖 + 测试回归"证明功能对等，不允许出现"旧路径残留做安全网"的技术债

### 设计

#### 执行骨架：子任务 A（纯 rename） + 子任务 B（legacy 删除）严格隔离

两件事风险完全不同，必须分开执行、分开验证，即使同版本发布也要分成独立 PR 以便出问题时独立回滚：

- **子任务 A — 纯 rename（零行为变化）**：100+ 类型的 identifier 改名。严格只改 name、不动逻辑。用 ts-morph codemod 在 AST 层替换，不可能在语义上削弱功能。
- **子任务 B — legacy 代码删除（每条走三问检查）**：删的是真代码，每一条必须通过下面的三问门槛才能动手。

#### 子任务 A：前缀 rename 规则

**保留前缀**（品牌信号）：
- `KodaXError` 及其子类（异常继承链）
- `KodaXClient`（门面 class）
- `KodaXOptions` / `KodaXCodingOptions`（产品级配置）
- `KodaXEvents`（回调总线）

**去前缀**（通用原语）：
- `KodaXMessage` → `Message`
- `KodaXContentBlock` → `ContentBlock`
- `KodaXToolDefinition` → `Tool`（合并 TS type）
- `KodaXSessionLineage` → 移除（替换为 `Session` + `LineageExtension`）
- `KodaXSessionCompactionEntry` → `CompactionEntry`
- `KodaXTokenUsage` → `TokenUsage`
- `KodaXAmaControllerDecision` → 内部化（不再 export）
- `KodaXReasoningMode` → `ReasoningMode`
- `KodaXProviderConfig` → `ProviderConfig`
- ... 等 ~80 个类型逐个改名

#### 子任务 B：Legacy 代码清除（每条走三问检查）

**门槛**：任何一条在开始动代码前必须能明确回答下面三个问题，**三问任意一个答不上来，这一条从本版移出**（不是"尽量删"，是"删得干净 or 不删"）：

1. **这段代码当前是否还在被实际调用？** —— 以 `grep` call sites 为准，不以文档说"已 deprecated"为准
2. **它提供的能力是否已被新机制完整覆盖？** —— 需要新旧**一一对应的对照表**，不允许"大致等价"模糊地带
3. **删除后回归如何保证？** —— 新路径必须有测试覆盖；测试不足的先补测试、再删代码

本版候选清单（8 项，每条的三问对照表在对应 PR 里提交）：

1. **`compactMessages()`** (`packages/agent/src/messages.ts`)：v0.7.23 已标 deprecated，本版直接 remove
2. **`--team` CLI flag 彻底移除**：ADR-017 定废。**勘误**：v1 设计稿与我上一轮的 spot-check 都说过 "CLI flag 已移除，本版只清 config.json 残留"，经 Q1 重查**均不成立** —— CLI flag 实际上完整保留在 `src/kodax_cli.ts`（commander 注册 + sunset handler 打 error + exit 1 + 多处 help 文案 + bash completion 字符串），config.json 的 `loadConfig()` 严格类型 schema 从来就没有 `team` 字段。**实际本版动作**：移除 CLI flag 全链（commander option、`CliOptions.team` 字段、`validateCliModeSelection` 里的 json-mode guard 条件、team help topic、全局 help 行、help topic 索引、bash completion 字符串、sunset handler block），同步更新 `README.md` 示例与旧测试指南归档提示。替代路径 `--agent-mode ama/sa`（FEATURE_027 自 v0.7.10 起提供）已经稳定若干版本
3. **`/project` 命令流程完整移除**（Commits A→D，共 −12,951 lines / 61 files）：AMA (Scout-first, FEATURE_061) 已完整覆盖，/project 流程无独占能力。按四层顺序执行保证每步 build 绿：
   - **Commit A — Layer 6 heuristics**：删除 `LONG_RUNNING_PROMPT` / `getLongRunningContext` / `detectLongRunningProjectContext` / `harness: 'project'` hint path + `'Project harness'` /provider scenario
   - **Commit B — Layer 4+5 CLI surface**：删除 `--init` / `--append` / `--overwrite` / `--auto-continue` / `--max-sessions` / `--max-hours` 六个 flag + `buildInitPrompt` / `getFeatureProgress` / `checkAllFeaturesComplete` / `readFeatureProgressSnapshot` 四个 helper + 它们的 barrel 再导出 + 相关测试
   - **Commit C — Layer 0-3 module**：删除 20 个 `project-*.ts` / `project-*.test.ts` 文件 + `commands-project-shim.test.ts` + `project-completer.ts`；移除 `LEGACY_PROJECT_COMMAND_NAMES` + `printProjectMigrationGuidance` stub；清空 `PROJECT_ARGS` / project JSON guards / `projectInitPrompt` 通路；`KodaXTaskSurface` 收紧为 `'cli' | 'repl' | 'plan'`
   - **Commit D — 尾清理 + 文档**：删除 `KODAX_FEATURES_FILE` / `KODAX_PROGRESS_FILE` 两个 orphan 常量；更新 README / README_CN / packages/coding/README.md；同时 FEATURE_054 scope 随之归零（AMA 收敛不再需要单独任务）
4. **归一化函数重复**：FEATURE_079 已把 7 个 `normalize*` 函数统一，本版删除旧的 duplicate
5. **YAML 归一化** (`sanitizeYaml` / `normalizeHooks` 在 discovery.ts + skill-loader.ts 重复，GLM F-6)：提取到 `@kodax/skills/shared/yaml.ts`
6. **Provider 配置双写** (GLM F-5)：内置 Provider 类 + `KODAX_PROVIDER_SNAPSHOTS` 合并为单一来源
7. **`getProvider()` 每次 `new` SDK 客户端** (GLM F-7)：改惰性单例
8. **旧 `OrchestrationTraceEvent` / `RepoIntelligenceTraceEvent`** (FEATURE_083 已 deprecated)：本版删除

#### 不做项（决策记录）

- **`@kodax/ai/src/cli-events/` + CLI-wrapper provider 跨包迁移** (GLM F-8, FEATURE_082 deferred)：设计稿 v1 曾计划把 `cli-events/` + `acp-base` / `codex-cli` / `gemini-cli` 三个 Provider 一起搬到 `@kodax/coding/src/cli-bridges/`，并把 static registry 改为 coding 侧 side-effect 注册。**经重新评估不做**，理由：
  - `cli-events/` 是同包内三个 CLI-wrapper Provider 的**内部实现细节**，不对外暴露
  - 三者在同一包内（`@kodax/ai/src/providers/` + `@kodax/ai/src/cli-events/`），**不存在循环依赖**
  - 外部消费者感知到的 API 仍是 `createProvider()` / provider registry，cli-events 不可见
  - "职责外溢"属于 ADR-001 **定位洁癖**问题（`@kodax/ai` 是否必须只包含 pure completion），不是**功能/正确性**问题
  - 违反 KodaX CLAUDE.md "3+ concrete use cases 才抽象"原则：无实际用户痛点（包尺寸、第三方单独引 CLI-bridge 的需求），跨包迁移风险（provider 注册时序、static registry side-effect、extension runtime 变更）没有对等收益
  - GLM F-8 标记为"已评估，按当前现状合规，若未来出现第三方单独引用 CLI-bridge 的具体需求再考虑"

#### "只增不减"的安全保障机制

不保留 `@deprecated` 类型别名作为兜底（用户裁决：不接受技术债式回退通道）。"功能只增不减"的保障完全落在以下三件事上：

1. **Codemod 精确度（100% 覆盖硬门槛）**
   - 基于 ts-morph 遍历所有 public `export`，任意遗漏一个都视为 codemod 缺陷
   - 命令：`npm run codemod:rename-to-v0.7.27`
   - 验收：codemod 跑完 `npm run build && npm run test` 必须**一次性全绿**；有任何失败一律回退 codemod 重做，不允许"手工补几处"
   - 抽查：人工挑选 10+ 文件双列对比 codemod 前后 diff，确认只改 identifier 不改语义

2. **功能对等对照表（子任务 B 核心交付物）**
   - 每一条 legacy 删除项在 PR 描述里列新旧一一对应：`compactMessages()` → 具体哪个 call path 替代；`--team` 字段 → AMA 哪个 mode 覆盖等
   - 对照表写不出来 = 三问第 2 问没过 = 这条不删

3. **外部消费者 codemod 同步发布**
   - 迁移文档：`docs/MIGRATION_0.7.27.md` 列出所有 rename + 示例
   - 独立 npm 包（scope 候选 `@kodax/codemod`，终稿待定）一键迁移 >=90% 下游代码；v0.7.27 发版公告明确指引
   - 下游契约：未跑 codemod 的消费者卡在 v0.7.26，不提供兼容别名做桥梁

### Contract

**Breaking**：100+ 类型 rename；`compactMessages()` / `--team` / `/project` 命令流程（含 `--init` / `--auto-continue` 等 CLI flag + `buildInitPrompt` / `getFeatureProgress` / `ProjectStorage` 等公开 API + 所有 `project-*.ts` 模块 + `KODAX_FEATURES_FILE` / `KODAX_PROGRESS_FILE` 两个常量）全部移除。`KodaXTaskSurface` 联合收紧至 `'cli' | 'repl' | 'plan'`。

**迁移支持（硬切，无 deprecated 回退层）**：
- Codemod 脚本一键迁移 >=90% 代码
- 迁移文档（`docs/MIGRATION_0.7.27.md`）列出所有 rename + 示例
- 下游升级路径：跑 codemod → build+test 全绿 → 升 v0.7.27；否则卡在 v0.7.26

### Non-Goals

- **不**修改运行时行为（纯 rename + legacy remove）
- **不**发起 v1.0.0 major bump（v0.7.x 内部清理）
- **不**改变包结构（包重组已在 FEATURE_082 完成）
- **不**做 `@kodax/ai/src/cli-events/` 跨包迁移（见"不做项"决策记录）
- **不**保留 `@deprecated` 类型别名兜底（见"只增不减"的安全保障机制）

## FEATURE_091: Repo-Intelligence Protocol Package Extraction

**Status**: Planned  
**Priority**: High  
**Category**: Internal

### 背景

- 公仓 KodaX 的 `packages/coding/src/repo-intelligence/premium-contract.ts` 是公仓-私仓-第三方 host 三方共享的契约
- 当前私仓 `KodaX-private/packages/repointel-core/vendor/repo-intelligence/public-bridge.js` 是**手工 vendor**（build 时从公仓复制）
- 这种 vendor 方式有三个问题：
  1. 契约变更时两边 build 时序敏感
  2. 没有 npm 语义化版本保护
  3. 第三方 host（codex/claude/opencode，见 `clients/repointel/`）只能复制引用，无法依赖一个稳定 npm 包
- 用户需求：保留 KodaX-private 的闭源价值，同时让 repointel 可被 KodaX 和第三方 coding agent 单独使用

### 设计

#### 新包：`@kodax/repointel-protocol`

内容（~200 行，纯类型 + 常量）：
- `REPOINTEL_CONTRACT_VERSION` 常量
- `RepoIntelligenceHost` / `RepoIntelligenceIntent` / `RepointelCommand` enum types
- `RepointelRequestPayload` / `RepointelRpcRequest` / `RepointelRpcResponse` interfaces
- `RepointelStatus` / `RepointelTraceInfo` 辅助类型

#### 消费者迁移

**消费者 1：公仓 KodaX**
- `packages/coding/src/repo-intelligence/premium-contract.ts` 内容迁到新包
- `premium-contract.ts` 改为 `export * from '@kodax/repointel-protocol'`（保留路径向后兼容，v0.7.40 前移除）
- `premium-client.ts` 引 `@kodax/repointel-protocol`

**消费者 2：私仓 KodaX-private**
- `repointel-core/vendor/repo-intelligence/public-bridge.js` 删除
- `repointel-core` / `repointel-daemon` / `repointel-cli` 三个 package.json 加 `@kodax/repointel-protocol` 依赖
- 导入路径从 `./vendor/repo-intelligence/*` 改为 `@kodax/repointel-protocol`
- `scripts/vendor-seed.mjs` 或类似 vendor 脚本移除

**消费者 3：第三方 host `clients/repointel/`**
- `install.mjs` / `doctor.mjs` / `demo.mjs` 改为可选依赖 `@kodax/repointel-protocol`（用于 schema 校验）
- `SKILL.md` 文档更新接入方式

#### 闭源价值保留

- 协议包只有**类型 + 常量**，不暴露私仓实现逻辑（engine、semantic index、cache、routing 仍在 `KodaX-private`）
- `premium-client.ts` 仍调用本地 daemon RPC endpoint，不暴露 daemon 的内部行为
- 第三方 host 能通过协议包理解"如何调 daemon"，但无法复现 daemon 本身

### Contract

**Public API（新 npm 包）**：`@kodax/repointel-protocol` 首个版本 `v1.0.0`，语义化版本保护契约变更。

**Internal**：
- 公仓 `premium-contract.ts` 变成 re-export shim
- 私仓三 package.json 增加 dep
- 第三方 `clients/repointel/scripts/*.mjs` 可选升级

### Non-Goals

- **不**改变 daemon 协议语义（只是把协议定义搬家）
- **不**合并公仓-私仓代码库
- **不**发布 daemon 的闭源内容

### Dependencies

- **前置**：无硬依赖（契约抽取独立于 SDK 基底重组）
- **后置**：无硬后置

### Testing

- 单元：协议包类型在公仓 `premium-client` 使用等价于旧 `premium-contract.ts`
- 集成：私仓 daemon 拆掉 vendor、用 npm dep 后，`repointel-daemon` + `repointel-cli` 全测试绿
- 集成：公仓 `repo-intelligence/query.ts` 对 daemon 的 RPC 调用结果与 v0.7.26 等价
- 集成：`clients/repointel/scripts/demo.mjs` 在接入新协议包后仍能跑 demo flow

### Risk

- **跨仓 coordination**：公仓先发 `@kodax/repointel-protocol@1.0.0`，私仓后升级依赖；需要短时间内完成两边 release 以避免跨仓不一致
- **向后兼容**：公仓保留 `premium-contract.ts` re-export 路径一段时间；私仓 vendor 移除可以稍晚

### Timeline

v0.7.27 合并 086 + 091。前缀清理 + 协议包抽取分别走各自 PR，可并行推进；本版发版前两边都要绿。
