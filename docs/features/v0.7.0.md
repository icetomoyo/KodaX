## 总览

`v0.7.0` 是当前 task engine 架构的基础版本。

今天回看，这个版本最重要的价值不是“引入更多角色”，而是把下面这些能力放到同一条主线上：

- task-first persistence
- intent-aware routing
- managed task artifacts
- provider-aware policy
- headless runtime substrate

后续版本对 AMA 的大部分简化，本质上都是在这个基础上做减法和收束，而不是推翻它。

## FEATURE_022: Adaptive Task Engine and Native Multi-Agent Control Plane

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_022` 是当前执行模型的 umbrella feature。

它现在承载的不是“默认重型多智能体”，而是：

- single-agent first
- harness on demand
- evidence-driven completion
- role separation with minimal graph

### 当前执行骨架

`FEATURE_022` 目前落成的产品语义是：

- `SA`：完全脱离 AMA 的 single-agent path
- `AMA-H0`：direct
- `AMA-H1`：checked-direct
- `AMA-H2`：`Planner -> Generator <-> Evaluator`

明确删除：

- `H3_MULTI_WORKER`
- 默认 `Lead`
- 默认 `Admission`
- `Contract Reviewer`

### 当前角色分工

- `Scout`
  - pre-harness only
  - 判断任务是否值得升级
  - 如果有 skill，读取完整 skill 并生成 `skill-map`
- `Planner`
  - 基于 `scope facts + overview evidence + skill-map` 生成 contract
- `Generator`
  - 基于 `deep evidence + full skill + contract` 执行
- `Evaluator`
  - 基于 `contract + handoff + skill-map` 做 targeted spot-check

### 为什么这样收敛

这版设计吸收了 Anthropic harness 的核心思想：

- `planner / generator / evaluator`
- contract negotiation
- artifact handoff
- independent evaluation

但不再保留“多层前置角色默认全开”的旧实现。

### 证据分层

AMA 现在显式区分：

- `scope facts`
- `overview evidence`
- `deep evidence`

角色边界：

- `Scout`: scope facts + 少量 overview
- `Planner`: scope facts + overview
- `Generator`: deep evidence
- `Evaluator`: contract/handoff + 定点 deep evidence

### Skill 集成

skill 仍然是 invocation/playbook，而不是另起一套多智能体协议。

进入 AMA 时，采用：

```text
Scout -> skill-map -> Planner / Generator / Evaluator
```

其中：

- `Planner` 只读 `skill-map`
- `Generator` 读完整 skill
- `Evaluator` 默认只读 `skill-map`，必要时 fallback 到 raw skill

### 用户可见语义

- 主预算语义：`Work used/total`
- `Round` 只在真实额外 pass 存在时显示
- 工具披露必须尽量显示目标对象/范围/命令
- Evaluator 的内部元评估不应泄露到最终用户答案

### Acceptance criteria

- [x] `SA` 完全脱离 AMA
- [x] AMA 只保留 `H0 / H1 / H2`
- [x] `Scout` 不进入 H2 主 graph
- [x] H2 固定为 `Planner -> Generator <-> Evaluator`
- [x] skill 进入 AMA 后使用 `skill-map`
- [x] `Work` 成为默认用户可见预算语义
- [x] Evaluator public answer 与内部 verdict 分离

### Implementation summary

`FEATURE_022` 当前实现已经包括：

- intent gate + scout stage
- `H0 / H1 / H2` harness selection
- contract / handoff / verdict block enforcement
- skill-aware managed task runtime
- global work budget + approval
- AMA transcript retention
- role-level tool policy enforcement

## FEATURE_025: Adaptive Task Intelligence and Harness Router

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_025` 现在的核心不是“把任务尽快推高到更重 harness”，而是：

- intent-first
- repo signals only shape execution, not intent
- simple tasks stay cheap

### 当前规则

- `conversation` / `lookup` 默认 direct
- 非 actionable 输入不得因 dirty repo 升到 H1/H2
- Scout 在 pre-harness 阶段决定是否升级
- H2 只在 contract / evidence / verification 需求成立时进入

### Acceptance criteria

- [x] greeting 不会再因 dirty repo 升级
- [x] lookup 默认 direct
- [x] H3 不再参与 routing
- [x] repo intelligence 只影响“怎么做”，不改变“在做什么”

## FEATURE_029: Provider Capability Transparency and Harness Policy

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### Overview

provider policy 继续影响 harness 选择，但其职责被收束为：

- 影响 execution strength
- 影响 verification strategy
- 影响 runtime behavior warnings

它不应替代 intent gate，也不应把非 actionable 输入推高到重 harness。

### Acceptance criteria

- [x] provider policy 影响 H1/H2 选择与 warning
- [x] 不再参与 H3 相关逻辑

## FEATURE_034: Extension and Capability Runtime

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_034` 为 task engine 提供统一的 runtime substrate：

- tool registry
- hooks
- invocation metadata
- skill / command integration
- surface-agnostic execution plumbing

### 当前重点

当前最重要的演进点是：

- builtin / discovered command 统一到同一 metadata 模型
- skill invocation metadata 能进入 managed runtime
- tool use / transcript / budget / verification 事件能跨 surface 复用

### Acceptance criteria

- [x] runtime 继续保持 headless
- [x] command / skill invocation 元数据能进入 execution context
- [x] AMA 能基于 runtime 元数据生成 `skill-map`
