## FEATURE_019: Session Tree & Rollback System

**Status**: Planned
**Priority**: High
**Category**: New

### 1. 需求概述

借鉴 pi-mono 的会话树机制，实现 KodaX 的分支导航和回滚系统。允许用户回溯到对话的任意节点并从那里继续，而不是简单的"撤销"操作。

**核心目标**:
1. **数据结构迁移** - 将扁平消息数组转换为树形结构 (id/parentId)
2. **树导航 UI** - `/tree` 命令可视化导航对话树
3. **分支摘要** - 切换分支时可选生成 LLM 摘要
4. **Git Checkpoint 扩展** - 可选扩展，stash 代码状态

**背景**:
- KodaX 当前是扁平数组结构，无树形导航
- pi-mono 使用 id/parentId 构建树，支持分支导航和总结
- 现代 AI-coding 工具都有回滚到某一步的操作

**Inspired by**: [pi-mono Session Tree](https://github.com/badlogic/pi-mono)

### 2. 影响范围

**与 pi-mono 对比**:

| Feature | KodaX (Current) | pi-mono |
|---------|-----------------|---------|
| **Session Format** | Flat message array | Tree with id/parentId |
| **Message Identity** | Array index only | 8-char hex ID + parentId |
| **Navigation** | Load/Resume only | `/tree` visual navigator |
| **Branching** | ❌ None | ✅ Navigate + continue = new branch |
| **Branch Summary** | ❌ None | ✅ LLM-generated when switching |
| **Fork to New Session** | ❌ None | ✅ `/fork` command |
| **Labels/Bookmarks** | ❌ None | ✅ Label any entry |
| **Compaction** | ✅ LLM summary | ✅ LLM summary + tracks file ops |

**需要创建的文件**:
- `packages/agent/src/session/types.ts` - Entry 类型定义
- `packages/agent/src/session/manager.ts` - SessionManager 类
- `packages/agent/src/session/migration.ts` - V1→V2 迁移
- `packages/repl/src/ui/components/tree-selector.tsx` - 树导航组件
- `packages/coding/src/extensions/git-checkpoint.ts` - Git 扩展

**需要修改的文件**:
- `packages/repl/src/interactive/storage.ts` - 支持新格式
- `packages/repl/src/interactive/commands.ts` - /tree, /fork 命令
- `packages/agent/src/compaction/compaction.ts` - 集成到树结构

### 3. 技术方案

#### 3.1 数据结构设计

```typescript
// packages/agent/src/session/types.ts

interface SessionEntryBase {
  type: string;
  id: string;              // 8-char hex ID
  parentId: string | null; // null = root
  timestamp: string;
}

interface MessageEntry extends SessionEntryBase {
  type: "message";
  message: KodaXMessage;
}

interface CompactionEntry extends SessionEntryBase {
  type: "compaction";
  summary: string;
  firstKeptEntryId: string;
  tokensBefore: number;
  details?: { readFiles?: string[]; modifiedFiles?: string[] };
}

interface BranchSummaryEntry extends SessionEntryBase {
  type: "branch_summary";
  fromId: string;       // Abandoned leaf
  summary: string;
  details?: { readFiles?: string[]; modifiedFiles?: string[] };
}

interface LabelEntry extends SessionEntryBase {
  type: "label";
  targetId: string;
  label: string | undefined;
}

type SessionEntry = MessageEntry | CompactionEntry | BranchSummaryEntry | LabelEntry;
```

#### 3.4 Git Checkpoint 扩展

```typescript
// packages/coding/src/extensions/git-checkpoint.ts

interface GitCheckpointConfig {
  enabled: boolean;
  autoRestore: boolean;  // Auto restore on branch switch
}

export function createGitCheckpointExtension(config: GitCheckpointConfig) {
  const checkpoints = new Map<string, string>(); // entryId -> stash ref

  return {
    name: 'git-checkpoint',

    // Before each LLM turn
    onTurnStart: async (entryId: string) => {
      const { stdout } = await exec('git', ['stash', 'create']);
      const ref = stdout.trim();
      if (ref) checkpoints.set(entryId, ref);
    },

    // When switching branches
    onBeforeTreeSwitch: async (fromId: string, toId: string) => {
      const checkpoint = checkpoints.get(toId);
      if (checkpoint && config.autoRestore) {
        await exec('git', ['stash', 'apply', checkpoint]);
      }
    },

    // Manual restore
    restoreCheckpoint: async (entryId: string) => {
      const ref = checkpoints.get(entryId);
      if (ref) {
        await exec('git', ['stash', 'apply', ref]);
        return true;
      }
      return false;
    }
  };
}
```

   - 每个 LLM turn 前创建 git stash

   - 确保各层正确集成

## FEATURE_026: Roadmap Integrity and Tracker Consistency Hardening

**Status**: Planned
**Priority**: High
**Category**: Internal

### 1. Overview

KodaX's product workflow depends on `FEATURE_LIST.md`, issue tracking, version design docs, and release metadata being trustworthy. Today those artifacts can drift when counts, statuses, planned versions, or design headers are edited manually.

This feature hardens the "Doc First" workflow so project truth is validated instead of merely assumed.

### 2. Goals

1. Detect drift across feature index rows, detail sections, summary blocks, and per-version design docs
2. Keep changelog, package version, and tracker metadata aligned
3. Reduce the amount of manual count/status maintenance required
4. Make Project Harness and future automation able to rely on repo metadata safely

### 3. Technical Direction

- add a tracker validator for `FEATURE_LIST.md` and `KNOWN_ISSUES.md`
- verify version summary counts against real index rows
- verify design-doc headers and referenced feature IDs
- add release-time and CI-time checks for document drift
- prefer regeneration of summaries where practical instead of hand-editing

### 4. Acceptance Criteria

- [ ] tracker validation can detect version/count/status drift automatically
- [ ] feature and issue summaries no longer depend on manual arithmetic
- [ ] release metadata mismatches are surfaced before release
- [ ] self-hosting project workflows can trust tracker data as repo truth

## FEATURE_029: Provider Adapter Transparency and Semantic Compatibility

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. Overview

KodaX currently presents native API providers and CLI-bridge adapters in the same provider list, but they do not have identical semantics. Bridge providers can lose context fidelity, MCP support, or reasoning behavior while still appearing equivalent at the UI layer.

This feature makes provider differences explicit and safer.

### 2. Goals

1. Distinguish native providers from bridge adapters in product and runtime surfaces
2. Prevent silent semantic downgrade when using bridge-backed providers
3. Improve trust when switching among providers
4. Make adapter behavior testable with explicit compatibility rules

### 3. Scope

- capability matrix for context handling, MCP, reasoning, and session behavior
- runtime warnings or gates when unsupported semantics are requested
- adapter-specific tests for multi-turn behavior and connector loss
- clearer config and `/provider` output

### 4. Acceptance Criteria

- [ ] users can see which providers are native vs bridge-backed
- [ ] unsupported adapter capabilities are explicit instead of silent
- [ ] bridge providers have semantic compatibility tests
- [ ] provider switching no longer implies false parity
