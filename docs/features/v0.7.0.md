## FEATURE_019: Session Tree & Rollback System

**Status**: Planned
**Priority**: High
**Category**: New

### 1. 需求概述

借鉴 pi-mono 的会话树机制，实现 KodaX 的分支导航和回滚系统。允许用户回溯到对话的任意节点并从那里继续，而不是简单的"撤销"操作。

**核心目标**:
1. **数据结构迁移** - 将扁平消息数组转换为树形结构 (id/parentId)
2. **树导航 UI** - `/tree` 命令可视化导航对话树
3. **分支摘要** - 切换分支时可选生成 LLM 摘要
4. **Git Checkpoint 扩展** - 可选扩展，stash 代码状态

**背景**:
- KodaX 当前是扁平数组结构，无树形导航
- pi-mono 使用 id/parentId 构建树，支持分支导航和总结
- 现代 AI-coding 工具都有回滚到某一步的操作

**Inspired by**: [pi-mono Session Tree](https://github.com/badlogic/pi-mono)

### 2. 影响范围

**与 pi-mono 对比**:

| Feature | KodaX (Current) | pi-mono |
|---------|-----------------|---------|
| **Session Format** | Flat message array | Tree with id/parentId |
| **Message Identity** | Array index only | 8-char hex ID + parentId |
| **Navigation** | Load/Resume only | `/tree` visual navigator |
| **Branching** | ❌ None | ✅ Navigate + continue = new branch |
| **Branch Summary** | ❌ None | ✅ LLM-generated when switching |
| **Fork to New Session** | ❌ None | ✅ `/fork` command |
| **Labels/Bookmarks** | ❌ None | ✅ Label any entry |
| **Compaction** | ✅ LLM summary | ✅ LLM summary + tracks file ops |

**需要创建的文件**:
- `packages/agent/src/session/types.ts` - Entry 类型定义
- `packages/agent/src/session/manager.ts` - SessionManager 类
- `packages/agent/src/session/migration.ts` - V1→V2 迁移
- `packages/repl/src/ui/components/tree-selector.tsx` - 树导航组件
- `packages/coding/src/extensions/git-checkpoint.ts` - Git 扩展

**需要修改的文件**:
- `packages/repl/src/interactive/storage.ts` - 支持新格式
- `packages/repl/src/interactive/commands.ts` - /tree, /fork 命令
- `packages/agent/src/compaction/compaction.ts` - 集成到树结构

#### 3.4 Git Checkpoint 扩展

```typescript
// packages/coding/src/extensions/git-checkpoint.ts

interface GitCheckpointConfig {
  enabled: boolean;
  autoRestore: boolean;  // Auto restore on branch switch
}

export function createGitCheckpointExtension(config: GitCheckpointConfig) {
  const checkpoints = new Map<string, string>(); // entryId -> stash ref

  return {
    name: 'git-checkpoint',

    // Before each LLM turn
    onTurnStart: async (entryId: string) => {
      const { stdout } = await exec('git', ['stash', 'create']);
      const ref = stdout.trim();
      if (ref) checkpoints.set(entryId, ref);
    },

    // When switching branches
    onBeforeTreeSwitch: async (fromId: string, toId: string) => {
      const checkpoint = checkpoints.get(toId);
      if (checkpoint && config.autoRestore) {
        await exec('git', ['stash', 'apply', checkpoint]);
      }
    },

    // Manual restore
    restoreCheckpoint: async (entryId: string) => {
      const ref = checkpoints.get(entryId);
      if (ref) {
        await exec('git', ['stash', 'apply', ref]);
        return true;
      }
      return false;
    }
  };
}
```

## FEATURE_026: Roadmap Integrity and Tracker Consistency Hardening

**Status**: Planned
**Priority**: High
**Category**: Internal

### 1. Overview

KodaX's product workflow depends on `FEATURE_LIST.md`, issue tracking, version design docs, and release metadata being trustworthy. Today those artifacts can drift when counts, statuses, planned versions, or design headers are edited manually.

This feature hardens the "Doc First" workflow so project truth is validated instead of merely assumed.

### 2. Goals

1. Detect drift across feature index rows, detail sections, summary blocks, and per-version design docs
2. Keep changelog, package version, and tracker metadata aligned
3. Reduce the amount of manual count/status maintenance required
4. Make Project Harness and future automation able to rely on repo metadata safely

### 3. Technical Direction

- add a tracker validator for `FEATURE_LIST.md` and `KNOWN_ISSUES.md`
- verify version summary counts against real index rows
- verify design-doc headers and referenced feature IDs
- add release-time and CI-time checks for document drift
- prefer regeneration of summaries where practical instead of hand-editing

### 4. Acceptance Criteria

- [ ] tracker validation can detect version/count/status drift automatically
- [ ] feature and issue summaries no longer depend on manual arithmetic
- [ ] release metadata mismatches are surfaced before release
- [ ] self-hosting project workflows can trust tracker data as repo truth

## FEATURE_029: Provider Adapter Transparency and Semantic Compatibility

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. Overview

KodaX currently presents native API providers and CLI-bridge adapters in the same provider list, but they do not have identical semantics. Bridge providers can lose context fidelity, MCP support, or reasoning behavior while still appearing equivalent at the UI layer.

This feature makes provider differences explicit and safer.

### 2. Goals

1. Distinguish native providers from bridge adapters in product and runtime surfaces
2. Prevent silent semantic downgrade when using bridge-backed providers
3. Improve trust when switching among providers
4. Make adapter behavior testable with explicit compatibility rules

### 3. Scope

- capability matrix for context handling, MCP, reasoning, and session behavior
- runtime warnings or gates when unsupported semantics are requested
- adapter-specific tests for multi-turn behavior and connector loss
- clearer config and `/provider` output

### 4. Acceptance Criteria

- [ ] users can see which providers are native vs bridge-backed
- [ ] unsupported adapter capabilities are explicit instead of silent
- [ ] bridge providers have semantic compatibility tests
- [ ] provider switching no longer implies false parity

## FEATURE_035: MCP Capability Provider

> 2026-03 redesign note: the normative design for FEATURE_035 is the "Redesign Summary" below. The older bridge-oriented draft is retained temporarily as historical context and should not be treated as the target implementation.

### Redesign Summary

#### Positioning

FEATURE_035 is no longer scoped as a REPL-only "MCP Bridge Runtime". It becomes the first large **Capability Provider** built on top of FEATURE_034.

Its job is:

- to represent MCP as a provider inside the capability runtime
- to support lazy discovery and typed description of tools, resources, and prompts
- to execute MCP capabilities without flattening everything into a single `mcp_call + tools.md` pattern

#### Why this redesign is needed

- A REPL-startup bridge would miss print mode, auto-continue, and other non-REPL paths that already call `runKodaX()`.
- MCP has three distinct capability families: tools, resources, and prompts. Treating them all as one markdown index throws away too much structure.
- MCP supports pagination, `list_changed` notifications, and structured tool output. A static `tools.md` file cannot be the main protocol surface.
- "Zero SDK dependency" is not user value. Correctness, spec alignment, and future support for resources/prompts matter more.

#### Runtime ownership

The MCP runtime should be session/runtime owned, not `InkREPL.tsx` owned.

```text
runKodaXSession()
  -> load extensions
  -> init capability registry
  -> init MCP capability provider(s) if enabled
  -> build model tool list
  -> run agent
```

REPL can still expose `/mcp status` and `/mcp refresh`, but it should not be the only place MCP exists.

#### v1 model-facing surface

For a low-risk rollout, v1 may expose a small MCP-prefixed tool family to the model:

```typescript
mcp_search({
  query: string,
  server?: string,
  kind?: 'tool' | 'resource' | 'prompt',
  limit?: number,
})

mcp_describe({ id: string })

mcp_call({
  id: string,
  args: Record<string, unknown>,
})

mcp_read_resource({ id: string })
```

This is still provider-neutral internally because all real behavior lives behind the capability provider abstraction from 034.

#### Capability and catalog model

```typescript
interface McpCatalogItem {
  id: string; // mcp:github:tool:create_issue
  serverId: string;
  kind: 'tool' | 'resource' | 'prompt';
  name: string;
  title?: string;
  summary: string;
  tags?: string[];
  risk?: 'read' | 'write' | 'network' | 'exec';
  annotations?: Record<string, unknown>;
  cachedAt: string;
}

interface McpCapabilityDescriptor extends McpCatalogItem {
  inputSchema?: JsonSchema;
  outputSchema?: JsonSchema;
  promptArgsSchema?: JsonSchema;
  uri?: string;
  mimeType?: string;
}
```

JSON should be the source of truth:

```text
~/.kodax/mcp/
  servers.json
  catalog/
    github.index.json
    github.items.json
    fs.index.json
    fs.items.json
  debug/
    github.tools.md
    fs.resources.md
```

Markdown is optional debug/export output, not the discovery protocol.

#### Connection and refresh strategy

Recommended config semantics:

```json
{
  "mcp": {
    "servers": {
      "github": {
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "connect": "lazy",
        "trust": "ask"
      },
      "fs": {
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"],
        "connect": "prewarm",
        "trust": "workspace"
      }
    }
  }
}
```

Semantics:

- `lazy`: connect on first search/describe/call/read
- `prewarm`: connect at session start
- `disabled`: keep config but do not activate

Refresh behavior:

- paginate through `tools/list`, `resources/list`, `prompts/list`
- cache catalog snapshots on disk
- mark provider dirty on `notifications/*/list_changed`
- refresh lazily on next search/describe
- if the server becomes unhealthy, reconnect on next use

#### Result shape and permissions

035 depends on a structured capability result shape in the runtime. MCP tool/resource outputs should not be flattened to a single string too early.

Design requirements:

- preserve `structuredContent` and declared `outputSchema` when available
- treat server annotations as hints, not as a security boundary
- align confirmation/trust behavior with KodaX permission modes
- raise confirmation level for destructive, open-world, network, or exec-like capabilities

#### Suggested file layout

```text
packages/coding/src/capabilities/providers/mcp/
  adapter.ts
  catalog.ts
  provider.ts
  runtime.ts

packages/repl/src/interactive/
  commands.ts   // /mcp status, /mcp refresh
```

#### Acceptance criteria rewrite

- [ ] MCP works in both REPL and non-REPL execution paths
- [ ] tools, resources, and prompts all enter the catalog
- [ ] `mcp_search`, `mcp_describe`, `mcp_call`, and `mcp_read_resource` work against the cached catalog/runtime
- [ ] catalog state is cached in JSON, with markdown only as optional debug output
- [ ] MCP servers reconnect on demand after failure
- [ ] the design does not require injecting every remote MCP tool directly into `KODAX_TOOLS`
- [ ] the design does not depend on "zero SDK dependency" as an architecture constraint

**Status**: Planned
**Priority**: High
**Category**: New

### 1. 概述

为 KodaX 提供 MCP 生态工具访问能力，无需内置 MCP SDK。采用持久化 Runtime 架构，在 REPL 启动时连接所有配置的 MCP server（长驻子进程），通过单一 `mcp_call` 工具暴露给 LLM，工具 schema 通过文件系统渐进披露。

**核心目标**:
1. **零 MCP SDK 依赖** — 纯 Node.js `child_process` + JSON-RPC 实现客户端（~100 行）
2. **持久化 Runtime** — REPL 启动时 spawn MCP server 子进程，session 期间保持活跃
3. **单一 mcp_call 工具** — 固定描述，不随 server 配置变化，不动态修改 KODAX_TOOLS
4. **渐进式披露** — 工具定义写入 `~/.kodax/mcp-bridge/` 用户级目录，LLM 按需 read 发现，智能在 LLM 推理而非启动时分析
5. **配置驱动** — 用户在 `~/.kodax/config.json` 中配置 `mcpServers`
6. **最小化实现** — 总计 ~325 行新增 + ~50 行修改，只做 `/mcp status` 命令

**背景**:
- KodaX 已决定不内置 MCP 支持（FEATURE_027 已删除）
- MCP 生态拥有大量高质量工具服务器（filesystem、github、postgres、brave-search 等）
- 行业标准：Claude Code、Cline、Cursor、Goose、OpenCode 全部采用持久连接模式
- 借鉴 Anthropic "Code Execution with MCP" 范式：工具定义按需加载，98.7% token 减少
- 单一 mcp_call 模式避免动态修改 KODAX_TOOLS 数组和 KODAX_TOOL_REQUIRED_PARAMS 白名单
- MCP 内容全部在用户级目录 `~/.kodax/`，不污染项目目录

**需要创建的文件**:
- `packages/repl/src/bridge/mcp-client.ts` — JSON-RPC over stdio 客户端（~100 行）

### 3. 技术方案

#### 3.1 零 SDK 的 MCP 客户端

```typescript
// packages/repl/src/bridge/mcp-client.ts
class MCPServerConnection {
  private proc: ChildProcess;
  private nextId = 1;
  private pending = new Map<number, { resolve, reject }>();
  private buffer = '';

  // 发送 JSON-RPC request，等待 response
  async send(method: string, params: Record<string, unknown>): Promise<unknown>;
  // 发送 JSON-RPC notification（无返回值）
  async notify(method: string, params: Record<string, unknown>): Promise<void>;
  // 启动连接：spawn → initialize → notifications/initialized → tools/list
  async connect(command: string, args: string[], env?: Record<string, string>): Promise<ToolDef[]>;
  // 调用工具
  async callTool(name: string, args: Record<string, unknown>): Promise<string>;
  // 优雅关闭：stdin.end → SIGTERM(2s) → SIGKILL(2s)
  async shutdown(): Promise<void>;
  // 状态
  get name(): string;
  get connected(): boolean;
}
```

关键实现点：
- stdout 用 `\n` 分割 JSON-RPC messages，处理跨 chunk 的 buffer
- stderr 直接 pipe 到 process.stderr（MCP server 的日志输出）
- 超时保护：request 30s 超时

#### 3.2 MCPRuntime 进程管理

## FEATURE_037: API Token Usage 真实值优先 + 估算回退

**Status**: Planned
**Priority**: Medium
**Category**: Enhancement

### 1. 需求概述

建立统一的 Token Usage 获取策略：当 API 响应中包含 token usage 字段时，优先使用 API 返回的真实 token 数量；当 API 未返回有效 usage 字段时，自动回退到现有的 `js-tiktoken` 客户端估算。

**核心目标**:
1. **消除估算误差** — 使用 API 返回的真实 token 用量替代客户端估算，提升 compaction/context 管理的精确度
2. **零降级风险** — 所有现有 Provider 均支持返回 usage，估算仅作为极端情况的兜底
3. **统一接口** — 扩展 `KodaXStreamResult` 添加 `usage` 字段，所有 Provider 走同一数据通道
4. **优先 + 回退** — 消费端优先使用真实 usage，仅在缺失时回退到 `estimateTokens()`

**背景**:
- 所有 Provider API 均支持返回 usage 数据，但 KodaX 当前**零提取**
- CLI bridge 解析器（Gemini/Codex）已提取 usage，但在 `pseudo-acp-server.ts` 的 ACP 协议转换中**丢失**
- compaction/context 管理完全依赖客户端 `js-tiktoken` 估算，与 API 实际 token 计数存在偏差
- `KodaXStreamResult` 接口**没有 usage 字段**，真实 token 数据无法传播到上层

### 2. 影响范围

**需要修改的文件**:

| 文件 | 改动说明 |
|------|---------|
| `packages/ai/src/types.ts` | `KodaXStreamResult` 添加可选 `usage` 字段 |
| `packages/ai/src/providers/anthropic.ts` | 从 `message_start` + `message_delta` 提取 usage |
| `packages/ai/src/providers/openai.ts` | 添加 `stream_options`，提取最终 chunk usage |
| `packages/ai/src/cli-events/pseudo-acp-server.ts` | 修复 `CLICompleteEvent` usage 丢失 |
| `packages/ai/src/cli-events/acp-base.ts` | 接收并传播 usage 数据 |
| `packages/agent/src/tokenizer.ts` | 添加 `getTokens(messages, usage?)` 优先使用真实值 |
| `packages/agent/src/compaction/compaction.ts` | 优先使用 `usage.totalTokens` |

**不需要修改的文件**:
- `packages/ai/src/cli-events/gemini-parser.ts` — 已正确解析 usage
- `packages/ai/src/cli-events/codex-parser.ts` — 已正确解析 usage

### 3. 技术方案

#### 3.1 KodaXStreamResult 扩展

```typescript
  inputTokens: number;
  outputTokens: number;
  totalTokens: number;

  toolBlocks: KodaXToolUseBlock[];
```

#### 3.2 Anthropic 协议族 usage 提取

Anthropic streaming 的 usage 分散在两个事件中：

- `message_start` → `message.usage.input_tokens`（包含缓存命中信息）
- `message_delta` → `usage.output_tokens`

```typescript
// packages/ai/src/providers/anthropic.ts

// 在 stream() 方法中增加局部变量追踪 usage
let inputTokens = 0;
let outputTokens = 0;

// message_start 事件
if (event.type === 'message_start') {
  inputTokens = event.message.usage?.input_tokens ?? 0;
}

// message_delta 事件
if (event.type === 'message_delta') {
  outputTokens = event.usage?.output_tokens ?? 0;
}

// 流结束后返回 usage
return {
  textBlocks,
  toolBlocks,
  thinkingBlocks,
  usage: (inputTokens > 0 || outputTokens > 0) ? {
    inputTokens,
    outputTokens,
    totalTokens: inputTokens + outputTokens,
  } : undefined,
};
```

#### 3.3 OpenAI 协议族 usage 提取

OpenAI 需要显式启用 `stream_options.include_usage`，usage 出现在最终 chunk 中：

```typescript
// packages/ai/src/providers/openai.ts

// 创建请求时添加 stream_options
const stream = await this.client.chat.completions.create({
  ...params,
  stream: true,
  stream_options: { include_usage: true },  // 新增
});

// 流处理中追踪 usage
let usage: KodaXUsage | undefined;

for await (const chunk of stream) {
  // ... 现有 delta 处理逻辑 ...

  // 提取 usage（通常在最终 chunk）
  if (chunk.usage) {
    usage = {
      inputTokens: chunk.usage.prompt_tokens,
      outputTokens: chunk.usage.completion_tokens,
      totalTokens: chunk.usage.total_tokens,
    };
  }
}

return {
  textBlocks,
  toolBlocks,
  thinkingBlocks: [],
  usage,
};
```

#### 3.5 消费端优先 + 回退策略

```typescript
// packages/agent/src/tokenizer.ts

export function getTokenCount(
  messages: KodaXMessage[],
  usage?: KodaXUsage,
): number {
  // 优先使用 API 返回的真实值
  if (usage?.totalTokens && usage.totalTokens > 0) {
    return usage.totalTokens;
  }
  // 回退到客户端估算
  return estimateTokens(messages);
}
```

```typescript
// compaction 判断中优先使用 usage
function needsCompaction(
  messages: KodaXMessage[],
  threshold: number,
  usage?: KodaXUsage,

  const tokens = getTokenCount(messages, usage);
  return tokens >= threshold;
}
```
