## 总览

`v0.7.0` 是这轮架构重置的起点版本。

它的任务是把新的核心骨架装进去：

- task-first persistence
- task abstraction 和 lifecycle
- native multi-agent control
- automatic harness routing
- provider-aware policy
- 能承载后续能力演进的 runtime substrate

这个版本有一个很重要的设计动作：
它有意把 `FEATURE_022` 扩成当前执行模型转向的 umbrella feature。
`FEATURE_019`、`FEATURE_025`、`FEATURE_029`、`FEATURE_034` 则是把这个扩大的 `022` 真正落地所需的支撑切片。

  -> FEATURE_025 harness router

## FEATURE_022: Adaptive Task Engine and Native Multi-Agent Control Plane

**Status**: Completed
**Priority**: High
**Category**: New

> 中文导读：
> 这是当前最核心的一份 feature 设计文档。
> 如果你想理解现在的 KodaX 为什么会有 `task engine`、`planner / generator / evaluator`、`/project` control surface，以及 session 分层，这份文档就是主入口。

### 概述

`FEATURE_022` 现在已经不只是“multi-agent control plane”那一小块。
它负责承接产品层从下面这种模式：

- command-led execution
- explicit mode switching
- 只有 `project` 才有长流程语义

转向下面这种模式：

- task-first execution
- 对非简单任务默认使用 native multi-agent role separation
- evidence-driven completion
- 让 `/project` 成为 managed task state 的 control surface

这就是本轮架构变化真正的归属 feature。

快速说明：

- `FEATURE_022` 不只是 “multi-agent feature”，它是 execution model 的 umbrella feature
- `user session` 指用户可以恢复的会话，例如 `kodax -c`
- `planner / generator / evaluator` 的 internal session 属于 managed-task runtime，不是产品层恢复入口
- verification 现在强调 visible evidence，non-terminal worker 的检查过程不应被黑盒化

### 为什么由 `022` 承接这次变化

- 它本来就是最接近新方向的既有 feature
- 这轮架构变化的核心是 execution shape，而不只是 storage 或 prompt
- 让 `022` 作为 umbrella feature，可以保持 roadmap 连续性，而不是再新造一个割裂的 ID
- 这样 feature planning 里就有一个明确 owner 来承接整次架构转向

### `022` 负责什么

`FEATURE_022` 负责：

- the first-class `Task` abstraction as the product's primary execution unit
- the managed-task lifecycle and state transitions
- role separation for `Lead`, `Planner`, `Generator`, `Evaluator`, and specialist workers
- the product contract that every non-trivial task is executed and judged through separated responsibilities
- the migration path that internalizes `Project Mode` into the task engine
- the integration contract across intake, routing, persistence, provider policy, and runtime substrate

### `022` 不负责什么

- persistence primitives and checkpoint storage
- routing heuristics and harness scoring policy
- provider capability truth
- extension and capability runtime loading
- roadmap validation mechanics

这些仍分别归属于 `019`、`025`、`029`、`034`、`026`。

### 与其他 `v0.7.0` feature 的关系

| Feature | How it supports `022` |
|---|---|
| `FEATURE_019` | supplies durable task lineage, checkpoints, and resumable state |
| `FEATURE_025` | decides which task/harness shape should be used |
| `FEATURE_029` | constrains task execution shape using provider truth |
| `FEATURE_034` | provides the runtime substrate the task engine runs on |
| `FEATURE_026` | keeps the roadmap and design docs aligned while the architecture shifts |

### Role model

`022` still contains the original control-plane responsibilities:

- role assignment
- task decomposition
- result synthesis
- evaluator independence

Default role sets remain:

| Complexity | Default role set |
|---|---|
| `L0` | one direct agent |
| `L1` | generator + evaluator |
| `L2` | planner + generator + evaluator |
| `L3` | lead + planner + one or more workers + evaluator |

### 对照 Anthropic harness 的差距与补齐

`FEATURE_022` 最开始参考的是 Anthropic 那篇 harness design 文章，但之前真正落地到代码时，只做到了“角色分离的一次性 DAG 编排”，还没完全做到“可见、可控、带反馈回路的 harness”。

下面这张表，用来说明差距原来在哪、现在补到了什么程度：

| 维度 | Anthropic harness 参考方向 | 这次补齐前的 `022` 落地 | 现在的 `022` 落地 |
|---|---|---|---|
| 角色分离 | `Generator` 与 `Evaluator` 明确分工 | 已具备 | 已具备，并保持为默认完成语义 |
| 反馈闭环 | `Evaluator` 失败后可把反馈送回 `Generator` 进入下一轮 | 缺失，只能单次 `generator -> evaluator` | 已补齐为显式、有限轮次的 refinement loop |
| contract negotiation | 开工前先把 success criteria / required evidence 谈清楚，并在需要时重谈 | 缺失，planner 更像自然语言 plan | 已补齐为 planner / lead 产出的结构化 contract block，加上 `contract-review` preflight gate，并写回 task contract / `contract.json` |
| loop 可见性 | 不是 hidden retry，而是可解释、可观察的 round | 缺失，用户只看到终态收口 | 已补齐，round 开始与 evaluator 追返会对外发事件 |
| handoff fidelity | 评审/验证应基于完整交接或 artifact，而不是短摘要 | 缺失，长文本 review 容易被截断成 summary | 已补齐为 artifact-first handoff：`handoff.json` / `handoff.md` + `feedback.json` / `feedback.md` + `result.json` + 精简 excerpt |
| transcript 可见性 | 非终态 worker 的关键输出仍可回看 | 缺失，Generator 输出可能只短暂出现然后消失 | 已补齐，非终态输出会作为 managed-task transcript 保留下来 |
| final answer semantics | 用户拿到的是 final review / final verdict，而不是“review 的 review” | 偏元评审语义 | 已改成 final user-facing answer contract |
| QA optionality | QA 是否介入不应是固定 yes/no，而应取决于任务是否超出当前 solo boundary | 缺失，进入 `H1/H2/H3` 就固定带 `Evaluator` | 已补齐为 `quality assurance mode=required|optional`，低风险 AMA 可以在 `contract-review` 后直接由 `Generator` 收口 |
| decision plane | 是否进入多角色 harness 应先看任务是否超出 solo boundary，而不是只看 task label | 主要靠粗粒度 `H0/H1/H2/H3` 规则，review / bugfix 容易过早升级 | 已补齐为 `soloBoundaryConfidence` + `needsIndependentQA` 双信号；LLM router 与 fallback 都会产出同一组信号，再由 runtime 做硬收口 |
| structured control retry | router / contract / verdict 这类 control-plane 结构化输出应该先尝试自修复，再决定 fallback 或 block | 路由失败和协议块缺失容易一次性退化 | 已补齐为最多 3 次的智能重试；router 失败后 heuristic fallback，contract / verdict 协议失败则直接 `blocked` |
| long-running policy | 轮次和成本应按任务形态、风险、上下文预算自适应，而且打住时应留下 continuation surface | 缺失，固定 3 轮 refinement | 已补齐为 harness / 风险 / project-long-running hint / context token 感知的 adaptive round budget，并持久化 `continuation.json` checkpoint |
| budget disclosure | budget 应帮助模型收口，而不是从第一轮就鼓励“还有很多轮可以继续探索” | worker 对预算边界无感，或者只能被硬切 | 已补齐为 `green / yellow / orange / red` 渐进披露；只有接近边界时才显示精确 round ratio，并允许结构化延期申请 |
| memory strategy | 长流程 worker 不应固定只有一种 session 策略 | worker session 语义较固定，主要靠实现细节决定是否复用 | 已补齐为 `continuous / compact / reset-handoff` 一等 memory strategy，并按角色和上下文压力自动选择 |
| verification rubric | verification 不应只是一组散的 checks，而应有可判定的 rubric / threshold / scorecard | 主要是 `requiredChecks` / `requiredEvidence` 文本约定 | 已补齐为 `rubricFamily`、`criteria[]`、`scorecard.json`，Evaluator 必须按 criterion / threshold 给出结论 |
| runtime-under-test contract | 前端 / API / DB 验证不应只靠 Evaluator 自己猜如何启动和检查系统 | 浏览器验证更多靠 prompt hint 和 capability hint | 已补齐为 `runtime-contract.json`，可显式描述 `startupCommand`、`baseUrl`、`uiFlows`、`apiChecks`、`dbChecks`、`fixtures` |
| 单 agent 退路 | 需要有清楚的 escape hatch 控制成本 | 只有自动 fallback，没有正式用户开关 | 由 `FEATURE_027` 提供 `SA / AMA` 模式协作收口 |

### Product commitments

1. Non-trivial tasks default to multi-agent execution.
2. Single-agent execution remains only as a low-complexity fallback.
3. `--team` is no longer the canonical product surface for multi-agent work.
4. `/project` stops being a separate future architecture and becomes a task-engine control surface.
5. Completion must remain compatible with evaluator-led, evidence-driven judgment.

### Detailed direction

This feature should carry forward the useful parts of the older multi-agent design instead of losing them during the roadmap rewrite.

The intended production-grade target is:

- native subagents rather than comma-separated parallel prompts
- explicit task decomposition and worker assignment
- clearer role and permission boundaries among agents
- evidence aggregation across workers
- stable cooperation with task persistence, evaluation, and future multi-surface workflows

The old `--team` path remains useful as orchestration plumbing, but it is not the target architecture because it lacks:

- role semantics
- contract negotiation
- evaluator authority
- durable task truth
- explicit evidence synthesis

### Boundary with `FEATURE_034`

`FEATURE_034` provides single-runtime programmability: extensions can participate in one running session, influence model/tool state, and queue follow-up work.

`FEATURE_022` still owns:

- task decomposition into subagents
- role and permission boundaries among agents
- evidence aggregation across workers
- a control plane that survives across surfaces

`034` should make orchestration possible to build on top of, but it should not collapse multi-agent control into the extension runtime itself.

### Suggested internal model

Recommended internal orchestration units:

- `TaskContract`
- `RoleAssignment`
- `WorkItem`
- `EvidenceBundle`
- `OrchestrationVerdict`

This keeps `022` centered on the execution model, while `019` and the DD own persistence details.

### 当前实现的详细设计

下面这些点，是这轮 `Adaptive Project Harness + AMA Harness` 真正已经落到 runtime 的结构化设计。

#### 1. 决策面：先判断 solo boundary，再决定 harness

现在不会再让模型直接拍板 `H0/H1/H2/H3`。
router 的职责变成先产出两类关键信号：

- `soloBoundaryConfidence`
- `needsIndependentQA`

这两个信号会同时出现在：

- provider-backed LLM routing
- heuristic fallback routing

然后再由 runtime 结合：

- repo intelligence
- task surface
- provider policy
- verification contract

做最终 harness 收口。

这意味着“小 review / 小 bugfix”不再天然至少升级到 `H1`；
只有它真的超出当前 solo boundary，或者确实值得付独立 QA 成本时，才会进入更重的 AMA harness。

#### 2. 控制面协议：router / contract / verdict 都走统一结构化重试

本轮把 control-plane 里的结构化判断点统一成了“最多 3 次智能重试”的协议：

- task router
- auto-reroute judge
- planner / lead 的 `kodax-task-contract`
- `contract-review`
- evaluator 的 `kodax-task-verdict`

策略是：

- 可恢复的 timeout / transport / 429 / 5xx / invalid JSON / schema invalid / 缺关键字段，先尝试重试
- router 3 次失败后退回 heuristic fallback
- contract / verdict 协议 3 次仍失败，则直接 `blocked`

这样 `022` 的完成判断就不再只是 prompt 约定，而是 runtime contract。

#### 3. Budget controller：统一管理 round、worker iter 和 continuation

本轮新增了统一的 `Harness Budget Controller`，它会为每次 managed task 维护：

- `global budget`
- `reserve pool`
- `planned rounds`
- 每个 worker 的 `soft budget`
- 每个 worker 的 `hard ceiling`

默认预算基线是：

- `H0 = 50`
- `H1 = 100`
- `H2 = 200`
- `H3 = 350`

然后再根据：

- `project / longRunning`
- `review / verify / debug / investigate`
- `requiresBrainstorm / systemic`
- 当前 context token footprint

做自动修正，并在预算不足以继续时输出 `continuation.json`。

#### 4. Progressive Budget Disclosure：只在接近边界时才暴露精确轮次

预算信息不会从第 1 轮开始就持续告诉模型“你现在在 `10/15` 轮”。
当前采用的是渐进披露：

- `green`：正常执行，只强调 focused execution
- `yellow`：开始收敛，但仍不暴露精确数字
- `orange`：接近边界，开始显示精确 round ratio，并禁止再开新探索分支
- `red`：最终窗口，必须在“完成 / blocked / 申请小额延期”之间收口

而且不同角色进入这些 zone 的节奏不同：

- `planner / contract-review` 更早收敛
- `evaluator` 比 `generator` 更早进入收口区
- `generator / implementation worker` 保留更长的执行窗口

#### 5. Automatic Extension Request：只在临界区允许模型申请少量续跑

本轮新增了 `kodax-budget-request` 协议块。

worker 只有在接近预算边界时，才可以申请：

- 再加 `1-3` 个 iter
- 说明为什么“再跑一点就能收口”
- 给出如果被拒时应该如何交付当前最完整结果

系统会用 reserve pool 自动审批这类小额延期；
如果 reserve 不足、或者申请不像“接近完成”而更像“继续开放探索”，则直接拒绝，并要求输出：

- 当前最完整结果
- 未完成原因
- continuation 建议

#### 6. Memory Strategy Controller：`continuous / compact / reset-handoff`

worker session 不再固定只有一种行为。
现在 `022` 已把 memory strategy 提升成一等控制面：

- `continuous`
- `compact`
- `reset-handoff`

其选择依据包括：

- provider 的 `sessionSupport`
- `contextFidelity`
- `longRunningSupport`
- 当前 round 的 token 压力
- repeated revise 之后的 progress delta
- 是否已经出现 context anxiety

默认上：

- `planner / evaluator` 更偏向 fresh session
- `generator / implementation worker` 更适合在 `continuous` 或 `compact` 下跨 round 复用 session
- 当 coherence 下降时，会自动退到 `reset-handoff`

其中 `compact` 不再只是“名字不同但仍然直接复用原 session”。
它现在会为 worker 生成一份压缩记忆种子，把：

- objective
- 当前 contract
- 最近一次 worker summary
- 最新 feedback artifact
- 关键 artifact 路径

压进新的 compact seed，再配合新的 round prompt 继续跑。
这让 `compact` 和 `continuous` / `reset-handoff` 真正形成了不同的行为层次。

#### 7. Verification v2：rubric、runtime contract、scorecard 一体化

verification contract 现在不再只是一组 loosely-structured checks。
它已经扩成了三个层面：

1. `rubricFamily`
2. `criteria[]`
3. `runtime`

其中：

- `criteria[]` 描述每个判定维度的 `threshold / weight / requiredEvidence`
- `runtime` 描述 live system 如何被验证，包括：
  - `startupCommand`
  - `cwd`
  - `env`
  - `readySignal`
  - `baseUrl`
  - `uiFlows`
  - `apiChecks`
  - `dbChecks`
  - `fixtures`

Evaluator 最终会产出 `scorecard.json`，并且：

- 任一 hard-threshold 不通过，就不能 accept
- final verdict 需要引用失败 criterion，而不是只给模糊的“感觉还不够”

除了 `runtime-contract.json`，runtime-under-test 现在还会生成 `runtime-execution.md`。
这份 guide 会把 startup、ready signal、base URL、UI/API/DB checks 组织成明确的执行协议，让 Validator / Evaluator 不只是“知道 contract 存在”，而是拿到一份更可直接执行的验证说明。

#### 8. `/project`：直接复用同一套 controller / contract / scorecard

`/project` 现在不只是“调用 task engine”而已。
它已经能把 feature 级的 verification / runtime 语义直接送进同一个 managed-task runtime：

- feature 级 verification contract
- feature 级 runtime contract
- feature 级 scorecard / continuation surface

也就是说，`Project harness` 和 `AMA harness` 现在已经共用同一套：

- routing signals
- budget controller
- memory strategy
- verification contract
- continuation artifacts

### 后续演进边界

本轮已经落地到 `022` 的，是 runtime 主路径。

下面这些 Phase 2 能力不再继续塞进 `FEATURE_022`，而是转交给 `v0.9.0` 的 `FEATURE_043` 承接：

- evaluator calibration loop
- pivot / branch management
- harness profiling / ablation
- safe workspace checkpoint surface

### Acceptance criteria

- [x] KodaX has a first-class task abstraction across CLI, REPL, and managed flows
- [x] non-trivial tasks can run with separated generator and evaluator roles
- [x] evaluator can request another explicit execution pass and re-run the relevant worker set within the same managed task
- [x] planner / lead can emit a structured task contract and update it across refinement rounds
- [x] H2 / H3 execute paths include a pre-execution `contract-review` gate before generator / specialist workers start
- [x] H2 / H3 refinement can re-enter planning roles when evaluator feedback invalidates the current contract or plan
- [x] refinement happens as a visible, bounded round loop rather than a hidden retry loop
- [x] QA participation can be adaptive rather than hard-wired for every AMA task, so low-risk implementation runs may omit the final evaluator while review / verify / high-risk runs still require it
- [x] routing exposes `soloBoundaryConfidence` / `needsIndependentQA`, and fallback routing can synthesize the same signals before runtime policy decides the final harness
- [x] task router and protocol-bearing control-plane judges support bounded structured retries, while contract / verdict protocol failures fail closed instead of silently soft-accepting
- [x] round budget is adaptive to harness profile, task risk, and current context footprint instead of fixed for every AMA task
- [x] managed tasks have a unified budget controller with reserve capacity, role-scoped worker ceilings, and budget-aware continuation behavior
- [x] budget disclosure is progressive, so exact round ratios only appear near the execution boundary instead of from the first round
- [x] near-boundary workers can request a small structured budget extension instead of being hard-cut mid-delivery
- [x] worker memory strategy is explicit and can switch among `continuous`, `compact`, and `reset-handoff`
- [x] long-form review and verification handoff is artifact-first (`handoff.json` / `handoff.md` / `result.json`) rather than summary-only transfer
- [x] refinement feedback itself is persisted as artifacts (`feedback.json` / `feedback.md`) and can be handed to the next round as a file-backed coordination surface
- [x] long-running runs emit a continuation checkpoint (`continuation.json`) when they stop in a revise / blocked state, so the next pass has a durable resume surface
- [x] verification contracts can express rubric criteria, runtime-under-test details, and scorecard-driven evaluator judgment
- [x] `/project` can emit feature-level verification / runtime contracts into the same managed-task control plane
- [x] non-terminal worker outputs remain inspectable after task completion
- [x] `/project` can inspect and control managed task state without owning a separate workflow model
- [x] append vs overwrite vs brainstorm can be represented as task-engine decisions
- [x] `019`, `025`, `029`, and `034` integrate under one product execution model
- [x] the architecture shift is traceable in roadmap and design docs through `FEATURE_022`

### Implementation summary

`FEATURE_022` now ships:

- a managed task engine with first-class `TaskContract`, `RoleAssignment`, `WorkItem`, `EvidenceBundle`, and `OrchestrationVerdict` artifacts
- native role-separated execution for `H1`, `H2`, and `H3` harness profiles on top of the existing routing and orchestration substrate
- planner / lead emitted `kodax-task-contract` blocks that are parsed back into the runtime and persisted as `contract.json`
- `contract-review` preflight workers for `H2` / `H3`, so contract quality is reviewed before implementation workers start instead of only after execution has already happened
- an explicit evaluator-led refinement loop for `AMA`, so `Generator` / worker sets can receive structured evaluator feedback and run another bounded pass before acceptance
- H2 / H3 refinement re-entering planner / lead when the evaluator requests a contract or planning correction, instead of forcing every retry through generator-only reruns
- adaptive `quality assurance mode=required|optional`, so low-risk implementation tasks can skip the terminal evaluator while review / verify / high-risk work still keeps evaluator authority
- a signal-driven decision plane where router output and fallback routing both produce `soloBoundaryConfidence`, `needsIndependentQA`, `routingSource`, and `routingAttempts`, and runtime policy then chooses the final harness
- bounded structured retries for router / contract / verdict control-plane decisions, so recoverable protocol failures get retried while unrecoverable ones fail closed
- an adaptive round-budget policy that scales refinement depth by harness profile, risk, brainstorm pressure, project long-running hints, and current context token footprint instead of pinning every task to 3 rounds
- a unified `Harness Budget Controller` with total budget, reserve pool, round ceilings, and worker-scoped soft/hard budgets
- `green / yellow / orange / red` progressive budget disclosure so workers only see precise round pressure when they actually need to converge
- `kodax-budget-request` extension blocks for near-boundary runs that need a very small additional execution window before they can safely conclude
- CLI, REPL, Plan Mode, and `/project` execution surfaces routed through the same managed-task entry point
- `/project` persistence for managed task state under `.agent/project/managed-task.json` and `.agent/project/managed-tasks/`
- memory-strategy-aware worker sessions (`continuous`, `compact`, `reset-handoff`) so builder-heavy roles can reuse context when it is still coherent, fall back to compact memory seeds when context pressure rises, and hand off cleanly when it is not
- prefixed text/thinking/tool observability for non-terminal workers so planner and evaluator verification remains visible to the user instead of collapsing into only tool/error events
- managed-task transcript retention for non-terminal outputs, so generator / planner / validator results remain inspectable after final evaluator收口
- session-persisted UI transcript snapshots for managed-task non-terminal outputs, so reload / resume does not drop planner / generator / validator history
- artifact-first dependency handoff with `handoff.json` and `handoff.md` per worker, while prompts now point downstream agents to those files plus `result.json` instead of relying on long embedded summaries
- file-backed refinement feedback artifacts (`feedback.json` / `feedback.md`) and a root-level `round-history.json` ledger so longer AMA runs keep a durable round-by-round record
- root-level `continuation.json` checkpoints that capture the latest feedback artifact, round history, and suggested next prompt when a managed task stops before acceptance
- verification v2 contracts with `rubricFamily`, `criteria[]`, `runtime`, `runtime-contract.json`, `runtime-execution.md`, and `scorecard.json`, so Evaluator judgment is threshold-backed rather than purely narrative and can point to a concrete runtime execution guide
- review-specific final-answer semantics where `Evaluator` delivers the final user-facing review, not a meta-review of the generator
- hard runtime enforcement of the evaluator closing contract, so missing `kodax-task-verdict` blocks now block the task instead of silently counting as acceptance
- explicit session layering: resumable user conversations remain the product-facing session surface, while managed-task worker sessions are internal execution scope only and do not participate in ordinary `kodax -c` recovery semantics
- project-harness integration that emits feature-scoped verification / runtime contracts into the same task-engine artifacts and continuation surface
- roadmap-visible completion tracking for the architecture shift under `FEATURE_022`

## FEATURE_019: Session Tree, Checkpoints, and Rewindable Task Runs

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

- `packages/coding/src/provider-policy.ts` already turns provider capability truth into routing constraints and harness-related warnings or blocks.

### Implementation plan

#### Phase 1: formalize the task-intelligence contract

Create a first-class routing result that can explain, in a structured and inspectable way:

- task classification
- complexity tier
- work intent (`append`, `overwrite`, `new`)
- brainstorm requirement
- selected harness profile
- provider-policy constraints that affected the decision

The immediate goal is to make routing output useful to both humans and downstream orchestration code without forcing hidden heuristics to stay buried in prompt text.

#### Phase 2: unify intake, provider policy, and harness selection

Move from "task routing" plus separate "provider policy" toward one intake pipeline that produces a single decision bundle:

1. normalize the user request and recent execution evidence

- coding-agent prompt assembly

- future `/project` control surfaces under `022`
- diagnostics or status surfaces that need to explain why a harness was chosen

The key requirement is that harness selection becomes explicit product state rather than an inferred side effect of prompt wording or provider hints.

#### Phase 4: adaptive retry and refinement policy

Extend the existing auto-reroute path so `025` also owns bounded adaptation after the first pass:

- escalate to planning or investigation when evidence is weak
- downgrade when the request is actually lightweight
- trigger brainstorm/discovery when ambiguity remains unresolved
- map repeated failure patterns to bounded repair playbooks

This should remain policy-driven and inspectable, not a hidden retry loop.

### Initial delivery slice for v0.7.x

4. expand tests so routing decisions are explainable, deterministic, and provider-aware

### Redesign Summary

#### Positioning

FEATURE_034 is no longer scoped as a narrow registry cleanup, and it is also not a full "pi-mono UI/plugin platform". It becomes the headless programmable runtime for KodaX.

It should make KodaX usable in three modes with one contract:

- as the built-in REPL/CLI product
- as an embeddable library for custom agent apps
- as a local extension host for rapid frontline experimentation

The runtime has four layers:

- **Extension Runtime** handles extension loading, hot reload, registration, lifecycle, disposables, failure isolation, provenance, and diagnostics.
- **Capability Runtime** handles capability discovery, description, execution, reading, prompt retrieval, and structured result transport.
- **Runtime Control Surface** lets extensions participate in a running agent session by influencing queued follow-ups, session-scoped state, active tools, and model/thinking behavior.
- **Host Adapters** expose optional command, skill-path, reload, and diagnostics surfaces without making REPL/TUI concerns part of the core runtime.

#### Why the scope upgrade is needed

- The current tool system still has three parallel sources of truth: handler registration, tool schema exposure, and required-params metadata.
- `CommandRegistry` and skill `pluginPaths` already expose partial extension seams, but there is no runtime that unifies them.
- `KodaXEvents` behaves like a host/UI callback surface and should remain a host adapter, not become the extension event bus.
- KodaX now needs to be programmable into custom agent apps, not only customized around the built-in REPL.
- Frontline users need fast local JS/TS extension authoring and hot reload.
- Extensions need to participate at selected critical loop nodes and drive session, state, and model behavior.
- FEATURE_035, FEATURE_038, FEATURE_018, FEATURE_028, and FEATURE_030 all depend on a stronger substrate than the earlier narrow 034 draft.
- The earlier "avoid TypeScript extensions entirely" posture was useful when KodaX was still collapsing surface area, but it is now too restrictive for the product direction.

#### Boundary with other roadmap features

| Feature | What 034 now provides | What remains in that feature |
|---------|-----------------------|------------------------------|
| FEATURE_019 | session-participant APIs, persisted extension state/records, queued follow-ups | tree persistence, rollback semantics, branch summaries, `/tree`, `/fork` |
| FEATURE_029 | provider registration provenance, runtime diagnostics, explicit hook metadata | semantic compatibility matrix, native-vs-bridge warnings/gates, adapter parity tests |
| FEATURE_035 | capability runtime, structured result transport, extension loading/hot reload, runtime control APIs | MCP catalog, transport/runtime, trust model, `/mcp` UX |
| FEATURE_038 | tool override semantics, built-in tool factories, diagnostics, explicit hook slots | sandbox policy, process isolation, `/sandbox`, provider selection for sandbox enforcement |
| FEATURE_018 | capability/result substrate for repository intelligence providers | repository intelligence indexing, storage, refresh policy, and query UX |
| FEATURE_028 | evidence-carrying result shape and capability slots | first-class retrieval, context, and evidence implementations |
| FEATURE_022 | single-runtime participation primitives | multi-agent planning, delegation, aggregation, control plane |
| FEATURE_030 | headless, surface-agnostic engine and host adapters | IDE/desktop/web delivery surfaces and transport |

#### Runtime boundary

The main runtime should live in `packages/coding`, not `packages/agent`.

- `packages/agent` remains the smaller generic foundation for session, compaction, and token management.
- `packages/coding` owns `runKodaX()`, tool exposure, and the agent loop, so 034 belongs there.

#### Design principles

- **Headless and library-first**. The runtime should be usable without REPL, Ink, or any specific UI host.
- **Few strong primitives**. KodaX should expose a small number of powerful, typed contracts rather than many ad hoc escape hatches.
- **Explicit mutability**. Mutable participation is allowed only through named hook contracts and runtime control APIs, not through a universal interception chain.
- **Definition-first registries**. Tools, capabilities, and provider registrations should each have a single atomic source of truth.
- **Provenance first**. Override origin, hook participation, and extension failures must be explainable in diagnostics.

- REPL integration is treated as a host adapter over the same runtime, not the primary extension surface.
- UI/editor/widget/theme APIs, universal middleware, and generic `setEditor()`-style customization are explicitly out of scope for FEATURE_034.

When reviewing or implementing against FEATURE_034, use the sections above as the source of truth:

- `Positioning`
- `Scope that should now move into FEATURE_034`
- `Scope that should not move into FEATURE_034`
- `Runtime control and extension API`
- `Event bus and explicit mutable hook points`
- `Acceptance criteria rewrite`

## Cross-feature decisions locked in v0.7.0

1. `Project Mode` stops being the main future-facing abstraction.
2. `--team` stops being the future-facing multi-agent abstraction.
3. `034` remains and is treated as prerequisite substrate, not competing architecture.
4. `025` becomes task intelligence for every request, not project-only logic.
5. `022` becomes the canonical multi-agent control plane.
