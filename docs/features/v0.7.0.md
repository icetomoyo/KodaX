# v0.7.0 Feature Designs

> Version: `v0.7.0`
>
> Status: Planned
>
> Theme: Engine foundation for the adaptive task system

---

## Overview

`v0.7.0` is the architectural reset release.

Its job is to install the new core:

- task-first persistence
- task abstraction and lifecycle
- native multi-agent control
- automatic harness routing
- provider-aware policy
- runtime substrate that future capabilities can build on

This release intentionally widens `FEATURE_022` so it becomes the umbrella feature for the new execution model. `FEATURE_019`, `FEATURE_025`, `FEATURE_029`, and `FEATURE_034` are enabling slices that make the widened `022` real.

### Dependency order

```text
FEATURE_034 runtime substrate
  -> FEATURE_029 provider policy
  -> FEATURE_019 task lineage and checkpoints
  -> FEATURE_025 harness router
    -> FEATURE_022 adaptive task engine + native multi-agent control plane

FEATURE_026 roadmap hygiene supports the release
```

## FEATURE_022: Adaptive Task Engine and Native Multi-Agent Control Plane

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

`FEATURE_022` is no longer just the "multi-agent control plane" slice. It becomes the umbrella feature that owns the product-level shift from:

- command-led execution
- explicit mode switching
- project-only long-running semantics

to:

- task-first execution
- native multi-agent role separation for non-trivial work
- evidence-driven completion
- `/project` as a control surface over managed task state

This is the feature that should carry the architecture change we discussed in this thread.

### Why `022` should carry this change

- it was already the closest existing feature to the new direction
- the new architecture is fundamentally about execution shape, not only storage or prompts
- using `022` as the umbrella keeps roadmap continuity instead of creating a disconnected new ID
- it gives the architecture shift one clear owner inside feature planning

### Owned responsibilities

`FEATURE_022` owns:

- the first-class `Task` abstraction as the product's primary execution unit
- the managed-task lifecycle and state transitions
- role separation for `Lead`, `Planner`, `Generator`, `Evaluator`, and specialist workers
- the product contract that every non-trivial task is executed and judged through separated responsibilities
- the migration path that internalizes `Project Mode` into the task engine
- the integration contract across intake, routing, persistence, provider policy, and runtime substrate

### It does not own

- persistence primitives and checkpoint storage
- routing heuristics and harness scoring policy
- provider capability truth
- extension and capability runtime loading
- roadmap validation mechanics

Those remain in `019`, `025`, `029`, `034`, and `026`.

### Relationship to other `v0.7.0` features

| Feature | How it supports `022` |
|---|---|
| `FEATURE_019` | supplies durable task lineage, checkpoints, and resumable state |
| `FEATURE_025` | decides which task/harness shape should be used |
| `FEATURE_029` | constrains task execution shape using provider truth |
| `FEATURE_034` | provides the runtime substrate the task engine runs on |
| `FEATURE_026` | keeps the roadmap and design docs aligned while the architecture shifts |

### Role model

`022` still contains the original control-plane responsibilities:

- role assignment
- task decomposition
- result synthesis
- evaluator independence

Default role sets remain:

| Complexity | Default role set |
|---|---|
| `L0` | one direct agent |
| `L1` | generator + evaluator |
| `L2` | planner + generator + evaluator |
| `L3` | lead + planner + one or more workers + evaluator |

### Product commitments

1. Non-trivial tasks default to multi-agent execution.
2. Single-agent execution remains only as a low-complexity fallback.
3. `--team` is no longer the canonical product surface for multi-agent work.
4. `/project` stops being a separate future architecture and becomes a task-engine control surface.
5. Completion must remain compatible with evaluator-led, evidence-driven judgment.

### Detailed direction

This feature should carry forward the useful parts of the older multi-agent design instead of losing them during the roadmap rewrite.

The intended production-grade target is:

- native subagents rather than comma-separated parallel prompts
- explicit task decomposition and worker assignment
- clearer role and permission boundaries among agents
- evidence aggregation across workers
- stable cooperation with task persistence, evaluation, and future multi-surface workflows

The old `--team` path remains useful as orchestration plumbing, but it is not the target architecture because it lacks:

- role semantics
- contract negotiation
- evaluator authority
- durable task truth
- explicit evidence synthesis

### Boundary with `FEATURE_034`

`FEATURE_034` provides single-runtime programmability: extensions can participate in one running session, influence model/tool state, and queue follow-up work.

`FEATURE_022` still owns:

- task decomposition into subagents
- role and permission boundaries among agents
- evidence aggregation across workers
- a control plane that survives across surfaces

`034` should make orchestration possible to build on top of, but it should not collapse multi-agent control into the extension runtime itself.

### Suggested internal model

Recommended internal orchestration units:

- `TaskContract`
- `RoleAssignment`
- `WorkItem`
- `EvidenceBundle`
- `OrchestrationVerdict`

This keeps `022` centered on the execution model, while `019` and the DD own persistence details.

### Acceptance criteria

- [ ] KodaX has a first-class task abstraction across CLI, REPL, and managed flows
- [ ] non-trivial tasks can run with separated generator and evaluator roles
- [ ] `/project` can inspect and control managed task state without owning a separate workflow model
- [ ] append vs overwrite vs brainstorm can be represented as task-engine decisions
- [ ] `019`, `025`, `029`, and `034` integrate under one product execution model
- [ ] the architecture shift is traceable in roadmap and design docs through `FEATURE_022`

## FEATURE_019: Session Tree, Checkpoints, and Rewindable Task Runs

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

KodaX currently treats session history and long-running work as mostly linear. `FEATURE_019` upgrades persistence from flat history into task-aware lineage with checkpoints and rewindable runs.

This feature is no longer only about chat tree visualization. It becomes the persistence backbone for managed tasks.

### Goals

1. Persist task lineage, not just flat message logs.
2. Add checkpoints that can be inspected and revisited safely.
3. Support branch-and-continue semantics for managed tasks.
4. Provide the storage model that `FEATURE_022` and `FEATURE_025` need.

### Scope

- task-aware session tree
- checkpoint index
- branch metadata
- replay-safe task resume points
- compatibility with old session storage

### Non-goals

- full automatic rollback engine
- git-history replacement
- hidden branch search system

### Design direction

Recommended persisted units:

- `session-tree/nodes.jsonl`
- `checkpoints/index.jsonl`
- task-local run metadata

Checkpoint payloads should reference artifacts and evidence rather than duplicating entire histories.

### Detailed design notes retained

The earlier draft for this feature contained implementation ideas that are still worth keeping attached to `019`.

#### Tree-shaped session entries

Useful retained direction:

- move from flat message arrays to tree-shaped entries with `id` / `parentId`
- treat messages, compactions, branch summaries, and labels as session entries
- keep branch navigation separate from git history

Recommended entry families:

- `message`
- `compaction`
- `branch_summary`
- `label`

#### Session manager responsibilities

Still useful as a target:

- append entries with identity and parent linkage
- track active leaf
- traverse branches and children
- build the effective context for the active branch
- support branch-and-continue semantics without corrupting the active path

#### Tree navigation UX

The earlier `/tree` design is still valuable implementation history:

- ASCII tree visualization
- active-path highlighting
- optional filtering and folding
- label editing
- branch-summary preview before switching

This remains compatible with the current decision to make `/project` a control surface rather than the center of the product.

#### Git checkpoint extension

Retained idea:

- git checkpoints should remain optional
- they may be implemented as a companion extension rather than core session semantics
- session lineage and code-state restoration should stay related but not fully fused

This keeps `019` focused on task/session truth while allowing code-state recovery to evolve more carefully.

#### Historical draft snapshot

The earlier design carried additional implementation detail that should remain preserved:

##### Persistence and entry families

Useful retained direction:

- move from flat message arrays to entries with `id` and `parentId`
- support entry families such as `message`, `compaction`, `branch_summary`, and `label`
- keep JSONL-friendly append semantics even after the model becomes tree-shaped

##### Session manager responsibilities

Useful retained target behaviors:

- append entries with identity and timestamps
- track the current leaf
- walk branch-to-root paths
- enumerate child branches
- build effective context for the active branch
- support branch-and-continue semantics cleanly

##### `/tree` control surface

Retained implementation ideas:

- ASCII tree visualization
- active-path highlighting
- optional filters such as user-only, labeled-only, or no-tools
- folding and unfolding
- label editing
- preview of branch summaries before switching

##### `/fork` semantics

Still useful:

- allow creating a new session from a selected tree node
- treat forked sessions as a deliberate export of one branch, not a replacement for in-place branching

##### Suggested implementation phases from the older draft

1. define entry types and session manager
2. add migration from old linear session format
3. add `/tree` UI and branch navigation
4. add optional branch summaries
5. add optional git-checkpoint companion behavior

##### Why this still matters

The architecture now frames `019` as task-aware lineage rather than only chat-tree UX, but the older session-tree mechanics are still the most concrete design material for making that lineage real.

### Acceptance criteria

- [ ] task runs can persist branchable lineage
- [ ] checkpoints can be created and revisited without corrupting active state
- [ ] long-running task resume points are durable
- [ ] old linear sessions can be migrated into the new model

## FEATURE_025: Adaptive Task Intelligence and Harness Router

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_025` is no longer "Adaptive Project Intelligence". It becomes the general task-intelligence layer for all requests and the main intake/routing slice underneath `FEATURE_022`.

This feature decides:

- what kind of task the user asked for
- how complex it is
- whether the request is append, overwrite, or new work
- whether brainstorm is needed
- which harness profile to use

### Why this redesign matters

The user should not have to think in terms of:

- `/project`
- brainstorm mode
- project mode
- manual multi-agent mode

The router should internalize those decisions.

### Scope

- intake normalization
- complexity scoring
- append vs overwrite inference
- brainstorm trigger policy
- harness-profile selection
- escalation and downgrade rules

### Non-goals

- cross-agent execution itself
- provider runtime registration
- evaluator implementation

### Default harness profiles

- `H0_DIRECT`
- `H1_EXECUTE_EVAL`
- `H2_PLAN_EXECUTE_EVAL`
- `H3_MULTI_WORKER`

### Detailed direction retained from the previous design

Although `025` was broadened from project intelligence into task intelligence, several detailed ideas from the previous design are still valuable and should remain attached to this feature.

#### Semantic rule compilation

The router should be able to compile stronger repo/task rules from:

- `AGENTS.md`
- `docs/HLD.md`
- `docs/ADR/**/*.md`
- workspace package structure
- CodeWiki summaries and invariants

Expected outputs remain declarative and bounded:

- architecture constraints
- allowed exceptions
- completion checklists
- repair playbooks

#### Declarative refiner behavior

`025` should not only classify tasks at intake time. It should also decide how the system adapts when work stalls or fails.

Useful retained direction:

- failure-pattern to playbook mapping that can evolve in config
- repo-specific repair actions
- bounded refinement with schema validation

#### Evaluator-facing task interpretation

The router and evaluator should share a clearer interpretation layer for:

- legality
- progress confidence
- evidence completeness
- quality delta
- retry or stall risk

This is not a hidden score machine. It is an inspectable explanation layer that helps KodaX decide whether to continue, retry, brainstorm, or escalate.

### Explicit non-goals carried forward

- no branch-search system
- no replay verifier
- no automatic rollback engine
- no executable policy generation

### Acceptance criteria

- [ ] every request flows through explicit intake and harness selection
- [ ] KodaX can infer append vs overwrite with inspectable reasons
- [ ] ambiguous requests can trigger brainstorm/discovery automatically
- [ ] harness profiles are selected without requiring users to choose a mode first

## FEATURE_029: Provider Capability Transparency and Harness Policy

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

KodaX currently exposes native providers and bridge-backed providers in one product surface, but their semantics differ. The new task engine cannot make safe routing decisions without explicit provider truth.

### Goals

1. Distinguish native vs bridge providers clearly.
2. Surface real capability differences that affect routing.
3. Prevent false parity in multi-agent and long-running flows.
4. Feed real provider constraints into `FEATURE_025`.

### Capability examples

- reasoning control
- tool-calling fidelity
- context reliability
- multimodal support
- MCP or capability integration behavior

### Relationship to `FEATURE_034`

`034` provides runtime slots and provider/capability provenance plumbing.

`029` defines what those capabilities mean to product behavior:

- warnings
- gating
- harness policy
- user-visible truth

### Detailed direction

`029` should make semantic differences explicit at three levels:

#### Capability matrix

The product should keep an explicit compatibility matrix for areas such as:

- context handling fidelity
- reasoning control
- MCP/capability support
- tool-calling behavior
- session behavior
- multimodal support

#### Runtime warnings and hard gates

When unsupported semantics are requested, KodaX should not silently degrade. It should warn or block in places such as:

- harness selection
- `/provider` or equivalent status surfaces
- long-running task startup
- multimodal/evidence-heavy flows

#### Adapter parity tests

Bridge-backed providers should have explicit parity checks for:

- multi-turn behavior
- context fidelity
- reasoning semantics
- MCP/capability loss
- session behavior

This keeps provider switching honest instead of suggesting false equivalence.

### User-facing surfaces

Recommended product surfaces for `029` include:

- clearer config truth for native vs bridge providers
- richer `/provider` output
- provider capability snapshots that can be attached to task decisions
- explicit notes when routing is constrained by provider semantics

### Acceptance criteria

- [ ] native and bridge providers are distinguishable in user-visible surfaces
- [ ] provider capability snapshots can influence harness selection
- [ ] unsupported semantics are warned or gated explicitly
- [ ] routing no longer assumes false provider equivalence

## FEATURE_034: Extension and Capability Runtime

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_034` was already heavily redesigned before this roadmap reset. That redesign remains valid.

There is no conflict between the updated `034` direction and the new task-engine architecture. In fact, `034` is the required runtime substrate that the new control plane builds on.

### What stays true

`034` remains responsible for:

- extension loading
- capability registration
- tool override semantics
- diagnostics and provenance
- structured capability result transport
- host-neutral runtime participation

### What it must not absorb

To avoid architectural drift, `034` must not become:

- the multi-agent control plane
- the harness router
- the owner of completion semantics
- a universal middleware system that swallows task logic

### Why it fits the new design

The new architecture needs a strong runtime substrate so that later capabilities such as:

- search
- MCP
- sandbox
- future task-aware tools

can integrate cleanly without rewriting the engine.

### Product boundary

`034` is about single-runtime programmability.

It enables:

- one runtime to host extensions and capabilities
- later features to register richer tool or capability families
- surfaces to share one execution substrate

It does not decide:

- which roles to run
- which harness profile to use
- whether a task is complete

### Acceptance criteria

- [ ] KodaX has a stable host-neutral extension and capability substrate
- [ ] tool override and provenance are explicit
- [ ] future capability packages can build on `034` without patching core internals
- [ ] `034` remains clearly below `022` and `025` in responsibility

### Restored Detailed Design Record

The following detailed draft was restored from the pre-reset `v0.7.0` design because it captures valuable implementation thinking and should remain part of the feature history.

> 2026-03 redesign note: the normative design for FEATURE_034 is the "Redesign Summary" below. The older draft text in this section is retained temporarily as historical context and should not be treated as the target architecture.

### Redesign Summary

#### Positioning

FEATURE_034 is no longer scoped as a narrow registry cleanup, and it is also not a full "pi-mono UI/plugin platform". It becomes the headless programmable runtime for KodaX.

It should make KodaX usable in three modes with one contract:

- as the built-in REPL/CLI product
- as an embeddable library for custom agent apps
- as a local extension host for rapid frontline experimentation

The runtime has four layers:

- **Extension Runtime** handles extension loading, hot reload, registration, lifecycle, disposables, failure isolation, provenance, and diagnostics.
- **Capability Runtime** handles capability discovery, description, execution, reading, prompt retrieval, and structured result transport.
- **Runtime Control Surface** lets extensions participate in a running agent session by influencing queued follow-ups, session-scoped state, active tools, and model/thinking behavior.
- **Host Adapters** expose optional command, skill-path, reload, and diagnostics surfaces without making REPL/TUI concerns part of the core runtime.

#### Why the scope upgrade is needed

- The current tool system still has three parallel sources of truth: handler registration, tool schema exposure, and required-params metadata.
- `CommandRegistry` and skill `pluginPaths` already expose partial extension seams, but there is no runtime that unifies them.
- `KodaXEvents` behaves like a host/UI callback surface and should remain a host adapter, not become the extension event bus.
- KodaX now needs to be programmable into custom agent apps, not only customized around the built-in REPL.
- Frontline users need fast local JS/TS extension authoring and hot reload.
- Extensions need to participate at selected critical loop nodes and drive session, state, and model behavior.
- FEATURE_035, FEATURE_038, FEATURE_018, FEATURE_028, and FEATURE_030 all depend on a stronger substrate than the earlier narrow 034 draft.
- The earlier "avoid TypeScript extensions entirely" posture was useful when KodaX was still collapsing surface area, but it is now too restrictive for the product direction.

#### Boundary with other roadmap features

| Feature | What 034 now provides | What remains in that feature |
|---------|-----------------------|------------------------------|
| FEATURE_019 | session-participant APIs, queued follow-ups, session-scoped extension state | tree persistence, rollback semantics, branch summaries, `/tree`, `/fork` |
| FEATURE_029 | provider registration provenance, runtime diagnostics, explicit hook metadata | semantic compatibility matrix, native-vs-bridge warnings/gates, adapter parity tests |
| FEATURE_035 | capability runtime, structured result transport, extension loading/hot reload, runtime control APIs | MCP catalog, transport/runtime, trust model, `/mcp` UX |
| FEATURE_038 | tool override semantics, built-in tool factories, diagnostics, explicit hook slots | sandbox policy, process isolation, `/sandbox`, provider selection for sandbox enforcement |
| FEATURE_018 | capability/result substrate for knowledge providers | CodeWiki indexing, storage, update policy, query UX |
| FEATURE_028 | evidence-carrying result shape and capability slots | first-class search/fetch/code-search implementations |
| FEATURE_022 | single-runtime participation primitives | multi-agent planning, delegation, aggregation, control plane |
| FEATURE_030 | headless, surface-agnostic engine and host adapters | IDE/desktop/web delivery surfaces and transport |

#### Runtime boundary

The main runtime should live in `packages/coding`, not `packages/agent`.

- `packages/agent` remains the smaller generic foundation for session, compaction, and token management.
- `packages/coding` owns `runKodaX()`, tool exposure, and the agent loop, so 034 belongs there.

#### Design principles

- **Headless and library-first**. The runtime should be usable without REPL, Ink, or any specific UI host.
- **Few strong primitives**. KodaX should expose a small number of powerful, typed contracts rather than many ad hoc escape hatches.
- **Explicit mutability**. Mutable participation is allowed only through named hook contracts and runtime control APIs, not through a universal interception chain.
- **Definition-first registries**. Tools, capabilities, and provider registrations should each have a single atomic source of truth.
- **Provenance first**. Override origin, hook participation, and extension failures must be explainable in diagnostics.

#### Event bus and explicit mutable hook points

The event bus should stay typed and primarily observe-only:

```typescript
type ExtensionEventMap = {
  'extension:loaded': { extensionId: string; source: string };
  'extension:error': { extensionId: string; error: Error };

  'session:start': { sessionId: string; cwd: string; gitRoot?: string };
  'session:end': { sessionId: string };

  'turn:start': { sessionId: string; turnId: string; userText: string };
  'turn:end': { sessionId: string; turnId: string; stopReason?: string };

  'tool:before': {
    toolName: string;
    input: Record<string, unknown>;
    source: 'builtin' | 'extension' | 'capability';
  };
  'tool:after': {
    toolName: string;
    result: CapabilityResult;
    durationMs: number;
  };
  'tool:error': {
    toolName: string;
    input: Record<string, unknown>;
    error: Error;
  };

  'provider:selected': { provider: string; model?: string };
  'provider:error': { provider: string; error: Error };
  'provider:rate-limit': { provider: string; attempt: number; delayMs: number };

  'compaction:before': { messageCount: number; estimatedTokens: number };
  'compaction:after': { tokensBefore: number; tokensAfter: number };

  'capability:search': { providerId: string; query: string; kind?: CapabilityKind };
  'capability:describe': { providerId: string; capabilityId: string };
  'capability:invoke': { providerId: string; capabilityId: string; kind: CapabilityKind };

```

Mutable participation should live in a separate, explicit hook layer:

```typescript
type ExtensionHookMap = {
  'tool:prepare': {
    toolName: string;
    input: Record<string, unknown>;
    replaceInput(next: Record<string, unknown>): void;
    replaceTool(nextToolName: string): void;
    block(reason: string): void;
  };
  'provider:prepare': {
    provider: string;
    model?: string;
    request: ProviderRequestEnvelope;
    replaceRequest(next: ProviderRequestEnvelope): void;
    replaceModel(nextModel: string): void;
    block(reason: string): void;
  };
  'turn:settle': {
    sessionId: string;
    turnId: string;
    stopReason?: string;
    queueUserMessage(message: string | KodaXMessage): void;
    setModelSelection(next: { provider?: string; model?: string }): void;
    setThinkingLevel(level: ReasoningLevel): void;
  };
  'session:hydrate': {
    sessionId: string;
    getState<T = unknown>(key: string): T | undefined;
    setState(key: string, value: unknown): void;
  };
};
```

This is intentionally **not** a universal middleware chain:

- most lifecycle notifications remain observe-only
- mutable behavior exists only at a small set of named contracts
- each mutable hook must have explicit merge/order diagnostics

`KodaXEvents` should continue to exist, but explicitly as a host callback adapter for REPL/CLI/UI concerns.

#### Implementation order

1. **Registry cleanup first**
   - convert local tools to atomic definitions
   - infer required params from schema
   - add same-name override semantics
   - expose built-in tool factories
   - remove `KODAX_TOOL_REQUIRED_PARAMS` as a separate truth source
2. **Capability and runtime control second**
   - define structured capability result transport
   - add capability/model provider registries
   - add runtime control APIs for session/tool/model participation
3. **Events and explicit hooks third**
   - add typed observe-only lifecycle events
   - add the small mutable hook layer with diagnostics
   - keep host callbacks separate
4. **Loader and hot reload fourth**
   - local JS/TS loading
   - disposables, provenance, diagnostics
   - host-agnostic reload support
5. **User-facing host adapters last**
   - `--extension`
   - config-based loading
   - REPL `/reload`
   - runtime diagnostics command/output

#### Acceptance criteria rewrite

- [ ] local tools are registered through a single atomic definition
- [ ] required params are derived from schema rather than maintained in a separate table
- [ ] same-name tool override and built-in tool factories are available to extensions
- [ ] extensions can register local tools, capability providers, and model providers
- [ ] extensions can observe lifecycle events and participate through the explicit mutable hook layer
- [ ] extensions can influence queued follow-ups, session-scoped state, active tools, and model/thinking behavior within a running session
- [ ] local JS/TS extensions can load and hot reload without REPL-specific coupling
- [ ] 034 runtime can initialize in both REPL and non-REPL execution paths
- [ ] host surfaces such as commands and skill paths are optional adapters, not core runtime requirements
- [ ] `KodaXEvents` remains usable but is clearly separated from the extension event bus
- [ ] extension failures, overrides, and hook participation surface in diagnostics without breaking the core agent loop

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### 1. 概述

仿照 pi-mono 的 Extension API，为 KodaX 设计统一的扩展接口。当前 KodaX 有 5 个分散的扩展面（Tool Map、CommandRegistry、Skill hooks、Provider registry、Shell hooks），但没有统一的 Extension API。此 feature 将这些分散的机制统一为一个 `KodaXExtensionAPI` 接口。

**核心目标**:
1. **统一 Extension 接口** — `registerTool`, `registerCommand`, `on(event)`, `registerProvider`, `setEditor`
2. **Agent 生命周期事件** — turn:start, turn:end, tool:before, tool:after, compaction:before/after
3. **原子化 Tool 注册** — `registerTool(name, handler, schema)` 替代分离的 Map + Array
4. **Plugin 加载** — 支持 `--extension ./my-ext.js` 加载扩展
5. **激活 Plugin source** — Skill 系统预留的 `'plugin'` 路径启用

**背景**:
- KodaX 当前扩展面：`registerTool()` (Map)、`CommandRegistry` (Class)、YAML frontmatter hooks (Shell)、`registerCustomProviders()` (Config)、`KodaXEvents` (Callbacks)
- Pi 的 Extension API 提供 28+ event hooks，一个 `api` 对象即可注册所有扩展
- KodaX Skill 系统已预留 `'plugin'` source 但 `pluginPaths: []` 未实现
- 当前 hooks 只能执行 shell 命令，无法编写 JS/TS 级别的 hook

**Inspired by**: [pi-mono Extension System](https://github.com/badlogic/pi-mono)

### 2. 影响范围

**需要创建的文件**:
- `packages/agent/src/extension/api.ts` — KodaXExtensionAPI 接口定义
- `packages/agent/src/extension/loader.ts` — Extension 加载器（JS 文件解析）
- `packages/agent/src/extension/events.ts` — Agent 生命周期事件系统

**需要修改的文件**:
- `packages/coding/src/tools/registry.ts` — 统一 registerTool 函数
- `packages/agent/src/agent.ts` — 注入生命周期事件
- `packages/ai/src/providers/registry.ts` — 开放 provider 注册
- `packages/skills/src/types.ts` — 激活 plugin source
- `src/kodax_cli.ts` — 添加 `--extension` CLI 参数

### 3. 技术方案

#### 3.1 KodaXExtensionAPI 接口

```typescript
// packages/agent/src/extension/api.ts

interface KodaXExtensionAPI {
  // Tool registration (atomic)
  registerTool(name: string, handler: ToolHandler, schema: KodaXToolDefinition): void;
  unregisterTool(name: string): void;

  // Command registration
  registerCommand(def: CommandDefinition): void;
  unregisterCommand(name: string): void;

  // Provider registration
  registerProvider(name: string, factory: () => KodaXBaseProvider): void;

  // Lifecycle events
  on(event: string, handler: (context: EventContext) => void | Promise<void>): void;
  off(event: string, handler: Function): void;

  // Editor integration
  setEditor(name: string): void;

  // Utility
  logger: ExtensionLogger;
  config: Readonly<Record<string, unknown>>;
}
```

#### 3.2 Agent 生命周期事件

```typescript
type AgentEvent =
  | 'session:start'
  | 'session:end'
  | 'turn:start'
  | 'turn:end'
  | 'tool:before'
  | 'tool:after'
  | 'compaction:before'
  | 'compaction:after'
  | 'provider:rate-limit'
  | 'provider:error';
```

#### 3.3 Extension 加载

```typescript
// --extension ./my-ext.js
export default function(api: KodaXExtensionAPI) {
  api.registerTool('my-tool', handler, schema);
  api.on('tool:before', async (ctx) => { /* ... */ });
  api.registerProvider('my-provider', factory);
}
```

### 4. 接口契约

```
kodax --extension ./my-ext.js
kodax --extension ./ext1.js --extension ./ext2.js
```

### 5. 实现步骤

**Phase 1: 接口定义**
1. 定义 `KodaXExtensionAPI` 接口
2. 定义事件类型和上下文
3. 创建 Extension 加载器

**Phase 2: 统一注册**
1. 重构 `registerTool` 为原子化函数
2. 暴露 CommandRegistry 注册
3. 开放 Provider 注册

**Phase 3: 生命周期事件**
1. 在 agent.ts 中注入事件触发点
2. 实现 middleware 链
3. 支持异步事件处理

**Phase 4: CLI 集成**
1. 添加 `--extension` 参数
2. 加载并初始化扩展
3. 错误处理和隔离

### 6. 验收标准

- [ ] 可以通过 `--extension` 加载 JS 扩展
- [ ] 扩展可以注册自定义工具并被 LLM 调用
- [ ] 扩展可以注册自定义命令并出现在 `/help` 中
- [ ] 扩展可以订阅 agent 生命周期事件
- [ ] 扩展可以注册新的 provider
- [ ] 扩展错误不影响核心功能
- [ ] Skill 的 `'plugin'` source 可以被激活

### 7. 设计决策

1. **Extension 格式**: ESM `.js` 文件
   - `export default function(api)` 签名
   - 不支持 TypeScript 直接加载（需要预编译）
   - 与 Pi 的 Extension 格式对齐

2. **Plugin 隔离**: 轻量级错误隔离
   - Extension 加载失败只警告，不崩溃
   - 不做沙箱隔离（信任用户自己的扩展）

3. **Implementation priority**: Phase 1 + 2 优先
   - 先实现工具/命令/Provider 注册
   - 生命周期事件可以后续迭代
