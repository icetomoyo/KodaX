# v0.7.38 Feature Designs

> 中文标题：`v0.7.38 Feature 设计`
>
> Version: `v0.7.38`
>
> Status: Planned (not yet released)
>
> Theme: Soft-Pause Protocol — 在 KodaX runner loop 引入 "tool 边界软暂停" 中间档信号，填补现有 `abort`（丢整个 turn 上下文）与 `pending-inputs`（要等到 turn 自然结束才注入）之间的 UX 缺口，让用户在长任务执行中能**轻量插话改方向**而不需要丢失 Planner 已花费的 token。

---

## 总览

KodaX 当前 REPL 只有两档 runner-level 信号：

| 操作 | 实现 | 行为 | 代价 |
|---|---|---|---|
| **不按键** | / | 按原计划继续 | 0 |
| **打字 + Enter** | [`pending-inputs.ts`](../../packages/repl/src/ui/utils/pending-inputs.ts)（`MAX_PENDING_INPUTS=5`）| 排队等当前 LLM turn 自然结束才注入 | 长 tool（如 `npm test` 90s+）时延迟到爽 |
| **Double-ESC** | `AbortController.abort()`（[InkREPL.tsx:3996-4047](../../packages/repl/src/ui/InkREPL.tsx#L3996)）| 整 turn 作废 | 丢上下文 + Planner 已花的 2000-6000 token 浪费 |

**真实痛点场景**：用户在 H2 任务里跑了一个 90 秒的 `npm test`，跑到一半想加一句"测试要包含 token rotation 场景"。当前两个选择都不好——`abort` 把 Planner 已花的 token 全作废、`pending-inputs` 要等 LLM 处理完测试结果 + 自己决定 turn 结束（可能 5+ 分钟）才生效。中间缺一档"当前 tool 跑完就停下听我说"。

业界主流（Claude Code `messageQueueManager`、OpenCode Effect fiber、Warp `FollowUpSubmitted`）的做法都是 **"迭代边界热注入 + 中断令牌"**：消息进入队列，agent 在每次迭代开始时 drain；用 AbortSignal 取消整个 stream 是另一档独立机制。KodaX 已经有 pending-inputs（队列）+ AbortController（中断），缺的是**让 runner loop 在 tool 边界主动暂停等用户**的 in-between 信号。

设计核心：runner loop 加一个 `paused: boolean` flag，**不引入新类型/状态机**——一个 boolean，在每次 tool call 完成后、下一轮 LLM 调用之前检查一次。代码增量约 100 行。

## FEATURE_111: Runner-Level Soft-Pause Protocol

**Status**: Planned
**Priority**: High
**Category**: Enhancement / UX

### 背景

#### 当前两档信号的覆盖空白

`pending-inputs.ts` 是 UI 层持有的字符串数组（无 runner 主动轮询机制）；runner-driven.ts 在 while loop 里只关心 `options.abortSignal` 和 tool call 返回值，turn 边界由 LLM 响应自然终止触发。这意味着：

- **快任务场景（tool p95 < 5s）**：pending-inputs 在 turn 边界注入与"tool 边界注入"差距小，体感几乎无感
- **慢任务场景（tool p95 > 30s，npm test / build / 数据库迁移）**：用户 ESC 后队列消息要等 tool 跑完 + LLM 处理完结果 + LLM 自己决定结束 turn 才生效，体感是"我按了暂停但完全没反应"

第二档（abort）虽然立刻生效，但代价是丢失整个 turn 的 LLM 上下文。在 H2 路径这是双倍浪费（Scout + Planner 已花的 token 全废）。

#### 业界惯例

- **Claude Code**：`messageQueueManager.enqueue()` priority='next'，runner 在每次 tool 边界 drain 队列消费 user message；`AbortController` 是独立路径用于真正取消（`QueryEngine.interrupt()`）
- **OpenCode**：Effect fiber `Stream.takeUntil` + `Effect.retry`，`SessionPrompt.cancel()` 通过 `Effect.onInterrupt` 钩子设 `aborted=true`；中断信号即刻取消 in-flight LLM 请求但 tool 完成前不强行打断
- **Warp**：`CancellationReason::FollowUpSubmitted` —— 用户发消息直接取消 stream，follow-up 作为同一会话新 exchange 继续。这是"每次 follow-up = abort + restart"的极端版

KodaX 选 Claude Code 风格"队列 + 边界注入 + 独立 abort"——已经有了 pending-inputs（队列）和 AbortController（abort），只缺**在 tool 边界主动等待**这一档。

### 设计

#### Runner 层：单 boolean flag（不是状态机）

`packages/coding/src/task-engine/runner-driven.ts` 在 runner loop 引入一个 `paused: boolean` 局部变量：

```ts
// 伪代码，关键逻辑
let paused = false;

// 每次 tool call 完成后、即将发起下一轮 LLM 调用之前
async function maybeWaitResume(): Promise<void> {
  while (paused && !options.abortSignal?.aborted) {
    // 等待 UI 层喂回一条 resume signal（任意内容或空字符串）
    const interjected = await events.waitForUserResume?.();
    paused = false;
    if (interjected && interjected.trim().length > 0) {
      // 把用户插话作为下一轮 user message，前置 <user-interjected> 标签
      pendingUserMessages.push({
        role: 'user',
        content: `<user-interjected mid-task>\n${interjected}\n</user-interjected mid-task>`,
      });
    }
    // 空内容 → resume 但不注入；LLM 看不到新 user message，按原计划继续
  }
}

// loop 内调用点（仅一个，关键）：
while (!done) {
  // 1. 跑当前 LLM turn → 拿 tool calls
  const llmResponse = await callLLM(...);
  // 2. 执行 tool calls（不在 tool 执行中检查 paused，保持 tool 原子性）
  for (const call of llmResponse.toolCalls) {
    const result = await executeTool(call, ctx);
    appendToolResult(result);
    // 3. tool 返回后、准备下一轮 LLM 调用之前，检查 paused
    await maybeWaitResume();
    if (options.abortSignal?.aborted) throw new AbortError();
  }
  // 4. 下一轮 LLM ...
}
```

**关键点**：
- **不在 tool 执行中检查**——保持 tool 原子性。`bash`/`run_tests` 跑 90 秒就让它跑完，不强行中断（强行中断在跨进程场景几乎不可能优雅实现，且会破坏 tool 内部状态）
- **不引入 `PauseToken` 新类型**——一个 boolean 足够，状态机的复杂度不值得
- **不与 `AbortController` 互斥**：`paused=true` 期间如果用户 abort，立刻退出（abort 优先级始终最高）

#### UI 层：单 ESC = soft-pause / Double-ESC = abort（既有不变）

InkREPL 的 ESC 处理（[InkREPL.tsx:3996-4047](../../packages/repl/src/ui/InkREPL.tsx#L3996) 既有 double-ESC 检测）扩展为三态：

| 操作 | UI 行为 | Runner 信号 |
|---|---|---|
| 按 1 次 ESC | spinner 文字变 `Paused (waiting for tool to finish)` | 设 `paused = true`；当前 tool 完成后 runner 进入 `maybeWaitResume()` 等待 |
| 当前 tool 跑完（runner 进入等待） | spinner 文字变 `Paused — type to inject, Enter to resume` | / |
| 输入新指令 + Enter | resume + 把新指令作为 user message 注入下一轮 LLM | `events.waitForUserResume()` 解析为该字符串 |
| 直接 Enter（空输入） | resume，不注入新内容 | `events.waitForUserResume()` 解析为空字符串 |
| 在 paused 状态下再按 ESC | 升级为 abort（既有 double-ESC 路径） | `AbortController.abort()`（既有） |

**为什么不区分"输入 Enter 注入" vs "空 Enter resume"两种语义**：辩论结论——LLM 能理解 `continue` / 空消息的语义，工程层硬编码两种 Enter 行为是把不必要的复杂度暴露给用户（"空 Enter 到底是什么意思"）。最小语义：**任意 Enter 都算 resume**，只是有内容时把它作为新 user message，没内容时不发。统一规则一目了然。

#### KodaXEvents 新增

`packages/coding/src/types.ts` 的 `KodaXEvents` 新增：

```ts
export interface KodaXEvents {
  // ... 既有
  /**
   * Runner 在 tool 边界检测到 paused=true 时调用，等待 UI 层喂回一条
   * resume signal。返回的字符串作为下一轮 user message 注入；返回空字符串
   * 表示 resume 但不注入。在 abort 期间 reject。
   *
   * FEATURE_111: Soft-pause protocol. Single signal type — boolean flag
   * + this resolver. No state machine, no PauseToken.
   */
  waitForUserResume?(): Promise<string>;
}
```

InkREPL 在 `setupEvents` 阶段注入实现：

```ts
const waitForUserResume = (): Promise<string> => new Promise((resolve, reject) => {
  // 把 resolve 挂在某个 ref 上，让 PromptInput 的 Enter handler 在
  // paused 状态下取出 input.value（可能为空字符串）调 resolve
  // ESC 在 paused 状态下调用 reject（升级为 abort）
});
```

#### Prompt 层：`<user-interjected mid-task>` 标签

注入下一轮 user message 时前置标签：

```
<user-interjected mid-task>
[用户输入的字面内容]
</user-interjected mid-task>
```

并在 system reminder 加一行：`If you see <user-interjected mid-task>, the user paused you mid-task to add information. Todo list and current plan are still active — only adjust if the user explicitly asks you to.`

**作用**：让 LLM 知道这是"插话"不是"新任务"，避免 LLM 把它当成 reroll 整个 plan 的信号。

**Prompt eval 触发**：本 feature 触及 system prompt（system-reminder 文本）和 role prompt（如何解读 `<user-interjected>` 标签），按 [CLAUDE.md FEATURE_104](../CLAUDE.md#prompt-eval-feature_104-v0729) 规则必须新增 eval case 在 `tests/soft-pause-interject.eval.ts`，覆盖以下场景：

1. 用户插话"补充测试场景" → LLM 应该继续执行原计划但把测试场景纳入下一步
2. 用户插话"算了换个方向" → LLM 应该明确暂停当前计划征求用户确认（不是直接换方向，因为标签语义是"插话"不是"新任务"）
3. 用户插话空字符串 → 这种情况 runner 不会注入 user message，eval case 验证 prompt 不会被触发即可（negative space）

### 验收标准

1. **单击 ESC 进入 soft-pause**：用户按一次 ESC，runner 不立即 abort；spinner 文字变 `Paused (waiting for tool to finish)`；当前 tool 跑完后变 `Paused — type to inject, Enter to resume`。
2. **空 Enter resume 不注入**：paused 状态下用户直接 Enter（空输入），runner 退出 `maybeWaitResume()` 继续下一轮 LLM 调用，**不**新增 user message；LLM 按原计划继续。
3. **有内容 Enter resume 并注入**：paused 状态下用户输入"测试要包含 token rotation 场景" + Enter，runner 把内容作为下一轮 user message 注入（前置 `<user-interjected mid-task>` 标签）；LLM 在下一轮看到该消息并相应调整。
4. **Pause 期间 abort**：paused 状态下用户再按 ESC（即 paused 后第二次 ESC，等价于 double-ESC），runner 走既有 abort 路径取消整个 turn。
5. **长 tool 不被强行打断**：用户在跑 90 秒 `npm test` 时按 ESC，**不**中断 npm test 进程；npm test 自然完成后 runner 才进入等待状态。这是 tool 原子性保证。
6. **Pending-inputs 路径不退化**：未按 ESC 时（非 paused 状态），用户在 streaming 中按 Enter 仍走既有 pending-inputs 路径排队；soft-pause 与 pending-inputs 正交不互相影响。
7. **Prompt eval 绿**：`tests/soft-pause-interject.eval.ts` 在 FEATURE_104 alias 表 8 alias 上 mean ≥80% 通过率（与 FEATURE_106 同一阈值），证明 LLM 能正确解读 `<user-interjected mid-task>` 标签的"插话不是新任务"语义。
8. **无回归**：`npm run test` / `npm run build` 全绿；既有 abort、pending-inputs、permission-gate、所有 AMA 路径测试通过。

### 非目标

- **不** 引入 `PauseToken` 新类型 / 三状态机 `running | paused | aborting`（一个 boolean 够）
- **不** 在 tool 执行**中**检查 paused（保持 tool 原子性，避免破坏 bash/test 等的内部状态）
- **不** 区分"空 Enter resume" vs "输入 Enter resume" 两种语义（统一"任意 Enter 都算 resume"，有内容则注入）
- **不** 替代 pending-inputs（两者并存，soft-pause 是用户主动按 ESC 触发的中间档；pending-inputs 是用户不按 ESC 时的后台队列）
- **不** 改 child agent 的 abort/pause 协议（child 仍由 parent runner 决定，soft-pause 只在 parent loop 实现）
- **不** 持久化 paused 状态（session 中断/恢复不保留 paused，恢复时一律为 running）
- **不** 给 paused 状态加超时（用户按 ESC 后没 Enter 就一直等，永远不会自动 timeout 进 abort——避免用户离开终端时被自动取消）

### 与 FEATURE_097 / FEATURE_110 的关系

- **与 FEATURE_097**（todo 列表可视化，v0.7.34）：正交。soft-pause 触发后 TodoListSurface 不变（不刷新、不 reset），spinner 文字变更但 todo 列表保持原状；resume 后正常推进。
- **与 FEATURE_110**（砍老 plan-mode，v0.7.34）：FEATURE_110 把 plan-mode 简化到只有路径 2 一条，soft-pause 不与 plan-mode 交互——`PermissionMode="plan"` 下用户照样可以 ESC soft-pause（注入意见后让 LLM 调整正在生成的 plan），符合 FEATURE_074 live predicate closure 语义。
