# v0.7.47 Feature Designs

> 中文标题：`v0.7.47 Feature 设计`
>
> Version: `v0.7.47`
>
> Status: Planned (not yet released)
>
> Theme: Session-Driven Reflective Prompt Patcher — 把 KodaX 真实 session trace 当作 offline trajectory pool，借鉴 GEPA (arXiv:2507.19457) 反思范式自动产出 prompt diff 候选，由 EVAL_GUIDELINES Layer 2 单轮探针验证后输出给开发者人工 PR 审核。**不替代 benchmark，是 benchmark 的上游 candidate generator**。

---

## 总览

KodaX 当前 prompt 迭代流程：人工读 trace → 改 prompt → FEATURE_104 prompt eval 验证 → PR review。FEATURE_107 (v0.7.32) 的"200+ cells 仅 5% 可决策信号"教训催生了 [EVAL_GUIDELINES](../../benchmark/EVAL_GUIDELINES.md)，把验证侧管住了，但**候选生成**这一环仍然是人工瓶颈：每个 prompt 改动想法都靠开发者读 trace 拍脑袋。

GEPA 论文 (arXiv:2507.19457) 提出反思式 prompt 进化：trajectory + execution feedback → reflector LLM → prompt patch。原版需要带 reward 的 task benchmark 跑 rollout——**这正是 EVAL_GUIDELINES 反模式 1+2 明确禁掉的**（OK rate 当指标 + 让 LLM 自由跑后解读）。

但 GEPA 的反思机制本身可以**离线化**用到 KodaX：

> 不跑 benchmark agent loop，而是把已发生的 session trace 当 trajectory pool，从中挖失败、让 reflector LLM 提改进、用罐头历史在 Layer 2 探针验证。

本 feature 把这个范式落地，全程**不**修改用户 session、**不**自动 apply prompt patch、**不**绕过 FEATURE_104 prompt eval。产出是给开发者的 prompt diff 候选 + 失败模式报告，仍然走 PR review 闭环。

**核心定位**：candidate generator，不是 evaluator。**最终决策权 100% 留给 FEATURE_104 prompt eval + 人工 review**。

## FEATURE_108: Session-Driven Reflective Prompt Patcher

**Status**: Planned
**Priority**: Medium
**Category**: Internal / Test Infrastructure

### 背景

#### 现状

KodaX 已有的 prompt 治理基础设施：

- `packages/session-lineage` — session trajectory tree 已记录（v0.7.24 FEATURE_082）
- `benchmark/harness/` — prompt-eval harness + judges + persist (FEATURE_104, v0.7.29 shipped)
- `benchmark/EVAL_GUIDELINES.md` — Layer 1/2/3 方法论 + 反模式清单 (v0.7.32 shipped)
- FEATURE_102 Phase 0 telemetry (v0.7.45 planned) — 标准化 trace + `objective_signals`
- FEATURE_105 advisor primitive (v0.7.46 planned) — cross-family advisor + capability alias

也就是说，**自动产出 prompt patch 候选所需的所有基础设施都已经在或即将在**——差的只是把"读 session 找失败 → 让 LLM 提改进 → 罐头历史验证"这条 pipeline 串起来。

#### 与已有基础设施的关系

| 已有 / 计划 | 提供的能力 | 本 feature 如何消费 |
|---|---|---|
| FEATURE_102 Phase 0 (v0.7.45) | telemetry trace + `objective_signals` 标准化 schema | 失败信号源（首选 hard signal） |
| FEATURE_104 (v0.7.29 shipped) | prompt-eval harness + judges + persist | Phase 3 罐头历史验证 + ship-decision 全 alias 验证 |
| FEATURE_105 (v0.7.46 planned) | cross-family advisor primitive + capability alias | Reflector 角色复用 `advisor:strong-reasoning` |
| `packages/session-lineage` | session trajectory tree | 备用素材源（telemetry 缺失时的 fallback） |
| EVAL_GUIDELINES (shipped) | Layer 1/2/3 + 反模式清单 | 验证流程的合规约束 + 反模式护栏 |

**依赖顺序硬性**：本 feature 必须在 v0.7.45 (FEATURE_102 Phase 0) 后 ship。v0.7.46 (FEATURE_105) 可并行设计但不强制 ship 后才能开工——若 FEATURE_105 延期，reflector 临时降级为单一固定 model（接受单家族盲区）。

### 核心判断（不能违反的设计原则）

1. **Candidate generator，不是 evaluator。** 本 feature 输出的是 prompt diff *候选*，必须经过 FEATURE_104 prompt eval 量化 + 人工 PR review 才能进 production prompt。**绝不 auto-apply**。
2. **离线 + 只读。** 不在用户活跃 session 期间运行；不修改任何 session / lineage 文件；只读 telemetry / session-lineage。
3. **失败信号必须是硬信号。** vitest 红 / tool error / `emit_handoff status="blocked"` / 用户在 ≤3 turn 内显式 negation——这些是优先信号。**"用户没追问" 不算 positive signal**（避免 reward hacking）。
4. **罐头历史验证不可省。** Reflector 提的 patch 必须在 canned history (失败 trajectory 截到失败前一步) 上跑 Layer 2 单轮探针，断言新 prompt **不再触发该失败模式** + **在 control case 上不退化**。
5. **不演化 evolutionary loop。** MVP 不做 Pareto frontier / quality-diversity / multi-generation。一次 evolution run = 一轮 reflection → 一份候选报告 → 开发者人工 review → 闭环。多代演化等真正出现"单代不够"的证据再加（CLAUDE.md "Abstract only after 3+ real cases"）。
6. **隐私 + opt-in。** session 是用户真实代码，离线分析等同把 session 内容喂给 reflector LLM。**默认 opt-out** + 路径/标识 redact + 显式同意才参与。
7. **失败簇 ≥3 才 reflect。** 单次失败可能是 noise；3+ 次同模式失败才喂 reflector，避免对偶发现象产出过拟合 patch（同 EVAL_GUIDELINES Layer 2 sample size 约束逻辑）。
8. **Reflector 不能 hallucinate prompt 片段。** Reflector 看到的当前 prompt 必须从 git HEAD 实读，候选 patch 中引用的 `current_prompt_excerpt` 必须能在 HEAD 中 grep 命中——否则候选直接拒绝。

### 实施路线（3 个 Phase，时间盒严格）

#### Phase 1 — Failure Mining + Cluster Report（2 天，Layer 1 成本 $0）

**目标**：从 telemetry / session-lineage 中识别失败簇，纯代码逻辑无 LLM。

**新增模块**：`packages/prompt-evolution/src/mine-failures.ts` + `src/signatures/`

**失败签名（每个 pattern 一个文件，机械化）**：
- `vitest-fail-then-blocked.ts` — vitest 失败后 ≤2 turn 内 emit_handoff blocked
- `tool-error-no-retry.ts` — 同 tool 同 input 失败 ≥3 次无变换
- `user-negation-followed.ts` — 用户消息含 negation 模式 ("不对" / "重做" / "你看错") 在 assistant tool_use 之后
- `objective-signal-fail.ts` — telemetry `objective_signals.{test|typecheck|lint}.result === 'fail'` 在 task 结束前
- `must-touch-miss.ts` — Scout 标记的 must-touch 文件在最终 patch 中未出现

**Phase 1 验收**：
- [ ] CLI `kodax evolve-prompts mine --since=<date>` 输出 cluster 报告
- [ ] 每个 pattern 有机械化签名（grep / state machine），不依赖 LLM
- [ ] cluster size < 3 时 skip 并退出码 0
- [ ] 路径 redact 默认开启 + unit test 覆盖
- [ ] 真实 dogfood：在最近 N=50 个开发者 session 上跑出至少 1 个 pattern_id 的 cluster

#### Phase 2 — Reflector + Patch Generation（2 天，Layer 2 成本 < $1/run）

**目标**：把 cluster 喂给 reflector LLM，产出强类型 PatchCandidate。

**新增模块**：`packages/prompt-evolution/src/reflect.ts`

**Reflector prompt 模板要点**（触发 FEATURE_104 prompt eval）：
- 输入：cluster 的 5 个代表 trajectory（已 redact）+ 当前 prompt 全文（从 git HEAD 实读）+ 失败信号 breakdown
- 强制返回 `PatchCandidate` 结构（runtime validator 验证）
- 必须填 `current_prompt_excerpt`（从输入 prompt 中精确摘出疑似引诱片段）
- 必须填 `proposed_diff`（unified diff 格式）
- 必须填 `proposed_assertion`（罐头历史探针该断言什么）
- "你不能调用任何 tool"（与 FEATURE_105 advisor 同范式）
- "你输出的是 *候选*，不是最终 prompt"（避免 reflector 写得过于自信）

**Phase 2 验收**：
- [ ] Reflector 复用 FEATURE_105 capability alias `advisor:strong-reasoning`
- [ ] 输出 `PatchCandidate` 强类型 + runtime validator 容错（解析失败降级为 `skip`，不 hard fail）
- [ ] Hallucination 防御：`current_prompt_excerpt` 不能在 git HEAD prompt 文件中 grep 命中 → 直接拒绝候选并 trace 记录
- [ ] reflector 不可达 fallback：返回空 candidate list，trace 记录但不 hard fail
- [ ] 触发 FEATURE_104 prompt eval（reflector prompt 本身改动也要 eval 验证）

#### Phase 3 — Canned History Validation（2 天，Layer 2 成本 ~$2/run）

**目标**：罐头历史 + Layer 2 单轮探针验证 patch 候选。

**新增模块**：`packages/prompt-evolution/src/validate.ts`

**罐头历史构造**：
- 从 cluster 的 5 个代表 trajectory 中，截取**失败发生前一步**作为 canned history
- 用户文本部分按 `redact_user_text` 配置决定是否 redact

**双探针验证**（每个候选 patch 跑两次 + 控制 case 5 个）：

| 探针 | system prompt | history | 期望断言 |
|---|---|---|---|
| baseline | 旧 prompt | canned history | 复现失败模式（≥80% 复现率） |
| patched | 新 prompt | canned history | 不出现失败模式（≥50pp 改善） |
| control × 5 | 新 prompt | 不相关 case 的 canned history | 不退化（≤2pp noise band） |

**预注册阈值**（pre-registered，跑前定下来不可改）：
- baseline 复现率 ≥80% → 否则 cluster 签名设计有问题，本轮 abort
- patched 改善 ≥50pp → 否则 verdict=`reject`
- control 退化 ≤2pp → 否则 verdict=`needs_human_review`

**Phase 3 验收**：
- [ ] 罐头历史能稳定复现 baseline failure (≥80%)
- [ ] verdict 三态机械化判定（pre-registered 阈值不可调）
- [ ] verdict=`ship_candidate` 自动产出 `benchmark/results/<ts>/prompt-patch-<id>.diff` + REPORT.md
- [ ] 总 LLM 成本 < $5 / 次完整 evolution run
- [ ] 跑出来的 candidate 必须能用 `git apply` 干净打上去（dry-run 验证）

### 接口契约

```typescript
// packages/prompt-evolution/src/types.ts

export interface FailureCluster {
  pattern_id: string;          // 'vitest-fail-then-blocked' 等
  signature_version: string;   // 签名实现的版本（变更时所有历史 cluster 失效）
  occurrences: number;         // ≥3 才入候选
  representative_traces: string[];  // trace_id, 取前 5 个最 representative
  failure_signal_breakdown: {
    tool_error_count: number;
    handoff_blocked_count: number;
    user_negation_count: number;
    objective_signal_fail_count: number;
    must_touch_miss_count: number;
  };
}

export interface PatchCandidate {
  pattern_id: string;
  current_prompt_file: string;          // 例如 'packages/coding/src/coding-preset.ts'
  current_prompt_excerpt: string;       // 必须能在 git HEAD 中 grep 命中
  proposed_diff: string;                // unified diff 格式
  reflector_rationale: string;          // 为什么这样改
  reflector_provider: string;
  reflector_model: string;
  proposed_assertion: string;           // 罐头历史探针该断言什么
  cross_family: boolean;                // reflector 与当前 executor 是否跨家族
}

export interface ValidationOutcome {
  pattern_id: string;
  patch: PatchCandidate;
  baseline_failure_rate: number;        // 旧 prompt + canned history 复现失败率
  patched_failure_rate: number;         // 新 prompt + canned history 失败率
  control_case_regression: number;      // 5 个 control case 上的退化均值
  verdict: 'ship_candidate' | 'reject' | 'needs_human_review';
  trace_artifacts: string[];            // benchmark/results/<ts>/ 下的产物路径
}
```

### 配置接口（草案，Phase 1 实施时定型）

```yaml
prompt_evolution:
  enabled: false                              # 默认 opt-out（隐私优先）
  min_cluster_size: 3                         # 失败簇 ≥3 才 reflect
  reflector_alias: advisor:strong-reasoning   # 复用 FEATURE_105 capability alias
  trace_source: telemetry                     # telemetry | session-lineage（前者优先）
  redact_paths: true                          # session 路径 redact
  redact_user_text: false                     # 用户文本默认保留（reflector 需要看用户反馈）
  canned_history_max_size_chars: 50000        # 截断阈值
  control_case_count: 5                       # 验证回归用的控制 case 数
  pre_registered_thresholds:
    baseline_reproduction_min: 0.80           # ≥80% 才视为有效复现
    patched_improvement_min_pp: 50            # ≥50pp 才 ship_candidate
    control_regression_max_pp: 2              # ≤2pp 才不打 needs_human_review
```

### CLI 接口

```bash
kodax evolve-prompts mine --since=<date>           # Phase 1: 仅挖簇出报告
kodax evolve-prompts reflect --pattern=<id>        # Phase 2: 产 patch 候选
kodax evolve-prompts validate --candidate=<file>   # Phase 3: 罐头历史验证
kodax evolve-prompts run --pattern=<id>            # 三 Phase 串跑
kodax evolve-prompts run --dry-run                 # 不调 LLM，仅 mine + 报告
```

### 明确不做（避免被未来误解）

- ❌ 自动 apply prompt patch 进 production
- ❌ 在用户活跃 session 期间运行
- ❌ Multi-generation evolutionary loop（GEPA 原版 Pareto / MAP-Elites）—— 等真有"单代不够"证据再加
- ❌ 用单次失败做 reflection（必须 cluster ≥3）
- ❌ 把 reflector 输出当 evaluator（reflector 提候选，FEATURE_104 prompt eval 才是判官）
- ❌ "用户没追问" 当 positive signal（reward hacking 风险）
- ❌ 跨 session 拼接 trajectory（每个 cluster 内 trace 独立）
- ❌ 接管 prompt 文件 git history（patch 只到 PR 候选阶段，不 commit）
- ❌ 自建 telemetry / signal pipeline（必须复用 FEATURE_102 Phase 0；FEATURE_102 没有的字段不能新加）
- ❌ 引 zod 或新外部依赖（用 TS 类型 + 手写 runtime validator，与 FEATURE_105 同范式）
- ❌ 让 reflector 调任何 tool 或 consult 第二个 reflector
- ❌ 把候选 patch 直接渲染给用户（候选只对开发者，进 PR 流程）
- ❌ 在 KodaX agent runtime 内联调用本 feature（offline-only，CLI 子命令独占入口）

### 风险与待解决问题（实施时需正面回答）

1. **Reflector 过拟合到簇内 trace**：可能产出 patch 在簇内有效但破坏其他场景。Mitigation: control case 验证（Phase 3 内置 5 个 control case） + ship_candidate 仍需开发者手动跑 FEATURE_104 全 alias 验证才能进 PR。
2. **隐私边界**：session 含用户代码。Mitigation: 默认 opt-out / 路径 redact / 用户显式同意 / reflector 用 KodaX 已配置的 advisor provider（不引入新供应商通道）。**单用户 CLI 场景下，"用户" = 开发者本人，opt-in 等于自我授权——文档需要明确说明这点避免误以为是 SaaS 模型**。
3. **失败签名的可机械化**：能否把"vitest 失败后 emit_handoff blocked"这种模式写成稳定的 grep / state machine？Mitigation: Phase 1 先做 5 个手挑模式签名，验证可机械化后再扩。失败签名版本化（`signature_version` 字段），变更时历史 cluster 失效。
4. **Reflector hallucinate 不存在的 prompt 片段**：reflector 可能凭印象写片段。Mitigation: `current_prompt_excerpt` 必须能在 git HEAD prompt 文件中 grep 命中，命中失败的候选直接拒绝。
5. **Cross-family reflector schema 错位**：复用 FEATURE_105 advisor，跨家族输出 schema 错位风险。Mitigation: 同 FEATURE_105 的 runtime validator 容错（解析失败降级为 skip，不 hard fail）。
6. **空跑无价值**：开发期可能没有足够 session 形成 cluster。Mitigation: tooling 在 cluster < 3 时直接 skip + 提示"等更多 session"，不强行降阈值。
7. **触发面与 EVAL_GUIDELINES 反模式 4 兼容**：EVAL_GUIDELINES 明确"探索期 1 alias、验证期多 alias"。本 feature Phase 2 reflector 用 1 alias（探索）；Phase 3 罐头探针用 1 alias（探索）；ship_candidate 后开发者手动跑 FEATURE_104 全 alias 才能进 PR。次序合规。
8. **FEATURE_102 Phase 0 schema 不稳**：Phase 0 telemetry 还在设计中，schema 可能变。Mitigation: 本 feature ship 时锁定一个 Phase 0 schema 版本，schema 变更时本 feature 必须同步升级（不允许 silent drift）。
9. **签名维护成本**：5 个签名是开局，长期可能膨胀到 20+。Mitigation: 每个签名必须自带 unit test 锁住"什么 trace 应该匹配 / 什么不该匹配"；签名超过 15 个时强制评审是否合并。
10. **与 FEATURE_050 (v0.8.0 Prompt Contracts) 的关系**：FEATURE_050 是 prompt 快照与回归 eval；本 feature 是 prompt 候选生成。两者**互补**——本 feature 产候选，FEATURE_050 锁基线 + 防回归。本 feature 不抢 FEATURE_050 scope。

### 一句话总结

**Session-Driven Reflective Prompt Patcher 是 EVAL_GUIDELINES 上游的候选生成层**：从真实 session 失败簇里让 reflector LLM 提 prompt diff 候选，用罐头历史 Layer 2 探针机械化验证，输出给开发者人工 PR 审核——不跑 benchmark agent loop、不 auto-apply、不破现有 prompt 治理流程，单次 evolution run < $5，**candidate generator 而非 evaluator**，最终决策权 100% 留给 FEATURE_104 prompt eval + 人工 review。
