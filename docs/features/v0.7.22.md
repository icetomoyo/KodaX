## 总览

v0.7.22 是 **SDK 基底段**（FEATURE_079-086）的第一步，也是整条路线图风险最低、体量最大的版本之一。

当前 [packages/coding/src/task-engine.ts](../../packages/coding/src/task-engine.ts) 达 **9034 行**，混合了 Scout 逻辑、H0/H1/H2 状态机、fanout 调度、evidence 获取、role prompt 装配、child-agent reduction、managed-task 生命周期、protocol 解析等多组责任。这个尺寸让任何后续结构性演进（Layer A 原语引入、Session 切分、协议升级）都无法安全进行。

这一版的唯一目标：**在完全不改变外部行为的前提下，把 task-engine.ts 拆成多个模块**，让后续 feature 有可落脚的代码表面。

### 背景

- `task-engine.ts` 是 **9034 行**的单文件，GLM 的代码审查把它列为最严重的结构性问题
- 上层 feature（080 引入 Agent 原语 / 081 拆 Session / 084 重写 Scout）都需要先在这个文件里定位到独立的"状态机入口"，但当前状态机与 glue code 交织
- 外部嵌入场景（SDK 消费者）通过 `runKodaX()` 看到的只有一只黑盒，中层能力（Scout、fanout、evidence）无法被单独引用

### 设计

**核心原则**：**pure extraction, zero behavior change**。不重命名外部 symbol、不改变执行路径、不更新 public API。所有改动都是"把代码搬到另一个文件，task-engine.ts 作为 re-export 门面"。

#### 拆分目标（子模块）

建议目录结构（最终可能微调）：

```
packages/coding/src/task-engine/
├── index.ts                 —— 门面：re-export 所有 public symbol，保持现有 import 路径
├── _internal/
│   ├── prompts/             —— role prompt builder（Scout / Planner / Generator / Evaluator）
│   ├── reducers/            —— pure reducers（round history、budget math、verdict merging）
│   ├── managed-task/        —— durable state helpers（contract / budget / scorecard 读写）
│   ├── evidence/            —— evidence acquisition（overview / diff-bundle / diff-slice / file-read）
│   ├── handoff/             —— reduction helpers、child-agent result merging
│   └── protocol/            —— managed-protocol 解析 shim（与 managed-protocol.ts 去重 — GLM F-4）
└── (task-engine.ts 保留为 index.ts 的别名，渐进替换 import 路径)
```

#### 必须保留的公共表面

- `runKodaX(options, prompt)` — 签名、返回值、事件时序 100% 不变
- `runManagedTask(...)` — FEATURE_076（v0.7.25）依赖它的 round-exit 语义，**表面 + 语义双保持**
- `runOrchestration(...)` — 已经是独立文件，不在本次迁移范围
- KodaXEvents 的所有回调时序
- `.kodax/.managed-tasks/{id}/` 目录结构与文件 schema

#### 消除的重复（GLM F-4）

把 `managed-protocol.ts` 与 `task-engine.ts` 中各有一份的 7 个 `normalize*` 函数（`normalizeManagedVerdictStatus`、`normalizeManagedNextHarness`、`normalizeManagedScoutHarness`、`normalizeManagedHandoffStatus`、`normalizeManagedEvidenceAcquisitionMode`、`normalizeManagedProjectionConfidence`、`normalizeStringListValue`）统一到 `_internal/protocol/normalize.ts`，两个来源引用同一实现。

### Contract

**Public API**: 零变化。所有既有的 import 继续有效。

**Internal**: 子模块间的依赖方向严格向内 — `prompts` / `reducers` / `evidence` / `handoff` 可被 top-level state machine 代码引用，但不能反向引用。`_internal/` 前缀是"包内私有"的约定信号。

### Non-Goals

- **不**引入 `KodaXAgent` primitive（那是 FEATURE_080）
- **不**修改状态机逻辑（H0/H1/H2 行为完全保持）
- **不**修改 Session / 压缩逻辑（那是 FEATURE_081）
- **不**优化性能（本版只重组，不优化）
- **不**加新测试（但要保证现有测试全绿）

### Dependencies

- **前置**：无。这是路线图起点
- **后置**：
  - FEATURE_080 需要在拆分后的代码上引入 Agent primitive
  - FEATURE_084（task-engine phase 2）需要基于本版的子模块结构做重写

### Testing

- 所有现有的 `packages/coding/src/task-engine.test.ts` 测试保持绿
- 所有 e2e `tests/` 测试保持绿
- 不新增功能测试（纯移动不需要新语义断言）
- 新增一条 import path 稳定性检查：扫描 public re-export 清单，确保零漂移

### Risk

- **主要风险**：意外引入行为变化（某个 glue code 被拆散后丢了隐式依赖）
- **缓解**：
  - 以小步提交推进（每步 ≤ 500 行迁移），每步跑全量测试
  - 一条"不改任何业务语义"的硬约束；任何涉及行为改动的 case 都推迟到后续 feature
  - 拆分完成后与 v0.7.20 的行为做一次 diff 对照（同 prompt → 同 output 的 sanity check）
