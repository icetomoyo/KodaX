> Version: `v0.9.0`
>
> Status: Planned
>
> Theme: Multimodal inputs, harness maturation, and repo-intelligence evolution

## 中文导读

`v0.9.0` 主要关注 multimodal artifact inputs。

它的核心意思是：KodaX 不再只接收文本，还要能把 image、screenshot、diagram 之类的 artifact 作为正式输入，接进 task engine，并参与 planning、generation、evaluation。

---

补充说明：

- `FEATURE_031` 负责把 image / screenshot / diagram 这类 artifact 真正接入 task engine
- `FEATURE_042` 继续推进 repo-intelligence 的增量刷新与 Java/C++ 结构语义

所以 `v0.9.0` 不再只是“多模态输入版本”，它同时也是 harness 第二阶段成熟化的版本。

## FEATURE_031: Multimodal Artifact Inputs

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

KodaX should be able to accept non-text artifacts such as images and feed them into the task engine in a provider-aware way.

This feature is not only about image upload convenience. It allows the planner, generator, and evaluator to reason over:

- screenshots
- UI mocks
- diagrams
- captured visual errors

### Goals

1. Support local image-path and clipboard-style image inputs.
2. Normalize multimodal inputs into task-engine-friendly content blocks.
3. Route multimodal work only to providers and harnesses that can support it.
4. Preserve artifact references in task state and evidence.

### Scope

- image attachment intake
- provider-aware image serialization
- artifact references in persisted task state
- evaluator access to visual artifacts

### Non-goals

- broad media editing
- video workflows
- full design-review product surface

### Relationship to other features

- `029` provides provider capability truth
- `025` decides whether a task needs multimodal-capable routing
- `018` and `028` may later index or retrieve artifact-related knowledge

### Acceptance criteria

- [ ] KodaX can accept image artifacts from local input paths
- [ ] multimodal inputs are stored as explicit task artifacts
- [ ] provider routing respects multimodal capability constraints
- [ ] visual artifacts can participate in evaluation and evidence flows

### Restored Detailed Design Notes

Useful retained points from the earlier draft:

- support both local image-path references and clipboard-style image input
- normalize images into provider-required formats such as base64 payloads or provider-native artifact references
- add image content-block support at the AI/provider layer for multimodal-capable backends
- ensure artifact serialization remains compatible with history persistence and future compaction rules
- keep size limits, format validation, and sensitive-path boundaries explicit

#### Historical user scenarios retained

- user provides a screenshot so KodaX can inspect an error
- user provides a UI mock so KodaX can generate or critique implementation
- user provides a chart or visual artifact so KodaX can extract, explain, or summarize it

## FEATURE_042: Incremental Repository Intelligence Refresh and Java/C++ Structural Semantics

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`v0.8.x` establishes task-aware repo intelligence, and Go/Rust can now follow the same bundled-parser path as Python. The next practical gaps are:

- refresh still rebuilds more of the repository-intelligence index than necessary
- Java and C++ still rely on heuristic structural parsing rather than bundled parser-backed extraction

`FEATURE_042` addresses those two gaps together. The goal is a faster and more balanced repository-intelligence substrate without expanding KodaX into a graph product.

This feature is explicitly about **quality and efficiency maturation**, not a new product surface. KodaX should keep the same repo-intelligence entry points while making refresh cheaper and Java/C++ evidence more trustworthy.

### Goals

1. Add file/module-scoped invalidation so intelligence refresh can reuse unchanged work.
2. Upgrade Java/C++ from heuristic structural parsing to bundled structural parsing where practical.
3. Improve confidence stability and token efficiency for large or mixed-language repositories.
4. Keep the public retrieval surface unchanged while strengthening the underlying evidence.

### Product principles

1. Keep the external surface minimal: `repo_overview`, `module_context`, `symbol_context`, `process_context`, `impact_estimate`, and `changed_scope` remain the only first-class repo-intelligence interfaces.
2. Push complexity inward: finer-grained invalidation and parser-backed analyzers stay internal implementation details.
3. Prefer deterministic local analysis over expensive model calls.
4. Fall back safely: when incremental state is invalid, incomplete, or low-confidence, KodaX must transparently rebuild more broadly rather than serving misleading intelligence.
5. Maintain inspectable artifacts under `.agent/repo-intelligence/` instead of introducing graph databases or opaque background services.

### Scope

- file-hash and module-hash invalidation metadata
- partial rebuild for changed files/modules/capsules
- bundled structural parsing for Java/C++
- tighter confidence calibration for Java/C++ retrieval results

### Deliverables

#### 042-A Incremental refresh engine

- extend the repo-intelligence manifest with per-file fingerprints, language, module ownership, and analyzer version metadata
- persist file-level analysis outputs so unchanged files can be reused between refreshes
- add an invalidation planner that decides among:
  - file-only reanalysis
  - module capsule rebuild
  - process capsule rebuild
  - full index rebuild
- keep fallback-to-full-rebuild as a first-class safety path when the invalidation planner is uncertain

#### 042-B Java/C++ parser-backed structural semantics

- replace the current line-regex-first extraction path for Java/C++ with bundled parser-backed structural extraction
- extract classes/interfaces/structs/enums, methods/functions, imports/includes, and qualified call names
- improve method ownership so retrieval surfaces can correctly attribute `Class.method` and `Namespace::Class::method` style symbols
- improve confidence scoring so Java/C++ repositories do not look artificially equivalent to TS/JS/Python, but also no longer collapse to low-confidence heuristic-only behavior

### Detailed design

#### Refresh pipeline

The refresh path should move from today's coarse "index is stale, rebuild almost everything" model to a staged planner:

1. Read the manifest and compare schema version, analyzer versions, and source fingerprints.
2. Compute changed files and deleted files.
3. Re-run analyzers only for changed files.
4. Rebuild module capsules only for modules that own changed files or whose dependency edges changed.
5. Rebuild process capsules only for modules whose entry symbols, dependencies, or first-hop calls changed.
6. Re-link call targets globally only for affected symbol names/modules; if the affected set is too large or ambiguous, fall back to a full re-link.
7. Persist the updated manifest and materialized indexes atomically.

#### Incremental artifacts

`FEATURE_042` should add or formalize the following internal artifacts:

- `repo-intelligence-manifest.json`
  - schema version
  - analyzer versions
  - workspace source fingerprint
  - per-file fingerprints
  - file -> module ownership
  - file -> language
- `file-analysis-index.json`
  - persisted file-level imports/symbols/calls/confidence
- existing module/symbol/process indexes remain the query-facing materialized view

#### Fallback policy

Incremental refresh is an optimization, not a correctness boundary. KodaX should rebuild more broadly when:

- the manifest is missing or schema-incompatible
- a parser upgrade changes analyzer semantics
- deleted or renamed files break module ownership assumptions
- dependency edges widen beyond the incremental planner's safe threshold
- the planner cannot prove that process capsules can be updated locally

#### Java/C++ semantic target

The target for `0.9.0` is **parser-backed structural semantics**, not full multi-language type resolution. That means:

- Java:
  - packages/imports
  - classes/interfaces/enums
  - methods/constructors
  - method-qualified calls where recoverable
- C++:
  - includes
  - namespaces/classes/structs/enums
  - out-of-line member definitions
  - qualified calls and method ownership where recoverable

Non-goal for `0.9.0`:

- full Java symbol resolution across inheritance and overload sets
- full C++ template, macro, and header/translation-unit parity

### Query-layer impact

`FEATURE_042` must not add new user-facing tools. Instead it should make existing results better:

- `module_context` should produce more stable dependency/dependent sets after small diffs
- `symbol_context` should attribute Java/C++ methods more accurately
- `process_context` should preserve unaffected capsules across small edits
- `impact_estimate` should stop over-broadening when only one file or module changed
- `changed_scope` should remain unchanged in interface, but become a stronger incremental invalidation input

### Router and task-engine impact

`025` and the task engine do not need a new public contract, but they should benefit from:

- lower latency for repo-intelligence prefetch on repeated tasks
- more stable `repoSignals.confidence` for Java/C++ repositories
- fewer unnecessary planner/investigation escalations caused by stale or overly broad refreshes

### Non-goals

- graph database or Cypher
- visual graph product surfaces
- full multi-language type-resolution parity with GitNexus
- changing the public retrieval surface away from `repo_overview / module_context / symbol_context / process_context / impact_estimate / changed_scope`

### Relationship to other features

- `018` owns the repository-intelligence substrate and persistence model
- `028` owns the retrieval surface that consumes those artifacts
- `025` benefits when routing can trust faster refreshes and more balanced cross-language confidence

### Acceptance criteria

- [ ] KodaX can refresh repo intelligence without rebuilding every unchanged module
- [ ] Java/C++ use bundled structural parsers for symbol/import/call extraction
- [ ] repo-intelligence confidence becomes more stable across TS/JS/Python/Go/Rust/Java/C++
- [ ] retrieval tools keep the same public surface while returning stronger evidence for Java/C++ repositories
- [ ] repeated repo-intelligence prefetch on small diffs is measurably cheaper than a cold rebuild
- [ ] incremental refresh has a documented and tested fallback path to full rebuild

## FEATURE_043: Harness Calibration, Pivoting, Profiling, and Safe Checkpoints

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_043` 承接 `FEATURE_022` 在 runtime 主路径已经落地之后的 Phase 2 能力。

它不是再发明一套新的 harness，而是把现有的 `Adaptive Project Harness + AMA Harness` 继续补成：

- 可校准
- 可 pivot
- 可 profiling
- 可安全 checkpoint

### Goals

1. 让 Evaluator 的质量可以被持续校准，而不只是靠人工感觉。
2. 让 harness 在发现当前方向不对时，可以显式 `pivot` 到新 branch，而不是只能沿原路径 `revise`。
3. 让系统知道“哪些 harness 组件真的带来了 lift”，避免长期只会越堆越重。
4. 让长流程执行在 workspace 层也有安全 checkpoint surface，而不只是 task artifact / continuation。

### Scope

- evaluator false-pass / false-fail capture
- evaluator regression corpus and calibration workflow
- `pivot` / branch directive and branch-scoped artifacts
- harness profiling and ablation records
- safe patch / diff checkpoints
- future sandbox worktree checkpoint hooks

### Deliverables

#### 043-A Evaluator calibration loop

- 记录 false-pass / false-fail case
- 建立 reviewer regression corpus
- 支持按 provider / model 维护 evaluator calibration prompt profile
- 让 QA 质量可以做 regression，而不只是看一次 run 的主观效果

#### 043-B Pivot / branch management

- 在 `accept / revise / blocked` 之外，引入 `pivot`
- 允许 Generator 在 rubric 停滞、趋势变差或实现方向明显不优时切换到新 branch
- 保留 branch-scoped artifact、feedback、scorecard 与 round history

#### 043-C Harness profiling / ablation

- 记录每次 run 中哪些 harness 组件被启用
- 记录这些组件对质量、成本、收敛速度的影响
- 支持按 provider / model / task family 形成 harness preset
- 为后续“最简单可行 harness”策略提供数据依据

#### 043-D Safe workspace checkpoints

- 默认生成非破坏性的 patch / diff snapshot
- 为将来的 sandbox worktree checkpoint 留出接口
- evaluator fail / pivot / continuation 时，能引用最近的 workspace checkpoint
- 不在用户主工作区执行未经明确策略允许的 destructive reset

### Relationship to other features

- `FEATURE_022` 负责当前 runtime 主路径和 control plane
- `FEATURE_027` 提供 `SA / AMA` 的用户级显式开关
- `FEATURE_029` 提供 provider capability truth
- `FEATURE_031` 与 `FEATURE_042` 的 artifact / intelligence 结果，都会成为 future calibration 与 profiling 的输入面

### Acceptance criteria

- [ ] Evaluator false-pass / false-fail case can be persisted and replayed as a regression corpus
- [ ] harness can emit a first-class `pivot` directive and carry branch-scoped artifacts forward
- [ ] harness profiling can explain which orchestration components were enabled and what cost / quality tradeoff they introduced
- [ ] safe patch / diff checkpoints exist for continuation / pivot / post-failure inspection
- [ ] future sandbox checkpoint support has a documented interface boundary without forcing destructive workspace resets
