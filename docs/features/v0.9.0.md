# v0.9.0 Feature 设计

> Version: `v0.9.0`
>
> Status: Planned
>
> Theme: Multimodal inputs, worktree-safe runtime, harness maturation, and repo-intelligence evolution

## 中文导读

`v0.9.0` 主要关注 multimodal artifact inputs。

它的核心意思是：KodaX 不再只接收文本，还要能把 image、screenshot、diagram 之类的 artifact 作为正式输入，接进 task engine，并参与 planning、generation、evaluation。

---

补充说明：

- `FEATURE_031` 负责把 image / screenshot / diagram 这类 artifact 真正接入 task engine
- `FEATURE_042` 继续推进 repo-intelligence 的增量刷新与 Java/C++ 结构语义
- `FEATURE_053` 负责把 canonical repo identity、worktree runtime、session resume/list 与 CLI/REPL/ACP 的 cwd 语义补齐
- `FEATURE_043` 承接 adaptive project / AMA harness 的 Phase 2 成熟化，但它消费 `FEATURE_053` 提供的 workspace isolation / checkpoint 边界，而不自己发明 worktree runtime

所以 `v0.9.0` 不再只是“多模态输入版本”，它同时也是 repo/worktree runtime 补完与 harness 第二阶段成熟化的版本。

## FEATURE_055: REPL Substrate Hardening and Summary-Only AMA UX

**Status**: Completed  
**Close-out note (2026-04-04)**: Implemented as a docs-first substrate hardening pass. The visible REPL shell remains frozen while transcript/selection helpers, input blur-paste closure, and AMA summary-only derivation were tightened without adding new multi-agent surface.
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_055` is the follow-up feature for the remaining REPL hardening work that should no longer live under completed `FEATURE_051`.

The visible KodaX REPL shell is already at the right product tier:

- transcript-first
- one main authority
- restrained footer/task/status shell
- no visible worker topology

The remaining gap is invisible substrate maturity:

- transcript / viewport / scroll closure
- host-aware selection / copy policy
- input focus / paste / cursor behavior
- summary-only AMA runtime mapping into the existing REPL shell

### Goals

1. Keep the current REPL visible shell frozen while continuing to harden transcript, scroll, selection, and input behavior.
2. Make terminal host capability truth fully drive selection / copy / browse behavior.
3. Keep AMA Tactical / AMA Managed / child fan-out summary-only inside the existing shell.
4. Preserve `SA` / `AMA` as the only user-visible execution semantics.

### Scope

- transcript owner / viewport owner refinement
- host-aware scroll / selection / copy-on-select closure
- input focus / paste / cursor substrate hardening
- summary-only AMA status derivation for existing REPL surfaces
- feature-doc alignment for `051/055/046/047/052/054`

### Non-goals

- new transcript chrome, task cockpit, or multi-panel REPL surface
- visible worker graph, teammate panel, task workspace, or revived team mode
- new top-level user-visible modes beyond `SA` / `AMA`
- re-opening `FEATURE_051` as an active visible-shell rollout

### REPL posture

`FEATURE_055` follows the KodaX shell-freeze rule:

- `StatusBar`, footer, task bar, help, message actions, and tool rows remain visually frozen
- transcript stays the only main screen
- follow-up work should improve hand feel and stability, not add more visible UI

### AMA boundary

Multi-agent runtime maturity is allowed here only as summary-only REPL mapping:

- `AMA Tactical` and `AMA Managed` remain internal harness profiles
- child fan-out may affect `AmaWorkStrip`, `BackgroundTaskBar`, and transcript live labels
- REPL surfaces do not grow a worker graph, teammate list, or task cockpit

### Relationship to other features

- `FEATURE_051` is the completed visible-shell close-out that hands remaining substrate hardening to `055`
- `FEATURE_046`, `FEATURE_047`, and `FEATURE_052` define the AMA substrate that `055` may summarize in the REPL
- `FEATURE_054` keeps AMA/project convergence on the architecture side without reopening old project-mode UI
- `FEATURE_043` and `FEATURE_053` continue to mature the harness/runtime beneath the same frozen REPL shell

### Acceptance criteria

- [x] transcript / viewport / scroll behavior became more internally coherent without visible shell drift
- [x] host-aware selection / copy policy is capability-driven and no longer scattered across the REPL
- [x] prompt input hardening improves paste / focus / cursor behavior without changing the visible input shell
- [x] AMA tactical / managed / child-fan-out states remain summary-only in the REPL
- [x] feature docs and index stay consistent about `051` being closed out and `055` carrying the remaining hardening work

### Delivered slices

- closed the remaining docs handoff from `FEATURE_051` and aligned `FEATURE_LIST` / `v0.8.0` / `v0.9.0` / feature index guidance
- tightened transcript selection/search derivation into REPL view-model helpers instead of leaving host/capability gating scattered in `InkREPL`
- hardened prompt blur / paste transient cleanup without changing the visible input shell
- kept AMA tactical/managed and child fan-out mapped only through existing summary surfaces

## FEATURE_031: Multimodal Artifact Inputs

**Status**: Planned
**Priority**: High
**Category**: New

### Overview

KodaX should be able to accept non-text artifacts such as images and feed them into the task engine in a provider-aware way.

This feature is not only about image upload convenience. It allows the planner, generator, and evaluator to reason over:

- screenshots
- UI mocks
- diagrams
- captured visual errors

### Goals

1. Support local image-path and clipboard-style image inputs.
2. Normalize multimodal inputs into task-engine-friendly content blocks.
3. Route multimodal work only to providers and harnesses that can support it.
4. Preserve artifact references in task state and evidence.

### Scope

- image attachment intake
- provider-aware image serialization
- artifact references in persisted task state
- evaluator access to visual artifacts

### Non-goals

- broad media editing
- video workflows
- full design-review product surface

### Relationship to other features

- `029` provides provider capability truth
- `025` decides whether a task needs multimodal-capable routing
- `018` and `028` may later index or retrieve artifact-related knowledge

### Acceptance criteria

- [ ] KodaX can accept image artifacts from local input paths
- [ ] multimodal inputs are stored as explicit task artifacts
- [ ] provider routing respects multimodal capability constraints
- [ ] visual artifacts can participate in evaluation and evidence flows

### Restored Detailed Design Notes

Useful retained points from the earlier draft:

- support both local image-path references and clipboard-style image input
- normalize images into provider-required formats such as base64 payloads or provider-native artifact references
- add image content-block support at the AI/provider layer for multimodal-capable backends
- ensure artifact serialization remains compatible with history persistence and future compaction rules
- keep size limits, format validation, and sensitive-path boundaries explicit

#### Historical user scenarios retained

- user provides a screenshot so KodaX can inspect an error
- user provides a UI mock so KodaX can generate or critique implementation
- user provides a chart or visual artifact so KodaX can extract, explain, or summarize it

## FEATURE_042: Incremental Repository Intelligence Refresh and Java/C++ Structural Semantics

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`v0.8.x` establishes task-aware repo intelligence, and Go/Rust can now follow the same bundled-parser path as Python. The next practical gaps are:

- refresh still rebuilds more of the repository-intelligence index than necessary
- Java and C++ still rely on heuristic structural parsing rather than bundled parser-backed extraction

`FEATURE_042` addresses those two gaps together. The goal is a faster and more balanced repository-intelligence substrate without expanding KodaX into a graph product.

This feature is explicitly about **quality and efficiency maturation**, not a new product surface. KodaX should keep the same repo-intelligence entry points while making refresh cheaper and Java/C++ evidence more trustworthy.

### Goals

1. Add file/module-scoped invalidation so intelligence refresh can reuse unchanged work.
2. Upgrade Java/C++ from heuristic structural parsing to bundled structural parsing where practical.
3. Improve confidence stability and token efficiency for large or mixed-language repositories.
4. Keep the public retrieval surface unchanged while strengthening the underlying evidence.

### Product principles

1. Keep the external surface minimal: `repo_overview`, `module_context`, `symbol_context`, `process_context`, `impact_estimate`, and `changed_scope` remain the only first-class repo-intelligence interfaces.
2. Push complexity inward: finer-grained invalidation and parser-backed analyzers stay internal implementation details.
3. Prefer deterministic local analysis over expensive model calls.
4. Fall back safely: when incremental state is invalid, incomplete, or low-confidence, KodaX must transparently rebuild more broadly rather than serving misleading intelligence.
5. Maintain inspectable artifacts under `.agent/repo-intelligence/` instead of introducing graph databases or opaque background services.

### Scope

- file-hash and module-hash invalidation metadata
- partial rebuild for changed files/modules/capsules
- bundled structural parsing for Java/C++
- tighter confidence calibration for Java/C++ retrieval results

### Deliverables

#### 042-A Incremental refresh engine

- extend the repo-intelligence manifest with per-file fingerprints, language, module ownership, and analyzer version metadata
- persist file-level analysis outputs so unchanged files can be reused between refreshes
- add an invalidation planner that decides among:
  - file-only reanalysis
  - module capsule rebuild
  - process capsule rebuild
  - full index rebuild
- keep fallback-to-full-rebuild as a first-class safety path when the invalidation planner is uncertain

#### 042-B Java/C++ parser-backed structural semantics

- replace the current line-regex-first extraction path for Java/C++ with bundled parser-backed structural extraction
- extract classes/interfaces/structs/enums, methods/functions, imports/includes, and qualified call names
- improve method ownership so retrieval surfaces can correctly attribute `Class.method` and `Namespace::Class::method` style symbols
- improve confidence scoring so Java/C++ repositories do not look artificially equivalent to TS/JS/Python, but also no longer collapse to low-confidence heuristic-only behavior

### Detailed design

#### Refresh pipeline

The refresh path should move from today's coarse "index is stale, rebuild almost everything" model to a staged planner:

1. Read the manifest and compare schema version, analyzer versions, and source fingerprints.
2. Compute changed files and deleted files.
3. Re-run analyzers only for changed files.
4. Rebuild module capsules only for modules that own changed files or whose dependency edges changed.
5. Rebuild process capsules only for modules whose entry symbols, dependencies, or first-hop calls changed.
6. Re-link call targets globally only for affected symbol names/modules; if the affected set is too large or ambiguous, fall back to a full re-link.
7. Persist the updated manifest and materialized indexes atomically.

#### Incremental artifacts

`FEATURE_042` should add or formalize the following internal artifacts:

- `repo-intelligence-manifest.json`
  - schema version
  - analyzer versions
  - workspace source fingerprint
  - per-file fingerprints
  - file -> module ownership
  - file -> language
- `file-analysis-index.json`
  - persisted file-level imports/symbols/calls/confidence
- existing module/symbol/process indexes remain the query-facing materialized view

#### Dirty snapshot strategy

The dirty-refresh path should stay intentionally simple:

- use a single clean git baseline as the only long-lived source of truth
- allow current dirty snapshots to be memoized for direct reuse when the same dirty state repeats
- share one resolved repo snapshot across overview, changed-scope, routing, and impact flows within a request
- treat truncated large-repo baselines conservatively: pure edits to baseline-known files may use the fast path, while deletes, renames, or baseline-external additions fall back to a full rebuild

This keeps the implementation aligned with KodaX's minimalist design philosophy: fewer cache layers, fewer hidden states, and explicit fallback to correctness when the dirty state becomes ambiguous.

#### Fallback policy

Incremental refresh is an optimization, not a correctness boundary. KodaX should rebuild more broadly when:

- the manifest is missing or schema-incompatible
- a parser upgrade changes analyzer semantics
- deleted or renamed files break module ownership assumptions
- dependency edges widen beyond the incremental planner's safe threshold
- the planner cannot prove that process capsules can be updated locally

#### Java/C++ semantic target

The target for `0.9.0` is **parser-backed structural semantics**, not full multi-language type resolution. That means:

- Java:
  - packages/imports
  - classes/interfaces/enums
  - methods/constructors
  - method-qualified calls where recoverable
- C++:
  - includes
  - namespaces/classes/structs/enums
  - out-of-line member definitions
  - qualified calls and method ownership where recoverable

Non-goal for `0.9.0`:

- full Java symbol resolution across inheritance and overload sets
- full C++ template, macro, and header/translation-unit parity

### Query-layer impact

`FEATURE_042` must not add new user-facing tools. Instead it should make existing results better:

- `module_context` should produce more stable dependency/dependent sets after small diffs
- `symbol_context` should attribute Java/C++ methods more accurately
- `process_context` should preserve unaffected capsules across small edits
- `impact_estimate` should stop over-broadening when only one file or module changed
- `changed_scope` should remain unchanged in interface, but become a stronger incremental invalidation input

### Router and task-engine impact

`025` and the task engine do not need a new public contract, but they should benefit from:

- lower latency for repo-intelligence prefetch on repeated tasks
- more stable `repoSignals.confidence` for Java/C++ repositories
- fewer unnecessary planner/investigation escalations caused by stale or overly broad refreshes

### Non-goals

- graph database or Cypher
- visual graph product surfaces
- full multi-language type-resolution parity with GitNexus
- changing the public retrieval surface away from `repo_overview / module_context / symbol_context / process_context / impact_estimate / changed_scope`

### Relationship to other features

- `018` owns the repository-intelligence substrate and persistence model
- `028` owns the retrieval surface that consumes those artifacts
- `025` benefits when routing can trust faster refreshes and more balanced cross-language confidence

### Acceptance criteria

- [ ] KodaX can refresh repo intelligence without rebuilding every unchanged module
- [ ] Java/C++ use bundled structural parsers for symbol/import/call extraction
- [ ] repo-intelligence confidence becomes more stable across TS/JS/Python/Go/Rust/Java/C++
- [ ] retrieval tools keep the same public surface while returning stronger evidence for Java/C++ repositories
- [ ] repeated repo-intelligence prefetch on small diffs is measurably cheaper than a cold rebuild
- [ ] incremental refresh has a documented and tested fallback path to full rebuild

## FEATURE_053: Canonical Repo Identity and Managed Worktree Runtime

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

Today KodaX still treats "git repo root", "active workspace root", and "execution cwd" as if they were the same thing.

That is acceptable in a single checkout, but it becomes a correctness bug once Git worktrees enter the picture:

- session list/resume can attach to the wrong checkout
- saved session metadata can be re-bound to the current process cwd
- ACP hosts cannot reliably restore the same workspace semantics as REPL/CLI
- future safe-checkpoint and pivot flows have no stable workspace-isolation boundary

`FEATURE_053` fixes that by introducing a first-class runtime model for:

- canonical repo identity
- current workspace/worktree root
- execution cwd
- optional managed worktree state

The purpose is not merely "support `git worktree add`". The real goal is to make KodaX work correctly across single-checkout, manually created worktree, and KodaX-managed worktree workflows without hiding unsafe assumptions in session or harness code.

### Goals

1. Separate `canonicalGitRoot`, `workspaceRoot`, and `executionCwd` across runtime and session contracts.
2. Make session save/list/resume operate across sibling worktrees of the same repo without silently resuming in the wrong checkout.
3. Support both user-created worktrees and KodaX-managed worktrees with safe keep/remove semantics.
4. Provide a stable workspace-isolation boundary that `FEATURE_043` can consume for safe checkpoint and pivot flows.

### Product principles

1. Detection before orchestration: existing user-created worktrees should already behave correctly before KodaX starts creating any new ones.
2. Separate identity from location: repo identity is not the same thing as the current checkout path or execution cwd.
3. Keep runtime ownership outside the core agent loop: CLI/REPL/ACP manage worktree lifecycle and cwd switching; the core agent consumes resolved context.
4. Fail safe on ambiguity: prefer explicit keep/remove decisions and conservative resume behavior over silent cwd drift or destructive cleanup.
5. Avoid nesting managed worktrees inside the main working tree.

### Scope

- canonical repo root resolution for main checkouts and worktrees
- runtime/session schema upgrades for repo/workspace/cwd identity
- cross-worktree session discovery, resume, and delete semantics
- worktree status/enter/exit surfaces for CLI, REPL, and ACP
- managed worktree lifecycle and cleanup policy
- runtime interface boundary for `FEATURE_043` workspace checkpoints

### Deliverables

#### 053-A Canonical repo and workspace identity

- resolve canonical repo identity from git common-dir semantics rather than assuming the current checkout path is the repo identity
- extend persisted session/runtime metadata with:
  - `canonicalGitRoot`
  - `workspaceRoot`
  - `executionCwd`
  - optional `worktree` descriptor
- restore saved workspace/cwd state on resume instead of recomputing everything from the current process cwd
- keep legacy `gitRoot` compatibility in `0.9.x`, but reinterpret it as the active workspace root rather than the cross-worktree repo identity

#### 053-B Existing worktree detection and attachment

- detect when the current cwd is already inside a user-created worktree
- surface current worktree status in runtime state and session metadata
- allow cross-worktree session listing/resume within the same canonical repo
- never auto-delete or take ownership of user-created worktrees

#### 053-C Managed worktree lifecycle

- add a KodaX-managed worktree flow for explicit isolation when the user or runtime asks for it
- provide user-visible surfaces such as:
  - CLI `--worktree <name>` or equivalent explicit entry
  - REPL `/worktree enter`
  - REPL `/worktree exit keep|remove`
  - REPL `/worktree status`
- carry the same workspace metadata through ACP-hosted sessions instead of collapsing everything to host cwd
- default managed worktree placement to a sibling hidden directory outside the main checkout, for example:
  - `<canonical-parent>/.kodax-worktrees/<repo-name>/<slug>`
- only `remove` KodaX-managed worktrees that were created under this feature's policy

#### 053-D Workspace isolation boundary for `FEATURE_043`

- expose a runtime-level checkpoint target abstraction that can refer to:
  - diff/patch snapshots in the current workspace
  - an optional isolated worktree target
- keep pivot/branch orchestration in `FEATURE_043`, not here
- make it possible for future checkpoint flows to reference a safe isolated workspace without destructive reset semantics

### Detailed design

#### Data model

The minimal runtime/session shape should become:

```ts
type WorktreeState = {
  path: string
  managed: boolean
  origin: 'detected' | 'created'
  branch?: string
  name?: string
}

type WorkspaceBinding = {
  canonicalGitRoot?: string
  workspaceRoot?: string
  executionCwd?: string
  worktree?: WorktreeState
}
```

Compatibility note:

- `gitRoot` remains in `0.9.x` so older call sites do not all break at once
- new logic should treat `gitRoot` as the active workspace root
- new cross-worktree semantics should key off `canonicalGitRoot`

#### Session semantics

Session storage must stop assuming "same repo" means "same checkout path".

New behavior:

1. save the resolved `WorkspaceBinding` with every persisted session
2. list sessions by `canonicalGitRoot`, not exact `workspaceRoot`
3. resume into the saved `workspaceRoot` / `executionCwd` when still available
4. if the saved worktree path no longer exists, degrade safely:
   - explain that the original worktree is gone
   - allow resume in the current workspace only as an explicit fallback

This turns worktree support into a correctness improvement for manual workflows even before KodaX exposes managed worktree creation.

#### Runtime surface ownership

Ownership should stay layered:

- core agent/task engine
  - consumes resolved workspace context
  - does not create/remove worktrees directly
- CLI/REPL/ACP runtime
  - resolves canonical repo/worktree state
  - changes cwd/workspace binding
  - manages enter/exit/status lifecycle

This boundary keeps worktree logic from leaking into unrelated planning, evaluation, or tool code.

#### Managed-path policy

Managed worktrees should default to a stable sibling directory outside the active checkout instead of living under the main repo tree.

Why:

- avoids recursive scanning/indexing surprises
- avoids editor confusion from nested repositories inside the primary workspace
- avoids mixing source files and KodaX-managed isolation directories

Future configuration may allow override, but `0.9.0` should start with one conservative default.

#### Safety policy

The safety contract should be explicit:

- `enter` may attach to a detected worktree or create a managed one
- `exit keep` detaches the KodaX session from the worktree but leaves files on disk
- `exit remove` only removes KodaX-managed worktrees created under this feature
- no destructive reset of the user's main workspace is introduced as part of worktree cleanup
- nested worktree creation from inside another managed worktree should resolve against the canonical repo root, not chain indefinitely

### Non-goals

- per-subagent automatic worktree fan-out
- tmux or multi-terminal orchestration
- sparse checkout or partial clone strategy
- full branch-management product UX
- destructive rollback/reset flows in the user's main workspace

### Relationship to other features

- `FEATURE_019` owns session-tree/checkpoint lineage, but not cross-worktree repo identity correctness
- `FEATURE_040` must carry the same workspace-binding semantics into ACP-hosted sessions
- `FEATURE_043` consumes this feature's worktree/checkpoint boundary for safe pivot and checkpoint flows
- `FEATURE_023` and `FEATURE_030` may later surface worktree state better in UI, but they do not own runtime correctness

### Suggested implementation phases

1. Phase 1: detect existing worktrees and persist `WorkspaceBinding` correctly across save/list/resume
2. Phase 2: add managed worktree enter/exit/status surfaces in CLI, REPL, and ACP
3. Phase 3: integrate isolated-workspace targets into `FEATURE_043` safe checkpoint and pivot flows

### Acceptance criteria

- [ ] KodaX can distinguish canonical repo identity from active workspace root and execution cwd
- [ ] sessions from sibling worktrees of the same repo can be listed and resumed without rebinding to the wrong checkout
- [ ] a user-created worktree can be detected and used without KodaX taking ownership of it
- [ ] KodaX can create, enter, exit, keep, and remove KodaX-managed worktrees through explicit runtime surfaces
- [ ] ACP-hosted sessions preserve the same workspace/worktree semantics as CLI/REPL sessions
- [ ] `FEATURE_043` has a documented workspace-isolation boundary it can consume without requiring destructive workspace reset

## FEATURE_043: Harness Calibration, Pivoting, Profiling, and Safe Checkpoints

**Status**: Planned
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_043` 承接 `FEATURE_022` 在 runtime 主路径已经落地之后的 Phase 2 能力。

它不是再发明一套新的 harness，而是把现有的 `Adaptive Project Harness + AMA Harness` 继续补成：

- 可校准
- 可 pivot
- 可 profiling
- 可安全 checkpoint

### Goals

1. 让 Evaluator 的质量可以被持续校准，而不只是靠人工感觉。
2. 让 harness 在发现当前方向不对时，可以显式 `pivot` 到新 branch，而不是只能沿原路径 `revise`。
3. 让系统知道“哪些 harness 组件真的带来了 lift”，避免长期只会越堆越重。
4. 让长流程执行在 workspace 层也有安全 checkpoint surface，而不只是 task artifact / continuation。

### Scope

- evaluator false-pass / false-fail capture
- evaluator regression corpus and calibration workflow
- `pivot` / branch directive and branch-scoped artifacts
- harness profiling and ablation records
- safe patch / diff checkpoints
- checkpoint hooks built on `FEATURE_053`'s worktree/runtime boundary

### Deliverables

#### 043-A Evaluator calibration loop

- 记录 false-pass / false-fail case
- 建立 reviewer regression corpus
- 支持按 provider / model 维护 evaluator calibration prompt profile
- 让 QA 质量可以做 regression，而不只是看一次 run 的主观效果

#### 043-B Pivot / branch management

- 在 `accept / revise / blocked` 之外，引入 `pivot`
- 允许 Generator 在 rubric 停滞、趋势变差或实现方向明显不优时切换到新 branch
- 保留 branch-scoped artifact、feedback、scorecard 与 round history

#### 043-C Harness profiling / ablation

- 记录每次 run 中哪些 harness 组件被启用
- 记录这些组件对质量、成本、收敛速度的影响
- 支持按 provider / model / task family 形成 harness preset
- 为后续“最简单可行 harness”策略提供数据依据

#### 043-D Safe workspace checkpoints

- 默认生成非破坏性的 patch / diff snapshot
- 通过 `FEATURE_053` 暴露的 runtime/worktree boundary 接入隔离式 workspace checkpoint
- evaluator fail / pivot / continuation 时，能引用最近的 workspace checkpoint
- 不在用户主工作区执行未经明确策略允许的 destructive reset

### Relationship to other features

- `FEATURE_022` 负责当前 runtime 主路径和 control plane
- `FEATURE_027` 提供 `SA / AMA` 的用户级显式开关
- `FEATURE_029` 提供 provider capability truth
- `FEATURE_053` 负责 canonical repo identity、worktree runtime 与 workspace isolation；`FEATURE_043` 只消费这条边界
- `FEATURE_031` 与 `FEATURE_042` 的 artifact / intelligence 结果，都会成为 future calibration 与 profiling 的输入面

### Acceptance criteria

- [ ] Evaluator false-pass / false-fail case can be persisted and replayed as a regression corpus
- [ ] harness can emit a first-class `pivot` directive and carry branch-scoped artifacts forward
- [ ] harness profiling can explain which orchestration components were enabled and what cost / quality tradeoff they introduced
- [ ] safe patch / diff checkpoints exist for continuation / pivot / post-failure inspection
- [ ] isolated-workspace checkpoint support can be layered on top of `FEATURE_053` without forcing destructive workspace resets
