## FEATURE_033: REPL Parallel Toggle (/parallel)

**Status**: Completed  
**Priority**: Medium  
**Category**: Enhancement

### 1. 概述

在 REPL 交互模式中添加 `/parallel` 命令，允许用户在运行时动态切换并行工具执行模式，无需重启 KodaX；切换结果会写入用户配置，并在后续正常启动时恢复。

**核心目标**:
1. **运行时切换**: `/parallel on` / `/parallel off` / `/parallel toggle` 动态切换，无需重启
2. **状态可见**: `/parallel`、`/status`、startup banner、classic status bar、Ink status bar 都能显示当前是 `parallel` 还是 `serial`
3. **持久化**: 选项保存到 `~/.kodax/config.json`，后续 REPL 正常启动可恢复
4. **CLI 优先级正确**: 只有显式传入 `-j/--parallel` 时才覆盖持久化配置；未传入时回落到配置文件
5. **公开 UI 一致**: 导出的 `App` 封装也能把 `parallel` 状态传给 `StatusBar`

**背景**:
- 原先 `--parallel` / `-j` 只能在 CLI 启动时设置
- REPL 中无法查看或切换 `parallel` 状态
- 状态栏无法帮助用户理解当前到底是串行还是并行执行
- 如果 CLI 入口把未传入的 `--parallel` 直接归一成 `false`，会破坏持久化恢复语义

### 2. 影响范围

**实现涉及文件**:
- `packages/repl/src/common/utils.ts` — 持久化 `parallel` 用户配置
- `packages/repl/src/commands/types.ts` — 扩展 `CurrentConfig` 与命令回调接口
- `packages/repl/src/interactive/commands.ts` — 注册 `/parallel` 命令并扩展 `/status`
- `packages/repl/src/interactive/repl.ts` — classic REPL 初始化、状态同步、banner 展示
- `packages/repl/src/interactive/status-bar.ts` — classic status bar 显示 `exec:parallel` / `exec:serial`
- `packages/repl/src/ui/InkREPL.tsx` — Ink REPL 初始化、状态同步、banner 展示
- `packages/repl/src/ui/components/StatusBar.tsx` — Ink status bar 显示 `parallel/serial`
- `packages/repl/src/ui/App.tsx` / `packages/repl/src/ui/types.ts` — 公开 UI API 透传 `parallel`
- `src/kodax_cli.ts` — 修复 CLI `parallel` 配置优先级与重启恢复逻辑

**依赖**:
- `KodaXOptions.parallel` — 已存在的执行选项字段
- REPL runtime options / command callbacks — 支持运行时更新
- Commander `getOptionValueSource()` — 用于区分 CLI 显式传参与默认值来源

### 3. 技术方案

#### 3.1 命令语义

```typescript
{
  name: 'parallel',
  aliases: ['pm'],
  description: 'Show or toggle parallel tool execution mode',

  // /parallel
  // /parallel on
  // /parallel off
  // /parallel toggle
}
```

#### 3.2 REPL 状态更新

```typescript
async function handleParallel(args: string): CommandResult {
  const current = currentConfig.parallel;
  const next = resolveParallelCommand(args, current);

  if (next.kind === 'error') {
    return errorMessage('Usage: /parallel [on|off|toggle]');
  }

  if (next.value !== current) {
    setParallel(next.value);
  }

  return statusMessage(`Execution mode: ${next.value ? 'parallel' : 'serial'}`);
}
```

#### 3.3 CLI 启动优先级

```typescript
function resolveCliParallel(
  program: Command,
  opts: Record<string, unknown>,
  config: { parallel?: boolean },
): boolean {
  if (program.getOptionValueSource('parallel') === 'cli') {
    return opts.parallel === true;
  }

  return config.parallel ?? false;
}
```

这个分支保证：
- 用户显式传 `-j/--parallel` 时，以 CLI 为准
- 用户没有传 `-j/--parallel` 时，不会把持久化状态错误覆盖成 `false`

### 4. 接口契约

```text
/parallel              # 显示当前执行模式
/parallel on           # 切换到 parallel
/parallel off          # 切换到 serial
/parallel toggle       # 在 parallel / serial 之间切换
```

**状态展示契约**:
- classic status bar 显示 `exec:parallel` / `exec:serial`
- Ink status bar 显示 `parallel` / `serial`
- `/status` 输出包含当前 execution mode
- startup banner 展示当前 execution mode，帮助用户在首轮输入前理解状态

### 5. 实现步骤

1. 在命令层添加 `/parallel` 与 `/status` 展示
2. 将 `parallel` 纳入 `CurrentConfig` 与 REPL runtime options
3. 将状态持久化到 `~/.kodax/config.json`
4. 在 classic REPL 和 Ink REPL 状态栏中可视化 execution mode
5. 修复 CLI 入口的配置优先级，确保重启后恢复持久化状态
6. 补齐导出 `App` 组件的 `parallel` prop 透传

### 6. 验收标准

- [x] `/parallel` 命令在 REPL 中可用
- [x] `/parallel on` 启用并行执行
- [x] `/parallel off` 禁用并行执行
- [x] `/parallel toggle` 正确切换状态
- [x] 无参数时显示当前状态
- [x] 状态变更在后续 tool 调用中生效
- [x] 配置写入 `~/.kodax/config.json`
- [x] REPL 正常重启后，在未显式传入 `-j/--parallel` 时恢复持久化状态
- [x] classic REPL / Ink REPL / 导出的 `App` 状态栏都能显示正确的 execution mode

### 7. 实现结果

- `/parallel [on|off|toggle]` 已接入 classic REPL 与 Ink REPL，并持久化到 `~/.kodax/config.json`
- `/status`、startup banner、classic status bar、Ink status bar 都会显示当前是 `parallel` 还是 `serial`
- classic REPL 与 Ink REPL 的 runtime options 都会同步更新 `KodaXOptions.parallel`
- CLI 入口已修复 `parallel` 选项优先级；未显式传入 `-j/--parallel` 时会回落到配置文件中的持久化状态
- 导出的 `App` UI 封装已支持 `parallel` prop，并向 `StatusBar` 正确透传
- 已补充 Vitest 覆盖，并通过 `npm run build:packages`

## FEATURE_036: DeepSeek Built-In Provider Support

**Status**: Completed  
**Priority**: Medium  
**Category**: Enhancement

### 1. Overview

KodaX now ships `deepseek` as the 11th built-in provider instead of requiring users to define it as a custom OpenAI-compatible endpoint.

This implementation intentionally does more than a registry-only add:

1. Registers `deepseek` with the official base URL `https://api.deepseek.com`
2. Uses `DEEPSEEK_API_KEY` as the built-in credential env var
3. Exposes both `deepseek-chat` and `deepseek-reasoner` in provider model discovery
4. Treats `deepseek-chat` as the default 128K model
5. Preserves OpenAI-compatible tool call history correctly across assistant/tool turns
6. Parses DeepSeek `reasoning_content` stream deltas into KodaX thinking blocks
7. Replays `reasoning_content` back to DeepSeek on tool turns where it is required

### 2. Scope

**Files changed**:

- `packages/ai/src/providers/base.ts` / `base.js`
- `packages/ai/src/providers/registry.ts` / `registry.js`
- `packages/ai/src/providers/openai.ts` / `openai.js`
- `packages/ai/src/providers/openai-reasoning-capability.test.ts`
- `packages/repl/src/common/utils.ts`
- `packages/repl/src/interactive/commands.ts`
- `packages/repl/src/ui/InkREPL.tsx`
- `packages/repl/src/interactive/provider-capabilities.test.ts`
- `tests/kodax_core.test.ts`
- `README.md`
- `README_CN.md`

### 3. Technical Design

#### 3.1 Built-in provider registration

`deepseek` is registered as an OpenAI-compatible built-in provider with:

- Base URL: `https://api.deepseek.com`
- API key env: `DEEPSEEK_API_KEY`
- Default model: `deepseek-chat`
- Additional model: `deepseek-reasoner`
- Context window: `128000`
- Max output tokens: `64000`

#### 3.2 Reasoning semantics

DeepSeek does not map cleanly to KodaX's existing `native-effort` bucket.

Actual shipped behavior:

- `deepseek-chat` uses provider-native thinking toggle via `extra_body.thinking = { type: 'enabled' }`
- `deepseek-reasoner` is exposed as a separate model and treated as model-selected reasoning, not `reasoning_effort`

That means FEATURE_036 explicitly does **not** claim "`deepseek-reasoner` = `native-effort`" anymore.

#### 3.3 Tool-call history replay

The previous OpenAI-compatible message conversion only forwarded text, which silently dropped:

- assistant `tool_use` blocks
- user `tool_result` blocks
- assistant thinking blocks

For DeepSeek this is especially important because tool turns can require the previous assistant message to carry both:

- `tool_calls`
- `reasoning_content`

The OpenAI-compatible conversion path now:

- serializes assistant `tool_use` blocks into OpenAI `tool_calls`
- serializes user `tool_result` blocks into `role: "tool"` messages
- preserves historical compacted `system` summary messages instead of discarding them
- includes DeepSeek `reasoning_content` only on assistant tool turns
- keeps non-tool assistant turns text-only, avoiding unnecessary replay of reasoning traces

#### 3.4 Stream parsing

DeepSeek `reasoning_content` streaming is now mapped into:

- `onThinkingDelta()`
- `onThinkingEnd()`
- `thinkingBlocks` in `KodaXStreamResult`

So the coding agent can persist those blocks and replay them on the next tool-result roundtrip.

### 4. User Contract

```bash
kodax --provider deepseek "Summarize this repository"
kodax --provider deepseek --model deepseek-reasoner "Think through this refactor plan"
```

Config example:

```json
{
  "provider": "deepseek",
  "model": "deepseek-chat",
  "reasoningMode": "auto"
}
```

User-visible behavior:

- `/model` now lists `deepseek`
- DeepSeek model lists include `deepseek-chat` and `deepseek-reasoner`
- REPL reasoning disclosure can resolve DeepSeek capability by active model
- built-in provider discovery, tests, and docs all count DeepSeek as a first-class provider

### 5. Implementation Notes

- The DeepSeek path is built on top of `KodaXOpenAICompatProvider`; no separate provider protocol was introduced
- OpenAI-compatible message replay is now materially more correct for all tool-using providers, not only DeepSeek
- DeepSeek support here covers built-in registration plus correct thinking/tool turn handling
- This feature does not add a DeepSeek-specific JSON mode or FIM-specific CLI surface

### 6. Acceptance Criteria

- [x] `--provider deepseek` can start KodaX / REPL flows
- [x] Default model is `deepseek-chat`
- [x] Users can switch to `deepseek-reasoner`
- [x] Built-in provider reads `DEEPSEEK_API_KEY`
- [x] Provider snapshot exposes 128K context window semantics
- [x] OpenAI-compatible tool call history is replayed correctly for DeepSeek turns
- [x] Historical compaction summaries are preserved for OpenAI-compatible providers
- [x] DeepSeek `reasoning_content` is streamed into KodaX thinking blocks
- [x] REPL/provider capability disclosure reflects `deepseek-chat` vs `deepseek-reasoner`

### 7. Verification

- `npm run build:packages`
- `npx vitest run packages/ai/src/providers/openai-reasoning-capability.test.ts packages/repl/src/interactive/provider-capabilities.test.ts tests/kodax_core.test.ts`

## FEATURE_039: Plan 模式双写白名单（项目说明文档 + 系统临时目录）

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### 1. 概述

当前 Plan 模式的核心价值是"先规划、后执行"，因此它默认应保持极强的写入约束，避免在规划阶段对工作区造成实质性变更。但完全禁止写入也带来了两个实际问题：

1. 无法把规划结果稳定落盘到项目内的固定说明文档
2. 无法安全地生成临时草稿、推导中间结果或一次性附件

这个 feature 为 Plan 模式增加一个严格收敛的双写白名单，在不放松整体安全边界的前提下，仅允许两个位置的文件修改：

1. 当前项目目录中的 `.agent/plan_mode_doc.md`
2. 当前操作系统的系统临时目录

除此之外，Plan 模式仍然保持禁止写入其他项目文件、用户主目录常规文件和任意非临时系统路径的行为。

### 2. 目标

1. **固定项目说明文档**
   - 允许 Plan 模式写入 `<workspace>/.agent/plan_mode_doc.md`
   - 该能力用于保存计划说明、分析摘要、待办拆分和阶段性结论
   - 白名单仅针对这个确定文件，不自动放开整个 `.agent/` 目录
   - **写入该路径不需要用户确认** — 这是 Plan 模式内建的放行规则，不触发 beforeToolExecute 审批

2. **支持系统临时目录**
   - 允许在系统临时目录内创建或修改临时文件
   - 适用于中间草稿、短生命周期导出物、调试附件和一次性分析结果
   - 临时目录白名单必须以真实系统临时目录解析结果为准，而不是仅凭路径名包含 `tmp`
   - **写入该路径不需要用户确认** — 同上，系统临时目录写入在 Plan 模式下自动放行

3. **LLM 感知：Plan 模式系统提示词增强**
   - Plan 模式的系统提示词应明确告知 LLM：这两个路径可用且无需确认
   - 提示 LLM 可以利用 `.agent/plan_mode_doc.md` 做规划记录、分析推导、待办拆分等
   - 提示 LLM 可以利用系统临时目录生成临时草稿、中间推导结果等
   - 这不是"允许 LLM 随意写入"，而是在已有的安全边界内让 LLM 知道有哪些落盘点可用

3. **跨平台语义清晰一致**
   - Windows: 使用 `%TEMP%` 或 `%TMP%`，典型路径为 `C:\Users\<user>\AppData\Local\Temp`
   - macOS: 优先使用 `$TMPDIR`，典型路径位于 `/var/folders/.../T/`
   - Linux: 优先使用 `$TMPDIR`，未设置时使用 `/tmp`
   - 实现需要把这些规则明确写入代码与文档，避免不同平台行为不一致

4. **白名单之外仍然严格禁止**
   - 不能因此放开普通项目文件
   - 不能放开任意 `.agent/*.md`
   - 不能放开工作区内自建的 `tmp/`、`temp/`、`.tmp/` 目录
   - 不能放开用户目录、桌面、下载目录等与 Plan 模式无关的位置

6. **跨模式一致性：Plan 放行的，其他模式也应放行**
   - Plan 是 KodaX 中权限控制最严格的模式
   - 如果 Plan 模式允许对这两个路径做免确认写入，则 Default 和 Accept-Edits 模式也应自然支持
   - 但 Default/Accept-Edits 模式本身已经允许更广泛的写入，所以这里的"一致性"指的是：
     - `.agent/plan_mode_doc.md` 在所有模式下都可写入，不因模式差异产生不同行为
     - 系统临时目录在所有模式下都可写入，不因模式差异产生不同行为
   - 实现建议：将这两个路径的白名单判定提取为模式无关的公共逻辑，所有模式共享

7. **路径校验必须可防绕过**
   - 统一做路径归一化和绝对路径解析
   - 防止 `..`、符号链接、junction、大小写差异和路径别名绕过白名单
   - 校验应发生在最终真实目标路径上，而不是用户原始输入字符串上

### 3. 范围

**预期涉及模块**:
- Plan/permission 模式判定逻辑
- 文件写入、编辑、重命名类工具的目标路径校验
- 错误提示与用户反馈文案
- Plan 模式相关文档与测试

**不在本 feature 范围内**:
- 放开整个 `.agent/` 目录
- 允许 Plan 模式执行任意 shell 写入
- 引入新的权限模式
- 允许写入工作区内任意缓存目录

### 4. 设计约束

#### 4.1 项目内允许写入的固定路径

唯一允许的项目内写入目标：

```text
<workspace>/.agent/plan_mode_doc.md
```

约束：
- 必须位于当前工作区根目录下
- 文件名必须精确匹配 `plan_mode_doc.md`
- 不接受同名软链接跳转到工作区外部
- 若 `.agent/` 不存在，可按需创建，但不能因此放开 `.agent/` 下其他文件

#### 4.2 系统临时目录约定

不同操作系统应按以下优先级解析系统临时目录：

```text
Windows:
  1. %TEMP%
  2. %TMP%
  Typical: C:\Users\<user>\AppData\Local\Temp

macOS:
  1. $TMPDIR
  Typical: /var/folders/.../T/

Linux:
  1. $TMPDIR
  2. /tmp
```

补充约束：
- 允许写入解析后的系统临时目录及其子路径
- 必须基于归一化后的真实路径判断是否位于临时目录内
- 不能把工作区里的 `tmp/`、`temp/` 或名字相似的目录误判为系统临时目录

#### 4.3 安全模型

Plan 模式中的写权限判断改为：

```text
allowWrite(path) =
  isWorkspacePlanDoc(path) ||
  isSystemTempPath(path)
```

但以下行为继续禁止：
- 写入任意普通源码、配置、测试或文档文件
- 在工作区任意位置新建非白名单文件
- 借助链接、相对路径或重命名把内容最终落到白名单之外

#### 4.4 Plan 模式系统提示词增强

Plan 模式下的系统提示词应增加以下引导，让 LLM 主动利用这两个落盘点：

```text
## Plan Mode: 可用写入路径

在 Plan 模式中，你可以将以下两个路径用于规划期间的讨论记录、分析推导和临时输出：

1. **项目规划文档**: <workspace>/.agent/plan_mode_doc.md
   - 用于保存计划说明、分析摘要、待办拆分、阶段性结论
   - 适合存放需要跨多轮对话保持的规划内容
   - 写入此文件不需要用户确认

2. **系统临时目录**: <system temp dir>
   - 用于生成临时草稿、中间推导结果、一次性分析附件
   - 适合存放短生命周期的辅助产出物
   - 写入此路径不需要用户确认

除上述两个位置外，Plan 模式禁止修改任何其他文件。
```

此提示词应在进入 Plan 模式时注入到系统提示词中，在 ExitPlanMode 时不需移除（因为只影响 Plan 模式的行为描述）。

#### 4.5 跨模式一致性

> 核心原则：Plan 是最严格的模式。Plan 放行的部分，在其他模式中不应被收回。

具体含义：

| 模式 | `.agent/plan_mode_doc.md` | 系统临时目录 | 其他项目文件 |
|------|--------------------------|-------------|-------------|
| **plan** | ✅ 免确认写入 | ✅ 免确认写入 | ❌ 禁止 |
| **default** | ✅ 可写入 | ✅ 可写入 | ⚠️ 需确认 |
| **accept-edits** | ✅ 可写入 | ✅ 可写入 | ✅ 自动放行 |
| **auto-in-project** | ✅ 可写入 | ✅ 可写入 | ✅ 自动放行 |

实现建议：
- 将白名单路径判定提取为模式无关的公共函数（如 `isAlwaysAllowedWritePath(path)`）
- 该函数在所有模式的权限判定中优先调用
- Plan 模式的特殊之处仅在于：白名单之外一律拒绝，而其他模式对白名单之外的路径仍有各自的判定逻辑

### 5. 用户可见行为

Plan 模式下：

- 写入 `.agent/plan_mode_doc.md` 应成功
- 写入系统临时目录应成功
- 写入其他任何位置应继续被拒绝
- 报错信息应明确告知当前仅允许的两个目标

推荐的错误提示方向：

```text
Plan mode only allows file modifications in:
1. <workspace>/.agent/plan_mode_doc.md
2. <system temp dir>
```

### 6. 验收标准

- [ ] Plan 模式仍默认禁止绝大多数文件写入
- [ ] `<workspace>/.agent/plan_mode_doc.md` 可以被创建和修改
- [ ] 仅这个固定项目文件被允许，`.agent/` 下其他文件仍被拒绝
- [ ] 写入这两个白名单路径**不需要用户确认**
- [ ] Windows 下按 `%TEMP%`/`%TMP%` 识别系统临时目录
- [ ] macOS 下按 `$TMPDIR` 识别系统临时目录
- [ ] Linux 下按 `$TMPDIR` 或 `/tmp` 识别系统临时目录
- [ ] 路径归一化后仍能拦截 `..`、符号链接或大小写绕过
- [ ] 工具报错信息能清楚说明允许写入的位置
- [ ] Plan 模式的系统提示词包含白名单路径引导，LLM 可主动利用这两个路径做规划和记录
- [ ] 白名单路径判定提取为模式无关的公共函数，Default/Accept-Edits 模式共享同一逻辑
- [ ] Default 模式下，这两个路径的写入不触发额外确认（行为与 Plan 一致：免确认）

### 7. 测试建议

1. 权限正向用例
   - 写入 `<workspace>/.agent/plan_mode_doc.md`
   - 写入系统临时目录中的新文件
   - 写入白名单路径**不触发 beforeToolExecute 审批回调**

2. 权限反向用例
   - 写入 `<workspace>/.agent/other.md`
   - 写入 `<workspace>/tmp/foo.txt`
   - 写入工作区普通源码文件
   - 写入用户目录中的非临时文件

3. 路径绕过用例
   - 使用 `..` 指向白名单外目标
   - 使用符号链接/junction 指向白名单外目标
   - Windows 大小写变化与短路径别名

4. 跨模式一致性用例
   - Default 模式下写入 `<workspace>/.agent/plan_mode_doc.md` 不触发额外确认
   - Accept-Edits 模式下写入系统临时目录不触发额外确认
   - 验证 `isAlwaysAllowedWritePath()` 在所有模式下返回一致结果

5. 提示词验证用例
   - 进入 Plan 模式后，验证系统提示词包含白名单路径引导文本
   - LLM 在 Plan 模式下主动写入 `.agent/plan_mode_doc.md` 做规划记录

### 8. 备注

这个 feature 的重点不是"让 Plan 模式能写更多地方"，而是为规划阶段补足两个必要且可控的落盘出口，同时继续保持 Plan 模式相对于 Default/Accept-Edits 模式更强的边界约束。

三个关键设计决策：
1. **免确认写入**：这两个白名单路径的写入不需要用户确认，这是 Plan 模式内建的放行规则，不是审批通过的
2. **提示词引导**：通过系统提示词让 LLM 主动利用这两个路径做规划记录，而不是靠 LLM 猜测
3. **跨模式一致性**：Plan 是最严格的模式，Plan 放行的路径在其他模式中也应自然可用（提取为公共逻辑）

## FEATURE_040: ACP Server 支持（供编辑器/IDE 直接调用）

**Status**: Completed
**Priority**: High
**Category**: New

### 1. 概述

当前仓库已经有 ACP 相关基础设施，例如 `acp-client`、`acp-base` 和 `pseudo-acp-server`。但这些能力主要服务于内部桥接和 provider 适配，尚未把 KodaX 作为一个正式、稳定、可被第三方宿主直接调用的 ACP Server 暴露出来。

这个 feature 的目标，是让 KodaX 可以作为标准 ACP agent runtime 被外部编辑器、IDE 和其他支持 ACP 的工具直接接入。换句话说，不再只是"KodaX 去连接别人"，而是"别人也可以把 KodaX 当成 ACP agent 来调用"。

### 2. 目标

1. **正式对外暴露 ACP Server**
   - 提供稳定的 KodaX ACP server 启动入口
   - 支持通过 stdio 或约定命令行方式被外部工具拉起
   - 不要求第三方了解 KodaX 内部 provider/bridge 细节

2. **支持外部宿主的核心会话流程**
   - 支持 `initialize`
   - 支持 `sessions/new`
   - 支持 `chat/prompt`
   - 支持 `chat/cancel`
   - 支持流式 session update 返回

3. **保留 KodaX 现有能力语义**
   - 对外暴露的仍是 KodaX 的真实工具执行、权限模式和中断语义
   - tool call / tool result / assistant text delta 应尽量保持与 REPL 体验一致
   - 外部编辑器不应因为 ACP 包装而丢失 KodaX 的关键运行反馈

4. **权限模型与 REPL 保持一致**
   - ACP 模式复用与 REPL 相同的四级权限模式：`plan` / `default` / `accept-edits` / `auto-in-project`
   - ACP 只是 transport，不引入独立权限系统
   - 宿主若不支持审批回传，默认应 fail-closed，而不是自动放行高风险操作

5. **为编辑器/IDE 接入提供稳定契约**
   - 明确启动命令、参数、cwd 处理方式
   - 明确 session 与权限请求语义
   - 明确哪些能力在 v1 支持，哪些能力暂不支持

6. **把内部 ACP 基础设施提升为产品能力**
   - 可以复用现有 `acp-client` / `pseudo-acp-server` / `acp-base` 的经验
   - 但最终交付必须是面向外部集成者的公共接口和文档，而不是内部实验能力

### 3. 范围

**预期包含**:
- 新增 KodaX 的 ACP server 启动入口
- ACP 握手、session、prompt、cancel 和流式通知
- 工具调用和执行状态向 ACP 事件映射
- 错误处理、能力声明和最基本的集成文档

**暂不要求在这个 feature 内完成**:
- 所有 ACP 扩展能力一次性全量实现
- GUI 宿主专属优化
- 针对每个编辑器单独做定制协议分支
- 新建另一套独立 agent runtime

### 4. 设计方向

#### 4.1 交付形态

建议提供类似下面的用户可见入口：

```bash
kodax acp serve
```

或等价的显式 server 模式参数，使外部工具可以用统一命令把 KodaX 作为 ACP server 拉起。

#### 4.2 协议职责

KodaX 在该模式下需要承担：

- ACP 连接初始化
- Session 创建与恢复
- Prompt 请求接收
- 取消信号处理
- 流式文本输出
- 工具调用与状态更新通知
- 权限请求与决策回传

#### 4.3 与现有 ACP 代码的关系

- `acp-client` 说明仓库已经具备 ACP 客户端侧基础能力
- `pseudo-acp-server` 说明已有事件映射和桥接思路
- `acp-base` 说明 ACP 已经进入 provider 抽象层

但这些基础还不足以直接等同于"外部工具可稳定调用的 KodaX ACP Server"。本 feature 需要把这些分散能力统一到正式的对外 server runtime 中。

#### 4.4 权限语义对齐 REPL

ACP 模式下的权限限制应与 REPL 模式保持一致：

- `plan`
  - 默认禁止写入与高风险操作
  - 如果 `FEATURE_039` 落地，则仅允许 `.agent/plan_mode_doc.md` 与系统临时目录写入
- `default`
  - 高风险操作需要显式审批
  - 通过 ACP 的 permission request 与外部宿主交互
- `accept-edits`
  - 文件编辑类操作可自动执行
  - 更高风险操作仍需要审批
- `auto-in-project`
  - 项目内允许的操作可自动执行
  - 仍受工作区边界和既有安全规则约束

关键要求：
- ACP server 必须复用 REPL 现有权限层/判定逻辑，而不是复制一份新规则
- 权限决定必须作用在真实工具执行前，而不是只做 UI 级提醒
- 外部宿主若不支持 permission request，KodaX 必须对需要审批的操作 fail-closed

#### 4.5 v1 能力边界

v1 建议优先保证：

- 单会话和多会话基本可用
- prompt/cancel 稳定
- 文本流更新稳定
- tool call / tool result 更新可观察
- 基础权限请求可回传

不要求 v1 一次性覆盖所有高级宿主能力，但必须让"编辑器把 KodaX 当 agent 调起来并稳定跑通一次完整任务"成为可行路径。

### 5. 用户可见价值

接入后，外部编辑器/IDE 可以：

- 直接拉起 KodaX 作为 ACP agent
- 向 KodaX 发送自然语言任务
- 接收流式响应与工具执行进度
- 复用 KodaX 现有的模型/provider/tooling 能力

这会让 KodaX 从"自己的 CLI/REPL 产品"进一步升级为"可被其他宿主嵌入的 agent runtime"。

### 6. 验收标准

- [x] KodaX 提供明确、稳定的 ACP server 启动入口
- [x] 外部工具可以完成 `initialize` 与 `sessions/new`
- [x] 外部工具可以通过 `chat/prompt` 发起请求
- [x] `chat/cancel` 可以中断正在运行的请求
- [x] assistant 文本可以通过流式 session update 返回
- [x] tool call / tool result 状态可以通过 ACP 更新事件观察到
- [x] ACP 模式复用与 REPL 相同的 `plan / default / accept-edits / auto-in-project` 权限语义
- [x] 默认模式下，需要审批的操作通过 ACP permission request 请求宿主确认
- [x] 宿主不支持 permission request 时，需要审批的操作 fail-closed，而不是自动放行
- [x] `plan` 模式下的限制与 REPL 一致；若 `FEATURE_039` 落地，则同步继承其双写白名单例外
- [x] README 或独立集成文档能说明如何把 KodaX 接入编辑器/IDE

### 6.1 Implementation Notes

- 新增 `src/acp_server.ts`，提供基于 `@agentclientprotocol/sdk` 的正式 ACP server runtime
- 新增 `kodax acp serve` CLI 入口，并补充 `kodax -h acp` 帮助主题
- ACP prompt 生命周期已接入 `runKodaX`，并桥接 assistant 文本流、tool call、tool result、mode update 与错误文本
- ACP 权限审批复用 REPL permission helpers，并支持 `allow_always` 将 session 从 `default` 切换到 `accept-edits`
- 新增 ACP 协议测试，覆盖流式通知、permission request、mode upgrade 与取消链路

### 7. 测试建议

1. 协议正向用例
   - 启动 ACP server 并完成 `initialize`
   - 创建 session 并发送 prompt
   - 接收完整文本流输出

2. 工具调用用例
   - 触发至少一个工具调用
   - 验证 tool call running/update 事件顺序

3. 中断与错误用例
   - 请求中发送 `chat/cancel`
   - 非法请求返回可诊断错误
   - server 退出时资源正确清理

4. 权限对齐用例
   - `plan` / `default` / `accept-edits` / `auto-in-project` 四种模式下，对同一工具调用得到与 REPL 一致的判定结果
   - 宿主支持 permission request 时，`default` 模式能够正确弹出审批并回传结果
   - 宿主不支持 permission request 时，需要审批的操作被拒绝

5. 集成用例
   - 用最小 ACP 客户端脚本拉起 KodaX 并完成一次任务
   - 验证 cwd、sessionId 和权限请求在外部宿主场景下可用

### 8. 备注

这个 feature 的核心不是"多一个协议名词"，而是把 KodaX 变成真正可以被外部产品嵌入的 agent 基础设施。对外可调用的 ACP Server 一旦成立，后续编辑器集成、IDE 插件和宿主化工作都会容易很多。
