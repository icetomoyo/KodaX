> 中文标题：`v0.7.48 Feature 设计`
>
> Version: `v0.7.48`
>
> Status: Planned (not yet released)
>
> Theme: Harness Observability Substrate — KodaX 的"agent-readable evidence layer"。读 arxiv:2604.25850 (AHE) 后给 KodaX 整体校准的最小必要补强：补 long-term memory 这一首类组件 + 把 self-modify 的 audit log 升级为 falsifiable contract（仅 prompt，零新字段）+ 守住 cross-family transfer 的 prose-prose 边界。零新依赖、零新 package、零新抽象层。

## 总览

读 [arxiv:2604.25850](https://arxiv.org/abs/2604.25850)（Agentic Harness Engineering, AHE）后对 KodaX 整体做了一次产品级校准。论文核心实证：同一个 GPT-5.4，opencode 47.2% → Codex CLI 71.9% → AHE 77.0%（**30pp harness gap**，比换 model 还大）；component ablation 显示 long-term memory 单独 +5.6pp（最大单贡献，Hard 任务上 memory-only 63.3% 超过 full AHE 53.3%）；**system prompt 单独 -2.3pp**（prose 不 transfer，structure transfer）；fix precision 33.7%（5×random）vs regression precision 11.8%（2×random，self-attribution 对 regression 基本盲）。

**对 KodaX 整体的判断**：

1. KodaX 7 类首类组件里只有 long-term memory 系统性缺失（grep `long.?term.?memory|cross.?session.?memory|persistent.?memory` 全仓 0 命中）。FEATURE_108 reflective patcher (v0.7.47) 想消费的"沉淀型经验"恰好就是这一层
2. FEATURE_090 audit log (v0.7.32) 是 passive ledger，没有 forward-looking prediction —— 论文 Decision observability 的 falsifiable contract 概念可以零新字段实现（在 `llmSummary` 自然语言里写 prediction，下次 self-modify 时 LLM 自己回看是否兑现）
3. KodaX 12 provider 是真实 cross-family pool，论文证 cross-family transfer +5-10pp 的硬约束是 **prose 不能按 provider 分叉**。需要一个机械单测守住这条边界，否则 v0.7.32-0.7.47 期间任何 prompt 改动都可能无意中破坏 transfer 红利

**本 feature 哲学校准**：经过两轮自我审视和两个对立子 agent 辩论（极简哲学捍卫者 + 产品长期价值评估者），从最初列的 14 条压缩到 3 条工程任务 + 1 个 sanity check + 2 条 CLAUDE.md 原则更新。所有砍掉的条目（cross-step risk pattern registry / rollback CLI / declarative middleware / outcome-driven tool description 优化机制 / fix-regression metric panel）要么违反"3+ real cases 才抽象"，要么把"Leverage LLM intelligence"误解为"工程化的抽象层"。本 feature 严格遵循：**LLM 在 markdown / JSONL 文件上读写 + user approve gate + 现有 tool 复用**，零新抽象层。

## FEATURE_109: Harness Observability Substrate — Long-Term Memory + Prediction Contract + Cross-Family Prose Guard

**Status**: Planned  
**Priority**: Medium  
**Category**: Internal / Substrate

### 背景

#### 现状

KodaX 截至 v0.7.47 已经具备的 harness substrate（对照论文 NexAU 7 类组件）：

| 论文组件类 | KodaX 对应 | 状态 |
|---|---|---|
| system prompt | [`agent-runtime/system-prompt-*.ts`](../../packages/coding/src/agent-runtime/) | ✅ |
| tool description | 50+ tool 的 `description` 字段 | ✅ |
| tool implementation | [`packages/coding/src/tools/`](../../packages/coding/src/tools/) | ✅ |
| middleware | [`agent-runtime/middleware/`](../../packages/coding/src/agent-runtime/middleware/) (13 个) | ✅ 比 AHE final 还多 |
| skill | [`packages/skills/`](../../packages/skills/) Agent Skills 标准 | ✅ |
| sub-agent | [`packages/agent/`](../../packages/agent/) + AMA Scout/Generator | ✅ |
| **long-term memory** | grep 全仓 0 命中 | ❌ |

KodaX 截至 v0.7.47 已有的 governance / observability：

- FEATURE_090 (v0.7.32) self-modify audit log + budget cap + guardrail ratchet（governance 层比论文严，user-as-final-authority 而非 sandbox）
- FEATURE_101 (v0.7.31) admission contract 8 invariants（机械可执行的 controllability，相对论文"non-deletable 约定"是优势）
- FEATURE_102 (v0.7.45) telemetry / Phase 0
- FEATURE_104 (v0.7.29) prompt-eval harness + EVAL_GUIDELINES
- FEATURE_105 (v0.7.46) cross-family advisor primitive
- FEATURE_108 (v0.7.47) session-driven reflective prompt patcher

#### 论文揭示的三个对 KodaX 整体有产品级意义的 finding

1. **Long-term memory 是最大单贡献组件**：+5.6pp 单独，Hard 任务上 memory-only 63.3% 超过 full AHE 53.3%。论文里就是 12 条 boundary-case lessons 的 markdown 文件（performance margin / queued-over-limit cancellation / evaluator-style closure / source-packaging layout）
2. **Prose 不 transfer，structure transfer**：system_prompt 单独 -2.3pp；tools/middleware/memory 各 +2-6pp。"factual harness structure transfers across tasks and models whereas prose-level strategy does not"
3. **Self-attribution 对 regression 基本盲**：fix precision 5×random vs regression precision 2×random。对 KodaX 所有 LLM-as-judge 用法（review / verdict / judges middleware）都是硬约束

#### 与 FEATURE_108 (v0.7.47) 的协同关系

FEATURE_108 reflective prompt patcher 的 Phase 1 mine-failures 在 telemetry / session-lineage 上挖失败簇。本 feature 的 §A1 long-term memory 是 FEATURE_108 reflector 的天然消费源 —— reflector 不仅看失败 trace，也看 KodaX agent 自己积累的 lessons.md，避免对已知 boundary case 重复提候选 patch。本 feature ship 在 FEATURE_108 之后，**给 FEATURE_108 的下一次迭代提供 substrate**，不构成 FEATURE_108 ship 的硬阻塞。

### 核心判断（不能违反的设计原则）

1. **不新增 package、不新增依赖、不新建抽象层**：long-term memory 是一个 markdown 文件 + 复用 self-modify 的 ask-user 通路；prediction contract 是改 prompt 不改 schema；prose guard 是 1 个单测
2. **LLM 在自然语言里工作，不要 schema/registry**：prediction 写在 `audit-log.ts` 已有的 `llmSummary` 字符串字段，不加新字段；下次 self-modify 时把上次的 `llmSummary` 文本喂回当前 LLM 让它自己判断兑现否
3. **Memory 是 LLM 写入，user approve 才落地**：复用 FEATURE_090 的 ask-user 通路；KodaX 单用户 CLI 威胁模型（DD §14.5）"workspace 完整性是 user 责任"在此延续
4. **Prose guard 是机械约束不是治理流程**：单测断言 prompt 文案不出现 provider 名字字符串；不引入"prompt 改动需 review"等流程化负担
5. **不演化、不自动化、不 emergent loop**：MVP 不做 LLM 自动决定何时 update memory（每次 update 必须 user approve）；不做 prediction 自动 verification（LLM 自己读自己判断）；不做 emergent reasoning loop
6. **3+ real cases 没到的方向不做**：cross-step risk pattern registry / rollback CLI / declarative middleware / outcome-driven tool description 优化机制 / fix-regression metric panel — 全部不做（详见 §显式不做）
7. **零运行时开销**：memory 是 markdown 读 / prediction 是 prompt 字段 / prose guard 是单测 —— 都不在热路径

### 设计

#### Slice A1 — Long-Term Memory：一个 markdown 文件 + 读写 + approve gate

**位置**：`<cwd>/.kodax/agent-memory.md`（与 FEATURE_090 的 `.kodax/constructed/_audit.jsonl` 同一目录树）

**格式**（约定，不是 schema）：

```markdown
# KodaX Agent Long-Term Memory

> 这个文件由 KodaX agent 写入，记录跨 session 的项目 boundary cases / 经验教训。
> 每次更新必须经过 user approve（CLI 提示）。
> User 可手动编辑此文件 —— agent 下次 session 自动读取。

## 2026-05-01: TypeScript strict mode 在本仓 tsconfig 之外仍为 true

context: ...
lesson: ...

## 2026-05-03: vitest 在 monorepo root 跑会拉到所有 package 的 test

context: ...
lesson: ...
```

**接口**（只新增三个最小函数，不抽象成 Repository / Manager）：

```ts
// packages/coding/src/construction/long-term-memory.ts

export interface MemoryReadOptions {
  readonly cwd?: string;
}

/** 读取整个 memory.md。文件不存在返回 ''。无解析、无结构化。 */
export async function readMemory(options?: MemoryReadOptions): Promise<string>;

export interface MemoryAppendInput {
  readonly cwd?: string;
  /** Agent 提议的新 lesson markdown 片段（已含 `## YYYY-MM-DD: title` heading）。 */
  readonly proposedAppend: string;
  /** 复用 FEATURE_090 ask-user 通路。返回 false = user 拒绝，append 不发生。 */
  readonly approve: () => Promise<boolean>;
}

/** Append 一段新 lesson 到文件末尾。审计日志（_audit.jsonl）写一条 `agent_memory_appended` 事件。 */
export async function appendMemory(input: MemoryAppendInput): Promise<{ appended: boolean }>;
```

**消费侧**（不强制接入，先给 FEATURE_108 reflector + 自由 KodaX agent prompt 提供 read 入口）：

- `runKodaX` 入口在构造 system prompt 时附加一段 "## Project lessons learned"，内容来自 `readMemory()`（截断到 ~2000 token 上限以防 prompt 膨胀）
- KodaX agent 想沉淀新 lesson 时，通过新 tool `propose_long_term_memory_append` 触发 `appendMemory()`（user 拒绝则 lesson 丢失，下次 agent 看到再决定要不要重提）

**显式不做**：

- ❌ Memory database / vector store / retrieval 抽象 / embedding —— 一个 markdown 文件 user 自己能读
- ❌ Memory diff / merge / conflict resolution —— append-only，user 手动编辑就是合并
- ❌ Cross-project memory（global memory）—— 范围限定 `<cwd>/.kodax/`，符合 KodaX project-local 一贯原则
- ❌ Auto-trim / 压缩 —— 文件膨胀时由 user 手动整理或删除，单用户 CLI 不需要 retention 策略
- ❌ Memory 不进 git 默认（写入 `.kodax/.gitignore` 默认条目，但 user 可手动 commit 个人版）

#### Slice A2 — Prediction Contract：改 prompt，零新字段

**修改点**：FEATURE_090 self-modify 流程的 LLM diff summary prompt（在 `audit-log.ts` 之外的 prompt 文件）。

**Prompt 增量**（追加到现有 LLM diff summary instructions 末尾）：

```
After describing what changed, add a section "Predicted impact":
- Which scenarios / failure patterns this change is expected to improve
- Which scenarios / failure patterns are at risk of regression
- A self-confidence note: "I am uncertain about regression detection — please flag any
  regression in the next session for explicit verification"

Write in natural language. No bullet schema, no JSON.
```

**回看机制**（同样改 prompt，无新代码）：

下次同 agent self-modify 触发时，在 LLM diff summary prompt 里 inject 上次 audit entry 的 `llmSummary` 全文：

```
[Previous self-modify session for agent '<name>']:
<上次 llmSummary 全文，含 Predicted impact 段>

[Your task this session]:
1. Look at the previous "Predicted impact" — did the predictions hold?
   For improvements: are the named scenarios actually better now?
   For regressions: did any of the at-risk scenarios actually regress?
2. Write your own diff summary for THIS session's change, including a fresh "Predicted impact" + a
   "Verification of previous prediction" section.
```

**新增 audit event kind**（minimal 增加，符合 [audit-log.ts:53](../../packages/coding/src/construction/audit-log.ts#L53) 现有 closed set 范式）：

- `'agent_memory_appended'`（Slice A1 用，本 slice 不直接产出此事件）

**不新增**：

- ❌ `predictedFixes` / `predictedRisks` / `verificationVerdict` 等 typed 字段 —— prediction 在 `llmSummary` 自然语言里
- ❌ Falsifiability check 函数 / 规则引擎 —— LLM 看自然语言自己判断
- ❌ History reader 自动喂回机制 —— Slice 实施时直接在 prompt 拼接代码里读上一条 entry，不抽象成"history reader"
- ❌ 跨轮 attribution 度量 / fix-regression precision-recall 仪表盘

**Acceptance**：

- [ ] FEATURE_090 self-modify 跑出的第一个 audit entry 的 `llmSummary` 包含明确的 "Predicted impact" 段
- [ ] 同一 agent 第二次 self-modify 时，LLM 在 `llmSummary` 里写出 "Verification of previous prediction" 段（基于上次的 prediction）
- [ ] Prompt 在 FEATURE_104 prompt eval 全 alias 下绿（任何 LLM-facing prompt 改动都触发 prompt eval，[CLAUDE.md](../../CLAUDE.md) 已规定）

### CLAUDE.md 原则更新

本 feature ship 时同步更新 [CLAUDE.md](../../CLAUDE.md) 加入两条原则（论文实证支撑）：

#### 原则 1: Provider 分支位置

> Provider-specific 分支只能出现在 LLM-call boundary（max_tokens / reasoning_effort / wire-level 协议参数等），**不能延伸到 prompt prose**。
>
> 论文证据：harness 跨 base model frozen-transfer +5-10pp（[arxiv:2604.25850](https://arxiv.org/abs/2604.25850) §4.3），远离 saturation 的 base model 收益最大；transfer 红利的前提是 prose 不分叉。
>
> KodaX 12 provider 是真实 cross-family pool。给 Anthropic 一份 prompt、给 Qwen 另一份等于把 12 provider 退化成 12 个独立产品。
>
> 机械守门：见 v0.7.48 FEATURE_109 Slice B2 单测。

#### 原则 2: LLM 自我评估的不可逆操作边界

> LLM 自我评估输出（review verdict / self-modify diff summary / judges middleware 输出 / advisor verification）**不能直接驱动不可逆操作**（git commit / push / 文件删除 / 部署 / 跨 session 状态写入），必须经过 user approve 或正交 control case 兜底。
>
> 论文证据：fix prediction 5×random（可靠），regression prediction 2×random（基本盲）（[arxiv:2604.25850](https://arxiv.org/abs/2604.25850) §4.4.2）。LLM 对自身行为的副作用判断有系统性偏差。
>
> KodaX 范式：所有 self-modify / 跨 session memory append / advisor verdict 都走 ask-user 通路；review/judge 输出供 user 决策不直接 ship。

### 显式不做（避免被未来误解）

#### 论文有数据但不做的方向（违反 KodaX 哲学）

- ❌ **Cross-step risk pattern registry / RiskPatternRegistry 数据结构**：现有 13 个 middleware (auto-reroute / mutation-reflection / scope-aware-harness-guardrail / edit-recovery / tool-outcome-tracking 等) hardcode 已 work；抽 registry 是为零次发生的扩展场景设计；论文 7 个 pattern 是 evolve agent *自己发现* 的，KodaX 还没积累到等价数据集
- ❌ **`kodax constructed rollback` CLI**：FEATURE_090 manifest version-immutable + budget cap 已够；真要 rollback 用 `git revert` + 删 `.kodax/constructed/agents/<name>/<v>/` 即可；写一段 troubleshooting 文档而非 CLI 命令
- ❌ **Tool description outcome-driven 优化机制**：论文 tools +3.3pp 主要靠 description 升级；但抽 "outcome 反馈调 description" 机制是巨大工程；正确路径是 §A1 + §A2 落地后让 KodaX agent 用现有 trace + memory 自己提 description PR 候选（emergent 而非工程化）
- ❌ **Self-modify 扩到 tool description**：governance scope 极大扩展；至少需要 §A1 + §A2 + 1 个月稳定数据 + 3+ 真实需求驱动才考虑
- ❌ **Declarative middleware（agent 自己写新 middleware 代码）**：扩 self-modify scope 到代码级；现有 13 个 middleware 已经声明式（在 declaration 里 enable）；agent 写 middleware 代码超出当前 governance 边界
- ❌ **跨轮 attribution 度量 / fix-regression precision-recall 仪表盘**：工程化 metric panel 在 single-user CLI 场景意义不明；KodaX 不跑 benchmark loop；正确做法是把 disclaimer 写进 LLM-as-judge prompt（CLAUDE.md 原则 2 已覆盖）

#### Slice 内的过度工程

- ❌ **Memory typed schema / structured field**：markdown free-form，LLM 自己组织
- ❌ **Memory retrieval / embedding / vector store**：read 整个文件 + 截断 token，足够 KodaX 项目级规模

- ❌ **History reader 抽象成独立 capability**：实施时直接在 prompt 拼接处读上一条 audit entry，不抽象

### 风险与待解决问题（实施时需正面回答）

1. **Memory 文件膨胀风险**：长期使用后 memory 可能膨胀到 prompt 注入超 token 预算。Mitigation: 注入时 truncate 到 ~2000 token（最近 N 条 lessons by ## heading）；user 手动整理 / 删除老 lessons 是预期工作流（单用户 CLI，user 直接编辑 markdown）
2. **Memory 内容质量风险**：LLM 可能写质量低的 lessons（重复、模糊、错误归因）。Mitigation: 每次 append 必须 user approve；user 是质量门
3. **Memory 跨项目错配**：用户在项目 A 学到的 lesson 可能在项目 B 不适用。Mitigation: 范围限定 `<cwd>/.kodax/`，不做 global memory；与 KodaX project-local 一贯原则一致
4. **Prediction 兑现判断的循环依赖**：LLM 判断"上次 prediction 是否兑现"本身受 self-attribution 偏差影响（论文 §4.4.2）。Mitigation: 接受这是 best-effort 信号；CLAUDE.md 原则 2 兜底（user approve 不可逆操作）；不把 LLM 的兑现判断作为自动决策依据
5. **Prose guard 误伤**：单测可能误伤合理用法（如文档里提到 provider 名字）。Mitigation: 单测只覆盖 4 个 prompt 文件家族里的常量字符串字面量，不扫整个 codebase；存量违规如不能简单修复，标记为已知 limitation 单独 issue 跟踪
6. **§A3 sanity check 不通过**：若 KodaX agent 不能用现有 tools 读 `.kodax/`，需补 ≤50 行胶水。Mitigation: ship 前必须验证；不能用就在 Phase 1/2/3 任意阶段顺手补
7. **与 FEATURE_108 的 sequencing**：FEATURE_108 (v0.7.47) 先 ship，本 feature (v0.7.48) 后 ship。FEATURE_108 ship 时无 memory 可读 = 它的 reflector 只读 trace；v0.7.48 ship 后 FEATURE_108 的下一次迭代可消费 memory。本 feature 的 Acceptance 要求 FEATURE_108 设计稿更新一段说明这个消费关系
