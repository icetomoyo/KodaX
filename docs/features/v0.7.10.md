## 总览

`v0.7.10` 在 `v0.7.0` 的 task engine 基础上补齐了三件事：

- 仓库智能底座
- `SA / AMA` 模式控制面
- retrieval / evidence tooling

这些能力后来都被继续收敛进当前更“极简且智能”的 AMA 设计里。

## FEATURE_018: Task-Aware Repository Intelligence Substrate

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_018` 提供 durable repository intelligence，供 task engine 查询。

它服务于：

- `Scout` 的 scope 判断
- `Planner` 的 overview evidence
- `Generator` / `Evaluator` 的定点取证

### 当前语义

repository intelligence 是 task-shaped retrieval substrate，而不是单独的 wiki 产品。

它继续提供：

- `repo_overview`
- `changed_scope`
- `module_context`
- `symbol_context`
- `process_context`
- `impact_estimate`

### Acceptance criteria

- [x] intelligence artifacts 可持久化
- [x] retrieval 结果具备 freshness / confidence / evidence 元数据
- [x] AMA 可把 intelligence 作为 `scope facts` / `overview evidence` 的一部分消费

## FEATURE_027: Adaptive Multi-Agent Mode Toggle and Team-Mode Sunset

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_027` 让 `SA / AMA` 成为正式的产品控制面。

今天它的正确语义是：

- `SA`：完全脱离 AMA 的 single-agent path
- `AMA`：允许 task engine 在 `H0 / H1 / H2` 中自适应选择

### Mode semantics

#### `AMA`

`AMA` = `Adaptive Multi-Agent`

当前只保留：

- `H0_DIRECT`
- `H1_EXECUTE_EVAL`
- `H2_PLAN_EXECUTE_EVAL`

`H3_MULTI_WORKER` 已删除。

当前补充语义：

- `H0` 的核心是最终单 agent 收口；它既可以是 direct path，也可以是 `Scout-complete H0`
- `read-only / docs-only` 默认停留在 `H0`；只有用户明确要求 `double-check / second pass / 更强审查` 时才进入 `H1`
- `read-only / docs-only` 永远不进入 `H2`
- `H1` 是 `Generator + 轻量 Evaluator` 的 lightweight checked-direct，不包含 `Planner`、contract negotiation 或默认多轮 refine
- `H2` 只服务真正长时的 `code / system` mutation work，并默认单主 pass

#### `SA`

`SA` = `Single-Agent`

当前语义：

- 不进入 Scout / managed worker graph
- 不创建 AMA breadcrumb / round / budget ceremony
- 直接走 single-agent runner
- 如果 skill 被激活，则完整 skill 在 direct path 中生效

### UX contract

用户需要明确感知：

- 当前处于 `SA` 还是 `AMA`
- `AMA` 的预算主语义是 `Work`
- `Round` 只在真实额外 pass 出现

### Relationship to `--team`

`--team` 与 `AMA` 不是同一个东西：

- `AMA` 是 task engine 的正式执行模式
- `--team` 是已退役的旧并行入口

### Acceptance criteria

- [x] 用户可以在 `SA / AMA` 间切换
- [x] 状态栏显示 `KodaX - SA` / `KodaX - AMA`
- [x] `SA` 真正绕开 AMA
- [x] `AMA` 只保留 `H0/H1/H2`
- [x] `--team` 不再是主产品故事的一部分

### Implementation summary

当前落地包括：

- CLI `--agent-mode sa|ama`
- REPL `/agent-mode`
- `Alt+M`
- 状态栏模式显示
- `SA` direct path
- `AMA` adaptive harness selection
- `Project + SA` lightweight run record

## FEATURE_028: First-Class Retrieval, Context, and Evidence Tooling

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### Overview

`FEATURE_028` 是 task engine 的 retrieval / evidence 工具层。

它的目标不是“工具越多越好”，而是：

- 让 `Scout / Planner / Generator / Evaluator` 有分层可用的取证工具
- 减少 ad hoc `grep -> read -> grep again` 的无结构循环

### 当前语义

AMA 现在会按证据层次消费这些工具：

- `scope facts`
- `overview evidence`
- `deep evidence`

大型 review 时，优先走：

```text
changed_scope -> repo_overview -> changed_diff_bundle -> changed_diff -> read
```

并且：

- `Planner` 默认止步于 bundle / overview 层
- `Generator` 才负责 deep diff paging
- `Evaluator` 只做 targeted spot-check
- `Scout / Planner / Evaluator` 跨轮会读取 compact same-role summary，而不是恢复完整私有历史
- `reviewScale`、repo 规模、改动文件数/行数只影响 evidence strategy，不再决定 `H0 / H1 / H2`

### Acceptance criteria

- [x] retrieval tools 具备 progressive disclosure
- [x] 大 review 能优先使用 bundle / scoped evidence
- [x] 工具摘要对用户尽量披露目标对象/范围
