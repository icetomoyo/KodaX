## FEATURE_018: Task-Aware Repository Intelligence Substrate

**Status**: Completed
**Priority**: High
**Category**: New

### Overview

`FEATURE_018` replaces the old "CodeWiki" product idea with a task-aware repository intelligence substrate.

Its job is to maintain local, incrementally refreshable, inspectable intelligence about the current codebase so KodaX can retrieve the right context for the current task without repeatedly reconstructing the repo from raw `grep` and `read` flows.

This substrate should serve:

- planner context
- generator context
- evaluator context
- repo rule synthesis
- task-scoped architectural understanding

It is not a wiki product, a graph-visualization product, or a documentation site generator. It is the repository intelligence layer that the task engine can query, refresh, and carry forward over time.

### Goals

1. Build durable repository intelligence that survives beyond one session.
2. Support incremental refresh instead of full re-ingestion.
3. Serve task-relevant context through progressive disclosure under token budgets.
4. Expose inspectable, repo-friendly artifacts rather than opaque memory blobs.
5. Support multiple languages through explicit capability and confidence tiers instead of pretending all languages have equal depth.

### Scope

- repository overview capsules
- module, symbol, process, impact, and changed-scope capsules
- incremental refresh policy
- freshness and confidence metadata
- language capability tiering
- evidence-friendly excerpts and file references

### Product shape

The primary shape of `018` is not "pages". The primary shape is task-native intelligence artifacts that later features can query.

Recommended first-class intelligence objects:

- `repo_overview`
- `module_context`
- `symbol_context`
- `process_context`
- `impact_estimate`
- `changed_scope`

If KodaX later emits human-readable summaries or onboarding briefs, those should be derived views of this substrate rather than the substrate itself.

### Detailed design notes retained

The older CodeWiki direction still contained several ideas worth keeping, but they now belong under a different abstraction:

- store intelligence as repo-friendly artifacts rather than opaque memory blobs
- support multiple knowledge types such as architecture overview, module boundaries, symbol relationships, process hints, diff-aware impact, and repo invariants
- prefer incremental refresh over full re-indexing
- keep query results progressively disclosed: summary first, deeper detail on demand

Recommended storage direction:

- `JSON + Markdown hybrid`, with JSON as the primary machine-facing store and Markdown reserved for selected readable summaries
- deterministic artifact layout under `.agent/` so task runs, routing, and evaluation can consume the same truth
- git-aware freshness metadata so KodaX can tell whether intelligence is current enough to trust

Recommended artifact families:

- manifest and freshness metadata
- file and symbol indexes
- module capsules
- process capsules
- changed-scope / diff summaries
- repo invariants and architecture constraints

### Progressive disclosure model

`018` should make repository knowledge consumable in layers:

1. summary capsule
2. scoped expansion capsule
3. evidence layer with exact file paths, symbol references, and relevant excerpts

This keeps the default response cheap while preserving a path to deeper context when the task truly needs it.

### Explicit non-goals

- no dedicated wiki product or standing documentation site
- no required graph database or Cypher interface
- no required visualization UI
- no promise of equal precision across all supported languages

#### Historical user scenarios retained

The earlier draft already captured several scenarios that still belong here:

1. new-project intake that generates an initial knowledge baseline
2. code-understanding queries such as architecture, auth-flow, or module-boundary questions
3. pre-change context loading before edits
4. post-change refresh so durable intelligence stays useful
5. diff-aware change analysis that helps planner and evaluator agree on scope

### Relationship to the task engine

`018` answers:

- what does KodaX know about this codebase
- which capsules are relevant to this task
- which facts are fresh enough to trust
- which invariants should guide planning and evaluation
- which areas and symbols the current change is likely to touch

It should feed:

- planner decomposition and contract building
- generator context loading before edits
- evaluator scope-checking and architectural review

It does not own task routing or verification logic directly, but it becomes one of the main substrates those systems rely on.

### Acceptance criteria

- [x] KodaX can build and refresh durable repository intelligence without requiring a wiki surface
- [x] planning, generation, and evaluation can consume repository intelligence capsules
- [x] repository intelligence is incremental, freshness-aware, and task-relevant
- [x] storage remains inspectable and repo-friendly
- [x] language support exposes explicit capability/confidence truth instead of false parity

### Implemented shape

- durable artifacts under `.agent/repo-intelligence/`
- `repo-overview.json`, `changed-scope.json`, `repo-intelligence-index.json`
- `repo-intelligence-manifest.json` with source fingerprint freshness metadata
- separate `module-index.json`, `symbol-index.json`, and `process-index.json`
- language-tiered symbol/module/process extraction for TS/JS/Python/Go/Java/Rust/C++
- AST-backed TS/JS symbol/import/call extraction via the TypeScript compiler API
- AST-backed Python function/class/method extraction via a bundled Node-side parser, without requiring a local Python runtime
- AST-backed Go/Rust structural symbol/import/call extraction via bundled parsers
- stronger structural method/receiver extraction for Java/C++
- prompt-time repository intelligence injection for task-aware agent context
- automatic active-module and active-impact injection for edit/review/refactor flows
- `025` routing can consume lightweight repo-intelligence signals to raise complexity, bias planning/investigation, and choose a safer harness
- low-confidence guidance that tells the agent when to fall back to `grep`/`read`
- managed-task evidence snapshots under each task workspace for repo overview, changed scope, active module, and impact

## FEATURE_027: Adaptive Multi-Agent Mode Toggle and Team-Mode Sunset

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### 概述

`FEATURE_022` 落地后，adaptive multi-agent execution 已经成为非简单任务的默认路径，但它也带来了更高的 token 成本与更重的执行形态。

`FEATURE_027` 的目标，是把"是否允许 adaptive multi-agent"变成一个正式、可见、可快速切换的产品控制项，而不是让用户只能通过 prompt 暗示、记忆隐含行为，或者误用旧的 `--team`。

这项 feature 需要同时回答 3 个问题：

1. 用户怎么显式切到 `Single-Agent`，以降低 token 消耗。
2. 用户怎么一眼看出当前是 `SA` 还是 `AMA`。
3. 旧的 `--team` 手动多 agent 模式，如何从产品表面退出，而不和新的 adaptive task engine 语义冲突。

### Goals

1. 提供正式的 agent-mode 开关，用于切换 `SA` 和 `AMA`。
2. 让当前模式在状态栏和启动信息中持续可见，而不是隐藏状态。
3. 支持低摩擦切换，包括 CLI、REPL 命令和快捷键。
4. 让 `SA` 真正降低 adaptive multi-agent 的 token 消耗，而不是只改 UI 标签。
5. 把 `--team` 从面向用户的主产品语义中移除或降级。

### Scope

- 持久化的 `agent mode` 偏好设置
- CLI 入口，例如 `--agent-mode sa|ama`
- REPL 命令入口，例如 `/agent-mode [sa|ama|toggle]`
- 快捷键切换，首选 `Alt+M`，并保留命令行回退入口
- 状态栏第一列与启动 banner 的模式显示
- routing / harness profile 的约束逻辑
- `--team` 的弃用、隐藏或移除策略

### Mode semantics

#### `AMA`

`AMA` = `Adaptive Multi-Agent`

这是默认模式，task engine 仍然会根据 task type、complexity、risk、provider capability 在 `H0 / H1 / H2 / H3` 之间自动选择。

#### `SA`

`SA` = `Single-Agent`

`SA` 不关闭 task routing、metadata、provider policy 或 managed-task artifact；它关闭的是 adaptive multi-agent role split。

换句话说：

- routing 仍然发生，用于理解任务和保留 task truth
- 但执行路径会被钳制到 single-agent execution
- 不再默认展开 `planner / generator / evaluator` 或 `lead / worker / validator` 这类角色链

设计约束：

- `SA` 的目标是明显减少 token 消耗，而不是只减少 UI 噪声
- `SA` 不应禁用显式、确定性的 verification，例如 project harness 或单独的 `/project verify`

### UX contract

需要明确一个统一的显示约定：

- 状态栏第一列显示 `KodaX - AMA` 或 `KodaX - SA`
- 模式切换后，状态栏立即刷新
- 启动 banner 和关键帮助输出也应反映当前模式
- 如果模式被持久化，恢复 session 后应保持一致

这项设计的目标不是"增加一个深层设置"，而是把当前执行语义变成用户能看见、能控制的状态。

### Command and shortcut shape

建议的控制面：

- CLI: `--agent-mode sa|ama`
- REPL: `/agent-mode`, `/agent-mode sa`, `/agent-mode ama`, `/agent-mode toggle`
- 快捷键: `Alt+M`

快捷键不是唯一入口，而是最高频入口。
如果终端环境不稳定或不支持 `Alt+M`，用户仍然必须能通过 REPL 命令完成相同切换。

### Relationship to `--team`

`--team` 与 `AMA` 不是一回事：

- `AMA` 是 task engine 的自适应执行模式
- `--team` 是旧的、手动指定多个 sub-agent 并行运行的入口

继续同时把两者都当成产品级模式，只会制造语义混乱。

因此 `027` 的方向应该是：

1. 不再把 `--team` 作为主产品故事的一部分
2. 从 help、文档、主流程中移除或弱化 `--team`
3. 如果短期不能彻底删除，也至少改成明确的 deprecated compatibility path
4. 保留 orchestration primitives 作为内部 runtime 能力，而不是继续保留 `--team` 作为未来入口

### Explicit non-goals

- 不新增第三种"手动多 agent 模式"来和 `AMA` 并列
- 不把 `SA` 设计成新的 prompt hack
- 不要求用户每次启动都重新选择模式
- 不把 `/parallel` 误当成 multi-agent 开关

### Acceptance criteria

- [x] 用户可以通过正式配置或运行时入口在 `SA` / `AMA` 之间切换
- [x] 状态栏第一列会明确显示 `KodaX - SA` 或 `KodaX - AMA`
- [x] 快捷键切换可用，且存在等价的命令回退入口
- [x] `SA` 会阻止 adaptive multi-agent role split，从而降低 token 消耗
- [x] `AMA` 继续保留当前 task-engine 的自适应 harness 选择
- [x] `--team` 被移除、隐藏，或至少变成明确的 deprecated 兼容入口

### Implementation summary

`FEATURE_027` 现在已经落地成一个正式的产品控制面，而不是文档里的设想：

- `KodaXOptions` 与用户配置里已经有持久化的 `agentMode`
- CLI 支持 `--agent-mode sa|ama`
- REPL 支持 `/agent-mode`, `/agent-mode sa`, `/agent-mode ama`, `/agent-mode toggle`
- 默认快捷键 `Alt+M` 可在运行中切换 `SA / AMA`
- 状态栏第一列显示 `KodaX - SA` / `KodaX - AMA`
- 启动 banner 也会显示当前模式，避免"切换了但看不出来"
- `SA` 不会关闭 task routing 或 managed-task artifact，而是把执行钳制到 single-agent path，避免展开 adaptive role split
- `AMA` 保留 `H0/H1/H2/H3` 自适应 harness 选择，并允许 `FEATURE_022` 的显式 refinement loop 生效
- `--team` 退出主产品语义，保留为 deprecated compatibility path；命令行仍可见其历史入口，但实际调用会直接提示弃用并拒绝继续执行

## FEATURE_028: First-Class Retrieval, Context, and Evidence Tooling

**Status**: Completed
**Priority**: High
**Category**: Enhancement

### Overview

KodaX needs retrieval tools that are stronger than shell fallbacks, more task-shaped than raw search, and more evidence-aware than ad hoc fetches.

`FEATURE_028` is the runtime query layer that turns local repository intelligence plus live evidence sources into structured, progressive, reusable result blocks for the task engine.

### Goals

1. Provide structured local-repo retrieval before broad shell-first exploration.
2. Preserve attribution, freshness, confidence, and evidence metadata.
3. Make retrieval outputs reusable by planner, generator, evaluator, and managed task artifacts.
4. Reduce ad hoc `grep -> read -> grep again` loops by returning task-shaped context bundles.
5. Support progressive disclosure so KodaX can stay token-efficient by default.

### Scope

- structured repository-intelligence queries
- structured code search and targeted expansion
- evidence-carrying result blocks
- retrieval freshness/confidence signaling and disclosure budgets
- task-evidence persistence for repo-intelligence snapshots

### Recommended user-facing query surfaces

The first useful query surfaces for `028` should be:

- `repo_overview`
- `module_context`
- `symbol_context`
- `process_context`
- `impact_estimate`
- `changed_scope`

These should prefer deterministic local repository intelligence first, then targeted code search, then broader fetch or web search when the task actually requires new external evidence.

### Detailed design notes retained

Useful previous direction that should remain attached to this feature:

- repository-aware context retrieval instead of only generic search
- code search that is stronger than shell-first grep flows
- evidence blocks that preserve source, freshness, summary, confidence, and reuse potential
- explicit room to add permissioned external research workflows later, without making them part of the v0.8.0 acceptance boundary

#### Result shape direction

Each retrieval result should be shaped for direct task-engine reuse rather than raw display:

- a short summary
- confidence and freshness notes
- evidence references
- suggested follow-up handles for deeper drill-down
- optional file or symbol excerpts only when needed

This keeps retrieval useful for both small-model execution and evaluator-led review loops.

### Relationship to other features

- `018` stores and refreshes repository intelligence
- `028` queries that intelligence and mixes it with live evidence when needed
- `025` may route research-heavy tasks toward search-capable harnesses
- `035` can later expose additional retrieval-capable providers through the same structured result shape

### Acceptance criteria

- [x] KodaX can query repository intelligence and targeted code evidence without raw shell dependence as the default path
- [x] retrieval outputs preserve attribution, freshness, and confidence metadata
- [x] retrieval outputs can feed task evidence, evaluation, and intelligence refresh
- [x] progressive disclosure keeps retrieval token-efficient by default
- [x] research-heavy tasks become meaningfully easier to execute and verify

### Implemented query surfaces

- `repo_overview`
- `module_context`
- `symbol_context`
- `process_context`
- `impact_estimate`
- `changed_scope`

Each surface now returns a short reusable capsule with freshness, confidence, evidence, and obvious next drill-down handles.

Managed tasks now persist task-scoped retrieval snapshots into their evidence bundle so planner/evaluator outputs can be reviewed against the same repository-intelligence snapshot used during execution.

## Cross-feature intent for v0.7.10

1. `018` makes KodaX maintain durable, task-aware repository intelligence.
2. `027` makes the adaptive multi-agent engine user-controllable and visible at the UX layer.
3. `028` makes KodaX retrieve the right local or external evidence with the right granularity.
