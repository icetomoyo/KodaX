# v0.7.21 Feature Designs

> 中文标题：`v0.7.21 Feature 设计`
>
> Version: `v0.7.21`
>
> Status: Planned
>
> Theme: prompt input lifecycle hardening

---

## 总览

`v0.7.21` 是一次克制的点版本，只做一件事：让 prompt 历史在 `Ctrl+O` 来回切换之后仍然可以用 Up 键召回。

当前 `<PromptComposer>` 只存在于 `promptFooterSurface` 这一个渲染槽；进入 transcript 模式时 `currentFooterSurface` 整体切到 `transcriptFooterSurface`，React 在同一位置看到不同的组件类型，于是把 PromptComposer 子树 unmount。它里面持有的 `useInputHistory` 状态跟着一起被 GC——**等价于把会话级数据当成组件级数据存了**。

`FEATURE_077` 把"历史条目数组"这一条最小必要的数据抬到模块作用域；其余导航位置和草稿占位保持组件作用域，避免引入任何新的行为。

---

## FEATURE_077: Session-Scoped Prompt Input History

**Status**: Planned (v0.7.21)
**Priority**: Medium
**Category**: Internal / UX Hardening
**Root-cause scope**: `@kodax/repl` —— 仅 `ui/hooks/useInputHistory.ts`

### Problem statement

#### 症状

已用 repro 验证过的触发路径：

1. 启动 `kodax`，submit `A`、`B`、`C`。按 Up 三次可依次回溯 `C → B → A`。
2. 打 `hello` 不提交。
3. 按一次 Ctrl+O（进 transcript）。
4. 再按一次 Ctrl+O（回 prompt）。
5. **观察到**：`hello` 消失；按 Up 没有任何反应。
6. submit 一条新 prompt `test`，再按 Up 能召回 `test`，但 `A/B/C` 永久找不回来。

#### 根因

| 环节 | 行为 | 是否 bug 源 |
|---|---|---|
| `<PromptComposer>` 只存在于 [`InkREPL.tsx:6649`](../../packages/repl/src/ui/InkREPL.tsx#L6649) 这一处 | 渲染树里唯一位置 | 前置条件 |
| [`InkREPL.tsx:6843-6845`](../../packages/repl/src/ui/InkREPL.tsx#L6843-L6845) `currentFooterSurface = isTranscriptMode ? transcriptFooterSurface : promptFooterSurface` | Ctrl+O 触发整段切换 | 前置条件 |
| [`PromptFooter.tsx:111`](../../packages/repl/src/ui/components/PromptFooter.tsx#L111) 直接 `{composer}` 渲染 | 同一 slot 出现不同组件类型 → React 销毁旧树 | 前置条件 |
| [`useInputHistory.ts:30`](../../packages/repl/src/ui/hooks/useInputHistory.ts#L30) 用 `useState<HistoryEntry[]>([])` | 状态挂在组件实例上 | **真正的 bug 点** |
| [`useInputHistory.ts:70`](../../packages/repl/src/ui/hooks/useInputHistory.ts#L70) `if (history.length === 0) return null` | remount 后命中 → 静默无反应 | 表现层 |

键盘事件、focus、handleKey、`shouldUseHistoryNavigation`、`navigateUp` 调用都正常。唯一的失败点是 `history` 数组是组件 state，而 prompt 历史在语义上是**会话级**数据。

### Goals

1. 让 prompt 历史在 `<PromptComposer>` unmount 后幸存。
2. 不引入任何用户可感知的行为变化（不改 Up/Down 语义、不改 submit 语义、不改 Ctrl+R 搜索语义）。
3. 改动面最小、无新抽象、无新 API。

### Non-goals

- 不修复 text buffer（未 submit 的草稿文本）在 Ctrl+O 后被清空——见 §Out of scope。
- 不加跨会话持久化（load/save 到磁盘）。
- 不改 transcript surface 切换机制。
- 不引入 Context Provider / 新 hook / 新 context——见 §Alternatives considered。

### Design

#### 最小必要改动：仅把 `history` 数组抬到模块作用域

`useInputHistory` 原来有三块 state：

| 变量 | 现在的作用域 | 语义 | remount 后需要保留吗？ |
|---|---|---|---|
| `history: HistoryEntry[]` | `useState` 组件级 | 所有已 submit 过的 prompts | **需要**（bug 的核心） |
| `historyIndexRef: number` | `useRef` 组件级 | 导航时的当前位置（`-1` = 未导航） | 不需要（submit 后本来就重置为 `-1`） |
| `tempInputRef: string` | `useRef` 组件级 | 首次按 Up 前的草稿文本 | 不需要（text buffer 也是组件级，两者保持一致） |

因此**只把 `history` 数组抬到模块作用域**，`historyIndexRef`/`tempInputRef` 保持 `useRef`——这样 remount 后 nav 位置自动重置为 `-1`，Up 键从"最近一条"开始，符合直觉。

```ts
// packages/repl/src/ui/hooks/useInputHistory.ts

import { useCallback, useRef } from "react";
import type { HistoryEntry } from "../types.js";

// 模块级：会话内的 prompt 历史数组
// 原因：<PromptComposer> 在 Ctrl+O 切换时会 unmount+remount，组件级 state 会随之 GC
// 见 docs/features/v0.7.21.md FEATURE_077
const historyStore: HistoryEntry[] = [];

/** 仅供测试使用，生产代码不应调用 */
export function __resetInputHistoryForTesting(): void {
  historyStore.length = 0;
}

export interface UseInputHistoryOptions {
  maxSize?: number;
  onSave?: (entry: HistoryEntry) => void;
}

export interface UseInputHistoryReturn {
  add: (text: string) => void;
  navigateUp: () => string | null;
  navigateDown: () => string | null;
  reset: () => void;
  saveTempInput: (text: string) => void;
}

export function useInputHistory(options: UseInputHistoryOptions = {}): UseInputHistoryReturn {
  const { maxSize = 1000, onSave } = options;
  const historyIndexRef = useRef<number>(-1);
  const tempInputRef = useRef<string>("");

  const add = useCallback((text: string) => {
    if (!text.trim()) return;
    const last = historyStore[historyStore.length - 1];
    if (last?.text !== text) {
      const entry: HistoryEntry = { text, timestamp: Date.now() };
      historyStore.push(entry);
      if (historyStore.length > maxSize) {
        historyStore.splice(0, historyStore.length - maxSize);
      }
      onSave?.(entry);
    }
    historyIndexRef.current = -1;
    tempInputRef.current = "";
  }, [maxSize, onSave]);

  const navigateUp = useCallback((): string | null => {
    if (historyStore.length === 0) return null;
    if (historyIndexRef.current === -1) {
      historyIndexRef.current = historyStore.length - 1;
      return historyStore[historyIndexRef.current]?.text ?? null;
    }
    if (historyIndexRef.current > 0) {
      historyIndexRef.current--;
      return historyStore[historyIndexRef.current]?.text ?? null;
    }
    return null;
  }, []);

  const navigateDown = useCallback((): string | null => {
    if (historyIndexRef.current === -1) return null;
    if (historyIndexRef.current < historyStore.length - 1) {
      historyIndexRef.current++;
      return historyStore[historyIndexRef.current]?.text ?? null;
    }
    historyIndexRef.current = -1;
    return tempInputRef.current;
  }, []);

  const reset = useCallback(() => {
    historyIndexRef.current = -1;
    tempInputRef.current = "";
  }, []);

  const saveTempInput = useCallback((text: string) => {
    if (historyIndexRef.current === -1) {
      tempInputRef.current = text;
    }
  }, []);

  return { add, navigateUp, navigateDown, reset, saveTempInput };
}
```

#### 额外一条顺带的小清理：收紧返回类型

当前 `UseInputHistoryReturn` 导出了 `history: HistoryEntry[]`、`currentIndex: number`、`getPrevious`、`getNext` 四项——**grep 全仓零消费**。本 feature 顺手把返回类型收紧到只保留实际被使用的 5 个函数。这不是为本 bug 做的，但因为同文件改动，一起处理避免 API 残留污染。

| 返回字段 | 全仓消费情况 | 处理 |
|---|---|---|
| `history` | 无 | **删** |
| `currentIndex` | 无 | **删** |
| `add` | `prompt-input-controller.ts:155` | 保留 |
| `navigateUp` | 同上 | 保留 |
| `navigateDown` | 同上 | 保留 |
| `reset` | 同上 | 保留 |
| `getPrevious` | 无 | **删** |
| `getNext` | 无 | **删** |
| `saveTempInput` | 同上 | 保留 |

### Test strategy

#### 新增：`useInputHistory.remount.test.ts`（回归门）

```ts
import { renderHook, act } from "@testing-library/react";
import { afterEach, describe, expect, it } from "vitest";
import {
  useInputHistory,
  __resetInputHistoryForTesting,
} from "./useInputHistory.js";

describe("useInputHistory — session-scoped history across remount", () => {
  afterEach(() => {
    __resetInputHistoryForTesting();
  });

  it("preserves history when the consuming component unmounts and remounts", () => {
    // 第一次 mount：积累历史
    const first = renderHook(() => useInputHistory());
    act(() => first.result.current.add("A"));
    act(() => first.result.current.add("B"));
    first.unmount();

    // 模拟 Ctrl+O 触发的 PromptComposer unmount+remount
    const second = renderHook(() => useInputHistory());

    // 新 mount 应该看到此前积累的 history
    let value: string | null = null;
    act(() => { value = second.result.current.navigateUp(); });
    expect(value).toBe("B");
    act(() => { value = second.result.current.navigateUp(); });
    expect(value).toBe("A");
  });

  it("resets navigation cursor on remount (index/tempInput are component-scoped)", () => {
    const first = renderHook(() => useInputHistory());
    act(() => first.result.current.add("A"));
    act(() => first.result.current.add("B"));
    act(() => first.result.current.navigateUp()); // index -> 1, returns "B"
    first.unmount();

    // remount 后 index 应该回到 -1（而不是残留 1）
    const second = renderHook(() => useInputHistory());
    let value: string | null = null;
    act(() => { value = second.result.current.navigateUp(); });
    expect(value).toBe("B"); // 如果 index 残留为 1，这里会返回 "A"
  });

  it("__resetInputHistoryForTesting clears the module store", () => {
    const { result } = renderHook(() => useInputHistory());
    act(() => result.current.add("A"));
    __resetInputHistoryForTesting();
    expect(result.current.navigateUp()).toBeNull();
  });
});
```

两条核心断言是本 feature 的**回归守门员**：第一条防止未来重构把 history 放回组件 state；第二条防止把 `historyIndexRef` 一起抬到模块级（会引入 nav 位置漂移）。

#### 已有测试的影响面

- `prompt-input-controller.test.ts` —— 用 mock，不真实走 `useInputHistory`，**零改动**。
- `App.test.tsx` / `InputPrompt.test.tsx` —— 若有断言涉及 history 行为，需要在 test setup 的 `afterEach` 里调 `__resetInputHistoryForTesting`，否则测试间串。这是模块级状态的标准 test hygiene。

### Alternatives considered

#### A. Context Provider（`InputHistoryProvider` + `useInputHistoryContext`）

在 `InkREPL` 根部挂一个 Provider 调用 `useInputHistory`，向下通过 Context 提供。沿用 `KeypressProvider` / `UIStateProvider` 模式。

**否决理由：**

- **零多 consumer 场景**：grep 全仓，`useInputHistory` 只有 1 个消费者（`prompt-input-controller.ts:155`）。CLAUDE.md 明确要求"NEVER create abstractions until you have 3+ concrete use cases"。
- **"匹配现有 provider 模式"不是充分条件**：`KeypressProvider` / `UIStateProvider` / `StreamingProvider` / `AutocompleteContextProvider` 每个都服务多个 consumer；模式对齐本身不是引入新抽象的正当动机。
- **新增 3 个导出名 + 1 个新文件 + 多一层间接跳转**，LLM 和人工读者都要多跳两步才能理解数据流向。直接读单个 hook 文件只要一跳。
- **并未为未来真正省成本**：即便将来加跨会话持久化，也是在 hook 内部加 init + `onSave` 两行，和 Context 架构在 provider 里加 `useEffect` 在成本上等价。"未来扩展性"在此处是幻觉。

FEATURE_073（Reference-Style Lineage）被 cancel 的首要理由就是 YAGNI——同样的判据应用到这里，Context Provider 也不应通过。

#### B. 纯模块级（把 `historyIndex` 和 `tempInput` 一并抬上去）

写起来更简洁，但**引入一个用户可能察觉的行为变化**：Ctrl+O 切换后再按 Up，导航位置会接续切换前的 `historyIndex` 而不是从最近一条重新开始。

否决理由：本 feature 承诺"不改变用户可感知的行为"。历史位置跨 toggle 漂移是一个新副作用，即便只是微小的 UX 偏移，也不值得搭载在这次 bug fix 里。

#### C. 在 Ctrl+O toggle 回调里把 history 显式存到一个 ref 再还原

需要在 `toggleTranscriptMode` / `exitTranscriptModeSurface` 里多加两条保存/恢复代码，跨越 hook 边界耦合 transcript state 和 input state——不仅没比模块级方案少几行，还引入跨模块耦合。否决。

### Out of scope

1. **Text buffer（未 submit 草稿）在 Ctrl+O 后被清空**  
   同源问题（`useTextBuffer` 也是组件本地 state），但涉及 cursor position / selection / undo-redo 栈 / paste buffer，上移风险面远大。用户主要痛点是历史召回，不是草稿保留。可作为独立 FEATURE_078+ 评估。

2. **跨会话历史持久化（`~/.kodax/history.jsonl`）**  
   `useInputHistory` 已经有 `onSave` 钩子。接入只需要在 hook 首次运行时从磁盘读入 `historyStore`（可以放到 module top-level 或一个 lazy init），并在 `onSave` 里追加写。本 feature **不实现**，但改完之后这条路径是畅通的。

3. **`<PromptComposer>` 的 unmount 本身**  
   理论上另一条路是让 PromptComposer 永远 mount、只换可见性。涉及 footer surface 的布局语义改动，风险大。保留现语义，只把会话级状态挪出 composer 生命周期。

### Verification plan

#### 手动测试（必须通过）

| # | 步骤 | 预期 |
|---|---|---|
| M1 | submit `A/B/C` → 按 Up 三次 | 依次显示 `C/B/A` |

| M3 | 在 M2 之后按 Up | **依次显示 `C/B/A`**（本 feature 的核心回归门） |
| M4 | 连续切换 Ctrl+O 10 次之后再按 Up | 历史仍完整可召回 |

| M6 | 按 Ctrl+R 历史搜索 `A` | 正常匹配到 `A`（验证没破坏 Ctrl+R 路径） |

### Rollout & risk

| 风险 | 评估 | 缓解 |
|---|---|---|
| 模块级数组在测试间串 | 中 | `__resetInputHistoryForTesting` + `afterEach` 覆盖；新测试文件已经这么做 |
| 多 REPL 实例串数据 | 不存在 | Ink 一个进程一个 REPL，CLAUDE.md 明确不为假想多实例铺路 |
| `onSave` 现在在 `add` 里触发，原来是 `setHistory` 回调触发 | 低 | 时序一致（都在 `add` 同步调用内）；项目当前无 `onSave` 使用者，行为变化不可观测 |
| 收紧返回类型破坏第三方代码 | 零 | 全仓只有一个消费者，只用 5 个函数；被删的 4 个字段全部无人读 |
| React 18 concurrent rendering 下的模块 state race | 不适用 | Ink 不走 concurrent mode |
| Fast Refresh / HMR 下模块被重载导致 history 被清空 | 低 | 仅开发态现象，cold start 后等效 |
| 回滚成本 | 低 | 单文件改动，git revert 即可 |

### Relationship to prior work

- `FEATURE_051`（v0.7.25 unreleased）冻结了 prompt/footer 可见 surface 档位——本 feature 严格遵循，不扩任何 visible surface。
- `FEATURE_055`（v0.9.0 unreleased）的 REPL substrate hardening 是同方向上更广的收口；本 feature 是最窄的一次切片，只解决一个具体 UX 回归。
- 与 `FEATURE_069`（Session Rewind & Shell Completion）虽然都涉及"跨 round 的用户状态"，但 069 是 session 级 rewind 语义，077 只收口 prompt history 一条数据流，无交叉。
