## 总览

`v0.7.21` 是一次克制的点版本，专门收口 prompt input 层面的生命周期耦合问题。

当前 `<PromptComposer>` 只存在于 `promptFooterSurface` 这一个渲染槽；一旦进入 transcript 模式（`Ctrl+O`），`currentFooterSurface` 会整体切换到 `transcriptFooterSurface`，React 在同一位置看到不同的组件类型（`PromptComposer` → `TranscriptModeFooter`），必然 unmount + remount。

这次 unmount 把本应属于**会话级**的 prompt 历史当成了**组件级**状态给扔掉了——用户体感就是"按了一次 Ctrl+O，上键就再也调不出历史了"。

`FEATURE_077` 的目标是把 prompt 历史从 composer 生命周期里解耦出来，以后不论 transcript 模式怎么切换、composer 实例被销毁几次，历史都留在会话层面。

## FEATURE_077: Session-Scoped Prompt Input History

**Status**: Planned (v0.7.21)
**Priority**: Medium
**Category**: Internal / UX Hardening
**Builds on**: `FEATURE_051` (v0.7.25 host-aware TUI substrate) 的 `usePromptInputController` 成熟体
**Root-cause scope**: `@kodax/repl` UI — `ui/hooks/useInputHistory.ts`, `ui/utils/prompt-input-controller.ts`, `ui/InkREPL.tsx`
**Companion reference**: Claude Code 的 input history 在 `KeypressManager` sibling scope 持有，不挂在 prompt surface 上——KodaX 此前缺这层隔离

### Problem statement

#### 症状

用户触发路径（已经用 hello 测试验证过）：

1. 启动 `kodax`，提交 prompt `A`、`B`、`C`。此时按 Up 可以依次回溯 `C → B → A`。
2. 在输入框里打 `hello`（不提交）。
3. 按 `Ctrl+O` 进入 transcript 模式。
4. 再按 `Ctrl+O` 回到 prompt 模式。
5. **观察到：** 输入框里的 `hello` 消失；按 Up 没有任何反应（既不报错也不回显历史）。
6. 此时如果提交一条新 prompt `test`，再按 Up 又能召回 `test`——但 `A/B/C` 永远找不回来了。

#### 根因

- `<PromptComposer>` 只在 [`InkREPL.tsx` 第 6649 行](../../packages/repl/src/ui/InkREPL.tsx#L6649) 一处渲染，嵌在 `promptFooterSurface` 里。
- [`InkREPL.tsx` 第 6843-6845 行](../../packages/repl/src/ui/InkREPL.tsx#L6843-L6845) 的 `currentFooterSurface = isTranscriptMode ? transcriptFooterSurface : promptFooterSurface` 会随 `Ctrl+O` 无条件在两个 surface 之间切换。
- `<PromptFooter>` 只是把 `composer` 当 React.ReactNode 直接 `{composer}` 渲染（[`PromptFooter.tsx` 第 111 行](../../packages/repl/src/ui/components/PromptFooter.tsx#L111)）；同一位置的子元素类型变了（`PromptComposer` vs `TranscriptModeFooter`）→ React 必然销毁旧树、挂载新树。
- Prompt 历史由 [`useInputHistory.ts`](../../packages/repl/src/ui/hooks/useInputHistory.ts) 里的 `useState<HistoryEntry[]>([])` + 两个 `useRef` 共同持有，而这个 hook 被 [`prompt-input-controller.ts:155`](../../packages/repl/src/ui/utils/prompt-input-controller.ts#L155) 在 `<PromptComposer>` 子树内部调用——`<PromptComposer>` unmount 时，hook 的 state 随组件实例一起被 GC。
- Remount 后 `history === []`，[`useInputHistory.ts:70`](../../packages/repl/src/ui/hooks/useInputHistory.ts#L70) 的 `navigateUp()` 静默 `return null`，[`prompt-input-controller.ts:320-322`](../../packages/repl/src/ui/utils/prompt-input-controller.ts#L320-L322) 的 `if (historyText !== null) setText(historyText);` 不命中——**键盘事件到位了，调用链也走完了，只是什么都没发生**。

#### 为什么这是架构问题而不是 surgical fix 问题

Prompt history 语义上就是**一次 REPL 会话的 session-level state**。把它绑在 `<PromptComposer>` 这个"某一次 footer 渲染的实例"上，是层级放错了——这也是为什么 transcript 切换能把它擦干净。

同一类别还有 `useTextBuffer`（unsubmitted 草稿文本）也被同样波及（测试 A 里 `hello` 消失就是这一条）。本 feature **只处理 history**，把草稿文本另做风险评估。见 §Out of scope。

### Goals

1. Prompt history 从 `<PromptComposer>` 生命周期里剥离，存活周期与 `InkREPL` 进程一致。
2. 切换 `Ctrl+O` 不再擦除历史；任意次数 transcript mode 进出之后 Up 键依然能召回最近 submit 过的 prompts。
3. 不新增 visible surface，不改变 `Ctrl+O` / Up / Down / Ctrl+R 的用户交互语义。
4. 遵循项目已有的 provider 模式（`KeypressProvider` / `UIStateProvider` / `StreamingProvider` / `AutocompleteContextProvider`），不引入新的状态管理抽象。
5. 保留未来接入跨会话持久化（磁盘 load / save）的清晰接入点，但本 feature **不实现**持久化。

### Design

#### 架构选择：Context Provider

在 `packages/repl/src/ui/contexts/` 新增 `InputHistoryContext.tsx`，沿用 `KeypressContext` 的模式：provider 在 `InkREPL` 外层调用一次 `useInputHistory()`，通过 context 向下游暴露。

选 Context 而不选 props 穿透的原因：

- `useInputHistory` 的返回值要从 `InkREPL` 根节点传到 `<PromptComposer>` → `<InputPrompt>` → `usePromptInputController`，中间有 3 层组件，每层都要多加 5 个 prop——跟项目已有的 `KeypressProvider` 风格不一致。
- `usePromptInputController` 签名已经有 9 个 option，再加 5 个 history 函数会让签名显著变胖。
- 历史状态跟 `KeypressManager` / `UIState` / `StreamingManager` 在职责层级上对等（都是 session-level UI infrastructure），应当走同一条通道。

#### InputHistoryContext 草图

```tsx
// packages/repl/src/ui/contexts/InputHistoryContext.tsx
import React, { createContext, useContext, type ReactNode } from "react";
import {
  useInputHistory,
  type UseInputHistoryOptions,
  type UseInputHistoryReturn,
} from "../hooks/useInputHistory.js";

const InputHistoryContext = createContext<UseInputHistoryReturn | null>(null);

export interface InputHistoryProviderProps {
  children: ReactNode;
  options?: UseInputHistoryOptions;
}

export function InputHistoryProvider({
  children,
  options,
}: InputHistoryProviderProps): React.ReactElement {
  const value = useInputHistory(options);
  return React.createElement(
    InputHistoryContext.Provider,
    { value },
    children,
  );
}

export function useInputHistoryContext(): UseInputHistoryReturn {
  const ctx = useContext(InputHistoryContext);
  if (!ctx) {
    throw new Error(
      "useInputHistoryContext must be used within an InputHistoryProvider",
    );
  }
  return ctx;
}
```

#### InkREPL wrapper 整合位置

```tsx
// InkREPL.tsx（第 6987 行附近）
const InkREPL: React.FC<InkREPLProps> = (props) => {
  const cwd = process.cwd();
  const gitRoot = props.options?.context?.gitRoot ?? undefined;

  return (
    <UIStateProvider>
      <StreamingProvider>
        <KeypressProvider>
          <ShortcutsProvider>
            <AutocompleteContextProvider cwd={cwd} gitRoot={gitRoot}>
              <InputHistoryProvider>{/* ← 新增 */}
                <InkREPLInner {...props} />
              </InputHistoryProvider>
            </AutocompleteContextProvider>
          </ShortcutsProvider>
        </KeypressProvider>
      </StreamingProvider>

```

Provider 放在 `AutocompleteContextProvider` 内侧、`InkREPLInner` 外侧的原因：它只需要挡住 `InkREPLInner` 内部任何 remount；和 autocomplete 是并列关系（autocomplete 状态本来就在外层持有），无前后依赖。

#### prompt-input-controller 切点

```ts
// prompt-input-controller.ts 顶部
- import { useInputHistory } from "../hooks/useInputHistory.js";
+ import { useInputHistoryContext } from "../contexts/InputHistoryContext.js";

// 第 155 行
```

### Test strategy

#### 1. 新增单元测试 `InputHistoryContext.test.tsx`

- Provider 正常渲染子组件
- `useInputHistoryContext` 在 Provider 外使用时抛 error
- Provider 内的两个消费者看到**同一个** history 状态（即：consumer A 调 `add("x")`，consumer B 调 `navigateUp()` 能返回 `"x"`）

#### 2. 回归测试（核心）

新增 `InputHistoryContext.remount.test.tsx`：

```tsx
// 伪代码
function Consumer() {
  const { add, navigateUp } = useInputHistoryContext();
  // ... exposes add/navigateUp via test harness
}

// Test:
// 1. Render <InputHistoryProvider><ConditionalMount><Consumer/></ConditionalMount></InputHistoryProvider>
// 2. Call add("A"), add("B")
// 3. Unmount ConditionalMount（模拟 PromptComposer 被销毁）
// 4. Remount ConditionalMount
// 5. Expect navigateUp() === "B", navigateUp() === "A"
```

这条测试是本 feature 的**回归守门员**——只要这条测试在，以后任何重构都不能再把 history 藏回 composer 子树。

#### 3. 已有测试的影响面

- `prompt-input-controller.test.ts` 用的是 mock，不真实调 `useInputHistory`——**无需改动**。
- `InputPrompt.test.tsx` 如果断言 `<InputPrompt>` 挂载时不依赖外部 provider，则需要在测试里补包一层 `<InputHistoryProvider>`（大概率只改 render helper）。
- `App.test.tsx` 已经通过 `runInkInteractiveMode` 路径走顶层 wrapper，provider 自动覆盖，无需改动。

### Out of scope

本 feature **有意识地不做**以下三件事，避免 scope creep：

1. **Unsubmitted 草稿文本保留**  
   测试 A 里 `hello` 消失是同源问题——`useTextBuffer` 也是组件本地 state。但 text buffer 除了文本还有光标位置、选区、undo/redo 栈、paste buffer、`isPasting` 瞬态 flag。整体上移会显著扩大改动面和回归风险，且用户抱怨主要集中在"上键召不回历史"，不在"草稿消失"。  
   **处理：** 本 feature 之后单独评估是否要补一个 `FEATURE_078: Composer Draft Persistence`，允许简单的 text 快照保留（不带 cursor/undo 复杂度）。

2. **跨会话历史持久化**  
   `useInputHistory` 已经有 `onSave` 钩子，`InputHistoryProvider` 也可以通过 `options` 传入——但本 feature 不接 disk I/O、不定义存储格式、不做迁移。  
   **处理：** 如果将来要做（与 `FEATURE_069` Session Rewind & Shell Completion 同类），在 `InputHistoryProvider` 里加一个 `useEffect` 读 `~/.kodax/history.jsonl` 作为 initial value 即可——本 feature 的架构已经让这件事变成纯粹的后续接线工作。

3. **`currentFooterSurface` 切换造成 `<PromptComposer>` unmount 本身**  
   理论上另一条修复路径是不再让 footer surface 整体切换，而是让 `<PromptComposer>` 始终 mount、靠可见性控制遮挡。这涉及 transcript footer / prompt footer 的布局语义改动，风险远大于本方案。  
   **处理：** 保留现有 surface 切换语义，只把容易丢失的 session state 抬到 composer 生命周期之上。

### Rollout & risk

- **兼容性**：`useInputHistory` hook 本身未改，未被 provider 包裹的调用点（如果有第三方直接 import 它）行为不变。项目内只有一处调用（`prompt-input-controller.ts:155`），不存在第三方风险。
- **失败模式**：如果 Provider 漏包（哪条渲染路径绕过 wrapper），`useInputHistoryContext` 会抛"must be used within an InputHistoryProvider"——这是**显式 fail-fast**，不会退化成静默 bug，比当前的"Up 静默无反应"更易诊断。
- **回滚成本**：四处改动全是新增 + 一行替换，回滚只需撤掉 provider + 切回 `useInputHistory()` 直调，原地可复。
